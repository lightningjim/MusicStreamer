# MusicStreamer — Flatpak Build

## Overview

This directory contains the Flatpak build infrastructure for MusicStreamer. The Flatpak
distributes as a sideload-installable `.flatpak` bundle and is a second Linux packaging
format alongside the AppImage (Phase 85).

- **App ID:** `io.github.kcreasey.MusicStreamer`
- **Runtime:** `org.kde.Platform//6.8` + `io.qt.PySide.BaseApp//6.8`
- **Codecs:** `org.freedesktop.Platform.ffmpeg-full//24.08` (AAC/MP3)
- **Manifest:** `io.github.kcreasey.MusicStreamer.yaml` (repo root)

## Open Question Empirical Findings

### OQ1: Internal Freedesktop SDK version of org.kde.Platform//6.8

Probe command run during Phase 86 Plan 01 Task 1:
```bash
flatpak run --command=bash org.kde.Platform//6.8 -c \
  "cat /usr/manifest.json | python3 -c 'import json,sys; d=json.load(sys.stdin); print(d.get(\"runtime-version\", \"N/A\"))'"
```

**Result:** `24.08`

This confirms that `org.kde.Platform//6.8` is built on Freedesktop SDK **24.08**. The
extensions `org.freedesktop.Platform.ffmpeg-full//24.08` and
`org.freedesktop.Sdk.Extension.node20//24.08` are correctly versioned and will align
with the KDE 6.8 runtime (assumptions A4 and A6 confirmed).

### OQ3: flatpak-pip-generator --runtime flag

`flatpak-pip-generator --runtime='org.kde.Sdk//6.8'` was used successfully (preferred
over `org.freedesktop.Sdk//24.08`). The tool resolves the python3 version from the
specified SDK to ensure ABI compatibility for any compiled pip extensions.

Note: `flatpak-pip-generator` was installed via `uv tool install flatpak-pip-generator`
(version 2026.5.28) and invoked as `~/.local/bin/flatpak_pip_generator`.

## Prerequisites

Install the build toolchain and runtimes (one-time setup):

```bash
# Install flatpak-builder (as a Flatpak — no sudo required)
flatpak install -y flathub org.flatpak.Builder

# Install runtimes and SDK extensions
flatpak install -y flathub \
  org.kde.Platform//6.8 \
  org.kde.Sdk//6.8 \
  io.qt.PySide.BaseApp//6.8 \
  org.freedesktop.Platform.ffmpeg-full//24.08 \
  org.freedesktop.Sdk.Extension.node20//24.08

# Install flatpak-pip-generator (for regenerating python3-modules.yaml)
uv tool install flatpak-pip-generator
```

## Local Build + Install

```bash
# From repo root:

# Build the Flatpak (local iteration — skip signing with SKIP_SIGN=1)
SKIP_SIGN=1 tools/linux-flatpak/build.sh

# Or manually:
flatpak run org.flatpak.Builder \
  --user \
  --repo=flatpak-repo \
  --force-clean \
  build-dir \
  io.github.kcreasey.MusicStreamer.yaml

# Install locally (no GPG verify for dev builds)
flatpak install --user --no-gpg-verify tools/linux-flatpak/artifacts/MusicStreamer-*.flatpak

# Run
flatpak run io.github.kcreasey.MusicStreamer
```

## SKIP_SIGN Local Iteration

For local development, set `SKIP_SIGN=1` to bypass GPG signing:

```bash
SKIP_SIGN=1 tools/linux-flatpak/build.sh
```

Without `SKIP_SIGN=1`, you must set `GPG_KEY_ID=<keyid>` to the passphrase-less signing
key. See CI signing documentation in `.github/workflows/linux-flatpak.yml`.

## Regenerating python3-modules.yaml

The `python3-modules.yaml` at repo root is generated by `flatpak-pip-generator` from
`tools/linux-flatpak/flatpak-requirements.txt`. Regenerate when pip deps change:

```bash
~/.local/bin/flatpak_pip_generator \
  --runtime='org.kde.Sdk//6.8' \
  --yaml \
  --requirements-file=tools/linux-flatpak/flatpak-requirements.txt \
  --output python3-modules
```

**NOTE:** `PySide6` is intentionally excluded from `flatpak-requirements.txt` — it is
provided by `io.qt.PySide.BaseApp//6.8`. Including it would cause an ABI conflict at
runtime (RESEARCH.md Pitfall 5).

## Publishing the .flatpak as a Release Asset (D-06)

The CI workflow (`.github/workflows/linux-flatpak.yml`) is `workflow_dispatch`-only
(D-07: no auto-publish). The workflow produces a signed `MusicStreamer-<version>.flatpak`
as a GitHub Actions build artifact but does NOT automatically upload it to a release tag.

To reconcile D-06 (the `.flatpak` ships as a GitHub release asset alongside the AppImage),
the operator must perform a one-time manual step per release:

1. **Trigger the CI workflow** from the GitHub Actions tab for the target commit/tag.
2. **Download the artifact** (`MusicStreamer-Flatpak`) from the completed CI run.
3. **Upload to the release tag** using `gh`:

   ```bash
   gh release upload <tag> MusicStreamer-<version>.flatpak
   ```

   Example:
   ```bash
   gh release upload v2.2.86 MusicStreamer-2.2.86.flatpak
   ```

This is the same release tag that carries the AppImage artifact from the AppImage CI
workflow. Both artifacts end up on the same GitHub Release page.

## Sandbox Permissions Summary

The Flatpak sandbox policy is the exact allow-list from CONTEXT.md D-01/FP-04/FP-05:

| Permission | Purpose |
|-----------|---------|
| `--share=network` | Stream playback, API calls |
| `--socket=pulseaudio` | Audio output |
| `--socket=wayland` | Wayland display |
| `--socket=fallback-x11` | X11 fallback |
| `--own-name=org.mpris.MediaPlayer2.MusicStreamer` | MPRIS2 media key control |
| `--filesystem=~/.local/share/musicstreamer:ro` | First-launch import wizard (read-only) |
| `--env=QTWEBENGINE_DISABLE_SANDBOX=1` | GBS.FM login via QtWebEngine |

**FORBIDDEN (deny-list enforced by drift-guard tests):**
- `--filesystem=home` — broad host filesystem access
- `--socket=session-bus` — unrestricted D-Bus access
