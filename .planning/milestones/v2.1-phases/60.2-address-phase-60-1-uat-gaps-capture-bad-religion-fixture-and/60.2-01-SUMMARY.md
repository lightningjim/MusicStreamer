## Wave 1 Fix Surface (for Plan 02 planner)

**File:** `musicstreamer/gbs_api.py`
**Class:** `_ArtistAlbumParser.handle_data` (the leading-text block discriminator)
**Lines:** 542-548
**Change:** Relax plural-only `startswith` to accept both forms:

```python
# BEFORE (production):
if lower.startswith("artists"):
    self._current_block = "artists"
elif lower.startswith("albums"):
    self._current_block = "albums"

# AFTER (Wave 1):
if lower.startswith("artist"):     # accepts "Artist:" AND "Artists:"
    self._current_block = "artists"
elif lower.startswith("album"):    # accepts "Album:" AND "Albums:"
    self._current_block = "albums"
```

`startswith("artist")` matches both "artist" and "artists" because "artists" begins with "artist". Same for albums. No regex needed; minimal-diff.

### Note on `test_clear_table_clears_spans` GREEN-by-coincidence

The plan predicted this test would be RED until Wave 2 added an explicit `clearSpans()` call to `_clear_table`. In practice, the test passes today because Qt's `QStandardItemModel.removeRows(0, rowCount)` (which `_clear_table` already calls) implicitly drops view-level spans when the rowCount drops to zero. Verified via direct probe:

```
after setSpan, span(0,0)= 4
after removeRows, span(0,0)= 1   <-- span dropped automatically
after fresh appendRow, span(0,0)= 1
```

The test still serves as a CONTRACT PIN — it locks in the post-`_clear_table` invariant. Wave 2 will add explicit `clearSpans()` for defense-in-depth (the implicit cleanup depends on rowCount→0; if a future change keeps rows alive, the contract stops holding). The test continues to pass either way.

This is consistent with the TDD fail-fast rule: I investigated the unexpected pass before proceeding, confirmed the test exercises the right invariant, and documented the finding rather than forcing a contrived RED.
