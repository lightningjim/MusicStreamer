# Phase 60.2: Address Phase 60.1 UAT Gaps — Bad-Religion Triage + Artist Drill Album Grouping — Research

**Researched:** 2026-05-06
**Domain:** Python `html.parser.HTMLParser` evolution (`_ArtistAlbumParser` triage + `_ArtistPageParser` album-state-carry) + PySide6 `QTableView`/`QStandardItemModel` row-spanning section-header rendering
**Confidence:** HIGH for the triage methodology, parser-state-carry mechanics, fixture-capture extension, test contract, validation surface, and security domain. MEDIUM-LOW for the bad-religion root-cause hypothesis ranking — the captured HTML decides; this research surfaces the inspection steps and candidate fix shapes but cannot determine the truth a priori.

## Summary

Phase 60.2 closes two specific Phase 60.1 UAT gaps. Both surfaces are well-scoped evolutions of code that shipped in Phase 60.1 (`musicstreamer/gbs_api.py` parsers + `musicstreamer/ui_qt/gbs_search_dialog.py` render loop), with **no new dependencies, no architectural moves, and no UI-SPEC layout changes**. The work breaks into three discrete tracks that can run mostly in parallel after the fixtures land:

1. **Track 1 — Bad-religion triage (SC1 closure):** Capture five new `/search` fixtures (`bad_religion`, `bad_company`, `iron_maiden`, `black_sabbath`, `death_cab_for_cutie`) by extending `scripts/gbs_capture_fixtures.sh`, then triage the `bad_religion` HTML against the working `foo_fighters` baseline. Outcome routes to either (a) a parser fix in `_ArtistAlbumParser` (with RED-first test from the captured fixture per D-06) or (b) a documented limitation in both code-comment and CONTEXT.md per D-05.
2. **Track 2 — `_ArtistPageParser` album-state-carry:** Replace the `_skip_current = "albumTitle" in row_class` deliberate-skip at `gbs_api.py:619` with a state-carrying read of the album anchor's text into `self._current_album`, then emit an `album` field on every subsequent song row's dict until the next `albumTitle` row resets it. Pre-first-`albumTitle` rows carry `album = ""` per D-10.
3. **Track 3 — Section-header span rows in the dialog:** Modify `_render_results()` in `gbs_search_dialog.py` to detect album-grouping rendering mode (drill-view), insert a span-row per album-group transition, set `setSpan(row, 0, 1, columnCount)`, populate with `"{album} ({n} songs)"` text, set `Qt.ItemFlag.ItemIsEnabled` (drops select+edit per Pattern T-40-04 plain-text + non-interactive contract), keep the per-row Add! button rendering unchanged for song rows, and skip the header row entirely when `album == ""` per D-11.

**Critical correction to CONTEXT.md terminology:** CONTEXT.md and the additional research context refer to "QTableWidget" and `QTableWidget.setSpan(...)`. The actual code (`gbs_search_dialog.py:328-341`) uses **`QTableView` + `QStandardItemModel`**, not `QTableWidget`. `QTableView.setSpan(row, col, rowSpan, colSpan)` exists and works identically [VERIFIED: `python3 -c "from PySide6.QtWidgets import QTableView; print(hasattr(QTableView, 'setSpan'))"` → `True` on PySide6 6.11.0, 2026-05-06]. The contract D-02 ("setSpan(row, 0, 1, columnCount); single QTableWidgetItem") translates to: `self._results_table.setSpan(row, 0, 1, self._model.columnCount())`; populate via a single `QStandardItem` whose `setFlags(Qt.ItemFlag.ItemIsEnabled)` removes editability and selectability while preserving theme rendering.

**Primary recommendation:** Plan 60.2 in **3 plans / 3 waves**, mirroring the Phase 60.1 wave structure: Wave 0 (RED + capture), Wave 1 GREEN (parser tier — bad-religion triage + `_ArtistPageParser` album field), Wave 2 GREEN (dialog tier — section-header rendering), Wave 3 (manual UAT signed). RED-first tests per D-06 (parser-fix path) and the `_ArtistPageParser` album-field-emission contract are both fixture-driven and follow the established Phase 60.1 TDD pattern verbatim.

## User Constraints (from CONTEXT.md)

### Locked Decisions

#### Album grouping render style (Area 1)
- **D-01:** Render album grouping in the artist drill-down via **section-header rows in the existing QTableWidget** (NOT an Album column, NOT both). Closest mapping to gbs.fm's source HTML structure (`<tr class="albumTitle">` separator rows); preserves the 4-column table width; adds scannable group boundaries without per-row text repetition.
- **D-02:** Implement the section header as a **spanning row inside the existing QTableWidget**: insert a row, call `setSpan(row, 0, 1, columnCount)`, populate with a single `QTableWidgetItem` containing the album name + song count, set bold font + background tint, set `Qt.ItemFlag.NoItemFlags` (non-selectable, non-editable). Keeps the dialog's widget type stable; minimal diff. Rejected: QTreeWidget swap (introduces widget-type asymmetry between search-results and drill view of the same dialog area); QTableView + QStandardItemModel + custom delegate (heavyweight; not justified for this scope).
- **D-03:** Section-header content is **album name + song count** in the form `"{album} ({n} songs)"` — e.g. `"The New Order (8 songs)"`. Count is `len(rows_in_group)` in the render path; no parser-side counting needed.

#### Triage→fix scope commitment (Area 2)
- **D-04:** Phase 60.2 commits to **triage + ship whatever fix the triage warrants**. SC1 closes one way or the other in this phase: parser fix (with RED-first test from the captured fixture) if triage points at a client-side bug; documented limitation (code comment + CONTEXT.md note) if it's confirmed server-side. Rejected: triage-only-split-fix-to-60.3 (one extra round-trip before SC1 actually closes); parser-fix-upfront-commitment (forces a phase replan if triage shows server-side).
- **D-05:** If triage lands on server-side, the documented limitation lives in **both** a code comment near `_ArtistAlbumParser` AND a note in `60.2-CONTEXT.md` (this file's "Specific Ideas" section gets updated post-triage with the finding). Both surfaces so neither rots alone — a future reader of the parser sees it inline; a future Claude reading planning history sees the rationale.
- **D-06:** If triage lands on client-side (parser bug), test contract is **RED test from `search_bad_religion_p1.html` first, GREEN fix second**. Mirrors Phase 60.1's locked D-03 TDD pattern (capture fixture → write failing parser test → fix → test goes green). Rejected: GREEN-test-with-the-fix (loses audit trail); fix-only-no-fixture-test (no permanent regression guard for this query class).

#### Multi-query fixture scope (Area 3)
- **D-07:** Capture **5 fixtures total** alongside the primary `search_bad_religion_p1.html`, to characterize the failure class:
  1. `search_bad_religion_p1.html` — primary defect repro (per Phase 60.1 UAT).
  2. `search_bad_company_p1.html` — same first word; isolates whether "bad" is the trigger or a position-1 prefix collision.
  3. `search_iron_maiden_p1.html` — popular two-word band, control case (rules in/out the "any multi-word with no special tokens" hypothesis beyond what `pearl jam` and `foo fighters` already covered in 60.1).
  4. `search_black_sabbath_p1.html` — control + tests whether words appearing in genre-tag fragments (`black metal`) interact badly with panel rendering.
  5. `search_death_cab_for_cutie_p1.html` — single-result hypothesis probe (a 4-word artist name with low collision likelihood; tests whether gbs.fm suppresses 1-row artist/album panels rather than rendering them).
- **D-08:** Filename pattern is `search_<query_with_underscores>_p1.html`, mirroring the existing `search_test_p1.html` and `search_multiword_p1.html` (Phase 60.1) conventions. The `_pN.html` suffix preserves room for page-2 captures if any probe ever returns paginated results.
- **D-09:** All 5 fixtures captured by extending `scripts/gbs_capture_fixtures.sh` (append after item 16 — the existing `search_multiword_p1.html` capture). Same dev-cookies invariant as the existing entries.

#### Album label fallback (Area 4)
- **D-10:** Parser carries **empty string `""`** as the album value for any song row that appears before the first `<tr class="albumTitle">` separator. Honest contract: the source HTML didn't tell us what album they belong to, parser doesn't fabricate. Rejected: `"Singles"` (introduces fake metadata; misleading for compilation cuts or HTML structural artifacts); `"Unknown"` (same fabrication risk; also visually ugly).
- **D-11:** Dialog renders empty-string album rows by **skipping the section-header row entirely** for that group — the rows render directly without a header above them. Honest visual: source HTML didn't group them, the UI doesn't pretend to. Rejected: synthetic "Other" divider (introduces UI text not present in source); subtle styling differentiation (italics + no header — code complexity not justified for an edge case the source doesn't even structure).

### Claude's Discretion

- Whether `_ArtistPageParser` carries the `album` value as a per-row dict field OR emits a parallel `album_groups: list[(album_name, start_idx, end_idx)]` structure — both satisfy the contract; planner picks based on which produces the cleaner diff. **Per-row field is simpler and idiomatic. Recommendation: per-row field.**
- Exact font weight / background tint values for section-header rows — UI polish; planner picks something consistent with the rest of the dialog's table styling. **Recommendation: bold via `QFont().setBold(True)` on the QStandardItem; no background tint (UI-SPEC `FLAG-01` no-design-system constraint forbids new colors / setStyleSheet — see Common Pitfall #5).** A bold-only differentiation is sufficient to scan and stays inside Phase 60.1 UI-SPEC's binding tokens.
- Whether the post-triage code comment for D-05 lives as a class-level docstring addition on `_ArtistAlbumParser` or as an inline comment near the parser's main loop — wherever a future reader is most likely to look first. **Recommendation: extend the existing class-level docstring at `gbs_api.py:494-507`**, in a new "Known limitations" section, plus a one-line `# See class docstring "Known limitations" for query-specific gaps.` near the `handle_data` block-discriminator at line 542. Two surfaces, one source of truth.
- Whether the bad-religion fixture captures full UTF-8 HTML or sanitized/trimmed-for-test — match the existing fixture style. **The existing fixtures (`search_test_p1.html`, `search_multiword_p1.html`, `artist_4803.html`, `album_1488.html`) are full UTF-8 HTML captures, byte-for-byte from `curl`.** Match that — no trimming.
- Whether the 4 probe fixtures (D-07 #2-#5) get their own RED-first parser tests beyond the primary `bad_religion` one. Probably not — they exist primarily as triage evidence + potential regression coverage if a pattern emerges. **Recommendation: ONE parametrized test that asserts each of the 5 fixtures produces the expected `_parse_artist_album_html(...)` shape *as observed in the captured HTML* (i.e., the test pins what the fixture has, not what we wish it had). The bad-religion entry of that parametrize gets RED-first treatment if the triage points client-side; the others are GREEN-from-the-start as regression baselines.**

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| `QTableView.setSpan(row, 0, 1, cols)` | Switch widget to `QTreeWidget` with album-name parent rows | UI-SPEC FLAG-01 + CONTEXT.md D-01/D-02 already reject the widget swap. `QTableView.setSpan` works identically to `QTableWidget.setSpan` (both inherit `setSpan` from `QTableView` which is `QTableWidget`'s base class) [VERIFIED]. No reason to swap. |
| Per-row `album` field on dict | Parallel `album_groups: list[(name, start_idx, end_idx)]` | Both satisfy D-01/D-02 contract. Per-row field is simpler diff (one new dict key), works with the existing `_render_results` shape, no new data structure. CONTEXT.md grants Claude's-discretion on this; recommend per-row. |
| `QStandardItem.setFlags(Qt.ItemFlag.NoItemFlags)` (literal CONTEXT D-02 reading) | `Qt.ItemFlag.ItemIsEnabled` | `NoItemFlags` (value 0) leaves the cell visually disabled (greyed). `ItemIsEnabled` alone keeps theme rendering while still removing select+edit. **Recommendation: ItemIsEnabled-only**, since the span row should be visually present (not greyed) but inert. Verified locally: `QStandardItem(); item.setFlags(Qt.ItemFlag.ItemIsEnabled); bool(item.flags() & Qt.ItemFlag.ItemIsSelectable)` → `False`; `bool(item.flags() & Qt.ItemFlag.ItemIsEditable)` → `False`; `bool(item.flags() & Qt.ItemFlag.ItemIsEnabled)` → `True` [VERIFIED 2026-05-06]. CONTEXT.md D-02's wording "non-selectable, non-editable" is satisfied either way. |
| Background tint via `setBackground()` or stylesheet | No tint, bold-only differentiation | UI-SPEC `FLAG-01` (no-design-system) and Phase 60.1 anti-pattern "DO NOT call `setStyleSheet` on... drill-down widget" both push toward bold-only. Bold + the layout-level `setSpan` (visually obvious row spanning all columns) is enough scannability; tint adds a color decision the codebase has no precedent for. |

```python
def __init__(self):
    super().__init__()
    self.results: list = []
    self._artist_name: str = ""
    self._page_title_seen: bool = False
    self._in_artist_table: bool = False
    self._row: Optional[dict] = None
    self._capture: Optional[str] = None  # extended: also "album_name"
    self._td_text: str = ""
    self._tds_in_row: int = 0
    self._ths_in_row: int = 0
    self._is_album_title_row: bool = False  # NEW: replaces _skip_current
    self._current_album: str = ""           # NEW: carries album name across rows

def handle_starttag(self, tag, attrs):
    ad = dict(attrs)
    if tag == "table" and "artist" in (ad.get("class") or ""):
        self._in_artist_table = True
        return
    if not self._in_artist_table:
        return
    if tag == "tr":
        self._row = {"songid": None, "title": "", "duration": "",
                     "add_url": "", "album": self._current_album}  # NEW: emit current album
        self._tds_in_row = 0
        self._ths_in_row = 0
        self._capture = None
        self._td_text = ""
        row_class = ad.get("class") or ""
        self._is_album_title_row = "albumTitle" in row_class
        return
    if self._is_album_title_row:
        # On albumTitle rows: capture the <a href="/album/X">{name}</a> anchor text
        # into self._current_album. The next regular <tr> will copy that value
        # into row["album"] via the handle_starttag(tr) branch above.
        if tag == "a" and ad.get("href", "").startswith("/album/"):
            self._capture = "album_name"
            self._current_album = ""  # reset for fresh capture
        return
    if tag == "th":
        self._ths_in_row += 1
        if not self._page_title_seen and "album" in (ad.get("class") or ""):
            self._capture = "page_title"
        return
    if tag == "td":
        # ... existing logic unchanged
        ...
    if tag == "a" and self._row is not None:
        # ... existing logic unchanged
        ...

def handle_data(self, data):
    if not self._in_artist_table:
        return
    if self._capture == "album_name":
        # Multi-chunk reassembly: append; final .strip() happens at handle_endtag
        self._current_album += data
        return
    if self._capture == "page_title" and not self._page_title_seen:
        self._artist_name += data
        return
    if self._is_album_title_row:
        return  # Not capturing — just skip the row's other text
    if self._capture == "title" and self._row is not None:
        self._row["title"] += data
        return
    if self._capture == "duration" and self._row is not None:
        self._row["duration"] += data

def handle_endtag(self, tag):
    if tag == "a" and self._capture == "album_name":
        # Multi-chunk reassembly: final strip
        self._current_album = self._current_album.strip()
        self._capture = None
        return
    if tag == "th" and self._capture == "page_title":
        self._artist_name = self._artist_name.strip()
        self._page_title_seen = True
        self._capture = None
        return
    if tag == "a" and self._capture == "title":
        self._capture = None
        return
    if tag == "tr":
        # Commit the row only if it's a real song row (NOT an albumTitle row,
        # NOT a header row, has songid).
        if (self._row is not None
                and not self._is_album_title_row
                and self._row.get("songid") is not None
                and self._tds_in_row >= 2):
            self._row["artist"] = self._artist_name or "(unknown)"
            self._row["title"] = (self._row.get("title") or "").strip()
            self._row["duration"] = (self._row.get("duration") or "").strip()
            self._row["add_url"] = self._row.get("add_url") or f"/add/{self._row['songid']}"
            # row["album"] was set in handle_starttag(tr) — already reflects current group
            self.results.append(self._row)
        self._row = None
        self._is_album_title_row = False
        self._tds_in_row = 0
        self._ths_in_row = 0
        self._capture = None
        return
    if tag == "table" and self._in_artist_table:
        self._in_artist_table = False
        # State reset on table close (defensive — handles partial HTML)
        self._current_album = ""
        self._is_album_title_row = False
```

**Three things to highlight:**

1. **Multi-chunk data reassembly:** `handle_data` may be called multiple times for a single text node when entity references are present (HTMLParser splits the chunk at entity boundaries even with `convert_charrefs=True`). The pattern above accumulates into `self._current_album` and strips ONLY at `</a>` / `</th>` close — mirroring the existing `_artist_name += data` page-title pattern at lines 655-657. Don't `.strip()` per-chunk.
2. **State reset on table close:** `_current_album` is reset on `</table>` so that a second `_ArtistPageParser` instance feeding a different fixture starts clean. Mirrors the `_in_artist_table = False` reset at line 695.
3. **Pre-first-albumTitle rows carry `album = ""`:** The `__init__` sets `self._current_album = ""` and that empty string is copied into `row["album"]` on every `<tr>` start until the first `<tr class="albumTitle">` updates `_current_album`. This satisfies D-10 by construction — no special-case branch needed.

**Recommended evolution** (single-kwarg signature change + a new render helper):

```python
def _render_results(self, *, clear_panels: bool = True,
                    group_by_album: bool = False) -> None:
    self._clear_table(clear_panels=clear_panels)
    if not self._results:
        self._show_inline_error("No results.")
        return
    self._hide_inline_error()

    if not group_by_album:
        # Original path: every row is a song row. Behavior unchanged.
        self._render_song_rows(self._results, base_idx=0)
        return

    # Drill-view path: insert per-album section-header span rows.
    # Group rows by their "album" field, preserving order. Empty-string albums
    # render WITHOUT a header per D-11.
    groups = self._group_rows_by_album(self._results)
    next_row = 0
    for album, rows_in_group in groups:
        if album != "":
            self._insert_album_section_header(next_row, album, len(rows_in_group))
            next_row += 1
        # Song rows beneath the header (or directly, if album == "")
        self._render_song_rows(rows_in_group, base_idx=next_row,
                               original_idx_offset=self._results.index(rows_in_group[0]))
        next_row += len(rows_in_group)

def _group_rows_by_album(self, results: list[dict]) -> list[tuple[str, list[dict]]]:
    """Preserve order; consecutive same-album rows form a group; empty-string
    albums form their own group (rendered without a header per D-11)."""
    groups: list[tuple[str, list[dict]]] = []
    for row in results:
        album = str(row.get("album", ""))
        if groups and groups[-1][0] == album:
            groups[-1][1].append(row)
        else:
            groups.append((album, [row]))
    return groups

def _insert_album_section_header(self, row: int, album: str, song_count: int) -> None:
    """Insert a span-row section header at the given table row index."""
    header_text = f"{album} ({song_count} songs)"
    item = QStandardItem(header_text)
    # Non-selectable + non-editable; visually present (not greyed → ItemIsEnabled retained)
    item.setFlags(Qt.ItemFlag.ItemIsEnabled)
    bold = QFont()
    bold.setBold(True)
    item.setFont(bold)
    # The model is column-count = 4 (Artist, Title, Duration, Add). Header lives in col 0,
    # the row's other 3 cells are filled with empty placeholder QStandardItems so the model
    # row has the correct shape. setSpan then visually merges them into one cell.
    placeholders = [QStandardItem("") for _ in range(self._model.columnCount() - 1)]
    for ph in placeholders:
        ph.setFlags(Qt.ItemFlag.ItemIsEnabled)
    self._model.appendRow([item, *placeholders])
    self._results_table.setSpan(row, 0, 1, self._model.columnCount())

def _render_song_rows(self, rows: list[dict], *, base_idx: int,
                     original_idx_offset: int = 0) -> None:
    """Append `rows` as song rows starting at `base_idx`. Add! button uses
    original_idx_offset + i so _on_submit_row reads the correct entry from
    self._results."""
    for i, result in enumerate(rows):
        artist = QStandardItem(str(result.get("artist", "")))
        title = QStandardItem(str(result.get("title", "")))
        duration = QStandardItem(str(result.get("duration", "")))
        placeholder = QStandardItem("")
        self._model.appendRow([artist, title, duration, placeholder])
        btn = QPushButton("Add!", self._results_table)
        btn.clicked.connect(self._make_submit_slot(original_idx_offset + i))  # QA-05
        self._submit_buttons.append(btn)
        self._results_table.setIndexWidget(
            self._model.index(base_idx + i, _COL_ADD), btn,
        )
```

**Key invariants:**

- **`setSpan(row, 0, 1, columnCount)` after `appendRow`:** The model row must exist before `setSpan` references it. PySide6 docs: `setSpan(row, column, rowSpan, columnSpan)` — `rowSpan=1` means "this single row spans `columnSpan` columns horizontally."
- **Add! button index offset:** Crucial subtlety. `_submit_buttons` and `_results` are 0-indexed by song-row position in `self._results`. After a span-row is inserted, the **table row index** and the **song-list index** diverge. The render path passes `original_idx_offset` so `_make_submit_slot` connects the click to the correct entry in `self._results`. `_on_submit_row(row_idx)` reads `self._results[row_idx]` — that index must remain a song-list index, not a table-row index.
- **Click-handler skip for header rows:** Because header rows do NOT call `setIndexWidget(row, _COL_ADD, btn)`, there's no Add! button on them; clicks on the span area do nothing. The `setFlags(ItemIsEnabled)` (no `ItemIsSelectable`) prevents row-selection from highlighting the header. **No additional click-discrimination logic is needed** — the contract emerges from the flags.
- **`_clear_table` must call `clearSpans()`:** A subsequent search after a drill-view will call `_clear_table` → `removeRows`. But `setSpan` state on `QTableView` lives on the view, not the model. **`_clear_table` MUST also call `self._results_table.clearSpans()`** to drop stale spans, otherwise the new search's first 4 columns of row 0 would be merged into one cell. See Common Pitfall #1.

### Anti-Patterns to Avoid

- **DO NOT** count song rows in the parser. D-03 specifies song count in the header is `len(rows_in_group)` computed by the render path. Parser stays a pure HTMLParser — no aggregation.
- **DO NOT** introduce a synthetic "Singles" / "Other" / "Unknown" album label for the empty-string group. D-10 / D-11 explicitly reject this.
- **DO NOT** swap `QTableView` → `QTableWidget` or `QTreeWidget`. CONTEXT.md D-01 mentions "QTableWidget" but the actual codebase uses `QTableView` + `QStandardItemModel`; both `QTableView.setSpan` and the contract D-02 describes work identically. The widget swap is rejected by D-01 and Phase 60.1's UI-SPEC FLAG-01.
- **DO NOT** call `setStyleSheet` on the section-header item or row. UI-SPEC FLAG-01 (no-design-system) is binding from Phase 60.1 and unchanged for 60.2. Bold-only differentiation via `QFont().setBold(True)`.
- **DO NOT** use HTML / RichText for the section-header text. T-40-04 plain-text invariant. The `f"{album} ({n} songs)"` string is plain UTF-8.
- **DO NOT** forget `clearSpans()` in `_clear_table`. Without it, a search after a drill-view leaves stale spans on rows 0..N (Common Pitfall #1).
- **DO NOT** use the table-row index as the song-list index in `_make_submit_slot`. After span-rows are inserted, table-row != song-list-row. Pass `original_idx_offset + i` so `_on_submit_row(idx)` reads `self._results[idx]` correctly.
- **DO NOT** invoke any of the 4 probe fixtures in 5+ separate parser tests. They exist as triage evidence + regression coverage; one parametrized test asserting their captured shape is enough.
- **DO NOT** speculate on the bad-religion root cause before the fixture is captured. D-04 explicitly mandates triage-then-fix, NOT fix-with-untested-hypothesis. Plan 1 captures + triages; Plan 2 ships whichever fix the triage warrants.

## Common Pitfalls

### Pitfall 1: `_clear_table` must call `clearSpans()` when transitioning out of drill-view

**What goes wrong:** User drills artist A → table renders with span rows at rows 0, 4, 9 → user clicks Back → search results restore via `_render_results(clear_panels=True, group_by_album=False)` → `_clear_table` → `removeRows`. But `setSpan` state lives on the `QTableView`, NOT the `QStandardItemModel`. Stale spans remain at rows 0, 4, 9. The next 3 search-result rows render with their first 4 columns visually merged into one cell.

**Why it happens:** PySide6's `QTableView.setSpan` is a view-level state. `clearSpans()` is the explicit reset. `removeRows` on the model does NOT clear them.

**How to avoid:** Add `self._results_table.clearSpans()` to `_clear_table`. Because `_clear_table` is called by every render path (search and drill-view), this is the single chokepoint where stale spans get cleared.

```python
def _clear_table(self, clear_panels: bool = True) -> None:
    self._model.removeRows(0, self._model.rowCount())
    self._submit_buttons = []
    self._results_table.clearSpans()  # NEW: drop stale spans (Phase 60.2 Pitfall 1)
    # ... rest unchanged
```

**Warning signs:** A test that drills, then clicks Back, then drills a different artist, then asserts `dlg._results_table.columnSpan(0, 0)` matches the new artist's first-album header. Without `clearSpans`, the stale spans from artist A persist.

### Pitfall 2: HTMLParser entity-boundary chunking for album-name capture

**What goes wrong:** Album names with `&amp;` (e.g., a hypothetical `"Sturm & Drang"`) split into multiple `handle_data` calls — `"Sturm "`, `"&"`, `" Drang"` — depending on `convert_charrefs` interaction. A naive parser that does `self._current_album = data.strip()` once captures only the last chunk.

**Why it happens:** Even with `convert_charrefs=True` (which auto-decodes `&amp;` to `&` and stops calling `handle_entityref`), the parser may still emit MULTIPLE `handle_data` calls for what looks like one text node when the chunk crosses a buffer boundary.

**How to avoid:** Accumulate via `+=` (the same pattern as the existing `_artist_name += data` capture at `gbs_api.py:655-657`). Strip ONLY at the closing tag (`</a>` for album-name; `</th>` for page-title).

```python
if self._capture == "album_name":
    self._current_album += data
    return  # do NOT strip yet
# ... in handle_endtag for tag == "a":
if self._capture == "album_name":
    self._current_album = self._current_album.strip()  # strip on close
    self._capture = None
```

**Warning signs:** A unit test feeding the parser an album name with `&amp;` (or, more pragmatically, leading whitespace as the Testament fixture has — `<td class="album" colspan=3> <a href="/album/69560"> Brotherhood of the Snake</a> </td>` has both internal-anchor leading space AND outer-cell whitespace). The captured text after strip should be exactly `"Brotherhood of the Snake"`.

### Pitfall 3: Add! button index drift when span rows are inserted

**What goes wrong:** Without offset accounting, `_make_submit_slot(table_row_idx)` connects to `_on_submit_row(table_row_idx)` which reads `self._results[table_row_idx]`. After span-rows shift song rows downward in the table, `self._results[3]` may correspond to table row 5, but the slot was wired with `idx=5`. `_on_submit_row(5)` then dies on `IndexError` or, worse, submits the wrong song.

**Why it happens:** `self._results` and `self._submit_buttons` are 0-indexed by song-list position. `setIndexWidget(model_idx, widget)` uses table-row-index. After mixing in non-song span-rows, those two indices diverge.

**How to avoid:** The render helper accepts both `base_idx` (table position for `setIndexWidget`) and `original_idx_offset` (song-list position for `_make_submit_slot`). The submit-slot closure captures `original_idx_offset + i` which is the song-list index, NOT the table-row index. Both `_submit_buttons.append(btn)` order AND `self._results` order remain song-list-indexed — that's what `_on_submit_row` reads.

**Warning signs:** A test that drills, asserts `len(dlg._submit_buttons) == song_count` (NOT `dlg._model.rowCount()`), AND clicks the Add! button on a song after a span row, asserting that `_GbsSubmitWorker` is dispatched with the correct songid.

### Pitfall 4: HTMLParser `convert_charrefs=True` is the default — no `handle_entityref` needed

**What goes wrong:** A planner adds `handle_entityref(name)` to `_ArtistPageParser` thinking they need to manually decode `&amp;` etc. They don't.

**Why it happens:** Python 3.5+ default for `HTMLParser` is `convert_charrefs=True` [VERIFIED locally 2026-05-06: `from html.parser import HTMLParser; HTMLParser().convert_charrefs == True`]. Phase 60.1 RESEARCH already documented this as Pitfall 11. The auto-decoded entity text arrives in `handle_data`.

**How to avoid:** No `handle_entityref` or `handle_charref` methods on the modified `_ArtistPageParser`. The pattern stays simple. Pitfall 11 from Phase 60.1 RESEARCH applies here unchanged.

**Warning signs:** Any reviewer flagging missing `handle_entityref` should be redirected to `convert_charrefs=True` default. Project precedent: existing `_SongRowParser`, `_ArtistAlbumParser`, `_QueueRowParser`, `_ArtistPageParser`, `_AlbumPageParser` all rely on this and have been verified working.

### Pitfall 5: UI-SPEC FLAG-01 binds 60.2 — no setStyleSheet, no new colors on the header

**What goes wrong:** A planner picks "subtle background tint" for the header (e.g., `setStyleSheet("background-color: #404040;")` or `item.setBackground(QColor("#202020"))`) thinking it adds visual differentiation.

**Why it happens:** CONTEXT.md D-02 says "bold font + background tint" — but Phase 60.1 UI-SPEC FLAG-01 is binding for the entire dialog scope ("This dialog has no theme tokens beyond `ERROR_COLOR_HEX`"). A new background color is a new design token.

**How to avoid:** Bold-only differentiation via `QFont().setBold(True)`. The `setSpan` itself provides the strongest visual signal (one cell spanning all columns is unmistakably different from a 4-cell row); bold reinforces it. Skip the tint.

**Warning signs:** Any plan task that calls `setStyleSheet` on the section-header item or row, or that assigns a new color to `_theme.py`, is wrong. UI-SPEC binding from Phase 60.1 unchanged.

### Pitfall 6: `_on_artist_drilled` is the single render-mode route — `_on_album_drilled` does NOT need album-grouping

**What goes wrong:** A planner mirrors the `_on_artist_drilled` change in `_on_album_drilled` thinking symmetry is good.

**Why it happens:** Album pages don't have `<tr class="albumTitle">` separators — they're a single album's track listing. Each row of `_AlbumPageParser` output has a per-row `artist` (the song's contributor), NOT a per-row `album`. Adding album-grouping to album-drill would group by song-artist (incorrect) or do nothing (since all rows have the same album implicitly — there is no `album` field in `_AlbumPageParser` output).

**How to avoid:** Only `_on_artist_drilled` calls `_render_results(group_by_album=True)`. `_on_album_drilled` keeps the default `group_by_album=False`.

**Warning signs:** A test of album-drill that asserts span rows in the table is wrong. Album drill should have `dlg._results_table.columnSpan(0, 0) == 1` for every row.

### Pitfall 7: Bad-religion fix MUST be RED-first, not GREEN-with-the-fix

**What goes wrong:** A planner, having read the captured fixture and identified the patch, writes the test and the fix in the same commit (or worse, just the fix).

**Why it happens:** D-06 is explicit: "RED test from `search_bad_religion_p1.html` first, GREEN fix second." Mirrors Phase 60.1 D-03's TDD pattern. If both land together, the project loses the audit trail showing that the captured fixture actually demonstrated the bug.

**How to avoid:** Plan 60.2-01 (Wave 0) writes the RED parser test against the captured fixture (no fix yet). The test FAILS. Plan 60.2-02 (Wave 1) ships the parser patch; the same test goes GREEN. Two commits. Two distinct verification runs.

**Warning signs:** Any wave-1 commit that includes both a new RED test AND its fix — split it.

### Pitfall 8: The 4 probe fixtures should NOT each get their own RED test

**What goes wrong:** A planner adds 5 separate RED tests (one per fixture) thinking each needs its own validation.

**Why it happens:** Probe fixtures (`bad_company`, `iron_maiden`, `black_sabbath`, `death_cab_for_cutie`) exist for **triage evidence** and **regression coverage** — not as primary defect repros. The bad-religion test is the load-bearing one. Adding 4 more RED tests for fixtures whose expected shape is already known dilutes the audit trail.

**How to avoid:** ONE parametrized test asserting the captured shape of each of the 5 fixtures (artist+album link counts, song row count). This serves as a regression baseline. The bad-religion entry of that parametrize gets the RED→GREEN treatment if triage points client-side; the others are GREEN-from-the-start. Per-fixture extra tests are unjustified.

**Warning signs:** A plan that has 5 separate `test_X_search_returns_artist_links` functions. Collapse to one parametrized.

### Pitfall 9: `clearSpans` ordering relative to `removeRows`

**What goes wrong:** Calling `clearSpans()` AFTER `removeRows` is a Qt-internal silent failure (the spans were associated with row indices that no longer exist; the call is a no-op or undefined behavior depending on Qt version).

**Why it happens:** The order matters: `clearSpans` must run while the rows still exist (or the call must reference rows that the model still knows about). Some Qt docs say `clearSpans` "removes all the row and column spans" without specifying the model state contract.

**How to avoid:** Call `clearSpans()` BEFORE `removeRows`:

```python
def _clear_table(self, clear_panels: bool = True) -> None:
    self._results_table.clearSpans()  # FIRST: drop spans while rows still exist
    self._model.removeRows(0, self._model.rowCount())
    self._submit_buttons = []
    # ... rest unchanged
```

**Warning signs:** A flickering visual artifact during drill→back→search transitions, or a test that drills and clicks Back and asserts `columnSpan(0, 0) == 1` failing intermittently.

### Pitfall 10: The single-result fixture (`death_cab_for_cutie`) MAY return paginated results

**What goes wrong:** The capture-script line `curl -sS -b "$COOKIES" "$BASE/search?query=death+cab+for+cutie&page=1"` only fetches page 1. If `death cab for cutie` returns multi-page results, page 1 alone may not exhibit the single-result behavior we're probing for.

**Why it happens:** GBS.FM's search pagination is `?page=N`. The first page may have artist/album panels and the second page may not (Phase 60.1 RESEARCH §Pitfall 1 hypothesis 1 noted: "page 2 of `test` has no Artist:/Album: blocks"). For probing single-result H4, page 1 is what we need; the full-result-set page count is informational.

**How to avoid:** Document in the capture script that we only need `page=1`. If post-capture the executor wants to verify the full result-set size, they can manually re-run with `page=2` etc. — but the committed fixture is `_p1.html` only.

**Warning signs:** If `grep -c '<a href="/song/' search_death_cab_for_cutie_p1.html` shows MANY song hits (e.g., >20), the fixture isn't actually single-result and H4 cannot be tested with this query. The planner should document the actual song-hit count in the triage SUMMARY.
