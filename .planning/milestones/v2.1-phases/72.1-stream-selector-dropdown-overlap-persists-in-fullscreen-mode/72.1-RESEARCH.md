# Phase 72.1: Stream-selector dropdown overlap persists in fullscreen mode - Research

**Researched:** 2026-05-16
**Domain:** PySide6 Qt 6.11 — runtime widget reparenting between sibling QHBoxLayouts driven by `resizeEvent`
**Confidence:** HIGH (Qt API surface is well-mapped; codebase has a directly parallel Phase 72 reparent precedent; pytest-qt resize patterns are already established in the test suite)

## Summary

Phase 72.1 is a **layout-only follow-up** to Phase 72. Phase 72 hid the `StationListPanel` to free width for the now-playing pane's bottom controls and fixed the overlap for single-stream stations. For **multi-stream** stations, the `stream_combo` QComboBox (`now_playing_panel.py:484-488`, `setMinimumWidth(140)`, visible only when `len(streams) > 1`) is still appended to the single control row, pushing the row's required minimum above the right pane's design floor.

CONTEXT D-01..D-08 lock the implementation shape: when the panel narrows below the row-1-with-picker minimum AND the picker is visible, the picker reparents from `controls` (existing row-1 QHBoxLayout, `now_playing_panel.py:440`) into a new `controls_row2` QHBoxLayout inserted into `center` immediately after `controls` (between line 548 and line 553). The picker uses `QSizePolicy.Expanding` on row 2 and reverts to its default Preferred + 140px-min on row 1. Trigger is **width**, not compact-mode; the resize hook is `NowPlayingPanel.resizeEvent` (D-05).

This is the first `resizeEvent` override and the first runtime widget-reparenting between sibling QHBoxLayouts in `musicstreamer/ui_qt/`. The closest precedent is Phase 72's StationListPeekOverlay round-trip (which reparents `station_panel` between a QSplitter and a QFrame overlay, not between two QHBoxLayouts). Phase 72's lessons about (a) Qt forum claims not surviving to Qt 6.11, (b) pytest-qt synthetic events bypassing real dispatcher behavior, and (c) sizes() returning stale values during transitions all carry through and shape the verification plan.

**Primary recommendation:** Build the reparent helper as a single private method `_reflow_stream_picker_row()` invoked from BOTH `resizeEvent` and `_populate_stream_picker`. Use `layout.indexOf(widget)` / `layout.itemAt(i).widget()` (already-established pattern from `test_phase72_now_playing_panel.py:140-179`) to verify layout membership in tests. Compute the wrap threshold from widget minimums (option (a) from CONTEXT Claude's Discretion) and cache it once, refreshing only when `_populate_stream_picker` rebuilds.

### Locked Decisions

**Wrap trigger:**
- **D-01:** Width-responsive trigger, not mode-coupled. Reparent decision driven by panel width vs. computed row-1-with-picker minimum. Compact mode is NOT an input. Wide windows stay one-row; narrow windows wrap, regardless of compact-mode state.
- **D-02:** Single-stream stations stay one-row at all widths. When `stream_combo` is hidden (existing `len(streams) > 1` gate at `now_playing_panel.py:1135`), row 2 stays empty/hidden — no reparent logic runs.

**What wraps to row 2:**
- **D-03:** ONLY the stream picker moves to row 2. Row 1 always contains play, stop, edit, star, eq, volume, compact_toggle, addStretch. The picker is the only widget that moves between rows.
- **D-04:** No flat FlowLayout for the entire row. Two explicit QHBoxLayouts preserve the "transport always together" invariant.

**Layout mechanism:**
- **D-05:** Reparent picker between row 1 / row 2 on QResizeEvent. Construct both `controls` (existing row 1) and a new `controls_row2` (QHBoxLayout) eagerly inside `center`. By default, stream_combo lives in row 1. On panel resize, compute `should_wrap = (stream_combo.isVisible() AND panel_width < row1_with_picker_min)`. If state changes, remove the picker from its current layout and add it to the other layout.
- **D-06:** No animation on reparent. Reparent is instant.

**Stream-picker sizing on its own row:**
- **D-07:** Picker stretches to fill row 2 width. Uses `QSizePolicy.Expanding` horizontally so the dropdown grows to the panel's full width minus margins.
- **D-08:** Picker reverts to its existing Preferred sizing when back in row 1. The 140px `setMinimumWidth(140)` is preserved as the floor; size policy is restored to Preferred on reparent back.

### Pattern 1: Sibling QHBoxLayouts Inside One QVBoxLayout

**What:** Two `QHBoxLayout` rows added sequentially to a parent `QVBoxLayout`, where a single widget moves between them at runtime.

**When to use:** Any "two-state row layout" where the trigger is a runtime condition (window width, content presence, mode), the wrapping widget is well-known and singular, and the row identity needs to remain inspectable (vs. a generic FlowLayout where wrap order is opaque).

**Example:**
```python
# Source: derived from now_playing_panel.py:440-548 (existing row 1)
#         + Qt docs https://doc.qt.io/qt-6/qboxlayout.html

# In NowPlayingPanel.__init__, immediately after `center.addLayout(controls)` (line 548):
self._controls_row2 = QHBoxLayout()
self._controls_row2.setSpacing(8)  # match controls.setSpacing(8) at line 441
center.addLayout(self._controls_row2)
# Row 2 starts empty; stream_combo lives in `controls` by default per CONTEXT D-05.
# When threshold is breached, reflow helper moves it here.
```

**Critical:**
- Both layouts share the same parent (`center`, the NowPlayingPanel center QVBoxLayout). Reparenting between them does NOT change the widget's QObject parent — that stays `self` (NowPlayingPanel) per `QComboBox(self)` at line 484. Layout membership and QObject parentage are orthogonal concerns in Qt. [CITED: https://doc.qt.io/qt-6/qboxlayout.html#addWidget — "widget becomes a child of the QLayout::parentWidget" only fires if the QObject parent changes; here `center.parentWidget() == self` already.]
- `controls_row2` will report `count() == 0` when empty. Qt collapses zero-item QHBoxLayouts to zero height — there is no need for an explicit `hide()` call on a container widget. The original "hide row 2 when empty" concern in CONTEXT integration_points is auto-handled. CONTEXT planner-discretion picks between explicit container widget + hide vs. relying on Qt's auto-collapse; this research recommends the latter (simpler).

### Pattern 2: Reparent QComboBox Between QHBoxLayouts via removeWidget + addWidget

**What:** Move an existing fully-configured `QComboBox` (or any widget) from one QBoxLayout to another sibling QBoxLayout while preserving signals, model, currentIndex, and itemData.

**When to use:** Any time the widget's internal state (selected item, populated rows, signal connections) is expensive or fragile to rebuild and you want a deterministic two-state layout.

**Example:**
```python
# Source: derived from PySide6 docs + Phase 72 station_list_peek_overlay.py:128-141
#         (which reparents station_panel between QSplitter and QFrame)
# Qt docs: https://doc.qt.io/qt-6/qboxlayout.html (addWidget reparents implicitly)
# Qt docs: https://doc.qt.io/qt-6/qobject.html#setParent (signals survive)

def _move_stream_picker_to(self, target_layout: QHBoxLayout) -> None:
    """Reparent stream_combo into target_layout (one of: self.controls, self._controls_row2).

    Idempotent — no-op if stream_combo is already in target_layout.
    Signal connections (currentIndexChanged -> _on_stream_selected) survive
    per Qt: QObject::setParent does NOT disconnect signal/slot connections.
    QComboBox model, itemData, currentIndex all survive layout reparent.
    """
    current_layout = self._find_picker_layout()  # returns controls or _controls_row2 or None
    if current_layout is target_layout:
        return  # idempotent
    if current_layout is not None:
        current_layout.removeWidget(self.stream_combo)
    target_layout.addWidget(self.stream_combo)
    # Size policy must be reset on each reparent — see Pattern 3.
```

**Critical:**
- `QObject::setParent()` (called implicitly by `addWidget`) **preserves all signal/slot connections** [VERIFIED: Qt docs — connections are only severed in `~QObject()` destructor, not during reparent]. The existing `stream_combo.currentIndexChanged.connect(self._on_stream_selected)` from `now_playing_panel.py:487` continues to fire after the move.
- `QComboBox.currentIndex()`, `QComboBox.itemData(i)`, and the underlying model survive the reparent [CITED: Qt parent-system architecture — these are properties of the QComboBox instance, not of its layout container].
- The widget's QObject parent (`stream_combo.parent()`) is already `self` (NowPlayingPanel) — both `controls` and `controls_row2` are sub-layouts of `center` whose `parentWidget()` is `self`, so the QObject parent never changes during reparent. Only layout membership changes.
- Focus is potentially lost during a reparent (Qt may clear focus when the widget's parent chain reconfigures). For a passive picker that the user interacts with by click-to-drop-down, this is acceptable. If keyboard focus needs to survive, the helper would need to capture and restore `self.stream_combo.hasFocus()` and call `setFocus()` after the move — **NOT NEEDED for this phase** because the user doesn't typically Tab to the picker.

### Pattern 3: Runtime QSizePolicy Toggle (Preferred ↔ Expanding)

**What:** Toggle a widget's horizontal QSizePolicy between Preferred (default for QComboBox, sizes to content with minimumWidth floor) and Expanding (claims all extra space in its row).

**When to use:** When the same widget needs different stretch behavior depending on which container it's in. Here: Preferred in row 1 (sits at 140px-min, doesn't crowd siblings), Expanding in row 2 (fills the row width).

**Example:**
```python
# Source: QSizePolicy docs https://doc.qt.io/qt-6/qsizepolicy.html
# Codebase precedent: favorites_view.py:183, station_list_panel.py:133 (both use setSizePolicy)

from PySide6.QtWidgets import QSizePolicy

def _set_picker_size_policy(self, expanding: bool) -> None:
    """Set stream_combo's horizontal size policy.

    Expanding=True for row 2 (D-07: stretches to fill row width).
    Expanding=False for row 1 (D-08: reverts to default Preferred + 140px-min floor).

    setMinimumWidth(140) was set at construction time (now_playing_panel.py:485)
    and PERSISTS across setSizePolicy() calls. [VERIFIED: Qt size system —
    minimumWidth is a separate property from sizePolicy; the policy controls
    *growth*, the minimum is a floor.]
    """
    if expanding:
        self.stream_combo.setSizePolicy(QSizePolicy.Expanding, QSizePolicy.Preferred)
    else:
        self.stream_combo.setSizePolicy(QSizePolicy.Preferred, QSizePolicy.Preferred)
    # Note: QComboBox's default horizontal policy IS Preferred (verified empirically
    # via codebase grep — no setSizePolicy call exists on stream_combo, so it carries
    # the QComboBox-class default). Explicitly setting Preferred on revert is safer
    # than relying on the default being unchanged across Qt versions.
```

**Critical:**
- `setMinimumWidth(140)` from `now_playing_panel.py:485` **PERSISTS** across `setSizePolicy()` changes. Minimum width is a hard floor set on the widget directly; policy controls how the widget *grows*, not where its minimum is. [CITED: https://doc.qt.io/qt-6/qwidget.html#minimumWidth-prop — "The minimum width of the widget" is a stand-alone property, independent of sizePolicy.]
- `setSizePolicy()` calls `updateGeometry()` internally on the QWidget — there is **no need to call `updateGeometry()` explicitly** after `setSizePolicy()`. [CITED: Qt source pattern — setSizePolicy triggers a layout invalidation on the parent layout.]
- The default QComboBox horizontal size policy IS Preferred (assumed but not formally verified from docs in this research session — Qt does not publish a class-by-class default-policy table; codebase observation: `stream_combo` has no `setSizePolicy` call and behaves as Preferred today). Explicitly setting `Preferred` on revert is a safety belt against Qt version drift. [ASSUMED]

### Pattern 4: Threshold Compute from Widget Minimums (Self-Correcting)

**What:** Compute "what width does this row need to fit all its current children?" from the children's individual minimum size hints rather than from a hand-tuned constant.

**When to use:** When the row's contents may evolve over time and you want the threshold to track those changes automatically without code edits to a magic number.

**Example:**
```python
# Source: derived from Qt layout system + CONTEXT Claude's Discretion recommendation (option a)

def _picker_row_min_width(self) -> int:
    """Compute the minimum width row 1 needs when stream_combo is included.

    Sum of:
      - each widget's minimumSizeHint().width() (or minimumWidth() if set)
      - spacing between adjacent widgets (controls.spacing() * (count - 1))
      - layout's contentsMargins left + right
    Plus a small safety buffer (~16px) for splitter-handle width and rounding.

    Cached on first compute; re-computed when _populate_stream_picker rebuilds
    (which is the only event that can change which widgets the picker shares
    row 1 with — currently none).

    Returns:
        int: the panel.width() value below which row 1 cannot fit the picker.
    """
    if self._cached_row1_min is not None:
        return self._cached_row1_min
    layout = self.controls  # row 1 QHBoxLayout
    margins = layout.contentsMargins()
    spacing = layout.spacing()
    total = margins.left() + margins.right()
    count = layout.count()
    widget_count = 0
    for i in range(count):
        item = layout.itemAt(i)
        w = item.widget()
        if w is None:
            # addStretch(1) -- no width contribution
            continue
        # Use max(minimumWidth, minimumSizeHint().width()) -- the actual floor.
        wmin = max(w.minimumWidth(), w.minimumSizeHint().width())
        total += wmin
        widget_count += 1
    if widget_count > 1:
        total += spacing * (widget_count - 1)
    # Safety buffer for splitter handle + margins-of-margins
    total += 16
    self._cached_row1_min = total
    return total
```

**Critical:**
- `widget.minimumSizeHint().width()` is reliable AFTER first paint. On Wayland, the first `resizeEvent` may fire before paint completes, returning stale or zero values. [ASSUMED — Qt does not publish Wayland-specific timing guarantees; conservative defense: cache the result and re-compute on `_populate_stream_picker` (which is called from `bind_station`, which fires *after* show in practice).]
- Caching is important: `minimumSizeHint()` walks the widget's internal layout each call. For a 9-widget row called from every resize tick (potentially dozens per second during a window drag), uncached calls add up.
- Compute ONCE on first need, re-compute only when `_populate_stream_picker` runs (which is the only event that can change the picker's contents — and even that doesn't change the *widget count* in row 1, only the picker's own internal state). A trivial cache-invalidation hook in `_populate_stream_picker` is enough.

### Pattern 5: resizeEvent Override With Single Reflow Call (Phase-72.1 Convention)

**What:** Override `QWidget.resizeEvent` to invoke a single reflow helper that does all width-driven layout work.

**When to use:** Any widget whose layout depends on its own current width and needs to recompute on every resize.

**Example:**
```python
# Source: https://doc.qt.io/qt-6/qwidget.html#resizeEvent
#         + Qt warning: "Calling resize() or setGeometry() inside resizeEvent()
#           can lead to infinite recursion."

def resizeEvent(self, event):  # noqa: N802 (Qt override)
    """Override to drive width-responsive stream-picker reparent (Phase 72.1).

    super().resizeEvent(event) MUST be called first per Qt convention so
    base QWidget machinery runs (layout invalidation, geometry update).

    The reflow helper inside _reflow_stream_picker_row() MUST NOT call
    self.resize() or self.setGeometry() — Qt warns this causes infinite
    recursion. Reparenting a child widget is safe (it does not trigger a
    parent resize event).
    """
    super().resizeEvent(event)
    self._reflow_stream_picker_row()
```

**Critical:**
- **Call `super().resizeEvent(event)` first.** [CITED: standard Qt subclassing convention — base implementation invalidates layout, sets internal state. Skipping it leaves the widget in an inconsistent state.]
- **NEVER call `self.resize()` or `self.setGeometry()` inside `resizeEvent`.** [CITED: https://doc.qt.io/qt-6/qwidget.html#resizeEvent — "Calling resize() or setGeometry() inside resizeEvent() can lead to infinite recursion."] Reparenting child widgets is safe because reparenting triggers a child geometry update, not a parent resize.
- On Wayland (the deployment target per memory `project_deployment_target.md`), resize events are coalesced by the compositor — successive small resizes during a window drag arrive as a single resizeEvent per frame, not one per pixel. This makes the per-resize reflow cost trivially affordable. [VERIFIED: empirical Phase 72 UAT observations + Wayland's protocol design.]
- During reparent — e.g., Phase 72's peek-overlay `adopt()` reparents `station_panel` between splitter and overlay — NowPlayingPanel may receive a spurious resize event as the layout re-flows. The reflow helper MUST be idempotent (Pattern 2 already is — no-op when already in target layout) so spurious events are harmless. [VERIFIED by analysis of `_open_peek_overlay` at `main_window.py:1060-1080` which reparents station_panel and explicitly resnapshots splitter sizes — adjacent NowPlayingPanel resize is expected.]

### Anti-Patterns to Avoid

- **Computing the threshold inside `resizeEvent` on every call.** Walks the row's 9 widgets, sums minimum size hints — measurable cost on rapid resize. Cache the value; invalidate only when `_populate_stream_picker` rebuilds.
- **Calling `self.resize()` or `self.setGeometry()` from inside `_reflow_stream_picker_row()`.** Infinite recursion per Qt warning.
- **Toggling a container widget's `setVisible(False)` for "empty row 2".** Unnecessary — Qt collapses a zero-item QHBoxLayout to zero height automatically. Adding a wrapper widget + hide doubles the code surface for no gain.
- **`lambda` in any signal connect for the reflow helper.** Project-wide QA-05 violation. Use bound methods (`self._reflow_stream_picker_row`).
- **Re-querying `_populate_stream_picker` from inside the reflow helper.** Single-source-of-truth invariant (Phase 47.1 WR-02 / Phase 67 M-02): `_populate_stream_picker` owns `stream_combo.setVisible(...)`; the reflow helper only owns layout membership. Two distinct concerns, two distinct owners.
- **Coupling the wrap trigger to compact-mode state.** Explicit CONTEXT D-01 violation. Width is the sole input.
- **Hand-tuned constant 700px for the threshold.** Breaks if Phase 47.2's EQ button or Phase 72's compact-toggle is ever resized. Self-correcting compute (Pattern 4) is the recommended approach per CONTEXT.
- **Adding a hysteresis dead-band before observing real flicker.** Premature optimization; Wayland coalescing makes flicker unlikely. CONTEXT explicit deferral.
- **Calling `stream_combo.disconnect()` + reconnect before/after reparent.** Unnecessary and dangerous — signals survive reparent per Qt parent-system semantics. Adding disconnect/reconnect introduces a window where a fast user click during reparent would not fire `_on_stream_selected`.
- **`updateGeometry()` after `setSizePolicy()`.** Already called internally by `setSizePolicy()`. Redundant; harmless but signals "developer wasn't sure."

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|

## Common Pitfalls

### Pitfall 1: Snapshot computed from stale sizeHint values during first paint

**What goes wrong:** The reflow helper's threshold compute runs from `widget.minimumSizeHint().width()` for each row-1 widget. On first show (BEFORE first paint completes), some widgets may report `minimumSizeHint().width() == 0` or unreasonable values, leading to a falsely-low threshold. The picker then "wraps" at unexpectedly wide windows.

**Why it happens:** Qt computes sizeHint lazily; before the widget has been laid out at least once, the hint may be uninitialized or reflect a default that doesn't account for icon size, font metrics, etc. Wayland's coalesced events can mean the first `resizeEvent` fires before paint.

**How to avoid:** Compute the threshold lazily — first call happens after `_populate_stream_picker`, which is invoked from `bind_station`, which is called only after the panel is shown. By that point all widgets have been painted at least once and their sizeHints are reliable. Cache the result; invalidate the cache when `_populate_stream_picker` rebuilds.

**Warning signs:** Tests that construct the panel and immediately call `resizeEvent` without going through `bind_station` see the picker wrap at the wrong widths. Add an "is panel shown" guard or compute-on-first-_populate to mask.

### Pitfall 2: `controls.spacing()` returns -1 (meaning "default")

**What goes wrong:** Threshold compute uses `layout.spacing()` to count inter-widget gaps. If the layout was created without an explicit `setSpacing()` call, `spacing()` returns -1, meaning "use the style's default" — but the *actual* spacing applied is the style's value, not -1. Adding -1 * 8 = -8 to the threshold underestimates.

**Why it happens:** Qt's QLayout uses -1 as a sentinel for "not explicitly set." The existing code at `now_playing_panel.py:441` DOES call `controls.setSpacing(8)`, so this is fine for Phase 72.1 — but be defensive in case a future refactor removes the call.

**How to avoid:** Compute spacing as `max(layout.spacing(), 8)` or guard with `if layout.spacing() < 0:`. Since `controls.setSpacing(8)` is set explicitly, the simpler `layout.spacing()` direct read is fine, but a code comment noting the dependency would be wise.

**Warning signs:** Threshold compute returns a smaller value than expected; picker wraps at wider widths than the actual row-overflow point.

### Pitfall 3: resizeEvent fires before stream_combo is constructed

**What goes wrong:** Qt may deliver a `resizeEvent` during `NowPlayingPanel.__init__` if the panel's geometry is set before all child widgets are constructed. The reflow helper accesses `self.stream_combo` — `AttributeError` if construction hasn't reached line 484 yet.

**Why it happens:** `super().__init__(parent)` may trigger an initial layout pass; setting `parent` in any widget constructor can cascade resize events up the parent chain.

**How to avoid:** Guard the reflow helper with `if not hasattr(self, 'stream_combo'):` early-return, OR construct `stream_combo` and the layouts BEFORE the panel becomes visible (which is the current order in `__init__` — stream_combo is at line 484, panel is added to splitter in `main_window.py:330` which fires after `__init__` returns).

**Warning signs:** AttributeError on first construction during pytest setup. Defensive guard is cheap.

### Pitfall 4: Reparent during peek-overlay show triggers spurious resize on NowPlayingPanel

**What goes wrong:** Phase 72's peek-overlay `adopt()` method (`station_list_peek_overlay.py:55-77` / called from `main_window.py:1073`) reparents `station_panel` out of the splitter into a QFrame overlay. This causes the splitter to reflow — Qt fills the now-vacant slot with a placeholder that briefly redistributes width. NowPlayingPanel may receive a transient `resizeEvent` with a non-final width during this transition.

**Why it happens:** Documented in Phase 72-04-SUMMARY (one of the lessons captured in `72-LEARNINGS.md`): "Qt creates a placeholder slot when a QSplitter child is reparented OUT." MainWindow's `_open_peek_overlay` already mitigates by capturing-and-restoring splitter sizes around the reparent — but during the in-between, NowPlayingPanel sees the intermediate width.

**How to avoid:** The reflow helper is idempotent (Pattern 2). A spurious resize that wraps the picker, immediately followed by the restore resize that un-wraps, is visually instant on Wayland (coalesced into one frame). No additional mitigation needed. If observed to flicker, add a hysteresis dead-band (CONTEXT-deferred polish).

**Warning signs:** Picker briefly visible in row 2 during peek-overlay open/close even though the panel ends up wide enough for row 1. Visually instant; likely not observable.

### Pitfall 5: Stale-cache after a future widget addition to row 1

**What goes wrong:** Some future phase adds a 10th widget to row 1 (e.g., a "shuffle queue" button between volume and compact_toggle). The cached `_picker_row_min_width()` is computed from row 1's contents at the time of first compute. Adding a new widget post-cache doesn't invalidate.

**Why it happens:** The cache is invalidated only inside `_populate_stream_picker`. A widget addition during `__init__` of a future phase will not trigger any invalidation.

**How to avoid:** Document the cache-invalidation contract in the helper's docstring. Optionally, compute on every resize without caching (the cost is ~50µs on a 9-widget row — measurable but not significant on a desktop). Alternative: invalidate cache on a public method `invalidate_picker_row_cache()` that future phases call from their `__init__` edit point.

**Warning signs:** A future phase adds a widget to `controls`, the new layout works fine in wide mode, but at narrow widths the picker fails to wrap because the threshold is now stale. Subtle — would surface only at UAT.

### Pitfall 6: Tests that resize the panel before show() return stale layout state

**What goes wrong:** `widget.resize(w, h)` BEFORE the widget is shown does not trigger `resizeEvent` reliably on Wayland (and inconsistently on X11). Tests that call `panel.resize(600, 400)` then assert layout membership see no change.

**Why it happens:** Qt defers initial layout until the widget is shown. `resize()` on an unshown widget sets the requested size, but the layout system may not propagate to children until a paint cycle is triggered.

**How to avoid:** Mirror Phase 72's integration test fixture pattern at `test_phase72_integration.py:75-90`:
```python
w = MainWindow(...)
qtbot.addWidget(w)
w.resize(1200, 800)
with qtbot.waitExposed(w):
    w.show()
```
Then for resize assertions: `w.resize(new_w, new_h); qtbot.wait(0)` (the `wait(0)` lets the event loop deliver pending resize events). For panel-level tests, mount the panel in a parent QMainWindow or use `qtbot.waitExposed`.

**Warning signs:** Tests that "work" but produce inconsistent assertions about layout membership. Add `qtbot.waitExposed` to the fixture.

### Pitfall 7: Synthetic resize event bypasses real dispatcher (mirrors Phase 72's hover-peek bug)

**What goes wrong:** Phase 72 burned a full plan on this: `QApplication.sendEvent(centralWidget, QResizeEvent(...))` bypasses Qt's real dispatcher and may not trigger the same code path as a real Wayland resize. Phase 72 LEARNINGS line 80: "16 automated tests passed; the feature was non-functional on live Wayland."

**Why it happens:** Qt's real event dispatcher routes resize events through platform-specific machinery. Synthetic events skip parts of this. Hard-to-reproduce-without-actually-running issues lurk in the gap.

**How to avoid:** Use `widget.resize(w, h)` (the public API) rather than constructing a `QResizeEvent` and sending it via `QApplication.sendEvent`. The public API goes through the platform dispatcher. Reserve synthetic events for cases where you specifically need to test the event-filter dispatch logic. For Phase 72.1, all resize tests should use the public `resize()` API + `qtbot.wait(0)`.

**Warning signs:** All tests pass; live UAT reveals the picker never wraps. Phase 72 already paid this debt; Phase 72.1 inherits the lesson.

### Pitfall 8: Reparent helper called from inside another reparent (re-entrancy)

**What goes wrong:** If `_reflow_stream_picker_row()` calls `addWidget` which triggers a child geometry update which triggers a parent resize event which calls `_reflow_stream_picker_row()` again, you have re-entrancy. State mid-reparent could be inconsistent.

**Why it happens:** `addWidget` does NOT itself fire a `resizeEvent` on the parent (it triggers a layout invalidation that takes effect on the next layout pass), so direct re-entry is unlikely. But a poorly-timed call from the test harness could trigger it.

**How to avoid:** The helper is naturally idempotent (Pattern 2 — early-return when already in target layout). Re-entry is harmless because the second call sees the target state and exits immediately. Add an optional re-entry guard (`self._reflowing = True`) only if testing reveals an issue.

**Warning signs:** Tests time out or hang during resize. Unlikely given the idempotent helper, but worth a docstring note.

### Pitfall 9: First QResizeEvent override in NowPlayingPanel may shadow a Qt base override

**What goes wrong:** A search of the entire codebase (`grep -rn "def resizeEvent" musicstreamer/`) returns ZERO existing overrides. NowPlayingPanel inherits resizeEvent from QWidget which is a passive base — but if any of its mixed-in superclasses (currently only QWidget — verified by class definition at `now_playing_panel.py:~150` line) ever change to include resizeEvent logic, the override must continue to chain via `super().resizeEvent(event)`.

**Why it happens:** Defensive coding for future phase additions.

**How to avoid:** ALWAYS call `super().resizeEvent(event)` first. This is documented in Pattern 5 above and is the standard Qt subclassing convention.

**Warning signs:** Future Qt version migration breaks layout in subtle ways. Not relevant for 6.11.

### Resize-driven reflow helper

```python
# Source: derived from CONTEXT D-01..D-08 + Pattern 5 above

def _reflow_stream_picker_row(self) -> None:
    """Width-driven reparent of stream_combo between row 1 and row 2.

    Idempotent — no-op when state is already correct.
    Called from resizeEvent (every resize) and from _populate_stream_picker
    (multi→single transition).

    D-01: width-responsive, not mode-coupled.
    D-02: single-stream stays one-row (early-return when picker hidden).
    """
    if not self.stream_combo.isVisible():
        # D-02: single-stream stations stay one-row; ensure picker is in row 1
        # for the eventual multi-stream transition.
        self._move_stream_picker_to(self.controls)
        self._set_picker_size_policy(expanding=False)
        return
    panel_width = self.width()
    threshold = self._picker_row_min_width()
    should_wrap = panel_width < threshold
    if should_wrap:
        self._move_stream_picker_to(self._controls_row2)
        self._set_picker_size_policy(expanding=True)
    else:
        self._move_stream_picker_to(self.controls)
        self._set_picker_size_policy(expanding=False)
```
