```
grep -nE "stream_combo\.set" musicstreamer/ui_qt/now_playing_panel.py
grep -nE "(stream_combo|stream_picker).*tooltip|setToolTip.*stream" -i
```

**Findings:**
- No `setToolTip`, `setPlaceholderText`, `setStatusTip`, `setWhatsThis`, `setAccessibleName`, or `setAccessibleDescription` calls were added to `stream_combo` or `_controls_row2` (verified — empty grep result). This matches `72.1-UI-SPEC.md §Copywriting Contract (Delta)` row 1 ("Picker dropdown items: unchanged") and row 2 ("Row 2 label: n/a — no label").
- The picker's items continue to come from `_populate_stream_picker` at `now_playing_panel.py:1151-1172`, which preserves the Phase 39 D-19..D-22 format plus the Phase 70 hi-res tier suffix at `:1168-1169`. No string changes in this phase.
- Silent reparent (no toast, no notification) matches `Phase 72 D-Discretion` precedent — verified by absence of any `toast`/`notify`/`show_message` calls in the 5 new helpers.

**Score rationale:** UI-SPEC explicitly declares "no new copy" as the contract, and the implementation honors that exactly. 4/4.

### Pillar 2: Visuals (3/4)

**Audit method:** Inspect row 2 construction site and reparent helpers against UI-SPEC §Row 2 Visual Contract table.

**Verified against contract:**
- Row 2 container is `QHBoxLayout` (not a wrapper widget) — `now_playing_panel.py:564`. ✓ matches UI-SPEC.
- Parent layout is `self.center` QVBoxLayout — `:566` `center.addLayout(self._controls_row2)`. ✓
- Inserted between `center.addLayout(self.controls)` at `:558` and `center.addWidget(self._stats_widget)` at `:572`. ✓ matches UI-SPEC insertion index requirement.
- Internal spacing 8px (`:565` `setSpacing(8)`) matches row 1's 8px (`:445`). ✓
- No `setContentsMargins` on row 2 (grep confirms 0 matches). ✓
- No `setVisible` on row 2 (grep confirms 0 matches — relies on Qt auto-collapse of zero-item layouts). ✓
- No `addStretch` on row 2 (grep confirms 0 matches — picker uses Expanding policy to fill). ✓
- Picker is the ONLY widget that moves between rows — `_move_stream_picker_to` at `:1302-1331` touches only `self.stream_combo`; all other row 1 widgets (play/stop/edit/star/eq/volume/compact_toggle/stretch) remain in their original positions.
- Row-1-return uses `insertWidget(self._picker_row1_index, ...)` at `:1328` — preserves visual ordering between `edit_btn` and `star_btn` per UI-SPEC §Tab Order.

**Real visual concern (1-point deduction):**
- **Self-describing picker on row 2 is the contract** (UI-SPEC §Copywriting Contract row 2: "Row 2 label: n/a — no label. Picker is self-describing"), and the implementation honors it. However, this means that when a user resizes narrow and the picker appears on a new visual row beneath the transport controls with no accompanying widget, label, or visual frame, the *only* indication of what the row contains is the picker's own dropdown text (e.g., "320 kbps MP3 — EU Mirror"). For an experienced user this is clear; for a first-time encounter at narrow width it could be momentarily disorienting. The human UAT recorded a related behavioral surprise on test 2 (`72.1-HUMAN-UAT.md:22`: "I don't feel like this is an issue" — note the user pre-empting concern). This is a contract-permitted choice, but it represents the one place visual hierarchy on row 2 is thin.

**Score rationale:** Contract substantially met; one defensible-but-thin visual hierarchy choice on the new row keeps this from 4/4. 3/4.

### Pillar 5: Spacing (3/4)

**Audit method:**
```
grep -nE "setSpacing|setContentsMargins|setMinimumWidth|setFixedWidth|setFixedSize|addStretch" -- controls/stream_combo/row2 lines
```

**Spacing matches contract:**
- Row 2 internal spacing: `self._controls_row2.setSpacing(8)` at `now_playing_panel.py:565`. ✓ matches row 1's `self.controls.setSpacing(8)` at `:445`. UI-SPEC §Spacing Scale Delta row 1.
- Row 2 vertical gap from row 1: inherits `center` QVBoxLayout default — no `setContentsMargins`, no explicit spacing. ✓ matches UI-SPEC §Spacing Scale Delta row 2 ("Claude's Discretion").
- Row 2 horizontal margins: inherits `center` margins. ✓ matches UI-SPEC §Spacing Scale Delta row 3.
- Row 2 vertical position: between `:558` (`center.addLayout(self.controls)`) and `:572` (`center.addWidget(self._stats_widget)`). ✓ matches UI-SPEC §Spacing Scale Delta row 4.
- Picker `setMinimumWidth(140)` at `:489` preserved unchanged — UI-SPEC §Picker Size Policy Table notes this floor stays as the row-1 minimum.

**Minor finding (1-point deduction):**
- **Phantom gap when row 2 is empty.** When the picker is on row 1 (multi-stream + wide OR single-stream + any width), `self._controls_row2` is an empty `QHBoxLayout` with `setSpacing(8)` and no widgets. Qt collapses the layout itself to zero height (which the contract relies on), but the `center` QVBoxLayout still allocates ONE spacing-gap slot between `controls` (above) and `_controls_row2` (below), and another between `_controls_row2` and `_stats_widget`. With `center.spacing()` defaulting to typical Qt 6 values (often ~6px), this means the gap between row 1 and `_stats_widget` is roughly `2 × center.spacing()` instead of `1 × center.spacing()` even when row 2 is empty.
  - **Evidence:** UI-SPEC §Spacing Scale Delta row 4 specifies "no `setContentsMargins` on row 2; `center`'s existing spacing handles the gap." The contract permits this — the cost is implicit, not visible to a user, but it's worth a note: the spacing budget between transport-controls and stats-widget is fixed at "2 spacings" regardless of whether row 2 is visible.
  - **Why this isn't a WARNING:** Human UAT did not flag the gap; the four pass-results in `72.1-HUMAN-UAT.md` include the wide-width state where the gap is present. The choice is contract-permitted and the alternative (conditionally calling `setSpacing(0)` on the parent on row-2-empty transitions) would add complexity disproportionate to the visual gain.

**Score rationale:** All contract values met exactly; one implicit-but-acceptable spacing-budget side effect. 3/4.

### Pillar 6: Experience Design (3/4)

**Audit method:** Trace all state transitions in the truth table; verify each path leads to a consistent layout state; check for animation, focus, signal preservation, and re-entry safety.

**State coverage verified:**
- **Multi-stream + wide → row 1, Preferred 140px floor.** Test 1 (`test_wide_multi_stream_picker_in_row_1`) ✓ GREEN.
- **Multi-stream + narrow → row 2, Expanding.** Test 2 (`test_narrow_multi_stream_picker_in_row_2`) ✓ GREEN. Picker fills row 2.
- **Single-stream + any width → row 1, hidden.** Tests 3 + 4 ✓ GREEN. D-02 enforced via `isHidden()` early-return at `now_playing_panel.py:1384`.
- **Round-trip preserves state.** Test 5 (`test_signal_survives_round_trip`) ✓ GREEN — `currentIndexChanged`, `currentIndex`, `itemData` all survive wide→narrow→wide.
- **Mid-narrow station change rehomes picker.** Test 6 (`test_multi_to_single_mid_narrow_rehomes_picker`) ✓ GREEN — extension hook at `:1187-1189`.
- **Compact mode is orthogonal to picker row.** Test 7 (`test_compact_mode_and_picker_wrap_independent`) ✓ GREEN — but the human UAT (`72.1-HUMAN-UAT.md:22`) noted user surprise: "This still brings the stream selector back to row 1 (though I don't feel like this is an issue)." The contract IS D-01 width-driven; toggling compact OFF widens the right pane, which pushes width past the threshold, which correctly returns the picker to row 1. Implementation is correct; **discoverability is the gap**.
- **Stop button rehomes picker.** WR-02 fix at `_on_stop_clicked` `:1000-1001` — symmetric with `_populate_stream_picker`'s hook, prevents the picker from being invisible-but-stranded in row 2's layout slot after a Stop click.

**Symmetric writer audit (WR-02):**
The contract requires `_populate_stream_picker` to be the single writer of `stream_combo.setVisible(...)` (Pattern E). The implementation respects this **for the visibility write itself** (`:1172`), but TWO sites perform the layout rehome on hidden-transitions: the extension hook (`:1187-1189`) and `_on_stop_clicked` (`:1000-1001`). This is the correct response to the original Pattern E (visibility-only single-writer); layout management is a separate concern. The symmetric handling at both sites is a positive — without WR-02 the picker would be visually consistent only after the next resize event.

**Re-entry safety:**
- `_move_stream_picker_to` is idempotent (`:1316-1320`) — same-target calls return without modification.
- `_reflow_stream_picker_row` early-returns on `panel_width <= 0` (`:1371`) — guards against Qt's spurious zero-size resize during teardown / first show.
- `hasattr(self, 'stream_combo')` / `hasattr(self, '_controls_row2')` guards (`:1366-1369`) — prevent reflow during `__init__` before widgets exist.

**Real concerns (1-point deduction):**

**6a — No hysteresis dead-band at the threshold.** `_reflow_stream_picker_row` at `:1390` uses a single `panel_width < threshold` comparison. On a resize event stream that crosses the threshold once, the picker reparents once. On a drag that oscillates exactly at the threshold, the picker could oscillate. CONTEXT explicitly defers this ("ship without; observe; only add if real flicker reported"), and `72.1-HUMAN-UAT.md` test 4 reports "pass — no flicker observed" — so the deferral is validated for the Wayland deployment target. But there is no test that locks "no oscillation when resize is exactly at threshold" — if a future contributor changes the threshold compute or the test fixture, regression would be silent. *Recommendation: add a pytest-qt test that calls `panel.resize(threshold-1, 800); panel.resize(threshold+1, 800); panel.resize(threshold-1, 800); ...` and asserts the picker reaches the expected row at each tick without intermediate-state assertions* — this would at least confirm idempotency under threshold-boundary churn.

**6b — Compact-mode return-to-row-1 discoverability.** Human UAT test 2 surfaced a small expectation mismatch ("toggling compact off brings the picker back to row 1"). Implementation is contract-correct (D-01 width-driven). The user's intuition ("picker stays where I last saw it") differs from the contract ("picker follows width"). This is fine — the contract is the source of truth — but it suggests one of two future improvements *if* a second user reports the same surprise: (a) a brief tooltip on the picker explaining "moves with window width", or (b) a documented note in user-facing release notes. Neither is required for ship.

**6c — No accessibility label or tab-order assertion for the row-2 position.** The picker's `accessibleName` is unchanged across reparent (the contract is "Qt default derived from object name / placeholder"). For screen-reader users, the picker sounds identical in row-1 vs row-2 positions — same `QComboBox`, same widget identity. UI-SPEC §Tab Order specifies that when picker is on row 2, tab order is `play → stop → edit → star → eq → volume → compact (row 1) → stream_combo (row 2)`. This is implicit in Qt's natural layout tab order, and the codebase does not call `setTabOrder()` explicitly. **No test asserts this**. For a screen-reader user, an explicit `setTabOrder()` call would be more robust against future widget insertions; for the keyboard-only sighted user, Qt's default usually does the right thing. Not a defect, but a robustness gap.

**Score rationale:** All four states correctly handled; symmetric WR-02 fix in place; signal/model/palette preservation locked by tests. Threshold-boundary behavior is unverified-but-deferred per contract, compact-mode return UX is correct-but-counterintuitive, tab-order is implicit-not-explicit. 3/4.
