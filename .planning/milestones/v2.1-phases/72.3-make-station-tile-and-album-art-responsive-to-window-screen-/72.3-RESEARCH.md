**Researched:** 2026-05-19

## Summary

Phase 72.3 replaces five hard-coded pixel literals in `now_playing_panel.py` (lines 339, 745, 1468, 1495, 1499) with a single tier-driven integer N selected from `{140, 180, 240}` based on `NowPlayingPanel.width()` evaluated inside the existing Phase 72.1 `resizeEvent` override at line 1149. CONTEXT-D-04 fixes the three tier values; CONTEXT-D-05 enforces logo-equals-cover at every tier (the +20 px bump of the cover at the medium tier is the only visible default-state change). CONTEXT-D-06 fixes the trigger (`self.width()`); CONTEXT-D-07 enforces bound-method-only (QA-05). UI-SPEC formalizes the truth table and the per-render-site refactor map.

The work breaks into: (1) a pure `_current_art_tier_size(self) -> int` helper, (2) an `_apply_art_tier(self)` orchestrator that diffs against a cached `self._current_art_tier`, (3) extending the existing `resizeEvent` with one bound-method call to `_apply_art_tier`, (4) refactoring three pixmap render call sites to read the tier, and (5) a new `tests/test_phase72_3_responsive_art.py` file mirroring the Phase 72.1 fixture / resize idiom.

**Primary recommendation:** Use **560 тЖТ 700 тЖТ 1100** as the two threshold cuts (planner-locked). Compute the tier in `_current_art_tier_size()` via plain `if/elif/else` against `self.width()` тАФ do NOT compute from widget minimums in this phase (Phase 72.1 already does that for the picker row; the art tier is independent of row-1 composition and a constant table is cleaner for testing). Cache the resulting tier integer on `self._current_art_tier`. The early-return-on-no-change guard is what prevents drag-thrash, not hysteresis.

### Pattern 1: Extend An Existing `resizeEvent` Override

**What:** Add a second bound-method call to a `resizeEvent` override that was added by a previous phase.

**When to use:** Whenever multiple width-driven behaviors share a single widget. Qt requires exactly ONE `resizeEvent` override per class.

**Example:**
```python
# Source: musicstreamer/ui_qt/now_playing_panel.py:1149-1159 (current state after Phase 72.1)
# Modification: ADD line 3 (the _apply_art_tier call).

def resizeEvent(self, event):  # noqa: N802 (Qt override)
    """Override to drive width-responsive layout decisions on the panel.

    Phase 72.1 / LAYOUT-02: stream-picker reparent on narrow widths.
    Phase 72.3 / LAYOUT-03: logo/cover tier retarget.

    Ordering per Pitfall 9 in 72.1-RESEARCH (super() FIRST is mandatory),
    then helpers in declared phase order. Both helpers are idempotent тАФ
    safe to call when state hasn't changed (see Pattern 2 below).
    """
    super().resizeEvent(event)
    self._reflow_stream_picker_row()  # Phase 72.1 / LAYOUT-02 (existing)
    self._apply_art_tier()             # Phase 72.3 / LAYOUT-03 (NEW)
```

**Critical:**
- **`super().resizeEvent(event)` MUST be called first.** [CITED: Qt subclassing convention тАФ base implementation invalidates layout and updates internal geometry. https://doc.qt.io/qt-6/qwidget.html#resizeEvent] Phase 72.1's research already paid this debt (Pitfall 9 at `72.1-RESEARCH.md:523-531`).
- **Order of helpers is `_reflow_stream_picker_row` THEN `_apply_art_tier`.** Neither helper depends on state mutated by the other (proof below). Either order would be functionally correct; the declared order is "behaviors in phase-shipping order" so the diff against post-72.1 state is a single inserted line.
- **Both helpers are idempotent.** `_reflow_stream_picker_row` is no-op when picker is already in the correct row (Phase 72.1 Pattern 2). `_apply_art_tier` is no-op when computed tier equals cached tier (Pattern 2 below). Spurious resize events (e.g., during peek-overlay open per Phase 72.1 Pitfall 4) are absorbed safely.
- **Independence proof:** `_reflow_stream_picker_row` mutates `stream_combo`'s layout membership and size policy. `_apply_art_tier` mutates `logo_label.size()`, `cover_label.size()`, and their pixmaps. The two mutation sets are disjoint. The picker is on row 2 only when the panel is at the narrow tier (panel < 700) тАФ but row-2 placement and tier-140 sizing are *correlated, not coupled* (they share width as input; neither reads the other's output state). [VERIFIED by inspection of `_reflow_stream_picker_row` at line 1305+ and the proposed `_apply_art_tier`.]

### Pattern 2: Cached-Diff Early-Return Guard

**What:** Recompute a derived value on every event; compare against a cached previous value; mutate widgets only when the value actually changed.

**When to use:** Any per-resize work whose output is a discrete value (tier number, layout slot, mode flag) and whose mutation cost is non-trivial (here: pixmap rescale + setFixedSize + layout invalidation cascade).

**Example:**
```python
# Source: derived from CONTEXT D-Claude-Discretion "Tier change detection"
#         + Phase 72.1's _reflow_stream_picker_row idempotence pattern

def _apply_art_tier(self) -> None:
    """Apply the current width-driven art tier; idempotent within a band.

    Recompute tier on every resize (cheap тАФ three integer comparisons).
    Mutate labels + bound pixmaps ONLY when the computed tier differs from
    the cached previous tier. Continuous drag within a single tier band
    is a no-op тАФ prevents pixmap re-render thrash.

    D-05: logo_label and cover_label receive the SAME N at every tier.
    D-Claude-Discretion: re-render the bound pixmap so the new label size
    is filled correctly (Qt does not auto-rescale because setScaledContents
    is False тАФ see now_playing_panel.py:341).
    """
    new_tier = self._current_art_tier_size()
    if new_tier == self._current_art_tier:
        return  # idempotent тАФ no-op within a tier band
    self._current_art_tier = new_tier
    self.logo_label.setFixedSize(new_tier, new_tier)
    self.cover_label.setFixedSize(new_tier, new_tier)
    # Re-render the bound pixmaps. The station logo is always rendered into
    # logo_label whenever a station is bound (see _show_station_logo call
    # sites in bind_station at line 807). The cover slot is more nuanced тАФ
    # see Pattern 4.
    self._rerender_art_for_current_tier()
```

**Critical:**
- **`new_tier == self._current_art_tier` short-circuit MUST be first.** Without it, a continuous splitter drag within a tier band re-renders both pixmaps on every resize tick. On Wayland the tick rate is one-per-frame; a 1-second drag would trigger ~60 unnecessary `QPixmap.scaled` calls.
- **`self._current_art_tier` starts as `None`** (sentinel). The first call (whether from `__init__` or from the first `resizeEvent`) always passes the diff guard and runs the full apply path.
- **`setFixedSize(N, N)` is a Qt-native sync mutation.** It calls `updateGeometry()` internally тАФ no need to call `update()` or `repaint()` explicitly. [CITED: https://doc.qt.io/qt-6/qwidget.html#setFixedSize] Layout invalidation propagates to the parent layout (the outer `QHBoxLayout`) and triggers a re-layout pass on the next event-loop tick. **It does NOT re-trigger `resizeEvent` on the panel** тАФ `setFixedSize` on a child widget invalidates the parent's layout but the panel's own geometry is unchanged (`width()` and `height()` are not affected by child-widget size changes). [VERIFIED: Phase 72.1's `setSizePolicy` calls on stream_combo do not produce infinite recursion in production тАФ same Qt invariant.]

### Pattern 3: Pure Width-To-Tier Function

**What:** A single integer-returning function that maps `self.width()` to one of three discrete tier values.

**When to use:** Whenever the trigger is a continuous numeric value (width in pixels) and the output is a finite set of discrete values (tier sizes). Pure functions are trivially testable.

**Example:**
```python
# Source: CONTEXT D-04 (three tiers) + CONTEXT Claude's Discretion (700/1100 thresholds)
#         + UI-SPEC Threshold widths table

def _current_art_tier_size(self) -> int:
    """Return the current art tier size in pixels (140, 180, or 240).

    Pure function тАФ reads only self.width(), returns an integer. Safe to
    call from any context without side effects.

    Thresholds (CONTEXT-locked at 700 and 1100; derivation in 72.3-RESEARCH
    ┬зThreshold Derivation):
        width < 700  ->  140 (narrow)
        700 <= width < 1100  ->  180 (medium / default)
        width >= 1100  ->  240 (wide)

    The narrow tier is the safe default when width < 560 (the panel's own
    setMinimumWidth floor) тАФ should not normally occur in production but
    can happen during initial layout pass before the splitter has assigned
    sizes. Tier 140 is sized to fit comfortably at the 560-px floor.
    """
    w = self.width()
    if w < 700:
        return 140
    if w < 1100:
        return 180
    return 240
```

**Critical:**
- **Predicate uses `<`, not `<=`.** The UI-SPEC Tier State Truth Table at `72.3-UI-SPEC.md:121-123` formalizes the half-open intervals as `560 тЙд w < 700 тЖТ narrow`, `700 тЙд w < 1100 тЖТ medium`, `w тЙе 1100 тЖТ wide`. The Python `if w < 700` / `elif w < 1100` / `else` chain implements exactly that.
- **No clamping for `w < 560`.** The 560-px floor is enforced by `self.setMinimumWidth(560)` at line 326; in production the panel's width never goes below 560. Tests that construct the panel before show may briefly see `w == 640` (Qt's default QWidget size hint) or other transient values тАФ the function still returns a valid tier integer in all cases.
- **No caching inside this function.** The cache lives on `self._current_art_tier` and is read/written by `_apply_art_tier` (Pattern 2). Keeping the pure function pure simplifies unit testing тАФ a test can `panel.resize(800, 600)` and assert `panel._current_art_tier_size() == 180` directly without going through `_apply_art_tier`.

### Pattern 4: Re-render Bound Pixmaps After Tier Change

**What:** When the tier changes, re-render whatever is currently shown in `logo_label` and `cover_label` at the new size.

**When to use:** Any time a label's `setFixedSize` value changes and `setScaledContents(False)` means Qt will NOT auto-rescale the existing pixmap to fit.

**Example:**
```python
# Source: derived from CONTEXT D-03 (three call sites must read central tier)
#         + cover_label state analysis (see Critical below)

def _rerender_art_for_current_tier(self) -> None:
    """Re-render logo and cover pixmaps at the current tier size.

    Called from _apply_art_tier whenever tier changes. The station logo
    in logo_label is always the station's station_art_path (or fallback).
    The cover slot is more nuanced:

      - If a cover-art-ready fetch has delivered real artwork, the cover
        label holds that artwork's pixmap. Reading the last-known
        cover-art path lets us replay _set_cover_pixmap at the new tier.

      - If no real cover has been delivered (or the most recent fetch
        returned no path), the cover label is showing the station logo
        via _show_station_logo_in_cover_slot тАФ replay that path.

    The simplest contract that avoids clobbering real cover art with the
    station logo fallback: track the last-known cover-art path on the
    panel and replay it when present; else call the logo-in-cover fallback.
    """
    # Logo path is unambiguous тАФ it always shows the station logo.
    self._show_station_logo()
    # Cover slot: distinguish "real cover present" from "fallback".
    if self._last_cover_path is not None:
        # Real cover was previously delivered тАФ replay at new tier.
        self._set_cover_pixmap(self._last_cover_path)
    else:
        # Showing the station-logo fallback in the cover slot.
        self._show_station_logo_in_cover_slot()
```

**Critical:**
- **The cover slot has TWO possible states at any moment** тАФ (a) real cover art delivered by `cover_art_ready` signal, (b) station-logo fallback rendered by `_show_station_logo_in_cover_slot`. Re-rendering by *unconditionally* calling `_show_station_logo_in_cover_slot` would clobber state (a) with state (b) on every tier change. The contract above requires tracking the most recent cover path.
- **Recommended state-tracking:** add `self._last_cover_path: Optional[str] = None` to `__init__`. Update it inside `_set_cover_pixmap` (set to the path argument) and inside `_show_station_logo_in_cover_slot` (set to `None`). Then `_rerender_art_for_current_tier` reads the cached value and branches.
- **Alternative (simpler but less correct):** read `self.cover_label.pixmap()` and inspect тАФ but Qt does not expose the original source path through `QLabel.pixmap()`, so the alternative loses information. The state-tracking approach is cleaner.
- **Stale-cover-path edge case:** if `_last_cover_path` points to a temp file that has since been cleaned up, `QPixmap(path)` in `_set_cover_pixmap` (line 1463) returns a null pixmap and falls back to `_show_station_logo_in_cover_slot` (existing code at line 1465). Behavior is correct; the user briefly saw the real cover then sees the logo fallback at the new tier. Acceptable.
- **CONTEXT D-03 + UI-SPEC Per-Render-Site Source-of-Truth Mapping** require all three call sites to read the central tier. After refactor, each site computes `N = self._current_art_tier or 180` (with the medium default for the brief window between `__init__` and the first `resizeEvent`). The cached-or-default expression keeps tests deterministic when the panel is exercised before exposure.

### Pattern 5: Initial-Tier Bootstrap

**What:** Decide the initial label size before any resize event has fired.

**When to use:** Phase shipping that introduces width-driven sizing and must not regress the visual appearance at construction time.

**Example:**
```python
# Source: CONTEXT integration_points + UI-SPEC Initial-Bind Contract

# In __init__, near line 297 (next to the existing 72.1 cache init):
self._current_art_tier: Optional[int] = None
self._last_cover_path: Optional[str] = None

# At line 339, keep the literal default of medium tier:
self.logo_label.setFixedSize(180, 180)  # default medium tier per UI-SPEC Initial-Bind Contract

# At line 745, change from 160 to 180 (the +20px bump per D-05 at medium tier):
self.cover_label.setFixedSize(180, 180)  # default medium tier per UI-SPEC + D-05

# Then on the first real resizeEvent (which fires automatically when the panel
# is added to the splitter in MainWindow.__init__), _apply_art_tier runs,
# computes the actual tier from the real width, and resizes if different
# from 180. No __init__-time call to _apply_art_tier is needed тАФ Qt's first
# layout pass triggers a resizeEvent reliably.
```

**Critical:**
- **CONTEXT's recommendation `__init__`-end call OR `QTimer.singleShot(0, ...)` is NOT needed.** The first `resizeEvent` fires automatically when (a) the panel is added to a layout that has visible parents, or (b) the parent layout assigns the panel a non-zero size. In production both happen during `MainWindow.__init__` (verified by the fact that Phase 72.1's `_reflow_stream_picker_row` correctly fires on first layout without any `__init__`-end call or `QTimer.singleShot` тАФ see Phase 72.1's Pitfall 6 and the `qtbot.waitExposed` pattern).
- **The cover label defaults to 180├Ч180, not 160├Ч160.** This is the D-05 cover-bump at the medium tier, applied at construction time. Tests that construct the panel and inspect `cover_label.size()` without resizing will see 180├Ч180 тАФ this is the new baseline.
- **Initial pixmap render is unaffected.** Lines 1495 and 1499 use `QSize(self._current_art_tier or 180, ...)` which evaluates to `QSize(180, 180)` when `_current_art_tier` is still `None`. The pixmap matches the label size at construction time.
- **Test instantiation outside a real layout** (e.g., `panel = NowPlayingPanel(...)` without `qtbot.addWidget` or without a parent QSplitter) may not trigger an initial `resizeEvent`. The medium-default-at-construction approach means such tests still see sane sizes. The `panel.show() + qtbot.waitExposed()` pattern in the fixture (already established in Phase 72.1) triggers `resizeEvent` once the panel is exposed.

### Anti-Patterns to Avoid

- **Computing the tier inside the helper that mutates** (combined `_reflow_art_tier()`). Forecloses unit-testing of the pure threshold logic; couples width-classification with widget mutation. CONTEXT recommends pattern (a) split for testability.
- **Calling `self.resize(...)` or `self.setGeometry(...)` inside `_apply_art_tier`.** Same Qt warning as Phase 72.1 (https://doc.qt.io/qt-6/qwidget.html#resizeEvent тАФ "Calling resize() or setGeometry() inside resizeEvent() can lead to infinite recursion"). `setFixedSize` on a CHILD widget is safe (only the parent geometry is the unsafe target).
- **Re-rendering the cover label via `_show_station_logo_in_cover_slot` unconditionally** after a tier change. Clobbers real cover art with the logo fallback. Pattern 4 above describes the correct contract.
- **Re-rendering on every resize regardless of tier change.** Wastes CPU on continuous drag (60 fps ├Ч 2 pixmap scales ├Ч `Qt.SmoothTransformation` = measurable). Pattern 2's cache guard is non-negotiable.
- **Using `widget.setScaledContents(True)` to let Qt auto-scale.** Would auto-rescale at the new label size but loses pixmap quality (Qt's auto-scale is `FastTransformation`, not `SmoothTransformation`). The codebase explicitly opts out at line 341 (`self.logo_label.setScaledContents(False)`); the cover label has no such call but the existing render paths use `_load_scaled_pixmap` which pre-scales тАФ matching the logo's contract. Keep as-is.
- **Adding hysteresis dead-band before observing real flicker.** Premature optimization; Wayland's coalesced resize events make oscillation unlikely. CONTEXT explicit deferral.
- **Recomputing thresholds from widget minimums.** Phase 72.1 needs to (its threshold depends on which widgets share row 1); Phase 72.3 does not (the art tier is independent of row composition). A hand-tuned constant table is correct here.
- **Coupling the tier trigger to compact-mode state.** Explicit CONTEXT D-06 violation. Width is the sole input.
- **`lambda` in any connection or override for tier helpers.** Project-wide QA-05. Use bound methods (`self._apply_art_tier`).

## Threshold Derivation

CONTEXT marks 700 / 1100 as "planner-discretion within recommended bands." This research locks both values and documents the geometric justification.

### Narrow тЖТ Medium threshold = 700 px

**Constraint floor:** Panel `setMinimumWidth(560)` at `now_playing_panel.py:326` тАФ the panel can never report width < 560 in production.

**Center-column budget at 560 px with tier 140:**
- Outer layout margins: 16 (left) + 16 (right) = 32 px
- Outer layout spacing: 24 ├Ч 2 (between three columns) = 48 px
- Logo column: 140 px
- Cover column: 140 px
- **Center column = 560 тИТ 32 тИТ 48 тИТ 140 тИТ 140 = 200 px**

**Row 1 minimum width** (post-Phase-72.1 with picker on row 2):
- play_pause_btn (36) + spacing (8) + stop_btn (36) + spacing (8) + edit_btn (36) + spacing (8) + star_btn (28) + spacing (8) + eq_toggle_btn (28) + spacing (8) + volume_slider (120) + spacing (8) + compact_mode_toggle_btn (28) + addStretch
- Fixed widget sum: 36 + 36 + 36 + 28 + 28 + 120 + 28 = 312 px
- Spacing sum: 6 ├Ч 8 = 48 px
- **Row 1 minimum = 360 px**

At the 560-px floor with tier 140, the center column gets 200 px тАФ which is **less than the 360-px row-1 minimum**. The picker must be on row 2 (Phase 72.1 enforces this), AND the row-1 controls must compress beyond their stated minimums. This is the existing post-Phase-72.1 state at the narrow tier; tier 140 doesn't make it worse, it doesn't help either.

**Wait тАФ the center column at 560 px is already too narrow for row 1 (200 < 360).** This means the entire post-72.1 layout is overflowing at 560 px even with tier 140 art. The bottom-bar overlap that Phase 72 / 72.1 ostensibly fixed reappears at the absolute floor with current widget sizes. **This is an observation, not a regression introduced by 72.3.** Phase 72 / 72.1 honored the 560-px floor but did not assert center-column budget. 72.3 doesn't introduce the issue тАФ but tier 140 is still correct (smaller art = larger center column = closer to fitting).

**Center-column budget at 700 px with tier 140:**
- 700 тИТ 32 тИТ 48 тИТ 140 тИТ 140 = **340 px** тАФ still below 360-px row-1 minimum.

**Center-column budget at 700 px with tier 180:**
- 700 тИТ 32 тИТ 48 тИТ 180 тИТ 180 = **260 px** тАФ clearly below 360.

Conclusion: **the tier 140 тЖТ 180 cut at 700 px is conservative.** At widths in [560, 700) the panel chooses tier 140 (smaller art, larger center column). Widths тЙе 700 can absorb the +40 px art bump (2 ├Ч 20 px per side) without making the center column dramatically worse than its already-overflowing state.

**Lock: narrow тЖТ medium at 700 px.** Symmetric with the Phase 72 UAT-04 "560тАУ700 px wide" overlap range. Tests assert tier=140 at 560/600/699 and tier=180 at 700/800/1099.

### Medium тЖТ Wide threshold = 1100 px

**Center-column budget at 1100 px with tier 180:**
- 1100 тИТ 32 тИТ 48 тИТ 180 тИТ 180 = **660 px** тАФ comfortable for row 1 (360-px min).

**Center-column budget at 1100 px with tier 240:**
- 1100 тИТ 32 тИТ 48 тИТ 240 тИТ 240 = **540 px** тАФ still well above 360-px min.

**Center-column budget at 1100 px with tier 240 plus picker on row 1** (Phase 72.1 picker_row_min тЙИ 564 px including the 140-min picker + spacing + margins):
- 1100 тИТ 32 тИТ 48 тИТ 240 тИТ 240 = 540 px тАФ slightly below 564-px row-1-with-picker minimum.
- This means at 1100-1119 px, with a multi-stream station, the picker is on row 2 AND art is tier 240. As width grows past ~1120 px, the picker rejoins row 1 (Phase 72.1's threshold) and tier 240 is comfortable.

**Lock: medium тЖТ wide at 1100 px.** Below 1100 the medium tier (180) preserves the historical default; at and above 1100 the +60 px art bump (2 ├Ч 30 px per side) is absorbed by the center column with 540 px to spare. Tests assert tier=180 at 800/1000/1099 and tier=240 at 1100/1400/1920.

## Common Pitfalls

### Pitfall 1: Re-entering `resizeEvent` via `setFixedSize`

**What goes wrong:** `_apply_art_tier` calls `setFixedSize(N, N)` on `logo_label` and `cover_label`. If `setFixedSize` triggers a parent-layout re-layout that fires `resizeEvent` on the panel, we recurse.

**Why it happens:** Qt's `setFixedSize` invalidates the parent layout and schedules a re-layout. The re-layout sizes the children to the new minimums. **It does NOT change the panel's own width or height** unless the layout's `sizeHint()` propagates upward past the panel's own size policy. In production the panel's width is set by the QSplitter (parent) тАФ the child labels growing by ┬▒20-60 px do not push past the splitter's assigned width.

**How to avoid:** Verify in tests that after `panel.resize(800, 600)` and the resulting `_apply_art_tier` run, `panel.width() == 800` (the panel's geometry was not perturbed). The cached-diff guard in Pattern 2 also ensures that any spurious recursion sees a no-op on the second entry. [VERIFIED: Phase 72.1's `setSizePolicy` calls on `stream_combo` follow the same pattern in production without recursion.]

**Warning signs:** Tests hang or recurse infinitely on the first resize. Stack trace shows alternating `resizeEvent` тЖТ `_apply_art_tier` тЖТ ... тАФ should not happen with the diff guard, but the guard is the canary.

### Pitfall 2: `_set_cover_pixmap` clobbering tier state on stale cover delivery

**What goes wrong:** A `cover_art_ready` signal delivers a pixmap path that was fetched before a tier change. `_set_cover_pixmap` reads the CURRENT `self._current_art_tier` (correct тАФ D-03) and scales the pixmap correctly. But: the cover-fetch-token check at line 1455 (`if token != self._cover_fetch_token: return`) may not exist for the tier-change case. The result: a cover for the previous station / previous tier is rendered into the cover slot at the *new* tier size. Looks fine visually because the size matches the label; functionally wrong if the user already switched stations.

**Why it happens:** The cover-fetch-token mechanism is keyed on station change, not tier change. A within-station tier change does not invalidate any in-flight cover-art-ready signal.

**How to avoid:** No mitigation needed. A cover-art-ready signal delivered during a tier transition simply renders the cover at the new tier. Since the cover content is for the currently-bound station (token matched), the visual result is correct: the user sees the right cover at the right size. This is a non-pitfall on careful analysis. [Documented for completeness.]

### Pitfall 3: `_last_cover_path` going stale after temp-file cleanup

**What goes wrong:** Phase 73's `cover_art_mb` writes downloaded CAA images to a temp file path that gets passed to `cover_art_ready`. A later tier change calls `_set_cover_pixmap(self._last_cover_path)` тАФ but the temp file has been cleaned up by the OS or by an explicit teardown.

**Why it happens:** `QPixmap(path)` for a nonexistent file returns a null pixmap. The existing code at lines 1463-1466 handles this: null pixmap тЖТ `_show_station_logo_in_cover_slot()` fallback. The user briefly saw a real cover and now sees the logo at the new tier тАФ a graceful degradation.

**How to avoid:** No mitigation needed; the existing null-pixmap fallback at line 1465 handles it correctly. Document in `_rerender_art_for_current_tier`'s docstring that `_last_cover_path` may point to a stale path тАФ null-pixmap fallback covers the case.

**Warning signs:** A user complaint "the album cover disappeared after I resized the window" combined with a long-running session where the temp directory was cleaned. Rare.

### Pitfall 4: First `resizeEvent` fires before `self._current_art_tier` is initialized

**What goes wrong:** Qt may deliver a `resizeEvent` during `super().__init__(parent)` before `self._current_art_tier = None` runs. `_apply_art_tier` then reads `self._current_art_tier` and crashes with `AttributeError`.

**Why it happens:** `super().__init__(parent)` can fire an initial layout pass with a child `resizeEvent` cascade. The class-attribute initialization order matters.

**How to avoid:** Initialize `self._current_art_tier: Optional[int] = None` and `self._last_cover_path: Optional[str] = None` **before** any `setMinimumWidth` / `setLayout` calls in `__init__`. Recommended position: line 297, immediately after `self._cached_row1_min: Optional[int] = None` (the existing Phase 72.1 cache init). This places the attribute set *before* line 326's `self.setMinimumWidth(560)` and line 331's `outer = QHBoxLayout(self)`, neither of which can fire a `resizeEvent` cascade because the panel has no visible parent yet.

### Pitfall 5: `_show_station_logo` AttributeError when `self._station` is None

**What goes wrong:** `_apply_art_tier` calls `_rerender_art_for_current_tier`, which calls `_show_station_logo` (line 1493). `_show_station_logo` reads `self._station.station_art_path`. If the panel was just constructed and no station is bound, `self._station` is `None` and the code at line 1494 already handles it (`path = self._station.station_art_path if self._station else None`). `_load_scaled_pixmap(None, ...)` falls back to `_FALLBACK_ICON`. No crash.

**Why it happens:** Defensive code already in place. Just confirming the path is safe to call from `_apply_art_tier` regardless of bind state.

**How to avoid:** No mitigation needed тАФ confirmed safe. Test with a non-bound panel: construct, resize wide, assert logo_label.size() == (240, 240) and that the pixmap is the fallback icon scaled to 240├Ч240.

### Pitfall 6: Tests that resize before `show()` see no `resizeEvent`

**What goes wrong:** `widget.resize(w, h)` BEFORE the widget is shown does not trigger `resizeEvent` reliably on Wayland (and inconsistently on X11). Tests that call `panel.resize(600, 400)` then assert `panel.logo_label.size()` see the default 180├Ч180, not the expected tier.

**Why it happens:** Qt defers initial layout until the widget is shown. `resize()` on an unshown widget sets the requested size, but the layout system may not propagate to children until a paint cycle is triggered.

**How to avoid:** Use the established Phase 72.1 fixture pattern (`tests/test_phase72_1_stream_picker_reflow.py:140-141`):
```python
panel.show()
qtbot.waitExposed(panel)
# THEN: panel.resize(800, 600) followed by direct assertions
```

The `waitExposed` call ensures the panel is actually shown by the platform compositor (offscreen plugin in tests, Wayland in production) before any resize-driven assertion.

**Warning signs:** Tests pass intermittently or fail when run individually but pass in suite. Always reproduce with `pytest tests/test_phase72_3_responsive_art.py::test_name -x` after a `pytest --collect-only` to verify isolation.

### Pitfall 7: Synthetic resize events bypassing the real dispatcher (mirrors Phase 72 / 72.1)

**What goes wrong:** `QApplication.sendEvent(panel, QResizeEvent(...))` bypasses Qt's real dispatcher and may not exercise the same code path as a real Wayland resize. Phase 72 burned a full plan on this тАФ see Phase 72.1 Pitfall 7 (`72.1-RESEARCH.md:503-511`).

**How to avoid:** Use `widget.resize(w, h)` (the public API). The public API goes through the platform dispatcher. Reserve `QApplication.sendEvent` for event-filter-dispatch-logic tests only тАФ not for layout-decision tests.

**Warning signs:** All tests pass; live UAT reveals the tier never changes. Phase 72 / 72.1 already paid this debt; Phase 72.3 inherits the lesson.

### Pitfall 8: `_apply_art_tier` called from inside a tier transition (re-entrancy)

**What goes wrong:** `_apply_art_tier` calls `setFixedSize` which schedules a layout pass, which fires `resizeEvent`, which calls `_apply_art_tier` again. The cached-diff guard makes the second call a no-op тАФ but if a future refactor accidentally clears the cache mid-call, the loop could run unbounded.

**How to avoid:** The cached-diff guard is the primary defense. Optional re-entry guard (`self._applying_tier = True` flag) is overkill for this phase but could be added if a future regression triggers it. Add a docstring note that the helper must not clear `self._current_art_tier` between read and assign.

**Warning signs:** Pytest tests time out or recurse. Stack trace shows alternating `resizeEvent` тЖФ `_apply_art_tier`. Cached-diff guard prevents.

### Pitfall 9: Tier flickers during compact-mode toggle

**What goes wrong:** Toggling compact mode (`Ctrl+B` from Phase 72) hides the StationListPanel and the QSplitter redistributes width. NowPlayingPanel's width jumps from ~840 (split) to ~1400 (fullscreen) instantly. `resizeEvent` fires; `_apply_art_tier` sees the new width; tier may flip from 180 to 240. Visible to the user as an instant art bump synchronous with the compact-toggle.

**Why it happens:** This is **intentional and correct**. CONTEXT D-06 specifies width-driven trigger regardless of compact-mode state. The art bump is a natural consequence тАФ the user moved to a wider effective pane and the art grew to match.

**How to avoid:** No mitigation needed. Document in tests: assert that compact-mode toggle at a window size that crosses a tier threshold produces the expected tier flip. Phase 72's tests already exercise compact-mode toggle; 72.3 adds a tier-assertion variant.

### Pitfall 10: `qtbot.waitExposed` is a context manager, not a function call

**What goes wrong:** Phase 72.1 uses `qtbot.waitExposed(panel)` as a function call (line 141 of `test_phase72_1_stream_picker_reflow.py`). Phase 72 integration test at `tests/test_phase72_integration.py:88` uses it as a context manager: `with qtbot.waitExposed(w): w.show()`. Both forms exist in pytest-qt; the function-call form returns the qWaitForWindowExposed-equivalent. Phase 72.1's pattern (function call after `show()`) works in practice.

**How to avoid:** Mirror Phase 72.1's exact idiom for consistency:

def _current_art_tier_size(self) -> int:
    """Return the current art tier size in pixels (140, 180, or 240).

    Pure function тАФ reads only self.width(). Maps the panel's own width to
    one of three discrete tier values per the UI-SPEC truth table:

        width <  700  ->  140  (narrow tier)

    Tier values are CONTEXT-locked at 140/180/240 (D-04); threshold values

### Apply-tier orchestrator

```python
# Source: CONTEXT D-Claude-Discretion (Tier change detection)
#         + 72.3-RESEARCH ┬зArchitecture Pattern 2 (cached diff)
#         + Pattern 4 (re-render with cover-state awareness)

def _apply_art_tier(self) -> None:
    """Apply the current width-driven art tier to logo and cover labels.

    Idempotent тАФ no-op when the computed tier matches self._current_art_tier.
    Called from resizeEvent (every panel resize) after Phase 72.1's
    _reflow_stream_picker_row.

    D-02 (single-tier source of truth): both labels receive the same N.
    D-05 (equal at every tier): logo and cover sizes are always equal.
    D-Claude-Discretion (re-render on tier change): pixmaps are re-rendered
    via the existing _show_station_logo / _set_cover_pixmap paths so the
    new label size is filled correctly (setScaledContents is False; Qt
    does not auto-rescale).
    """
    new_tier = self._current_art_tier_size()
    if new_tier == self._current_art_tier:
        return  # idempotent тАФ no-op within a tier band
    self._current_art_tier = new_tier
    self.logo_label.setFixedSize(new_tier, new_tier)
    self.cover_label.setFixedSize(new_tier, new_tier)
    # Re-render logo (always shows station_art_path or fallback icon).
    self._show_station_logo()
    # Re-render cover slot: branch on whether a real cover is loaded.
    if self._last_cover_path is not None:
        self._set_cover_pixmap(self._last_cover_path)
    else:
        self._show_station_logo_in_cover_slot()
```

## Open Questions

1. **Should `_last_cover_path` be cleared on `bind_station`?**
   - **What we know:** `bind_station` (line 785) resets several "last" fields (`self._last_cover_icy = None`, `self._last_icy_title = ""`). It also calls `_show_station_logo_in_cover_slot()` (line 808) before the cover-art-fetch worker can deliver real art.
   - **What's unclear:** If `_show_station_logo_in_cover_slot()` resets `self._last_cover_path = None` (per Pattern 4's contract), the bind path is correct. If the user re-binds the same station and a cover-art-fetch was already in flight, the in-flight cover might land *before* `_show_station_logo_in_cover_slot` runs. Order is: bind_station тЖТ `_show_station_logo_in_cover_slot` тЖТ cover-art-fetch returns тЖТ `_set_cover_pixmap`. The reset happens first; the cover arrives second; the last-path tracking ends in the correct state.
   - **Recommendation:** No explicit reset needed in `bind_station`; the existing call to `_show_station_logo_in_cover_slot` at line 808 handles it. Document in tests: bind-station with no in-flight cover sets `_last_cover_path` to None; bind-station followed by cover-art-ready sets it to the new path.

2. **Should the planner expose `_apply_art_tier` as a public method for MainWindow to call (e.g., after a compact-mode toggle)?**
   - **What we know:** Compact-mode toggle changes the panel's width via the QSplitter, which fires `resizeEvent`, which calls `_apply_art_tier`. No MainWindow involvement needed.
   - **What's unclear:** A potential future "force re-tier" surface (e.g., theme change or settings dialog). Not requested by this phase.
   - **Recommendation:** Keep `_apply_art_tier` private. If a future phase needs an external trigger, a one-line public wrapper can be added.

3. **Does the YouTube 16:9 letterbox aspect ratio survive `Qt.SmoothTransformation` at sub-pixel-fractional sizes?**
   - **What we know:** `Qt.KeepAspectRatio` produces a rectangle with the same aspect ratio as the source, fit into the target QSize. For a 16:9 source at 240├Ч135 target, the math is exact (240 ├Ч 9/16 = 135). At tier 180: 180 ├Ч 9/16 = 101.25 тЖТ Qt rounds to 101. At tier 140: 140 ├Ч 9/16 = 78.75 тЖТ Qt rounds to 79.
   - **What's unclear:** Whether the rounding introduces a 1-pixel aspect-ratio drift visible to the user. For a 16:9 source at 180├Ч101, the actual aspect is 1.782 vs. 1.778 (16/9). Difference is 0.2%.
   - **Recommendation:** Test asserts aspect-ratio within ┬▒1 pixel tolerance, not exact equality. `assert abs(pixmap.width() / pixmap.height() - 16/9) < 0.01` is sufficient.
