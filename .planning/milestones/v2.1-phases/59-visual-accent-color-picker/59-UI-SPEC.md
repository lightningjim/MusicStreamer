# Phase 59 — UI Design Contract

> Visual and interaction contract for the AccentColorDialog rewrite (`musicstreamer/ui_qt/accent_color_dialog.py`). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Stack note:** This is a PySide6 6.11 / Qt 6 desktop app on Linux X11 (DPR=1.0) with Windows as a secondary target. The shadcn / 8-point-grid / 60-30-10 palette framing in the template is adapted to a Qt-native context. Most pixel-level visual properties are determined by Qt's style on the deployment target (Fusion or system) and we **inherit** them rather than re-specify them. Sections below are explicit about what the wrapper OWNS vs what the inner `QColorDialog` INHERITS.

## Spacing Scale

Multiples of 4 — applies only to the wrapper `QDialog`. Inner `QColorDialog` uses Qt-default spacing and is NOT re-specified.

| Token | Value | Usage |
|-------|-------|-------|
| sm | 8px | Wrapper `QVBoxLayout` content margins (all four edges); spacing between the embedded `QColorDialog` widget and the `QDialogButtonBox` row |
| md | 16px | NOT USED in this dialog — today's `setContentsMargins(16, 16, 16, 16)` + `setSpacing(16)` is reduced to 8/8/8/8 (sm) so the wrapper does not visually inflate beyond the inner dialog's intrinsic 522×387 footprint (RESEARCH Pitfall 7) |

**Wrapper layout (locked):**
```
QVBoxLayout(self)
    .setContentsMargins(8, 8, 8, 8)
    .setSpacing(8)
    .addWidget(self._inner)              # QColorDialog (NoButtons | DontUseNativeDialog)
    .addWidget(self._button_box)         # QDialogButtonBox: Apply | Reset | Cancel
```

**Sizing exceptions (locked):**
- **Drop today's `setMinimumWidth(360)`** (per RESEARCH Pitfall 7 — the inner `QColorDialog` reports `sizeHint() == QSize(522, 387)` and dominates the wrapper minimum; the 360px hint is dead). Let the wrapper size to the inner widget's natural footprint plus the 8px margins + button row.
- **Do not set `setFixedSize` or `setMinimumSize` on the wrapper.** The dialog should be free to grow if a future Qt version adjusts `QColorDialog`'s intrinsic size; the user's window manager handles centering on parent.
- **No exceptions for icon-only touch targets.** No icon-only buttons in this dialog.

## Color

This phase does not introduce new color tokens. It is a UI for *choosing* the existing `accent_color` value (the Highlight palette role override). The wrapper itself is not themed beyond Qt defaults.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `QPalette.Window` (Qt default for the deployment target) | Wrapper `QDialog` background; inner `QColorDialog` background |
| Secondary (30%) | `QPalette.Button` (Qt default) | `QDialogButtonBox` button surfaces |
| Accent (10%) | `QPalette.Highlight` — currently selected color in the dialog (LIVE) | The accent value being previewed: applied to all consumers of `palette(highlight)` (Stop button, slider sub-page via `build_accent_qss`, chips, segments, filter toggles) |
| Destructive | `_theme.ERROR_COLOR_HEX = "#c0392b"` | NOT USED in this dialog — kept here for the reserved-list audit |

**Accent reserved for (system-wide, not just this dialog):**
- The `Stop` playback button (`palette(highlight)` background)
- The `QSlider::sub-page:horizontal` (volume + buffer indicator) via `build_accent_qss` — written to `paths.accent_css_path()` on Apply
- Filter chip / segment / station-row selection backgrounds (consumers of `palette(highlight)` in their per-widget QSS, see `accent_utils.apply_accent_palette` rationale)
- `QPalette.HighlightedText` (forced to white inside `apply_accent_palette` for legibility on the new highlight)

**Destructive actions in this dialog:** none. Reset is a recoverable action (clears `accent_color` setting; theme + snapshot still in effect; dialog stays open). It is NOT styled with the destructive color.

**Color-flash guard (RESEARCH Pitfall 3):** if `repo.get_setting("accent_color", "")` returns a value that fails `_is_valid_hex(...)`, the wrapper MUST fall back to `ACCENT_COLOR_DEFAULT` before calling `setCurrentColor`. A bare `QColor(invalid_hex)` constructs to `#000000` and would flash the entire app's accent black for one frame on dialog open.

## Interaction & Feedback Contract

This section is owned by Phase 59 and is the load-bearing UX shape for the design.

### Modality & lifecycle

- Wrapper is **modal**: `setModal(True)` (D-10). User must Apply / Reset+(Apply|Cancel) / Cancel before resuming main window interaction.
- **Window-manager close (X button, Alt+F4 on Windows, super-q on some Linux WMs)** routes through `QDialog.reject()` — same code path as the Cancel button. Snapshot is restored. No additional "Are you sure?" confirm dialog (changes were preview-only and are losslessly reverted).
- **Esc key** routes through `reject()` (Qt default for dialogs with a RejectRole button) — restores snapshot.
- **Enter / Return** routes through the AcceptRole default button (Apply). The wrapper MUST call `apply_btn.setDefault(True)` so Enter triggers Apply, matching today's `accent_color_dialog.py:112` implicit default.

### Button row

| Button | Role | Order | Default? | Triggers |
|--------|------|-------|----------|----------|
| Apply | `QDialogButtonBox.AcceptRole` | leftmost | **YES** (`setDefault(True)`) | `_on_apply` → `repo.set_setting` + write QSS file + `accept()` (D-14) |
| Reset | `QDialogButtonBox.ResetRole` | middle | no | `_on_reset` → clear setting + restore snapshot palette + `setCurrentColor(ACCENT_COLOR_DEFAULT)` + dialog stays open (D-15) |
| Cancel | `QDialogButtonBox.RejectRole` | rightmost | no (Esc still triggers it) | `reject()` → restore snapshot palette + QSS + `super().reject()` (D-13) |

**Button ordering rationale:** matches today's `accent_color_dialog.py:111-118` exactly, which itself follows the project convention (Apply | secondary | Cancel) — see `EqualizerDialog`, `AccountsDialog`. Note this is the GNOME / Linux convention; on Windows users typically expect (OK | Cancel | secondary), but maintaining cross-platform consistency with the rest of the app outweighs platform-native ordering for a single dialog.

### Live preview

- `QColorDialog.currentColorChanged(QColor)` fires on every user interaction with the inner widget: dragging through the hue ring, clicking a Custom Colors slot, typing in the hex field, dragging the sat-val square, or picking with the eyedropper.
- The connected slot is a **bound method** (`self._on_color_changed`, NOT a lambda — QA-05 ban on self-capturing lambdas).
- Slot body (locked, D-11 + D-12):
  ```python
  def _on_color_changed(self, color: QColor) -> None:
      self._current_hex = color.name()  # lowercase #rrggbb
      apply_accent_palette(QApplication.instance(), self._current_hex)
  ```
- **No throttle / no QTimer coalesce.** `apply_accent_palette` is cheap (palette swap + 1-line QSS via `setStyleSheet`); Qt batches repaints; no flicker is expected on the Linux X11 DPR=1.0 target. If real-world flicker is observed during UAT, promote to `QTimer.singleShot(50, …)` debounce — capture the alternative in PLAN.md per CONTEXT.md "Claude's Discretion".
- **Visual feedback expectations** (verified perceptually during UAT):
  1. Dragging through the hue ring updates the Stop button's background color in real time, smoothly (no stutter, no momentary black flash).
  2. Dragging through the hue ring updates the volume slider's sub-page color in real time, smoothly.
  3. Picking with the eyedropper applies the picked color the moment the user clicks (single-shot, no preview-while-hovering — Qt's eyedropper convention).
  4. Reset: the entire app's accent visually returns to **default blue** (`#3584e4`) within one frame (~16ms). The picker's wheel + numeric fields also visually return to blue.
  5. Cancel: the entire app's accent visually returns to whatever was in effect when the dialog opened (saved hex, or unset → theme/system default), within one frame.

### Pre-population on open (D-17 + Pitfall 6)

1. `saved = repo.get_setting("accent_color", "")`.
2. `initial = saved if _is_valid_hex(saved) else ACCENT_COLOR_DEFAULT`.
3. **Wire `currentColorChanged` to `_on_color_changed` BEFORE** calling `setCurrentColor(QColor(initial))` — OR — call `setCurrentColor` first and then explicitly set `self._current_hex = initial` before connecting (RESEARCH Pitfall 6). Either approach is acceptable; the wrapper MUST guarantee `self._current_hex == initial` after `__init__` returns.
4. The picker visually reflects the initial color (wheel + sat-val square + numeric fields + hex field all in sync — Qt handles the cross-field syncing internally).

### Snapshot & restore (D-13)

- Snapshot taken in `__init__` BEFORE any preview wiring:
  ```python
  app = QApplication.instance()
  self._original_palette = app.palette()    # QPalette is value-copy; safe to retain
  self._original_qss = app.styleSheet()     # QString-equivalent; value-copy
  ```
- Cancel / X / Esc → `reject()`:
  ```python
  app.setPalette(self._original_palette)
  app.setStyleSheet(self._original_qss)
  super().reject()
  ```
- Reset → `_on_reset` → `reset_accent_palette(app, self._original_palette)` (which is `app.setPalette(snapshot); app.setStyleSheet("")` per `accent_utils.py:68-71`).

### Idempotent reseed of Custom Colors row (D-03)

In `__init__`, BEFORE constructing the inner `QColorDialog`:
```python
for idx, hex_val in enumerate(ACCENT_PRESETS):  # 0..7
    QColorDialog.setCustomColor(idx, QColor(hex_val))
```
Slots 8..15 are NOT touched — they remain user-editable within a session and die with the process (D-16). The reseed is idempotent — opening the dialog twice in one session restores slots 0..7 back to the curated presets even if the user edited a slot via the inner dialog's "Add to Custom Colors" button.

## Accessibility & Keyboard Support

The wrapper inherits Qt's accessibility infrastructure for screen readers (orca on Linux, Narrator on Windows) — both report dialog title + currently-focused widget label automatically. No additional `setAccessibleName` / `setAccessibleDescription` calls are required because all our owned widgets (the 3 buttons) have user-visible labels that double as accessible names by default.

| Surface | Behavior | Owner |
|---------|----------|-------|
| Tab order on open | Tab traverses **into** the inner `QColorDialog` first (Qt's natural focus chain — wheel/sat-val/numeric fields/hex field/eyedropper/Custom Colors slots), then to Apply → Reset → Cancel | Qt-default + wrapper layout order |
| Initial focus on open | First tab-stop in the inner `QColorDialog` (typically the Basic Colors row or the hue ring, Qt-style-dependent) | Qt default |
| Esc key | Trigger `reject()` (Cancel + restore snapshot) | Qt default for dialogs |
| Enter / Return key | Trigger Apply (the AcceptRole default button) | OWNED — wrapper MUST call `apply_btn.setDefault(True)` |
| Eyedropper keyboard activation | Tab to "Pick Screen Color" button → Space/Enter activates → cursor changes to crosshair → arrow keys move screen cursor (Qt 6 default) | Qt default — INHERITED |
| Custom Colors slot keyboard activation | Tab to slot → Space/Enter selects → fires `currentColorChanged` | Qt default — INHERITED |
| Screen reader on dialog open | Reads window title "Accent Color" + reads first focused widget's accessible name | Qt-default infrastructure |
| Screen reader on button focus | Reads button label ("Apply" / "Reset" / "Cancel") | Qt-default infrastructure |
| High-contrast / color-blind users | The picker itself shows numeric R/G/B/H/S/V values (inherited from `QColorDialog`) which is the gold-standard accessibility affordance for color pickers — users who can't visually distinguish the wheel can type/read exact values | INHERITED — do not theme away |

**Forbidden:** do NOT call `setFocusPolicy(Qt.NoFocus)` on any wrapper widget — that breaks keyboard navigation. Do NOT call `setVisible(False)` on the eyedropper / Custom Colors row to "simplify" the picker — that defeats ACCENT-02's "visual color picker, in addition to the existing 8 presets and hex entry" requirement.

**Linux X11 DPR=1.0 deployment target — accessibility caveats:**
- The eyedropper has known mouse-grab issues on some compositors (RESEARCH Pitfall 4 — WARNING-severity). Document as a known limitation in PLAN.md; do not gate phase completion on perfect eyedropper accuracy across all Linux desktops.

## States Inventory

The wrapper has these runtime states. All are user-driven; none are async/network-bound.

| State | Trigger | Visual signal | Persistence side effect |
|-------|---------|--------------|-------------------------|
| **Idle (just opened)** | `__init__` complete | Wheel + sat-val + numeric fields all show `initial` (saved hex or `ACCENT_COLOR_DEFAULT`); app accent matches `initial` (already applied at startup, or by `setCurrentColor` triggering `currentColorChanged`) | none — read-only of repo |
| **Previewing** | User drags wheel / clicks slot / types hex / drags sat-val / picks with eyedropper | App accent live-updates everywhere `palette(highlight)` is consumed; picker reflects the chosen color | none — only `QApplication.palette()` + `styleSheet()` mutated in process memory |
| **Applied** | User clicks Apply (`accept()` triggered) | Dialog closes; app accent stays at `self._current_hex` | `repo.set_setting("accent_color", hex)` + write `paths.accent_css_path()` (try/except OSError — non-fatal) |
| **Reset (in progress)** | User clicks Reset | App accent visually returns to `ACCENT_COLOR_DEFAULT`; picker wheel/fields visually return to blue; dialog stays open | `repo.set_setting("accent_color", "")` + (optional) remove `paths.accent_css_path()` per CONTEXT.md D-15.6 (Claude's Discretion — planner picks: leave alone, write empty, or `os.remove(...)` wrapped in try/except OSError) |
| **Cancelled** | User clicks Cancel / Esc / X | Dialog closes; app accent restored to snapshot (whatever was in effect when dialog opened) | none — repo not touched |
| **Defensive: corrupt saved hex** | `repo.get_setting("accent_color")` returns a non-`#rrggbb` value | Picker opens with `ACCENT_COLOR_DEFAULT` (blue); app accent does NOT flash black | none — fallback path is silent |
| **Defensive: Apply with empty `_current_hex`** | Apply clicked when `_current_hex == ""` (e.g., right after Reset, before any new selection) | No-op — `self.accept()` is NOT called; dialog stays open | none — guarded by `_is_valid_hex(self._current_hex)` check |

**Note on the "Idle (just opened)" state:** because `setCurrentColor` emits `currentColorChanged` for an actual change-from-prior-state, opening the dialog when no `accent_color` is saved (so initial = DEFAULT) AND the app's current palette has a different highlight than DEFAULT will trigger one initial preview emission. This is harmless — `apply_accent_palette` repaints to DEFAULT and the user sees the app accent shift to blue if it wasn't already blue. Document this in PLAN.md so UAT testers don't flag it as a regression.
