  <how-to-verify>
**Setup:**
1. Confirm running on a Linux X11 desktop (deployment target per project memory: DPR=1.0). Wayland is out of scope — if running under Wayland, switch to an X11 session for this UAT.
2. Launch the app from a terminal so any tracebacks are visible: `python3 -m musicstreamer` or whatever the local launch command is.

**SC #1 — Visual color picker present alongside 8 swatches and hex field**
1. Open the hamburger menu → click `Accent Color`.
2. Observe the dialog. Confirm visible:
   - HSV wheel OR hue slider (Qt-style-dependent — Linux Fusion typically renders a hue ring + sat-val square)
   - Saturation/value square
   - R, G, B numeric input fields
   - H, S, V numeric input fields
   - Hex field (built into QColorDialog)
   - "Pick Screen Color" eyedropper button
   - **Custom Colors row** with the first 8 slots showing the ACCENT_PRESETS values (Blue, Green, etc. — confirm 8 distinct non-white colors in slots 0..7)
   - Wrapper button row at the bottom showing exactly: `Apply` | `Reset` | `Cancel` (left to right)
3. **Record:** SC #1 PASS / FAIL + screenshot path or 1-line description of what's visible.

**SC #2 — Live preview during drag**
1. With the dialog open, drag through the hue ring (or hue slider) at normal mouse speed.
2. While dragging, watch the app's **Stop** playback button background and the **volume slider sub-page** (the colored portion to the left of the slider knob).
3. Both should update in real time, smoothly, with no visible flicker, no momentary black flash, no stutter.
4. Drag through the sat-val square similarly. Same expectation.
5. **Record:** SC #2 PASS / FAIL. If flicker observed, log "FLICKER OBSERVED — promote to QTimer.singleShot(50, ...) per CONTEXT.md D-11 Claude's Discretion alternative; recommend follow-up Plan 04". If no flicker, log "No flicker — no-throttle ship correct".

**SC #3 — Hex entry still works**
1. Click into the QColorDialog's hex field (typically labeled `HTML:` or similar).
2. Clear the existing value and type `#9141ac` (Purple), press Enter or Tab.
3. Confirm the wheel + sat-val + R/G/B + H/S/V fields all update to reflect Purple.
4. Confirm the app's Stop button + volume slider sub-page update to Purple in real time.
5. **Record:** SC #3 PASS / FAIL.

**SC #4 — Persistence across app restart**
1. With the dialog still open, pick a distinctive non-default color (e.g., bright orange `#ff8800` — distinct enough to recognize on relaunch).
2. Click **Apply**. Dialog closes. App accent stays orange.
3. Quit the app **fully** (not minimize — File → Quit, or close all windows + ensure process exits).
4. Relaunch the app.
5. On first paint after relaunch (before opening any dialogs), confirm the Stop button + volume slider sub-page render in orange (the chosen color persisted across the restart).
6. **Record:** SC #4 PASS / FAIL.

**Eyedropper UAT (T-59-UAT row in VALIDATION.md)**
1. Open Accent Color again.
2. Click the **"Pick Screen Color"** button.
3. Cursor changes to a crosshair. Move the cursor anywhere on screen.
4. Click on a recognizable color — e.g., a desktop wallpaper region, a colored region in another app's window.
5. Confirm:
   - The picked color appears in the dialog's wheel + numeric fields + hex field
   - The app's accent live-previews to that picked color (Stop button, volume slider sub-page)
6. **Pitfall 4 awareness:** if the picked color does NOT match what was under the cursor (off-by-one z-order issues on certain X11 compositors), document the compositor (`echo $XDG_CURRENT_DESKTOP` and `echo $DESKTOP_SESSION`) but do NOT fail the phase — RESEARCH downgrades this to WARNING per project memory ("Linux X11 DPR=1.0 deployment target; HiDPI/Retina/Wayland-fractional findings downgrade from CRITICAL to WARNING").
7. **Record:** Eyedropper PASS / WARNING / FAIL.

**Reset UAT (D-15)**
1. Open Accent Color. Pick a non-default color (e.g., red `#e62d42`). Confirm live preview applies.
2. Click **Reset**.
3. Confirm:
   - App accent visually returns to **default blue (`#3584e4`)** within ~1 frame
   - Picker wheel + numeric fields visually return to blue
   - Dialog **STAYS OPEN** (does not close)
4. Click Cancel to close.
5. Relaunch the app (full quit + relaunch).
6. On first paint after relaunch, confirm the accent is **default blue** (Reset cleared the saved hex; main_window.py:189-192 falls back to whatever the theme/system default is — for Phase 59's pre-Phase-66 era, the app palette default).
7. **Record:** Reset PASS / FAIL.

**Cancel UAT (D-13)**
1. Set a known accent — e.g., open Accent Color, pick green `#33d17a`, click Apply. Confirm app accent is green.
2. Open Accent Color again. Pick red `#e62d42`. Confirm live preview to red.
3. Click **Cancel** (or press Esc).
4. Confirm:
   - App accent **returns to green** within ~1 frame (the snapshot from when the dialog opened)
   - No save happened (the green from step 1 is still the persisted value)
5. Open Accent Color one more time, observe picker opens with green. Click **the X / window-manager close button**.
6. Confirm same behavior as Cancel — accent stays green, no save.
7. **Record:** Cancel/Esc/X PASS / FAIL.

**Idempotent reseed UAT (D-03)**
1. Open Accent Color. In the Custom Colors row, click an empty slot (e.g., slot 8 — the first slot AFTER the 8 ACCENT_PRESETS). Use the inner dialog's "Add to Custom Colors" button (or whatever Qt's mechanism is on the platform) to set that slot to a custom color.
2. Now click on slot 0 (which should be the first ACCENT_PRESET — Blue).
3. Edit slot 0 — set it to a different color (e.g., neon green) using the same "Add to Custom Colors" mechanism. This simulates the user accidentally overwriting a preset.
4. Click **Cancel**. Dialog closes.
5. Reopen Accent Color via the hamburger menu.
6. Observe slot 0 in the Custom Colors row.
7. **Expected:** slot 0 is back to ACCENT_PRESETS[0] (Blue) — D-03 idempotent reseed runs in `__init__` and overwrites whatever the user did to slots 0..7 in the previous open. Slots 8..15 may retain user edits within the session (and die with the process).
8. **Record:** Idempotent reseed PASS / FAIL. Note: this attestation is the runtime evidence that complements T-59-A (which only proves the seed loop runs once at initial construction).
