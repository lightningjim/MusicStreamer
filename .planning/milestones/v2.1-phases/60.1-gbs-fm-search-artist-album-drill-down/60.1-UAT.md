## Deviations / Open Issues

- **Pre-existing baseline failures (out of scope for this phase):** 11 unit-test failures + 1 C-level abort across `test_media_keys_mpris2`, `test_station_list_panel`, `test_twitch_auth`, `test_edit_station_dialog`, `test_import_dialog_qt`. All reproducible on the pre-phase HEAD (`c3d9164`); not introduced by Phase 60.1. Tracked separately for a future cleanup phase.
- **D-04 multi-word triage outcome (per Plan 02 SUMMARY):** Hypothesis 1 (server omits panels for multi-word queries) was REFUTED for `foo fighters` via fixture inspection. UAT confirms it's NOT a blanket multi-word issue — `foo fighters` and `pearl jam` both yield artist/album panels. **However, the originally reported Issue A reproduces for the specific query `bad religion`:** zero artist links, zero album links, only song results. Bad Religion is unambiguously a band with multiple albums catalogued on gbs.fm — the absence of panels for this specific query is a real defect (server-side response shape OR client-side parser edge case that the foo+fighters fixture didn't surface). **Phase 60.1.1 must capture `tests/fixtures/gbs/search_bad_religion_p1.html` and triage from the captured HTML** before deciding whether the fix is parser-side or "document server-side limitation".
- **SC1 NEW DEFECT — artist drill-down renders songs as undifferentiated flat list (no album grouping):** Per UAT note, when an artist drill-down successfully loads (e.g., a working multi-word artist), all songs are shown in a single flat table without any visual grouping by album. The artist-page HTML structure interleaves `<tr class="albumTitle">` separator rows between each album's track list (see `tests/fixtures/gbs/artist_4803.html` — Testament's catalog has multiple albums each preceded by an albumTitle row). Plan 02's `_ArtistPageParser` deliberately SKIPS those separator rows and emits flat song dicts keyed only on `artist=<page-title>`. The dialog renders that flat list directly with no album column, no section headers, no grouping. **This wasn't a Plan 60.1 contract violation** (Plan 02 explicitly chose flat output) — it's a UX regression revealed by live UAT. Phase 60.1.1 should either:
  - Add an `album` field to each song row in `_ArtistPageParser` (carry the most-recent albumTitle text as state across handle_starttag/handle_endtag), then render an Album column in the artist drill-down view, OR
  - Emit album section headers in the table itself (visual dividers between albums in the artist drill view).
- **SC4 keyword-filter doc oversight:** Plan 04 specifies `drill_down or back_button or pre_drill or click_drills_down` for the dialog test count, which catches 6/7 (misses `test_new_search_while_drilling_abandons_snapshot` due to `drilling` vs `drill_down`). Broadening to `drill or back_button or pre_drill` returns 7/7 — both confirmations recorded above.
- **Auth-expired path passed via login-gate (different copy than planned toast):** UAT confirms the auth-expired path works, but instead of the toast string `"GBS.FM session expired — reconnect via Accounts"`, the dialog re-locks to its login-gate state with the existing copy `"Log in via -> GBS.FM to search and submit songs"`. Tester explicitly notes this UX is preferred ("honestly better than the toast itself"). Disposition: keep the current login-gate behavior; do NOT switch to a toast. Phase 60.1.1 may want to confirm whether `_GbsArtistWorker` / `_GbsAlbumWorker` `error('auth_expired')` signal handlers actually emit the toast (they may be unreached because the login gate triggers earlier) — purely informational.
