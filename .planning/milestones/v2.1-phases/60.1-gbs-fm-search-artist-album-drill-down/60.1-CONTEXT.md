<domain>
## Phase Boundary

Replace Phase 60-11's Shape 4 free-text-search fallback (D-11a, now superseded) with **real drill-down navigation** into `/artist/<id>` and `/album/<id>` pages, AND fix the multi-word query bug (Round-2 UAT Issue A) where artist/album results are dropped when the search input contains spaces.

**In scope:**
- New `_AlbumPageParser` in `musicstreamer/gbs_api.py` (album drill-down)
- Extend `_ArtistPageParser` work that the roadmap signals (artist drill-down)
- Fix multi-word query handling in `_SongRowParser` / `_ArtistAlbumParser` (search response)
- New per-drill workers in `musicstreamer/ui_qt/gbs_search_dialog.py` (`_GbsArtistWorker`, `_GbsAlbumWorker`)
- Inline drill-down render with Back-button breadcrumb pattern in `GBSSearchDialog`
- Test contract: delete the two Shape-4-locking tests; replace with multi-word + drill-down tests
- New multi-word search fixture (TDD): `tests/fixtures/gbs/search_multiword_p1.html` (or similar)

**Out of scope:**
- Album drill-down via external browser (rejected in favor of in-dialog parsing)
- Multi-word fix for `_ArtistPageParser` / `_AlbumPageParser` (drill-down uses href, not text â€” multi-word names are moot for the click path; only the search-response parsers need the fix)
- Refactoring the existing single-word click path that already works in Phase 60-11
- Any new UX patterns beyond inline drill-down + Back button (no sub-dialogs, no accordion, no external browser)

<decisions>
## Implementation Decisions

### Album parser strategy
- **D-01:** Ship `_AlbumPageParser` as part of Phase 60.1 â€” closes SC #2 fully in-phase. Best-effort acceptable given GBS.FM's HTML stability profile. Rejected alternatives: defer + disable click; defer + external browser; keep Shape 4 fallback.
- **D-02:** `_AlbumPageParser` locks onto the **first `<table width="620">` inside `<div class="playlist">`**. Belt-and-suspenders combination of container class (consistent across search/artist/album surfaces) AND width attribute. Rejected: pure positional (Nth table); content-based heuristic (`/song/` href detection in rows); reusing `_SongRowParser` with a loosened gate (risks song-search regression).

### Multi-word query bug (Issue A)
- **D-03:** TDD approach â€” Plan 1 captures a multi-word search response (e.g., `query=foo+fighters` or `query=pearl+jam`) into `tests/fixtures/gbs/search_multiword_p1.html` (or similar â€” exact filename at planner discretion), writes a failing parser test, THEN fixes. Rejected: live-only diagnosis (no permanent regression guard); speculative encoding patch (high regression risk); deferring to a separate phase (would block drill-down for multi-word artists).
- **D-04:** Fix scope is **search response parsers only** â€” `_SongRowParser` AND `_ArtistAlbumParser`. The drill-down flow uses the link's `href` (NOT its text), so multi-word artist/album NAMES become moot for the click path. Only the search-page extraction needs to survive multi-word queries. Rejected: also fixing artist-page / album-page parsers for multi-word inputs (out of scope per D-04's logic).

### Drill-down UX
- **D-05:** Drill-down **replaces the song-results table inline**, with a "Back to search" button that restores the prior search state. Browser-style breadcrumb pattern. Search input + Artist/Album panels stay in place visually (or are restored on Back). Rejected: hide-search-show-drilled (more disruptive); separate sub-dialog (user has historically avoided dialog stacking); inline accordion under entry (incompatible with `QListWidget`).
- **D-06:** **New per-drill workers** in `gbs_search_dialog.py`: `_GbsArtistWorker` and `_GbsAlbumWorker`, each with its own `finished` signal and dedicated slot in `GBSSearchDialog`. Type-safe; clear separation of search vs drill-down state machines. Rejected: extending `_GbsSearchWorker.finished` with a mode-flag str (couples three response shapes into one signal); dialog-side state with simple workers (mode tracking via instance var is implicit and bug-prone); separate top-level `gbs_api` functions (covered by D-06 worker structure but out-of-scope to add new top-level surface beyond what the workers need).

### Test contract evolution
- **D-07:** **Delete** `test_artist_click_kicks_free_text_search` and `test_album_click_kicks_free_text_search` in the same commit that lands the drill-down implementation. D-11a Shape 4 is gone â€” the tests are dead weight. Git blame preserves the old contract for anyone who needs the history. Replacement tests (`test_artist_click_drills_down_into_artist_page`, `test_album_click_drills_down_into_album_page`, plus multi-word search-response tests per D-03) cover the new behavior. Rejected: rename + rewrite in place (cosmetic; doesn't actually preserve more history than a delete + new test); `@pytest.mark.skip(reason="superseded")` (pollutes the suite list with permanently-dead entries); keep + add (doubles surface for the same behavior).

### Claude's Discretion
- Exact filename of the multi-word fixture (e.g., `search_multiword_p1.html` vs `search_foo_fighters_p1.html`) â€” planner picks based on what's typical for the test suite.
- Whether `_GbsArtistWorker` and `_GbsAlbumWorker` share a common base class or duplicate the Qt worker boilerplate â€” planner decides based on diff against `_GbsSearchWorker`'s structure.
- Whether the Back button is a separate `QPushButton` next to pagination or replaces the pagination row entirely while in drill-down mode â€” UX detail; either satisfies D-05.
- Whether to fold the artist-page parsing logic ("locks onto `<table class="artist">`") into a single `_ArtistPageParser` class or split into a class hierarchy mirroring `_SongRowParser` â€” planner decides.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`_SongRowParser` HTMLParser pattern** (`gbs_api.py:305-`): Class-based HTMLParser with start-tag gate, in-row state machine, and `self.results` accumulator. New `_ArtistPageParser` and `_AlbumPageParser` mirror this shape with their own gates (`<table class="artist">` and "first `<table width="620">` inside `<div class="playlist">`" respectively).
- **`_ArtistAlbumParser`** (`gbs_api.py:496-`): Already handles the search-response Artist/Album blocks. Phase 60.1 extends test coverage but doesn't redesign the class.
- **`_GbsSearchWorker`** (`gbs_search_dialog.py:53-78`): QThread + finished signal pattern. New `_GbsArtistWorker` and `_GbsAlbumWorker` (D-06) clone this shape with their own URL targets and signal payloads.
- **`_open_with_cookies` / `load_auth_context`** (`gbs_api.py:92, 219-`): Authentication flow. Reused unchanged for `/artist/<id>` and `/album/<id>` fetches â€” these pages require the same cookie context as `/search`.

### Established Patterns
- **Plain-text invariant (T-40-04):** All `QListWidget.addItem()` calls use plain text by default; `href` values stored as `Qt.ItemDataRole.UserRole`. The drill-down click handler reads the stored href, not the displayed text. Already enforced in 60-11; preserved by Phase 60.1.
- **Worker â†’ finished signal â†’ slot routing:** Each new worker owns a typed signal that the dialog connects to a dedicated slot. No multi-mode signal (D-06 rejection of "Signal(list, int, int, str)" pattern).
- **Fixture-driven parser tests:** Every parser has a captured-HTML fixture; tests parse the fixture and assert on extracted shape. Phase 60.1 follows this for the new multi-word search fixture (D-03) and the artist/album page parsers.
- **D-08e ignore-block docstring** (`_SongRowParser` lines 293-295): The "ignores `<p class="artists">` / `<p class="albums">` blocks" docstring is now factually wrong post-60-11. Plan should update this comment when touching the class for D-04.

### Integration Points
- **Drill-down click â†’ URL fetch â†’ parse â†’ re-render**: New flow connects `_artist_list.itemClicked` â†’ instantiate `_GbsArtistWorker(href)` â†’ `worker.finished.connect(_on_artist_drilled)` â†’ render catalog into the existing results table + show Back button.
- **Back button â†’ restore search state**: Need to preserve last-known `(query, page, results, artist_links, album_links)` tuple before drill-down begins; Back restores it via the existing render path. Implementation lives in the dialog state machine; tests verify both the down-and-back round-trip preserves search state.

<deferred>
## Deferred Ideas

- **Multi-word fix for artist-page / album-page parsers**: Out of scope per D-04. If a future surface uses the artist/album text (not href), this would matter. Not Phase 60.1's problem.
- **Pagination within an artist's catalog**: `/artist/<id>` may return multiple pages of songs for prolific artists. Phase 60.1 ships single-page drill-down (renders whatever the first page returns); pagination within drill-down view is a future enhancement.
- **Album drill-down via external browser (`QDesktopServices.openUrl`)**: Rejected for in-dialog parsing per D-01, but if `_AlbumPageParser` proves brittle in production, this is the natural fallback for a Phase 60.2.
- **Sub-dialog navigation pattern**: Rejected for inline replacement per D-05. If multi-level drill-down ever becomes a need (artist â†’ album â†’ song), the breadcrumb-stack approach scales further than sub-dialogs would.

</deferred>
