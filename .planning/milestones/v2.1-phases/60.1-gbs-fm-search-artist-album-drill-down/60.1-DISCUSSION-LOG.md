## Album parser strategy

### Round 1: How to handle the album drill-down surface?

| Option | Description | Selected |
|--------|-------------|----------|
| Ship `_AlbumPageParser` now | Best-effort parser using width=620 + structural heuristics. Brittle to GBS.FM HTML changes since there's no deterministic class gate. Closes SC #2 fully in this phase. | тЬУ |
| Defer album parser, disable click | Ship `_ArtistPageParser` only. Album entries become non-clickable. SC #2 partially deferred. Cleaner scope. | |
| Defer album parser, click opens browser | Album click calls `QDesktopServices.openUrl('https://gbs.fm/album/<id>')`. SC #2 met by side-stepping. Adds external-navigation pattern. | |
| Keep Shape 4 fallback for albums | Album click stays as today's free-text search of the album name. Lowest churn but SC #2 not really met. | |

**User's choice:** Ship `_AlbumPageParser` now.

### Round 2: Which structural cue should `_AlbumPageParser` use?

| Option | Description | Selected |
|--------|-------------|----------|
| Container div + width attr | Match: first `<table>` inside `<div class="playlist">` with `width="620"`. Belt-and-suspenders combination. | тЬУ |
| Table with `/song/` hrefs | Find first `<table>` whose `<tr>`s contain at least one `<a href="/song/<id>">`. Self-validating against content shape. | |
| Pure positional | Take the Nth `<table>` on the page. Simplest code, breaks if layout adds/removes a table elsewhere. | |
| Reuse `_SongRowParser` | Loosen `_SongRowParser`'s gate to also match unclassed `<table width="620">`. Risk: breaks song-search path. | |

**User's choice:** Container div + width attr.

## Multi-word query bug (Issue A)

### Round 1: Approach to the multi-word query bug?

| Option | Description | Selected |
|--------|-------------|----------|
| Capture fixture first (TDD) | Plan 1 captures multi-word search response, writes failing test, THEN fixes. Most rigorous. | тЬУ |
| Diagnose inline against live GBS.FM first | Read `gbs_api.search()` URL construction by eyeball + run live request. Faster start; risk of fix-without-fixture. | |

| Defer Issue A to a separate phase | Phase 60.1 focuses ONLY on drill-down. Risk: drill-down on multi-word artist still won't work. | |

**User's choice:** Capture fixture first (TDD).

### Round 2: Scope of the multi-word fix?

| Option | Description | Selected |
|--------|-------------|----------|
| Search response only | Verify `_SongRowParser` AND `_ArtistAlbumParser` correctly extract entries from multi-word-query response. Drill-down (using href, not text) makes multi-word names moot for the click path. | тЬУ |
| Search + artist-page response | Also capture `/artist/<id>` for an artist whose name has spaces. Verify `_ArtistPageParser` handles title/header text correctly. | |
| All three: search + artist + album | Capture multi-word fixtures for all three parsers and add tests for each. Most thorough; biggest fixture-capture cost. | |
| Defer scope until fixture is captured | Plan 1 just captures one multi-word search fixture; failures dictate scope. Pragmatic; minimum upfront commitment. | |

## Drill-down UX in the dialog

### Round 1: Where should the artist's/album's catalog render?

| Option | Description | Selected |
|--------|-------------|----------|
| Replace results table inline + Back button | The song-results `QTableView` is repurposed for the catalog. New 'Back to search' button restores prior state. Browser-style breadcrumb. Lowest layout churn. | тЬУ |
| Hide search panels, show drilled view | Hide Artist/Album panels and search input; replace with 'Viewing: ...' header + catalog table + Back button. Cleaner focus. | |
| Open separate sub-dialog | Spawn new modal showing the catalog. Easiest to implement (no state-restore logic) but adds dialog stacking. | |
| Inline expand below the entry | Click expands inline accordion under the clicked entry. Most compact; complex layout management; QListWidget doesn't natively support row expansion. | |

**User's choice:** Replace results table inline + Back button.

### Round 2: How should the dialog track drill-down mode?

| Option | Description | Selected |
|--------|-------------|----------|
| New worker + signal per drill | Add `_GbsArtistWorker` and `_GbsAlbumWorker` (mirroring `_GbsSearchWorker`) with their own finished signals. Type-safe; clear separation. | тЬУ |
| Reuse search worker, add mode flag | Extend `_GbsSearchWorker.finished` to `Signal(list, int, int, str)` where str is mode. Less code; couples three responses into one signal shape. | |
| Dialog-side state, worker stays simple | `_GbsSearchWorker` runs whatever URL is given; dialog tracks mode via instance var. Simplest worker. | |
| Dispatch via separate API surface | `gbs_api` gets `fetch_artist_page()` / `fetch_album_page()` as new top-level functions. Each gets its own worker class. Maximum separation; biggest test surface. | |

**User's choice:** New worker + signal per drill.
