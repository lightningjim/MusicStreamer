# Phase 60.1: GBS.FM Search Artist/Album Drill-Down — Research

**Researched:** 2026-05-06
**Domain:** PySide6 dialog state machine + Python `html.parser.HTMLParser` subclasses against captured GBS.FM HTML fixtures
**Confidence:** HIGH (code surfaces verified line-by-line; fixtures exercised via live parser; all decisions already locked in CONTEXT.md)

## Summary

Phase 60.1 evolves three existing surfaces in the codebase:

1. **`musicstreamer/gbs_api.py`** — adds two new HTMLParser subclasses (`_ArtistPageParser`, `_AlbumPageParser`) plus two new fetch functions (`fetch_artist_songs`, `fetch_album_songs`) for `/artist/<id>` and `/album/<id>` drill-downs; also patches `_SongRowParser` and `_ArtistAlbumParser` to defend against the multi-word search-response failure mode (Issue A).
2. **`musicstreamer/ui_qt/gbs_search_dialog.py`** — replaces the Phase 60-11 D-11a Shape 4 free-text fallback with **inline drill-down navigation** (Back-button breadcrumb pattern per UI-SPEC), adds two new per-drill workers (`_GbsArtistWorker`, `_GbsAlbumWorker`) with their own typed `finished` signals (D-06), and wires a single `self._pre_drill_state` snapshot/restore on the existing `_artist_list` / `_album_list` `itemActivated` (or `itemClicked`) signals.
3. **Tests** — deletes `test_artist_click_kicks_free_text_search` and `test_album_click_kicks_free_text_search` (D-07) and replaces them with multi-word-fixture-driven parser tests + drill-down round-trip tests, all following the existing `qtbot` + `monkeypatch.setattr(gbs_api, "_open_with_cookies", ...)` pattern.

The phase is fixture-driven and backed by an already-captured pair of HTML files (`tests/fixtures/gbs/artist_4803.html`, `tests/fixtures/gbs/album_1488.html`) plus the search baseline (`tests/fixtures/gbs/search_test_p1.html`). The multi-word search-response fixture **MUST be captured by Plan 1** (TDD-RED step per D-03) before any parser fix is attempted; without that fixture the multi-word root cause is unprovable.

The artist page (`/artist/4803`) and album page (`/album/1488`) have **different table shapes** — confirmed by direct grep of the captured fixtures. The artist page uses `<table width="620" class="artist">` with a 3-column body (Title / Length / Add) and `<tr class="albumTitle">` separator rows that group songs by album. The album page uses an UNCLASSED `<table width="620">` inside `<div class="playlist">` with a 6-column body (Artist / Title / Format / Bitrate / Length / Add). These are NOT compatible with `_SongRowParser`'s `<table class="songs">` gate, which is why Phase 60-11 incorrectly locked Shape 4. They also differ from EACH OTHER, so the two new parsers are NOT a single shared class — they are separate state machines.

**Primary recommendation:** Plan in 4 tasks (Wave 0 multi-word fixture capture + RED tests; Wave 1 parser GREEN: both `_ArtistPageParser` and `_AlbumPageParser` plus the multi-word fix; Wave 2 dialog drill-down GREEN: workers, snapshot/restore, Back button per UI-SPEC; Wave 3 integration tests + D-07 deletes). Use TDD throughout (D-03 mandates RED-first for the multi-word fix; the diagnostic value of the captured fixture is load-bearing).

### Pattern 1: HTMLParser subclass with start-tag gate + in-row state machine

Both `_ArtistPageParser` and `_AlbumPageParser` follow the **exact same shape** as `_SongRowParser` (`gbs_api.py:305-381`). The state-machine elements:

```python
# Source: musicstreamer/gbs_api.py:305-381 (existing _SongRowParser)
class _SongRowParser(HTMLParser):
    """Anchors on <table class='songs'>; collects <tr> rows."""
    _SONG_RE = re.compile(r"^/song/(\d+)$")
    _ADD_RE = re.compile(r"^/add/(\d+)$")
    _ARTIST_RE = re.compile(r"^/artist/(\d+)$")

    def __init__(self):
        super().__init__()
        self._in_songs_table = False     # <-- the gate flag
        self._row: Optional[dict] = None  # <-- per-row scratchpad
        self._capture: Optional[str] = None  # <-- which field is currently being collected
        self._td_text = ""               # <-- text accumulator for the current <td>
        self._tds_in_row = 0             # <-- positional column counter
        self.results: list = []          # <-- final output

    def handle_starttag(self, tag, attrs):
        a = dict(attrs)
        if tag == "table" and "songs" in (a.get("class") or ""):
            self._in_songs_table = True
            return
        if not self._in_songs_table:
            return
        if tag == "tr":
            self._row = {}
            self._tds_in_row = 0
            return
        if tag == "td":
            self._td_text = ""
            self._tds_in_row += 1
            return
        if tag == "a" and self._row is not None:
            href = a.get("href", "")
            m_song = self._SONG_RE.match(href)
            m_add = self._ADD_RE.match(href)
            m_artist = self._ARTIST_RE.match(href)
            if m_song:
                self._row["songid"] = int(m_song.group(1))
                self._capture = "title"
            elif m_add:
                self._row["add_url"] = href
            elif m_artist:
                self._capture = "artist"
    # ... handle_endtag commits row on </tr> if it has songid+artist+title
    # ... handle_data appends to self._row[self._capture] AND self._td_text
```

`_ArtistPageParser` differs in two ways: the gate is `<table class="artist">` (matched on the `class` attribute, not the `width`), and the artist name is fixed ONCE from the page title `<tr><th class='album' colspan='3'>Testament</th></tr>` rather than per-row. Album-separator rows (`<tr class="albumTitle">`) have a 1-cell `<td colspan="3">` and must be SKIPPED — they don't have a songid. Two-`<th>` header row also skipped.

`_AlbumPageParser` differs from both: the gate per D-02 is **first `<table width="620">` inside `<div class="playlist">`**. Belt-and-suspenders means the parser must track `_in_playlist_div` (set on `<div class="playlist">`, cleared on `</div>` if depth-tracked) AND seek the first `<table>` whose `width="620"` AND `class != "artist"` (artist pages also have width=620 but with class="artist" — explicit exclusion is safer than just-missing-class). Row shape is 6 columns and the artist name comes from `<a href="/artist/Y">` in column 0.

```python
    finished = Signal(list, int, int)              # (results, page, total_pages)
    metadata_ready = Signal(list, list)            # (artist_links, album_links) — 60-11
    error = Signal(str)                            # 'auth_expired' sentinel OR raw msg

    def __init__(self, query: str, page: int, cookies, parent=None):
        super().__init__(parent)
        self._query = query
        self._page = page
        self._cookies = cookies

    def run(self):
        try:
            out = gbs_api.search(self._query, self._page, self._cookies)
            # ORDERING INVARIANT: finished BEFORE metadata_ready (60-11)
            self.finished.emit(
                list(out.get("results", [])),
                int(out.get("page", self._page)),
                int(out.get("total_pages", self._page)),
            )
            self.metadata_ready.emit(
                list(out.get("artist_links", [])),
                list(out.get("album_links", [])),
            )
        except Exception as exc:
            if isinstance(exc, gbs_api.GbsAuthExpiredError):
                self.error.emit("auth_expired")
            else:
                self.error.emit(str(exc))
```

### Pattern 3: Snapshot-and-restore for Back button

Per UI-SPEC.md Delta 5 + judgment-call #8: a single `self._pre_drill_state: Optional[dict]` captured exactly once at click time.

```python
# Implementation pattern (NEW for Phase 60.1)
def _snapshot_pre_drill_state(self) -> dict:
    """Capture the dialog state to restore on Back. Called once per drill-down session."""
    return {
        "results": list(self._results),
        "current_page": self._current_page,
        "total_pages": self._total_pages,
        "current_query": self._current_query,
        "search_version": self._search_version,

            {"text": self._artist_list.item(i).text(),
             "url": self._artist_list.item(i).data(Qt.ItemDataRole.UserRole) or ""}
            for i in range(self._artist_list.count())
        ],

            {"text": self._album_list.item(i).text(),
             "url": self._album_list.item(i).data(Qt.ItemDataRole.UserRole) or ""}
            for i in range(self._album_list.count())
        ],
        "artist_panel_visible": not self._artist_list.isHidden(),
        "album_panel_visible": not self._album_list.isHidden(),
        "prev_enabled": self._prev_btn.isEnabled(),
        "next_enabled": self._next_btn.isEnabled(),
    }

def _restore_pre_drill_state(self, state: dict) -> None:
    """Inverse of _snapshot_pre_drill_state. Called from _on_back_clicked."""
    self._results = list(state["results"])
    self._current_page = state["current_page"]
    self._total_pages = state["total_pages"]
    self._current_query = state["current_query"]
    # search_version intentionally NOT restored — drill-down doesn't bump it,
    # so the live counter is already correct.
    self._render_results()  # repopulates _results_table from self._results
    # Re-populate panels via the same path as _on_metadata_ready
    self._on_metadata_ready(state["artist_links"], state["album_links"])
    # Page-row state
    self._page_label.setText(f"Page {self._current_page} of {self._total_pages}")
    self._prev_btn.setEnabled(state["prev_enabled"])
    self._next_btn.setEnabled(state["next_enabled"])
    # Top-row + bottom-row swap back
    self._page_label.setVisible(True)
    self._breadcrumb_label.setVisible(False)
    self._prev_btn.setVisible(True)
    self._next_btn.setVisible(True)
    self._back_btn.setVisible(False)
    self._hide_inline_error()
```

### Anti-Patterns to Avoid

- **DO NOT** snapshot pre-drill state on every artist/album click. UI-SPEC FLAG-03 calls this out: a click WHILE in drill-down must NOT re-snapshot (would overwrite the original search state with the partial drilled state). Guard pattern: `if self._pre_drill_state is None: self._pre_drill_state = self._snapshot_pre_drill_state()`.
- **DO NOT** introduce a new `metadata_ready`-style signal on the per-drill workers. Each worker emits ONE typed `finished(list)` signal — that's it. The drill-down response has no separate "artist/album link" payload (artist pages and album pages don't have `<p class="artists">` blocks, per the captured fixtures — see Common Pitfalls #4).
- **DO NOT** re-introduce the Phase 60-11 ORDERING INVARIANT comment block on the new workers. That invariant only applies to `_GbsSearchWorker` because `_clear_table` hides the panels and `_on_metadata_ready` re-shows them. The drill-down render path explicitly RE-SHOWS the panels from snapshot per Delta 3 — a different invariant. Document the new invariant in the dialog slot, not the worker.
- **DO NOT** call `setStyleSheet` on `_back_btn` or `_breadcrumb_label`. UI-SPEC's "no new colors" / "no new typography" constraints are binding (FLAG-01).
- **DO NOT** snapshot state if `_pre_drill_state is not None` (already drilling). The user clicked another artist/album entry — replace the drilled view, but keep the original snapshot.
- **DO NOT** use `lambda` in `.connect()` calls. Project convention QA-05 (Pitfall 10) — the existing test `test_no_self_capturing_lambdas_in_dialog` (lines 181-187) greps for `lambda` in connect lines. Use bound methods or closure factories like `_make_submit_slot`.
- **DO NOT** use HTML / RichText for any `QListWidget.addItem` or `QStandardItem` — T-40-04 PlainText invariant. Existing `_on_metadata_ready` complies; new code must too.
- **DO NOT** rely on `_GbsSearchWorker.metadata_ready` being received before/after a drill-down click. The dialog state machine must be robust to: user types query → search returns → user clicks artist BEFORE metadata_ready fires (rare, but Qt's signal queue isn't deterministic across cross-thread emits).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTML attribute parsing | regex on the raw string | `HTMLParser.handle_starttag(tag, attrs)` (already in use) | regex misses entity-encoded attribute values; the existing parsers consume `attrs` as a list of `(name, value)` tuples that are already entity-decoded by stdlib. |
| URL extraction from links | regex on the full anchor block | `dict(attrs).get("href", "")` inside `handle_starttag(tag="a", ...)` | Same parser already extracts `/song/X` and `/artist/Y` patterns via `re.match` on the href value alone — the host parser is HTMLParser, the regex narrows to the URL slot. |
| Cookie-bearing GET | new `urlopen` call | `_open_with_cookies(url, cookies, timeout=_TIMEOUT_READ)` (`gbs_api.py:146-160`) | Already handles 302→/accounts/login/ and raises `GbsAuthExpiredError`. Reuse for all `/artist/<id>` and `/album/<id>` fetches. |
| Cookies path discovery | hand-coded XDG path | `paths.gbs_cookies_path()` and `gbs_api.load_auth_context()` | Project's standard cookie-loading path; auto-handles corruption check via `cookie_utils`. |
| Auth-expired error pathway | catch `urllib.error.HTTPError` | `try: ... except gbs_api.GbsAuthExpiredError` | The conversion from raw HTTPError to `GbsAuthExpiredError` lives in `_open_with_cookies` already. Workers only need to catch the typed exception. |
| Plain-text rendering for QListWidgetItem | `setHtml`, `<b>`, etc. | `QListWidgetItem(str(text))` and `setData(Qt.ItemDataRole.UserRole, url)` | T-40-04 invariant; existing `_on_metadata_ready` already does this. |
| Worker-thread cleanup | manual `quit()`/`wait()` | Set `self._search_worker = None` on `finished`/`error` slot end; assign new worker over old | The dialog already does this for `_GbsSearchWorker` and `_GbsSubmitWorker` (search worker stored on `self._search_worker`, etc.). Mirror for `self._artist_drill_worker` / `self._album_drill_worker`. |

**Key insight:** Phase 60-02's parser/worker/dialog scaffolding is fully reusable for drill-down. The only new thing is the **page-specific HTMLParser shape** (one parser per drill target type) and the **state-machine snapshot dict**. No new dependencies, no new abstractions.

## Common Pitfalls

### Pitfall 1: Multi-word query parser regression — UNCONFIRMED ROOT CAUSE

**What goes wrong:** Per the Round-2 UAT report (Issue A), clicking a multi-word artist name in the Phase 60-11 panel triggers a Shape 4 free-text search where the result page shows song results but NO Artist:/Album: panels. Tested live with the captured single-word `tests/fixtures/gbs/search_test_p1.html` baseline, the parser correctly extracts 45 artist links (including multi-word names like `"Hugh Jackman, Keala Settle, Daniel Everidge, Zendaya & The Greatest Showman Ensemble"` — verified via direct `_parse_artist_album_html` invocation 2026-05-06). So the parser handles multi-word NAMES; the bug must be at multi-word QUERY response time.

**Why it happens (HYPOTHESES, not verified):**

The multi-word fixture has not yet been captured. CONTEXT.md D-03 mandates capture-fixture-first, then test, then fix. The most likely causes (ranked by plausibility):

1. **GBS.FM omits `<p class="artists">` blocks for multi-word queries entirely.** The single-word query "test" returns artist+album panels, but a multi-word query "foo fighters" might return only the song table. This would mean the parser is correct and the user-visible feature is intentionally absent server-side. Fix: nothing to fix; document and warn user.
2. **Different leading-text node for the multi-word response.** GBS.FM may emit `<p class="artists">No artist matches for "foo fighters"</p>` or similar, in which case the current `_ArtistAlbumParser`'s "first non-empty data node decides the block category" logic (`gbs_api.py:543-550`) sees text starting with `"No"` and never sets `_current_block`. Result: every `<a>` inside the block is silently dropped (the `target = artist_links if _current_block == "artists" else album_links` line is in the `if _collect_anchor_text and ...` branch which is never reached without `_current_block` set). Fix: relax the leading-text discriminator to also accept `"No artist"`/`"No album"` markers and treat the block as empty.
3. **Different paragraph class.** GBS.FM may use `<p class="albums">` (singular section name) for the album block on multi-word queries instead of the documented-as-shared `<p class="artists">`. The current parser ONLY checks for `class="artists"`. Fix: also accept `class="albums"`.
4. **Query-encoding side effect.** Already debunked: `urllib.parse.urlencode({"query": "foo fighters", "page": 1})` yields `query=foo+fighters&page=1` (verified 2026-05-06 locally). `+` is the standard form encoding for space; GBS.FM's Django-default URL parsing decodes it to `"foo fighters"`. NOT THE BUG.
5. **Fragment discriminator confusion.** On the search-response page, `<p class="artists">` appears TWICE (Artists block + Albums block). The current parser's reset-on-`</p>` logic (`gbs_api.py:530-532`) handles that fine for single-word fixtures. For multi-word fixtures, perhaps the markup nests differently (e.g., a `<br>` or stray `</p>` closes the block early). Verifiable only against captured HTML.

**How to avoid:** Plan 1 MUST capture `tests/fixtures/gbs/search_<multi_word>_p1.html` (D-03). Then write a failing parser test that asserts a multi-word response produces non-empty `artist_links`/`album_links` (assuming the response actually has panels — if it doesn't, the test asserts the empty-list case and we accept that hypothesis 1 is the truth). Compare the captured fixture to `search_test_p1.html` byte-for-byte to identify the structural delta.

**Warning signs:** A multi-word response with `<p class="artists">` but NO matching artists has different copy than the single-word case. Diff the captured fixture against `search_test_p1.html` at the line-388 region (Artists block start) before writing the parser fix.

### Pitfall 2: Album page table-shape ambiguity (D-02 belt-and-suspenders)

**What goes wrong:** A naive "first `<table>` inside `<div class='playlist'>`" parser would also match the artist page (whose table is also the first one in the same container, but with `class="artist"`). If the same parser runs against an artist page by mistake (e.g., a router bug), the album parser would extract garbage rows.

**Why it happens:** GBS.FM reuses `<div class="playlist">` for SEARCH RESPONSES, ARTIST PAGES, AND ALBUM PAGES — confirmed by direct grep of all three fixtures (lines 374, 374, 374 respectively). Search response: `<table width="620" class="songs">`. Artist page: `<table width="620" class="artist">`. Album page: `<table width="620">` (UNCLASSED). All inside `<div class="playlist">`.

**How to avoid:** D-02's belt-and-suspenders is correct — `_AlbumPageParser` must require BOTH `width="620"` AND `class != "artist"` (and `class != "songs"`). A first-match algorithm:

```python
def handle_starttag(self, tag, attrs):
    ad = dict(attrs)
    if tag == "div" and "playlist" in (ad.get("class") or ""):
        self._in_playlist_div = True
        return
    if not self._in_playlist_div or self._table_locked:
        return
    if tag == "table" and ad.get("width") == "620":
        # D-02 belt-and-suspenders: NOT class="artist" and NOT class="songs"
        cls = ad.get("class") or ""
        if "artist" in cls or "songs" in cls:
            return  # this is the artist or songs table; not the album-page table
        # First UNCLASSED width=620 table inside playlist → lock onto it
        self._in_album_table = True
        self._table_locked = True
```

**Warning signs:** A unit test for `_AlbumPageParser` against `tests/fixtures/gbs/artist_4803.html` MUST return `[]` (empty results — the artist-class table is correctly rejected). Add this as a regression test.

### Pitfall 3: `_clear_table()` side-effect hiding panels mid-drill-down render

**What goes wrong:** `_clear_table()` (`gbs_search_dialog.py:438-452`) defensively hides `_artist_list` and `_album_list` panels. This is correct for a fresh search (the new search-response's `_on_metadata_ready` re-shows them). But during drill-down, Delta 3 of UI-SPEC.md mandates the panels REMAIN VISIBLE (so the user sees the source they drilled from). If the drill-down render slot calls `_clear_table()` first, the panels disappear.

**Why it happens:** Phase 60-11's `_clear_table()` was written assuming the only callers are `_kick_search_worker` (always followed by `_on_metadata_ready` re-show) and `_render_results` itself. The drill-down path is a NEW caller pattern.

**How to avoid:** UI-SPEC offers two resolutions; the planner must pick one explicitly:
- **Option A:** Add a `clear_panels: bool = True` kwarg to `_clear_table`. Drill-down render slot calls `_clear_table(clear_panels=False)`. Cleanest.
- **Option B:** Save `(artist_links, album_links)` from current panel contents into local variables, call `_clear_table()`, then re-populate via `_on_metadata_ready(saved_artist, saved_album)`. More verbose but no API change.

Recommendation: **Option A.** It's a single-line signature change and the kwarg name self-documents the intent. The existing `_clear_table()` body already has two `if hasattr(self, "_artist_list"):` blocks that wrap the hide; gating those on `clear_panels` is a clean diff.

**Warning signs:** A drill-down test that asserts `not dlg._artist_list.isHidden()` AFTER `_on_artist_drilled` fires. Add this as a regression test.

### Pitfall 4: Artist/album drill-down pages have NO `<p class="artists">` blocks

**What goes wrong:** A planner might assume the per-drill workers also need to populate Artist/Album panels (via a `metadata_ready`-style signal). They don't. Direct grep against the captured fixtures confirms:

```
$ grep -c '<p class="artists"' tests/fixtures/gbs/artist_4803.html
0
$ grep -c '<p class="artists"' tests/fixtures/gbs/album_1488.html

```

**Why it happens:** The Artist/Album panels are **search-response-only** UI affordances (page 1 of `/search?query=...`). Drilled pages don't show "Other artists you might like" or "Other albums" — they just show the catalog table.

**How to avoid:** `_GbsArtistWorker.finished` and `_GbsAlbumWorker.finished` emit ONE list (the drilled songs). The dialog slot does NOT touch `_artist_list` or `_album_list` content during drill-down render. The panels stay populated FROM the snapshot (Delta 3 + Pitfall 3).

**Warning signs:** Any plan task that adds a `metadata_ready` signal to `_GbsArtistWorker` or `_GbsAlbumWorker` is wrong — reject it.

### Pitfall 5: Artist-page `<tr class="albumTitle">` mid-table separator rows

**What goes wrong:** The artist page groups songs by album using `<tr class="albumTitle"><td colspan="3"><a href="/album/X">...</a></td></tr>` interspersed with the song rows. A naive parser that treats every `<tr>` inside `<table class="artist">` as a song row will produce phantom rows with songid=None / artist="(blank)" / title=(album name).

**Why it happens:** Artist-page table shape from `tests/fixtures/gbs/artist_4803.html:384-486`:
```html
<tr class="albumTitle">
  <td class="album" colspan=3> <a href="/album/69560"> Brotherhood of the Snake</a> </td>
</tr>
<tr class="odd">
  <td><a href="/song/563811">Brotherhood of the Snake</a></td>
  <td>4:14</td>
  <td><a class="boxed add" href="/add/563811">Add!</a></td>
</tr>
```

**How to avoid:** `_ArtistPageParser.handle_starttag` for `<tr>` MUST inspect the row's `class` attribute and SKIP rows where `"albumTitle"` is present. Mirror the `_QueueRowParser` pattern (`gbs_api.py:417`) that skips `class="playing"` and `class="history"`. Also skip the header row (`<tr><th>Title</th>...</tr>`) — recognizable because the row's first `<td>` is actually a `<th>`, OR by tracking a flag set when `<th>` is seen instead of `<td>`. Simplest: gate on `<td>` presence within the row's TR; if NO `<td>` is seen before `</tr>`, drop the row.

The page-title row `<tr><th class='album' colspan='3'>Testament</th></tr>` is ALSO `<th>`-only — same skip rule covers both.

**Bonus extraction:** On `<tr class="albumTitle">`, the `<a href="/album/X">` IS a real album link. A future enhancement (out of scope for Phase 60.1) could expose those as additional drill-down targets. For now, just skip cleanly.

**Warning signs:** A test that drills `/artist/4803` and asserts `len(results) == NN` where NN is the song count, NOT the song count + album-separator count. Read the fixture to count songs (Testament has many albums; the test should count `<a href="/song/`). A simpler assertion: every result has `songid` as a non-None int.

### Pitfall 6: Album-page artist comes from a per-row `<a href="/artist/Y">` (not the page header)

### Pitfall 9: `_back_btn` race during in-flight fetch

**What goes wrong:** User clicks Artist A → fetch dispatches → user clicks Back BEFORE `finished` fires → snapshot is restored → fetch THEN completes and tries to render drilled view over the restored search.

**Why it happens:** Qt signals are async; the worker thread keeps running even if the UI navigates away. The completed `finished` signal will still fire on the dialog's slot.

**How to avoid:** UI-SPEC Interaction States line 204 mandates `_back_btn` is DISABLED during fetch. Two approaches:
- **Approach A:** Disable `_back_btn` in `_on_artist_link_activated` immediately after dispatching the worker. Re-enable in `_on_artist_drilled`.
- **Approach B:** Track an "active drill worker" reference; the slot checks `if worker is self._artist_drill_worker` (similar to the `search_version` token pattern in `_on_submit_finished`). If stale, drop the rendered result.

Recommendation: **both**. Approach A prevents the user from triggering the race in the first place; Approach B defends against the case where the user hits the Close button (`x` corner of the dialog) mid-fetch and then re-opens — `self._artist_drill_worker = None` after every settled callback ensures stale callbacks no-op.

**Warning signs:** A test `test_drill_down_disables_back_button_during_fetch` that constructs the dialog, dispatches a drill click, asserts `_back_btn.isEnabled() == False`, then manually fires the worker's `finished` signal and asserts `_back_btn.isEnabled() == True`.

### Pitfall 10: New search WHILE in drill-down doesn't restore snapshot

**What goes wrong:** User drills into Artist A → types in search box → presses Enter. UI-SPEC line 228 mandates this exits drill-down by abandoning the snapshot (snap = None, breadcrumb hidden, page label shown, panels reset by the new search). If the implementation accidentally restores the snapshot before kicking the new search, the user sees their old search results flash before the new search completes.

**How to avoid:** `_start_search` MUST clear `self._pre_drill_state = None` and reset all drill-down chrome (`_back_btn.setVisible(False)`, `_breadcrumb_label.setVisible(False)`, `_page_label.setVisible(True)`, `_prev_btn.setVisible(True)`, `_next_btn.setVisible(True)`) at the TOP of the method, before `_kick_search_worker` is called.

**Warning signs:** A test `test_new_search_while_drilling_abandons_snapshot` that drills, then calls `_start_search()`, then asserts `dlg._pre_drill_state is None` and `dlg._back_btn.isHidden()`.

### Pitfall 11: HTMLParser `convert_charrefs` is True by default — entity refs in `handle_data`, NOT `handle_entityref`

**What goes wrong:** A planner might add a `handle_entityref(name)` method to one of the new parsers thinking they need to manually decode `&amp;` etc. They don't.

**Why it happens:** Python 3.5+ default for `HTMLParser` is `convert_charrefs=True`, which auto-decodes entity refs and char refs into plain text inside `handle_data`. Verified locally 2026-05-06: `from html.parser import HTMLParser; HTMLParser().convert_charrefs == True`. The existing parsers in the file rely on this — `&amp;` → `&` in `handle_data(data)` calls.

**How to avoid:** No `handle_entityref` or `handle_charref` methods. The new parsers stay simple.

**Warning signs:** Any reviewer who flags missing `handle_entityref` should be redirected to `convert_charrefs=True` default. Project Pattern: existing `_SongRowParser`, `_ArtistAlbumParser`, `_QueueRowParser` all rely on this and have been verified working.

```python
# Source: musicstreamer/ui_qt/gbs_search_dialog.py:107-136 (_GbsSubmitWorker)
class _GbsSubmitWorker(QThread):
    """Phase 60 D-08 / GBS-01e: gbs_api.submit on a worker thread."""
    finished = Signal(str, int)          # (message_text, row_idx)
    error = Signal(str, int)             # (msg, row_idx)

    def __init__(self, songid: int, cookies, row_idx: int,
                 search_version: int, parent=None):
        super().__init__(parent)
        self._songid = songid
        self._cookies = cookies
        self._row_idx = row_idx
        self.search_version = search_version

    def run(self):
        try:
            msg = gbs_api.submit(self._songid, self._cookies)
            self.finished.emit(msg, self._row_idx)
        except Exception as exc:
            if isinstance(exc, gbs_api.GbsAuthExpiredError):
                self.error.emit("auth_expired", self._row_idx)
            else:
                self.error.emit(str(exc), self._row_idx)
```

### Known Threat Patterns for {Python + PySide6 + GBS.FM HTML scraping}

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| URL injection via crafted Artist/Album anchor href (e.g., `href="javascript:..."` or `href="//evil.example/..."`) | Tampering / Spoofing | `_on_artist_link_activated` / `_on_album_link_activated` MUST regex-match `^/artist/(\d+)$` or `^/album/(\d+)$` and reject hrefs that don't match. Otherwise the dialog could be tricked into fetching `https://evil.example/...` (relative-URL resolution by `_open_with_cookies` would fail benignly since `f"{GBS_BASE}/artist/{artist_id}"` requires an int, but the regex defense is still required). Phase 60-11 Shape 4 fallback used `item.text()` to dodge this risk; the drill-down path moves the risk back into href territory. |
| HTML injection in artist/album/song names rendered in the dialog | Information Disclosure | T-40-04 PlainText invariant. `QListWidgetItem(str(text))`, `QStandardItem(str(text))`, `QLabel.setText(str(text))` are all PlainText by default. Existing tests grep for `setHtml` / `Qt.RichText` to enforce. New `_breadcrumb_label.setText(f"Viewing artist: {name}")` MUST set `_breadcrumb_label.setTextFormat(Qt.TextFormat.PlainText)` explicitly (UI-SPEC Typography section). |
| DoS via pathological `/artist/<id>` page (10000+ rows) | DoS | Out of scope — single-page drill-down per Deferred Ideas. If a future page response exceeds plausible limits, the parser still completes; the dialog renders with QTableView's virtual scroll. No mitigation needed for this phase. |
| Auth-expired during drill-down fetch | Spoofing / Information Disclosure | Workers catch `GbsAuthExpiredError` and emit `error("auth_expired")`; dialog slot toasts and re-locks the login gate. Pattern reused from `_GbsSearchWorker.error` and `_on_search_error` lines 348-350. |
| Captured fixture leaks real session cookies | Information Disclosure | Phase 60-11 Task 0's pattern: cookie-expiry hard date guard PLUS `grep -E 'csrftoken=[A-Za-z0-9]{8,}' tests/fixtures/gbs/<file>.html` post-capture verifying 0 matches. The `curl -b "$COOKIES"` flag sends cookies in the request header but does NOT echo them into response body. New fixtures must be sanity-checked the same way before committing. |
| Race between Back-button click and worker finished signal | Tampering (state) | Pitfall 9 mitigation: disable `_back_btn` during fetch + sender-token check in worker callback (mirror the `search_version` pattern in `_on_submit_finished`). |
