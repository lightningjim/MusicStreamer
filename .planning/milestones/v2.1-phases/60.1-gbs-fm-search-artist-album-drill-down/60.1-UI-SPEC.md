# Phase 60.1 — UI Design Contract

> Visual and interaction contract for the GBS.FM Search dialog drill-down surface. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Stack note:** PySide6 (Qt 6.x) widget toolkit. No QML, no shadcn, no project-wide design system. Sole shared visual token is `musicstreamer.ui_qt._theme.ERROR_COLOR_HEX = "#c0392b"`. All other styling inherits Qt's Fusion / system palette. Spacing and typography below describe the deltas this phase introduces against the existing dialog (already locked in `gbs_search_dialog.py:_build_ui()`); the contract preserves muscle memory rather than reskinning.

Phase 60.1 introduces NO new colors. The only color token in scope is the existing error red.

**Drill-down state visual cue — color policy:** "You are in drill-down mode" is signaled via *layout and copy* (the breadcrumb label + the Back button presence), NOT via a background color change, border highlight, or accent recolor. This keeps Phase 60.1 inside the project's no-design-system constraint and avoids competing with Qt's selection highlight.

## Layout Deltas (binding for Phase 60.1)

The existing `_build_ui()` layout in `gbs_search_dialog.py` is preserved verbatim outside the deltas listed below. **No widget is removed**; the drill-down view re-skins the existing widgets in place.

### Existing layout (preserved)

```
[QHBoxLayout: top_row]
  ├── _search_edit (QLineEdit, stretch=1)
  ├── _search_btn  (QPushButton "Search")
  └── _page_label  (QLabel "Page N of M")
[_progress] (QProgressBar, indeterminate, hidden when idle)
[_artist_label + _artist_list] (hidden when no matches; max-h 80px)
[_album_label + _album_list]  (hidden when no matches; max-h 80px)
[_results_table] (QTableView, stretch=1)
[_error_label] (inline, hidden when empty)
[QHBoxLayout: page_row]
  ├── _prev_btn (QPushButton "← Prev")
  ├── stretch
  └── _next_btn (QPushButton "Next →")
[QDialogButtonBox: Close]
```

### Drill-down deltas (Phase 60.1)

**Delta 1 — Top row dual-purpose:** While in drill-down mode, `_page_label` is hidden (`setVisible(False)`) and a NEW `_breadcrumb_label` (QLabel, plain text, no font override) takes its slot in `top_row`. The breadcrumb label uses copy `Viewing artist: {name}` or `Viewing album: {name}` per Copywriting Contract. On Back-to-search, `_breadcrumb_label.setVisible(False)` and `_page_label.setVisible(True)` restore the prior state.

**Delta 2 — Pagination row toggles to Back-button row:** While in drill-down mode, `_prev_btn` and `_next_btn` are hidden (`setVisible(False)`) and a NEW `_back_btn` (QPushButton with copy `← Back to search`) is shown in their place inside the same `page_row` `QHBoxLayout`. Layout shape: `[_back_btn] [stretch]` (left-aligned, mirroring the leftward arrow). On Back-to-search click, `_back_btn.setVisible(False)`, `_prev_btn` and `_next_btn` restore to their pre-drill enabled/disabled state.

  Rationale for replacing rather than appending: keeps total row count stable (no new vertical space, no dialog resize). Matches the per-CONTEXT.md "Claude's discretion" call to either-place-or-replace; this UI-SPEC pins it to **replace**.

**Delta 3 — Search input + Artist/Album panels remain VISIBLE during drill-down.** Per CONTEXT.md D-05 ("Search input + Artist/Album panels stay visible"), `_search_edit`, `_search_btn`, `_artist_list`, `_album_list` (and their headers) are NOT hidden by entering drill-down mode. They retain whatever visibility/content they had at click time. Clicking another artist/album entry while already in drill-down mode is permitted and triggers a new drill-down fetch (replacing the current drilled view, no breadcrumb stack).

**Delta 4 — Results table content swap:** `_results_table` itself is not removed or recreated. The drill-down `finished` slot calls `_clear_table()` (which already clears the rows AND hides the panels — but the panels MUST be re-shown by the caller to honor Delta 3, since the original panel state is what the user clicked from), then re-populates rows with the drilled artist/album catalog using the SAME `_render_results()` path. The Add! per-row buttons remain functional in drill-down mode (a song from an artist's catalog is still a submittable song).

  **Implementation note for the executor:** `_clear_table()`'s side-effect of hiding the artist/album panels (defensive code from Phase 60-11 to avoid stale-panel flicker after a NEW search) conflicts with Delta 3 in the drill-down render path. The drill-down render slot must restore panel visibility from saved pre-drill state. Save `(artist_links, album_links)` (the lists already in `_artist_list` / `_album_list` via UserRole) before calling `_clear_table()`, then re-populate the panels from saved state. Alternatively, refactor `_clear_table()` to take a `clear_panels: bool = True` kwarg and pass `False` from the drill-down render path. Either approach satisfies the contract; planner picks.

**Delta 5 — State-restoration on Back:** The Back button restores the dialog to the EXACT state captured before drill-down began. State to preserve and restore:

| Saved state | Source | Restored to |
|-------------|--------|-------------|
| `_results` list | `self._results` at click time | re-render via `_render_results()` |
| `_current_page` / `_total_pages` | dialog state at click time | reassigned; `_page_label` re-set with `Page N of M` |
| `_prev_btn.isEnabled()` / `_next_btn.isEnabled()` | derived from `_current_page > 1` / `< _total_pages` | recomputed on restore |
| `_artist_list` / `_album_list` populated items + visibility | snapshot of `(artist_links, album_links)` taken at click time | re-populated via the same logic as `_on_metadata_ready` |
| `_search_edit.text()` | unchanged during drill-down (Delta 3) | no action needed |
| Inline error label | hidden on Back (a fresh state shouldn't carry forward a drill-down error) | `_hide_inline_error()` |

The state snapshot is captured ONCE at the moment the user clicks an Artist/Album panel entry, BEFORE the drill-down worker is dispatched. Snapshot lives on `self._pre_drill_state: Optional[dict]`. Set to `None` after Back is clicked.

## Interaction States

For each interactive widget added or repurposed in Phase 60.1, the following state matrix is binding:

### `_artist_list` / `_album_list` (existing widgets, behavior changed)

| State | Behavior | Visual |
|-------|----------|--------|
| Idle (no search yet OR no Artist/Album matches) | Hidden | n/a (panel not in layout) |
| Populated (page-1 search with matches) | Visible, scrollable, max-height 80px | Qt default list rendering |
| Item hovered | Qt default hover highlight | system theme |
| Item clicked (Phase 60.1 NEW) | Triggers drill-down: snapshot pre-drill state → dispatch `_GbsArtistWorker` or `_GbsAlbumWorker` → show `_progress` bar → on `finished`, render drilled catalog into `_results_table` and switch top row + bottom row to drill-down deltas | breadcrumb appears, Back button replaces pagination, table re-renders with drilled songs |
| Item clicked while ALREADY in drill-down mode | Permitted. Snapshot is NOT updated (the original snapshot still represents the search state to restore on Back). New drill-down replaces the current drilled view. | breadcrumb label updates to new artist/album name; table re-renders |

### `_back_btn` (new)

| State | Behavior | Visual |
|-------|----------|--------|
| Hidden (default — search mode) | not in layout (hidden) | n/a |
| Enabled (drill-down mode, fetch settled) | Click → restore `_pre_drill_state` per Delta 5 → hide `_back_btn` → show `_prev_btn` / `_next_btn` → hide breadcrumb → show `_page_label` | standard QPushButton, no override |
| Disabled (drill-down fetch in flight) | Disabled to prevent mid-fetch retreat that would race with the worker callback. Re-enabled on `finished` or `error`. | Qt default disabled state (greyed) |
| Disabled while another drill-down is in flight | Same as above | Qt default disabled state |

### `_breadcrumb_label` (new)

| State | Behavior | Visual |
|-------|----------|--------|
| Hidden (default) | not visible; `_page_label` occupies the slot | n/a |
| Visible (drill-down mode, post-fetch) | Set text to `Viewing artist: {name}` or `Viewing album: {name}`, `setVisible(True)`. `_page_label.setVisible(False)`. | plain text label, no color override |

### `_progress` (existing widget, reused for drill-down)

| State | Behavior | Visual |
|-------|----------|--------|
| Hidden | default | n/a |
| Visible during search | unchanged | indeterminate bar |
| Visible during drill-down fetch (Phase 60.1 NEW use) | identical to search-fetch behavior. The same widget, the same `setVisible(True)` / `setVisible(False)` toggles. | indeterminate bar |

  **Rationale (judgment call):** Reusing the existing progress bar avoids introducing a second loading indicator and reads as "the dialog is fetching" regardless of which fetch is in flight. The breadcrumb label only appears post-fetch, which makes the loading affordance unambiguous (no half-rendered breadcrumb during a fetch).

### `_search_edit` / `_search_btn` (existing — interaction in drill-down)

| State | Behavior |
|-------|----------|
| User edits and presses Enter / clicks Search WHILE in drill-down mode | A new search is dispatched. This implicitly exits drill-down mode (`_pre_drill_state = None`, `_back_btn.setVisible(False)`, breadcrumb hidden, `_page_label` shown, panels and pagination return to search-mode behavior driven by the new search response). Snapshot is not restored — the user's intent is "abandon drill-down, do a new search." This must be explicit in the planned slot. |

## Out-of-Scope (negative space)

The following are explicitly OUT of scope for Phase 60.1 and MUST NOT appear in the implementation:

- Background color, border, or accent recolor to indicate drill-down mode (covered by breadcrumb + Back-button instead)
- Bold, italic, or font-size override on the breadcrumb label
- A separate "Loading…" text label during drill-down fetch (`_progress` bar carries the affordance)
- A second progress indicator dedicated to drill-down
- Pagination within the drilled artist's catalog (deferred per CONTEXT.md)
- Multi-level breadcrumb stack (artist → album → song); breadcrumb is single-level
- Confirmation dialog before retreating via Back (Back is non-destructive)
- A new icon library or any SVG/PNG icons (text + UTF-8 arrows only)
- Multi-word query input validation, hint text, or live feedback (silent fix per parser)
- Resize of `setMinimumSize(640, 480)`
- A new color in `_theme.py`
- A new `setStyleSheet` call on `_back_btn`, `_breadcrumb_label`, or any drill-down widget

## Judgment Calls Made by This UI-SPEC (no upstream answer)

1. **Back-button placement:** PINNED to "replaces pagination row entirely while in drill-down" rather than "next to pagination." Keeps row count stable, avoids dialog resize.
2. **Drill-down state visual cue:** PINNED to breadcrumb label (`Viewing artist: {name}`) replacing `Page N of M` in the top row. No background or border color change. Distinguishes artist vs album surface in the copy itself.
3. **Loading state during drill-down fetch:** PINNED to reuse existing `_progress` `QProgressBar`. No new indicator, no new label. Single source of "the dialog is busy" affordance.
4. **Empty-state for zero-row drill-down:** PINNED to inline `_error_label` (the dialog's existing inline-message slot) with copy `No songs found for {name}.` Defensive only — shouldn't occur in practice. Accepts the existing red color since the situation is anomalous.
5. **Multi-word input feedback:** PINNED to silent (no UI surface). The fix is parser-side per CONTEXT.md D-04; the user's existing mental model ("type words, press Enter, get results") shouldn't change.
6. **Click on Artist/Album panel WHILE already in drill-down:** PINNED to "permitted, replaces current drilled view, snapshot is preserved as the original search state." User can hop between drilled artists without re-searching, and Back always returns to the original search.
7. **New search WHILE in drill-down:** PINNED to "abandons drill-down, snapshot discarded." User intent is unambiguous.
8. **Snapshot strategy:** PINNED to a single `self._pre_drill_state: Optional[dict]` captured pre-dispatch, restored on Back, cleared on (a) Back, (b) new search, or (c) dialog dismiss. Planner may choose either the kwarg-on-`_clear_table` approach or the snapshot-and-restore-panels approach for Delta 4.

## Risks / Flags for Checker

- **FLAG-01 (no-design-system tradeoff):** This dialog has no theme tokens beyond `ERROR_COLOR_HEX`. Standard checker dimensions (60/30/10 color split, accent-reserved-for) are evaluated against Qt's system palette rather than explicit tokens. The contract complies in spirit (selection highlight is the sole accent) but cannot prove compliance via grep. Document this as PASS-with-context.
- **FLAG-02 (error-label re-use for empty drill-down):** The `No songs found for {name}.` copy lands in the same red `_error_label` widget used for actual errors. This is technically a misuse of the slot (information vs error). Acceptable because (a) the situation is defensive-only and shouldn't occur in practice, (b) introducing a second informational-label widget would expand the surface area beyond Phase 60.1's scope.
- **FLAG-03 (snapshot capture timing):** Delta 5 mandates capturing pre-drill state BEFORE dispatching the worker. If a user spam-clicks an Artist entry while a drill-down fetch is in flight, the snapshot must NOT be re-captured (would overwrite the original search state with the partially-rendered drilled state). Mitigation in Interaction States table: `_back_btn` is disabled during fetch, but the panels themselves remain enabled — the "click while in drill-down" path must explicitly check `if self._pre_drill_state is None: snapshot()` before snapshotting.
- **RISK-01 (Qt panel-hide side effect):** `_clear_table()` defensively hides the artist/album panels (Phase 60-11 ordering invariant). Drill-down render path must NOT inherit that side effect or the panels disappear when the drilled catalog renders, violating Delta 3. Planner must address (kwarg or refactor or save+restore).
- **RISK-02 (multi-line breadcrumb truncation):** Long artist names (e.g., the actual GBS.FM record `"#gbs-fm's greatest shits"` for an album, ~25 chars) plus the prefix `Viewing artist: ` push close to typical dialog width at the 640px minimum. QLabel will wrap, possibly pushing the layout. Accepted: QLabel default word-wrap behavior is acceptable for v1; if it causes layout jump, planner can add `setMaximumWidth` or elide. Not contract-binding.
