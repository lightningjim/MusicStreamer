# Phase 60.4 Plan 01: gbs_api.fetch_user_tokens Capability + Test Scaffold Summary

**One-liner:** Adds the `fetch_user_tokens(cookies) -> Optional[int]` data-layer capability that scrapes the GBS.FM home page for the per-user token count, plus 6 unit tests locking happy-path / singular-variant / zero / parse-miss / network-error / auth-expired branches and a conftest spec extension so Plan 02's dialog tests can mock the new symbol.

---

## What was built

### `musicstreamer/gbs_api.py`

1. **New module-level constant** `_TOKEN_RE = re.compile(r"You have (\d+) tokens?", re.IGNORECASE)` — placed adjacent to the existing `_LINKED_SONGID_RE` precedent. The `?` quantifier covers both plural (`48 tokens`) and singular (`1 token`) prose per Pitfall D; `re.IGNORECASE` is defensive against future template-text capitalisation drift.

2. **New public function** `fetch_user_tokens(cookies: http.cookiejar.MozillaCookieJar) -> Optional[int]` — placed after `_fold_ajax_events` and before the `# ---------- Capability 4: Vote on now-playing ----------` section, with its own `# ---------- Phase 60.4 / GBS-01-followup-T: User token count ----------` section header. Behaviour:
   - GETs `${GBS_BASE}/` via the existing `_open_with_cookies` helper (which already raises `GbsAuthExpiredError` on 302 → `/accounts/login/` — D-T8 path).
   - Parses the response with `_TOKEN_RE.search(html_str)` and returns `int(match.group(1))` on success (D-T1).
   - Returns `None` on parse miss, `urllib.error.HTTPError` (5xx / 4xx-other), `urllib.error.URLError`, `OSError`, or `int()` conversion failure — all D-T2 paths. Each path emits a single `_log.warning(...)` line; no toast (Pitfall 3).
   - **Does NOT** re-handle 302 → login: `GbsAuthExpiredError` propagates from `_open_with_cookies` directly so callers (Plan 02 dialog) can route to the existing auth-expired UX.
