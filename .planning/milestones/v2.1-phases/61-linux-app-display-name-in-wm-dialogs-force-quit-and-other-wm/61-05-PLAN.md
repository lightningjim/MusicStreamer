<objective>
Close the BUG-08 regression that Plan 04's UAT exposed: the GNOME force-quit dialog (and the dock icon) still showed `org.lightningjim.MusicStreamer` + a generic gear icon when MusicStreamer was launched via `uv run musicstreamer` from PyCharm's embedded terminal, even after Plans 02+03 shipped correct .desktop files and a correct wayland app_id. Diagnostic captured 2026-05-05 (see "Diagnostic addendum" below) traced this to PyCharm exporting `XDG_ACTIVATION_TOKEN` (and `DESKTOP_STARTUP_ID`) into its terminal env, which `uv run` and the venv inherit. Qt's wayland integration then forwards that token via `xdg_activation_v1.activate(<PyCharm-token>, wl_surface#N)` for our first window, and mutter binds the surface to PyCharm's launch context ŌĆö short-circuiting the wayland-app-id ŌåÆ .desktop basename match.

Fix: strip both env vars at the top of `_run_gui()` before any Qt code runs. Two `os.environ.pop()` calls in a tiny helper, plus a unit test, plus a UAT re-run from PyCharm's terminal to confirm the dialog now reads "MusicStreamer".

Output: ~6 LOC of production code, ~25 LOC of tests, an appended `## POST-FIX-2` section in `61-DIAGNOSTIC-LOG.md` flipping the D-16 gate to PASS, and full pytest suite GREEN.

Note: This does NOT add a new architectural decision; the rationale is documented inline in the helper's docstring and in the `## POST-FIX-2` diagnostic note. If the project later wants a numbered decision record, that can be a separate edit to `61-DISCUSSION-LOG.md`.

## Diagnostic addendum (captured 2026-05-05 ŌĆö TWO root causes, not one)

### Initial hypothesis (env-inheritance) ŌĆö partial truth, not the fix

Wayland-debug trace from `WAYLAND_DEBUG=client uv run python -m musicstreamer` in **PyCharm terminal**:

```
xdg_toplevel#42.set_title("MusicStreamer")                                          ŌåÉ OK (Plan 02)
xdg_toplevel#42.set_app_id("org.lightningjim.MusicStreamer")                        ŌåÉ OK (Plan 02)
xdg_activation_v1#35.activate("gnome-shell/PyCharm 2026.1.1/8345-7-hurricane_...", wl_surface#37)   ŌåÉ suspicious
```

PyCharm's terminal exports `XDG_ACTIVATION_TOKEN`/`DESKTOP_STARTUP_ID` into every child shell; Qt then forwards the stale token via `xdg_activation_v1.activate()` for our first surface. We hypothesized this token was binding our window to PyCharm's launch context and shipped the env-strip helper (Task 1).

### Second-pass verification ŌĆö env-strip didn't close BUG-08

Kyle re-ran the UAT against the new build from a **gnome-terminal** (no PyCharm token in env). Bug still present: dock and force-quit dialog still showed `org.lightningjim.MusicStreamer` + gear icon. Wire trace from clean env:

```
xdg_toplevel#42.set_title("MusicStreamer")
xdg_toplevel#42.set_app_id("org.lightningjim.MusicStreamer")
xdg_activation_token_v1#44.set_app_id("org.lightningjim.MusicStreamer")
xdg_activation_v1#35.activate("cfa391e6-0828-4166-9ece-4e82c6b45592_TIME0", wl_surface#37)
```

Qt mints its OWN activation token (uuid-style, `_TIME0` suffix = no input event) regardless of env state. The env-strip removes a stale token but doesn't suppress the self-generated one. The "PyCharm token forwarding" was real but not the load-bearing cause; mutter ignores app-self-activation tokens for matching purposes.

### Real root cause (cgroup name)

Mutter's `meta_window_get_unit_cgroup()` reads `/proc/<pid>/cgroup` and parses the v2 unified line for `app-<reverse-dns>-<token>.scope`. The `<reverse-dns>` segment IS the app id mutter uses to look up the .desktop file. For terminal launches, the cgroup is inherited from the parent shell's scope:

| Launch path | Cgroup leaf scope | Mutter parses app id as | Match result |
|---|---|---|---|
| Activities/dock click (gnome-shell launcher) | `app-org.lightningjim.MusicStreamer-<gshell-token>.scope` | `org.lightningjim.MusicStreamer` | Ō£ō matches our .desktop |
| `uv run` from gnome-terminal | `app-org.gnome.Terminal-<id>.scope` (or whatever the terminal's scope is) | `org.gnome.Terminal` | Ō£Ś wrong app, no match for our window |
| `uv run` from PyCharm terminal | `app-gnome-jetbrains-pycharm-<uuid>.scope` | `gnome-jetbrains-pycharm-<uuid-prefix>` | Ō£Ś wrong app |

This was confirmed empirically: a wrapper script `scripts/dev-launch.sh` that exec's musicstreamer via `systemd-run --user --scope --unit=app-org.lightningjim.MusicStreamer-$$.scope` puts the process in the right cgroup, and mutter immediately matches the window correctly (Kyle confirmed dock icon + tooltip render correctly when launched this way).

### Two fixes shipped ŌĆö both kept

| Fix | What it addresses | Required for end users? |
|-----|-------------------|-------------------------|
| Task 1: `_strip_inherited_activation_tokens()` | Stale `XDG_ACTIVATION_TOKEN` from JetBrains terminals (or any other launcher that leaks it) | No (defensive hygiene; harmless no-op when var is absent) |
| Task 3: `scripts/dev-launch.sh` | Mutter's cgroup-based appŌåöwindow matching for terminal-launched dev runs | No (end users launch via Activities/dock; this is dev-only) |

Conclusion: BUG-08 user-facing surfaces (Activities/dock/force-quit dialog under normal launches) were closed by Plans 02+03. The Plan 04 UAT FAIL surfaced a *dev-launch-only* edge case caused by terminal cgroup inheritance, which Plan 05 closes via `scripts/dev-launch.sh`. The env-strip is retained as defensive hygiene because it's a real (if cheaper) bug that would surface later in any launcher that leaks the activation token.

## Trust Boundaries

`os.environ.pop()` mutates the process's own env. No cross-process effect, no IPC, no filesystem write, no network. Single trust boundary: the values being stripped were untrusted input (inherited from an arbitrary parent process). Stripping is strictly defensive ŌĆö we are reducing the attack surface of what mutter sees from us, not adding any new authority.

## STRIDE Threat Register
