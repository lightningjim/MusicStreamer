# Phase 72.4: Volume slider / compact-toggle overlap on narrow lower-screen widths - Research

**Researched:** 2026-05-19
**Domain:** PySide6 / Qt 6.11 — second-tier width-driven reparent of a TWO-widget "volume cluster" between sibling QHBoxLayouts, extending the Phase 72.1 mechanism
**Confidence:** HIGH (direct codebase precedent at `now_playing_panel.py:1170-1432`; Qt API surface fully mapped; pytest-qt patterns established in `tests/test_phase72_1_stream_picker_reflow.py`)

## Summary

Phase 72.4 is a **second wrap tier** layered on top of the Phase 72.1 stream-picker reflow. When `panel.width()` drops below a NEW threshold (the row-1 minimum WITHOUT the volume cluster), the **volume slider + compact-mode toggle** wrap together as an indivisible unit to a wrap row. Three target rows are possible depending on stream count: in single-stream mode the cluster lands on `self._controls_row2` (which is otherwise empty since the picker is hidden); in multi-stream mode the cluster lands on a NEW `self._controls_row3` (because row 2 already holds the picker per 72.1).

CONTEXT.md D-01..D-14 and UI-SPEC.md lock every visible decision. This research answers the eight planning-specific HOW questions in the objective: which threshold-computation strategy to pick (generalize `_picker_row_min_width` via parameters — option (a)), the canonical Qt 6 setFixedWidth↔Expanding round-trip (three-call sequence: `setMinimumWidth(0)` + `setMaximumWidth(QWIDGETSIZE_MAX)` + `setSizePolicy(Expanding, Fixed)`), the cluster-reparent pattern (single helper that walks both widgets in order, mirroring the row-1-return precedent at `now_playing_panel.py:1338-1367`), the test-coverage envelope (eight sample points across the six-state truth table + signal/state survival + idempotency + widen-back), the new pitfalls specific to two-widget cluster wrapping, the resizeEvent call order (picker → volume-cluster → art-tier, already established in 72.3), and the `LAYOUT-04` requirement row format.

**Primary recommendation:** Generalize `_picker_row_min_width()` into `_row1_min_width(include_picker: bool, include_volume_cluster: bool) -> int` with a two-key cache `dict[tuple[bool, bool], int]`. Add `_reflow_volume_cluster()` as a sibling predicate to `_reflow_stream_picker_row()` invoked from the same `resizeEvent` override between the picker reflow and the art-tier reflow. The volume slider's size-policy round-trip uses the three-call canonical sequence (verified against Qt 6 docs). The cluster reparent walks `(volume_slider, compact_mode_toggle_btn)` in left-to-right order via a single helper, using `insertWidget(self._volume_cluster_row1_index, ...)` for the row-1 return path (mirrors the 72.1 `_picker_row1_index` pattern at line 509).

### Pattern 1: Generalized Threshold Helper (Parameterized Cache)

**What:** Replace `_picker_row_min_width()` (single-purpose, computes row-1-with-picker minimum) with `_row1_min_width(include_picker: bool, include_volume_cluster: bool) -> int` that supports four threshold variants and shares the outer-chrome calculation across all of them.

**When to use:** When two related thresholds need to be computed from the same outer-chrome walk and share most of their widget enumeration. Avoids code duplication and outer-chrome drift between the two helpers.

**Example:**
```python
# Source: derived from now_playing_panel.py:1260-1336 (existing _picker_row_min_width)
#         + CONTEXT 72.4 Claude's Discretion (option a — generalized helper)

def _row1_min_width(
    self,
    include_picker: bool,
    include_volume_cluster: bool,
) -> int:
    """Compute the panel.width() floor below which row 1 cannot fit.

    Phase 72.1 / LAYOUT-02 was a single-purpose `_picker_row_min_width()`.
    Phase 72.4 / LAYOUT-04 generalizes this to support TWO thresholds:
      - include_picker=True,  include_volume_cluster=True  -> row-1-with-everything (72.1 picker-wrap threshold)
      - include_picker=False, include_volume_cluster=True  -> row-1-without-picker, with-cluster (single-stream picker hidden)
      - include_picker=True,  include_volume_cluster=False -> row-1-with-picker, without-cluster (multi-stream cluster-wrap threshold)
      - include_picker=False, include_volume_cluster=False -> row-1-without-anything (single-stream + cluster-wrapped — the floor)

    Cache key is the (include_picker, include_volume_cluster) tuple. Cache
    invalidated by _populate_stream_picker (defensive — widget count is
    unchanged by 72.4's scope, but keeps cache invariant tight; Pitfall 5
    in 72.1-RESEARCH).
    """
    cache_key = (include_picker, include_volume_cluster)
    if cache_key in self._row1_min_cache:
        return self._row1_min_cache[cache_key]

    layout = self.controls  # row 1 QHBoxLayout
    margins = layout.contentsMargins()
    spacing = layout.spacing()
    total = margins.left() + margins.right()
    widget_count = 0

    # Walk row 1 widgets. Per-widget include flags decide which ones count.
    picker_in_row1 = layout.indexOf(self.stream_combo) >= 0
    volume_in_row1 = layout.indexOf(self.volume_slider) >= 0
    compact_in_row1 = layout.indexOf(self.compact_mode_toggle_btn) >= 0

    for i in range(layout.count()):
        item = layout.itemAt(i)
        w = item.widget()
        if w is None:
            continue  # addStretch -- no width contribution
        # Per-widget exclusions based on include flags + current layout state.
        if w is self.stream_combo and not include_picker:
            continue
        if w is self.volume_slider and not include_volume_cluster:
            continue
        if w is self.compact_mode_toggle_btn and not include_volume_cluster:
            continue
        wmin = max(w.minimumWidth(), w.minimumSizeHint().width())
        total += wmin
        widget_count += 1

    # If picker is currently NOT in row 1 but include_picker=True, add it.
    # (Layout-agnostic contract — mirrors 72.1 CR-01 fix at line 1305-1312.)
    if include_picker and not picker_in_row1:
        total += max(
            self.stream_combo.minimumWidth(),
            self.stream_combo.minimumSizeHint().width(),
        )
        widget_count += 1
    # If cluster widgets are currently NOT in row 1 but include_volume_cluster=True, add them.
    if include_volume_cluster and not volume_in_row1:
        total += max(
            self.volume_slider.minimumWidth(),
            self.volume_slider.minimumSizeHint().width(),
        )
        widget_count += 1
    if include_volume_cluster and not compact_in_row1:
        total += max(
            self.compact_mode_toggle_btn.minimumWidth(),
            self.compact_mode_toggle_btn.minimumSizeHint().width(),
        )
        widget_count += 1

    if widget_count > 1:
        total += spacing * (widget_count - 1)

    # Outer chrome walk (unchanged from 72.1 — logo + cover + outer margins/spacing).
    outer = self.layout()
    if outer is not None:
        om = outer.contentsMargins()
        total += om.left() + om.right()
        for i in range(outer.count()):
            item = outer.itemAt(i)
            w = item.widget()
            if w is not None:
                wmin = max(w.minimumWidth(), w.minimumSizeHint().width())
                total += wmin
        if outer.count() > 1:
            total += outer.spacing() * (outer.count() - 1)

    total += 16  # Safety buffer (splitter handle + rounding) — preserved from 72.1.

    self._row1_min_cache[cache_key] = total
    return total
```

**Critical:**
- Cache type changes from `Optional[int]` to `dict[tuple[bool, bool], int]`. Init at `__init__` to `{}`. The `_cached_row1_min: Optional[int]` field at `now_playing_panel.py:297` is **renamed/repurposed** to `self._row1_min_cache: dict[tuple[bool, bool], int] = {}`.
- The existing `_picker_row_min_width()` callers (line 1425) call into the new generalized helper. Provide a thin shim or update call site:
  ```python
  # Old: threshold = self._picker_row_min_width()
  # New: threshold = self._row1_min_width(include_picker=True, include_volume_cluster=True)
  ```
- `_populate_stream_picker` at line 1213 currently sets `self._cached_row1_min = None`. New form: `self._row1_min_cache.clear()`. Same semantics.
- The layout-agnostic contract from 72.1 CR-01 (lines 1305-1312) MUST be preserved for both picker and cluster widgets. The helper computes "row-1 if these widgets WERE there" regardless of where they currently are.

### Pattern 2: Cluster Reparent Helper (Two-Widget Unit)

**What:** Single helper method `_move_volume_cluster_to(target_layout)` that walks both cluster widgets (`volume_slider`, `compact_mode_toggle_btn`) in left-to-right order, performing the reparent as a unit. Idempotent via `indexOf` membership check on EITHER widget (the cluster-as-unit invariant D-03 guarantees they're always together).

**When to use:** When a fixed-membership group of widgets needs to move together between layouts. Single helper consolidates the order invariant; parallel helpers spread it across call sites.

**Example:**
```python
# Source: derived from now_playing_panel.py:1338-1367 (existing _move_stream_picker_to)
#         + CONTEXT D-01..D-03 (cluster-as-unit)

def _move_volume_cluster_to(self, target_layout: QHBoxLayout) -> None:
    """Reparent the volume cluster (volume_slider + compact_mode_toggle_btn)
    into target_layout idempotently as a single unit.

    Cluster-as-unit invariant (CONTEXT D-03): both widgets move together or
    neither does. The left-to-right order on the wrap row is volume_slider →
    compact_mode_toggle_btn (D-02), preserving row-1 reading order. On row-1
    return, both widgets are inserted at their original position via
    insertWidget(self._volume_cluster_row1_index, ...) — analogous to 72.1
    Pitfall 6 / Pattern B for the picker's _picker_row1_index.

    Pattern 2 from 72.1-RESEARCH (lines 228-263): QObject::setParent (called
    implicitly by addWidget) does NOT disconnect signal/slot connections;
    QSlider.value() and QToolButton.isChecked() / icon() / toolTip() survive
    reparent. Drag-in-progress on the slider also survives (Qt routes the
    in-flight mouse grab through the same widget regardless of parent).

    Use the volume_slider as the membership probe (cluster invariant — if
    volume is in layout X then compact-toggle is also in layout X).
    """
    # Determine current home using volume_slider as the probe.
    current_in_row1 = self.controls.indexOf(self.volume_slider) >= 0
    current_in_row2 = self._controls_row2.indexOf(self.volume_slider) >= 0
    current_in_row3 = self._controls_row3.indexOf(self.volume_slider) >= 0

    # Idempotent early-returns (Pattern 2 -- avoids spurious reparent churn).
    if target_layout is self.controls and current_in_row1:
        return
    if target_layout is self._controls_row2 and current_in_row2:
        return
    if target_layout is self._controls_row3 and current_in_row3:
        return

    # Remove BOTH cluster members from whichever layout currently owns them.
    # Cluster-as-unit invariant: both are always in the same layout.
    current = (
        self.controls if current_in_row1
        else self._controls_row2 if current_in_row2
        else self._controls_row3 if current_in_row3
        else None
    )
    if current is not None:
        # Order doesn't matter for removal; layout repacks after.
        current.removeWidget(self.volume_slider)
        current.removeWidget(self.compact_mode_toggle_btn)

    # Add to target -- target-specific to preserve row-1 insertion position.
    if target_layout is self.controls:
        # Row-1 return: insert at the captured original position so the
        # cluster sits between eq_toggle_btn and addStretch(1) — its
        # original visual location. Insert volume FIRST, then compact-toggle
        # at idx+1 (D-02 left-to-right order).
        self.controls.insertWidget(
            self._volume_cluster_row1_index, self.volume_slider
        )
        self.controls.insertWidget(
            self._volume_cluster_row1_index + 1, self.compact_mode_toggle_btn
        )
    else:
        # Wrap row (row 2 or row 3): addWidget in left-to-right order.
        # No addStretch -- volume's Expanding policy pins compact-toggle right.
        target_layout.addWidget(self.volume_slider)
        target_layout.addWidget(self.compact_mode_toggle_btn)
```

**Critical:**
- The `_volume_cluster_row1_index` field is captured at construction time AFTER `self.controls.addWidget(self.volume_slider)` (line 546) and BEFORE `self.controls.addWidget(self.compact_mode_toggle_btn)` (line 566). At that point, volume_slider's index is the cluster's left-edge position. On insertWidget back, the +1 offset is the compact-toggle's slot.
- **Critical insertWidget gotcha:** When you insertWidget at index N then insertWidget at index N+1, both go to the right place because the first insertion pushes everything to the right BEFORE the second insertion. This is the same convention 72.1 didn't have to handle (only one widget). If reversed (insert at N+1 first then N), the order would be wrong. **Insert volume first at N, then compact-toggle at N+1.**
- Using `volume_slider` as the membership probe is safe because of the cluster-as-unit invariant (D-03). If the invariant is ever broken (a bug puts volume in row 1 and compact in row 2), the helper would still produce a correct end-state for volume's target while compact would stay where it was — a partial fix that surfaces the bug. Defensive but acceptable.
- `removeWidget` is safe to call even if a widget isn't in the layout (Qt no-ops). But calling it twice from row 1 with `current` set to row 1 is the normal flow; no double-remove.
- The signal connections on both widgets survive — `valueChanged.connect(self._on_volume_changed_live)` at line 774, `sliderReleased.connect(self._on_volume_released)` at line 775, `toggled.connect(self._on_compact_btn_toggled)` at line 565 [VERIFIED via codebase grep + 72.1 Pattern 2 verification].

### Pattern 3: Volume Slider Size-Policy Round-Trip (Canonical Qt 6)

**What:** Toggle the volume slider between `setFixedWidth(120)` (row 1) and `setSizePolicy(Expanding, Fixed)` with cleared min/max width (wrap row). The canonical Qt 6 sequence requires explicitly resetting the min and max width constraints set by `setFixedWidth` before applying the new policy.

**When to use:** Any time a widget that has been locked with `setFixedWidth()` needs to grow via a non-Fixed size policy.

**Example:**
```python
# Source: derived from Qt 6 QWidget docs https://doc.qt.io/qt-6/qwidget.html#setFixedWidth
#         [VERIFIED 2026-05-19 via WebFetch — setFixedWidth sets minimum=maximum=fixed_w internally;
#          undoing requires resetting both bounds, NOT just setSizePolicy()]
#         + now_playing_panel.py:1369-1383 (_set_picker_size_policy precedent)

# Module-level constants
from PySide6.QtWidgets import QSizePolicy, QWIDGETSIZE_MAX

def _set_volume_size_policy(self, expanding: bool) -> None:
    """Toggle volume slider horizontal sizing for row 1 (Fixed 120) vs wrap row (Expanding).

    D-09: Expanding on wrap row (volume stretches to fill the row's free width
    minus the 28px compact-toggle).
    D-10: Fixed 120 on row 1 (restores the pre-72.4 baseline at line 544).

    CANONICAL Qt 6 ROUND-TRIP (verified via Qt docs 2026-05-19):
    setFixedWidth(120) at construction sets BOTH minimumWidth=120 AND
    maximumWidth=120 internally. Undoing this requires:
      (1) setMinimumWidth(0)              -- release the floor
      (2) setMaximumWidth(QWIDGETSIZE_MAX) -- release the cap (16777215)
      (3) setSizePolicy(Expanding, Fixed) -- enable growth

    Calling only setSizePolicy() after setFixedWidth() does NOT make the
    widget grow -- the min=max=120 constraints from setFixedWidth still
    pin the width. This is the key Qt 6 invariant the 72.4 helper must
    honor that 72.1's _set_picker_size_policy did not need to (the picker
    only ever called setMinimumWidth(140), not setFixedWidth).

    On row-1 return, setFixedWidth(120) restores both bounds in one call AND
    sets the QSizePolicy via Qt's internal sync; explicitly calling
    setSizePolicy(Fixed, Fixed) is a belt-and-braces safety belt (matches
    72.1's explicit Preferred-on-revert pattern at line 1383).
    """
    if expanding:
        # Release the setFixedWidth(120) constraint set at line 544.
        self.volume_slider.setMinimumWidth(0)
        self.volume_slider.setMaximumWidth(QWIDGETSIZE_MAX)
        self.volume_slider.setSizePolicy(QSizePolicy.Expanding, QSizePolicy.Fixed)
    else:
        # Restore row-1 baseline.
        self.volume_slider.setFixedWidth(120)
        # Belt and braces: setFixedWidth internally syncs policy to Fixed/Fixed,
        # but explicitly setting it documents intent and guards against Qt
        # version drift (mirrors 72.1's Preferred-on-revert at line 1383).
        self.volume_slider.setSizePolicy(QSizePolicy.Fixed, QSizePolicy.Fixed)
    # Compact-toggle's setFixedSize(28, 28) at line 556 is NEVER mutated.
    # D-11: 28×28 fixed in every state.
```

**Critical:**
- `QWIDGETSIZE_MAX = 16777215` (2^24 - 1) [CITED: Qt 6 QWidget docs — "maximum size property defaults to a value of 16777215"].
- The three-call sequence is the canonical Qt 6 pattern. Skipping step (1) or (2) leaves the min=max=120 constraints in place, and the slider stays 120px wide even after `setSizePolicy(Expanding, ...)`.
- `QSizePolicy.Fixed` (not `Preferred`) for the vertical policy. QSlider's vertical size is essentially fixed (it's a single-line control); using `Preferred` would technically work but `Fixed` matches the slider's intrinsic geometry [VERIFIED: codebase grep — no other QSlider in the project sets a non-Fixed vertical policy].
- On row-1 return, `setFixedWidth(120)` is sufficient on its own (it resets both min and max to 120). The explicit `setSizePolicy(Fixed, Fixed)` is a safety belt mirroring 72.1's pattern at line 1383.
- The slider's `setTickPosition(QSlider.NoTicks)`, `setRange(0, 100)`, and any current value() all survive these calls — only width constraints and size policy are mutated [VERIFIED: Qt 6 docs — these are independent properties].

### Anti-Patterns to Avoid

- **Two parallel helpers `_move_volume_slider_to` + `_move_compact_toggle_to`.** Spreads the cluster-as-unit invariant across two call sites; one helper that walks both widgets in order is the established pattern from CONTEXT D-03.
- **Reading `panel.layout()` repeatedly inside `_row1_min_width`.** Cache the outer chrome walk inside the dict cache; the (False, False) variant doesn't need it but the (True, True) and (True, False) and (False, True) variants all do — and they all share the same chrome.
- **`addStretch(1)` on the wrap row.** Explicit CONTEXT D-11 violation. Volume's Expanding policy already consumes free space; adding stretch would compete with Expanding and produce non-deterministic right-edge alignment for compact-toggle.
- **Toggling compact-toggle's size on reparent.** Explicit CONTEXT D-11 violation. Compact-toggle stays 28×28 fixed in every state.
- **Calling only `setSizePolicy(Expanding, Fixed)` without first resetting min/max width.** The `setFixedWidth(120)` constraint from construction (line 544) persists through `setSizePolicy()` because setFixedWidth pins min=max=120 directly. Pattern 3 documents the canonical three-call sequence.
- **Computing the threshold inside `resizeEvent` on every call.** Walks the row's 8-9 widgets + outer chrome — measurable cost on rapid resize. The parameterized cache makes this once-per-(include_picker, include_volume_cluster) variant.
- **`lambda` in any signal connect inside `_reflow_volume_cluster` / `_move_volume_cluster_to` / `_set_volume_size_policy` / `_row1_min_width`.** Project-wide QA-05 violation. Use bound methods. Source-grep test enforces (per 72.1 negative-2 pattern).
- **Coupling the wrap trigger to compact-mode state.** Explicit CONTEXT D-12 violation. Width is the sole input.
- **Hand-tuned constant for the cluster threshold.** Breaks when row 1 widgets change in future phases. Use the generalized helper.
- **Hiding `_controls_row3` explicitly when empty.** Qt collapses an empty QHBoxLayout to zero height; explicit hide doubles code surface for no gain (72.1 ran for weeks without a row 2 wrapper).
- **Reading `self.stream_combo.isVisible()` instead of `isHidden()` for stream-count branching.** `isVisible()` returns False on any widget whose top-level ancestor has not been shown; pre-show contexts (tests) would misclassify single-stream as multi-stream. Use `isHidden()` (intended-visibility semantics). 72.1 burned a full plan on this at line 1410-1419.

## Common Pitfalls

### Pitfall 1: setFixedWidth + setSizePolicy interaction (NEW for 72.4)

**What goes wrong:** The implementer calls `setSizePolicy(QSizePolicy.Expanding, QSizePolicy.Fixed)` on the volume slider when wrapping, but forgets to first reset the min/max width constraints set by `setFixedWidth(120)` at construction. The slider stays 120px wide on the wrap row instead of expanding to fill the available width.

**Why it happens:** `setFixedWidth(120)` is implemented by Qt as `setMinimumWidth(120); setMaximumWidth(120)` — it pins both bounds directly. `setSizePolicy()` only controls how the widget *grows within* its [min, max] range; if min==max==120, the policy has no room to expand.

**How to avoid:** Use the canonical three-call sequence from Pattern 3:
```python
slider.setMinimumWidth(0)
slider.setMaximumWidth(QWIDGETSIZE_MAX)  # 16777215
slider.setSizePolicy(QSizePolicy.Expanding, QSizePolicy.Fixed)
```

**Warning signs:** Manual UAT on narrow widths: cluster wraps to a new row but volume slider remains 120px wide; compact-toggle sits 8px to the right of volume's end rather than pinned to the row's right edge.

### Pitfall 2: insertWidget order matters (TWO widgets at adjacent slots)

**What goes wrong:** On row-1 return, the implementer inserts compact-toggle at `_volume_cluster_row1_index + 1` BEFORE inserting volume at `_volume_cluster_row1_index`. Result: compact-toggle ends up where volume should be, and volume ends up one slot to the right.

**Why it happens:** `insertWidget(idx, w)` pushes everything at index ≥ idx to the right. If you insert compact-toggle at idx=N+1 first (when N+1 is currently the slot of the trailing `addStretch`), it goes there and the stretch moves to N+2. Then insertWidget at idx=N for volume: volume goes to N, compact-toggle stays at N+1. Result is correct IF stretch was at N+1 before, but if you reverse the order...

**How to avoid:** Always insert in left-to-right order: volume FIRST at `_volume_cluster_row1_index`, then compact-toggle at `_volume_cluster_row1_index + 1`. The mental model: "I'm filling slots left-to-right."

**Warning signs:** Widen-back test fails an order assertion — the cluster appears in row 1 with compact-toggle before volume (reversed).

### Pitfall 3: Cluster invariant break — only one widget moved

**What goes wrong:** A test harness or a bug in `_move_volume_cluster_to` removes only one of the two cluster widgets, leaving the other stranded. The cluster-as-unit invariant (D-03) is violated; subsequent `_reflow_volume_cluster` calls see partial state.

**Why it happens:** `removeWidget(volume_slider)` succeeds and `removeWidget(compact_mode_toggle_btn)` raises or no-ops. The membership probe (which reads volume's index) reports "not here" but compact-toggle is still stranded.

**How to avoid:** The helper performs both removes unconditionally (Qt no-ops if not present). Idempotent probe reads volume's index, but the helper assumes the cluster invariant. Add a debug-only invariant assertion in the helper if paranoid:
```python
# Invariant guard (cheap in production; would fire only on bug):
volume_layout = (
    self.controls if self.controls.indexOf(self.volume_slider) >= 0
    else self._controls_row2 if self._controls_row2.indexOf(self.volume_slider) >= 0
    else self._controls_row3 if self._controls_row3.indexOf(self.volume_slider) >= 0
    else None
)
compact_layout = (
    self.controls if self.controls.indexOf(self.compact_mode_toggle_btn) >= 0
    else self._controls_row2 if self._controls_row2.indexOf(self.compact_mode_toggle_btn) >= 0
    else self._controls_row3 if self._controls_row3.indexOf(self.compact_mode_toggle_btn) >= 0
    else None
)
assert volume_layout is compact_layout, "cluster-as-unit invariant broken (D-03)"
```

**Warning signs:** Tests fail with cluster split across two rows. Defensive guard catches it.

### Pitfall 4: Stream-count branch reads wrong predicate (isVisible vs isHidden)

**What goes wrong:** `_reflow_volume_cluster` reads `self.stream_combo.isVisible()` to decide row 2 vs row 3. In a test context where the panel hasn't been `.show()`ed, `isVisible()` returns False even for a multi-stream station (because the top-level ancestor isn't shown). The helper picks row 2, but row 2 is supposed to hold the picker (for the multi-stream test). Cluster collides with picker on row 2.

**Why it happens:** Qt's `isVisible()` returns False on any widget whose top-level ancestor has not been shown, regardless of explicit `setVisible(...)` state. 72.1 burned a full plan on this at line 1410-1419 and pinned the lesson: use `isHidden()` for intended-visibility semantics. `_populate_stream_picker` at line 1208 calls `setVisible(len > 1)`, which sets `isHidden()` correctly even pre-show.

**How to avoid:** Use `self.stream_combo.isHidden()` for the stream-count branch. Matches the 72.1 pattern at line 1420.

**Warning signs:** Test `test_narrow_multi_stream_cluster_on_row_3` fails because the cluster lands on row 2 instead of row 3. UAT-only: works fine after first interactive resize because `.show()` has fired.

### Pitfall 5: Spurious resize during __init__ before _controls_row3 exists

**What goes wrong:** A Qt internal layout pass fires `resizeEvent` during `__init__` before the new `self._controls_row3` field is constructed. `_reflow_volume_cluster` accesses `self._controls_row3` → AttributeError.

**Why it happens:** Same root cause as 72.1 Pitfall 3 (line 455-463). Setting `parent` on any widget constructor can cascade resize events up the parent chain. The first construction pass may fire before all layout containers exist.

**How to avoid:** Defensive guard at the top of `_reflow_volume_cluster`:
```python
if not hasattr(self, "volume_slider"):
    return
if not hasattr(self, "compact_mode_toggle_btn"):
    return
if not hasattr(self, "_controls_row3"):
    return
```
Same shape as 72.1's defensive guard at lines 1402-1405.

**Warning signs:** AttributeError on first construction during pytest setup. Defensive guards make the helper safe regardless.

### Pitfall 6: Zero-width resize during teardown

**What goes wrong:** During pytest teardown, the panel is torn down. A resize event with `width() == 0` fires; `_reflow_volume_cluster` computes `should_wrap = 0 < threshold == True` and tries to reparent — but child widgets may already be in mid-destruction. Crash.

**Why it happens:** Same root cause as 72.1 Pitfall 5 and the Open Question 5 resolution (line 686). Qt fires spurious zero-size resizes during teardown.

**How to avoid:** Defensive guard:
```python
panel_width = self.width()
if panel_width <= 0:
    return
```
Same shape as 72.1's at line 1406-1408.

**Warning signs:** Random pytest crashes during teardown. Defensive guard masks.

### Pitfall 7: Cached threshold stale after future widget addition

**What goes wrong:** A future phase adds a 10th widget to row 1 (e.g., a "shuffle queue" button). The cached `_row1_min_cache[(True, False)]` is computed at first call and doesn't auto-invalidate. The cluster wraps at a stale-low threshold.

**Why it happens:** Cache is invalidated only by `_populate_stream_picker.clear()`. Widget additions during `__init__` of a future phase don't trigger any invalidation.

**How to avoid:** Document the cache-invalidation contract in the helper's docstring. Add a public method `invalidate_row1_min_cache()` that future phases call from their `__init__` edit point if they add widgets to row 1. Same pattern surfaced by 72.1 Pitfall 5 (line 477-484).

**Warning signs:** A future phase adds a row-1 widget; the wide layout works fine but the cluster wraps at the wrong width. Surface only at UAT.

### Pitfall 8: Multi-multi station change mid-very-narrow flips cluster to row 1 (CR-01 echo)

**What goes wrong:** User switches between two multi-stream stations while the panel is in the very-narrow state (cluster on row 3, picker on row 2). `_populate_stream_picker` clears the threshold cache. On the next reflow, `_row1_min_width(include_picker=True, include_volume_cluster=False)` walks `self.controls` — but the cluster isn't there (it's in row 3) and the picker isn't there (it's in row 2). The walk sees only play/stop/edit/star/eq → threshold drops by ~190px. Predicate misclassifies as wide; cluster flips back to row 1; picker also flips back; user sees an overlap.

**Why it happens:** This is exactly the same shape as the 72.1 CR-01 bug at lines 1305-1312. The fix is the same: make the threshold layout-agnostic by adding the widget's contribution explicitly when it's currently NOT in the target layout (row 1).

**How to avoid:** The generalized `_row1_min_width` MUST preserve the layout-agnostic contract from 72.1 CR-01. Pattern 1 above demonstrates the correct shape: for each widget the caller requests via include_X=True, add its contribution explicitly if it's currently NOT in row 1.

**Warning signs:** Test `test_multi_to_multi_mid_very_narrow_keeps_cluster_on_row_3` (analogous to 72.1's `test_multi_to_multi_mid_narrow_keeps_picker_on_row_2`) fails after a bind_station between two multi-stream stations at narrow widths.

### Pitfall 9: Reparent during Phase 72 peek-overlay open (echo of 72.1 Pitfall 4)

**What goes wrong:** Phase 72's `_open_peek_overlay` (`main_window.py:1060-1090`) reparents `station_panel` out of the splitter. NowPlayingPanel receives a spurious intermediate-width resize event during the transition. The cluster briefly wraps to row 2/3 and then unwraps when the final width is set.

**Why it happens:** Documented in 72-LEARNINGS. The reflow helper is idempotent, so a brief mid-transition wrap followed by unwrap is harmless visually (Wayland coalesces into one frame).

**How to avoid:** Helper is naturally idempotent. No additional mitigation unless real flicker observed (then add hysteresis dead-band — CONTEXT Deferred).

**Warning signs:** Cluster briefly visible on row 2/3 during peek-overlay open. Visually instant on Wayland; likely not observable.

### Pitfall 10: Compact-toggle clicked while in mid-reparent

**What goes wrong:** User clicks the compact-toggle button at the exact instant a resize event fires that reparents the cluster. The `toggled` signal fires after the reparent — but by then the compact-toggle is in a new layout. Does the signal still reach `_on_compact_btn_toggled`?

**Why it happens:** Concern is whether the signal connection is briefly disconnected during reparent.

**How to avoid:** Qt's `QObject::setParent()` (called implicitly by addWidget/removeWidget) does NOT disconnect signal/slot connections [CITED: Qt 6 QObject docs — connections are only severed in `~QObject()` destructor, not during reparent]. The `toggled.connect(self._on_compact_btn_toggled)` from line 565 is invariant across reparents. 72.1 Pattern 2 verified the same for stream_combo's `currentIndexChanged` connection.

**Warning signs:** None — the signal connection is safe by Qt design. Test `test_compact_toggle_signal_survives_reparent` pins the invariant.

### Pitfall 11: Volume slider drag-in-progress when reparent fires

**What goes wrong:** User is dragging the volume slider (mouse pressed, value changing) at the instant a resize event fires that reparents the cluster. Does the drag continue? Does `valueChanged` keep firing?

**Why it happens:** Concern is whether the mouse grab survives the reparent.

**How to avoid:** Qt routes the mouse grab through the same widget regardless of parent. The slider's value continues to update from mouse movement; `valueChanged` and `sliderReleased` continue to fire on the same connections (line 774-775). This is a Qt platform-default behavior, verified by 72.1's signal-survival tests.

**Warning signs:** None — drag-during-reparent is safe by Qt design. Test the invariant if paranoid (extension to the existing 72.1 signal-survival test).

### New `_reflow_volume_cluster` predicate

```python
# Source: derived from CONTEXT D-12..D-14 + Pattern 4 above
#         + now_playing_panel.py:1385-1432 (analogous _reflow_stream_picker_row shape)

def _reflow_volume_cluster(self) -> None:
    """Width-driven reflow of the volume cluster (D-01..D-14 / LAYOUT-04).

    Wraps (volume_slider, compact_mode_toggle_btn) as a unit off row 1 when
    panel.width() < threshold. Target row depends on stream count:
      - single-stream (picker hidden) -> _controls_row2 (D-07)
      - multi-stream  (picker visible) -> _controls_row3 (D-04)

    Defensive guards (Pitfalls 5/6):
      - hasattr volume_slider / compact_mode_toggle_btn / _controls_row3.
      - panel_width <= 0 (spurious zero-size resize during teardown).

    Idempotency (D-14): _move_volume_cluster_to early-returns when the
    cluster is already in the target layout. Spurious or duplicate calls
    are no-ops.

    Trigger is width, NOT mode (D-12): compact-mode state, sidebar
    visibility, and _is_compact are NOT inputs.
    """
    # Defensive guards (Pitfalls 5, 6).
    if not hasattr(self, "volume_slider"):
        return
    if not hasattr(self, "compact_mode_toggle_btn"):
        return
    if not hasattr(self, "_controls_row3"):
        return
    panel_width = self.width()
    if panel_width <= 0:
        return

    # Threshold compute (Pattern 1 — layout-agnostic via include flags).
    # include_picker reflects "would the picker be in row 1 in the wide state?"
    # which is "is the picker visible at all?" Single-stream (hidden) -> False;
    # multi-stream (visible) -> True. Use isHidden() (Pitfall 4 -- 72.1 line 1410-1419).
    include_picker = not self.stream_combo.isHidden()
    threshold = self._row1_min_width(
        include_picker=include_picker,
        include_volume_cluster=False,  # cluster wrap threshold
    )
    should_wrap = panel_width < threshold

    # Target-row selection (D-07 + D-04).
    if not should_wrap:
        target = self.controls
    elif self.stream_combo.isHidden():
        # Single-stream narrow -> row 2 (cluster shares with hidden picker).
        target = self._controls_row2
    else:
        # Multi-stream very narrow -> row 3 (picker has row 2).
        target = self._controls_row3

    self._move_volume_cluster_to(target)
    self._set_volume_size_policy(expanding=should_wrap)
```

### Layout-membership assertion helper for tests (extend from 72.1)

```python
# Source: tests/test_phase72_1_stream_picker_reflow.py:191-217 (existing helper for picker)
# 72.4 extends with a sibling helper for the cluster (both widgets in same target).

def _assert_volume_cluster_in_layout(
    panel: NowPlayingPanel, expected_layout: QHBoxLayout, msg: str = ""
) -> None:
    """Assert BOTH cluster widgets are in `expected_layout` (and ONLY there).

    Cluster-as-unit invariant (CONTEXT D-03): volume_slider and
    compact_mode_toggle_btn must be in the same layout in every state.
    Also asserts the cluster preserves left-to-right order (volume first,
    compact second per CONTEXT D-02).
    """
    # Both widgets in target.
    assert expected_layout.indexOf(panel.volume_slider) >= 0, (
        f"{msg}volume_slider not in expected layout"
    )
    assert expected_layout.indexOf(panel.compact_mode_toggle_btn) >= 0, (
        f"{msg}compact_mode_toggle_btn not in expected layout"
    )
    # Neither in any other layout (anti double-membership).
    others = [
        layout for layout in (panel.controls, panel._controls_row2, panel._controls_row3)
        if layout is not expected_layout
    ]
    for other in others:
        assert other.indexOf(panel.volume_slider) == -1, (
            f"{msg}volume_slider unexpectedly present in another layout"
        )
        assert other.indexOf(panel.compact_mode_toggle_btn) == -1, (
            f"{msg}compact_mode_toggle_btn unexpectedly present in another layout"
        )
    # Left-to-right order (D-02): volume before compact.
    vol_idx = expected_layout.indexOf(panel.volume_slider)
    comp_idx = expected_layout.indexOf(panel.compact_mode_toggle_btn)
    assert vol_idx < comp_idx, (
        f"{msg}cluster order broken — volume must come before compact-toggle "
        f"(volume={vol_idx}, compact={comp_idx})"
    )
```
