## Summary

Phase 60.3 is a surgical fix in **one file** (`musicstreamer/ui_qt/now_playing_panel.py`) wiring an existing data path (`/ajax` → `state["icy_title"]` → `_on_gbs_playlist_ready`) to an existing UI surface (`icy_label` + `_last_icy_title` + `_fetch_cover_art_async` + `star_btn`). All decisions D-01 through D-08 are already locked in `60.3-CONTEXT.md`; this research surfaces only the *implementation knowledge* required to plan the change well.

The plumbing already exists: the `_GbsPollWorker` chain (Plan 60-05), the 15s `_gbs_poll_timer`, the token-discard machinery (`_gbs_poll_token`), the worker-retention slot (`self._gbs_poll_worker`), the GBS-context detection idiom (`is_gbs = ...`), and the cookies-existence login signal (D-04 from Phase 60). The fix amounts to: (a) add a `setText` + source-flag flip in `_on_gbs_playlist_ready`; (b) gate `on_title_changed` writes on the source flag; (c) add a kick-on-tag-change inside `on_title_changed` with worker-in-flight skip. Test fixture extension is minimal — the GBS test patterns at `tests/test_now_playing_panel.py:940-1170` are the playbook; the existing `mock_gbs_api` fixture already returns the right `state["icy_title"]` shape.

**Primary recommendation:** Implement as a **3-task wave** in this order:
1. Introduce the `_gbs_label_source` tri-state flag + lifecycle hooks (D-05/D-07/D-08 plumbing) — pure additive, no behavior change yet.
2. Wire `_on_gbs_playlist_ready` to write `icy_label` and refresh star/cover-art (D-01/D-06/D-07) — visible defect fix.
3. Wire the kick-on-tag-change debounce inside `on_title_changed` (D-03/D-04) and the no-downgrade guard (D-05) — race-condition tightening.

This ordering means each task's tests are an isolatable contract; the tri-state flag tasks land first so subsequent tasks can read it.

### Pattern 1: Tri-state source flag (recommended for D-05/D-07/D-08)

**What:** A simple string attribute on the panel that tracks who last wrote `icy_label`.
**When to use:** Any time `on_title_changed` and `_on_gbs_playlist_ready` would race for write authority.
**Example:**
```python
# Source: project pattern — mirrors existing _gbs_current_entryid (Optional[int]) at now_playing_panel.py:431
# Initialize in __init__ alongside other GBS state (~line 441):
self._gbs_label_source: Optional[str] = None  # 'ajax' | 'icy' | None

# Reset in _refresh_gbs_visibility entry/exit (~line 891 + line 901):
self._gbs_label_source = None

# Flip in _on_gbs_playlist_ready when a non-empty icy_title is in state (~line 977):
self._gbs_label_source = 'ajax'

# Flip in on_title_changed GBS branch (~line 529, after the icy_disabled gate):
if is_gbs and self._gbs_label_source == 'ajax':
    return  # D-05 no-downgrade guard
# ... existing setText/_last_icy_title writes happen here ...
if is_gbs:
    self._gbs_label_source = 'icy'

# Flip in _on_gbs_playlist_error auth_expired branch (~line 1003):
self._gbs_label_source = 'icy'  # D-08 fallback
```

**Why a string enum, not a boolean:** Three meaningful states (`None` = bridge window with /ajax pending, `'icy'` = ICY-owned (pre-/ajax bridge OR auth_expired fallback), `'ajax'` = /ajax-owned). The bridge gating in D-07 reads `_gbs_label_source == 'ajax' OR (logged_out OR auth_expired) == True` — this is exactly three states.

**What:** Extract the icy_label/`_last_icy_title`/source-flag/star/cover-art update from `_on_gbs_playlist_ready` into a helper.

```python
# Source: project pattern — mirrors _apply_vote_buttons_enabled at now_playing_panel.py:1012-1020
def _apply_gbs_icy_label(self, icy_title: str) -> None:
    """Phase 60.3 D-01/D-06/D-07: /ajax stamps the canonical Artist - Title.

    Last-writer-wins (D-02). Coupling point per D-06 — _last_icy_title and
    cover-art lookup string both follow.

        return

    self._gbs_label_source = 'ajax'
```

### Anti-Patterns to Avoid
- **Capturing `self` in a lambda for any new connection.** All new connections must be bound methods (QA-05; Pitfall 10 of 60-RESEARCH).
- **Reading `_current_highlighted_vote()` style "ask the widget what it knows" inferences.** The post-event widget state is unreliable for control logic — use explicit flags. (BLOCKER 1 fix from Plan 60-05 burned this in: see `now_playing_panel.py:1041-1054` comment block.)
- **Restarting the 15s timer on the kick.** Would reset the next-tick delay, double-pace the polling, and diverge from gbs.fm's `DELAY=15000` web UI cadence (D-06a from Phase 60).
- **Comparing label text to derive the source flag** (rejected in CONTEXT D-05 alternative). Two writers (on_title_changed, _on_gbs_playlist_ready) racing on the same QLabel make text-comparison brittle; a flag is cheaper and clearer.
- **Writing `icy_label` from `_on_gbs_playlist_ready` when `state.get("icy_title")` is empty.** The /ajax cold-start response can race the playlist-events stream — empty strings would clobber the bridge ICY value. Always guard `if icy_title:` before stamping.
- **Forgetting the `icy_disabled` gate.** In `on_title_changed`, the existing `icy_disabled` early-return at line 524 must remain *before* any GBS branch. Per CONTEXT Claude's Discretion: also skip /ajax stamping when `icy_disabled=True` (consistency: user has explicitly disabled track display for that station).

## Common Pitfalls

### Pitfall 1: Cross-thread title_changed slot reentrancy

**What goes wrong:** `on_title_changed` is wired to `Player.title_changed` (a Qt Signal that auto-queues from the GstBusLoopThread to the main thread; `player.py:446` comment confirms "auto-queued cross-thread to main"). Adding a kick that calls `_on_gbs_poll_tick()` inside the slot creates a `_GbsPollWorker.start()` call. `QThread.start()` is safe to call from the main thread (which is where the slot is running, since title_changed is queued), so this is fine — but a careless reader might assume the slot runs on the bus-loop thread and try to dispatch defensively.

**Why it happens:** The bus-loop thread + Qt-queued-signal pattern (Phase 43.1 / `qt-glib-bus-threading.md`) makes thread-affinity reasoning subtle.

**How to avoid:** State explicitly in code comment: "on_title_changed runs on the Qt main thread (auto-queued from player.title_changed). Direct method calls to _on_gbs_poll_tick are safe; no QTimer.singleShot needed." Anchor the comment to `player.py:446`.

**Warning signs:** Any sign of a `Signal.emit` or `QTimer.singleShot(0, ...)` appearing in 60.3 diff inside the on_title_changed kick path indicates a reviewer misunderstood thread affinity.

### Pitfall 2: Stale /ajax stamps overwriting a fresh ICY title

**What goes wrong:** Track changes server-side. ICY arrives quickly (~1-2s). `on_title_changed` writes `Title` and kicks `_on_gbs_poll_tick`. But there's an in-flight worker (D-04 skip kicks). The in-flight worker's response carries the **previous** track's `icy_title`. `_on_gbs_playlist_ready` writes that stale `Artist - Title` to `icy_label`, overwriting the fresh-but-bare ICY value.

**Why it happens:** Token-discard machinery (`_gbs_poll_token`) only protects against responses that arrive after a *new* poll has been kicked. The in-flight worker has the current token; its response is "valid" by token but stale by entryid.

**How to avoid:** **Compare entryids.** In `_on_gbs_playlist_ready`, after the existing entryid handling, only write `icy_label` from `state["icy_title"]` if the response's `now_playing_entryid` matches what the panel currently believes is the active entryid (i.e., `self._gbs_current_entryid` after the existing update at line 964). On a track transition, the in-flight worker's response will have the *new* entryid (because /ajax is event-stream-based — see CONTEXT specifics line 128: "the in-flight poll *will* see the new entryid since /ajax is event-stream-based"). If by some race it has the *old* entryid, skip the label-stamp; the next poll (kicked by D-03 or scheduled by the 15s timer) will catch up.

**Pragmatic alternative (simpler):** Trust CONTEXT's claim that /ajax always carries the up-to-date metadata when polled, even mid-track-transition. Don't add entryid comparison in 60.3; if a stale-stamp regression surfaces, fix it then. **Recommended for 60.3 plan:** trust /ajax's atomicity (it matches CONTEXT D-02's "Last-writer-wins" model) and skip the entryid-coupling. Document this as a known-acceptable risk in plan-checker comments.

**Warning signs:** Tests that exercise rapid track transitions (within 1-2s of each other) intermittently show the wrong label. If observed during 60.3 verify-work, escalate to entryid-coupling.

### Pitfall 3: `_GbsPollWorker.isRunning()` race window

**What goes wrong:** D-04 reads `self._gbs_poll_worker.isRunning()`. There is a tiny window between `worker.run()` finishing (`playlist_ready.emit` returns) and `worker.isFinished()` becoming true on the consuming side. During that window, `isRunning()` returns False but `_on_gbs_playlist_ready` hasn't fired yet. A kick during that window creates a second worker — duplicate poll, duplicate token bump. Stale-discard catches the older worker's response, so functionally harmless, but extra network work.

**Why it happens:** QThread state machine has a "finished but not yet collected" sub-state.

**How to avoid:** Accept the duplicate-poll risk. The token-discard in `_on_gbs_playlist_ready` (line 941) already discards the older worker's response; the user sees the same final state. The cost is one extra `/ajax` request — acceptable within the rate-limit envelope (Pitfall 5 of 60-RESEARCH: 15s baseline, occasional duplicates are fine; Phase 60 RESEARCH A11 marks rate-limit cadence as MEDIUM-confidence but tolerant).

**Warning signs:** None — silent. Surface only as a "why are there 2 _GbsPollWorker instances at once?" debug log if the project ever adds polling visibility.

### Pitfall 4: `_refresh_gbs_visibility` clearing the source flag too aggressively

**What goes wrong:** `_refresh_gbs_visibility` is called from `bind_station` (line 517). If a user re-binds the **same** GBS station (e.g., clicks the station list entry it's already on), the method runs again, resets `_gbs_label_source` to `None`, and the next `on_title_changed` could kick a redundant poll. Worse, if this re-bind happens between `on_title_changed` (which set flag to `'icy'`) and the next /ajax response, the source flag is lost.

**Why it happens:** Plan 60-05 made `_refresh_gbs_visibility` the single call site for GBS context entry/exit (Phase 64 D-04 invariant). It runs on every bind, not just on actual context transitions.

**How to avoid:** Only reset `_gbs_label_source` inside `_refresh_gbs_visibility` when `should_show` would change the visibility. Read the current `_gbs_playlist_widget.isVisible()` (or `isHidden()` per the existing convention at `test_now_playing_panel.py:1001`) at method entry; reset the flag only on actual transition. Alternatively, accept the redundant reset (since `_on_gbs_playlist_ready` will flip it back to `'ajax'` on the next /ajax response, which is kicked by the same `_refresh_gbs_visibility` path at line 896).

**Recommended:** Accept the redundant reset. The window where the flag is `None` after re-bind is small (one `/ajax` round-trip); the only visible cost is one disabled-then-enabled flip on the star button. Tests should cover the rebind case explicitly.

**Warning signs:** Test `test_refresh_gbs_visibility_runs_once_per_bind_station` (line 1113) provides the spy; extend the same pattern to assert flag behavior on rebind.

### Pitfall 5: Ordering of writes inside on_title_changed when GBS branch is added

**What goes wrong:** The existing `on_title_changed` body (lines 523-549) does, in order: icy_disabled gate → `setText` → `_last_icy_title` write → `_update_star_enabled` → favorites lookup + star icon → cover-art lookup. Adding a "no-downgrade guard" at the top (after icy_disabled gate) and a kick at the bottom is straightforward, but the "set source flag to 'icy'" line must happen **after** the setText (so the post-write state is consistent) and **before** the kick (so the kick's eventual `_on_gbs_playlist_ready` sees the right pre-/ajax state — though it will overwrite `'icy'` to `'ajax'` anyway).

**Why it happens:** Three writers (label, last-icy-title, source-flag) need consistent ordering across slots.

**How to avoid:** Pick this order in the GBS branch of `on_title_changed`:
1. `icy_disabled` gate (existing) — early return.
2. **NEW:** `is_gbs = ...` derivation (or read once and store).
3. **NEW:** No-downgrade guard: `if is_gbs and self._gbs_label_source == 'ajax': return`.
4. `setText` + `_last_icy_title` write (existing).
5. **NEW:** `if is_gbs: self._gbs_label_source = 'icy'`.
6. `_update_star_enabled` (existing).
7. Favorites lookup + star icon (existing).
8. Cover-art lookup (existing) — **subject to D-07 gating per source flag** (see Pitfall 6).
9. **NEW:** `if is_gbs and not self._gbs_poll_in_flight(): self._on_gbs_poll_tick()`.

**Warning signs:** Test that emits `on_title_changed` then immediately asserts on `_gbs_label_source` and `_last_icy_title` will catch ordering mistakes.

### Pitfall 6: D-07 gating — when to suppress cover-art and disable star during the bridge window

**What goes wrong:** D-07 says: during the bridge window (`_gbs_label_source is None`, logged in to GBS, /ajax not yet stamped), star is disabled and `_fetch_cover_art_async` is suppressed. But in the existing code, `_update_star_enabled` reads `bool(self._last_icy_title)` (line 624) — which becomes `True` as soon as `on_title_changed` writes the bare ICY title. So the star will *briefly* become enabled when `on_title_changed` fires before /ajax responds — a flicker.

**Why it happens:** Two coupling points (`_last_icy_title` and `_gbs_label_source`) need to agree.

**How to avoid:** Update `_update_star_enabled` to read both:
```python
def _update_star_enabled(self) -> None:

    # Phase 60.3 D-07: during bridge window in GBS context, star is disabled
    # until /ajax stamps the canonical Artist - Title.
    if has_track and self._station.provider_name == "GBS.FM":
        if self._is_gbs_logged_in() and self._gbs_label_source != 'ajax':
            has_track = False
    self.star_btn.setEnabled(has_track)
    if not has_track:
        ...
```

For cover-art suppression, gate the call inside `on_title_changed`:
```python
if (
    title
    and not is_junk_title(title)
    and self._station is not None
    and title != self._last_cover_icy
    # Phase 60.3 D-07: during bridge window in GBS context, suppress
    # cover-art lookup; _on_gbs_playlist_ready will fire the lookup with
    # the full Artist - Title once /ajax stamps.
    and not (
        is_gbs
        and self._is_gbs_logged_in()
        and self._gbs_label_source != 'ajax'

    self._last_cover_icy = title
    self._fetch_cover_art_async(title)
```

**Warning signs:** Star button flickers on first track during a GBS station bind; cover art briefly fetches the bare title before being replaced by the full-string fetch.

### Pitfall 7: Test isolation — `paths._root_override` + cookies file across multiple GBS tests

**What goes wrong:** Existing GBS tests use `monkeypatch.setattr(paths, "_root_override", str(tmp_path))` + create a fake cookies file at `paths.gbs_cookies_path()` to simulate logged-in state. New 60.3 tests for D-08 (auth_expired fallback) need to either delete the cookies file mid-test or rely on `_on_gbs_playlist_error("auth_expired")` to make the panel "act logged out" without actually unlinking. The existing handler hides the widget + stops the timer (line 1001-1003) but does NOT delete the cookies file — the next bind would re-show the widget. For D-08 tests, prefer driving the auth_expired branch directly via the signal handler rather than file manipulation; mirrors the existing `test_gbs_auth_expired_hides_widget_no_toast` (line 1092).

**Why it happens:** The cookies-file-existence gate is a UX shortcut, not the full auth-state machine.

**How to avoid:** New 60.3 D-08 tests should:
1. Set up logged-in state (cookies file present, _refresh_gbs_visibility called).
2. Call `panel._on_gbs_playlist_error(token, "auth_expired")` directly.
3. Assert `_gbs_label_source == 'icy'` (the D-08 flip).
4. Call `panel.on_title_changed("Some Track")` and assert `icy_label.text() == "Some Track"` and `_last_icy_title == "Some Track"` (D-08 fallback: ICY writes win when /ajax is impossible).

**Warning signs:** Tests that try to delete `paths.gbs_cookies_path()` mid-test will hit pytest cleanup issues; the file is created via `os.makedirs(str(tmp_path), exist_ok=True)` + an `open(...).write` — the next test's `tmp_path` is fresh, but within a single test, file manipulation is fragile.
