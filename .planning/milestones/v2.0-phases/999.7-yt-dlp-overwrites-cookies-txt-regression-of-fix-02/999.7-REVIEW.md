# Phase 999.7: Code Review Report

**Reviewed:** 2026-04-24
**Depth:** standard
**Files Reviewed:** 11
**Status:** issues_found

## Summary

Phase 999.7 restores FIX-02 temp-copy protection on the yt-dlp library API path. The implementation is structurally sound: a single helper module (`cookie_utils.py`) centralises both the corruption predicate and the `temp_cookies_copy` contextmanager, and both call sites (`yt_import.scan_playlist` and `Player._youtube_resolve_worker`) use them consistently. Test coverage is strong — `FakeYDLSaveCookies` faithfully reproduces the real `save_cookies()` `__exit__` side effect, and FIX-02-a through FIX-02-e plus NEW-a through NEW-d cover the new behaviour end-to-end.

No critical or security issues found. One warning around cross-thread `Signal` emission semantics that is technically correct under Qt's `AutoConnection` but worth pinning down explicitly. Four info-level items: a docstring/code mismatch, a redundant `os.path.exists` guard, a benign race window between the corruption check and the temp-copy, and a minor concurrency note for rapid back-to-back YouTube resolves.

## Warnings

### WR-01: `cookies_cleared` Signal lacks explicit `QueuedConnection` documentation at emit site

**File:** `musicstreamer/player.py:483-485` (emit) and `musicstreamer/ui_qt/main_window.py:224` (connect)
**Issue:** `Player.cookies_cleared` is emitted from `_youtube_resolve_worker`, which runs on a bare `threading.Thread` daemon (not a `QThread`, see `_play_youtube` line 463-465). The slot `MainWindow.show_toast` lives on the main thread. The connection in `main_window.py:224` uses the implicit `Qt.AutoConnection`, which marshals correctly when the receiver is QObject-affined to the main thread — so the actual behaviour is right.

However, every other cross-thread Signal in this file (`twitch_resolved`, `youtube_resolved`, `youtube_resolution_failed`, `_cancel_timers_requested`, `_error_recovery_requested`) is wired with an *explicit* `Qt.ConnectionType.QueuedConnection` in `Player.__init__` (lines 148-163), and the class header comment at lines 78-83 specifically calls out that bare `QTimer.singleShot` from bus-loop threads is broken. `cookies_cleared` is the lone cross-thread Signal in this class that relies on auto-connection, and it's connected externally rather than internally. This is a quiet inconsistency that future readers (or a future bus-loop port of this worker) could trip on.

The risk is not bug-shaped today — `AutoConnection` resolves to queued because emitter and receiver have different thread affinities. The risk is regression-shaped: if anyone later adds an internal `Player`-side connection to `cookies_cleared` (e.g. for logging) without an explicit queued type, or if the receiver ends up on the same thread as the emitter for some reason, the toast call would land on the worker thread and likely crash on first widget access.

**Fix:** Either (a) wire an explicit dummy connection in `Player.__init__` to anchor the contract, or (b) add a one-line comment at the emit site documenting the cross-thread invariant. Preferred: option (b), no behaviour change:

```python
# musicstreamer/player.py near line 483
# cookies_cleared is emitted from this worker thread; receivers (e.g.
# MainWindow.show_toast) must be on the main thread so Qt.AutoConnection
# resolves to QueuedConnection. Mirrors the youtube_resolved contract above.
self.cookies_cleared.emit(
    "YouTube cookies cleared — re-import via Accounts menu."
)
```

## Info

### IN-01: `is_cookie_file_corrupted` slices first 4 lines, not first 4 *non-blank* lines

**File:** `musicstreamer/cookie_utils.py:34-49`
**Issue:** Docstring says "scanning the first 4 non-blank lines" but the implementation is `head.splitlines()[:4]` which includes blank lines. With the real yt-dlp save format the marker is on line 2, well inside the slice, so this never causes a false negative in practice. It is purely a doc/code mismatch.

**Fix:** Either tighten the code to match the doc, or simplify the doc to match the code. The latter is less risky:

```python
"""...Detects the yt-dlp save header by scanning the first 4 lines of the
file (real yt-dlp save header lives on line 2)..."""
```

### IN-02: Redundant `os.path.exists` guard before `is_cookie_file_corrupted`

**File:** `musicstreamer/yt_import.py:56-57` and `musicstreamer/player.py:480-481`
**Issue:** Both call sites do:

```python
if os.path.exists(canonical) and cookie_utils.is_cookie_file_corrupted(canonical):
```

`is_cookie_file_corrupted` already swallows `OSError` (including missing-file) and returns `False`, per its docstring. The `os.path.exists` short-circuit is therefore redundant. Not a bug — and avoiding the open() call when the file is missing is a microoptimization — but it duplicates the responsibility for "missing file means not corrupted."

**Fix:** Optional tightening — drop the existence check:

```python
if cookie_utils.is_cookie_file_corrupted(canonical):
    constants.clear_cookies()
    ...
```

Leaving it as-is is also fine; the redundancy is defensive.

### IN-03: TOCTOU between corruption check and `temp_cookies_copy`

**File:** `musicstreamer/yt_import.py:55-73` and `musicstreamer/player.py:479-505`
**Issue:** The corruption check + `clear_cookies()` runs, then `temp_cookies_copy()` is entered, which re-checks `os.path.exists`. There is a tiny window where another process or thread could re-create a (still-corrupted) cookies.txt between the two checks. In single-user MusicStreamer (per the project memory) this is essentially impossible; flagging only because the comment "MUST run BEFORE building opts" implies an ordering invariant that's slightly weaker than it sounds — what actually matters is that *nothing in this thread* writes the canonical between the two steps, which is true by construction.

**Fix:** No code change needed. If you want to make the invariant explicit, add a one-line comment:

```python
# Phase 999.7 corruption check — runs BEFORE temp_cookies_copy() so a
# corrupted canonical is wiped before we copy it into the temp file.
# Single-process app ⇒ no TOCTOU risk between the two checks.
canonical = paths.cookies_path()
```

### IN-04: Concurrent YouTube resolves can race on `clear_cookies()`

**File:** `musicstreamer/player.py:467-485`
**Issue:** `_youtube_resolve_worker` runs on a daemon `threading.Thread` (line 463-465). The failover path can call `_play_youtube` on rapid back-to-back stream advances. If two workers are alive simultaneously and both detect a corrupted canonical:

1. Worker A: detects corruption, calls `clear_cookies()` (atomic `os.remove`).
2. Worker B: `os.path.exists(canonical)` returns False (already cleared), skips clear.
3. Both emit `cookies_cleared` — wait, no, only A emits because B's exists check fails. Good.

Actually the failure mode is benign: `clear_cookies()` itself is `os.path.exists` → `os.remove`, which races safely (second remove finds no file and returns False). And `temp_cookies_copy` re-checks `os.path.exists` after the clear, so a second worker that arrives after the canonical was wiped just yields `None` and runs without cookies. No crash, no corruption.

The only observable artifact: in a tight race two toasts could fire ("YouTube cookies cleared..." × 2). Acceptable.

**Fix:** No change needed. Documenting for future debugging if duplicate toasts appear in the wild.

### IN-05: `_youtube_resolve_worker` broad `except Exception` masks programming errors

**File:** `musicstreamer/player.py:511-513`
**Issue:** Pre-existing pattern (not introduced this phase). The broad `except Exception` catches everything yt-dlp throws, but also catches `AttributeError`, `KeyError`, etc. that would indicate a bug in our opts construction. Worth noting because Phase 999.9 already burned cycles on a silent-no-formats failure that this catch-all could similarly hide.

**Fix:** Out of scope for 999.7 — leaving as-is. If a future yt-dlp version changes API surfaces and this worker silently emits `youtube_resolution_failed("'NoneType' object has no attribute 'get'")` flag the catch-all then.
