# Phase 999.7: yt-dlp overwrites cookies.txt — regression of FIX-02 - Context

**Gathered:** 2026-04-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Restore the Phase 23 (v1.5 FIX-02) protection that kept yt-dlp from rewriting the user's imported `~/.local/share/musicstreamer/cookies.txt`. The regression was introduced when Plan 35-06 replaced the mpv subprocess with the `yt_dlp.YoutubeDL` library API and wired `opts["cookiefile"] = paths.cookies_path()` directly, dropping the `shutil.copy2 → temp → pass temp path` wrapper that the subprocess path had.

In scope:
- Wrap yt-dlp library calls in `musicstreamer/player.py::_youtube_resolve_worker` and `musicstreamer/yt_import.py::scan_playlist` so `cookiefile` points at a per-call temp copy, not the canonical path.
- Recover already-corrupted `cookies.txt` files (yt-dlp serialized format) at the read sites.
- Restore / adapt the v1.5 `tests/test_cookies.py` coverage to the library-API call sites.

Out of scope:
- Changing the OAuth import flow (`oauth_helper._GoogleWindow`), the three `cookie_import_dialog` write paths, or `paths.cookies_path()` itself — these all write clean Netscape format and are not the source of the regression.
- Twitch silent-playback issue (Phase 999.8) and any other YouTube playback work.
- Re-opening the mpv-fallback decision (Plan 35-06 stays locked).

</domain>

<decisions>
## Implementation Decisions

### Recovery of already-corrupted cookies.txt
- **D-01:** If `cookies.txt` content starts with the yt-dlp marker line (`# This file is generated by yt-dlp.  Do not edit.` or similar — detect by first non-blank line containing `generated by yt-dlp`), treat it as corrupted.
- **D-02:** On detection: delete the file (auto-clear) and surface a toast via the existing Player / main-window toast path: "YouTube cookies cleared — re-import via Accounts menu." Single-user project, no data-loss concern (the file is already broken). Proceed with the current yt-dlp call without cookies.
- **D-03:** Detection runs at the two yt-dlp read sites — inside `_youtube_resolve_worker` (before building `opts`) and inside `scan_playlist` (before building `opts`). No startup-time check, no duplicate check elsewhere. Triggers only when the user actually exercises YouTube.

### Temp-copy mechanism (Claude's Discretion)
- **D-04:** Use `tempfile.mkstemp(prefix="ms-cookies-", suffix=".txt")` + `os.close(fd)` + `shutil.copy2(paths.cookies_path(), tmp_path)`. Cross-platform (avoids `NamedTemporaryFile`'s Windows open-handle restriction for v2.0 parity). Pass `tmp_path` as `opts["cookiefile"]`. `try/finally` unlink to guarantee cleanup even if `extract_info` raises.
- **D-05:** Factor the copy + cleanup into a single shared helper — recommend a small context manager (e.g., `with _temp_cookies_copy() as cookiefile: opts["cookiefile"] = cookiefile`) in a shared module (`musicstreamer/cookie_utils.py` or extend `musicstreamer/paths.py`) so `player.py` and `yt_import.py` do not duplicate the copy/cleanup boilerplate.

### Copy-failure fallback (Claude's Discretion)
- **D-06:** If `shutil.copy2` raises `OSError` (disk full, permissions), proceed without cookies — do not set `opts["cookiefile"]`. Mirrors the v1.5 Phase 23 pattern. No user-facing error; the yt-dlp call may still succeed for non-age-gated content.

### Test strategy (Claude's Discretion)
- **D-07:** Adapt the v1.5 `tests/test_cookies.py` temp-copy / fallback / original-unchanged tests to the library API. Monkeypatch `yt_dlp.YoutubeDL` to capture the `opts` dict passed in; assert `opts["cookiefile"]` ≠ `paths.cookies_path()` and that the file at `paths.cookies_path()` is byte-identical before/after the call.
- **D-08:** Add a new test for the corruption-detection recovery path — pre-seed `cookies.txt` with the yt-dlp marker, call `scan_playlist` / resolve, assert the file is gone afterward, and the relevant code path fired the toast signal.

### Scope
- **D-09:** Only `player.py::_youtube_resolve_worker` and `yt_import.py::scan_playlist` read cookies.txt for yt-dlp. Both must route through the shared temp-copy helper. No other read sites to fix.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `musicstreamer.paths.cookies_path()` — canonical cookies path, already used by every read/write site.
- `musicstreamer.constants.clear_cookies()` — existing unlink helper; fits the auto-clear branch.
- Player toast path — `Player.playback_error.emit(str)` pattern + main-window toast subscription already exists for surfacing operational messages; the corruption toast can reuse this signal or a dedicated sibling signal. `yt_import.scan_playlist` is called from `ImportDialog`, which has its own toast/status surface.
- v1.5 `tests/test_cookies.py` (in archived phase dir) — test structure for temp-copy + fallback + unchanged-original is the template to port.

### Established Patterns
- Worker-thread + queued Qt signal for yt-dlp calls (see `_youtube_resolve_worker`) — temp-copy setup/cleanup must live inside the worker so cleanup runs on the same thread that owns the temp file path.
- Cross-thread state: `GLib.idle_add` (GTK) was replaced by Qt queued signals (`youtube_resolved`, `youtube_resolution_failed`). A corruption toast from a worker must emit a queued signal, not call UI directly.
- `try/finally` for tempfile cleanup is the project norm.

### Integration Points
- `Player._youtube_resolve_worker` — wrap the `YoutubeDL(opts)` context manager's `opts["cookiefile"]` assignment with the temp-copy helper.
- `yt_import.scan_playlist` — same wrapping; synchronous call from the import worker thread, simpler lifetime.
- New shared helper module (`cookie_utils.py` or extend `paths.py`) — single home for `is_cookie_file_corrupted()` and `temp_cookies_copy()` context manager so both call sites stay one-line changes.
- New/extended toast signal wiring — either a dedicated `Player.cookies_cleared.emit("...")` or reuse the existing error-toast surface.

</code_context>

<specifics>
## Specific Ideas

- Reproduced 2026-04-24 during Phase 999.3 UAT: file now starts with `# This file is generated by yt-dlp.  Do not edit.` instead of `# Netscape HTTP Cookie File`.
- Single-user project — auto-clearing the corrupted file is acceptable because the data is already unusable; the tradeoff is simpler UX vs. preserving broken bytes.
- Windows compatibility (v2.0 milestone) is the reason `mkstemp` + manual unlink is preferred over `NamedTemporaryFile(delete=True)` — Windows cannot reopen a still-open `NamedTemporaryFile`.

</specifics>
