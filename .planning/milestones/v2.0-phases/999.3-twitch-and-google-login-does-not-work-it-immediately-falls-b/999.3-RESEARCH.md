## Summary

**The blocking finding of this research is a token-compatibility issue, not a flow-design issue.** Three independent facts land on the same conclusion:

1. **Twitch does not support PKCE for the Authorization Code flow** (as of Dec 2024, community threads from Twitch maintainers confirm PKCE is still unsupported and on no published roadmap). The locked D-04 approach cannot be implemented as stated. D-06 fallbacks are in play.
2. **streamlink's Twitch plugin uses Twitch's private GraphQL API** (`gql.twitch.tv/gql`), not the public Helix REST API. A token obtained from a newly-registered dev.twitch.tv app (what D-01 proposes) is a **Helix Bearer token** and is **not accepted by the GQL endpoint**. The upstream streamlink maintainer (bastimeyer) states this explicitly: "3rd party websites/tools can't authenticate users on [GQL]."
3. **streamlink already works for public Twitch streams without any token** (plugin falls back to its own hardcoded `kimne78kx3ncx6brgo4mv6wki5h1ko` client-id for anonymous GQL requests). The token only enables subscriber-only streams, VODs, and Turbo ad-suppression — none of which are in MusicStreamer's stated requirements.

**The current code's hidden assumption** — that piggybacking on Twitch's own web client_id + implicit flow would hand back a GQL-compatible session token — was the *only* thing that could have worked, and is apparently what broke (Twitch likely tightened implicit-flow behavior for the web client_id, or the unloadable `http://localhost` redirect never triggered `urlChanged` in the first place).

**Primary recommendation:** Before implementing any of D-01..D-11 as written, the user (Kyle) needs to decide among three viable paths. The planner MUST loop back to the user with this finding before locking a plan. My recommendation, in priority order:

1. **RECOMMENDED — Path Z (descope Twitch auth entirely):** Remove the Twitch token capture flow. Delete `_TwitchWindow` and the AccountsDialog "Connect Twitch" button. streamlink works unauthenticated for the public streams MusicStreamer plays. This eliminates the bug, eliminates 300+ lines of code, eliminates the new loopback-server attack surface, and costs the user only the ability to stream their own subscriber-only / VOD content (which has never been a stated requirement). **Lowest effort, lowest risk, eliminates the problem class.**
2. **Path Y (device code flow with proper app registration, documented as "for ad-suppression only"):** Implement Twitch Device Code Flow (DCF — generally available, public-client-compatible, no client_secret, no loopback server needed, no QWebEngineView needed at all). User gets a short code, visits twitch.tv/activate in their own browser, enters the code. **But the resulting Bearer token still doesn't work with streamlink's GQL calls**, so this path only makes sense if we *also* rewrite `_play_twitch` to use Helix (giving up streamlink's stream resolution entirely — very large change, likely infeasible within v1.5 scope).
3. **Path X (keep implicit flow + Twitch web client_id, rebuild the capture mechanism properly):** This preserves the one token type that actually works with streamlink. Switch the capture from fragile `urlChanged` to an HTML bounce-page served from a local loopback listener that parses the URL fragment in JS and POSTs it to the listener. This is what D-06(b) describes. Keeps all existing functionality and fixes the bug. Violates no technical constraint, but retains the "piggyback" pattern the user explicitly rejected in the discussion phase.

**The planner cannot proceed to PLAN.md until the user chooses a path** because each path produces a radically different plan: Path Z is ~5 tasks of deletion; Path Y is a 2-week refactor; Path X is a focused bug-fix close to the original CONTEXT.md shape but with a fundamentally different client_id.

### Pattern 1: PKCE (RFC 7636) — stdlib-only

```python
# [CITED: RFC 7636 §4.1-4.2]
import base64
import hashlib
import secrets

def _pkce_pair() -> tuple[str, str]:
    """Return (code_verifier, code_challenge).

    Verifier is 43-128 chars from unreserved alphabet (RFC 7636 §4.1).
    secrets.token_urlsafe(32) produces 43 chars from the URL-safe alphabet.
    Challenge is base64url(sha256(verifier)) without padding (RFC 7636 §4.2).
    """
    verifier = secrets.token_urlsafe(32)
    digest = hashlib.sha256(verifier.encode("ascii")).digest()
    challenge = base64.urlsafe_b64encode(digest).rstrip(b"=").decode("ascii")
    return verifier, challenge
```

### Pattern 2: Loopback server thread-safe signaling

```python
# [CITED: Qt Forum "using pyqtSignal in BaseHTTPRequestsHandler"]
# Key insight: emitting a Qt Signal from a BaseHTTPRequestHandler thread IS
# thread-safe when the connected slot is Qt.QueuedConnection. The signal
# crosses the thread boundary via Qt's event queue.
#
# Simpler alternative (preferred — no QObject parent needed for the handler):
# use a threading.Event + a mutable container, polled on the main thread by QTimer.

import threading
from http.server import BaseHTTPRequestHandler, HTTPServer
from urllib.parse import urlparse, parse_qs

class _RedirectHandler(BaseHTTPRequestHandler):
    server: "_OAuthServer"  # type: ignore[assignment]

    def do_GET(self):  # noqa: N802
        parsed = urlparse(self.path)
        params = parse_qs(parsed.query)
        self.server.result = {
            "code": params.get("code", [None])[0],
            "state": params.get("state", [None])[0],
            "error": params.get("error", [None])[0],
        }
        self.server.done.set()
        self.send_response(200)
        self.send_header("Content-Type", "text/html; charset=utf-8")
        self.end_headers()
        self.wfile.write(
            b"<html><body><p>Login complete. You can close this window.</p></body></html>"
        )

    def log_message(self, *args, **kwargs):  # silence default stderr access log
        return

class _OAuthServer(HTTPServer):
    def __init__(self, port: int):
        super().__init__(("127.0.0.1", port), _RedirectHandler)  # bind 127.0.0.1 ONLY
        self.result: dict | None = None
        self.done = threading.Event()
```

### Anti-Patterns to Avoid

- **Binding loopback server to `0.0.0.0` or `localhost`.** RFC 8252 §8.3 requires 127.0.0.1 (IPv4) or ::1 (IPv6) to avoid listening on non-loopback interfaces. `localhost` may resolve to either and is NOT RECOMMENDED by RFC 8252. **Bind explicitly to `"127.0.0.1"`.**
- **Keeping the loopback server alive after the redirect.** RFC 8252 §8.3: "close it once the response is returned." Call `HTTPServer.shutdown()` from the main thread after `server.done.is_set()`.
- **Logging the redirect URL or any query params.** D-10 prohibits it. Scrub to just `{category, detail_sanitized}` where detail is either an exception class name, HTTP status code, or a short error message — NEVER a URL, NEVER the token, NEVER the PKCE verifier.
- **Writing the log file from the subprocess AND the parent.** Single-writer rule. The subprocess emits structured events on stderr; the parent is the sole log-file writer.
- **Relying on `urlChanged` for loopback redirect capture.** This is the current bug. Once the browser hits `http://localhost:PORT`, the page won't load (ERR_CONNECTION_REFUSED if no server, or a legit 200 from our server), but neither case guarantees `urlChanged` fires reliably for the target URL with fragment/query parameters. **The loopback server is the capture mechanism; `urlChanged` is only a fallback for diagnostics.**
- **Hand-rolling log rotation.** Use `logging.handlers.RotatingFileHandler(..., maxBytes=64*1024, backupCount=2)`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| PKCE verifier/challenge | Custom CSPRNG, custom base64-url encoding | `secrets.token_urlsafe`, `hashlib.sha256`, `base64.urlsafe_b64encode` | Standard primitives, correct by construction. |
| Loopback redirect HTTP listener | Raw socket server | `http.server.HTTPServer` + `BaseHTTPRequestHandler` | stdlib, single-request lifecycle, Keep-Alive-free. |
| URL query parsing | String splitting | `urllib.parse.parse_qs` | Handles URL-encoding correctly. |
| Token endpoint POST | Manual socket | `urllib.request.urlopen` with `Request(method="POST", data=urlencoded)` | Already handles redirects, TLS, 4xx errors. |
| Log rotation | Manual truncation | `logging.handlers.RotatingFileHandler` | Handles file locking and rotation atomically. |
| State parameter | Skipping it | `secrets.token_urlsafe(16)` validated on redirect | CSRF protection is required per OAuth 2.1 / RFC 8252 §8.9. |

## Common Pitfalls

### Pitfall 1: "I implemented PKCE and Twitch returned `invalid_request: client_secret is required`"

**What goes wrong:** Token endpoint rejects the exchange because Twitch does not support PKCE.
**Why it happens:** [VERIFIED: Twitch Developer Forums Dec 2024 thread "Still no support for auth code flow with PKCE"] Twitch has not implemented PKCE. D-04 as written is not executable.
**How to avoid:** Do not attempt D-04. Plan MUST pick a D-06 fallback OR a different token strategy altogether.
**Warning signs:** 400 from `https://id.twitch.tv/oauth2/token` with message mentioning `client_secret`.

### Pitfall 2: "The new Twitch token we captured doesn't work with streamlink"

**What goes wrong:** `session.set_plugin_option("twitch", "api-header", [("Authorization", f"OAuth {token}")])` silently has no effect (or 401s), because streamlink calls `gql.twitch.tv/gql` which only accepts session tokens from Twitch's web frontend — NOT Helix tokens from registered apps.
**Why it happens:** [CITED: github.com/streamlink/streamlink/discussions/4400, #6648] Two independent token systems at Twitch. Helix = public API (registered apps, Bearer, scoped). GQL = private web API (session token, `OAuth` prefix, full-account auth). `OAuth <helix_token>` is not a valid GQL auth header.
**How to avoid:** Before locking a plan, verify which streams MusicStreamer actually needs to access. If public-only → no token needed, drop the feature. If subscriber/VOD → must keep web-client_id implicit flow (D-06b).
**Warning signs:** streamlink resolves public streams fine, but subscriber-only streams fail with 403, and the token file appears populated but has no effect.

### Pitfall 3: Thread-safety between HTTPServer handler and Qt event loop

**What goes wrong:** Handler mutates state that the main Qt thread is reading → race or GIL-protected-but-logically-wrong reads.
**Why it happens:** `BaseHTTPRequestHandler.do_GET` runs in the HTTPServer's thread, Qt slots run in the main thread.
**How to avoid:** Use a `threading.Event` + one write from handler, one read from main thread after `.wait()` or `QTimer`-based poll. Or emit a `Signal` from the handler (Qt signals cross threads via QueuedConnection safely). **Never** call QWidget methods from the handler thread.
**Warning signs:** Intermittent crashes, "QObject::setParent: Cannot set parent, new parent is in a different thread."

### Pitfall 4: Port collision / TIME_WAIT

**What goes wrong:** Second login attempt fails with `OSError: [Errno 98] Address already in use` because the previous subprocess's socket is in TIME_WAIT.
**Why it happens:** TCP sockets linger in TIME_WAIT after close. Fixed port + rapid retry → collision.
**How to avoid:** `HTTPServer.allow_reuse_address = True` (sets `SO_REUSEADDR`). RFC 8252 actually prefers *ephemeral* ports (OS picks free port at bind-time, client reads `server.server_address[1]` and includes it in authorize URL + redirect_uri). **Recommend ephemeral port.** [CITED: RFC 8252 §7.3] Twitch's loopback exception allows port to vary per auth request.
**Warning signs:** "Connect Twitch" works the first time, fails the second time until a minute passes.

### Pitfall 5: Twitch registered redirect_uri mismatch

**What goes wrong:** Token endpoint returns `redirect_mismatch` even though the URIs look identical.
**Why it happens:** [CITED: Twitch Dev Forums multiple threads] Twitch requires the redirect_uri in the token-exchange POST to EXACTLY match the one in the authorize URL. Trailing slashes, scheme case, and port all count. For loopback, Twitch reportedly enforces exact port match despite RFC 8252 §8.4 guidance allowing port flex. **Safer to use a fixed port (D-03) and register that exact URL.**
**How to avoid:** Fixed port. Planner picks one (suggest 17823 or 47731 — IANA-unassigned). Document the exact URL Kyle must paste into the Twitch registration form.
**Warning signs:** 400 `invalid_request` with `redirect_mismatch` message.

### Pitfall 6: User closes the login window mid-flow

**What goes wrong:** HTTPServer thread keeps running forever (serve_forever blocks), subprocess hangs, QProcess never emits `finished`.
**Why it happens:** serve_forever() only exits on shutdown(). QMainWindow.closeEvent needs to trigger that shutdown.
**How to avoid:** In `_TwitchWindow.closeEvent`: emit `WindowClosedBeforeLogin` stderr event, call `server.shutdown()`, then `QApplication.exit(1)`.
**Warning signs:** "Connecting..." label in AccountsDialog stays forever after user closes browser window.

### Pitfall 7: GQL client_id drift

**What goes wrong:** Even if we keep the "piggyback" implicit flow (D-06b), Twitch could silently change what `kimne78kx3ncx6brgo4mv6wki5h1ko` returns or rotate the web client_id.
**Why it happens:** It's Twitch's own internal ID, not stable public API. [CITED: github.com/streamlink/streamlink/discussions/4320]
**How to avoid:** This is a risk inherent to the GQL path. streamlink itself hardcodes this value; we'd be sharing its fate. If Twitch rotates it, MusicStreamer and streamlink break together — but that's already true.
**Warning signs:** Sudden Twitch failures affecting MusicStreamer that also affect streamlink CLI users.

# Source: RFC 7636 §4, Twitch docs https://dev.twitch.tv/docs/authentication/getting-tokens-oauth/

### Example 2: Token exchange (Authorization Code — with or without PKCE verifier)

```python
# Source: https://dev.twitch.tv/docs/authentication/getting-tokens-oauth/#authorization-code-grant-flow
import json
from urllib.parse import urlencode
from urllib.request import Request, urlopen
from urllib.error import HTTPError

def exchange_code(client_id: str, code: str, redirect_uri: str,
                  verifier: str | None = None, client_secret: str | None = None) -> dict:
    """Exchange authorization code for access token.

    Pass verifier for PKCE flow (will fail on current Twitch).
    Pass client_secret for classic auth-code flow (works today).
    """
    data = {
        "client_id": client_id,
        "code": code,
        "redirect_uri": redirect_uri,
        "grant_type": "authorization_code",
    }
    if verifier is not None:
        data["code_verifier"] = verifier
    if client_secret is not None:
        data["client_secret"] = client_secret

    req = Request(
        "https://id.twitch.tv/oauth2/token",
        data=urlencode(data).encode("ascii"),
        method="POST",
        headers={"Content-Type": "application/x-www-form-urlencoded"},
    )
    try:
        with urlopen(req, timeout=30) as resp:
            return json.loads(resp.read().decode("utf-8"))
    except HTTPError as exc:
        body = exc.read().decode("utf-8", errors="replace")
        raise RuntimeError(f"token_endpoint_{exc.code}: {body}") from exc
```

### Example 3: Device Code Flow initiation (Path Y alternative)

```python
# Source: https://dev.twitch.tv/docs/authentication/getting-tokens-oauth/#device-code-grant-flow
# No loopback server, no QWebEngineView needed.
import json, time
from urllib.parse import urlencode
from urllib.request import Request, urlopen

def start_device_flow(client_id: str, scopes: list[str]) -> dict:
    req = Request(
        "https://id.twitch.tv/oauth2/device",
        data=urlencode({"client_id": client_id, "scopes": " ".join(scopes)}).encode("ascii"),
        method="POST",
        headers={"Content-Type": "application/x-www-form-urlencoded"},
    )
    with urlopen(req, timeout=30) as resp:
        return json.loads(resp.read())
    # → {"device_code": ..., "user_code": "ABCDEFGH",
    #    "verification_uri": "https://www.twitch.tv/activate?...",
    #    "expires_in": 1800, "interval": 5}

def poll_for_token(client_id: str, device_code: str, scopes: list[str],
                   interval: int, expires_in: int) -> dict:
    deadline = time.monotonic() + expires_in
    while time.monotonic() < deadline:
        time.sleep(interval)
        try:
            req = Request(
                "https://id.twitch.tv/oauth2/token",
                data=urlencode({
                    "client_id": client_id,
                    "scopes": " ".join(scopes),
                    "device_code": device_code,
                    "grant_type": "urn:ietf:params:oauth:grant-type:device_code",
                }).encode("ascii"),
                method="POST",
                headers={"Content-Type": "application/x-www-form-urlencoded"},
            )
            with urlopen(req, timeout=30) as resp:
                return json.loads(resp.read())
        except HTTPError as exc:
            # "authorization_pending" → keep polling. Others → raise.
            body = exc.read().decode("utf-8", errors="replace")
            if "authorization_pending" in body:
                continue
            raise
    raise TimeoutError("device_code_expired")
```
