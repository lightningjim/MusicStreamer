## Summary

Phase 43.1 replaces the `NotImplementedError` stub in `smtc.py` with a real `WindowsMediaKeysBackend`. All scaffolding (factory, base class, signals, art cache, MainWindow wiring) was completed in Phase 41 — this phase is purely the Windows leaf.

The winrt Python projection (pywinrt 3.2.1, June 2025) uses lowercase-dotted module names (`winrt.windows.media.playback`), `add_<event>()` subscription returning tokens, and `remove_<event>(token)` for detachment. Event callbacks run on a winrt threadpool thread; emitting Qt signals from the callback is safe because Qt's queued-connection mechanism dispatches to the main thread (D-02 confirmed valid).

One gap discovered: D-03 specifies `RandomAccessStreamReference.create_from_uri(Uri("file:///..."))` for cover art, but the official WinRT docs confirm `CreateFromUri` only accepts `http`, `https`, `ms-appx`, and `ms-appdata` URI schemes — NOT `file://`. The workaround is `InMemoryRandomAccessStream` + `DataWriter`: read the PNG bytes in Python, write into the in-memory stream, then wrap with `RandomAccessStreamReference.create_from_stream()`. This adds ~5 lines per publish but avoids any async StorageFile API call. Alternatively, omit thumbnail on failure with a logged warning — art is nice-to-have. The planner should pick one approach and treat it as a discrete task.

**Primary recommendation:** Implement the winrt backend as a near-mirror of `mpris2.py`'s structure. Replace the `file://` URI pattern with the `InMemoryRandomAccessStream` approach for thumbnail delivery.

### Pattern 3: Button Pressed Callback

```python
def _on_button_pressed(
    self,
    sender,   # SystemMediaTransportControls — typed for clarity; winrt object
    args,     # SystemMediaTransportControlsButtonPressedEventArgs
) -> None:
    btn = args.button
    if btn == SystemMediaTransportControlsButton.PLAY:
        self.play_pause_requested.emit()
    elif btn == SystemMediaTransportControlsButton.PAUSE:

    elif btn == SystemMediaTransportControlsButton.STOP:
```

### Pattern 5: DisplayUpdater / Thumbnail

```python
def publish_metadata(
    self,
    station: Station | None,
    title: str,
    cover_pixmap: QPixmap | None,
) -> None:
    # deferred imports — same function-body pattern
    from winrt.windows.media import MediaPlaybackType
    from winrt.windows.storage.streams import (
        InMemoryRandomAccessStream, DataWriter,
        RandomAccessStreamReference,
    )

    du = self._smtc.display_updater
    du.type = MediaPlaybackType.MUSIC

    if station is None:
        du.music_properties.title = ""
        du.music_properties.artist = ""
        du.thumbnail = None
    else:
        du.music_properties.title = title or station.name
        du.music_properties.artist = station.name

        # Thumbnail: write PNG bytes into InMemoryRandomAccessStream
        # (file:// URIs are NOT accepted by RandomAccessStreamReference.create_from_uri
        # — only http/https/ms-appx/ms-appdata are valid URI schemes per MS docs)
        art_path = write_cover_png(cover_pixmap, station.id) if cover_pixmap else None
        if art_path:
            try:
                stream = InMemoryRandomAccessStream()
                writer = DataWriter(stream)
                with open(art_path, "rb") as f:
                    writer.write_bytes(f.read())
                writer.store()   # NOTE: StoreAsync() in WinRT — see pitfall #3
                stream.seek(0)
                thumb_ref = RandomAccessStreamReference.create_from_stream(stream)
                du.thumbnail = thumb_ref
            except Exception as exc:
                _log.warning("SMTC thumbnail write failed: %s", exc)
                du.thumbnail = None
        else:
            du.thumbnail = None

    du.update()
```

[ASSUMED] — `writer.store()` vs `await writer.store_async()`: pywinrt wraps `StoreAsync` as `store_async()`. In a non-asyncio context, the synchronous `.get()` or `.get_results()` may be callable. Planner should confirm or use the thread-safe asyncio path via `asyncio.run()` in a helper. See Pitfall #3.

### Anti-Patterns to Avoid

- **Module-level winrt imports:** Breaks Linux test discovery. All winrt imports inside function/method bodies only.
- **`file://` URIs with `RandomAccessStreamReference.create_from_uri`:** These URIs are not in the supported scheme list (http/https/ms-appx/ms-appdata). Will silently fail or raise at runtime.
- **`+=` event subscription syntax:** pywinrt uses `add_<event>()` / `remove_<event>()` — not the C# `+=` operator. Using `+=` won't work in Python.
- **Calling `set_state(PLAYING)` on the MediaPlayer:** Don't use the MediaPlayer's playback — it's a conduit to get SMTC, not the audio engine. Only the SMTC properties and display updater are used.
- **Omitting `command_manager.is_enabled = False`:** Without this, MediaPlayer's automatic command routing intercepts button presses before your `button_pressed` handler gets them. The manual handler will never fire.
- **Not storing the event token:** `remove_button_pressed()` requires the exact token returned by `add_button_pressed()`. Without it, the handler cannot be detached at shutdown, causing callbacks after the object is destroyed.

## Common Pitfalls

### Pitfall 1: `command_manager.is_enabled = False` is MANDATORY

**What goes wrong:** If you skip this step, MediaPlayer's built-in command routing intercepts button presses. Your `button_pressed` handler fires for neither Play nor Pause — the event just doesn't arrive.

**Why it happens:** By default, `MediaPlayer` wires SMTC button events to its own `CommandManager`, which routes them internally. This binding supersedes any `add_button_pressed` handler you attach.

**How to avoid:** Set `self._media_player.command_manager.is_enabled = False` immediately after getting the SMTC reference, before subscribing to `button_pressed`.

**Warning signs:** Handler never called during manual media-key testing on Win11 VM.

[CITED: https://learn.microsoft.com/en-us/windows/apps/develop/media-playback/system-media-transport-controls — "If you are going to manually control the SMTC, you should disable the automatic integration provided by MediaPlayer by setting the CommandManager.IsEnabled property to false."]

### Pitfall 2: `file://` URIs not accepted by `RandomAccessStreamReference.create_from_uri`

**What goes wrong:** D-03 originally specified `RandomAccessStreamReference.create_from_uri(Uri("file:///C:/path/art.png"))`. This will fail — the WinRT docs state the valid URI schemes are `http`, `https`, `ms-appx`, and `ms-appdata` only.

**Why it happens:** `RandomAccessStreamReference.CreateFromUri` is designed for networked or app-package resources, not arbitrary filesystem paths.

**How to avoid:** Use `InMemoryRandomAccessStream` + `DataWriter` to load the PNG bytes, then `RandomAccessStreamReference.create_from_stream(stream)`. Alternatively, accept no thumbnail and clear `du.thumbnail = None` — SMTC will show the app icon as fallback.

**Warning signs:** No thumbnail visible in SMTC overlay, possibly a silent error or exception during construction.

[CITED: https://learn.microsoft.com/en-us/uwp/api/windows.storage.streams.randomaccessstreamreference.createfromuri?view=winrt-22621 — "The valid Uniform Resource Identifier (URI) schemes are http, https, ms-appx, and ms-appdata."]

### Pitfall 3: DataWriter.store_async() in a non-asyncio context

**What goes wrong:** `DataWriter.store_async()` returns an `IAsyncOperation`. Calling `.get_results()` synchronously on the main thread while in an STA context may deadlock (COM requires re-entrant message pump for STA synchronous blocking).

**Why it happens:** Qt's main thread is STA on Windows. Blocking an STA thread on a winrt async operation can deadlock if the operation needs to marshal back to the calling thread.

**How to avoid:** Use a short-lived `asyncio.run()` call for the PNG-to-stream write in a helper function, or run `store_async()` in a `concurrent.futures.ThreadPoolExecutor`. The simplest safe approach: write PNG bytes directly to the in-memory stream in synchronous chunks using `InMemoryRandomAccessStream.write_async()` via asyncio, OR pre-read the PNG bytes with Python's standard `open()` and avoid `DataWriter` entirely — write raw bytes using a direct stream write method if one is available.

**Warning signs:** App freeze on first `publish_metadata()` call; no error message (pure deadlock).

[ASSUMED] — COM STA deadlock risk with blocking winrt async calls is a known issue with pywinrt on GUI threads. Confirmed by the bleak `allow_sta()` pattern and pywinrt's own recommendation to use `call_soon_threadsafe` for asyncio contexts. The exact behavior with MediaPlayer/DataWriter on Qt STA is not empirically verified for this specific API on this machine — planner should add a UAT step to verify thumbnail delivery works without deadlock.

### Pitfall 4: Event token not stored → crash at shutdown

**What goes wrong:** If `add_button_pressed` return value is discarded, `shutdown()` cannot call `remove_button_pressed()`. The callback remains registered and may fire after the `WindowsMediaKeysBackend` is destroyed, crashing with a use-after-free or null-pointer error.

**How to avoid:** `self._bp_token = self._smtc.add_button_pressed(self._on_button_pressed)` — always store the token.

**Warning signs:** Crash or `RuntimeError` at app close on Windows.

[VERIFIED: pywinrt projection readme — "All event handlers must be removed before Python runtime shutdown to prevent crashes from callbacks after Python exits."]

### Pitfall 5: SMTC state leaked after crash

**What goes wrong:** If the process exits without calling `shutdown()`, the SMTC session name (MusicStreamer) may remain visible in the Windows media overlay until the OS garbage-collects the COM object. This is cosmetic and recovers automatically within 5-30 seconds.

**Why it happens:** COM reference counting — the MediaPlayer COM object lives until released or GC'd.

**How to avoid:** No action needed in normal shutdown. For abnormal termination, Windows handles recovery automatically. Not a blocking issue.

### Pitfall 6: winrt-runtime not installed

**What goes wrong:** `winrt-Windows.Media.Playback` has an undeclared runtime dependency on `winrt-runtime` (the C extension providing the Python/WinRT bridge). Without it, importing any `winrt.windows.*` package raises `ImportError` or `ModuleNotFoundError`.

**How to avoid:** `winrt-runtime` is pulled as a transitive dependency by any `winrt-Windows.*` package — `pip install -e ".[windows]"` installs it automatically. No explicit listing needed, but optionally add to `pyproject.toml` for clarity.

[VERIFIED: pywinrt text_to_speech sample requires `winrt-runtime` in its inline script dependencies alongside `winrt-Windows.Media.Playback`]

### Thumbnail via InMemoryRandomAccessStream (D-03 workaround)

```python
# Source: [ASSUMED] — pattern synthesized from WinRT API docs
# file:// URIs are NOT accepted by create_from_uri; use in-memory stream instead
from winrt.windows.storage.streams import (
    InMemoryRandomAccessStream, DataWriter, RandomAccessStreamReference
)
import asyncio

async def _png_to_stream_ref(path: str):
    stream = InMemoryRandomAccessStream()
    writer = DataWriter(stream)
    with open(path, "rb") as f:
        writer.write_bytes(f.read())
    await writer.store_async()
    stream.seek(0)
    return RandomAccessStreamReference.create_from_stream(stream)

# Call from non-asyncio context:
thumb_ref = asyncio.run(_png_to_stream_ref(art_path))
```

**Planner note:** The asyncio.run() approach creates a fresh event loop for each `publish_metadata()` call. This is safe but potentially slow for frequent calls. Since MusicStreamer ICY updates arrive every 2-5 minutes, this is acceptable. If profiling shows overhead, switch to a dedicated asyncio thread (one persistent loop, pass coroutines via `asyncio.run_coroutine_threadsafe`).

```python
def shutdown(self) -> None:
```

## Open Questions

1. **DataWriter sync vs async in non-asyncio context**
   - What we know: pywinrt wraps WinRT `StoreAsync` as `store_async()` returning an `IAsyncOperation`. The projection readme recommends asyncio for async operations.
   - What's unclear: Whether `asyncio.run()` inside a Qt main thread raises ("cannot nest event loops") or works because Qt doesn't use asyncio.
   - Recommendation: Test both approaches in Wave 0 implementation. If `asyncio.run()` works (Qt doesn't run an asyncio loop), use it. If it conflicts, use `concurrent.futures.ThreadPoolExecutor` with `loop.run_in_executor()` or just read PNG bytes synchronously and find a synchronous write path on `InMemoryRandomAccessStream`.

2. **Enum member casing verification**
   - What we know: pywinrt uses UPPER_SNAKE_CASE for enum members based on naming convention docs.
   - What's unclear: Whether `SystemMediaTransportControlsButton` follows this exactly or uses a different convention.
   - Recommendation: Add a tiny `python -c "from winrt.windows.media import SystemMediaTransportControlsButton; print(list(SystemMediaTransportControlsButton))"` diagnostic to Wave 1 UAT checklist.

3. **Thumbnail clearing (`du.thumbnail = None`)**
   - What we know: D-08 specifies setting thumbnail; clearing is not explicitly documented.
   - What's unclear: Whether `None` is valid or if a no-op sentinel is needed.
   - Recommendation: Try `du.thumbnail = None`; if it raises, omit the assignment when no art is available (leave stale thumbnail from previous station).
