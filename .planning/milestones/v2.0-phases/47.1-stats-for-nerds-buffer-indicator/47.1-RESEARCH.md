## Summary

Phase 47.1 wires GStreamer's `message::buffering` onto the existing `playbin3` bus in `player.py`, emits a Qt `Signal(int)` carrying the percent (0–100), and surfaces it in `NowPlayingPanel` as a mini `QProgressBar` + `QLabel` inside a `QFormLayout`-backed stats widget gated by a checkable hamburger-menu `QAction`. All three integration points (player.py, now_playing_panel.py, main_window.py) follow patterns already established in the codebase for bus handlers, settings persistence, and menu wiring — this is additive, low-risk, and has no feature-flag ambiguity.

**Primary recommendation:** Mirror existing `_on_gst_tag` verbatim for the new `_on_gst_buffering` handler (bus-loop thread → Qt signal emission, which auto-queues to the main thread). Mirror `volume` for `show_stats_for_nerds` persistence (read at panel construction, write on change). Mirror `now_playing_panel.bind_station`'s layout pattern for adding the stats widget as the last item in the center `QVBoxLayout`. No new dependencies, no novel patterns.

## 2. GStreamer Buffering — API Surface and Threading

### `Gst.Message.parse_buffering()` signature

`[CITED: https://lazka.github.io/pgi-docs/Gst-1.0/classes/Message.html]` formally documents the method as returning a tuple `(percent: int,)`. **In PyGObject actual call sites**, single-out-parameter methods are flattened — `parse_buffering()` returns a bare `int`. Confirm at implementation time; if the tuple form is returned, unpack `percent, = msg.parse_buffering()`. Range is 0–100.

**Reference handler pattern (mirrors `_on_gst_error` + `_on_gst_tag`):**

```python
def _on_gst_buffering(self, bus, msg) -> None:
    """Bus-loop-thread handler: parse buffer percent and emit Qt signal.
    Runs on GstBusLoopThread (not main thread). Must only emit signals,
    never touch Qt widgets directly. Emission is auto-queued cross-thread.
    De-dups on unchanged percent (D-14) to avoid UI churn."""
    percent = msg.parse_buffering()  # int, 0-100 (PyGObject flattens single-out-param)
    if percent == self._last_buffer_percent:
        return
    self._last_buffer_percent = percent
    self.buffer_percent.emit(percent)
```

### Emission frequency / thresholds

- `[CITED: gstreamer.freedesktop.org queue2 docs]` queue2 (embedded in `playbin3`) posts `BUFFERING` messages **event-driven on low-watermark / high-watermark crossings**, not at fixed time intervals. Defaults depend on queue2 internals; for `playbin3` with network sources, the queue typically emits frequent updates during initial fill (0→100%) and again if fill drops below the low watermark during playback.
- **Practical implication for de-dup (D-14):** most emissions will carry distinct percents during fill; redundant emissions are rare but possible (the hysteresis logic can re-post 100% after fill completes). The de-dup guard is cheap insurance.
- `[ASSUMED]` No QTimer coalescing needed — bus emission volume on playbin3 is modest (tens of messages, not thousands, per play session). If UAT shows lag, revisit.

### Threading contract

The existing bus architecture is already fully set up for this:

- `bus.add_signal_watch()` dispatches handlers on the `GstBusLoopThread` daemon (player.py:34, 90). `[VERIFIED: player.py docstring + _on_gst_tag/_on_gst_error comments]`.
- Qt `Signal.emit()` across threads is auto-queued — PySide6 marshals the slot call to the receiving object's affinity thread. `[VERIFIED: player.py:301 comment "auto-queued cross-thread to main", and pitfall-2 comments at lines 93-95]`.
- **Never call `QWidget.setValue` or `QLabel.setText` from a bus handler** — always emit the signal and let the slot (on the main thread) do the widget mutation. This is the canonical pattern; `_on_gst_tag` follows it exactly.

**Confidence:** HIGH for threading. MEDIUM for `parse_buffering` return type (PyGObject flattening convention is well-established but the formal binding is a tuple; test double should handle both forms — see §4).

## 3. Qt Patterns — QProgressBar + QFormLayout + Checkable QAction

### QProgressBar

`[CITED: Qt for Python PySide6 docs]`

- `bar = QProgressBar(parent)`
- `bar.setRange(0, 100)` → 0..100 scale
- `bar.setTextVisible(False)` → suppresses Qt's built-in `{N}%` overlay so the separate `QLabel` is the sole percent display (D-01)
- `bar.setValue(int)` → cheap, repaints only on change
- Fixed width: `bar.setFixedWidth(120)` (D-02)
- **No stylesheet needed** for default chunk rendering — Fusion style renders a clean horizontal progress bar out of the box.

### QFormLayout

`[CITED: Qt docs + grepped project usage: edit_station_dialog.py:242, import_dialog.py:246]`

- `form = QFormLayout()` on a container `QWidget`, then `container.setLayout(form)` (or `QFormLayout(container)`).
- Row API accepts either `addRow(str_label, QWidget)` **or** `addRow(QLabel, QWidget)` — the explicit `QLabel` form is preferred here because we need to apply the muted palette to the row label (D-10, §6). String form auto-constructs a `QLabel` with default palette, which we can't restyle cleanly.
- For the value side, wrap `QProgressBar + QLabel` in a single `QHBoxLayout` inside a throwaway `QWidget` to hand `addRow` one widget (avoids breaking the form's label→value grid).

**Pattern:**

```python
stats_wrapper = QWidget(self)
form = QFormLayout(stats_wrapper)
form.setContentsMargins(0, 0, 0, 0)  # align with existing center rows

buffer_label = QLabel("Buffer", stats_wrapper)
# Apply muted palette here (§6)

value_row = QWidget(stats_wrapper)
value_layout = QHBoxLayout(value_row)
value_layout.setContentsMargins(0, 0, 0, 0)
value_layout.setSpacing(6)

self.buffer_bar = QProgressBar(value_row)
self.buffer_bar.setRange(0, 100)
self.buffer_bar.setTextVisible(False)
self.buffer_bar.setFixedWidth(120)
self.buffer_pct_label = QLabel("0%", value_row)
value_layout.addWidget(self.buffer_bar)
value_layout.addWidget(self.buffer_pct_label)
value_layout.addStretch(1)

form.addRow(buffer_label, value_row)

self._stats_widget = stats_wrapper
center.addWidget(self._stats_widget)
```

### Checkable QAction in menu

`[CITED: Qt QAction docs + grepped project usage: now_playing_panel.py:210 setCheckable(True) on QToolButton, settings_import_dialog.py:139 toggled.connect]`

**Note:** No existing `QAction.setCheckable(True)` in `main_window.py` — hamburger menu currently uses non-checkable actions via `menu.addAction("Text")`. The API is identical:

```python
self._act_stats = self._menu.addAction("Stats for Nerds")
self._act_stats.setCheckable(True)
self._act_stats.setChecked(initial_state_bool)  # no signal emitted by setChecked
self._act_stats.toggled.connect(self._on_stats_toggled)  # slot receives bool
```

- `setChecked(...)` does **not** emit `toggled` (safe to call during construction).
- `toggled.connect(slot)` — slot signature is `slot(checked: bool)`.
- Prefer `toggled` over `triggered` for checkable actions — `triggered` fires for both clicks and programmatic changes; `toggled` fires only on state change.

**Confidence:** HIGH — pattern confirmed against PySide6 docs and the existing `settings_import_dialog.py:139` `toggled.connect` usage.

## 5. Settings-Key Contract

### `Repo.get_setting(key, default) -> str`

`[VERIFIED: musicstreamer/repo.py:348-352]`

```python
def get_setting(self, key: str, default: str) -> str:
    row = self.con.execute("SELECT value FROM settings WHERE key = ?", (key,)).fetchone()
    return row["value"] if row else default
```

- **Return type:** `str`, always. Default must be a `str`.
- **Missing-key behavior:** returns `default` verbatim — no type coercion.
- **Call signature:** `self._repo.get_setting("show_stats_for_nerds", "0")` returns `"0"` on fresh install (D-04, D-05).

### `Repo.set_setting(key, value)`

`[VERIFIED: musicstreamer/repo.py:354-359]`

```python
def set_setting(self, key: str, value: str):
    self.con.execute("INSERT OR REPLACE INTO settings(key, value) VALUES (?, ?)", (key, value))
    self.con.commit()
```

- Value must be a `str` (SQLite will coerce, but the type hint is `str`).
- Commits immediately — no batching.

### Conversion pattern for `"0"`/`"1"` bools

**No existing precedent in `musicstreamer/` for `"0"`/`"1"` settings** (`[VERIFIED: grep "set_setting\(.*\"0\"|set_setting\(.*\"1\""` returns no matches). The `icy_disabled` pattern uses `int(bool)` against a dedicated station column, not the settings key/value table.

Recommended idiom (matches D-04 precisely):

```python
# Read (panel construction, MainWindow construction):
show = self._repo.get_setting("show_stats_for_nerds", "0") == "1"

# Write (QAction.toggled slot):
self._repo.set_setting("show_stats_for_nerds", "1" if checked else "0")
```

**Do NOT use `int(...)` or `bool(...)` coercion** — `bool("0")` is `True` in Python (non-empty string), which is a classic footgun. Strict string equality is safe and matches what `volume` / `accent_color` do for their own value spaces.

**Confidence:** HIGH.

## 6. Theme Token Lookup — Muted Label Color (D-10)

### Finding

`[VERIFIED: read of musicstreamer/ui_qt/_theme.py and full grep of _theme.py usage]`

**Phase 46 `_theme.py` does NOT define a muted/secondary-text color token.** Current tokens are:

- `ERROR_COLOR_HEX: str = "#c0392b"` — destructive/error foreground (red)
- `ERROR_COLOR_QCOLOR: QColor = QColor("#c0392b")`
- `STATION_ICON_SIZE: int = 32`

That's the full public surface. No muted/secondary token exists.

### Recommendation

Two options for the planner — **option A preferred:**

**Option A (preferred, minimal):** Use Qt's built-in `QPalette.Disabled` role for the row-label color without adding a new theme token. Qt's Fusion style (enforced on Windows per PORT-07) and native Linux styles both render `Disabled` text in a muted gray that respects the active palette — it's the Qt-idiomatic "secondary text" surface.

```python
from PySide6.QtGui import QPalette
row_label = QLabel("Buffer")
palette = row_label.palette()
muted = palette.color(QPalette.Disabled, QPalette.WindowText)
palette.setColor(QPalette.WindowText, muted)
row_label.setPalette(palette)
```

This does NOT disable the label (it's read-only display; disabling is semantically wrong), but lifts the muted color from the palette and applies it to the normal role. Works with any accent color (UI-11) and dark mode (PORT-07) because it reads the live palette.

**Option B (defer-the-token):** Add `MUTED_TEXT_QCOLOR: QColor = ... ` to `_theme.py` as part of this phase. This matches Phase 46's pattern but expands scope beyond the locked decisions. Only pursue if planner confirms the user wants a concrete hex. The `palette().color(QPalette.Disabled, ...)` form is self-tuning against dark mode; a hardcoded hex is not.

**Recommendation for planner:** Go with Option A — it's what D-10 implicitly describes ("palette().color(QPalette.Disabled, ...) or whatever the existing secondary-text token is") and it matches Phase 46's intent without expanding theme token scope.

**Confidence:** HIGH that no token exists; MEDIUM on which approach D-10 intended (planner should lock in `/gsd-discuss-phase` if unclear — currently researcher's best read of D-10 is Option A).

## 8. Pitfalls / Gotchas

### Pitfall 1: `parse_buffering()` return type (tuple vs int)
**What goes wrong:** Formal PyGObject binding says `(percent,)` tuple; most real-world PyGObject releases flatten single-out-params to bare scalars. If the handler writes `percent = msg.parse_buffering()` and the binding returns a tuple, the Signal will emit a tuple, and `QProgressBar.setValue(tuple)` throws.
**Why it happens:** PyGObject documentation is inconsistent; projects test against whatever version they're compiled against.
**How to avoid:** Defensive parse:
```python
result = msg.parse_buffering()
percent = result[0] if isinstance(result, tuple) else int(result)
```
OR write the test double to return both forms and assert the handler handles both.
**Warning signs:** `TypeError: setValue expected int`, or percent emitted as a one-element tuple.

### Pitfall 2: Cross-thread widget mutation
**What goes wrong:** Calling `self.buffer_bar.setValue(...)` directly from `_on_gst_buffering` would touch a Qt widget from the bus-loop thread, which is undefined behavior (crashes, silent data corruption, Heisenbugs).
**Why it happens:** Bus handlers run on `GstBusLoopThread` (daemon); widgets are affined to the main thread.
**How to avoid:** Handler emits Qt Signal only. Signal.emit is thread-safe; queued-connection slot dispatch marshals back to main. `_on_gst_tag` at player.py:291-301 is the reference.
**Warning signs:** Sporadic crashes, `QBackingStore::endPaint()` warnings, assertions like "Cannot send events to objects owned by a different thread".

### Pitfall 3: De-dup reset on new URL
**What goes wrong:** User plays Stream A that hits 100% buffer, then picks Stream B whose first message is also 100%. Without a reset, the Signal never emits for Stream B because `_last_buffer_percent == 100`. UI freezes at Stream A's last value.
**Why it happens:** De-dup persists across plays.
**How to avoid:** Reset `self._last_buffer_percent = -1` inside `_try_next_stream()` (line ~340, right after `self._current_stream = stream`). Sentinel `-1` guarantees first real 0-100 emits.
**Warning signs:** Progress bar stuck at previous stream's value after station change.

### Pitfall 4: `setVisible(False)` vs row-level hide in QFormLayout
**What goes wrong:** If the planner tries to hide individual `QFormLayout` **rows** (e.g. via `form.rowWrapper(i).setVisible(False)` or `form.labelForField(w).setVisible(False)`), Qt's QFormLayout still reserves the vertical space for the row even when both widgets are hidden. Result: visible empty gap.
**Why it happens:** QFormLayout stretch/sizing is set at widget add time; hiding children doesn't release their row geometry.
**How to avoid:** D-07 locks this correctly — hide the **whole stats wrapper `QWidget`** via `self._stats_widget.setVisible(False)`. The wrapper's parent (`center` QVBoxLayout) properly collapses hidden child widgets.
**Warning signs:** Dead vertical space in center column when stats is "hidden".

### Pitfall 5: Default `"0"` bool conversion footgun
**What goes wrong:** Writing `if self._repo.get_setting("show_stats_for_nerds", "0"):` is always truthy because the non-empty string `"0"` evaluates to `True` in Python.
**Why it happens:** String-to-bool coercion in Python is existence-based, not content-based.
**How to avoid:** Explicit string comparison: `setting == "1"`. Covered by the convention in §5.
**Warning signs:** Stats widget visible on fresh install despite default `"0"`.

### Pitfall 6: `isVisible()` returns False on unshown parent
**What goes wrong:** Widget tests assert `stats_widget.isVisible() is True` after construction; fails because the top-level window hasn't been shown.
**Why it happens:** Qt's `isVisible()` reflects the full visibility chain — a widget is "visible" only if all its ancestors are visible AND the top-level is shown.
**How to avoid:** Prefer `not stats_widget.isHidden()` (reflects the explicit flag, not the chain) OR call `panel.show()` in the test before asserting.
**Warning signs:** Tests pass in isolation but fail when the panel is constructed without a shown window.

### Pitfall 7: Menu action insertion position vs separator placement
**What goes wrong:** Planner inserts "Stats for Nerds" between Accounts and the existing separator, so it ends up in the same visual group as Accounts — not a "new group" per D-03.
**Why it happens:** `main_window.py:142` already has `self._menu.addSeparator()` after Accounts. D-03 requires the new action in its OWN group, meaning it should appear AFTER that existing separator, with ANOTHER separator inserted after it (before the current Export/Import group at line 144).
**How to avoid:** Follow the line table in §1 exactly — insert the new action AFTER line 142's existing separator, then add a new `self._menu.addSeparator()` before the existing `self._act_export = self._menu.addAction("Export Settings")` at line 147.
**Warning signs:** Stats toggle appears visually attached to Accounts group.
