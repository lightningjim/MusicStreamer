# Phase 40.1 Plan 05: ICY Disable Toggle Summary

Fix the "Disable ICY" regression by gating `on_title_changed` in the now-playing panel (early return when `station.icy_disabled`) and rebinding the panel after EditStationDialog save so the flag takes effect without restart.

## Root Cause

Per RESEARCH.md, the `Station.icy_disabled` flag **was** being persisted correctly — `repo.update_station()` writes `int(icy_disabled)` and `list_stations()` reads `bool(r["icy_disabled"])`. The regression lived entirely on the consumption side:

1. `Player._on_gst_tag()` fires `title_changed` unconditionally. The Player does not hold a Station reference on the GLib thread, so gating there would require scope creep (passing/caching a Station into the GStreamer callback path).
2. `NowPlayingPanel.on_title_changed()` DID have `self._station` on the Qt main thread, but neither checked `icy_disabled` before overwriting `icy_label` nor before calling `_fetch_cover_art_async()` — which is what triggers iTunes lookups.
3. `MainWindow._sync_now_playing_station()` only reassigned `self.now_playing._station = fresh` after EditStationDialog save. That updated the reference, but did NOT re-run `bind_station()` — so toggling "Disable ICY" in the edit dialog had no visible effect until the user restarted.

## Fix

### Task 1 — Regression tests (commit `1950611`)

Appended four tests to `tests/test_now_playing_panel.py`:

| Test | Assertion |
|------|-----------|
| `test_icy_disabled_suppresses_all` | After `bind_station` with `icy_disabled=True`, `icy_label` shows station name; `on_title_changed(track)` does NOT call `_fetch_cover_art_async` and does NOT overwrite `icy_label`. |
| `test_icy_disabled_suppresses_itunes_call` | `cover_art.fetch_cover_art` is never invoked when `icy_disabled=True`. |
| `test_icy_enabled_still_updates_title` | Control: `icy_disabled=False` still updates label + triggers cover-art fetch. |
| `test_icy_disabled_rebind_takes_effect` | Rebinding with a fresh `icy_disabled=True` Station suppresses subsequent titles (no stale reference). |

Three of four failed against HEAD (the control test passed, as expected).

### Task 2 — Panel gate (commit `291d085`)

`musicstreamer/ui_qt/now_playing_panel.py`:

- `on_title_changed`: early-return at top of the method when `self._station.icy_disabled`. No label overwrite, no cover-art fetch, no star button state change.
- `bind_station`: when the incoming station has `icy_disabled=True`, set `icy_label` to `station.name` instead of `""`. This gives the slot a sensible persistent label rather than an empty field.

`Player` and `cover_art.py` untouched per the plan's Anti-Pattern note — the gate is at the panel, so `fetch_cover_art` is never called in the disabled path.

### Task 3 — MainWindow rebind (commit `b0f4fc7`)

`musicstreamer/ui_qt/main_window.py::_sync_now_playing_station`:

- Replaced raw `self.now_playing._station = fresh` assignment with a guarded `self.now_playing.bind_station(updated_station)` call.
- Guard: only rebind when the edited station's id matches the currently-bound station (don't disturb the panel if the user edited an unrelated station).
- `station_saved` signal signature unchanged (emits no args). Fresh Station is re-read via `self._repo.get_station(station_id)` as before — no signal API change.

## Why Player Was NOT Modified

Per RESEARCH.md anti-pattern: `Player._on_gst_tag` runs on the GLib main-loop thread and has no `Station` reference — only `_current_station_name`. Teaching Player about the per-station flag would require either threading a `Station` object into the GStreamer callback path (Qt/GLib thread boundary hazards) or a Qt→GLib cross-thread flag lookup (adds mutex surface). Neither is justified when the Qt-side slot already has `self._station` in a thread-safe context. Gate in the panel, full stop.
