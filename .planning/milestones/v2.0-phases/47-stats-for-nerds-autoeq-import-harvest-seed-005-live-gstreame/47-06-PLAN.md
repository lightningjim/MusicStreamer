<patterns>
**PLS format reference (inline — no external doc lookup needed):**
Standard Shoutcast/DI.fm PLS files have this shape:
```
[playlist]
numberofentries=2
File1=http://primary.server.com:8000/listen
File2=http://fallback.server.com:8000/listen
Length1=-1
Length2=-1
Version=2
```
The `FileN=` lines (where N starts at 1) are the stream URLs in preference order. All AA premium PLS files have exactly 2 entries. Robust parsing: scan every line, match `FileN=` regex where N is any positive integer, preserve file order (not N order — though they usually coincide).

**Recommended parse:** use `re.match(r"^File(\d+)=(.+)$", line)` on each line; collect `(int(n), url)` pairs; sort by n; extract urls. This handles non-contiguous numbering (rare but legal). For AA (always contiguous), this is equivalent to iterating in file order.

**Fallback behavior (preserve existing):** on HTTP failure or missing File= lines, `_resolve_pls` currently returns the pls_url string itself — keep that, but return it as `[pls_url]` (list form). Callers taking `[0]` still get the pls_url on failure.
</patterns>
