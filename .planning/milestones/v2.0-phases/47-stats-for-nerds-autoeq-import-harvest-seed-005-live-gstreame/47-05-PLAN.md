<objective>
Fix UAT gap 5 (and its side-effect gap 4): add an in-flight guard to `Player._handle_gst_error_recovery` so that multiple GStreamer bus errors emitted for a single failing URL coalesce into exactly ONE failover-queue advance. Root cause is fully diagnosed in `.planning/debug/stream-exhausted-premature.md` — this plan lifts that fix verbatim.

**Symptom:** On an AA station where only the low-quality stream is functional (higher qualities broken), the app reports "Stream exhausted" instead of falling through to the working low stream. Reason: playbin3 emits N cascading bus errors per broken URL (source + demuxer + decoder during pipeline teardown); each error schedules an independent `QTimer.singleShot(0, _handle_gst_error_recovery)` which pops the next stream. For a 3-stream station, 3 cascading errors for the FIRST URL drain all 3 queue entries in milliseconds → spurious "Stream exhausted". This ALSO masks failover-visibility (gap 4): the toast "Stream failed, trying next…" fires but scrolls past in milliseconds instead of the expected BUFFER_DURATION_S window per stream.

**Fix (verbatim from debug session):** Add `self._recovery_in_flight: bool` guard. Set in `_handle_gst_error_recovery` before calling `_try_next_stream`. Clear via `QTimer.singleShot(0, self._clear_recovery_guard)` AFTER `_try_next_stream` returns so same-URL duplicate recoveries no-op but next-URL recoveries still fire. Reset the flag in `play()`, `play_stream()`, `stop()`, `pause()`.

**Thread note:** `_on_gst_error` runs on the bus-loop daemon thread but only SCHEDULES a recovery via `QTimer.singleShot(0, ...)`. The guard is read/written exclusively inside `_handle_gst_error_recovery`, `_clear_recovery_guard`, and the play/stop/pause reset sites — all main-thread. No cross-thread race.

Purpose: Close UAT gap 5 (severity: major — highest-value gap, core failover feature doesn't work end-to-end without it). Side-effect fix for gap 4 (failover toast becomes visible again because cascading errors no longer drain the queue in milliseconds — each stream gets its full BUFFER_DURATION_S window).

Output: ~10 lines added to `player.py` (init + guard + helper + 4 resets); two regression tests in `test_player_failover.py`.

**Deferred-clear via singleShot(0)** (debug analysis rationale): setting `_recovery_in_flight = False` at the end of `_handle_gst_error_recovery` synchronously would race any singleShot callbacks already queued behind us in the main-thread event queue. Scheduling the clear via `QTimer.singleShot(0, self._clear_recovery_guard)` ensures already-queued recovery callbacks from the old URL drain FIRST (and see True), and only then the guard clears for future URL failures.
