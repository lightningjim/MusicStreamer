PySide6 QtDBus pitfalls (from QtDBus docs + community notes):
- Python types don't auto-map to D-Bus `a{sv}`. Use `QDBusArgument` or let QtDBus marshal `dict` → `a{sv}` via properties returning `dict`. Values must be `QVariant`-compatible (str, int, float, bool, list of str).
- `QDBusAbstractAdaptor` subclasses MUST set the class-level attribute `Q_CLASSINFO("D-Bus Interface", "org.mpris.MediaPlayer2")` (via `__classinfos__` / `QObject.Q_CLASSINFO` — in PySide6 this is done via class-level `__dbus_interface__` or by passing `info_map` to the adaptor base; the canonical PySide6 pattern is `@QtCore.ClassInfo(...)`. See https://doc.qt.io/qtforpython-6/PySide6/QtDBus/QDBusAbstractAdaptor.html).
- `org.freedesktop.DBus.Properties` is auto-generated by QtDBus if the adaptor declares `@Property(...)` for each property. `PropertiesChanged` is NOT auto-emitted — you call `QDBusMessage.createSignal()` + `QDBusConnection.send()` manually when a property changes.
- Object registration: `connection.registerObject("/org/mpris/MediaPlayer2", parent_qobject, QDBusConnection.RegisterOption.ExportAdaptors)` — the adaptors are picked up from the parent's children.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| D-Bus session bus → `LinuxMprisBackend` | Any process on the session bus can call our MPRIS methods (Raise/Quit/PlayPause/Stop/Next/Previous) and read properties |
| `QPixmap` (from UI) → PNG file on disk | Pixmap binary written to user cache dir; consumed by external media-session overlay |
| Station model → MPRIS Metadata dict | Station name + ICY title serialized to D-Bus `a{sv}`; consumed by OS overlay which may render it in a system UI |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-41-04 | Tampering | MPRIS slot methods (PlayPause/Stop/Next/Previous) | accept | Any local session-bus client can trigger these, but the worst outcome is a stream toggle — no data exfiltration, no privilege escalation. Single-user desktop app threat model. |
| T-41-05 | Information Disclosure | MPRIS `Metadata` property (xesam:title, xesam:artist, mpris:artUrl) | accept | Station name + ICY title are already visible in the app UI; publishing them to the session bus exposes nothing not already visible to any process with access to the user's X/Wayland session. artUrl points to a file in user cache — same trust zone. |
| T-41-06 | Injection (rich-text/markup) | `xesam:title` = ICY track title from stream | mitigate | ICY titles pass through `_fix_icy_encoding` in player.py (latin-1 → UTF-8). MPRIS consumers render these as plain text. Do NOT pass HTML-escape or markup-parse — the MPRIS spec defines these as plain strings; GNOME/KDE overlays render them literally. Add a unit test that passes `"<script>alert(1)</script>"` and asserts the metadata dict `xesam:title` is exactly that string (no double-escape, no strip). |
| T-41-07 | D-Bus introspection leak | `org.freedesktop.DBus.Introspectable` auto-exposed | accept | QtDBus auto-generates introspection XML for registered adaptors. Contents (interface names, method signatures, property names) match the public MPRIS2 spec and expose no proprietary info. |
| T-41-08 | Denial-of-Service | `publish_metadata` called at high frequency writes PNG to disk each time | mitigate | Per D-05, `title_changed` emissions are rare (every 2-5 min). Per D-04, cover art file is overwritten in place — no file churn. `write_cover_png` only runs when pixmap changes (Plan 03 adds the caller-side guard: track last station.id + pixmap identity). If a pathological ICY stream somehow emits every 100ms, worst case is ~10 PNG writes/sec to a single file — bounded, not unbounded growth. |
| T-41-09 | Path traversal | Cover-art filename built from `station.id` | mitigate | `station.id` is an integer SQLite primary key — not user-controlled text. f-string `f"{station.id}.png"` is safe. Add unit test confirming `write_cover_png` accepts only int station_id (if callers pass string, fail explicit). |
| T-41-10 | Service name squatting | `org.mpris.MediaPlayer2.musicstreamer` registered without `DBUS_NAME_FLAG_REPLACE_EXISTING` | accept | Two concurrent MusicStreamer instances: second `registerService` fails; backend raises; factory returns NoOp per D-06. User-visible effect: second instance has no media keys. This matches 41-CONTEXT.md "Out of scope: Multiple concurrent MPRIS2 instances". |
