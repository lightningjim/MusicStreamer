## Summary

Phase 999.1 is a tightly-scoped Qt UX addition — three files touched, zero new dependencies, zero schema changes. The plumbing is already in place: `repo.create_station()` and `repo.delete_station()` both exist (verified in `musicstreamer/repo.py:253` and `:286`) and `create_station()` currently has **no callers in the codebase** — it is a dormant primitive introduced for exactly this use case. Discovery and Import dialogs persist stations via their own raw SQL paths, so D-03's pre-create/delete-on-reject lifecycle is fully isolated from those flows.

The dialog already overrides `reject()` and `closeEvent()` (for `_shutdown_logo_fetch_worker`, `musicstreamer/ui_qt/edit_station_dialog.py:671-677`). The cleanup-on-reject for `is_new=True` drops cleanly into both of those existing overrides — no new override is needed, just an additional call inside each. `_add_stream_row()` is a ready-made helper for D-05's pre-added blank row (`:443`). `StationListPanel.refresh_model()` already rebuilds the tree for D-07's post-save refresh (`:293`), but there is **no existing select-by-id helper** on the panel — one must be added.

**Primary recommendation:** Add a single constructor kwarg `is_new: bool = False` to `EditStationDialog`; branch only on (a) pre-add one blank row in `_build_ui`/`_populate`, and (b) delete-on-reject inside the existing `reject()` and `closeEvent()` overrides. Add a `select_station(station_id: int)` public method to `StationListPanel` and a `new_station_requested` signal. Wire both menu action and "+" button through a single `MainWindow._on_new_station_clicked` slot.

### Pattern 1: Hamburger menu action registration

**When to use:** Every menu entry in MainWindow.
**Example:**
```python

act_discover.triggered.connect(self._open_discovery_dialog)

Use `QMenu.insertAction(before, action)` OR rebuild the Group 1 block in the correct order. Simpler: since actions are appended in order, just move the new-station creation lines to BEFORE the Discover lines — no `insertAction` needed. QA-05: bound method, no lambda.

### Pattern 2: Dialog launch with signal wiring
**What:** Re-fetch station → construct dialog → connect `station_saved`/`station_deleted` → `exec()`.
**When to use:** Every time `EditStationDialog` is invoked.
**Example:**
```python
# Source: VERIFIED musicstreamer/ui_qt/main_window.py:340-350
def _on_edit_requested(self, station: Station) -> None:
    fresh = self._repo.get_station(station.id)
    if fresh is None:
        return
    dlg = EditStationDialog(fresh, self._player, self._repo, parent=self)
    dlg.station_saved.connect(self._refresh_station_list)
    dlg.station_saved.connect(lambda: self._sync_now_playing_station(fresh.id))
    dlg.station_deleted.connect(self._on_station_deleted)
    dlg.exec()
```
**For this phase:** The new-station slot mirrors this but (a) creates the placeholder first, (b) passes `is_new=True`, (c) connects `station_saved` additionally to a lambda that calls `self.station_panel.select_station(new_id)`. QA-05 caveat: the existing pattern already uses `lambda: self._sync_now_playing_station(fresh.id)` — a self-capturing lambda — so this pattern is accepted for id-capturing signal connections in practice. (Pure `.connect(bound_method)` preferred where possible; bind id via `functools.partial` if strict QA-05 enforced.)

### Pattern 3: Dialog reject/close override
**What:** Override `reject()` and `closeEvent()` to do cleanup before super call.
**When to use:** Dialog owns transient resources that need explicit teardown.
**Example:**
```python
# Source: VERIFIED musicstreamer/ui_qt/edit_station_dialog.py:671-677
def closeEvent(self, event):  # noqa: N802 (Qt override)
    self._shutdown_logo_fetch_worker()
    super().closeEvent(event)

def reject(self) -> None:
    self._shutdown_logo_fetch_worker()
    super().reject()
```
**For this phase:** Add `if self._is_new: self._repo.delete_station(self._station.id)` INSIDE both overrides, BEFORE `_shutdown_logo_fetch_worker()`. Important: `_on_save` (line 683) calls `self.accept()` which does NOT route through `reject()` — so the delete-on-reject branch will never fire after a successful save. But `_on_delete` (line 763) ALSO calls `self.accept()` after `delete_station()` — so deleting a new station via the explicit Delete button is a no-op double-delete risk. Mitigation: `delete_station()` uses `DELETE FROM ... WHERE id = ?` (repo.py:287) — SQLite silently no-ops on missing id, so a second delete is safe. Still worth a flag-flip (`self._is_new = False` after save or delete) to make intent explicit.

### Anti-Patterns to Avoid
- **Branching inside `_on_save` for new vs. edit:** D-03 explicitly avoids this. The placeholder row exists in DB, so `update_station()` works identically — no branch needed in save path.
- **Self-capturing lambda on menu/button `triggered`:** Violates QA-05 (main_window.py:283 explicitly notes "no self-capturing lambdas"). Use bound methods.
- **Delete on reject WITHOUT `is_new` guard:** would orphan-delete the station whenever an existing-station edit is cancelled. Guard is mandatory.
- **Selecting new station via `QTreeView.setCurrentIndex` on the source model:** tree uses a proxy (`StationFilterProxyModel`, station_list_panel.py:251). Must map source → proxy index or the selection won't render. See Pitfall 2.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Station row creation | Raw SQL `INSERT INTO stations` | `repo.create_station()` | Exists at repo.py:253, returns new id, commits transaction |

## Common Pitfalls

### Pitfall 1: Delete-on-reject fires after Save
**What goes wrong:** If `_on_save` calls `self.accept()` and `self.accept()` somehow routes through `reject()`, the just-saved station gets deleted.
**Why it happens:** Qt's `QDialog.accept()` and `reject()` are distinct code paths — `accept()` does NOT invoke `reject()`. [VERIFIED: Qt semantics] But a programmer might accidentally wire `button_box.accepted.connect(self.reject)` instead of `self._on_save`.
**How to avoid:** Keep existing wiring `button_box.accepted.connect(self._on_save)` (line 363). The delete-on-reject branch lives only in `reject()` and `closeEvent()`, both of which fire only when the user discards.
**Warning signs:** A new station disappears from the list immediately after clicking Save.

### Pitfall 2: Programmatic selection uses source index, not proxy
**What goes wrong:** `station_panel.select_station(new_id)` calls `tree.setCurrentIndex(source_idx)` and nothing visually selects — because the tree is bound to `self._proxy`, not `self.model`.
**Why it happens:** `StationListPanel` uses `StationFilterProxyModel` between the view and the source (station_list_panel.py:251-253). Every index going into the view must be mapped through the proxy.
**How to avoid:** `select_station()` must do: (1) find node in source model with matching `station.id`, (2) build source `QModelIndex` for that node, (3) call `proxy.mapFromSource(source_idx)`, (4) `tree.setCurrentIndex(proxy_idx)` and `tree.scrollTo(proxy_idx)`.
**Warning signs:** Test asserts `tree.currentIndex().isValid()` passes but visual selection highlight is absent.
**Design note:** `StationTreeModel` currently exposes only `station_for_index(index) -> Station` (station_tree_model.py:53). Inverse lookup (`index_for_station_id`) does not exist. Simplest path: iterate `tree.model().rowCount()` in the view and compare `.station.id` — or add `index_for_station_id(sid)` helper on `StationTreeModel` for testability.

### Pitfall 3: Placeholder row survives a crash
**What goes wrong:** User clicks "New Station", app crashes before they hit Save OR Discard. Placeholder row with name "New Station" and no streams persists in DB.
**Why it happens:** Crash-safety cleanup isn't part of this phase.
**How to avoid:** D-06 accepts this — name-required is the only validation, so a recovered "New Station" placeholder is not corrupt data, just an undesirable empty shell the user can manually edit or delete. Acceptable tradeoff for scope; an orphan-sweep on startup would be a future enhancement.
**Warning signs:** User reports "why is there a station called 'New Station' with no URL?"
**Decision note:** This is a KNOWN limitation, documented for transparency. Not a blocker.

### Pitfall 4: Discover/Import dialogs open-stale `self._station` reference
**What goes wrong:** Discover Stations dialog runs concurrently with the new-station flow; both write `stations` rows; concurrent delete of the new-station placeholder interferes.
**Why it happens:** It doesn't — Qt dialogs are modal (`exec()` blocks). Only one station CRUD dialog can be open at a time. Discover and Import use separate, non-nested `exec()` calls from the hamburger menu. [VERIFIED: main_window.py:505, :512]
**How to avoid:** Nothing to do — modal enforcement is the guarantee. But be aware this is why modality matters. Do not make `EditStationDialog` non-modal for this phase.
**Warning signs:** Concurrent edit attempts produce visibly stale state.

### Pitfall 5: Provider combo "blank" isn't really blank
**What goes wrong:** Freshly-created station has `provider_id=NULL`, `provider_name=None`. `_populate` sets `provider_combo.setCurrentText(None)` which may produce a stringified "None".
**Why it happens:** Qt's `QComboBox.setCurrentText()` coerces `None` to `""` on PySide6 in most cases, but this is worth verifying in a test.
**How to avoid:** In `_populate`, use `self.provider_combo.setCurrentText(station.provider_name or "")`. Check the existing `_populate` logic — if it already does `or ""`, no change.
**Warning signs:** Test `test_new_station_provider_combo_blank` fails with combo text being "None".

### Example 5: `StationListPanel.select_station(station_id)` (D-07)
```python
# Source: derived from station_tree_model.py:53 + proxy mapping

def select_station(self, station_id: int) -> None:
    """Programmatically select a station in the tree by id (D-07).

    Switches to Stations mode if needed so the user can see their new station.
    """
    # Ensure we're on the Stations page so the selection is visible
    if self._stack.currentIndex() != 0:
        self._on_stations_clicked()

    # Find source index by walking the tree
    for prov_row in range(self.model.rowCount()):
        prov_idx = self.model.index(prov_row, 0)
        for child_row in range(self.model.rowCount(prov_idx)):
            child_idx = self.model.index(child_row, 0, prov_idx)
            station = self.model.station_for_index(child_idx)
            if station is not None and station.id == station_id:
                # CRITICAL (Pitfall 2): map through proxy
                proxy_idx = self._proxy.mapFromSource(child_idx)
                self.tree.setCurrentIndex(proxy_idx)
                self.tree.scrollTo(proxy_idx)
                return
```
