# Phase 999.1: Add "New Station" primary action - Context

**Gathered:** 2026-04-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Provide a direct UX path to create a station from scratch. Currently users can only obtain new stations via **Discover Stations** (AudioAddict/Radio-Browser) or **Import Stations** (YouTube/playlist). This phase adds entry points that open `EditStationDialog` with a blank station model so users can hand-create a station.

**In scope:** Menu entry + "+" button, transient blank-station lifecycle, post-save selection, default field values, minimal validation.
**Out of scope:** Any new fields in `EditStationDialog`, any changes to existing edit/save flow for existing stations, any streaming-source changes.

</domain>

<decisions>
## Implementation Decisions

### Entry Point Placement
- **D-01:** Add "New Station" as a menu action in the existing hamburger menu (`main_window.py` Group 1, above Discover Stations / Import Stations). Menu label: **"New Station"**.
- **D-02:** Also add a small "+" button in `StationListPanel`'s header row, right-aligned (QToolButton). Both entry points call the same handler so there is one code path for launching the dialog.

### New-Station Flow
- **D-03:** Pre-create an empty station row in the DB (via `repo.create_station()`) on menu/button click, then open `EditStationDialog` normally. This keeps `EditStationDialog` untouched — `update_station(id, ...)`, `assets.copy_asset_for_station(id, ...)`, and stream insert/update all continue to work against a real `station.id`. No "new mode" branching inside the dialog.
- **D-04:** On Cancel (button_box rejected) AND on window-close (X / closeEvent / reject()), delete the placeholder row via `repo.delete_station(id)` so empty-shell orphans cannot accumulate. Only applies when the dialog was opened in "new" mode — detect via a flag passed to the dialog (e.g. `is_new=True`) or by comparing station state on open.
- **D-05:** The dialog should open with one blank row pre-added to the streams table so the user can type a URL immediately. Existing "Add" button handles subsequent rows.

### Save Validation
- **D-06:** Keep current validation — only non-empty name is required to save. No stricter rule for new stations; consistent with edit mode. Users may scaffold a station and fill streams/provider/tags later.

### Post-Save Behavior
- **D-07:** After `station_saved` fires for a new station, call `_refresh_station_list()` (existing pattern) and then programmatically select the newly-created station in the tree so it's focused/visible. Do NOT auto-play.
- **D-08:** Provider field opens blank (editable combo). `ensure_provider("")` already returns None. No last-used-provider pre-fill.

### Claude's Discretion
- Exact icon / theme-icon name for the "+" button (Qt standard icon vs. theme icon).
- Internal signature for distinguishing new vs. edit mode (constructor flag, factory classmethod `EditStationDialog.for_new(...)`, or sentinel `station.id == -1`) — planner/executor choose whichever keeps the diff small and the existing test suite green.
- Tree-selection mechanism for D-07 — select-by-id helper if one exists in `StationListPanel`, otherwise a small addition.
- Whether to wire the hamburger action and the "+" button via a shared private slot `_on_new_station_clicked` or via a public helper.

### Folded Todos
None.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`EditStationDialog`** (`ui_qt/edit_station_dialog.py`): accepts a `Station` and handles name/provider/tags/logo/streams/icy/delete. Works unchanged for a pre-created empty station — the core reason D-03 (pre-create row) was chosen.
- **`repo.create_station()` / `repo.delete_station()`**: placeholder-row lifecycle primitives already in place.
- **Hamburger menu pattern** (`main_window.py:119-163`): `self._menu.addAction(label)` → `.triggered.connect(slot)` → slot opens dialog. `_open_discovery_dialog`, `_open_import_dialog`, etc. are the template.
- **`_on_edit_requested` handler** (`main_window.py:340`): canonical signal wiring pattern for dialog launch — re-fetches from DB, connects `station_saved` to `_refresh_station_list`, connects `station_deleted` to `_on_station_deleted`, calls `exec()`.
- **`StationListPanel.edit_requested`** signal: existing channel used by right-click edit — the "+" button lives in the panel but emits a new signal to `main_window`.

### Established Patterns
- **Menu actions are bound slots on `MainWindow`**, not lambdas (QA-05 in prior phases).
- **Dialogs are modal (`exec()`)** and refresh the station list via `station_saved` signal.
- **`_sync_now_playing_station`** is called after save in case the edited station is the one playing — not relevant for a brand-new station, but the post-save wiring should still include the `_refresh_station_list` connection.
- **Add/Remove row buttons** in streams table are already implemented; the "pre-add one row" requirement just calls the existing add-row helper once during dialog setup when `is_new=True`.

### Integration Points
- `MainWindow.__init__` (Group 1 of hamburger menu): insert `New Station` action above `Discover Stations`.
- `StationListPanel` header row: add a `QToolButton` with "+" symbol/icon; emit a new signal (e.g. `new_station_requested`) that `MainWindow` connects to the same slot as the menu action.
- `MainWindow._on_new_station_clicked` (new slot): calls `repo.create_station()` with defaults, fetches fresh Station, launches `EditStationDialog(fresh, ..., is_new=True)`, wires `station_saved` → refresh + select, `station_deleted` → existing `_on_station_deleted` (for explicit delete button), handles reject/close → cleanup delete.
- `EditStationDialog`: add `is_new: bool = False` constructor flag; when True, (a) pre-add a blank streams row, (b) override reject/closeEvent to call `repo.delete_station(id)` before accept.

</code_context>

<specifics>
## Specific Ideas

- The trigger ROADMAP entry calls out that this gap was discovered during Phase 46 UAT (logo-fetch AA-no-key flow required an existing station to reach the dialog). Any test/UAT for this phase should include: open dialog via menu → save name-only → station appears in list → select it → edit again → all expected fields still functional.
- "New Station" wording matches dialog title ("Edit Station") so the parallel is clear.

</specifics>
