# Phase 999.9: YouTube playback broken — yt-dlp "No video formats found" regression - Context

**Gathered:** 2026-04-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Diagnose and fix the YouTube live-stream playback regression surfaced 2026-04-24 during Phase 999.7 UAT. Symptom: `_youtube_resolve_worker` calls `yt_dlp.YoutubeDL.extract_info(URL)` and the extractor returns `ERROR: [youtube] <id>: No video formats found!`, triggering `youtube_resolution_failed` → failover.

Pre-ruled-out causes (do NOT re-investigate):
- **Cookie corruption** — Phase 999.7 byte-equality proof holds: canonical `cookies.txt` sha256 unchanged before/after the failing call. FIX-02 is working.
- **Node missing** — Confirmed `which node` resolves on this install (fnm-managed). The EJS PO-token solver has a runtime to call.
- **yt-dlp out of date** — Already on the latest stable release (2026.03.17).

Open hypotheses (in priority order, set by the lightweight diagnostic):
1. **yt-dlp upstream lag** — YouTube shipped a client-side change after 2026.03.17 that the current default player_client cannot handle.
2. **`extractor_args` API drift** — `youtubepot-jsruntime.remote_components=['ejs:github']` may be a renamed/removed key in the installed yt-dlp version. The EJS PO-token plugin's API has been moving target.
3. **Cookie expiration** (NOT corruption) — Imported Google cookies have a max-age; Google can also rotate session tokens server-side. Different bug class from 999.7.

In scope:
- Reproduce in terminal with `uv run yt-dlp <URL>` and `--extractor-args` variants — capture which player_client (if any) returns formats.
- If a working player_client exists, set it as the default in `_youtube_resolve_worker` opts.
- If NO player_client works → upstream is broken → ship a user-facing toast ("YouTube temporarily broken — try again later") + log lesson for `/schedule` follow-up to revisit when yt-dlp ships a fix.
- Verify the fix in-app against the failing URL (LoFi Girl or whichever station first reproduced).

Out of scope:
- Cookie re-import as a diagnostic step (skipped — trust 999.7's verification; if cookies turn out to be the culprit, re-open this phase).
- Automated regression test for this path (skipped — manual UAT per single-user pragmatic policy).
- Pinning yt-dlp to git HEAD (rejected — git HEAD breaks more often than it fixes).
- Switching the YouTube extraction strategy entirely (e.g., a different library) — out of scope; would be a new phase.

</domain>

### Mitigation strategy (conditional on diagnostic outcome)
- **D-03:** **Working player_client found** → set it as default in `musicstreamer/player.py::_youtube_resolve_worker` opts via `extractor_args={"youtube": {"player_client": ["<client>"]}}`. Preserve the existing `youtubepot-jsruntime.remote_components` config — researcher must verify that key is still honored in yt-dlp 2026.03.17 before declaring the merge.
- **D-04:** **No player_client works** (truly upstream) → minimal user-facing toast on `youtube_resolution_failed` for YouTube live URLs ("YouTube playback is temporarily broken — yt-dlp upstream issue, try again after a yt-dlp update"). Existing failover behavior stays.
- **D-05:** If diagnosis reveals the `youtubepot-jsruntime` extractor_args key was renamed/removed in 2026.03.17 (hypothesis 2), update to the current API per yt-dlp source/docs at probe time. Treat this as the most likely "1-line fix" outcome (parallel to Phase 999.8's `set_plugin_option` → `set_option`).

- **D-07:** Probe order for player_clients: researcher decides based on current yt-dlp issue tracker / docs. Default candidates: `ios`, `web_safari`, `android`, `mweb`, `tv`, `web` (the current default). Probe ALL candidates so the diagnostic captures the full failure-mode matrix in one pass.
