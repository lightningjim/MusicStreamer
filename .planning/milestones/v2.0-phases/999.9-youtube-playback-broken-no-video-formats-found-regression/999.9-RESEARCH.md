# Phase 999.9: YouTube playback broken — yt-dlp "No video formats found" regression - Research

**Researched:** 2026-04-24
**Domain:** yt-dlp YouTube extractor configuration (player_client + remote_components)
**Confidence:** HIGH

## Summary

The most likely root cause is **API drift in the EJS PO-token solver configuration**, NOT YouTube-side breakage. yt-dlp 2026.03.17 has moved `remote_components` from an `extractor_args` namespace key (`youtubepot-jsruntime.remote_components`) to a **top-level YoutubeDL parameter** (CLI flag `--remote-components`). The current `extractor_args` dict at `musicstreamer/player.py:488-490` references the obsolete `youtubepot-jsruntime` namespace, which the installed yt-dlp **silently ignores** — meaning the EJS GitHub script fetcher is no longer being enabled, and YouTube's `n` / signature challenges fail, producing `No video formats found!`.

This matches CONTEXT D-05's "most likely 1-line fix" hypothesis. Probe matrix in Section 1 still runs first (cheap, confirms which branch); but if Branch C resolves it before testing each player_client, that's expected.

**Primary recommendation:** Run probe matrix once. If `--remote-components ejs:github` flag fixes things alone (no `--extractor-args` needed), apply Branch C: replace the dead `extractor_args` entry with `remote_components=["ejs:github"]` in the opts dict. If a player_client also helps, layer Branch A on top.

## yt-dlp 2026.03.17 API Status — Critical Finding

[VERIFIED: yt-dlp source inspection at `.venv/lib/python3.13/site-packages/yt_dlp/`]

### `youtubepot-jsruntime` extractor_args namespace: **REMOVED**

The string `youtubepot-jsruntime` does not appear anywhere in yt-dlp 2026.03.17 source. Only `youtubepot` (no `-jsruntime` suffix) survives, and it serves a different role (PO token providers, not JS runtime config).

```bash
$ grep -rn "youtubepot-jsruntime\|jsruntime" .venv/lib/python3.13/site-packages/yt_dlp/
# No matches for 'youtubepot-jsruntime'.
# 'youtubepot' alone exists at extractor/youtube/pot/_director.py:352 — different purpose.
```

### `remote_components` is now a top-level YoutubeDL param

In 2026.03.17, `remote_components` is read via `self.ie.get_param('remote_components')` — that's a top-level YoutubeDL kwarg, NOT `extractor_args`.

| Source location | Evidence |
|-----------------|----------|
| `yt_dlp/__init__.py:955` | `'remote_components': opts.remote_components,` — wired from CLI to params dict |
| `yt_dlp/YoutubeDL.py:738-739` | `self.params['remote_components'] = set(...)` — normalized at YoutubeDL init |
| `yt_dlp/options.py:481-496` | CLI flag is `--remote-components ejs:github` (or `ejs:npm`) |
| `yt_dlp/extractor/youtube/jsc/_builtin/ejs.py:292` | `if 'ejs:github' not in (self.ie.get_param('remote_components') or ()):` — reader site |

**Implication:** The current `extractor_args={"youtubepot-jsruntime": {"remote_components": ["ejs:github"]}}` at `musicstreamer/player.py:488-490` is a no-op. yt-dlp silently ignores unknown extractor_args namespaces. The EJS GitHub script fetcher is currently disabled in MusicStreamer.

### Player clients available in 2026.03.17

[VERIFIED: `INNERTUBE_CLIENTS` dict in `_base.py:97-413`]

User-requestable clients (those without leading `_`): `web`, `web_safari`, `web_embedded`, `web_music`, `web_creator`, `android`, `android_vr`, `ios`, `mweb`, `tv`, `tv_downgraded`, `tv_simply`.

**Probe candidates** (covers the diverse extraction backends): `web`, `web_safari`, `mweb`, `ios`, `android`, `tv`. Skip `web_music` / `web_creator` (auth-only) and `web_embedded` / `tv_embedded` (rarely useful for live).

### Existing extractor_args namespaces that ARE still recognized (for reference)

| Namespace | Purpose |
|-----------|---------|
| `youtube` | Player client, formats, etc. (e.g. `--extractor-args "youtube:player_client=ios"`) |
| `youtube-ejs` | EJS-specific dev settings (`dev`, `repo`, `script_version`) — see ejs.py:112 |
| `youtubejsc-<provider>` | Per-JS-challenge-provider settings (e.g. `youtubejsc-ejs`) — see jsc/_director.py:202 |
| `youtubepot-<provider>` | Per-PO-token-provider settings — see pot/_director.py:352 |

`youtubepot-jsruntime` is NOT in this list — the `-jsruntime` suffix never existed in 2026.03.17.

## Conditional Fix Shape Decision Tree

### Branch A: Pin a working player_client

Edit `musicstreamer/player.py:488-490`:

```python
opts = {
    "quiet": True,
    "no_warnings": True,
    "skip_download": True,
    "format": "best[protocol^=m3u8]/bestaudio/best",
    "extractor_args": {
        "youtube": {"player_client": ["<WORKING_CLIENT>"]},
    },
    "remote_components": ["ejs:github"],  # was: extractor_args["youtubepot-jsruntime"]
}
```

**Layered with Branch C** (always): the dead `youtubepot-jsruntime` entry is replaced by the top-level `remote_components` kwarg.

### Branch B: User-facing toast (no working client)

No edit to `_youtube_resolve_worker`. Add a YouTube-specific toast in `musicstreamer/ui_qt/main_window.py`. The cleanest hook is connecting `youtube_resolution_failed` directly (separate from generic `playback_error`) — but that signal is currently consumed by the Player's own `_on_youtube_resolution_failed` which emits `playback_error` (player.py:524-528). Two options:

1. **Wire a new MainWindow handler for `youtube_resolution_failed`** (cleanest):
   ```python
   # main_window.py, near line 224 (next to cookies_cleared connection)
   self._player.youtube_resolution_failed.connect(self._on_youtube_failed)

   def _on_youtube_failed(self, _msg: str) -> None:
       self.show_toast(
           "YouTube playback temporarily broken — yt-dlp upstream issue. "
           "Try again after a yt-dlp update."
       )
   ```
   This fires BEFORE `_on_playback_error`, so the user sees the descriptive YouTube toast instead of the generic "Playback error: ..." toast. (Both will fire — order is connection-order; if the generic toast steps on the YouTube one, the planner can either disconnect/special-case in `_on_playback_error` or move the YouTube toast to a longer duration.)

2. **Special-case in `_on_playback_error`** (simpler, less explicit):
   ```python
   def _on_playback_error(self, message: str) -> None:
       if message.startswith("YouTube resolve failed"):
           self.show_toast("YouTube playback temporarily broken — try again after yt-dlp update.")
           return
       truncated = message[:80] + "…" if len(message) > 80 else message
       self.show_toast(f"Playback error: {truncated}")
   ```
   Relies on the prefix wired at player.py:527 (`f"YouTube resolve failed: {msg}"`). Lighter touch but couples to a string literal. **Recommended for D-02 lightweight scope.**

### Branch C: Replace dead extractor_args namespace (most likely outcome)

```python
opts = {
    "quiet": True,
    "no_warnings": True,
    "skip_download": True,
    "format": "best[protocol^=m3u8]/bestaudio/best",
    "remote_components": ["ejs:github"],  # top-level param in yt-dlp 2026.03.17

```

That's it. The entire `extractor_args` dict can be removed if no per-client pin is needed (Branch A not layered). Verify against `yt_dlp/__init__.py:955` and `yt_dlp/YoutubeDL.py:738`.

**Why this is the most likely fix:** the silent ignore of an unknown extractor_args namespace means the EJS GitHub fetcher has been disabled in MusicStreamer since the user's yt-dlp was upgraded to a version where the `-jsruntime` suffix was dropped. The probe will confirm this in row 7 vs rows 1-6.

## Common Pitfalls

### Pitfall 1: extractor_args dict-merge silently loses keys
**What goes wrong:** Replacing `extractor_args={"youtubepot-jsruntime": {...}}` with `extractor_args={"youtube": {"player_client": [...]}}` in Branch A would have lost the EJS config — except the EJS config was already a no-op. **Mitigation:** in Branch A+C combined, keep BOTH `extractor_args["youtube"]` AND top-level `remote_components` (different namespaces, no conflict).

### Pitfall 2: cookie_utils invariant
**What goes wrong:** Moving the `yt_dlp.YoutubeDL(opts)` call outside the `with cookie_utils.temp_cookies_copy()` block reopens the Phase 999.7 corruption regression. **Mitigation:** the fix is a single-line edit to the `opts` dict literal at lines 488-490 — never touch the surrounding `with` block (lines 498-506).

### Pitfall 3: Signal connection order in Branch B option 1
**What goes wrong:** If `_player.youtube_resolution_failed` is connected to a new MainWindow handler AND the Player's `_on_youtube_resolution_failed` (which emits `playback_error`) also fires, the user could see two toasts. **Mitigation:** prefer Branch B option 2 (special-case in `_on_playback_error`) for the lighter D-02 scope.

### Pitfall 4: LoFi Girl live ID rotation
**What goes wrong:** YouTube live IDs occasionally change (channel re-streams). A hardcoded URL in the probe may 404 unrelated to the bug. **Mitigation:** planner instructs executor to grab the URL from the user's MusicStreamer station list at probe time, not from research.
