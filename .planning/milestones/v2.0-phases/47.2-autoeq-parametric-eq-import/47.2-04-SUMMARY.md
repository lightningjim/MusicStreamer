### musicstreamer/settings_export.py (MODIFIED)

**Write side** — `build_zip`: new loop after the logos-writing loop, inside the same `with zipfile.ZipFile(dest_path, "w")` block:
```python
eq_dir = paths.eq_profiles_dir()
if os.path.isdir(eq_dir):
    for fname in sorted(os.listdir(eq_dir)):
        src = os.path.join(eq_dir, fname)
        if os.path.isfile(src) and fname.endswith(".txt"):
            zf.write(src, f"eq-profiles/{fname}")
```

**Read side** — `commit_import`: new loop INSIDE the existing `with zipfile.ZipFile(preview.zip_path, "r") as zf:` block, after the station/logo extraction loop, still inside `with repo.con:`. Reuses the already-computed `zip_names` set:
```python
eq_dir = paths.eq_profiles_dir()
os.makedirs(eq_dir, exist_ok=True)
for name in zip_names:
    if not name.startswith("eq-profiles/") or not name.endswith(".txt"):
        continue
    basename = posixpath.basename(name)
    if not basename:
        continue  # skip directory entries
    with zf.open(name) as src, open(
        os.path.join(eq_dir, basename), "wb"
    ) as dst:
        dst.write(src.read())
```

**Placement choice:** inside the existing `with zipfile.ZipFile ... with repo.con:` block, alongside logo extraction. Rationale: same `zip_names` set, single `_validate_zip_members` call, mirrors logo handling pattern. Write goes to the filesystem directly (no pending-list needed — profiles are small text files and the idempotent overwrite is safe for reruns; unlike logos which live under `assets/{station_id}/` and depend on DB commit succeeding first).

**Defense strengthening** (`_validate_zip_members`): added a `..` path-component check before `posixpath.normpath`. Root cause: the original guard only rejected `..` at the top level (`..` or `../*`), but `posixpath.normpath("eq-profiles/../evil.txt")` collapses to `evil.txt`, which passes the original check while escaping the intended subdirectory. Fix: reject any member whose split path contains `..`. See Deviations below.

None of these were in Plan 04's scope and none needed further modification.

### [Rule 2 - Missing Security] Hardened `_validate_zip_members` against nested `..`

- **Found during:** Task 2 GREEN — `test_eq_profiles_path_traversal_rejected` failed even after implementing the extraction loop because the existing `_validate_zip_members` did not reject the hostile member.
- **Root cause:** `posixpath.normpath("eq-profiles/../evil.txt")` collapses to `"evil.txt"`, which is neither `..` nor `../*` — the original guard was rooted at the top of the archive only and let the hostile name through. The extraction loop itself still wrote only inside `eq_dir` (it uses `posixpath.basename` on the raw name, which returns `"evil.txt"` joined to `eq_dir`), so exploitability is minimal — but the threat register's T-47.2.04-01 disposition is "mitigate" via `_validate_zip_members`, and the plan's acceptance criteria expect the validator to reject it.
- **Fix:** Added `if ".." in fname.split("/"): raise ValueError(...)` ahead of the existing normpath check. Rejects any `..` path component at any depth. No collateral damage — legitimate names like `foo..bar.jpg` do not contain a standalone `..` component.
- **Files modified:** `musicstreamer/settings_export.py` (lines inside `_validate_zip_members`).
- **Commit:** Bundled into the GREEN commit `aab5ce0`.
- **Regression check:** All 20 pre-existing settings_export tests still pass (including `test_preview_path_traversal` which uses a top-level `../evil.txt`).
