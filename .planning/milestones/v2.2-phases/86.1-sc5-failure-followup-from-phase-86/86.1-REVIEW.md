# Phase 86.1: Code Review Report

**Reviewed:** 2026-06-05T02:37:49Z
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

The phase fixes the SC5 privacy defect: in the Flatpak sandbox the `:ro` host
mount makes `src != dest`, so the legacy `run_migration()` copy path silently
ingested 0600 host secrets (`cookies.txt`, `twitch-token.txt`) into the sandbox
without consent. The fix adds `is_sandboxed()` and an early-return guard in
`run_migration()`, and wires the consent-based `FlatpakImportWizard` into
`_run_gui` behind a two-conjunct gate.

**Core security/privacy correctness is sound.** The three load-bearing
properties hold:

1. **No secret leakage.** When `is_sandboxed()` is True, `run_migration()`
   returns before `_copy_tree_nondestructive` is ever reached
   (`migration.py:54-56`). Zero file copies occur. `test_migration_sandboxed_skips_copy`
   asserts both secrets are absent from dest.
2. **Marker/copy ordering is correct.** The sandbox guard runs *before* the
   same-path check and the copy branch, so a sandboxed process can never fall
   through to a copy. The marker is written so the buffer_log ordering invariant
   (`__main__.py:199`) is preserved.
3. **Startup gate is correctly conjunctive.** `_maybe_offer_import` gates on
   `is_sandboxed() AND should_offer_import_wizard()`, so native Linux (where
   host path == sandbox path) never gets the offer. The wizard is wired as the
   sole consent-based import path.

The findings below are robustness and maintainability concerns, not breaks in
the security guarantee. No BLOCKER-tier defects were found.

## Warnings

### WR-01: `should_offer_import_wizard()` omits the sandbox precondition — latent native-Linux footgun

**File:** `musicstreamer/flatpak_first_launch.py:104-117`
**Issue:** The function's correctness as a "should the wizard appear" decision
depends entirely on the caller ALSO checking `is_sandboxed()`. On native Linux
with legacy data, `has_unsandboxed_data()` is True and the offer flag is absent,
so `should_offer_import_wizard()` returns **True**. The only thing preventing a
native-Linux user from being offered a self-import is the `is_sandboxed() and ...`
conjunction in `__main__._maybe_offer_import` (`__main__.py:268`). The helper's
docstring (lines 104-117) describes its return contract without mentioning this
precondition, and the security boundary is not enforced where the decision is
made. Any future caller — a "Import from host settings…" menu item, a refactor,
or a second entry point — that calls `should_offer_import_wizard()` alone would
re-introduce the exact class of defect this phase fixes (offering to import
host data into a context where `_HOST_DATA_DIR == data_dir()`).
**Fix:** Make the helper self-guarding so the security invariant is enforced at
the source of truth, not at every call site:
```python
def should_offer_import_wizard() -> bool:
    # Wizard only makes sense inside the sandbox: on native Linux the host
    # path IS the sandbox path, so there is nothing to import.
    if not is_sandboxed():
        return False
    if not has_unsandboxed_data():
        return False
    return not os.path.isfile(import_offered_flag_path())
```
The `__main__.py:268` gate then becomes redundant-but-harmless defense in depth.
Update the docstring to state the sandbox precondition explicitly.

### WR-02: Unhandled exception in `_maybe_offer_import` Qt slot can crash the deferred timer callback

**File:** `musicstreamer/__main__.py:263-277`
**Issue:** `_maybe_offer_import` is invoked from `QTimer.singleShot(0, ...)`. The
body performs three lazy imports, two filesystem-probing calls
(`is_sandboxed()`, `should_offer_import_wizard()` which does `os.path.isfile`),
and a `FlatpakImportWizard` construction — none wrapped in a try/except. If any
raises (e.g., `paths.data_dir()` resolves to an unwritable/odd location,
`os.path.isfile` raises on a malformed path, or the wizard import/constructor
throws), the exception propagates out of the Qt slot. Qt's default behavior for
an exception in a singleShot slot is implementation-dependent and can terminate
the application or be silently swallowed depending on the PySide6 build — either
way the user loses the offer with no diagnostic. This runs on the happy startup
path of every sandboxed launch with host data.
**Fix:** Wrap the body defensively so a wizard-offer failure degrades gracefully
to "no offer this launch" rather than risking the GUI:
```python
def _maybe_offer_import() -> None:
    try:
        from musicstreamer import flatpak_first_launch, paths
        from musicstreamer.ui_qt.flatpak_import_wizard import FlatpakImportWizard
        if flatpak_first_launch.is_sandboxed() and flatpak_first_launch.should_offer_import_wizard():
            FlatpakImportWizard(
                paths.db_path(),
                toast_callback=window.show_toast,
                parent=window,
            ).exec()
    except Exception:
        logging.getLogger(__name__).exception("flatpak import-wizard offer failed")
```

### WR-03: `write_offered_flag()` swallows no errors but `open(..., "a").close()` leaks the fd on partial failure / uses bare open without context manager

**File:** `musicstreamer/flatpak_first_launch.py:139`
**Issue:** `open(flag_path, "a").close()` does not use a `with` context manager.
This violates the project convention (CLAUDE.md / standard Python guidance flags
missing `with` for file operations) and, more concretely, if `open()` succeeds
but `.close()` is interrupted, or if a future edit adds a write between open and
close, the file descriptor can leak. More importantly: the function performs no
error handling around `os.makedirs` / `open`. Because this flag is the offer-once
guarantee (D-03), a silent failure here means the wizard will be re-offered on
every launch (annoying but safe) — however an *uncaught* exception here, when
called from the wizard's `reject()` / `_on_commit_done` paths, will propagate
into Qt close handling. The flag write is offer-once bookkeeping; it should never
be able to take down a dialog close.
**Fix:** Use a context manager and treat the flag write as best-effort:
```python
def write_offered_flag() -> None:
    flag_path = import_offered_flag_path()
    try:
        os.makedirs(os.path.dirname(flag_path), exist_ok=True)
        with open(flag_path, "a"):
            pass  # touch: create if absent, leave content untouched
    except OSError:
        logging.getLogger(__name__).warning("could not write offer-once flag", exc_info=True)
```

### WR-04: `paths.db_path()` passed to the wizard is effectively dead — commit worker reopens via `db_connect()`

**File:** `musicstreamer/__main__.py:269-273` (cross-file: `musicstreamer/ui_qt/flatpak_import_wizard.py:85,170-175`)
**Issue:** `_run_gui` constructs the wizard with `paths.db_path()` as the
`sandbox_db_path` positional arg. The wizard stores it as `self._sandbox_db_path`
(line 175), but the actual commit happens in `_ImportCommitWorker.run()` which
opens a fresh connection via `Repo(db_connect())` (`flatpak_import_wizard.py:85`)
and ignores `_sandbox_db_path` entirely. So the constructor argument that *looks*
like it controls the import target does not. This is dead-parameter coupling: a
reader of `_run_gui` reasonably believes the import is targeted at `paths.db_path()`,
but changing that argument would have no effect. It also means the import commits
against a SEPARATE sqlite connection while `_run_gui`'s long-lived `con`/`repo`
(`__main__.py:220-226`) hold the same file open — the MainWindow will show stale
data until it re-queries. Not a corruption risk (sqlite handles concurrent
connections) but a correctness-of-display gap and a maintenance trap.
**Fix:** Either (a) make the commit worker actually use `self._sandbox_db_path`
so the wiring is honest, or (b) drop the unused parameter and document that the
wizard always targets `db_connect()`. If MainWindow must reflect imported data,
emit/handle `import_complete` to trigger a repo refresh. This finding is on the
in-scope `__main__.py` call site; the worker body is in the out-of-scope wizard
file but is the reason the argument is inert.

## Info

### IN-01: Duplicated `import os` inside functions despite module-level import

**File:** `musicstreamer/__main__.py:170`
**Issue:** `_strip_inherited_activation_tokens` does `import os` locally even
though the module is structured around lazy imports elsewhere. `os` is a stdlib
module with no Qt/GStreamer cost; the local import adds noise and inconsistency
(other functions rely on top-level imports). Not introduced by this phase but
adjacent to the reviewed changes.
**Fix:** Hoist `import os` to module level (it is cheap and already imported in
sibling modules) and drop the function-local import.

### IN-02: Magic filename strings duplicated across modules

**File:** `musicstreamer/flatpak_first_launch.py:55,101` and `musicstreamer/migration.py:32`
**Issue:** The literal `"musicstreamer.sqlite3"` appears in `flatpak_first_launch.py:55`
(`_HOST_DB`) and is the same filename `paths.db_path()` constructs
(`paths.py:39`); `~/.local/share/musicstreamer` is hard-coded in both
`flatpak_first_launch.py:54` (`_HOST_DATA_DIR`) and `migration.py:32`
(`_LEGACY_LINUX`). These two host-path constants are conceptually the same value
(the legacy Linux data dir) maintained in two places. If the DB filename or
legacy path ever changes, the sandbox-detection probe and the migration source
can silently diverge.
**Fix:** Define the legacy host data dir and DB filename once (e.g., a shared
constant in `paths.py` or a small `_legacy.py`) and have both modules import it,
so `_HOST_DATA_DIR`, `_LEGACY_LINUX`, and the DB basename share one source of
truth.

### IN-03: `should_offer_import_wizard()` re-derives the flag path on every call

**File:** `musicstreamer/flatpak_first_launch.py:117`
**Issue:** Minor — `import_offered_flag_path()` does a late `from musicstreamer
import paths` import on each invocation. Functionally fine and intentional (keeps
the module Qt-free at import time per the module docstring), but the repeated
import-inside-function pattern is worth a one-line comment at the call site in
`should_offer_import_wizard()` so future readers don't "optimize" it to a
top-level import and break the Qt-free guarantee.
**Fix:** No code change required; optionally add `# late import (Qt-free module)`
to clarify intent at line 99.

## Notes on test coverage (positive)

The test suite for this phase is strong and adversarial in the right places:
- `test_migration_sandboxed_skips_copy` directly asserts both `cookies.txt` and
  `musicstreamer.sqlite3` are absent from dest while the marker IS written —
  this is the core privacy guarantee and it is verified.
- `test_flatpak_import_wizard_wiring.py` covers all four gate cases including the
  critical native-Linux case (case 2) where `should_offer` alone is True but the
  composed gate is False. NOTE: this same test (lines 92-95) is itself the proof
  of WR-01 — `should_offer_import_wizard()` returning True while not sandboxed is
  the documented-but-unenforced footgun.
- `test_migration_copy_preserves_mode_bits` guards the 0600 invariant on the
  legitimate (non-sandboxed) copy path.

One gap: there is no test asserting `run_migration()`'s **ordering** — that the
sandbox guard short-circuits *before* the same-path/copy branches even when
`src == dest` would also be true inside a sandbox. The existing test uses
distinct `src`/`dst`. A regression that moved the `is_sandboxed()` check below
the same-path check would still pass current tests in the divergent-path case.
Consider adding a sandboxed test where `_LEGACY_LINUX == paths.data_dir()` to
pin the guard ordering.
