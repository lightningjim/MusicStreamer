## What Was Built

### Task 1: FlatpakImportWizard offer wired in _run_gui

Modified `musicstreamer/__main__.py` `_run_gui()`:

- Added local helper `_maybe_offer_import()` defined after `window.show()` and before `return app.exec()`
- Gate: `flatpak_first_launch.is_sandboxed() and flatpak_first_launch.should_offer_import_wizard()`
- When gate is True: constructs `FlatpakImportWizard(paths.db_path(), toast_callback=window.show_toast, parent=window)` and calls `.exec()` (modal)
- Scheduled via `QTimer.singleShot(0, _maybe_offer_import)` so modal renders over shown main window at event loop start
- All four imports (`flatpak_first_launch`, `paths`, `FlatpakImportWizard`, `QTimer`) are lazy inside `_run_gui` — matches existing style; `_run_smoke` stays Qt-light and `ui_qt`-free (D-05/D-24)
- `_run_smoke` untouched — no `FlatpakImportWizard` or `ui_qt` in smoke path

### Task 2: Unit coverage of the wizard-offer gate (four cases)

Created `tests/test_flatpak_import_wizard_wiring.py`:

- `test_sandboxed_with_data_and_no_flag_gate_is_true` — Case 1: sandboxed + host DB + no flag → gate True (offer fires)
- `test_not_sandboxed_with_data_gate_is_false` — Case 2: not sandboxed + host DB → gate False (native Linux, proves `is_sandboxed()` conjunct blocks it; also asserts `should_offer_import_wizard()` alone is True to isolate the cause)
- `test_sandboxed_with_data_and_flag_present_gate_is_false` — Case 3: sandboxed + data + `write_offered_flag()` called → gate False (offer-once D-03)
- `test_sandboxed_no_host_db_gate_is_false` — Case 4: sandboxed + no host DB → gate False (fresh Flatpak user)
- No Qt imports — runs headless in CI
- Monkeypatches `_FLATPAK_INFO`, `_HOST_DB`, and `paths._root_override` exactly as the existing test suite
