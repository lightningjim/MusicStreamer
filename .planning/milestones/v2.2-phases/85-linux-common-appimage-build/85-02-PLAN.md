2. **Add D-09 fail-fast gate** — insert near the top of `build.sh`, AFTER the `set -euo pipefail` line and BEFORE `HERE=...`. **CRITICAL: this gate MUST appear before the first `docker run` / `docker build` invocation in the script** so that a missing-key failure surfaces in <100ms rather than waiting on a Docker daemon probe. This positioning is enforced by the structural source-grep gate in this task's `<verify>` block (warning-1 + warning-2 mitigation from plan-checker pass).
   ```bash
   # D-09 fail-fast: signing is mandatory for release artifacts; SKIP_SIGN=1 is the
   # local-iteration escape hatch (CI never sets SKIP_SIGN per D-09).
   # POSITIONING: this block MUST stay above the first `docker run`/`docker build`
   # call (currently around line 91 from Plan 85-01). Moving it below docker
   # invocations would cause unset-key runs to hang on a daemon probe instead of
   # failing fast — see warning #2 in the Plan 85-02 checker pass.
   if [[ -z "${GPG_KEY_ID:-}" && "${SKIP_SIGN:-0}" != "1" ]]; then
     echo "BUILD_FAIL reason=gpg_key_unset (set GPG_KEY_ID=<keyid> or SKIP_SIGN=1 for local iteration; CI must set the key) (D-09)" >&2
     exit 5
   fi
   ```

3. **Add D-08 signing step** — insert AFTER the GLIBC scan completes successfully (after the final `case "$GLIBC_MAX" in ... esac` block and BEFORE the final `echo "BUILD_OK appimage=$APPIMG glibc=$GLIBC_MAX"` line). The signature must cover the final artifact bytes (D-08):
   ```bash
   # D-08 / PKG-LIN-APP-10: GPG-sign the produced AppImage. The signature is a
   # detached, armored sidecar at <appimage>.sig and is published alongside
   # the AppImage in the release artifact set. linuxdeploy itself supports a
   # --sign flag, but the standalone gpg2 invocation here keeps the signing
   # surface visible at the build-driver level (easier to audit / disable).
   if [[ "${SKIP_SIGN:-0}" != "1" ]]; then
     gpg2 --detach-sign --armor --local-user "$GPG_KEY_ID" --output "${APPIMG}.sig" "$APPIMG" \
       || { echo "BUILD_FAIL reason=signing_failed (gpg2 --detach-sign exited non-zero for key=$GPG_KEY_ID)" >&2; exit 6; }
     echo "SIGN_OK signature=${APPIMG}.sig key=$GPG_KEY_ID"
   else
     echo "SIGN_SKIPPED SKIP_SIGN=1 (local-iteration mode; no .sig sidecar produced)"
   fi
   ```

   The `--output "${APPIMG}.sig"` flag makes the destination explicit (defaults vary by GPG version); `--armor` produces an ASCII-armored signature (`.sig` per AppImage convention rather than `.asc`, though both are accepted by `gpg --verify`).

4. Update the final `echo "BUILD_OK ..."` line to include the signature path when present:

   ```bash
   else
     echo "BUILD_OK appimage=$APPIMG glibc=$GLIBC_MAX signature=skipped"
   fi
   ```

Do NOT touch the `gpg-agent` setup or `GNUPGHOME` env (Plan 85-03 handles ephemeral GNUPGHOME for CI; local developer runs use the host's default keyring). Do NOT hardcode any specific key ID or fingerprint anywhere (per D-09).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host environment → `GPG_KEY_ID` env var | Build driver reads the value; logs it in `SIGN_OK`/`SIGN_SKIPPED` messages (key ID is public metadata; private key material never crosses this boundary). |
| host GPG agent → signing operation | `gpg2 --detach-sign` invokes the host gpg-agent for private-key access; passphrase prompt or agent pinentry is the developer's responsibility. |
| AppImage `.upd_info` section | Embedded zsync URL points at github.com/kcreasey/MusicStreamer; downstream AppImageUpdate clients will resolve this URL to fetch updates. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-85-02-01 | Information disclosure | `set -x` inside docker bash -c body (Plan 85-01 leaves this enabled for debugging) | accept | `GPG_KEY_ID` is the public key ID, not the private key material. `gpg2 --detach-sign` invocation happens OUTSIDE the docker container (after AppImage is copied to host `artifacts/`) so the container's `set -x` never sees the signing step. Private key material lives in the host gpg-agent and never crosses into the container. |
| T-85-02-02 | Tampering | embedded zsync update URL → future AppImageUpdate clients | mitigate | The signature sidecar (PKG-LIN-APP-10) is the user-side guard against hostile takeover of the GitHub Releases endpoint. Plan 85-05's README must document the canonical signing key fingerprint so users can verify out-of-band before trusting an update. (README documentation is Plan 85-05's surface; this plan establishes the contract.) |
| T-85-02-03 | Spoofing | unsigned AppImage masquerading as a release | mitigate | The D-09 fail-fast gate (exit 5) ensures `build.sh` cannot silently produce an unsigned release artifact. CI workflow (Plan 85-03) does not set `SKIP_SIGN`, so the only path to an unsigned AppImage is a developer's explicit `SKIP_SIGN=1` for local iteration — which produces the literal `signature=skipped` token in stdout, easily greppable in any pre-release verification. The Task-2 structural source-grep gate (awk position check) also prevents a future refactor from accidentally moving the fail-fast block below the docker probe (which would mask the signing-bypass failure as an unrelated daemon hang). |
| T-85-02-04 | Elevation of privilege | `gpg2 --local-user "$GPG_KEY_ID"` with unsanitized env var | accept | `GPG_KEY_ID` is sourced from the developer's environment, not user input. The argument is passed as a single shell-quoted token (`"$GPG_KEY_ID"`); shell injection requires the developer to have already compromised their own shell. |
| T-85-02-SC | Tampering | this plan introduces no new npm/pip/cargo installs | n/a | No package legitimacy gate required. |
</threat_model>
