<objective>
Promote the Phase 85a spike build pipeline to production at `tools/linux-build/`. Three concerns:

1. **Single source-of-truth bundle** (D-01, D-02, D-03): `environment.yml` is the only place package lists live. `build.sh` parses it at build start to synthesize `CONDA_PACKAGES` for linuxdeploy-plugin-conda, then a post-step `pip install --no-deps .` adds the application code without re-resolving the conda graph.
2. **Production AppRun** (Pitfalls 19, 20): final exec is `python -m musicstreamer`; `PULSE_PROP` is exported for deterministic PipeWire identity.
3. **Relocate-from-spike** (Claude's Discretion per D-07 / PATTERNS.md): `Dockerfile`, `pins.env`, `verify-pins.sh` move from `.planning/spikes/85a-linux-packaging-spike/` to `tools/linux-build/` so production builds don't reach into the planning directory.

Purpose: the entire downstream phase (signing, CI, smoke) depends on `tools/linux-build/build.sh` producing a working AppImage. This plan delivers that core.

Output: a runnable production build that, when invoked from the host as `bash tools/linux-build/build.sh`, produces `tools/linux-build/artifacts/MusicStreamer-<version>-x86_64.AppImage` with GLIBC ≤ 2.35, the same conda plugin set as the Windows installer, and an AppRun that launches the real `musicstreamer` module.
</objective>
