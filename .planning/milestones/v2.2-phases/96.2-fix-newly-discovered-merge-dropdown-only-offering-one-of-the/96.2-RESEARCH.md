## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** The newly-discovered merge-target dropdown (`_DiscoverRowWidget._map_combo`) MUST offer all of this provider's saved stations, independent of the `live_url_syncs_from_channel` flag. Source it from a new `repo.list_stations_for_provider(provider_id)` (modeled on `list_flagged_stations_for_provider`, flag predicate dropped), NOT the existing flagged query.
- **D-02:** Scope boundary — only the merge dropdown changes. The REMAP section stays sourced from `list_flagged_stations_for_provider`. Do not broaden the REMAP set.
- **D-03:** Merging onto an unflagged station MUST NOT change that station's `live_url_syncs_from_channel` flag. Re-point stream URL and title anchor only.
- **D-04:** Present stations alphabetically by name — keep `ORDER BY s.name COLLATE NOCASE`.

- Whether to add a dedicated `list_stations_for_provider` repo query vs. filtering an existing list in the dialog.
- Which `stream_id` the `remap` path re-points (preferred_stream_id vs first/primary stream) — verify against existing `apply_refresh` remap behavior.
- Whether the empty-state combo text "(no flagged stations)" should be reworded.

### Deferred Ideas (OUT OF SCOPE)
- Cross-provider / all-YouTube target set.
- Auto-enrolling merged stations into live-sync (flipping `live_url_syncs_from_channel` on merge).
- Stale-only filtering / flagged-first ordering / URL disambiguation hints in the dropdown.

## Summary

Phase 96.2 is a tight, single-file-plus-one-repo-method bugfix. The root cause is confirmed and the fix path is fully mechanical: the `_DiscoverRowWidget` receives `self._flagged_stations` (the flag-filtered list) at `_populate_rows` line 815, but it should receive an all-provider list instead.

The fix requires exactly three code edits: (1) a new `repo.list_stations_for_provider(provider_id)` query (copy of `list_flagged_stations_for_provider` with `AND s.live_url_syncs_from_channel = 1` removed), (2) a second dialog-level load `self._all_provider_stations` alongside `self._flagged_stations` in `LiveRefreshDialog.__init__`, and (3) passing `self._all_provider_stations` instead of `self._flagged_stations` at the `_DiscoverRowWidget` call site in `_populate_rows`. The `_DiscoverRowWidget` constructor signature and interior loop are unchanged except for the list it iterates and the empty-state fallback string.

No new packages. No schema changes. No changes to `apply_refresh`, the WR-03/WR-01 guards, or the REMAP path. All existing tests remain green; a small set of new tests is added to cover D-01, D-02, and D-03.

**Primary recommendation:** One new repo query + two dialog edits. Everything else is unchanged.

### Pattern 1: New Repo Query (`list_stations_for_provider`)

**What:** Copy of `list_flagged_stations_for_provider` with the `AND s.live_url_syncs_from_channel = 1` predicate removed. Same SELECT, same Station-building loop, same `ORDER BY s.name COLLATE NOCASE`.

```python
# Source: repo.py:1067 (list_flagged_stations_for_provider) — verified in session
def list_stations_for_provider(self, provider_id: int) -> List[Station]:
    """Phase 96.2 D-01: all stations for this provider, flag-independent.

    Copy of list_flagged_stations_for_provider with the
    live_url_syncs_from_channel predicate removed. Used as the merge-target
    dropdown source for newly-discovered rows. ORDER BY s.name COLLATE NOCASE
    satisfies D-04.
    """
    rows = self.con.execute(
        """
        SELECT s.*, p.name AS provider_name, p.avatar_path AS provider_avatar_path
        FROM stations s
        LEFT JOIN providers p ON p.id = s.provider_id
        WHERE s.provider_id = ?
        ORDER BY s.name COLLATE NOCASE
        """,
        (provider_id,),
    ).fetchall()
    return [
        Station(
            id=r["id"],
            name=r["name"],
            provider_id=r["provider_id"],
            provider_name=r["provider_name"],
            tags=r["tags"] or "",
            station_art_path=r["station_art_path"],
            album_fallback_path=r["album_fallback_path"],
            icy_disabled=bool(r["icy_disabled"]),
            cover_art_source=r["cover_art_source"] or "auto",
            last_played_at=r["last_played_at"],
            is_favorite=bool(r["is_favorite"]),
            preferred_stream_id=r["preferred_stream_id"],
            streams=self.list_streams(r["id"]),
            prerolls=self.list_prerolls(r["id"]),
            prerolls_fetched_at=r["prerolls_fetched_at"],
            channel_avatar_path=r["channel_avatar_path"],
            provider_avatar_path=r["provider_avatar_path"],
            live_url_syncs_from_channel=bool(r["live_url_syncs_from_channel"]),
            live_url_title_anchor=r["live_url_title_anchor"],
        )
        for r in rows
    ]
```

- **Deduplicating the all-provider list against flagged:** No dedup needed or wanted. An unflagged station in the all-provider list that happens to also be flagged just appears once (all-provider is a superset; flagged ones appear in it naturally).

### Q2: Threading the new list into `_DiscoverRowWidget` without touching REMAP

[VERIFIED: codebase — live_refresh_dialog.py:707-820]

The pattern is clear:
- `self._flagged_stations` stays at line 708 — feeds `_RowWidget` (REMAP) loop at line 790.
- `self._all_provider_stations` is loaded immediately after line 709 — feeds `_DiscoverRowWidget` loop at line 812-819.

### Q3: Remap apply path — which `stream_id`, and does WR-03 still hold?

[VERIFIED: codebase — live_refresh_dialog.py:586-627, 208-255]

`build_staged_change` in map mode (lines 600-618) picks the stream via:
1. Iterate `chosen.streams`, find the one with `position == 1`.
2. If none found, fall back to `chosen.streams[0]`.

This is the same logic for flagged and unflagged stations — it is purely a function of the Station's `.streams` list, which is always populated by `list_streams(r["id"])` in the Station-building loop. An unflagged station with exactly one stream (the common case for a YBC station that has a stale live URL) will correctly resolve to `streams[0]`.

The WR-03 URL duplicate guard (lines 220-232) and WR-01 station_id duplicate guard (lines 241-255) operate on the staged-change dicts, which are the same shape regardless of whether the target was flagged. Both guards remain fully effective.

**Edge case:** A station with zero streams (`chosen.streams` is empty) will result in `primary_stream_id = None`, and `build_staged_change` returns `{"action": "remap", "stream_id": None, ...}`. In `apply_refresh` (line 268-274), `primary = next(...)` falls through to `streams[0] if streams else None`; if streams is empty, `primary is None` triggers `continue` (line 274 defensive skip). This is an acceptable fail-safe — a station with no streams cannot be remapped, and the user would see no mutation for that row.

### Q4: D-03 — confirm sync flag is not touched in merge path

[VERIFIED: codebase — live_refresh_dialog.py:260-290, repo.py:1023-1035]

The `remap` branch in `apply_refresh` (lines 260-290) calls exactly two repo methods:
1. `repo.update_stream(...)` — updates URL and non-URL fields in `station_streams` table.
2. `repo.set_live_url_title_anchor(station_id, new_title)` — updates `stations.live_url_title_anchor`.

`repo.set_live_url_syncs_from_channel` is called only in the `add` branch (line 306) for newly-added stations. It is never called in the `remap` branch. D-03 is already satisfied by the existing code — no change required to `apply_refresh`.

## Common Pitfalls

### Pitfall 1: Accidentally broadening the REMAP section
**What goes wrong:** Passing `self._all_provider_stations` to `_RowWidget` or using it as the `_flagged_stations` loop in `_populate_rows`. This violates D-02 — REMAP rows must only list flagged stations.
**Why it happens:** Both loops are adjacent in `_populate_rows`; a search-and-replace would hit both.
**How to avoid:** Only change line 815 (`_DiscoverRowWidget` call site), not lines 790-799 (`_RowWidget` loop).
**Warning sign:** A REMAP row appears for a station that has `live_url_syncs_from_channel = False`.

### Pitfall 2: Reusing `self._flagged_stations` for the discover combo
**What goes wrong:** Forgetting to add `self._all_provider_stations` and just renaming the existing slot.
**Why it happens:** Looks like a rename fix; it is actually an additive load.
**How to avoid:** After the fix, `LiveRefreshDialog.__init__` must call both `list_flagged_stations_for_provider` AND `list_stations_for_provider` and store them in two separate instance attributes.

### Pitfall 3: Empty-state guard disabling map radio on a non-empty all-provider list
**What goes wrong:** If there are 19 provider stations but 0 flagged, the old code disabled the map radio because `flagged_stations` was empty. After the fix, `all_provider_stations` will be non-empty, so the guard must check the new list.
**Why it happens:** The guard at line 551-553 checks `if not flagged_stations`. After the param rename, this references the correct new list automatically — but only if the param rename is consistent.
**How to avoid:** Confirm that the empty-state guard references the same variable name used in the combo-population loop above it.

### Pitfall 4: Forgetting `list_streams` per station is called in the new query
**What goes wrong:** A test inserts a station but forgets to insert a stream; `station.streams` is empty; `build_staged_change` falls through to the zero-streams defensive skip.
**Why it happens:** `list_stations_for_provider` calls `self.list_streams(r["id"])` like the template — which returns `[]` for stream-less stations.
**How to avoid:** In tests that exercise the remap path against the new query, insert at least one stream for the target station.

### Pitfall 5: Station-building loop field mismatch
**What goes wrong:** Omitting a field from the Station constructor in the new query (e.g., `prerolls_fetched_at`, `provider_avatar_path`) — Station is a dataclass and will raise TypeError.
**Why it happens:** The Station constructor has grown to 16+ fields across many phases.
**How to avoid:** Copy the Station-building block verbatim from `list_flagged_stations_for_provider` — do not reconstruct it from memory.
