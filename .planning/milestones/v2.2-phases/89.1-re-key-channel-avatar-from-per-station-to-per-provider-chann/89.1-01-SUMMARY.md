## What Was Built

**providers.avatar_path schema (D-11):** Idempotent `ALTER TABLE providers ADD COLUMN avatar_path TEXT` appended after the Phase 89A `channel_avatar_path` ALTER in `db_init`. Uses the established try/except OperationalError idiom. Confirmed: providers has no rebuild block (zero hits for `providers_new`), so Pitfall-2 ordering constraint does not apply.

**Idempotent backfill (D-01/D-02/D-03):** Runs on every `db_init` startup after the ALTER. Queries the most-recently-updated station per provider (correlated `MAX(updated_at)` subquery) that has a `channel_avatar_path` and whose provider lacks `avatar_path`. Uses `shutil.copy2` (not `os.rename`) for crash-safety (T-89.1-02). Commits `providers.avatar_path` before running the cleanup pass that deletes old per-station files (D-01 ordering guarantee). Skips `provider_id IS NULL` stations via WHERE clause. `NOT EXISTS` guard ensures idempotency.

**write_provider_avatar (D-10):** Atomic `tempfile.mkstemp + os.replace` writer in `assets.py`, keyed by `{provider_id}.png`. Mirror of `write_channel_avatar`; does not modify it. `provider_id` is int from SQLite PK — no traversal risk (T-89.1-01).

**update_provider_avatar_path (D-09):** Dedicated `UPDATE providers SET avatar_path = ? WHERE id = ?` Repo method placed immediately after `update_channel_avatar_path`. Pitfall-5 rationale documented in docstring. `update_channel_avatar_path` retained with deprecation comment; no new production callers added (D-04).

**Station.provider_avatar_path (D-11):** New `Optional[str] = None` field appended after `channel_avatar_path` in models.py. All four mappers updated: `list_stations`, `get_station`, `list_recently_played`, `list_favorite_stations` each add `p.avatar_path AS provider_avatar_path` to SELECT and `provider_avatar_path=r["provider_avatar_path"]` to constructor (Pitfall-6 — none missed).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Schema guard before backfill SQL**
- **Found during:** Task 2 (GREEN)
- **Issue:** `test_bitrate_kbps_migration_adds_column` builds a minimal stations schema without `provider_id`; the backfill SQL `s.provider_id` failed with `OperationalError: no such column` on that schema
- **Fix:** Added PRAGMA table_info guard before running the backfill — checks both `channel_avatar_path` and `provider_id` columns present before executing the SQL
- **Files modified:** musicstreamer/repo.py
- **Commit:** b2a15a66

**2. [Rule 1 - Bug] SameFileError when station_id == provider_id**
- **Found during:** Task 2 (GREEN)
- **Issue:** In an in-memory DB, the first provider gets `id=1` and the first station gets `id=1`. `shutil.copy2('1.png', '1.png')` raises `SameFileError` (a subclass of `OSError`), which the `except OSError: continue` clause caught — skipping the DB update entirely
- **Fix:** Added `os.path.abspath(_src) != os.path.abspath(_dst)` guard before the copy; when same-file, the file is already in place so just proceed to the DB UPDATE
- **Files modified:** musicstreamer/repo.py
- **Commit:** b2a15a66

**3. [Rule 1 - Bug] Cleanup pass deleted newly-written provider file**
- **Found during:** Task 2 (GREEN)
- **Issue:** After copying `{station_b_id}.png` → `{provider_id}.png`, the cleanup pass deleted `{station_a_id}.png` — which happened to be `{provider_id}.png` because `station_a_id == provider_id`. Result: provider-keyed file was deleted immediately after creation
- **Fix:** Added `AND s.channel_avatar_path != p.avatar_path` to the cleanup SELECT, preventing deletion of paths that are already the committed provider avatar
- **Files modified:** musicstreamer/repo.py
- **Commit:** b2a15a66

**4. [Rule 1 - Bug] Test fixture trigger interference in sibling-wins test**
- **Found during:** Task 1 (test authoring) / confirmed in Task 2
- **Issue:** The `stations_updated_at` trigger fires on EVERY `UPDATE stations`, so direct `UPDATE stations SET updated_at = '2020...'` triggered the trigger which reset `updated_at = datetime('now')` — making both siblings have the same timestamp
- **Fix:** Test drops the trigger with `DROP TRIGGER IF EXISTS stations_updated_at`, sets timestamps directly, then recreates the trigger. Tracked as a test-isolation fix, not a production code change
- **Files modified:** tests/test_repo.py
- **Commit:** b2a15a66

**5. [Rule 1 - Bug] Test setup: station_id == provider_id in backfill_copies test**
- **Found during:** Task 2
- **Issue:** `test_backfill_copies_avatar_to_provider_keyed_location` expected the old per-station file to be deleted, but when `station_id == provider_id`, the cleanup pass exclusion (Fix 3) correctly skips deletion (the station path IS the provider path). The test's assertion was wrong for this case
- **Fix:** Inserted a dummy provider first so `provider_id != station_id`, making the copy path non-trivial and the delete assertion valid
- **Files modified:** tests/test_repo.py
- **Commit:** b2a15a66
