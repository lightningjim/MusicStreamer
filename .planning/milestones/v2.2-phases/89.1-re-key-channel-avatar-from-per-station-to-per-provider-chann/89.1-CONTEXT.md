# Phase 89.1: Re-key channel avatar from per-station to per-provider - Context

**Gathered:** 2026-06-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Move the YouTube channel-avatar from the **Station** to the **Provider**. Phase 89 stored
`channel_avatar_path` on each `stations` row and keyed the cached PNG by `station_id`
(`assets/channel-avatars/{station_id}.png`), so every sibling stream of the same YouTube
channel (e.g. multiple Lofi Girl streams) fetched and stored an identical avatar redundantly.
For YouTube imports the provider IS the channel (`yt_import.py:120` sets provider =
`playlist_channel` / `playlist_uploader` / `uploader`; `providers.name` is UNIQUE).

This phase: add `avatar_path` to the `providers` table, key the cached PNG by `provider_id`,
persist via a provider-keyed update method, and resolve it in `now_playing_panel.bind_station`
and `EditStationDialog` via the station's `provider_id` so all siblings reuse **one fetch and
one file**. Fall back to the station thumbnail when `provider_id` is null. Migrate the existing
per-station avatars and deprecate the old `stations.channel_avatar_path` column.

**Out of scope (their own phases):** new avatar *sources* — Twitch Helix `profile_image_url`
(Phase 89b, ART-AVATAR-04) and provider brand-avatar fallback (Phase 89c, ART-AVATAR-11/12).
This phase is a re-key of the existing YouTube avatar mechanism only.

</domain>

<decisions>
## Implementation Decisions

### Existing-avatar migration *(discussed)*
- **D-01:** **Backfill at `db_init()` + clean orphans.** Add a one-time, idempotent backfill
  pass: for each station with a non-null `channel_avatar_path` AND a non-null `provider_id`,
  copy/rename its on-disk PNG to the provider-keyed location (`{provider_id}.png`) and set
  `providers.avatar_path`. After a provider's avatar is written, **delete the old per-station
  `{station_id}.png` file** so the `channel-avatars/` dir does not accumulate stale duplicates.
  Result: existing users keep their avatars with **zero network re-fetch** on upgrade, and no
  orphaned per-station files linger.
- **D-02:** **Conflict resolution — most recently updated sibling wins.** When several sibling
  stations of one provider each have a stored avatar (normally identical; can differ if the
  channel changed its picture or was fetched at different times), the backfill selects the
  avatar from the sibling with the newest `updated_at` (highest `station_id` is an acceptable
  proxy if `updated_at` ordering is awkward in the query). Reflects the freshest fetch of the
  channel's current picture. First-writer-wins per provider after that (idempotent: skip if
  `providers.avatar_path` already set).
- **D-03:** Backfill MUST be **idempotent and rollback-safe**, following the Phase 89A migration
  discipline: re-running `db_init()` does not re-copy or error; the pass is a no-op once
  `providers.avatar_path` is populated and the per-station file is gone. Stations with
  `provider_id` NULL are skipped (cannot be keyed by provider) — their avatar is intentionally
  dropped (see D-06 fallback).

### Old `stations.channel_avatar_path` fate *(discussed)*
- **D-04:** **Clean cutover — deprecate the station column.** After backfill, avatar resolution
  reads ONLY the provider avatar (via `station.provider_id`). Stop **writing**
  `stations.channel_avatar_path` (remove the `update_channel_avatar_path(station_id, ...)` write
  path / repoint it to the provider). Leave the column in the schema (SQLite cannot `DROP`
  pre-3.35; the project's additive pattern never drops) with a deprecation comment noting it is
  superseded by `providers.avatar_path` as of Phase 89.1. No dual-read fallback — backfill
  already preserved every avatar, so a single resolution path is correct and unambiguous.
- **D-05:** Update the existing readers — `now_playing_panel.bind_station` (the Phase 89-04 swap)
  and `cover_art._channel_avatar_lookup` — to resolve via the provider avatar instead of
  `station.channel_avatar_path`. The `_channel_avatar_lookup` "never raises / no thread launch"
  contract (WR-04, Pitfall 7) is preserved; only the field it reads changes.

### Avatar resolution / fallback chain *(discussed)*
- **D-06:** **Resolution order:** (1) provider avatar via `station.provider_id` if
  `providers.avatar_path` is set and the file exists on disk; (2) `provider_id` NULL → station
  thumbnail (per the phase goal); (3) `provider_id` set but provider avatar not yet fetched
  (e.g. brand-new channel) → **fall through to today's normal cover/logo resolution unchanged**
  (no avatar swap). The re-key changes WHERE the avatar comes from, not the existing non-avatar
  fallback behavior.

### EditStationDialog behavior *(discussed)*
- **D-07:** **Reuse on open, no network fetch.** When `EditStationDialog` opens on a station whose
  provider ALREADY has a stored avatar, show that provider avatar in the preview immediately and
  **skip the network fetch**. The debounced auto-fetch (`_AvatarFetchWorker`) fires only when the
  provider has NO avatar yet. This is the core dedup goal — one fetch per channel, not per station.
- **D-08:** **"Refresh avatar" refreshes the provider and updates all siblings, with an explicit
  shared-effect hint.** Clicking Refresh re-fetches from the network and overwrites the single
  per-provider file; every sibling stream of that channel then shows the new avatar. Surface the
  shared effect in the UI (e.g. button tooltip/label/status text noting it updates the whole
  channel) so the user is not surprised that other stations changed.

### Persistence *(locked from goal + Phase 89 precedent)*
- **D-09:** Add a **provider-keyed persist method** (mirroring `repo.update_channel_avatar_path`
  but keyed by `provider_id`, e.g. `update_provider_avatar_path(provider_id, path)`). Like its
  Phase 89 predecessor it must NOT route through a broad provider/station update, to avoid the
  silent-reset-on-unrelated-save hazard (Phase 89 D-13 / RESEARCH Pitfall 5). Avatar file writes
  remain atomic via `tempfile.mkstemp` + `os.replace` in the same directory (Phase 89A D-12).

### Storage layout *(locked from goal)*
- **D-10:** Cached PNG keyed by **`provider_id`**: `assets/channel-avatars/{provider_id}.png`,
  path stored relative to `paths.data_dir()` (mirroring the existing `write_channel_avatar`
  relative-path convention). The existing `assets.write_channel_avatar(station_id, data)` writer
  is re-keyed to take `provider_id` (or a sibling provider-keyed writer is added) — planner's call
  on signature shape; the layout (flat dir, `{provider_id}.png`) is fixed.

### Schema migration *(locked from Phase 89A precedent — not re-discussed)*
- **D-11:** Add `providers.avatar_path TEXT` via the established idempotent idiom (`try: ALTER
  TABLE providers ADD COLUMN avatar_path TEXT / except sqlite3.OperationalError: pass`). Unlike
  the `stations` additions, `providers` is NOT subject to the `stations_new` rebuild block, so
  the Pitfall-2 ordering constraint does not apply here — but verify the `providers` table has no
  analogous rebuild before placing the ALTER. Nullable `TEXT`, no DEFAULT (NULL = no avatar).

### Claude's Discretion
- Exact backfill query shape (the `updated_at` vs `station_id` tiebreak mechanics in D-02).
- Whether `write_channel_avatar` is re-keyed in place or a provider-keyed sibling writer is added (D-10).
- Exact wording/placement of the Refresh shared-effect UI hint (D-08).
- Test fixture style for the backfill migration (follow Phase 89A migration test pattern).

</decisions>

<specifics>
## Specific Ideas

- The dedup target is concrete: multiple **Lofi Girl** YouTube streams (sibling stations, same
  channel/provider) should produce exactly **one** avatar fetch and **one** cached file after this
  phase, where Phase 89 produced one per station.
- Mirror existing Phase 89 / 89A patterns rather than inventing new ones: idempotent additive
  ALTER, atomic file write, dedicated non-silent-reset persist method.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `repo.update_channel_avatar_path(station_id, path)` — exact template for a provider-keyed
  `update_provider_avatar_path(provider_id, path)` (same non-silent-reset rationale, Pitfall 5).
- `repo.ensure_provider(name)` / the `LEFT JOIN providers p ON p.id = s.provider_id` mappers —
  `provider_id` is already on every Station row, so resolution needs no new join plumbing.
- `assets.write_channel_avatar(station_id, data)` — atomic `mkstemp`+`os.replace` writer to
  re-key by `provider_id`; returns a `data_dir()`-relative path.
- Phase 89A idempotent additive-ALTER idiom — copy for `providers.avatar_path`.

### Established Patterns
- **Non-silent-reset persist (Pitfall 5):** avatar writes go through a dedicated single-column
  update, NOT a broad save, so unrelated edits don't wipe the avatar. Preserve for the provider.
- **Atomic file write (Phase 89A D-12):** `tempfile.mkstemp` in the destination dir + `os.replace`.
- **`_channel_avatar_lookup` contract (WR-04 / Pitfall 7):** never raises into the Qt callback
  chain; no QPixmap / no thread launch in the lookup. Only change the field read, not the contract.
- **Idempotent migration (Phase 89A D-04/D-07):** double-run `db_init()` is a no-op; the backfill
  must share this property.

### Integration Points
- `repo.py:db_init()` — new `providers.avatar_path` ALTER + one-time backfill pass (idempotent).
- `repo.py` — new `update_provider_avatar_path`; deprecate/remove `update_channel_avatar_path` write path.
- `assets.py` — re-key `write_channel_avatar` (or add provider-keyed sibling) + orphan cleanup.
- `cover_art._channel_avatar_lookup` + `now_playing_panel.bind_station` — resolve via provider avatar.
- `ui_qt/edit_station_dialog.py` — reuse-on-open (skip fetch when provider has avatar); Refresh
  overwrites the per-provider file + shared-effect hint.
