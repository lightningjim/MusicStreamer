## Summary

This is a small, self-contained enhancement: detect the fingerprint of a text-mode-corrupted ZIP backup (the Phase 88 / G4 incident) and surface an actionable `QMessageBox` dialog instead of the generic truncating toast. The implementation touches three locations: `settings_export.py` (detection logic + new exception class), `_ImportPreviewWorker.run()` in `main_window.py` (emit a sentinel string for the text-mode case), and `_on_import_preview_error` in `main_window.py` (branch: dialog vs toast).

The CRLF detection algorithm is simple and reliable: read the whole file in binary mode, check that it starts with the ZIP local-file-header magic `PK\x03\x04` (0x50 0x4B 0x03 0x04), and check that the byte sequence `\x0d\x0a` appears anywhere in the content. Because the pre-condition is `zipfile.BadZipFile` (the file cannot open as a valid ZIP), a false positive from a legitimate ZIP that naturally contains `\x0d\x0a` in its compressed data is impossible — such a file would have opened without error. The existing `_GbsImportWorker` sentinel-string pattern (`"auth_expired"`) is the right discriminator idiom to follow.

**Primary recommendation:** Add `TextModeCorruptionError(ValueError)` to `settings_export.py`, detect and raise it inside the `except zipfile.BadZipFile` block, catch it specifically in `_ImportPreviewWorker.run()` and emit the sentinel `"TEXT_MODE_ZIP"`, then branch in `_on_import_preview_error` to show a `QMessageBox.warning` modal for the sentinel and the existing toast for all other errors.

### Pattern 1: CRLF Detection Helper

**What:** Pure stdlib function, no Qt, reads whole file in binary mode, checks PK\x03\x04 magic AND presence of `\x0d\x0a`.

**When to use:** Called only inside `except zipfile.BadZipFile` in `preview_import`. Never called on a successfully-opened ZIP (false positives are impossible in that context).

```python
# Source: verified by live Python 3 stdlib experiment (2026-06-19)
def _is_text_mode_corrupted_zip(path: str) -> bool:
    """Return True iff *path* begins with ZIP local-file-header magic and
    contains at least one injected CRLF (0x0D 0x0A) byte pair.

    Precondition: called only after zipfile.BadZipFile is raised, so a
    legitimate ZIP that naturally contains 0x0D0A in its compressed payload
    cannot reach this function (it would have opened successfully).
    Reading the full file is correct: settings ZIPs are small and DEFLATE
    streams occasionally place their first 0x0A past byte 512.
    """
    try:
        with open(path, "rb") as fh:
            data = fh.read()
    except OSError:
        return False
    return data[:4] == b"PK\x03\x04" and b"\x0d\x0a" in data
```

**ZIP local-file-header facts (PKWARE spec, verified):**
- Bytes 0–3: signature `0x50 0x4B 0x03 0x04` (never altered by text-mode since none are 0x0A)
- Bytes 14–17: CRC-32 (4 arbitrary bytes; frequently contains 0x0A values)
- Bytes 18–25: compressed/uncompressed size fields (arbitrary)

Because the CRC-32 field spans 4 arbitrary bytes, the probability that at least one byte is 0x0A is high for any non-trivial zip, making the `\x0d\x0a` scan reliable in practice. [VERIFIED: live Python experiment against generated ZIPs, 2026-06-19]

**False-positive analysis (D-01 requirement):**
- A legitimate ZIP with `\x0d\x0a` in its compressed data: would have opened normally via `zipfile.ZipFile` → `BadZipFile` would never be raised → detection function never called. Zero false positives. [VERIFIED]
- A truncated ZIP: starts with `PK\x03\x04` but typically has no injected `\x0d\x0a` pairs (the missing/offset data doesn't add CRLFs). Falls through to generic message. [VERIFIED]
- A text file or non-ZIP garbage: will not start with `PK\x03\x04` → returns False → generic message. [VERIFIED]

### Anti-Patterns to Avoid

- **Scanning only first N bytes with a fixed small window:** The DEFLATE output for small settings-only ZIPs (no logos) can have zero `0x0A` bytes in the first 512 bytes even when the full file has some. Read the full file — settings ZIPs are small.
- **Auto-repair (collapsing 0x0D0A → 0x0A):** Explicitly forbidden by D-01. Would clobber legitimate binary byte pairs in zip payload.
- **Changing the `error: Signal(str)` to `Signal(str, str)`:** Unnecessary complexity. The sentinel string approach threads through the existing signal contract unchanged.
- **Modifying the flatpak import wizard path:** D-04 explicitly excludes `flatpak_import_wizard.py`. The wizard's `preview_import` call is on a zip it built itself from binary `build_zip`, so text-mode corruption cannot occur there.
- **Branching inside `settings_export.preview_import` on the exception message string:** The caller must not re-parse `str(exc)` — the subclass type carries the discriminator cleanly.

## Common Pitfalls

### Pitfall 1: Reading only N bytes and missing corruption in small clean-payload ZIPs
**What goes wrong:** A settings-only backup with no logo files and favorable DEFLATE output may have no `0x0A` bytes in its first 512 bytes, yet will still have `0x0A` bytes later (e.g. in the Central Directory entries or deeper in the DEFLATE stream). Scanning only 512 bytes would miss these and fall through to the generic message even though it was text-mode corrupted.
**Why it happens:** DEFLATE output is pseudorandom; the probability that no `0x0A` appears in first 512 bytes is non-trivial for small payloads.
**How to avoid:** Read the full file. Settings ZIPs are small (typically 10 KB–500 KB); there is no performance reason to limit the window.
**Warning signs:** If a test fixture using a pure-deflate small zip finds `_is_text_mode_corrupted_zip` returns `False` even after mangling, the window was too small.

### Pitfall 2: Catching TextModeCorruptionError after (not before) the generic except Exception
**What goes wrong:** If `except Exception as exc` appears before `except settings_export.TextModeCorruptionError`, Python matches the more general clause first and the sentinel is never emitted. The user sees the toast with the sentinel string instead of the dialog.
**Why it happens:** Python `except` clauses are checked in order.
**How to avoid:** The specific `except settings_export.TextModeCorruptionError` catch must come first in `_ImportPreviewWorker.run()`.
**Warning signs:** Unit test of worker emits `"TEXT_MODE_ZIP"` as a string through `str(exc)` instead of the sentinel via the specific branch.

### Pitfall 3: Forgetting to call `self._end_busy()` before the dialog
**What goes wrong:** `_on_import_preview_error` must call `self._end_busy()` before showing the dialog. If omitted, the busy spinner stays active while the modal blocks.
**Why it happens:** Existing code calls `_end_busy()` first, but the new branch must also do so.
**How to avoid:** `_end_busy()` call is the first line of `_on_import_preview_error` already — the new `if msg == "TEXT_MODE_ZIP"` branch comes after it, so `_end_busy()` runs in both branches naturally.
**Warning signs:** Spinner still animating while dialog is shown.

### Pitfall 4: The D-03 fallback message text
**What goes wrong:** D-03 changes the non-text-mode `BadZipFile` message from bare `"Not a valid ZIP archive"` to `"Not a valid ZIP archive — the file may be incomplete or corrupted."`. The existing `test_preview_invalid_zip` test uses `match="Not a valid ZIP archive"` — this regex still matches the new longer string (it is a substring match), so no test update is required. But new tests should use the full D-03 string when asserting the non-text-mode path.
**How to avoid:** Confirm `pytest.raises(ValueError, match="Not a valid ZIP archive")` still passes (it does, since `re.search` is used, not `==`).

### Complete `_is_text_mode_corrupted_zip` with ZIP spec facts

```python
# Source: PKWARE ZIP Application Note (local file header structure),
# verified by live Python experiment 2026-06-19.
def _is_text_mode_corrupted_zip(path: str) -> bool:
    """Return True iff the file at *path* has the fingerprint of a ZIP that
    was corrupted by a text-mode file transfer (LF 0x0A rewritten to CRLF
    0x0D 0x0A by the transfer layer).

    Detection requires ALL of:
    - First 4 bytes are the ZIP local-file-header signature 0x50 0x4B 0x03 0x04.
      (Bytes 2-3 are 0x03/0x04 — neither is 0x0A — so text-mode never alters
      the magic. The signature is thus reliable post-corruption.)
    - The file contains at least one 0x0D 0x0A byte pair (injected by the
      LF->CRLF conversion).

    This function is called only after zipfile.BadZipFile is raised, so a
    legitimate binary ZIP that happens to contain 0x0D0A in its compressed
    payload cannot reach here (it would have opened successfully).

    The full file is read because DEFLATE output for small payloads may place
    the first 0x0A byte past the 512-byte mark. Settings ZIPs are small
    (typically <500 KB) so this is not a performance concern.
    """
    try:
        with open(path, "rb") as fh:
            data = fh.read()
    except OSError:
        return False
    return data[:4] == b"PK\x03\x04" and b"\x0d\x0a" in data
```

## Open Questions (RESOLVED)

1. **Minimum file size to attempt detection**
   - What we know: `fh.read()` on an empty file or 0-byte file returns `b""`, which does not start with `PK\x03\x04` — returns `False` safely.
   - What's unclear: whether to add an early-return for files under, say, 22 bytes (the minimum valid ZIP is ~22 bytes).
   - RESOLVED: No special-case needed. The `data[:4] == b"PK\x03\x04"` check already filters short/empty files correctly. Plan 01 Task 1 implements exactly this guard; no early-return added.

2. **LF-only removal on reverse transfer (Windows→Linux text mode)**
   - What we know: The G4 incident was Linux→Windows (LF→CRLF). The reverse (Windows→Linux, CRLF→LF) would REMOVE bytes from zip structures that happen to have `0x0D 0x0A` pairs, NOT add them.
   - What's unclear: Should we detect the reverse direction?
   - RESOLVED: Out of scope for this phase. CONTEXT.md D-01 specifies detecting only "injected 0x0D 0x0A runs"; the reverse direction is covered by the D-03 generic+hint message ("the file may be incomplete or corrupted"). No plan work needed.
