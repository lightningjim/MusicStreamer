## Summary

This phase fixes a PyInstaller-frozen Windows bug where the in-app OAuth/login
launcher used `sys.executable -m musicstreamer.oauth_helper` (silently ignored by
a frozen exe). The core fix â€” `_make_oauth_launch_args` in `subprocess_utils.py`,
the `--oauth-helper` argv dispatch + `_run_oauth_helper` in `__main__.py`, and the
build.ps1 OAUTH HELPER GUARD â€” is correct and well-traced. I verified the argv flow
through `main()` end-to-end:

- `MusicStreamer.exe --oauth-helper --self-test` â†’ argparse consumes `--oauth-helper`,
  `remaining` carries `--self-test`, `_run_oauth_helper` returns 0 before any
  QApplication. Build guard contract holds.
- Normal dispatch â†’ `remaining` = `[prog, --mode, <m>]`, `_run_oauth_helper` sets
  `sys.argv`, `oauth_helper.main()`'s strict `parse_args()` reads `[--mode, <m>]`. Holds.
- The T-40-05 contract (`sys.executable` program, args as list, never `shell=True`,
  mode is a hardcoded literal) is preserved in both branches.
- WR-03 guard-reset ordering on `FailedToStart` (reset `None` FIRST, then
  `_update_status`, then fallback) is correct, and the `FailedToStart`-specific
  gating correctly avoids double-triggering on `Crashed` (which still emits `finished`).

The dominant defect: the fix wired `errorOccurred` handlers into **two** of the
**three** rewired QProcess launch sites. The third â€” `CookieImportDialog._on_google_login`
â€” was switched to the frozen-safe launcher but left WITHOUT an `errorOccurred` handler,
reintroducing the exact silent dead-end this phase exists to eliminate, on the frozen
Windows bundle.

## Critical Issues

### CR-01: CookieImportDialog Google/cookie login has no `errorOccurred` handler â€” silent dead-end on frozen launch failure

**File:** `musicstreamer/ui_qt/cookie_import_dialog.py:292-330`
**Issue:**
`_on_google_login` was rewired in this phase to launch via `_make_oauth_launch_args`
(the frozen-exe `--oauth-helper` dispatch path), exactly like the two `accounts_dialog`
call sites. But unlike those two sites, it connects ONLY `finished` â€” no
`errorOccurred`. Per Qt, when a QProcess fails to start (`FailedToStart`),
`finished` is **never emitted**. The consequences on a frozen Windows bundle where
the dispatch regresses (the precise failure mode this phase guards against):

- `_on_google_process_finished` never runs.
- `self._google_btn` stays disabled, status stuck at `"Logging in..."` forever.
- `self._google_process` is never cleared (non-None), so the dialog is wedged and
  the subprocess reference leaks.
- The user is left at a silent dead-end â€” the symptom this phase's WR-03/D-02 work
  was specifically chartered to remove (cf. `accounts_dialog` GBS fallback + Twitch
  guard reset).

This is a BLOCKER: it is a correctness regression in a rewired call site, in the
exact scenario the phase targets, and it ships an interactive UI into a permanently
hung state.

**Fix:** Mirror the `accounts_dialog` pattern â€” wire `errorOccurred` and reset state
on `FailedToStart`:
```python
def _on_google_login(self) -> None:
    self._hide_error()
    self._google_btn.setEnabled(False)
    self._google_status_label.setText("Logging in...")
    self._google_status_label.setVisible(True)

    process = QProcess(self)
    self._google_process = process
    process.finished.connect(self._on_google_process_finished)
    process.errorOccurred.connect(self._on_google_process_error)  # NEW
    from musicstreamer.subprocess_utils import _make_oauth_launch_args
    program, args = _make_oauth_launch_args(self._oauth_mode or "google")
    process.start(program, args)

def _on_google_process_error(self, error: QProcess.ProcessError) -> None:
    if error == QProcess.ProcessError.FailedToStart:
        self._google_process = None
        self._google_btn.setEnabled(True)
        self._google_status_label.setVisible(False)
        self._show_error("Login helper failed to start.")
```
Add a corresponding source-correctness test alongside the new accounts_dialog
error-handler tests.

## Warnings

### WR-01: Unused `import sys` in cookie_import_dialog after the launch rewrite

**File:** `musicstreamer/ui_qt/cookie_import_dialog.py:31`
**Issue:** The only executable use of `sys` was `sys.executable` in the old
`process.start(sys.executable, [...])` call, which this phase replaced with
`_make_oauth_launch_args(...)`. The remaining `sys` references are in a docstring
(line 25) and a comment (line 302). `import sys` at line 31 is now dead.
**Fix:** Remove `import sys` from line 31.

### WR-02: Twitch `errorOccurred` handler leaves no user-visible signal on launch failure

**File:** `musicstreamer/ui_qt/accounts_dialog.py:654-668`
**Issue:** `_on_oauth_login_error` handles `FailedToStart` by resetting
`self._oauth_proc = None` and calling `_update_status()` (button re-enabled,
"Not connected"), plus a `_log.warning`. Unlike the GBS handler (which surfaces the
cookie-import fallback dialog) and unlike CR-01's recommended fix, there is no
user-facing surface â€” the only signal of a launch failure is a silent flip back to
"Not connected" and a log line the user never sees. A user who clicks "Connect
Twitch" on a broken frozen bundle sees the button momentarily disable then re-enable
with no explanation, which reads as "nothing happened." Twitch has no fallback path
(GBS-anchor only, per D-02), but it should still inform the user.
**Fix:** Surface a brief failure dialog/toast on `FailedToStart`, e.g. reuse
`_show_failure_dialog("twitch", "SubprocessCrash", "Login helper failed to start")`
so the user gets the same category-dialog UX as a post-start failure.

### WR-03: Forwarded Qt flags would break `oauth_helper`'s strict `parse_args` in the frozen dispatch path

**File:** `musicstreamer/__main__.py:48-73, 386-391`
**Issue:** `_run_oauth_helper` strips only `--oauth-helper` from argv, then sets
`sys.argv` and calls `oauth_helper.main()`, which uses `argparse.parse_args()`
(strict â€” errors exit 2 on any unrecognized arg). In the dispatched path, `main()`'s
`parse_known_args` routes any token it does not own into `qt_argv` â†’ `remaining`. I
verified that if a Qt flag (e.g. `-platform offscreen`) is present alongside
`--oauth-helper`, it survives into `remaining` and makes `oauth_helper.main()`
error-exit 2 ("unrecognized arguments: -platform offscreen"). The current call sites
only pass `["--oauth-helper", "--mode", <m>]`, so this is not triggerable today, but
the dispatch arm is documented as the general frozen-exe re-exec entry and is one
launch-site change away from breaking. The stated rationale ("strip `--oauth-helper`
so oauth_helper's argparse cannot error") under-protects: anything OTHER than
`--oauth-helper`/`--mode` reaching `remaining` defeats it.
**Fix:** Forward only the recognized helper args rather than the whole `remaining`
tail, e.g. reconstruct `sys.argv = [argv[0], "--mode", mode]` from a locally parsed
`--mode`, or use `parse_known_args` inside `oauth_helper.main()` and ignore the rest.
At minimum, document that only `--mode <m>` may accompany `--oauth-helper`.

## Info

### IN-01: Strip-`--oauth-helper` in `_run_oauth_helper` is dead in the real dispatch path

**File:** `musicstreamer/__main__.py:70`
**Issue:** In the production path, `main()`'s `parse_known_args` already consumes the
registered `--oauth-helper` store_true flag, so it is absent from `remaining` by the
time `_run_oauth_helper` runs â€” the list-comprehension strip is a no-op except when
`_run_oauth_helper` is called directly (as the unit test does). Harmless belt-and-
braces, but the docstring (Pitfall 1 / T-88.2-03) implies it is load-bearing on the
dispatched path, which it is not. Consider a one-line comment noting the strip only
matters for direct invocation.

### IN-02: Test reloads `subprocess_utils` module, risking cross-test binding staleness

**File:** `tests/test_launcher_frozen_dispatch.py:28-32, 47-49`
**Issue:** `test_frozen_branch_*` / `test_non_frozen_branch_*` call
`importlib.reload(su)` after monkeypatching `sys.frozen`/`sys.executable`. Modules
that did `from musicstreamer.subprocess_utils import _make_oauth_launch_args` at
import time (e.g. `accounts_dialog`) keep the pre-reload function object, so a later
test in the same session could observe a different module identity than the live app
uses. No current failure (the helper reads `sys` at call time, not import time, so
the reload is not even necessary for correctness), but the reload adds global state
churn for no behavioral gain. Consider dropping the `importlib.reload` and asserting
directly â€” `getattr(sys, "frozen", False)` is evaluated per-call.
