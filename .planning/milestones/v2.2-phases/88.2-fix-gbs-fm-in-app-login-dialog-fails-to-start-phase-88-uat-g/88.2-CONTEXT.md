# Phase 88.2: Fix GBS.FM in-app login dialog fails to start (Phase 88 UAT G3) - Context

**Gathered:** 2026-06-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the in-app GBS.FM login so it actually starts on the bundled (PyInstaller-frozen)
Windows build, unblocking playback (UAT-10 / VER-02-J / Phase 88 UAT gap G3). The
identical subprocess-launch mechanism also breaks Twitch and Google/YouTube in-app login
on the frozen build, so this phase repairs the **shared launcher** rather than a single
caller. Scope is the launch mechanism + its failure handling + regression guards — not new
auth providers, not the QtWebEngine login windows themselves (they work once invoked).

**Root cause (identified during discussion scouting):**
`accounts_dialog._launch_gbs_login_subprocess()` (and `_launch_oauth_subprocess()` for
Twitch, and the Google path) launch the login via
`QProcess.start(sys.executable, ["-m", "musicstreamer.oauth_helper", "--mode", "<m>"])`.
In a frozen build `sys.executable` is `MusicStreamer.exe`, **not** a Python interpreter —
it silently ignores `-m musicstreamer.oauth_helper`, the QtWebEngine login window never
opens, and the user sees nothing. `musicstreamer/__main__.py:main()` already multiplexes
alternate entrypoints via argv flags (`--smoke`, `--check-mediakeys`), so a frozen-exe
re-exec dispatch for the login helper fits the existing house style.

</domain>

<decisions>
## Implementation Decisions

### Fix breadth — repair the shared subprocess launcher (all three logins)
- **D-01:** Fix the **one shared `sys.executable -m musicstreamer.oauth_helper` launch
  mechanism** so GBS.FM, Twitch, and Google/YouTube in-app login all work on the frozen
  Windows build. They share the identical broken code path
  (`accounts_dialog._launch_gbs_login_subprocess` line ~405, `_launch_oauth_subprocess`
  line ~383, and the Google/cookie-import oauth launch). GBS.FM / UAT-10 is the
  **verification anchor**; Twitch + Google are fixed as a free consequence of repairing
  the shared launcher. Chosen over a GBS-only patch that would knowingly ship two more
  broken logins.

### Failure UX — graceful fallback to cookie import
- **D-02:** When the login subprocess **fails to start**, the dialog must **auto-surface
  the existing GBS.FM File/Paste cookie-import path** (already wired via
  `_on_gbs_import_clicked` → `CookieImportDialog` with `cookies_path=paths.gbs_cookies_path`)
  with a short explanatory message — the user is never dead-ended on a silent failure.
  This makes the login resilient even if the frozen re-exec regresses again. The Twitch /
  Google equivalents should degrade comparably where a cookie-import fallback exists; GBS
  is the anchor.

### Diagnostics — no more silent "nothing happens"
- **D-03:** Wire QProcess launch-failure detection (`errorOccurred` / `FailedToStart` /
  non-launch) to a **visible log line** so "login fails to start" always leaves a
  diagnosable trace. Mirrors Phase 88.1 D-03's anti-silent-degradation stance (backend
  selection + fallback reason logged at a visible level). The failure log and the D-02
  fallback are driven off the same detected condition.

### Build-time regression guard — two layers (mirrors 88.1 D-04/D-05)
- **D-04:** Add a **Linux-runnable static/unit test** that locks the fix: assert the login
  launch resolves to a working frozen-exe invocation when `sys.frozen` is set (e.g. the
  launcher chooses the re-exec/argv-dispatch form rather than `-m module` under frozen
  conditions). Catches a regression of the root cause in normal CI **without** a Windows
  build — the executor runs on Linux.
- **D-05:** Extend `build.ps1` / the frozen `--smoke`-style harness with a **runtime guard
  that proves the frozen exe can dispatch the oauth-helper entrypoint** (e.g. a headless
  `MusicStreamer.exe <login-helper-flag> --self-test` that exits 0 without opening a
  window), failing the Windows build with a **new exit code** if the helper entrypoint
  isn't reachable. Follows the existing exit-code-guard house style (exit 3 = smoke,
  8/9/10 = bundle guards, 11 = SMTC from 88.1).
- **Why both:** the static test guards the launcher logic in CI; the runtime smoke
  assertion proves the frozen exe actually reaches the helper entrypoint on the build
  machine.

### Completion boundary — Linux fix+guard now, frozen-Windows UAT deferred to 88-03
- **D-06:** Phase 88.2 **closes on the Linux/CI side**: the shared-launcher fix + D-02
  fallback + D-03 logging + the D-04 Linux static test land and pass in CI. The executor
  runs on Linux and **cannot** run the Windows build or click through the QtWebEngine
  login.
- **D-07:** The frozen-Windows **UAT-10 re-test** (GBS.FM in-app login opens, authenticates,
  audio plays) and the `build.ps1` D-05 runtime guard both run in the **consolidated
  Win11 VM session (88-03)**, alongside the 88.1 VM-deferred items. `must_haves` MUST be
  split:
  - **Linux-verifiable now:** launcher chooses a frozen-safe invocation under `sys.frozen`;
    D-04 static test passes; launch-failure logging + cookie-import fallback wired and
    unit-tested; `build.ps1` contains the new helper-entrypoint smoke assertion +
    documented exit code.
  - **VM-deferred (88-03):** frozen `MusicStreamer.exe` opens the GBS.FM login window,
    login succeeds, GBS.FM audio plays (UAT-10); Twitch + Google login also open from the
    frozen exe.

### Claude's Discretion
- The exact frozen-exe dispatch shape — preferred approach is an argv-multiplexed flag in
  `__main__.py` (e.g. `MusicStreamer.exe --oauth-helper --mode gbs`) routed to
  `oauth_helper.main()` before the GUI builds, with the launcher choosing
  `sys.executable <flag>` when `sys.frozen` else the current `-m` form. Researcher/planner
  may confirm vs. alternatives (PyInstaller multipackage second exe, in-process QtWebEngine
  — the latter discouraged; Phase 76 chose a subprocess specifically to isolate
  QtWebEngine).
- The exact new `build.ps1` exit-code number and the precise self-test mechanism.
- Log level / wording for launch-failure diagnostics.
- Whether `runtime_hook.py` needs any env setup for the re-exec'd helper process.

</decisions>
