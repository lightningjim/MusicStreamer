<objective>
Close the major UAT defect (87.1-HUMAN-UAT.md `## Gaps`, Test 1, severity major, status diagnosed): the inline "GBS session expired" prompt now appears correctly on real expiry, but the `oauth_helper --mode gbs` login window **auto-launches on detection** instead of waiting for the user to click "Log in again". This defeats the locked user-initiated, non-dismissive contract (D-01/D-02 / GBS-AUTH-EXP-01) and makes UAT Test 2 (the cancel flow) untestable.

Root cause (pinned from the UAT, NOT to be re-investigated):
1. `now_playing_panel.py:_on_gbs_playlist_error` auth_expired branch reveals the prompt AND THEN calls `self._gbs_relogin_handler.notify_expiry_detected()`, which unconditionally launches the subprocess (only single-flight guarded, not user-intent guarded).
2. `attach_gbs_marquee_worker` wires `worker.auth_expired` straight to `handler.notify_expiry_detected`, so the marquee poller also auto-launches the login window with no user action.

Purpose: separate "a consumer detected expiry → ensure the inline prompt is shown" from "the user clicked 'Log in again' → launch the subprocess". Detection events must NEVER spawn the window; only the button does.

Output: a new shared `_on_gbs_expiry_detected` slot on the panel that reveals the prompt (the existing reveal logic, extracted/reused), both detection paths rewired to it, the launch left solely on `_on_gbs_relogin_clicked`, and a dual-path regression test observing QProcess start.
</objective>
