# Phase 87.1 Plan 04: _open_with_cookies Auth-Expiry Detection Fix Summary

Fix the upstream auth-expiry DETECTION in `_open_with_cookies` that the inline re-login prompt (Plans 01-03) depends on. urllib's default `HTTPRedirectHandler` was auto-following 302 → /accounts/login/, making the detection branch dead code for GET redirects. Also fixed a marquee propagation defect found during audit.

## What Was Built

**Task 1 (RED):** Added 5 real-chain regression tests in `tests/test_gbs_api.py` (plus `_make_fake_200_response` helper) that patch at `urllib.request.AbstractHTTPHandler.do_open` — the same deep-transport level as the existing T13 tests — so the full opener chain (including redirect handlers) runs against real CPython code. Two tests fail RED (login-redirect cases), two pass (non-login redirect guard, 200 happy-path), one marquee propagation test added in Task 3.

**Task 2 (GREEN):** Fixed `_open_with_cookies` in `musicstreamer/gbs_api.py`. Detection approach: **PREFERRED — no-redirect handler** mirroring the existing `_open_no_redirect` pattern. A `_NoRedirect` inner class overrides `http_error_302/301/303/307` to return `fp` (raw response) instead of following. After `opener.open()` returns with a stopped 3xx response, the Location header is inspected:
- If `/accounts/login/` is in Location → `GbsAuthExpiredError` raised
- If non-login 3xx → re-issued with a default redirect-following opener (preserves legitimate redirects)
- The original `except HTTPError/302` belt-and-braces block is retained for error-style redirects

The ALTERNATIVE approach (`geturl()` check after auto-follow) was not used because the fake `do_open` returning a 302 repeatedly causes infinite redirect loops in tests, confirming the no-redirect approach is more robust.

**Task 3 (AUDIT):** Traced `_fetch_marquee` (gbs_marquee.py). Found that `GbsAuthExpiredError` (a subclass of `GbsApiError → Exception`) was caught and swallowed by the generic `except Exception` belt-and-suspenders before it could propagate to `GbsMarqueeWorker._on_tick`'s `except gbs_api.GbsAuthExpiredError` block (which emits `auth_expired`). Added a narrow `except gbs_api.GbsAuthExpiredError: raise` BEFORE the generic handler. Added `test_fetch_marquee_propagates_auth_expired_error` using the same do_open pattern to pin the propagation contract.

## Detection Approach Choice

Used the **no-redirect handler** (PREFERRED). Justification:
1. Mirrors the already-existing `_open_no_redirect` in the same file — consistent, familiar pattern
2. Stops the chain at the first 302 so Location is visible before any redirect is followed
3. Avoids the infinite-loop failure mode when `do_open` returns a 302 response to each re-try (observed in RED test runs)
4. Enables clean inspect-then-re-follow: non-login 3xx is re-issued with a follow-opener, preserving existing behavior for legitimate redirects

## Marquee Audit Finding

`_fetch_marquee` docstring stated `GbsAuthExpiredError` "propagates to caller (`GbsMarqueeWorker._on_tick`)". This was **incorrect** — the generic `except Exception` (line 362 pre-fix) swallowed it. `_on_tick`'s `except gbs_api.GbsAuthExpiredError` block (which emits `auth_expired` Signal per GBS-AUTH-EXP-02) was therefore unreachable even after the Task 2 detection fix. Fixed with a narrow re-raise before the generic handler.
