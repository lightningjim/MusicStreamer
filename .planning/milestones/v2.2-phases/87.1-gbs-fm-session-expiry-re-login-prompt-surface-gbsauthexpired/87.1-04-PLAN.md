<objective>
Fix the upstream auth-expiry DETECTION that the Phase 87.1 inline re-login prompt depends on. Human UAT (87.1-HUMAN-UAT.md, gap GBS-AUTH-EXP-01) reproduced live that `_open_with_cookies` never raises `GbsAuthExpiredError` on a real expired session: urllib's default `HTTPRedirectHandler` auto-follows the server's `302 → /accounts/login/` and returns the resulting `200` login page, making the existing `except HTTPError`/302 branch (gbs_api.py:169-172) DEAD CODE for GET redirects. `fetch_active_playlist` then runs `json.loads(login_page_html)` → `JSONDecodeError`, which `_GbsPollWorker.run` routes to the generic-error (silent-fallback) branch — so the correctly-built, unit-tested inline prompt can never appear in production.

Purpose: Make detection actually fire so the entire downstream UX (already shipped in plans 01-03) becomes reachable. One fix in `_open_with_cookies` repairs every GET caller (`fetch_active_playlist`, `fetch_user_tokens`, `search`, `fetch_artist_songs`, `fetch_album_songs`, and the marquee `_fetch_marquee`).

Output: A repaired `_open_with_cookies` plus a regression test that exercises the REAL urllib redirect chain (the existing tests mock `_open_with_cookies` itself, which is exactly why this bug passed verification).
</objective>
