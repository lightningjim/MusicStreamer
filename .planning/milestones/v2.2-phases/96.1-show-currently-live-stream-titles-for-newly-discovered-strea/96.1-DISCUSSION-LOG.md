## Symptom (what's actually wrong today) — multiSelect

| Option | Description | Selected |
|--------|-------------|----------|
| Titles show blank/wrong | New streams appear but title is empty/'Untitled'/stale — can't tell which is which | ✓ |
| Titles buried/unclear | Titles present but unclear which are NEW vs map to existing stations | |
| Can't merge into existing | New stream can only be ADDed, not pointed onto an existing station | |
| Duplicates / double-listed | Same stream shows in a station dropdown AND as an ADD row | |

**User's choice:** Titles show blank/wrong.
**Notes:** Confirms the real gap is data-fetch: `scan_playlist` returns blank/`"Untitled"` for just-started lives (flat `/streams` tab). Pre-question scouting had verified the scan returns titles and the dialog displays them — so the defect is the *quality* of the flat-scan title for live entries, not a missing display.

## Claude's Discretion

- Exact "blank" predicate (empty vs `"Untitled"` vs whitespace); session caching of resolved titles.
- Concurrent vs sequential blank-resolution; progress affordance while resolving.
- Section placement/ordering and precise row layout.
- Location of the single-video title-resolution helper (new `yt_import` helper vs reuse `player._youtube_resolve_worker`).

## Deferred Ideas

- Map/merge into ANY station (not just flagged).
- Proactive in-UI conflict greying of already-targeted stations.
- Thumbnail / "started X ago" disambiguators.
- Always-resolve-every-entry titles (rejected for cost/fragility).
