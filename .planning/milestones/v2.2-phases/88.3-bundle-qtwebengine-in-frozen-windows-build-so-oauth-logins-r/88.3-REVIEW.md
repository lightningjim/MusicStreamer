# Phase 88.3: Code Review Report

**Reviewed:** 2026-06-13
**Depth:** standard (all ultracode dimensions)
**Files Reviewed:** 13
**Status:** issues_found

## Summary

Reviewed the B1 isolated-helper-bundle implementation: `oauth_helper.py` frozen as its
own PyInstaller exe from an isolated pip-PySide6 6.11.0 env, the conda main bundle
shipping no WebEngine, `build.ps1` building both artifacts with conda/GStreamer stripped
off PATH and a SHA256 ABI-match assertion, and the Inno installer shipping both artifacts.

The core architecture is sound. The launch-target contract (`_make_oauth_launch_args`),
the spec WebEngine partitioning, the PATH-sanitization with restore-on-every-exit, and
the test drift-guards are all correct under adversarial tracing. `build.ps1` is clean
ASCII (the PS 5.1 / cp1252 constraint holds — verified byte-scan, no non-ASCII found).

No BLOCKER-class defects survived refutation. The findings are: (1) the build's central
G6-regression-catching ABI guard can silently no-op and ship a broken bundle, (2) a
reachable-but-untriggered `sys.exit(2)` trap in the conda process's `--oauth-helper`
non-self-test arm, (3) two stale/incorrect README facts (PySide6 version, install path)
that will mislead a future maintainer, plus test-state-leak and documentation-drift items.

## Warnings

### WR-01: ABI-coherence guard silently degrades to a no-op (ships a potentially broken bundle)

**File:** `packaging/windows/build.ps1:493-506`
**Issue:** The SHA256 ABI-match check between the bundled `Qt6Core.dll` and the venv's
`Qt6Core.dll` is the documented "G6 root-cause regression catch" (the entire rationale of
the PATH-sanitization block). But it runs only inside
`if ((Test-Path $bundledQt6Core) -and (Test-Path $venvQt6Core))`. If either path fails to
resolve — e.g. a future PyInstaller version changes the `_internal\PySide6\` layout, or the
venv site-packages path differs — the `else` branch merely prints a yellow warning and the
build continues to Inno Setup. The exact failure this guard exists to catch (a wrong-ABI
`Qt6Core.dll` baked in from a contaminated PATH) could coincide with a layout change that
makes `$bundledQt6Core` unresolvable, so the guard skips and the broken helper ships. A
guard whose miss-mode is "pass silently" provides false confidence.
**Fix:** Treat an unresolvable path as a hard failure, not a skip:
```powershell
if (-not ((Test-Path $bundledQt6Core) -and (Test-Path $venvQt6Core))) {
    Write-Host "BUILD_FAIL reason=helper_qt6core_abi_guard_indeterminate bundled='$bundledQt6Core' venv='$venvQt6Core' hint='could not locate both Qt6Core.dll copies; layout may have changed -- cannot prove ABI coherence'" -ForegroundColor Red
    exit 14
}
# ... then the hash comparison as today
```

### WR-02: `_run_oauth_helper` non-self-test arm hits a `sys.exit(2)` trap inside the conda bundle

**File:** `musicstreamer/__main__.py:65-73` (with `musicstreamer/oauth_helper.py:77-86`)
**Issue:** Under B1 the conda `MusicStreamer.exe` never carries WebEngine, yet
`_run_oauth_helper`'s non-`--self-test` branch does `from musicstreamer.oauth_helper import
main` (line 71). Importing `oauth_helper` triggers its module-level
`from PySide6.QtWebEngineWidgets import QWebEngineView`, which in the conda bundle is
guaranteed to be absent and calls `sys.exit(2)` (oauth_helper.py:86). So
`MusicStreamer.exe --oauth-helper --mode twitch` (without `--self-test`) would hard-exit 2
from the wrong process. The B1 launcher (`_make_oauth_launch_args`) never produces this
invocation, so it is not currently triggered — but it is reachable dead-trap code: any
future caller, manual invocation, or test that passes `--oauth-helper --mode X` to the
conda exe gets a bare exit 2 with no diagnostic. The function's own docstring (line 56-60)
even describes this branch as the live "Normal" path, which is misleading under B1.
**Fix:** Either remove the non-self-test branch entirely (B1 never uses it in the conda
process) and emit a clear diagnostic, or guard the import:
```python
if "--self-test" in argv:
    return 0
# B1: the conda exe must NEVER run the WebEngine helper in-process. It launches the
# separate oauth_helper.exe via QProcess. Any other --oauth-helper invocation is a bug.
print("ERROR: --oauth-helper in-process dispatch is not supported in the conda bundle "
      "(B1: use the separate oauth_helper.exe).", file=sys.stderr, flush=True)
return 2
```

### WR-03: README states the wrong helper PySide6 version (6.10.1 vs pinned 6.11.0)

**File:** `packaging/windows/README.md:202`
**Issue:** The file-map row reads `oauth-helper-requirements.txt | B1: isolated pip env
pins (PySide6 6.10.1 + pyinstaller)`. The requirements file pins `6.11.0`
(`oauth-helper-requirements.txt:40-41`), and the 6.10.1→6.11.0 bump is the explicit
88.3-05 Kasada fix (commit `46f30049`, and the long rationale comment in the requirements
file lines 11-17). A maintainer reading the README would believe the helper still runs
Chromium 134, which is exactly the version Twitch's Kasada rejects — the bug this phase
fixed. Stale docs here re-introduce the diagnosis dead-end the phase set out to remove.
**Fix:** Update the README row to `PySide6 6.11.0`:
```
| `oauth-helper-requirements.txt`  | B1: isolated pip env pins (PySide6 6.11.0 + pyinstaller)   |
```

### WR-04: README states the wrong install location (`%LOCALAPPDATA%\MusicStreamer`)

**File:** `packaging/windows/README.md:134`
**Issue:** "Install behavior" says the installer installs `to %LOCALAPPDATA%\MusicStreamer`.
The Inno script installs to `{userpf}\MusicStreamer`, which Inno itself documents as
`%LOCALAPPDATA%\Programs\MusicStreamer` (`MusicStreamer.iss:20, 26`). The `.iss` header
(lines 20-25) explicitly moved OFF `{localappdata}\MusicStreamer` because it collided with
the platformdirs `musicstreamer` data folder on case-insensitive NTFS and broke
uninstall. The README still documents the abandoned, broken path. This will misdirect
anyone debugging an install/uninstall issue to the wrong directory.
**Fix:** Correct the path to match the `.iss`:
```
to `%LOCALAPPDATA%\Programs\MusicStreamer`, and adds a single Start Menu shortcut.
```

### WR-05: `subprocess_utils` docstring + `.iss` comment reference a nonexistent function/module

**File:** `musicstreamer/subprocess_utils.py:44` and `packaging/windows/MusicStreamer.iss:68`
**Issue:** Both comments claim the path is resolved by `_oauth_helper_exe()` in
`musicstreamer/oauth_launcher.py`. Neither exists: there is no `oauth_launcher.py` in the
package (`ls` confirms ENOENT), and the resolver is `_make_oauth_launch_args` in
`subprocess_utils.py`. The `.iss` comment is load-bearing documentation that ties the
install path to the launcher's expectation; pointing it at a phantom function means a
maintainer verifying the path contract cannot find the referenced code. The actual contract
IS correct (helper installs to `{app}\oauth_helper\oauth_helper.exe`; launcher resolves
`Path(sys.executable).parent / "oauth_helper" / "oauth_helper.exe"` =
`{app}\oauth_helper\oauth_helper.exe`), so this is a doc defect, not a path bug.
**Fix:** Replace both references with `musicstreamer/subprocess_utils.py::_make_oauth_launch_args`.

### WR-06: test mutates global `sys.argv` without restoring it (cross-test state leak)

**File:** `tests/test_launcher_frozen_dispatch.py:110-119`
**Issue:** `test_run_oauth_helper_strips_oauth_helper_flag` calls
`_run_oauth_helper(["prog", "--oauth-helper", "--mode", "gbs"])`, which executes
`sys.argv = [a for a in argv if a != "--oauth-helper"]` (`__main__.py:70`), permanently
replacing the process `sys.argv` with `["prog", "--mode", "gbs"]`. The test never restores
it (no `monkeypatch.setattr(sys, "argv", ...)`), so any later test in the same worker that
reads `sys.argv` sees the mutated value. This is a latent flaky-ordering hazard. The sibling
test in `test_oauth_launch_path.py` correctly uses `monkeypatch` for all `sys` mutations;
this one should too.
**Fix:** Guard `sys.argv` via monkeypatch so it auto-restores:
```python
def test_run_oauth_helper_strips_oauth_helper_flag(monkeypatch):
    monkeypatch.setattr(sys, "argv", list(sys.argv))  # snapshot+auto-restore
    ...
```

## Info

### IN-01: `importlib.reload(su)` in frozen-dispatch test is unnecessary

**File:** `tests/test_launcher_frozen_dispatch.py:40-42, 61-63`
**Issue:** Both tests `importlib.reload` `subprocess_utils` after monkeypatching `sys`, with
a comment claiming the module reads `sys` "at module scope." It does not —
`_make_oauth_launch_args` reads `getattr(sys, "frozen", False)` and `sys.executable` at call
time (verified in `subprocess_utils.py:68-72`). The reload is dead ceremony that risks
import-cache surprises and contradicts the (correct) comment in `test_oauth_launch_path.py`
which says the function "reads sys.frozen / sys.executable on each call, not at import time."
**Fix:** Delete the `importlib.reload` lines and the misleading comment; import the function
directly like `test_oauth_launch_path.py` does.

### IN-02: `_run_oauth_helper` returns unreachable `0` after `_oauth_main()`

**File:** `musicstreamer/__main__.py:72-73`
**Issue:** `_oauth_main()` calls `sys.exit` internally, so `return 0` on line 73 is
unreachable in production (the code comment acknowledges this). Harmless, but it is dead
code that the test suite only reaches because the test stub raises `SystemExit` which the
test swallows. Low priority; documented as such.
**Fix:** Leave as-is or convert to `raise AssertionError("oauth_helper.main() did not exit")`
to make a contract violation loud rather than silently returning success.

### IN-03: `README.md` table row PySide6 version drift is untested

**File:** `tests/test_packaging_helper_build_spec.py:346-375`
**Issue:** `test_helper_requirements_pinned` asserts the `6.11.0` pin in the requirements
file, and `test_readme_documents_conda_free_python` asserts the conda-free prereq prose,
but nothing guards the README's file-map version string against drift from the requirements
pin. WR-03 (the 6.10.1 staleness) slipped through precisely because no test couples the two.
**Fix:** Add an assertion that the README does not mention a PySide6 version other than the
pinned one, or that it omits a version number from that row entirely.

### IN-04: `_GoogleWindow` success path emits no structured `Success` event

**File:** `musicstreamer/oauth_helper.py:435-440`
**Issue:** `_TwitchCookieWindow` and `_GbsLoginWindow` both emit `_emit_event("Success", ...)`
on the happy path; `_GoogleWindow._flush_cookies` does not (it just prints cookies and
quits). The caller's `_parse_oauth_stderr` keys off the last `category`-bearing event, so a
successful Google login produces no `Success` diagnostic, only the implicit exit 0. The file
marks the Google path "UNTOUCHED — working as-is," so this is consistency/observability
debt, not a regression introduced by this phase.
**Fix:** For diagnostic parity, emit `_emit_event("Success", detail="")` before
`QApplication.quit()` in `_GoogleWindow._flush_cookies` (out of scope for 88.3 if the Google
contract is frozen — note only).
