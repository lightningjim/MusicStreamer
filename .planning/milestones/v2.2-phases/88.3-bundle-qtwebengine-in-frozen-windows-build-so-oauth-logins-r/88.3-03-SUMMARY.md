# Phase 88.3 Plan 03: Rewire OAuth Launch Path (B1) + Twitch Platform UA Summary

**One-liner:** B1 launch target rewired — `_make_oauth_launch_args` frozen branch now resolves the sibling `oauth_helper.exe` path instead of self-re-execing; `_CHROME_UA` made platform-appropriate (Windows NT token on win32) to fix Twitch UA rejection.

## What Was Built

### Task 1: Point the frozen launch target at the separate oauth_helper.exe

`musicstreamer/subprocess_utils.py` — `_make_oauth_launch_args` frozen branch rewritten:

**Before (B0 / Phase 88.2):**
```python
if getattr(sys, "frozen", False):
    return sys.executable, ["--oauth-helper", "--mode", mode]
```
This re-executed `MusicStreamer.exe --oauth-helper`, forcing QtWebEngine into the conda bundle and causing the Phase 88.3 G6 DLL-load-failed ABI conflict.

**After (B1 / this plan):**
```python
if getattr(sys, "frozen", False):
    from pathlib import Path
    helper = Path(sys.executable).parent / "oauth_helper" / "oauth_helper.exe"
    return str(helper), ["--mode", mode]
```
The conda main exe now spawns the **separate** `oauth_helper.exe` built from the isolated pip PySide6-Addons env (spike 001). The helper carries its own WebEngine Qt bundle, completely decoupled from the conda stack.

Key properties:
- Path resolution: `{app}\oauth_helper\oauth_helper.exe` (Inno install layout from 88.3-04)
- No `--oauth-helper` self-dispatch flag (removed — it was only needed for self-re-exec)
- Source/dev/Linux branch: preserved byte-for-byte (`-m musicstreamer.oauth_helper --mode <mode>`)
- `accounts_dialog.py`: untouched — `errorOccurred` → cookie-import fallback still fires on FailedToStart
- T-40-05 contract: fixed sibling path (no PATH lookup), args as list, mode as call-site literal

### Task 2: Make _CHROME_UA platform-appropriate (Twitch Windows fix)

`musicstreamer/oauth_helper.py` — hardcoded Linux UA replaced with platform branch:

```python
if sys.platform == "win32":
    _CHROME_UA = "Mozilla/5.0 (Windows NT 10.0; Win64; x64) ..."
else:
    _CHROME_UA = "Mozilla/5.0 (X11; Linux x86_64) ..."  # unchanged
```

Spike 001 finding: on the frozen Windows exe, Twitch's server rejected login with "Your browser is not currently supported". Root cause: the UA advertised `(X11; Linux x86_64)` but the real Chromium emitted `Sec-CH-UA-Platform: "Windows"` — the mismatch triggered Twitch's UA check. The Linux token works on Linux because Chromium's platform hint matches. Fix: advertise the Windows NT token on win32.

`QTWEBENGINE_CHROMIUM_FLAGS` assignment preserved before any QtWebEngine import (load-bearing order).
