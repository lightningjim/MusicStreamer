# Phase 40.1: Code Review Report

**Reviewed:** 2026-04-14
**Depth:** standard
**Files Reviewed:** 13
**Status:** issues_found

## Summary

Phase 40.1 is a bug-fix sweep with generally solid Qt discipline: worker threads are stored as instance attrs for lifetime safety, cross-thread signal emissions use queued connections (explicitly or via Signal default), `Qt.PlainText` is enforced on ICY labels, and resource/temp-file cleanup is largely handled. No security issues and no critical bugs.

The main concerns cluster around `_LogoFetchWorker` in `edit_station_dialog.py`:
1. Rapid URL typing creates orphaned running workers whose tmp files leak.
2. Dialog close does not wait for in-flight workers, so tmp files from late completions are leaked and `QThread: Destroyed while thread is still running` warnings are possible.
3. `_on_logo_fetched` does not unlink tmp on the exception path.

A minor defensive-coding gap in `NowPlayingPanel._on_cover_art_ready` (`int()` without try/except) is also flagged. Tests are well-structured and include strong regression locks for 40.1-01, 40.1-05, and 40.1-06.

## Warnings

### WR-01: `_LogoFetchWorker` orphan on rapid URL typing — tmp file leak

**File:** `musicstreamer/ui_qt/edit_station_dialog.py:435-444`
**Issue:** When the debounce timer fires while a prior worker is still running, `_on_url_timer_timeout` disconnects the previous worker's `finished` signal and creates a new one. The orphaned `QThread` keeps running `urllib.request.urlretrieve` to completion; the tmp file it creates is never consumed and never unlinked. Under bad-network conditions (slow DNS, stalled HTTP) multiple orphans accumulate and leak `.jpg`/`.png` files into the OS temp dir.

**Fix:** Use a monotonically-increasing fetch token (mirror `NowPlayingPanel._cover_fetch_token`) so the slot always runs and can unlink stale tmp paths:

```python
# __init__
self._logo_fetch_token: int = 0

# _on_url_timer_timeout
self._logo_fetch_token += 1
token = self._logo_fetch_token
self._logo_fetch_worker = _LogoFetchWorker(url, token, self)
self._logo_fetch_worker.finished.connect(self._on_logo_fetched)  # no disconnect
self._logo_fetch_worker.start()

# _on_logo_fetched(self, tmp_path: str, token: int)
if token != self._logo_fetch_token:
    if tmp_path:
        try: os.unlink(tmp_path)
        except OSError: pass
    return
# ... existing success path ...
```

### WR-02: `_LogoFetchWorker` survives dialog close — tmp file leak + potential QThread destroyed warning

**File:** `musicstreamer/ui_qt/edit_station_dialog.py:517-531`
**Issue:** `closeEvent` and `reject` disconnect `finished` but do not `wait()` on the running worker. If `urlretrieve` completes after the dialog's Python object is garbage collected, the emitted tmp path is dropped (file leaked). If the `QThread` C++ object is destroyed while still running, Qt logs `QThread: Destroyed while thread is still running` and abort semantics are implementation-defined.

**Fix:** Either (a) bound-wait with a short timeout on close:

```python
def closeEvent(self, event):
    if self._logo_fetch_worker is not None and self._logo_fetch_worker.isRunning():
        # Let the worker finish HTTP so its tmp file can be cleaned up,
        # but cap the wait so UI close is never blocked long.
        self._logo_fetch_worker.finished.disconnect()
        self._logo_fetch_worker.wait(2000)
    super().closeEvent(event)
```

or (b) pair with WR-01's token scheme and let `_on_logo_fetched` unlink stale tmp files even after the dialog is a zombie (bind the slot to a module-level janitor, not `self`).

### WR-03: `_on_logo_fetched` does not unlink tmp on exception paths

**File:** `musicstreamer/ui_qt/edit_station_dialog.py:489-509`
**Issue:** If `assets.copy_asset_for_station` raises, `os.unlink(tmp_path)` on line 507 is never reached. Less critically, the `if not tmp_path or not os.path.exists(tmp_path)` branch at line 491 also does not attempt unlink on the truthy-but-missing path case (race / external deletion) — safe today but fragile.

**Fix:** Wrap the consumption in try/finally:

```python
def _on_logo_fetched(self, tmp_path: str) -> None:
    self._fetch_logo_btn.setEnabled(True)
    if not tmp_path or not os.path.exists(tmp_path):
        # existing fetch-failed branch
        ...
        return
    try:
        old_path = self._logo_path
        rel = assets.copy_asset_for_station(self._station.id, tmp_path, "station_art")
        self._logo_path = rel
        self._refresh_logo_preview()
        self._invalidate_cache_for(old_path)
        self._logo_status.setText("Fetched")
    finally:
        try: os.unlink(tmp_path)
        except OSError: pass
```

### WR-04: `_on_cover_art_ready` crashes event loop on non-numeric token

**File:** `musicstreamer/ui_qt/now_playing_panel.py:476-479`
**Issue:** `int(token_str)` has no fallback. All current emission sites are controlled, so this cannot fire today, but a future refactor to `cover_art.py` or the adapter could emit a malformed payload and `ValueError` would propagate out of the slot. Defensive code here is cheap and preserves the slot-never-raises contract recommended for Qt queued connections.

**Fix:**
```python
def _on_cover_art_ready(self, payload: str) -> None:
    token_str, _, path = payload.partition(":")
    try:
        token = int(token_str)
    except ValueError:
        return
    if token != self._cover_fetch_token:
        return
    if not path:
        self._show_station_logo_in_cover_slot()
        return
    self._set_cover_pixmap(path)
```

## Info

### IN-01: `_on_fetch_logo_clicked` can race with debounce-triggered fetch

**File:** `musicstreamer/ui_qt/edit_station_dialog.py:446-453`
**Issue:** Manual "Fetch from URL" stops the timer then calls `_on_url_timer_timeout`, which disables the fetch button. But between the click arriving and the slot executing, a timer-fired fetch could already be in flight. Behavior is safe (the new fetch replaces it via the existing disconnect path) but see WR-01 for the tmp-file cost.
**Fix:** Adopting the WR-01 token pattern resolves this incidentally.

### IN-02: `_AaImportWorker.progress` callback relies on queued connection

**File:** `musicstreamer/ui_qt/import_dialog.py:138-152`
**Issue:** `on_progress=lambda cur, tot: self.progress.emit(cur, tot)` is invoked from the worker thread because `aa_import.import_stations_multi` runs on that thread. This is safe because `self.progress.connect(..., Qt.QueuedConnection)` on line 425 is explicitly queued. No bug — just worth a comment so future maintainers do not switch to `Qt.DirectConnection`.
**Fix:** Add a short comment at line 144 noting the queued-connection requirement.

### IN-03: `test_yt_scan_passes_through` starts a real QThread (timing dependency)

**File:** `tests/test_import_dialog_qt.py:197-224`
**Issue:** All other import-dialog tests explicitly bypass threading per the module docstring strategy; this one starts a real worker with a 3s `waitSignal` timeout. It is the correct choice for this specific regression (the bug was in `_YtScanWorker.run`, not in a direct slot), but a future maintainer might "simplify" it into a direct-slot test and reopen the 40.1-01 regression.
**Fix:** Add a short comment explaining why this test intentionally exercises the worker thread.

### IN-04: `_logo_status` QLabel has no `Qt.PlainText` lockdown

**File:** `musicstreamer/ui_qt/edit_station_dialog.py:179`
**Issue:** `_logo_status` only ever displays fixed strings ("Fetching…", "Fetched", "Fetch failed", "Fetch not supported for this URL", "Enter a URL first"), so there is no injection surface today. Noted only for consistency with the project's `Qt.PlainText` defense-in-depth pattern (`icy_label`, `name_provider_label`).
**Fix (optional):** `self._logo_status.setTextFormat(Qt.PlainText)` during construction.
