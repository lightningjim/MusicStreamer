# Phase 40.1 Plan 06: Playing Timer Summary

Restored the v1.5 now-playing elapsed-time display (stuck at 0:00 since the v2.0 Qt port) by adding a 1Hz wall-clock QTimer producer for `Player.elapsed_updated`.

## Root Cause

`Player.elapsed_updated = Signal(int)` was declared in `musicstreamer/player.py` and wired `main_window.py:141 → NowPlayingPanel.on_elapsed_updated`. The consumer slot correctly formats seconds. **But no producer ever emitted the signal.** Grep verified: zero `elapsed_updated.emit` call sites anywhere in the codebase before this plan. The signal was a declared-and-wired no-op left over from the Qt port.

## Implementation

1. Added `self._elapsed_seconds: int = 0` and `self._elapsed_timer = QTimer(self)` (1000ms interval) in `Player.__init__`, wired `timeout → _on_elapsed_tick`.
2. Added `_on_elapsed_tick`: `self._elapsed_seconds += 1; self.elapsed_updated.emit(self._elapsed_seconds)`.
3. In `_try_next_stream`, seeded the timer only when `_is_first_attempt` is True (fresh user-initiated play). **Failover deliberately does not reset** — user sees continuous elapsed time across transparent stream swaps.
4. `pause()` stops the timer (no reset).
5. `stop()` stops the timer AND resets `_elapsed_seconds` to 0.
6. Timer is intentionally **not** included in `_cancel_timers()` — that helper is called by GStreamer bus handlers during failover recovery and must not pause elapsed time.

## Signal Semantics

| Event | Effect on counter | Effect on timer |
|-------|-------------------|-----------------|
| `play(station)` / `play_stream(stream)` | reset to 0 | start |
| failover (next stream attempt) | keep value | keep running |
| `pause()` | keep value | stop |
| `stop()` | reset to 0 | stop |
| 1Hz tick | increment, emit `(n)` | (continues) |
