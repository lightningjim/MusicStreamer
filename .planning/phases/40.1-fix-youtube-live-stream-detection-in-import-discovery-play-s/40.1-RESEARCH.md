## Summary

All five bugs are regression regressions introduced during the v2.0 Qt port. Every bug has a clear root cause that can be verified directly in the codebase — no speculative diagnosis required.

**Fix 1 (YT import):** `_YtScanWorker.run()` in `import_dialog.py:64` applies a post-filter `[e for e in results if e.get("is_live") is True]` after calling `yt_import.scan_playlist()`. But `scan_playlist()` already filters via `_entry_is_live()` and the returned dicts have keys `{"title", "url", "provider"}` — no `is_live` key. Every result gets dropped. Fix: delete line 64, pass `results` directly to `finished.emit(live)` (rename to `finished.emit(results)`).

**Fix 2 (Discovery icons):** `_on_search_finished` creates `QPushButton("Play")` and later `.setText("Stop")` in `_on_play_row`. Replace each play button with a single toggle button using `QIcon.fromTheme("media-playback-start", ...)` swapping to `"media-playback-stop"`. The bundled SVG set already ships `-symbolic` variants; the DiscoveryDialog uses non-symbolic names. Either add non-symbolic aliases to icons.qrc or reference the `-symbolic` variants.

**Fix 3 (AA/YT feedback):** `_AaImportWorker.finished = Signal(int)` and `_YtImportWorker.finished = Signal(int)` both discard the skipped count. The `run()` methods already extract only `result[0]`. Fix: change both to `Signal(int, int)`, emit both values, update `_on_aa_import_complete` and `_on_yt_import_complete` to show context-sensitive wording per D-07/D-08/D-09/D-10.

**Fix 4 (Station logos):** `_load_scaled_pixmap(path, size)` in `now_playing_panel.py` takes a relative path from DB (e.g. `assets/12/station_art.png`) but `QPixmap(path)` needs an absolute path. The v1.5 code called `os.path.join(DATA_DIR, rel_path)` before passing to the image loader. The Qt port uses the relative path directly. Same issue in `station_list_panel._load_station_icon()`. Fix: resolve relative paths against `paths.data_dir()` before passing to `QPixmap()`. For `EditStationDialog`: add logo picker row with `QFileDialog` + auto-fetch worker.

**Fix 5 (ICY disable):** `station.icy_disabled` is persisted correctly (verified: `repo.update_station()` writes `icy_disabled`, `list_stations()` reads `bool(r["icy_disabled"])`). The bug is in `Player._on_gst_tag()` and `NowPlayingPanel.on_title_changed()` — neither checks `icy_disabled`. The Player doesn't hold a reference to the current station at all; it only stores `_current_station_name`. The panel holds `_station` but never gates the cover art fetch or title emission on `_station.icy_disabled`. Fix is in the panel's `on_title_changed()` — check `self._station.icy_disabled` before calling `self._fetch_cover_art_async()` and before updating `icy_label`. The Player's `_on_gst_tag` handler can't easily check station state (runs on GLib thread, has no station reference); the simplest correct approach is to suppress in `on_title_changed` which runs on the Qt main thread.

**Primary recommendation:** All five fixes are surgical and isolated. Plan two waves: Wave 1 for the three import/UI fixes (no cross-file dependencies), Wave 2 for logos + ICY disable (require coordinated changes across panel + dialog/player).

### Fix 1: YT Import Double-Filter

**File:** `musicstreamer/ui_qt/import_dialog.py:64`

**Current code:** [VERIFIED: codebase read]
```python
def run(self):
    try:
        results = yt_import.scan_playlist(self._url)
        live = [e for e in results if e.get("is_live") is True]  # BUG
        self.finished.emit(live)
```

**Why it fails:** `yt_import.scan_playlist()` returns `[{"title": ..., "url": ..., "provider": ...}]` — no `is_live` key. The comprehension filters every element, `live` is always `[]`, `finished.emit([])` fires, `_on_yt_scan_complete` sees empty list and shows "No live streams found." [VERIFIED: codebase read of `yt_import.scan_playlist` return statement lines 72-79]

**Fix:** [VERIFIED: codebase]
```python
def run(self):
    try:
        results = yt_import.scan_playlist(self._url)
        self.finished.emit(results)
```

### Fix 4: Station Logo Render Path

**Files:** `musicstreamer/ui_qt/now_playing_panel.py` and `musicstreamer/ui_qt/station_list_panel.py`

**Root cause confirmed:** [VERIFIED: codebase]

`assets.copy_asset_for_station()` returns `rel = os.path.relpath(dst, paths.data_dir())` — a relative path like `assets/12/station_art.png`. This is what's stored in `station.station_art_path`.

`_load_scaled_pixmap(path, size)` in now_playing_panel.py passes `path` directly to `QPixmap(path)`. `QPixmap` needs an absolute path. On Linux, `QPixmap("assets/12/station_art.png")` resolves relative to the process CWD, not the data directory, so it fails silently and `pix.isNull()` returns True → falls back to generic icon.

`_load_station_icon(station)` in station_list_panel.py has the same issue: `path = station.station_art_path or _FALLBACK_ICON` then `QPixmap(path)`.

**Fix pattern:** [ASSUMED — standard Qt approach] Resolve against `paths.data_dir()` before passing to `QPixmap`:
```python
import os
from musicstreamer import paths

def _resolve_art_path(rel_path: Optional[str]) -> Optional[str]:
    """Convert relative asset path to absolute, or return None if path is None."""
    if not rel_path:
        return None
    if os.path.isabs(rel_path):
        return rel_path  # already absolute (future-proofing)
    return os.path.join(paths.data_dir(), rel_path)
```

Update `_load_scaled_pixmap` and `_load_station_icon` to call this resolver before `QPixmap(path)`.

**QPixmapCache invalidation:** `station_list_panel._load_station_icon` uses `QPixmapCache` keyed on `f"station-logo:{path}"`. The cache key uses the relative path, so after fixing to use absolute paths the key will change — this is fine (new entries, old stale entries evict naturally). However, after a logo is updated, the cache must be invalidated. Cache invalidation call: `QPixmapCache.remove(key)` or just use the absolute path as the key from the start.

### Fix 5: ICY Disable Toggle

**Root cause confirmed:** [VERIFIED: codebase]

`station.icy_disabled` is **correctly persisted** — `repo.update_station()` writes `int(icy_disabled)` and `list_stations()` reads `bool(r["icy_disabled"])`. The DB schema has the column. Persistence is NOT the bug.

`Player._on_gst_tag()` emits `self.title_changed.emit(title)` unconditionally — no station reference, no ICY disable check. [VERIFIED: player.py lines 247-257]

`NowPlayingPanel.on_title_changed(title)`:
1. Sets `icy_label.setText(title)` — no `icy_disabled` check
2. Calls `self._fetch_cover_art_async(title)` — no `icy_disabled` check

The panel holds `self._station` (set in `bind_station()`). So the check belongs here:

```python
def on_title_changed(self, title: str) -> None:
    if self._station and self._station.icy_disabled:
        return  # suppress all ICY updates when disabled
    # ... existing body
```

**cover_art.py:** `fetch_cover_art()` takes no station context — it only receives an ICY string. The suppression can happen entirely in `on_title_changed` before `_fetch_cover_art_async` is called. No changes to `cover_art.py` are required. [VERIFIED: cover_art.py API]

**Player `title_changed` signal:** When `icy_disabled=True`, the tag bus message still fires from GStreamer and `Player._on_gst_tag` still emits `title_changed`. The panel suppresses it. An alternative is to gate in `Player` itself, but that requires Player to hold a station reference, which is architectural scope creep. Panel gating is correct and sufficient.

**Track title display fallback (D-15):** When icy_disabled=True and a new track tag arrives, the `icy_label` should show the station name. Currently `bind_station()` does `self.icy_label.setText("")` — change to set it to `station.name` when `icy_disabled=True`. Then `on_title_changed` early-return prevents overwriting it.

## Common Pitfalls

### Pitfall 1: QPixmapCache stale after logo update
**What goes wrong:** After a user picks a new logo for a station, the station list still shows the old icon because `_load_station_icon` reads from `QPixmapCache` using the old path as key.
**How to avoid:** After `copy_asset_for_station()` updates `station.station_art_path`, call `QPixmapCache.remove(f"station-logo:{old_path}")` or clear the relevant entry. Alternatively, key the cache on absolute path + mtime.
**Warning signs:** Logo updates in EditStationDialog don't appear in the station list until restart.

### Pitfall 2: Signal type mismatch crashes at runtime
**What goes wrong:** If `_AaImportWorker.finished` is changed to `Signal(int, int)` but `_on_aa_import_complete` still takes `(self, count: int)`, Qt raises a `TypeError` at signal emission time.
**How to avoid:** Change both the signal definition AND the connected slot signature atomically. Update the test assertion in `test_import_dialog_qt.py:117` to match.

### Pitfall 3: _YtScanWorker rename
**What goes wrong:** After removing line 64 (`live = ...`), the variable `live` in `self.finished.emit(live)` no longer exists.
**How to avoid:** Replace both lines together: delete `live = [...]` AND change `self.finished.emit(live)` to `self.finished.emit(results)`.

### Pitfall 4: EditStationDialog _on_save doesn't persist new logo path
**What goes wrong:** `_on_save` calls `repo.update_station(station.id, ..., station.station_art_path, ...)` using the original `station.station_art_path`. If the user picked a new logo, that in-memory update isn't reflected unless the dialog stores the new path and passes it.
**How to avoid:** Add `self._logo_path: Optional[str]` initialized from `station.station_art_path`. Update it when file picker or auto-fetch completes. Pass `self._logo_path` (not `station.station_art_path`) to `repo.update_station`.

### Pitfall 5: Auto-fetch worker lifetime
**What goes wrong:** `_LogoFetchWorker` starts, user closes the dialog, worker finishes and emits signal onto a deleted widget → `RuntimeError: Internal C++ object already deleted`.
**How to avoid:** Store worker as `self._logo_fetch_worker`. In `closeEvent`/`reject`, check `if self._logo_fetch_worker and self._logo_fetch_worker.isRunning(): self._logo_fetch_worker.finished.disconnect()` before worker completes. Or use the token pattern from `NowPlayingPanel._cover_fetch_token`.

### Pitfall 6: ICY disable check uses stale station reference
**What goes wrong:** `bind_station()` stores `self._station`. If the station is edited (icy_disabled toggled) while playing, the panel's `self._station` has the old value. ICY titles keep appearing after the toggle.
**How to avoid:** After `EditStationDialog.station_saved` signal, MainWindow refreshes station list AND rebinds the now-playing panel's station using the fresh station from repo. This is already the required flow for station list refresh — just ensure `now_playing.bind_station(updated_station)` is called (or the `_station` reference is refreshed).
