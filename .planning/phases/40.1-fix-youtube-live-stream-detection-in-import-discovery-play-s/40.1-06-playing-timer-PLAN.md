<objective>
Fix the elapsed-time display regression in the v2.0 Qt port. The now-playing "playing timer" (elapsed_label in now_playing_panel.py) stays stuck at "0:00" and never advances while a station is playing. v1.5 GTK displayed elapsed time correctly.

**Root cause (verified by reading the code):** `Player.elapsed_updated = Signal(int)` is declared at `musicstreamer/player.py:67` with the comment "Phase 30 reserved". It is wired in `main_window.py:141` to `NowPlayingPanel.on_elapsed_updated`, which formats seconds correctly (confirmed in `now_playing_panel.py:307-314`). BUT: no code path anywhere in the codebase ever emits `elapsed_updated`. `grep "elapsed_updated\\.emit"` returns zero matches. The signal is declared-and-wired but the producer was never implemented during the Qt port.

**Fix:** Add a `QTimer` (1000ms interval, repeating) on `Player` that:
- Starts on each `_try_next_stream` first-attempt playback begin
- Ticks once per second, increments an internal counter, emits `elapsed_updated(seconds)`
- Resets the counter to 0 on a new `play()` / `play_stream()` (first attempt)
- Stops on `stop()` and `pause()`
- Does NOT reset on failover (elapsed time is wall-clock since user pressed play, not since current stream URI load — matches v1.5 behavior and user expectation that failover is transparent)

Scope is strictly the elapsed-time display. NO seek bar, NO total duration, NO volume changes (per scope constraint from orchestrator).

Purpose: Restore v1.5 elapsed-time display parity in the v2.0 Qt port.
Output: QTimer-driven elapsed emissions from Player + two regression tests (one unit at the Player level, one at the NowPlayingPanel level).
</objective>
