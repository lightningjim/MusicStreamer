# Phase 96.2: Fix Newly-Discovered Merge Dropdown - Context

**Gathered:** 2026-06-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the **map/merge target dropdown** on each "Newly discovered on channel" row in `LiveRefreshDialog` (`musicstreamer/ui_qt/live_refresh_dialog.py`). Today it only offers stations with `live_url_syncs_from_channel = 1`, so a user with 19 saved stations on a provider sees only the 1 that happens to be flagged — making merge/map near-useless when a station's live stream dies.

**Confirmed root cause:** the dialog loads `self._flagged_stations = repo.list_flagged_stations_for_provider(provider_id)` (live_refresh_dialog.py:708) and **reuses that same flagged-only list** as the merge-target dropdown source for newly-discovered rows (passed at live_refresh_dialog.py:815, consumed at `_DiscoverRowWidget._map_combo`, lines 545–552). `list_flagged_stations_for_provider` filters `WHERE s.live_url_syncs_from_channel = 1` (repo.py:1081). With only 1 of 19 stations flagged, the dropdown offers 1.

**Scope anchor:** This is the explicit **un-deferral** of Phase 96.1's deferred idea "Map/merge into ANY station (not just flagged)" — narrowed to **same-provider** stations. YouTube-only and manual review-and-confirm invariants carry forward. Targeted bugfix to one dropdown; not a new capability.
</domain>

<decisions>
## Implementation Decisions

### Merge-target dropdown scope (the core fix)
- **D-01:** The newly-discovered merge-target dropdown (`_DiscoverRowWidget._map_combo`) MUST offer **all of this provider's saved stations**, independent of the `live_url_syncs_from_channel` flag — not just the flagged ones. Source it from a provider-scoped, flag-independent station list (a new `repo.list_stations_for_provider(provider_id)` modeled on `list_flagged_stations_for_provider`, or equivalent), NOT the existing flagged query.
- **D-02:** **Scope boundary — only the merge dropdown changes.** The REMAP section (existing flagged rows whose live URL is being re-checked) stays sourced from `list_flagged_stations_for_provider` and is correctly flag-scoped. Do not broaden the REMAP set; this phase touches only the newly-discovered merge-target dropdown source.

### Merge behavior
- **D-03:** Merging a discovered live stream onto a station that was **not** previously live-syncing MUST NOT change that station's `live_url_syncs_from_channel` flag. Re-point the station's stream URL / title anchor only (carry forward Phase 96.1 D-06/D-07 `remap` change shape) — do **not** auto-enroll the station into future auto-refresh. Merge stays a one-time fix; the station won't auto-appear in future refresh dialogs as a REMAP row.

### Dropdown presentation
- **D-04:** Present the (now ~19+) stations **alphabetically by name** — keep `ORDER BY s.name COLLATE NOCASE`. No flagged-first / stale-first grouping and no extra per-row disambiguation hint (e.g. current stream URL) this phase.

### Claude's Discretion
- Whether to add a dedicated `list_stations_for_provider` repo query vs. filtering an existing list in the dialog. The flagged query is the obvious copy-from template (same Station-building loop, drop the flag predicate).
- Which `stream_id` on the chosen target station the `remap` path re-points (e.g. `preferred_stream_id` vs first/primary stream) — verify against the existing `apply_refresh` remap behavior so a multi-stream target re-points the intended stream.
- Whether the empty-state combo text "(no flagged stations)" (live_refresh_dialog.py:552) should be reworded now that the source is all-provider stations (e.g. "(no stations for this provider)").
</decisions>

<specifics>
## Specific Ideas

- Lived symptom: the user has ~19 saved stations on the Yellow Brick Cinema provider, but when a newly-discovered live stream appears in the refresh dialog, the "Map target" dropdown only lets them pick 1 of those 19 — so they can't merge the new live stream onto the station whose stream actually died.
- "Merge onto station X" means re-point station X's stale primary stream to the newly-discovered live URL (the Phase 96.1 D-06/D-07 remap semantics), just now selectable across all of the provider's stations.
</specifics>
