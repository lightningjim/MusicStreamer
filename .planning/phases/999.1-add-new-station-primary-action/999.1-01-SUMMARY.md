## Key Decisions

### D-1: Pre-existing provider-combo guard — no change needed
Task 01-01 action step 2 asked to confirm whether `_populate` already had the `station.provider_name or ""` guard. Verified at line 396 of the pre-edit file: `self.provider_combo.setCurrentText(station.provider_name or "")`. No modification was made for D-08; the behavior was already correct. Commit message for Task 01-01 explicitly notes this.

### D-2: Flag-flip ordering inside reject/closeEvent
Placed `self._is_new = False` IMMEDIATELY after `self._repo.delete_station(self._station.id)` and BEFORE the existing `self._shutdown_logo_fetch_worker()` + `super().reject()/closeEvent()` calls. Rationale: if Qt double-routes the teardown (e.g. closeEvent then reject), the flag-flip guarantees exactly one delete regardless of call order. `repo.delete_station` is a silent no-op on missing id (repo.py:286-288), so a double-delete is harmless — but the flag-flip makes intent explicit and satisfies the Pitfall 2 double-delete concern called out in the plan.

### D-3: SAVE-CLEANUP minimum-diff insertion
Only ONE line was inserted in `_on_save`: `self._is_new = False` between `self.station_saved.emit()` and `self.accept()`. No other line in `_on_save` was modified. This matches the plan's explicit D-03 "no branching in _on_save" requirement and the minimum-diff acceptance criterion.
