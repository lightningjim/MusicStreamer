<objective>
Plan 87-05 ships the user-facing AnnouncementBanner widget and wires it to the marquee-ready signal from Plan 87-03's `GbsMarqueeWorker`. The banner is the second of Phase 87's two visual deliverables (themed-logo override from Plan 87-04 is the first).

Visibility predicate (GBS-MARQ-03 literal): banner visible iff (a) bound station is GBS.FM, (b) first-segment announcement is non-empty after parse, (c) SHA-256 hash of the first-segment is NOT in the in-memory `_dismissed_announcement_hashes` set. Dismissal via inline × button adds the current hash to the set. Same marquee text re-fetched on the next 60s tick → same hash → banner stays hidden. Marquee text changes → new hash → banner re-appears.

Banner content rendering (GBS-MARQ-04 / D-14): first-segment text with internal `|` characters replaced by `\n` so QLabel's `wordWrap=True` wraps at pipe boundaries when present. Qt.TextFormat.PlainText is non-negotiable (CONVENTIONS T-40-04 + V5 input validation in the threat model — defeats HTML/JS injection in operator-controlled marquee body).

Pitfall #5 (Phase 71 RichText baseline): banner is PlainText-only. Adding the banner does NOT increase the `EXPECTED_RICHTEXT_COUNT = 3` constant in test_constants_drift.py. Verify after this plan ships that `grep -c "setTextFormat(Qt.RichText)" musicstreamer/` returns 3 (unchanged from baseline) and the Phase 71 drift-guard stays green.

Banner parenting (87-RESEARCH.md Open Question 4 resolution — option c): wrap NowPlayingPanel's existing layout in a new outer `QVBoxLayout` where row 0 = banner, row 1 = the existing layout. Banner takes full panel width. Visual UAT at narrow / medium / wide widths (per memory `feedback_ui_bug_verify_with_extremes.md`).
