# Phase 47.1: Code Review Report

**Reviewed:** 2026-04-18
**Depth:** standard
**Files Reviewed:** 3
**Status:** issues_found

## Summary

Phase 47.1 adds a GStreamer `message::buffering` bus subscription that emits a de-duped `buffer_percent(int)` signal, a hidden-by-default "Stats for Nerds" widget in `NowPlayingPanel` (QFormLayout + QProgressBar + percent label), and a checkable hamburger-menu action in `MainWindow` that persists the toggle to the settings repo.

The implementation is small, well-commented, and correctly observes the Qt/GStreamer thread boundary (bus handler only emits signals). No security issues and no critical bugs. Two warnings relate to (a) a potential `AttributeError` at startup if `set_buffer_percent` is called before `_build_stats_widget` runs to completion, and (b) an initial-visibility value that is read twice (once in `NowPlayingPanel.__init__`, once implicitly via the menu-action toggle handler not firing at startup), which could desynchronize the panel from a third "Stats for Nerds" setting source. Remaining items are informational.

## Warnings

### WR-01: `set_buffer_percent` can be called before `buffer_bar` / `buffer_pct_label` exist

**File:** `musicstreamer/ui_qt/now_playing_panel.py:364-367`
**Issue:** `set_buffer_percent` writes directly to `self.buffer_bar` and `self.buffer_pct_label`. Those attributes are only created inside `_build_stats_widget`, which is called from `__init__` near the end of construction (line 230). Construction order in `MainWindow.__init__` is:

1. `NowPlayingPanel(player, repo, parent=...)` is constructed (line 180 of `main_window.py`).
2. `self._player.buffer_percent.connect(self.now_playing.set_buffer_percent)` is wired (line 211).

That ordering is safe in isolation, but `buffer_percent` is a queued cross-thread signal emitted from the GStreamer bus-loop thread. If GStreamer ever delivered an early buffering message between `Player.__init__` (which calls `add_signal_watch` at line 91 — before `MainWindow` finishes constructing the panel) and the wiring on line 211, nothing would happen (no slot connected yet — harmless). However, the real risk is the reverse: if any other code path (tests, future refactor) instantiates `NowPlayingPanel` but skips `_build_stats_widget` (e.g. a subclass override), calls into `set_buffer_percent` will raise `AttributeError` rather than silently no-op. Given the Phase 47.1 spec guarantees the widget is "always constructed", this is a defensiveness concern rather than a current bug.

**Fix:** Add a guard, or pre-declare the attributes at the top of `__init__` so `hasattr` is True before `_build_stats_widget` finishes:
```python
def set_buffer_percent(self, percent: int) -> None:
    bar = getattr(self, "buffer_bar", None)
    lbl = getattr(self, "buffer_pct_label", None)
    if bar is None or lbl is None:
        return
    bar.setValue(int(percent))
    lbl.setText(f"{int(percent)}%")
```

### WR-02: Initial-visibility of stats widget read twice; menu and panel can desync

**File:** `musicstreamer/ui_qt/now_playing_panel.py:232-233` and `musicstreamer/ui_qt/main_window.py:144-150`
**Issue:** Two independent readers of the same setting:

- `NowPlayingPanel.__init__` reads `show_stats_for_nerds` and calls `self._stats_widget.setVisible(show_stats)`.
- `MainWindow.__init__` reads `show_stats_for_nerds` and calls `self._act_stats.setChecked(...)` — but `setChecked` on a `QAction` does **not** fire the `toggled` signal when called from code (Qt emits `toggled` only on user interaction or via `setChecked(!currentState)`; in practice Qt does emit on state change, but only if the state actually changes — starting from unchecked and setting to unchecked is a no-op). The net effect: the panel's visibility is set by `NowPlayingPanel`, and the menu's checked state is set by `MainWindow`. They are consistent today because both read the same key, but any drift (e.g., future code that sets the setting after panel construction but before menu construction, or the other way around) will silently desynchronize the checkmark from the actual panel state.

**Fix:** Single source of truth — let `MainWindow` drive the panel after both are constructed:
```python
# In MainWindow.__init__, after self.now_playing and self._act_stats both exist:
initial = self._repo.get_setting("show_stats_for_nerds", "0") == "1"
self._act_stats.setChecked(initial)
self.now_playing.set_stats_visible(initial)  # explicit, avoids relying on toggled-on-setChecked
```
And remove lines 232-233 of `now_playing_panel.py` (the widget defaults to hidden via the `setVisible(False)` you can set unconditionally in `_build_stats_widget`).

## Info

### IN-01: `_on_gst_buffering` tuple-vs-int unpack may mask upstream API changes

**File:** `musicstreamer/player.py:313-315`
**Issue:** `msg.parse_buffering()` in PyGObject 3.x consistently returns a bare int (the percent). The tuple branch (`result[0] if isinstance(result, tuple) else int(result)`) is defensive against a PyGObject flattening quirk that is documented in the plan's Pitfall 1 notes, but swallowing both cases silently means if a future PyGObject change returned, e.g., a `(percent, mode, avg_in, avg_out, buffering_left)` tuple (the underlying `gst_message_parse_buffering_stats` shape), only `percent` is consumed and the extra fields are silently dropped. Today that's correct — only percent is needed — but a comment noting that the other tuple positions are the stats shape (mode/avg_in/avg_out/buffering_left) would help future maintainers decide whether to surface them.

**Fix:** Extend the inline comment:
```python
# PyGObject may flatten single-out-param to bare int OR return tuple
# (percent[, mode, avg_in, avg_out, buffering_left] — stats fields unused in v2.0).
percent = result[0] if isinstance(result, tuple) else int(result)
```

### IN-02: `set_buffer_percent` does `int(percent)` twice redundantly

**File:** `musicstreamer/ui_qt/now_playing_panel.py:366-367`
**Issue:** `int(percent)` is called twice in the same slot. Cheap, but the signal is declared `Signal(int)` and Qt guarantees the payload is already an int, so both casts are no-ops. Readability nit.

**Fix:**
```python
def set_buffer_percent(self, percent: int) -> None:
    self.buffer_bar.setValue(percent)
    self.buffer_pct_label.setText(f"{percent}%")
```

### IN-03: Muted-label palette mutation may not follow system theme changes

**File:** `musicstreamer/ui_qt/now_playing_panel.py:571-576`
**Issue:** The "Buffer" row label's palette is captured once at construction (`pal = buffer_row_label.palette()` ... `setPalette(pal)`). If the user changes the system palette or the app's accent palette at runtime (the accent-color dialog exists in this codebase — `apply_accent_palette` in `accent_utils`), the muted color will be frozen to the palette that was active at panel-construction time. Cosmetic only; no functional impact.

**Fix:** Either accept this as cosmetic, or switch to a stylesheet-based approach so the color resolves at paint time:
```python
buffer_row_label.setStyleSheet("color: palette(disabled-text);")
```
(Skip if the project prefers palette-API over stylesheets — consistent with existing patterns.)
