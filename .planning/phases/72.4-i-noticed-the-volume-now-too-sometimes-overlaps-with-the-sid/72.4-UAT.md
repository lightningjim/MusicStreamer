## Tests

### 1. Visual no-overlap at 560px right-pane width (the reproducer)

**Setup:** Launch app on lower display. Sidebar expanded (compact OFF).
**Action:** Drag splitter or resize main window until right pane is at its **560px minimum**.
**Expected:**
- Volume slider does NOT visually overlap the compact-toggle button
- Both fully visible — cluster sits on row 3 (multi-stream) or row 2 (single-stream)
- Transport+actions cluster on row 1 fits without truncation

**Status:** ✅ PASSED (perceived bug was a background-illustration illusion)
**Notes:**
- Initial UAT report flagged Test 1 as failing on launch + after compact-toggle uncollapse.
- Three rounds of hardening fixes (showEvent + QTimer.singleShot defer, 100ms re-fire, nuclear clamp with explicit min/max + layout invalidate+activate) all applied — bug persisted in user's perception.
- Diagnostic stderr build (commit f2ed0b4) instrumented every lifecycle point and confirmed at every panel.width (837 / 714 / 1024 across both monitors): `slider.w=120, sliderMin/Max=(120,120), policy=Fixed, loc=row1`. Qt's internal state was always correct.
- Definitive test: at volume 0% (no orange filled portion visible), user confirmed the "long dark/teal line extending past the compact-toggle" STAYS VISIBLE — it is the headband stroke of the headphones background illustration at the same vertical level as the row 1 controls, NOT the slider's unfilled track.
- All four UAT-1 commits reverted at 550c3d6. Phase 72.4 ships as originally implemented at 1b09055 + 3842340.
- Lesson learned for future UAT rounds: when visual reports suggest layout corruption, request a 0% / 100% volume sweep BEFORE pursuing any code change — the volume-state test would have ruled out the bug in five minutes instead of three rounds of hardening.
