# Phase 72.4: Volume slider / compact-toggle overlap on narrow lower-screen widths - Context

**Gathered:** 2026-05-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Extend the Phase 72.1 width-responsive reflow mechanism to fix a **second wrap tier**: at narrow now-playing-panel widths the **volume slider (120px fixed) + compact-mode toggle button (28x28 fixed)** still overlap each other and/or neighbouring controls, EVEN AFTER `stream_combo` has wrapped to row 2. Phase 72.1 explicitly deferred this case (`.planning/phases/72.1-stream-selector-dropdown-overlap-persists-in-fullscreen-mode/72.1-CONTEXT.md` line 32: *"Changes to the volume slider (still 120px fixed width). Volume is NEVER what wraps; if width is below row 1's minimum even without the picker, that's a separate phase."*) — this IS that separate phase.

**Measured overflow source** (post-Phase 72.1/72.3 layout — `now_playing_panel.py:454–569`):
Row 1 control order (multi-stream, wide) =
`play(36) → stop(36) → edit(36) → stream_combo(min 140) → star(28) → eq(28) → volume(120 fixed) → compact_toggle(28) → addStretch(1)`.
After 72.1 reflow (panel < threshold ⇒ picker wraps to row 2), row 1 becomes:
`play(36) → stop(36) → edit(36) → star(28) → eq(28) → volume(120) → compact_toggle(28) → addStretch(1)`
= **312px** of fixed widgets + 6×8px spacing (48px) = **~360px** center-column minimum.
panel.width() includes outer chrome (logo column + outer spacing×2 + cover column + outer margins). At Phase 72.3's smallest tier (140×140 logo + 140×140 cover) + outer spacing/margins (~32px) = **~672px** panel-width floor for the post-72.1 row 1 to actually fit. Below that floor (i.e., near the 560px right-pane minimum), the row 1 widgets visually overlap — the user's small lower display sits squarely in the **560–672px** danger band.

**Trigger context (from user report):** "The volume now, too, sometimes overlaps with the side hide/expand button, on the lower screen when the side bar is expanded." Sidebar EXPANDED (compact mode OFF) is the reproducer — the sidebar's 280px floor + right-pane shrinks toward 560px = no margin for the secondary cluster. Compact ON typically gives the right pane >840px, well above the floor, so the bug doesn't fire there. But because the trigger is **width**, not mode, the same overflow would manifest in compact ON if the user shrank the window further; the fix must be width-driven (mirrors 72.1 D-01).

**In scope:**
- Add a **second wrap tier** to the now-playing-panel responsive layout. When the post-72.1 row 1 minimum exceeds the available center-column width, reparent the **volume slider + compact-mode toggle** as a unit ("volume cluster") off row 1 onto a new row.
- Row 1 keeps the **transport + actions cluster** invariant: `play → stop → edit → star → eq → addStretch(1)`. (Plus `stream_combo` when not wrapped.) Star and EQ remain on row 1 — they sit visually with the transport group, the unit that survives.
- **Volume cluster** wraps as a unit: `volume_slider → compact_mode_toggle_btn`, in that left-to-right order (preserves the row-1 reading order). Volume uses `QSizePolicy(Expanding, Fixed)` on the wrap row; compact-toggle stays 28×28 fixed on the right. addStretch is NOT needed on the wrap row — the volume slider's Expanding policy already consumes free space.
- **Three-row state** appears only at the narrowest widths AND multi-stream:
  - Row 1: `play / stop / edit / star / eq / addStretch`
  - Row 2: `stream_combo` (Expanding, full-width — existing 72.1 D-07)
  - Row 3: `volume_slider (Expanding) / compact_mode_toggle_btn (Fixed 28)` — NEW
- **Two-row state** for single-stream OR mid-narrow multi-stream:
  - Single-stream narrow: row 1 = transport+actions cluster; row 2 = volume cluster. (Picker hidden, never participates.)
  - Multi-stream mid-narrow: row 1 = transport+actions cluster + volume cluster; row 2 = picker. (Existing 72.1 behavior — preserved.)
- **Single-stream stations get the responsive treatment** for the volume cluster (supersedes 72.1 D-02 *for the volume cluster only* — the picker invariant is unchanged because the picker isn't present in single-stream). 72.1 D-02 said "single-stream stations stay one-row at all widths"; that scoped to the picker. Phase 72.4 introduces a second wrap dimension that does apply to single-stream.
- **Width-responsive trigger, not mode-coupled.** Volume cluster wraps based on panel.width() vs. computed row-1-without-volume-cluster minimum. Compact-mode state, sidebar visibility, and stream count are NOT inputs to the wrap decision (stream count only affects whether row 2 also holds the picker — independent dimension).
- **Reuse the existing `resizeEvent` override** (`now_playing_panel.py` — already extended by 72.1 for `_reflow_stream_picker_row` and by 72.3 for `_apply_art_tier`). Add a third reflow helper invoked from the same override. No new event hook.
- **Idempotent reparent** following 72.1 Pattern 2 (D-05): widget reparent via `addWidget` does not break signal/slot connections; `valueChanged` / `sliderReleased` / `toggled` connections survive. State checks (`indexOf(...) >= 0`) before reparent — no-op if already in target layout.
- **No animation** on reparent (mirrors 72.1 D-06 and 72 D-Claude-Discretion peek-overlay). Instant show/hide.
- **A new row layout** is added eagerly inside `center` (the NowPlayingPanel center QVBoxLayout), beneath the existing `self._controls_row2`. Likely name: `self._controls_row3`. Like `_controls_row2`, it stays empty (zero-height collapse via Qt's hidden-widget handling) when the volume cluster is on row 1.
- **Cached threshold**, invalidated alongside the existing `_cached_row1_min` invalidations. The new threshold is "what panel.width() floor is needed for row 1 to fit WITHOUT the volume cluster, with or without the picker depending on stream count" — planner finalizes the expression but the cached-threshold + invalidate-on-population-change pattern is locked from 72.1.
- **Tests at the pytest-qt level**, extending the 72.1 test surface (`tests/test_phase72_1_now_playing_panel.py` or equivalent):
  - Wide width → both clusters on row 1 (multi-stream) / row 1 (single-stream).
  - Mid-narrow + multi-stream → picker on row 2, volume cluster on row 1 (existing 72.1 unchanged).
  - Very narrow + multi-stream → picker on row 2, volume cluster on row 3.
  - Narrow + single-stream → volume cluster on row 2 (row 3 not used).
  - Reparent preserves `valueChanged` / `sliderReleased` signal connectivity and current volume value.
  - Reparent preserves compact-toggle's `toggled` connectivity and current checked state.
  - Idempotent: calling `_reflow_volume_cluster()` twice in a row produces no widget motion on the second call.
  - On panel widen back across threshold, volume cluster returns to row 1 in the correct position (between eq and addStretch — i.e., between the eq button and the row-1 trailing stretch).

**Out of scope:**
- Changes to `stream_combo` reflow behavior. Phase 72.1's row 1↔row 2 reparent is preserved as-is. The new helper for the volume cluster is independent of `_reflow_stream_picker_row`.
- Changes to MainWindow `setMinimumWidth(560)` floor (`main_window.py:310`). 72.4 makes that floor honored for the secondary cluster; it does not lower or raise the floor.
- Reordering row 1 widgets in the wide-state layout. Wide-layout order is preserved exactly: play/stop/edit/(picker)/star/eq/volume/compact-toggle/stretch.
- Volume slider's MINIMUM width or behavior on row 1. Row 1 keeps the existing `setFixedWidth(120)`. Only when on the wrap row does volume's size policy change (Expanding). On reparent back to row 1, restore the fixed 120 — implementation choice (toggle `setFixedWidth(120)` vs `setSizePolicy(...)+setMaximumWidth(120)`) is planner-discretion.
- Compact-toggle button size or behavior. 28×28 fixed on both row 1 and the wrap row. Connection to `compact_mode_toggled` Signal is untouched.
- Hamburger menu entry for the toggle (already explicitly excluded by Phase 72 D-01). Not revisited.
- The Phase 72.2 initial-launch sizing bug. That's a **timing/state-machine** bug (overlap on launch that resolves only after a resize); 72.4 is a **structural-overflow** bug (overlap at narrow widths that requires wrapping more). 72.2 will be its own phase; 72.4 must not depend on 72.2 being landed first.
- Hysteresis dead-band to suppress flicker exactly at the threshold. Planner-discretion (recommendation: ship without; observe; mirror 72.1 D-Claude-Discretion).
- Animation / slide / fade on the wrap. Instant (D-06 of 72.1 mirrored).
- Touch-friendly hit-target adjustments to volume or compact-toggle. Deployment is Wayland desktop with mouse (memory `project_deployment_target.md`).
- Accessibility tab-order / focus-order audit between the new rows. Implicit assumption: Qt's default top-down + left-to-right traversal is acceptable; if a future polish phase wants explicit tab-order curation that's separate scope.

</domain>

<decisions>
## Implementation Decisions

### What wraps to the new row

- **D-01:** **Volume slider + compact-mode toggle wrap together as a single "volume cluster" unit.** When narrow, both move off row 1 together; row 1 keeps the play/stop/edit/star/eq transport+actions cluster. Star and EQ remain on row 1 because they read visually with the transport group and are the natural row-1 floor. (User Q on Wrap unit: "Volume + compact-toggle".)
- **D-02:** **Volume cluster left-to-right order on the wrap row: `volume_slider → compact_mode_toggle_btn`.** Preserves the row-1 reading order so user mental-model survives the wrap. Compact-toggle on the right edge of the wrap row mirrors its row-1 right-edge position.
- **D-03:** **Cluster wraps as a unit — never splits across rows.** Both widgets move together or neither does. Avoids the dangling-orphan visual that "volume alone" would produce (compact-toggle stranded with no neighbor).

### Coexistence with stream_combo

- **D-04:** **NEW row 3 below row 2 when multi-stream + very narrow.** Three explicit rows: row 1 = play/stop/edit/star/eq, row 2 = stream_combo (existing 72.1 Expanding behavior), row 3 = volume + compact-toggle. (User Q on Coexist: "New row 3, below the picker".)
- **D-05:** **Picker invariant from 72.1 is preserved.** Row 2 = stream_combo (full-width Expanding) is unchanged from Phase 72.1 D-07. Phase 72.4 does NOT revisit the picker's row-2 sizing or position; it ADDS a row 3 beneath. The two wrap tiers are independent — picker wraps on its own threshold, volume cluster wraps on its own threshold.
- **D-06:** **Wrap tiers fire in order as width shrinks** (natural consequence of independent thresholds, but worth stating): (1) widest → all on row 1, (2) mid → picker drops to row 2 (existing 72.1), (3) narrowest → volume cluster drops to row 3 (new). Reverse order on widen.
- **D-07:** **For single-stream stations (no picker), the volume cluster wraps to row 2** (the existing `self._controls_row2` QHBoxLayout, currently always empty for single-stream). Row 3 is not used in single-stream state. (User Q on Single-stream: "Volume cluster wraps to row 2".)
- **D-08:** **72.1 D-02 ('single-stream stations stay one-row at all widths') is SUPERSEDED FOR THE VOLUME CLUSTER ONLY.** 72.1 D-02 still holds for the picker (single-stream → picker hidden → never wraps); but the volume cluster does respond to width for both stream counts. Downstream agents should treat 72.1 D-02 as "scoped to the picker dimension"; the volume-cluster dimension is governed by D-07 above.

### Volume slider sizing on the wrap row

- **D-09:** **Volume slider uses `QSizePolicy(Expanding, Fixed)` when on its wrap row.** Stretches to consume the row's free width (minus the compact-toggle's 28px); improves precision on narrow displays where precision matters most. Mirrors 72.1 D-07 (picker stretches on row 2). (User Q on Volume size: "Stretch (Expanding) to fill the row".)
- **D-10:** **Volume slider reverts to `setFixedWidth(120)` Fixed sizing when on row 1.** Identical visual to the pre-72.4 wide-state layout. Implementation: planner chooses between (a) `setFixedWidth(120)` ↔ `setMinimumWidth(0); setMaximumWidth(QWIDGETSIZE_MAX); setSizePolicy(Expanding, Fixed)` toggle pairs, or (b) a wrapper container that owns the sizing while the slider stays plain Expanding-or-not. Either is acceptable; (a) is closer to 72.1's pattern.
- **D-11:** **Compact-toggle stays 28×28 fixed on the wrap row.** Identical visual to row 1. No size-policy change for the compact-toggle on reparent. addStretch is NOT added to the wrap row — the volume slider's Expanding policy consumes the free space, leaving the compact-toggle pinned to the right edge.

### Trigger mechanism

- **D-12:** **Width-responsive, NOT mode-coupled.** The reflow decision is driven by `panel.width()` vs. a computed "row-1 minimum WITHOUT the volume cluster" threshold. Compact-mode state, sidebar visibility, and current `_is_compact` are NOT inputs. Same rationale as 72.1 D-01: the bug fires when the right pane is narrow, regardless of why it's narrow. (Implicit from D-12 carrying forward 72.1 D-01.)
- **D-13:** **Reuse the existing `resizeEvent` override on `NowPlayingPanel`.** The override already invokes `_reflow_stream_picker_row` (72.1) and `_apply_art_tier` (72.3). Phase 72.4 adds a third call to a new helper (e.g., `_reflow_volume_cluster`). Order of calls inside `resizeEvent` is planner-discretion but logically: picker reflow first (determines row 2 occupancy), volume-cluster reflow second (uses the post-picker row 1 occupancy to compute its threshold), art tier last (cosmetic).
- **D-14:** **Idempotency via `indexOf(...)` check before reparent** (72.1 Pattern 2 reused). No-op if the cluster is already in the target layout. Spurious resize events during `__init__` and zero-width resizes during teardown are guarded against by the same `hasattr` + `panel_width <= 0` defensive checks 72.1 established.

### Claude's Discretion

- **Exact threshold expression for the volume-cluster wrap decision.** Likely candidates: (a) extend `_picker_row_min_width()` into a generalized `_row1_min_width(include_picker, include_volume_cluster)` helper with two cached values, (b) introduce a sibling `_volume_cluster_wrap_min_width()` that computes row-1-without-volume independently, (c) hand-tune a constant. Recommendation: (a) — generalized helper avoids drift between the two thresholds and consolidates the outer-chrome calculation 72.1 already worked out (`_picker_row_min_width` lines 1260-1336 in `now_playing_panel.py`). Planner finalizes.
- **Cache invalidation hook for the new threshold.** 72.1's `_cached_row1_min` is invalidated by `_populate_stream_picker` (Plan 03 hook). The volume-cluster threshold has no equivalent population-change hook (volume slider and compact-toggle are constructed once and never repopulated). It only needs invalidation if eq_toggle / star button sizes change, which they don't. Likely: cache once, never invalidate. Planner confirms.
- **Where exactly to insert the new row 3 inside `center`.** Most-natural insertion is `center.addLayout(self._controls_row3)` immediately after `center.addLayout(self._controls_row2)` (currently at `now_playing_panel.py:577`). Planner verifies there's no widget between row 2 and the stats widget that should sit above row 3.
- **Helper name and signature.** Recommendation: `_reflow_volume_cluster(self) -> None` mirroring `_reflow_stream_picker_row` for grep-affinity. Internal helpers: `_move_volume_cluster_to(target_layout: QHBoxLayout)` (analogue of `_move_stream_picker_to`), `_set_volume_size_policy(expanding: bool)` (analogue of `_set_picker_size_policy`). Planner picks final names.
- **Whether row 3 should have explicit `setSpacing(8)` and `setContentsMargins` matching row 1/2.** Recommendation: yes, `_controls_row3.setSpacing(8)` to mirror row 1 and row 2. No explicit `setContentsMargins`; inherit `center`'s margins.
- **Hysteresis dead-band** to suppress flicker at the second threshold boundary. Recommendation: ship without; observe in real use; add only if observed flicker is real (mirrors 72.1's same conclusion). Wayland resize coalescing tends to suppress this in practice.
- **Test file location and naming.** Recommendation: extend the existing 72.1 test file (`tests/test_phase72_1_now_playing_panel.py` if it exists, or whichever file holds the 72.1 reflow tests) with a `class TestPhase72_4VolumeClusterReflow` block. Avoid creating a parallel file unless test cardinality demands it. Planner picks.
- **Order of reflow calls inside `resizeEvent`.** Recommendation: (1) `_reflow_stream_picker_row` (72.1), (2) `_reflow_volume_cluster` (new), (3) `_apply_art_tier` (72.3). The picker-first order matters because the volume-cluster threshold may want to see whether picker is in row 1 or row 2 (if the cached helper is generalized per option (a) above, the picker-occupancy state isn't actually a runtime input — but the call order is still sensible because layout-affecting reflows run before cosmetic ones).
- **REQUIREMENTS.md entry.** Phase 72.4 likely registers a new requirement code (e.g., `LAYOUT-04`) as the continuation of LAYOUT-01 (Phase 72), LAYOUT-02 (Phase 72.1), LAYOUT-03 (Phase 72.3). Planner picks the code and writes the table row.

</decisions>

### Established Patterns
- **QSplitter + QSizePolicy hand-offs:** The right pane (NowPlayingPanel) honors its `setMinimumWidth(560)` floor; row 1 must always fit at that floor or the layout overflows. 72.4 ensures row 1 fits at 560 by wrapping the volume cluster to a new row.
- **Single-source-of-truth panel-state visibility** (Phase 47.1 WR-02, Phase 67 M-02, Phase 72 D-Claude pattern): the layout state (which widgets are in which row) is driven by `resizeEvent` calling the reflow helpers; nothing else reads or writes the row membership. 72.4 follows this — no other code path manipulates `_controls_row2` / `_controls_row3` membership.
- **`Qt.AutoConnection` for in-thread signal wiring:** volume slider's `valueChanged` and compact-toggle's `toggled` are main-thread signals; default AutoConnection is correct on both row 1 and the wrap row. Reparent doesn't change thread affinity.
- **Phase 72.1 defensive guards** (`hasattr`, `panel_width <= 0`) — must be applied identically to the new helper. Spurious resize events during `__init__` and zero-size resizes during teardown are real on this codepath.

- **Reproducer anchor:** "The volume now, too, sometimes overlaps with the side hide/expand button, on the lower screen when the side bar is expanded." - **"Too" is significant:** The user said "the volume now, **too**" — Phase 72.1 fixed the stream_combo overlap; the user is reporting that the next-tier overflow now bites the volume + compact-toggle. Implied user mental model: "you fixed the picker overlap; the next overlap is happening here." This phase closes that next tier.
- **Wrap unit precedence:** The user's mental groupings are (a) transport (play/stop) as a unit, (b) actions (edit/star/eq) as a unit, (c) volume + compact-toggle as a unit on the right edge. The "cluster" terminology in 72.4 D-01..D-03 reflects this. Star and EQ stay on row 1 because they sit visually adjacent to the transport widgets and read as part of the transport+actions block at narrow widths.
- **Single-stream behavior is deliberately diverged from 72.1 D-02.** 72.1's "single-stream stations stay one-row" was scoped to the picker (which is hidden in single-stream so naturally never wraps). 72.4 introduces a new wrap dimension that applies to both stream counts — single-stream gets two rows at narrow widths, multi-stream gets three rows. User explicitly chose "Volume cluster wraps to row 2" for single-stream.
- **Reference shortcut:** The volume slider Expanding policy on its wrap row mirrors the picker's row-2 Expanding policy from Phase 72.1 D-07. Consistent UX precedent within the now-playing pane: "wrapped widgets stretch."
- **Deployment correction (still applies):** Memory `project_deployment_target.md` confirms Wayland-only deployment. No X11 considerations. Manual verification on the small lower display closes the loop on the user's reproducer.

<deferred>
## Deferred Ideas

- **Hysteresis dead-band at the volume-cluster threshold.** Same shape as 72.1's deferred hysteresis — ship without, observe in practice, add only if real flicker is observed when dragging the splitter exactly at the threshold. Wayland resize coalescing typically masks this; the user's split-second flickering would be the only signal.
- **Animation on the wrap.** Phase 72 / 72.1 / 72.4 all ship without animation. If the instant wrap feels jarring once shipped, a future micro-polish phase could add a 100-150ms QPropertyAnimation on height or opacity for the wrap row's appearance. Low priority.
- **Accessibility tab-order / focus-order curation between the new rows.** Qt's default top-down + left-to-right traversal should be acceptable, but if a keyboard-navigation user reports awkward tab traversal between row 1, row 2, and row 3, a polish phase could install an explicit focus chain. Not raised this discussion.
- **Per-display-geometry layout memory.** "Remember that this monitor always shows three rows." User declined per-display memory in Phase 72; same here. Layout is purely width-driven.
- **A FlowLayout for the entire row.** Same rejection as 72.1 D-04 — a FlowLayout would let widgets wrap arbitrarily, breaking the "cluster as a unit" invariant from 72.4 D-01..D-03.
- **Volume slider replacement with a popover button or icon-only control.** Some music apps use a speaker-icon button that opens a popover slider; this would let volume disappear entirely on narrow widths and free even more row-1 space. Different feature; if the user ever wants this, separate phase.
- **Compact-toggle hide-on-narrow** (i.e., wrap it AWAY from the row entirely when narrow, on the assumption that the user can use Ctrl+B). User did not raise this; the chosen "wrap with volume" keeps the button reachable.
- **Touch-friendly hit-target adjustments.** Deployment is desktop with mouse (memory). If a touchscreen device enters scope, a future phase would revisit volume slider thickness and tap targets across all three rows.
- **Phase 72.2 cross-pollination.** 72.2 (initial-launch sizing bug — pending) may share root cause with the layout-state-machine that 72.4 is extending. If 72.2 turns out to be "the resizeEvent doesn't fire on initial show," 72.4's helpers may need to be invoked explicitly from `showEvent` too. That's 72.2's problem to solve; 72.4 ships independently and 72.2's fix would just add a `showEvent` call site that invokes ALL three reflow helpers (picker, volume cluster, art tier).
- **Generalizing the reflow framework.** After 72.1 + 72.4, the now-playing panel has TWO wrap tiers + an art-tier. If a third wrap is ever needed (e.g., the stats widget needs to wrap), a "reflow registry" pattern could replace the hand-listed calls in `resizeEvent`. Premature now; revisit if/when a third dimension appears.

</deferred>
