<decisions>
## Implementation Decisions

### Album grouping render style (Area 1)
- **D-01:** Render album grouping in the artist drill-down via **section-header rows in the existing QTableWidget** (NOT an Album column, NOT both). Closest mapping to gbs.fm's source HTML structure (`<tr class="albumTitle">` separator rows); preserves the 4-column table width; adds scannable group boundaries without per-row text repetition.
- **D-02:** Implement the section header as a **spanning row inside the existing QTableWidget**: insert a row, call `setSpan(row, 0, 1, columnCount)`, populate with a single `QTableWidgetItem` containing the album name + song count, set bold font + background tint, set `Qt.ItemFlag.NoItemFlags` (non-selectable, non-editable). Keeps the dialog's widget type stable; minimal diff. Rejected: QTreeWidget swap (introduces widget-type asymmetry between search-results and drill view of the same dialog area); QTableView + QStandardItemModel + custom delegate (heavyweight; not justified for this scope).
- **D-03:** Section-header content is **album name + song count** in the form `"{album} ({n} songs)"` — e.g. `"The New Order (8 songs)"`. Count is `len(rows_in_group)` in the render path; no parser-side counting needed.

### Triage→fix scope commitment (Area 2)
- **D-04:** Phase 60.2 commits to **triage + ship whatever fix the triage warrants**. SC1 closes one way or the other in this phase: parser fix (with RED-first test from the captured fixture) if triage points at a client-side bug; documented limitation (code comment + CONTEXT.md note) if it's confirmed server-side. Rejected: triage-only-split-fix-to-60.3 (one extra round-trip before SC1 actually closes); parser-fix-upfront-commitment (forces a phase replan if triage shows server-side).
- **D-05:** If triage lands on server-side, the documented limitation lives in **both** a code comment near `_ArtistAlbumParser` AND a note in `60.2-CONTEXT.md` (this file's "Specific Ideas" section gets updated post-triage with the finding). Both surfaces so neither rots alone — a future reader of the parser sees it inline; a future Claude reading planning history sees the rationale.
- **D-06:** If triage lands on client-side (parser bug), test contract is **RED test from `search_bad_religion_p1.html` first, GREEN fix second**. Mirrors Phase 60.1's locked D-03 TDD pattern (capture fixture → write failing parser test → fix → test goes green). Rejected: GREEN-test-with-the-fix (loses audit trail); fix-only-no-fixture-test (no permanent regression guard for this query class).

### Multi-query fixture scope (Area 3)
- **D-07:** Capture **5 fixtures total** alongside the primary `search_bad_religion_p1.html`, to characterize the failure class:
  1. `search_bad_religion_p1.html` — primary defect repro (per Phase 60.1 UAT).
  2. `search_bad_company_p1.html` — same first word; isolates whether "bad" is the trigger or a position-1 prefix collision.
  3. `search_iron_maiden_p1.html` — popular two-word band, control case (rules in/out the "any multi-word with no special tokens" hypothesis beyond what `pearl jam` and `foo fighters` already covered in 60.1).
  4. `search_black_sabbath_p1.html` — control + tests whether words appearing in genre-tag fragments (`black metal`) interact badly with panel rendering.
  5. `search_death_cab_for_cutie_p1.html` — single-result hypothesis probe (a 4-word artist name with low collision likelihood; tests whether gbs.fm suppresses 1-row artist/album panels rather than rendering them).
- **D-08:** Filename pattern is `search_<query_with_underscores>_p1.html`, mirroring the existing `search_test_p1.html` and `search_multiword_p1.html` (Phase 60.1) conventions. The `_pN.html` suffix preserves room for page-2 captures if any probe ever returns paginated results.
- **D-09:** All 5 fixtures captured by extending `scripts/gbs_capture_fixtures.sh` (append after item 16 — the existing `search_multiword_p1.html` capture). Same dev-cookies invariant as the existing entries.

### Album label fallback (Area 4)
- **D-10:** Parser carries **empty string `""`** as the album value for any song row that appears before the first `<tr class="albumTitle">` separator. Honest contract: the source HTML didn't tell us what album they belong to, parser doesn't fabricate. Rejected: `"Singles"` (introduces fake metadata; misleading for compilation cuts or HTML structural artifacts); `"Unknown"` (same fabrication risk; also visually ugly).
- **D-11:** Dialog renders empty-string album rows by **skipping the section-header row entirely** for that group — the rows render directly without a header above them. Honest visual: source HTML didn't group them, the UI doesn't pretend to. Rejected: synthetic "Other" divider (introduces UI text not present in source); subtle styling differentiation (italics + no header — code complexity not justified for an edge case the source doesn't even structure).

### Claude's Discretion
- Whether `_ArtistPageParser` carries the `album` value as a per-row dict field OR emits a parallel `album_groups: list[(album_name, start_idx, end_idx)]` structure — both satisfy the contract; planner picks based on which produces the cleaner diff. Per-row field is simpler and idiomatic.
- Exact font weight / background tint values for section-header rows — UI polish; planner picks something consistent with the rest of the dialog's table styling.
- Whether the post-triage code comment for D-05 lives as a class-level docstring addition on `_ArtistAlbumParser` or as an inline comment near the parser's main loop — wherever a future reader is most likely to look first.
- Whether the bad-religion fixture captures full UTF-8 HTML or sanitized/trimmed-for-test — match the existing fixture style in `tests/fixtures/gbs/search_*_p1.html`.
- Whether the 4 probe fixtures (D-07 #2-#5) get their own RED-first parser tests beyond the primary `bad_religion` one. Probably not — they exist primarily as triage evidence + potential regression coverage if a pattern emerges; planner can decide whether each gets a parametrized test row or just a mention in the triage report.

## Existing Code Insights

### Reusable Assets
- **`_ArtistPageParser` HTMLParser pattern** (`gbs_api.py:574+`): The class already detects `albumTitle` rows (`row_class = "albumTitle"` at line 619) — it just sets `_skip_current = True` and discards them. The diff for D-01/D-10 is "instead of skipping, capture the album text into `self._current_album` and emit it as `row["album"]` for subsequent rows; reset on each new `albumTitle` encounter."
- **`QTableWidget.setSpan(row, col, rowSpan, colSpan)`**: Established Qt idiom for full-width header rows in a QTableWidget. Used elsewhere in PySide6 dialog patterns. Pairs with `Qt.ItemFlag.NoItemFlags` for non-selectable, non-editable header cells.
- **Phase 60.1 fixture-driven test pattern** (test_gbs_api.py): Every parser has a captured-HTML fixture; tests parse the fixture and assert on extracted shape. 60.2 follows this for both the bad-religion fixture (RED-first if parser bug) and the `_ArtistPageParser` album-field-emission tests.
- **`scripts/gbs_capture_fixtures.sh`** (already extended through item 16 by Phase 60.1): Capture-script convention — `curl -sS -b "$COOKIES" "$BASE/search?query=<q>&page=1" > "$OUT/search_<q>_p1.html"`. 5 new lines append cleanly.

### Established Patterns
- **`<tr class="albumTitle">` shape:** Source HTML on `/artist/<id>` interleaves `<tr class="albumTitle">` rows (containing album title in a `<th>` or single `<td>` with bold styling) between groups of song rows. The Testament fixture (`artist_4803.html`) is the canonical example.
- **Plain-text invariant (T-40-04):** All `QListWidget`/`QTableWidget` items use plain text by default; `href` and song IDs stored as `Qt.ItemDataRole.UserRole`. 60.2 preserves this — section-header rows store no UserRole data (they're decorative).
- **Honest-data principle:** Parsers emit empty strings for fields the source HTML didn't provide rather than fabricated defaults. Already established in `_SongRowParser` (empty `add_url` if no `/add/N` anchor present).

### Integration Points
- **Parser → Dialog → Table render path:** `fetch_artist_songs(href)` returns `list[dict]` with `{songid, artist, title, duration, add_url, album}` (album field is the new addition). `_GbsArtistWorker.finished.emit(rows)` carries this list to `_on_artist_drilled` which calls `_render_results(rows, mode="artist_drill")`. The new code path in `_render_results` groups rows by `album` and inserts span-row headers between groups (skipping the header for the empty-string group per D-11).
- **Bad-religion triage workflow:** Capture script runs → `search_bad_religion_p1.html` lands in `tests/fixtures/gbs/` → planner reads the captured HTML and inspects: (a) does it contain `<p class="artists">` / `<p class="albums">` blocks at all? (b) are those blocks empty? (c) are they present-but-malformed in a way `_ArtistAlbumParser` chokes on? (d) any other shape signal. Triage outcome routes to either D-06 (RED-first parser fix) or D-05 (documented limitation).

## Specific Ideas

- **Triage hypotheses (informed by 60.1-UAT.md notes):**
  - H1: gbs.fm omits the artist/album panels for `bad religion` server-side (e.g., too many song matches → panels suppressed; special-cased substring; result-cardinality below a panel-rendering threshold).
  - H2: `_ArtistAlbumParser` has an edge-case bug — perhaps related to space encoding, anchor-text empty cases, or apostrophes — that the foo-fighters fixture didn't surface.
  - H3: Both — server returns a slightly different response shape that exposes a parser bug.
  - H4 (added during discussion): gbs.fm suppresses panels when only ONE artist/album hit would appear (single-result threshold). The `death_cab_for_cutie` probe directly tests this.
- **`bad religion` UAT note (verbatim):** "It now takes each table per album into a single undifferentiated table of songs for artists that work. However it seems to still specifically be tripping on 'bad religion' as I am still only getting song results; no artist, no album. Bad Religion is definitely a band and also has a self-titled album among others, which neither are showing up here."
- **Working drill-down example:** The Testament artist page (`/artist/4803`) is the canonical multi-album artist with `<tr class="albumTitle">` separators — used as the primary fixture for `_ArtistPageParser` album-field-emission testing.
