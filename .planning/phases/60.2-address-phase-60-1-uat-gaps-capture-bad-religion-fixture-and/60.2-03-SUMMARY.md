original_idx = 0       # song-list index (advances len(rows) per group; NEVER counts headers)

**2. [Acceptance-grep over-broadness] Pitfall 5 grep flagged docstring mentions of `setStyleSheet`/`setBackground`**

- **Found during:** Task 2 acceptance check.
- **Issue:** Plan acceptance criterion: `awk '/def _insert_album_section_header/,/^    def [^_]/' ... | grep -c "setStyleSheet\|setBackground"` should return 0. Two issues with the criterion as-written:
  1. The awk slice pattern `^    def [^_]` matches a `def` whose name does NOT start with underscore — but every method in the file starts with underscore. So awk extends the slice past the helper's actual end, picking up unrelated code.
  2. Even with a tighter scope, the helper's docstring legitimately MENTIONS `setStyleSheet` and `setBackground` (in the prohibition: "NO setStyleSheet, NO setBackground"). A literal grep cannot distinguish docstring mentions from actual method invocations.
- **Resolution:** Re-validated using a Python regex that strips the docstring before counting actual method invocations (`re.findall(r'\.setStyleSheet\(', body_no_doc)`). Confirmed zero actual `setStyleSheet` / `setBackground` CALLS in the helper body. The docstring mentions are intentional inline documentation of the prohibition; they reinforce the contract for future editors.
- **Files modified:** None (acceptance-criterion check method adjusted; no production code change).
- **Justification:** The plan's intent is clearly "no actual style intrusion in the rendering path," not "no string occurrence anywhere in the function." A docstring that documents the prohibition is the OPPOSITE of a violation — it's a defense against future regressions. Treating this as criterion-tooling adjustment, not a Rule 1/2/3 fix.
- **Impact:** None on production code. The same logic applies to the `_on_album_drilled` Pitfall 6 grep, which the same Python script verifies returns 0 occurrences within the precise method body.
