## Decision

- **Selected hypothesis:** H2 — `_ArtistAlbumParser` discriminator-text edge case (singular vs plural).
- **Routing:** **D-06** (client-side parser fix in Wave 1).
- **Evidence:** Direct parser output above shows zero artists/albums extracted for every fixture whose discriminator is singular ("Artist:" / "Album:"), and correct extraction for every fixture whose discriminator is plural. The captured HTML demonstrably contains the data; the parser drops it.

### Exact Fix Surface for Wave 1

**File:** `musicstreamer/gbs_api.py`
**Lines:** 542-548
**Class:** `_ArtistAlbumParser.handle_data` (the leading-text block discriminator)

**Current code (production, refused fixture data):**
```python
if self._current_block is None:
    lower = txt.lower()
    if lower.startswith("artists"):
        self._current_block = "artists"
    elif lower.startswith("albums"):
        self._current_block = "albums"
    return
```

**Proposed Wave 1 fix:** Accept both singular and plural forms.
```python
if self._current_block is None:
    lower = txt.lower()
    if lower.startswith("artist"):     # accepts "Artist:" AND "Artists:"
        self._current_block = "artists"
    elif lower.startswith("album"):    # accepts "Album:" AND "Albums:"
        self._current_block = "albums"
    return
```

`startswith("artist")` matches both "artist" and "artists" (and "artist:" and "artists:") because "artists" starts with "artist". Same for albums. No regex needed; minimal-diff.

**Pitfall mitigation noted for Wave 1 planner:** The discriminator text may also include trailing colons or other whitespace in the captured chunk — `.startswith("artist")` handles all observed variants without needing to also strip the prefix or check for ending colons.
