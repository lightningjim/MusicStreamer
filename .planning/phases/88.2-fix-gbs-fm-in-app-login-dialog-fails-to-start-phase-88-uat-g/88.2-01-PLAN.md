<objective>
Repair the shared oauth-helper subprocess launcher so GBS.FM, Twitch, and Google/YouTube
in-app login all start on the PyInstaller-frozen Windows build (D-01). The root cause:
all three call sites launch `QProcess.start(sys.executable, ["-m", "musicstreamer.oauth_helper", ...])`,
but in a frozen bundle `sys.executable` is `MusicStreamer.exe` (not a Python interpreter),
so `-m musicstreamer.oauth_helper` is silently ignored and the login window never opens.

Purpose: Unblock UAT-10 / VER-02-J (Phase 88 UAT gap G3). The fix is the verification anchor
for GBS.FM; Twitch + Google are fixed as a free consequence of repairing the shared launcher.

Output: A frozen-safe `_make_oauth_launch_args` helper, a `--oauth-helper` argv dispatch arm
in `__main__.py` (with a `--self-test` headless short-circuit for the Plan 02 build guard),
all three call sites rewired, and a Linux-runnable D-04 frozen/non-frozen dispatch test.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| main process → re-exec'd frozen helper subprocess | argv crosses from `_make_oauth_launch_args` into `QProcess.start` → `MusicStreamer.exe --oauth-helper` → `__main__.main()` |
| attacker-controlled argv → __main__ dispatch | `--oauth-helper` / `--self-test` flags are a new frozen-exe entrypoint reachable by anyone who can invoke the exe |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-88.2-01 | Tampering | `_make_oauth_launch_args` QProcess launch | mitigate | Preserve T-40-05: program is always `sys.executable` (no PATH lookup), args passed as a list (never a shell string), never `shell=True`; `mode` is a hardcoded call-site literal ("twitch"/"gbs"/`self._oauth_mode or "google"`), never free user input |
| T-88.2-02 | Elevation of Privilege | `--oauth-helper` / `--self-test` entrypoint in `__main__.py` | accept | `--self-test` returns 0 and does nothing (no file/network/shell side effects); `--mode` is choice-constrained by oauth_helper's argparse (`choices=["twitch","google","gbs"]`); the entrypoint only ever launches the same already-trusted login windows — no new capability is exposed to attacker argv |
| T-88.2-03 | Information Disclosure | `_run_oauth_helper` argv forwarding | mitigate | Strip `--oauth-helper` before forwarding so oauth_helper's `parse_args` cannot error/leak via "unrecognized arguments" exit; no secrets are placed on the command line (cookies flow over stdout, unchanged) |
| T-88.2-SC | Tampering | npm/pip/cargo installs | accept | No package-manager installs in this phase (RESEARCH.md §Package Legitimacy Audit: not applicable — zero new dependencies); slopcheck N/A |
</threat_model>
