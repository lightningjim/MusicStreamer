## Summary

The GBS.FM in-app login (and identically for Twitch and Google/YouTube) fails to start in the PyInstaller-frozen Windows build because `accounts_dialog._launch_gbs_login_subprocess()` and `_launch_oauth_subprocess()` (and `cookie_import_dialog._on_google_login()`) all call `QProcess.start(sys.executable, ["-m", "musicstreamer.oauth_helper", "--mode", "<m>"])`. In a frozen bundle, `sys.executable` is `MusicStreamer.exe`, not a Python interpreter. The frozen exe ignores `-m musicstreamer.oauth_helper` silently — the login window never opens and the user sees nothing.

The fix follows the established house style in `__main__.py:main()`: add a new argv flag (e.g. `--oauth-helper`) to the existing `parse_known_args` dispatch table, dispatch it to `oauth_helper.main()` before the GUI builds, and change the three call sites to branch on `getattr(sys, "frozen", False)`: frozen → `QProcess.start(sys.executable, ["--oauth-helper", "--mode", m])`, non-frozen → unchanged `["-m", "musicstreamer.oauth_helper", "--mode", m]`. This is the identical pattern already used by `--smoke` and `--check-mediakeys`.

`oauth_helper.main()` builds its own `QApplication` (confirmed at line 459 of `oauth_helper.py`). Since the argv-dispatch in `__main__.py:main()` fires before `_run_gui` ever touches `QApplication`, there is no double-`QApplication` risk — the early-return from the dispatch means `_run_gui` is never reached. The only open question for the frozen path is whether `runtime_hook.py`'s GIO/typelib env setup executes before `oauth_helper.main()` — it does (runtime hooks run at frozen-exe startup, before Python code runs).

**Primary recommendation:** Add `--oauth-helper` argv dispatch to `__main__.py`, update the three call sites in `accounts_dialog.py` and `cookie_import_dialog.py`, wire `errorOccurred` signal to log + GBS fallback in `accounts_dialog.py`, add a pure-Python D-04 monkeypatch test in `tests/test_launcher_frozen_dispatch.py`, add a `--oauth-helper --self-test` flag for the D-05 build.ps1 guard (exit 12), and a source-text drift-guard in `tests/test_packaging_oauth_launcher_spec.py`.

### Pattern 1: PyInstaller frozen-exe re-exec dispatch (`__main__.py`)

**What:** `sys.frozen` (set to `True` by PyInstaller) distinguishes frozen-exe from dev. In the frozen branch, re-exec via `sys.executable <new-flag>` instead of `python -m module`. `__main__.py` already dispatches `--smoke` and `--check-mediakeys` this way.

**When to use:** Any time the frozen exe must run a sub-entrypoint as a subprocess. The `-m module` form is silently ignored by the frozen exe.

**Confirmed from codebase:**

```python
# Source: musicstreamer/__main__.py:310-356 (confirmed)
# Existing dispatch pattern — add --oauth-helper here:

parser.add_argument(
    "--oauth-helper",
    action="store_true",
    default=False,
    help=(
        "Phase 88.2 D-05: frozen-exe re-exec entry for login helper. "
        "Routes argv to oauth_helper.main() before GUI builds."
    ),
)
# ... also add --self-test under --oauth-helper for D-05 headless guard

args, qt_argv = parser.parse_known_args(argv[1:])
remaining = [argv[0], *qt_argv]

if args.oauth_helper:
    return _run_oauth_helper(remaining)   # new dispatch arm
if args.check_mediakeys:
    return _run_check_mediakeys(remaining)
if args.smoke is not None:
    return _run_smoke(remaining, args.smoke)
return _run_gui(remaining)
```

```python
# _run_oauth_helper: dispatch to oauth_helper.main() — no QApplication construction here
# (oauth_helper.main() builds its own). Must short-circuit before _run_gui.
def _run_oauth_helper(argv: list[str]) -> int:
    # Phase 88.2 D-05: --self-test exits 0 immediately (headless smoke guard).
    if "--self-test" in argv:
        return 0
    # Drop the --oauth-helper flag so oauth_helper.main() gets clean argv.
    import sys as _sys
    _sys.argv = [a for a in argv if a != "--oauth-helper"]
    from musicstreamer.oauth_helper import main as _oauth_main
    _oauth_main()
    return 0  # oauth_helper.main() calls sys.exit internally; this is unreachable
```

**Important:** `parse_known_args` must strip `--oauth-helper` from argv before forwarding to `oauth_helper.main()`, which uses `argparse` itself (`--mode` is required). The recommended approach passes `--mode` through and strips only `--oauth-helper`.

### Pattern 2: Frozen-safe subprocess launch helper (`accounts_dialog.py`)

**What:** A shared helper function that selects `(program, args)` based on `getattr(sys, "frozen", False)`. Keeps the frozen-branch logic in one place, testable with monkeypatch.

**When to use:** All three call sites — replaces the three inlined `QProcess.start(sys.executable, ["-m", ...])` calls.

```python
# Source: inferred from __main__.py --check-mediakeys pattern + sys.frozen convention
# Place in accounts_dialog.py (or a shared subprocess_utils helper if both dialogs use it).

import logging
_log = logging.getLogger(__name__)

def _make_oauth_launch_args(mode: str) -> tuple[str, list[str]]:
    """Return (program, args) for launching oauth_helper --mode <mode>.

    Phase 88.2 D-01: when frozen (sys.executable is MusicStreamer.exe),
    re-exec via --oauth-helper argv dispatch in __main__.py.main().
    When not frozen (dev), use the standard -m module form (T-40-05).
    """
    if getattr(sys, "frozen", False):
        return sys.executable, ["--oauth-helper", "--mode", mode]
    return sys.executable, ["-m", "musicstreamer.oauth_helper", "--mode", mode]
```

### Pattern 3: QProcess `errorOccurred` signal for launch-failure (D-02/D-03)

**What:** `QProcess.errorOccurred(QProcess.ProcessError)` fires when the process fails to start (FailedToStart) or has other errors. Critically, `finished` is NOT emitted on FailedToStart — so the existing `_on_gbs_login_finished` / `_on_oauth_finished` handlers never fire. The failure handler must hang off `errorOccurred`.

**Confirmed from Qt docs / PySide6 API [ASSUMED — training knowledge]:** `QProcess.ProcessError.FailedToStart` is the enum value; `QProcess.errorOccurred` is the signal.

**WR-03 re-entrancy guard interaction (critical):** On `FailedToStart`, `_gbs_login_proc` still holds the QProcess reference (it was never cleared by a `finished` signal). The `errorOccurred` handler must:
1. Log the failure.
2. Reset `_gbs_login_proc = None` (so the user can retry and so the re-entrancy guard doesn't permanently block re-entry).
3. Call `_update_status()` (restores "Connect to GBS.FM…" button state).
4. Auto-open `CookieImportDialog` (D-02).

```python
# Wire on QProcess construction, in _launch_gbs_login_subprocess:
self._gbs_login_proc.errorOccurred.connect(self._on_gbs_login_error)

# Handler:
def _on_gbs_login_error(self, error: QProcess.ProcessError) -> None:
    """Phase 88.2 D-02/D-03: handle QProcess launch failure.
    fired when FailedToStart (or other QProcess errors); finished is
    NOT emitted in the FailedToStart case, so this is the sole handler.
    WR-03: reset _gbs_login_proc so retry is possible.
    """
    _log.warning(
        "GBS login subprocess failed to start (QProcess.ProcessError=%s); "
        "falling back to cookie-import dialog.",
        error,
    )
    self._gbs_login_proc = None  # WR-03 re-entrancy guard reset
    self._update_status()
    # D-02: auto-surface cookie import
    self._on_gbs_import_clicked()
```

Similarly wire `errorOccurred` on `_oauth_proc` (Twitch) — though Twitch has no cookie-import fallback, at minimum reset the guard and log. The GBS anchor is the requirement; Twitch/Google are free fixes.

### Anti-Patterns to Avoid

- **Double-`QApplication`:** `_run_oauth_helper` must NOT construct a `QCoreApplication` or `QApplication`. `oauth_helper.main()` builds its own. The dispatch short-circuits before `_run_gui`.
- **`sys.argv` mutation in `__main__.py`:** Avoid mutating `sys.argv` globally. Pass `argv` explicitly via the existing `argv: list[str]` argument chain.
- **`-m` form in frozen context:** `sys.executable ["-m", "musicstreamer.oauth_helper"]` is the root cause. Never use `-m` in the frozen branch.
- **Leaking `--oauth-helper` into `oauth_helper.main()` argparse:** `oauth_helper.main()` calls `argparse.ArgumentParser.parse_args()` (not `parse_known_args`), so an unrecognized `--oauth-helper` flag will cause it to error. The dispatch must strip `--oauth-helper` from argv before calling `oauth_helper.main()`.
- **Connecting both `errorOccurred` AND `finished` without coordination:** If `errorOccurred` resets `_gbs_login_proc = None` and `finished` also fires (it does NOT fire on FailedToStart, but may fire on other errors), there could be a double-reset. Safest: make the `finished` handler robust to `proc = self._gbs_login_proc; self._gbs_login_proc = None` (already existing pattern in `_on_gbs_login_finished`) — `proc` will be `None` if `errorOccurred` already cleared it, and the early `if proc is None: return` guards work.
- **`console=False` and stdout for the self-test:** The frozen exe is built with `console=False` (build.ps1 / spec). Exit code is the contract; do not rely on stdout from the self-test. build.ps1's `$LASTEXITCODE` check is sufficient.

## Common Pitfalls

### Pitfall 1: `oauth_helper.main()` calls `argparse.parse_args()`, not `parse_known_args()`

**What goes wrong:** `oauth_helper.main()` uses `parser.parse_args()` (line 452 confirmed). If `--oauth-helper` is still in `sys.argv` when it runs, argparse errors with "unrecognized arguments: --oauth-helper" and exits 2.
**Why it happens:** `__main__.py`'s `parse_known_args` strips only args it knows; remaining unknown args are forwarded. `--oauth-helper` must be stripped from the forwarded argv.
**How to avoid:** In `_run_oauth_helper(argv)`, filter out `--oauth-helper` before calling `oauth_helper.main()`. The cleanest approach: manipulate `sys.argv` to be `[argv[0], "--mode", <extracted_mode>]` or simply let `oauth_helper.main()` re-parse from `sys.argv` after stripping `--oauth-helper`. [VERIFIED: codebase read of `oauth_helper.py` line 452 + `__main__.py:349`]
**Warning signs:** `oauth_helper.main()` error-exits 2 in CI with "unrecognized arguments".

### Pitfall 2: `finished` is NOT emitted on `QProcess.ProcessError.FailedToStart`

**What goes wrong:** If only `finished` is connected (current code), a `FailedToStart` error leaves `_gbs_login_proc` non-None permanently. The WR-03 re-entrancy guard then blocks all future login attempts for the session.
**Why it happens:** Qt does not emit `finished` when the process never started. This is confirmed Qt behavior. [ASSUMED — training knowledge; matches the signal documentation]
**How to avoid:** Connect `errorOccurred` separately. In `errorOccurred` handler, reset `_gbs_login_proc = None` and call `_update_status()` before the fallback.
**Warning signs:** "Connect to GBS.FM…" button stays disabled after a failed start; "Connecting…" label persists.

### Pitfall 3: `--self-test` must not require a display server

**What goes wrong:** The D-05 self-test `MusicStreamer.exe --oauth-helper --self-test` runs on the Windows build VM (not a display server concern there), but the pattern must not construct `QApplication` or open any window.
**Why it happens:** Build VMs may not have a full display session or SMTC registration.
**How to avoid:** `_run_oauth_helper` detects `--self-test` in `argv` and returns 0 immediately, before any `QApplication` or `oauth_helper.main()` is called. This mirrors `--check-mediakeys` which uses `QCoreApplication` (headless). [VERIFIED: codebase read of `_run_check_mediakeys`]
**Warning signs:** build.ps1 D-05 guard hangs or opens a login window during the build.

### Pitfall 4: `console=False` makes stdout unreachable on the frozen Windows build

**What goes wrong:** The frozen exe is built with `console=False` (MusicStreamer.spec line 189). Any `print()` to stdout is silently discarded. The existing `--check-mediakeys` already works around this by printing to stderr and relying on the exit code — the D-05 self-test must follow the same discipline.
**Why it happens:** `console=False` is intentional (no console window flash). It's a known pyinstaller behavior. [VERIFIED: MusicStreamer.spec line `console=False`]
**How to avoid:** Exit-code only for the self-test contract. build.ps1 branches on `$LASTEXITCODE`, not on stdout parsing.

### Pitfall 5: `runtime_hook.py` env setup runs before `oauth_helper.main()` — no extra wiring needed

**What goes wrong (potential concern):** Does the re-exec'd frozen helper process get the correct `GIO_EXTRA_MODULES`, `GI_TYPELIB_PATH`, and `GST_PLUGIN_SCANNER` env vars needed for Qt/GStreamer to function?
**Resolution:** Yes. Runtime hooks run at frozen-exe startup, before any Python code executes, including `__main__.py`. Whether the frozen exe is the main GUI process or a re-exec'd helper, `runtime_hook.py` has already run and set the env vars. The subprocess (the re-exec'd helper) is a fresh process launched by QProcess — it gets its own runtime hooks automatically because it is also the frozen exe. [VERIFIED: runtime_hook.py logic — sets env vars unconditionally at module level; applies to any invocation of the frozen exe]
**Warning signs:** QtWebEngine fails to find typelibs or TLS module in the re-exec'd helper. If this happens, `--oauth-helper` mode may need to set additional env vars — but this is not expected given the runtime-hook design.

### Pitfall 6: QProcess `errorOccurred` may fire for errors other than FailedToStart

**What goes wrong:** `errorOccurred` fires for all `QProcess.ProcessError` values (FailedToStart, Crashed, Timedout, ReadError, WriteError, UnknownError). The D-02 fallback should ideally only fire on `FailedToStart` (process never launched), not on `Crashed` (which is handled by `finished` with non-zero exit code already feeding `_classify_and_show_failure`).
**How to avoid:** In `_on_gbs_login_error`, check `if error == QProcess.ProcessError.FailedToStart:` before auto-opening the cookie-import fallback. For other error types, log at WARNING but let `finished` handle the UX (it already calls `_classify_and_show_failure`).
**Warning signs:** `CookieImportDialog` opens unexpectedly after a crash (Crashed → both `errorOccurred` and `finished` fire; without the FailedToStart check, D-02 triggers twice).

## D-04 Linux Static Test — Design

### Test file: `tests/test_launcher_frozen_dispatch.py`

**Shape:** Pure Python, no QProcess execution, no Windows build. Uses `monkeypatch` to set `sys.frozen = True` and `sys.executable = "/fake/MusicStreamer.exe"`, then calls the shared launch helper (or invokes `_make_oauth_launch_args`) and asserts the returned `(program, args)` tuple.

Mirrors the shape of `tests/test_media_keys_smtc.py` Task 2b tests (confirmed in codebase — they monkeypatch `QCoreApplication` and call `_run_check_mediakeys` directly).

```python
# tests/test_launcher_frozen_dispatch.py — illustrative shape

def test_frozen_branch_uses_exe_argv_dispatch(monkeypatch):
    """D-04: when sys.frozen is True, launcher returns (sys.executable, ['--oauth-helper', '--mode', m])."""
    monkeypatch.setattr(sys, "frozen", True, raising=False)
    monkeypatch.setattr(sys, "executable", "/fake/MusicStreamer.exe")
    from musicstreamer.subprocess_utils import _make_oauth_launch_args
    prog, args = _make_oauth_launch_args("gbs")
    assert prog == "/fake/MusicStreamer.exe"
    assert "--oauth-helper" in args
    assert "--mode" in args
    assert "gbs" in args
    assert "-m" not in args

def test_non_frozen_branch_uses_module_form(monkeypatch):
```
