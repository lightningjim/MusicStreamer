# Phase 999.9 Plan 01: YouTube Playback Regression Fix ‚Äî Summary

**Real root cause: yt-dlp's library API does not auto-discover JS runtimes the way the CLI does.**
**Fix: `js_runtimes={"node": {"path": None}}` added to YoutubeDL opts; YouTube live playback restored.**

## Outcome

UAT approved 2026-04-24. LoFi Girl plays end-to-end. Cookies sha256 unchanged across UAT and persists through reboot ‚Äî Phase 999.7 byte-identity invariant intact.

## What actually happened

The plan-as-written assumed the bug was either (a) the dead `youtubepot-jsruntime` extractor_args namespace, or (b) needing to pin a working `player_client`. The terminal probe matrix supported (b) ‚Äî `web`, `web_safari`, `mweb` all returned valid HLS URLs from the CLI. Branch A fix shipped: pin `player_client=web`, remove dead namespace.

**In-app UAT immediately failed** with the same "No video formats found!" error for both the original station (`jXAEIWcGXwE`) and the canonical probe URL (`jfKfPfyJRdk`) ‚Äî even though both URLs still succeeded from the CLI in the same session. That meant Branch A wasn't the real fix; the divergence was elsewhere.

Replicating the exact in-app opts dict inline via `uv run python` reproduced the failure and surfaced the warning the in-app `quiet=True` had been suppressing:

```
WARNING: [youtube] jXAEIWcGXwE: n challenge solving failed: Some formats may
be missing. Ensure you have a supported JavaScript runtime and challenge
solver script distribution installed.
```

Two more probes narrowed it:
- `yt_dlp_ejs` package is NOT installed in the venv ‚Äî bundled solver is unavailable.
- Adding `remote_components=["ejs:github"]` did NOT help the library API.
- Adding `js_runtimes={"node": {"path": None}}` DID ‚Äî yt-dlp logged `[jsc:node] Solving JS challenges using node`, then resolved a valid HLS URL.

The CLI auto-discovers Node when deno isn't installed; the library API does not. That's the bug. The dead extractor_args namespace and the player_client rotation were both red herrings (the namespace cleanup is still legitimate housekeeping, but it wasn't load-bearing for the symptom).

## Lesson

When the CLI succeeds but the library API fails with the same opts, the gap is almost always a default the CLI's `_real_main` auto-populates that the library does not. The first diagnostic should be: replicate the exact in-app opts via `uv run python` and compare verbose output against `uv run yt-dlp -v`. Chasing `player_client` rotation or `extractor_args` API drift before that step (as this phase did) is wasted motion ‚Äî the symptom looks identical regardless of which default is missing.

This is the second phase in two days where a yt-dlp/streamlink upgrade silently dropped/changed an API the project depends on (Phase 999.8 was `Streamlink.set_plugin_option` ‚Üí `set_option`; this one is library auto-discovery of `js_runtimes`). Both surfaced as silent failures inside daemon worker threads. The 2026-05-01 scheduled sweep (`trig_0135THdv9pRxjvFoJVrD5ifh`) covers the daemon-thread silencing; nothing structural prevents the next CLI-vs-library divergence yet.
