# Phase 87.1: Code Review Report

**Reviewed:** 2026-06-17T00:00:00Z
**Depth:** standard
**Files Reviewed:** 4
**Status:** issues_found

## Summary

This gap-closure run reviews the GBS auth-expiry detection rewrite (Plan 87.1-04).
The core fix in `_open_with_cookies` is **correct**: installing a `_NoRedirect`
handler that returns `fp` stops the redirect chain at the first 302 so the
`Location` header is inspectable, and the `getattr(resp, "status", None)` /
`resp.headers.get("Location")` reads are verified valid against the real CPython
`http.client.HTTPResponse` surface (status, headers, and `__enter__` all present).
The login-substring match correctly handles both absolute and relative `Location`
URLs, and the happy-path 200 JSON return is preserved. The marquee exception
narrowing (`except gbs_api.GbsAuthExpiredError: raise` before the generic
`except Exception`) is correct and necessary — without it the auth-expired signal
would be unreachable. All eight changed-path tests pass.

The rewrite did, however, introduce a **second outbound HTTP request** on the
non-login-redirect branch (re-issue with a follow-opener). That re-issue path
(a) violates the project-wide GET-with-side-effects no-retry invariant for the
vote/ajax callers, (b) is unguarded against `HTTPError` for a malformed
(Location-less) 3xx, and (c) is not actually exercised for read-correctness by
any test. None rise to BLOCKER because all require gbs.fm to emit a non-login
3xx on a GET that today returns 200 or a login-302 — bounded but real.

## Warnings

### WR-01: Non-login redirect re-issue double-fetches GET-with-side-effects (vote) — violates Pitfall 7 no-retry invariant

**File:** `musicstreamer/gbs_api.py:204-226`
**Issue:** When `_open_with_cookies` sees a non-login 3xx, it closes the first
response and **re-issues the entire request** through a fresh redirect-following
opener:

```python
if getattr(resp, "status", None) in (301, 302, 303, 307):
    ...
    follow_opener = urllib.request.build_opener(
        urllib.request.HTTPCookieProcessor(cookies)
    )
    return follow_opener.open(
        urllib.request.Request(url, headers={"User-Agent": _USER_AGENT}),
        timeout=timeout,
    )
```

`vote_now_playing` (line 424) and `fetch_active_playlist`/`/ajax` requests all go
through `_open_with_cookies`. The module docstring (Pitfall 7, line 23) and
`vote_now_playing`'s own docstring explicitly state these are **GET-with-side-effects
with NO retries**. The old implementation issued exactly one request (the default
`HTTPRedirectHandler` followed redirects in-band on the *same* logical fetch). The
new code issues the original GET, then a *second* GET of the same URL — so a vote
that the server answers with a 302 would be cast twice. This is a behavioral
regression against a documented invariant.
**Fix:** Do not blind-re-issue. Either follow the redirect from the *already-received*
response (read its `Location` and open the target once, not the original URL again),
or — simpler and matching the documented "stop at first redirect" intent — treat any
non-login 3xx the same way `_open_no_redirect` does and return the stopped response,
letting callers that expect JSON fail loudly rather than silently double-firing a
side-effecting GET. At minimum, gate the re-issue so it never runs for the vote path.

### WR-02: Malformed (Location-less) non-login 3xx escapes as raw HTTPError past the try/except

**File:** `musicstreamer/gbs_api.py:196-226`
**Issue:** The `except urllib.error.HTTPError` block (lines 197-202) wraps only the
*first* `opener.open(req, ...)`. The re-issue `follow_opener.open(...)` at lines
223-226 is outside any try/except. If a non-login 3xx arrives with a missing or
empty `Location` header, the login-substring check is False (`"/accounts/login/" in ""`),
control falls into the re-issue branch, and the default `HTTPRedirectHandler` raises
`HTTPError` (it cannot follow a redirect with no `Location`). That `HTTPError`
propagates uncaught out of `_open_with_cookies`. Callers that catch `HTTPError`
(`search`, `fetch_user_tokens`, `fetch_artist_songs`, `fetch_album_songs`) degrade
gracefully, but `fetch_active_playlist` and `vote_now_playing` have **no** `HTTPError`
handler, so they surface a raw `HTTPError` instead of the documented degraded path.
**Fix:** Wrap the re-issue `.open()` in the same `try/except urllib.error.HTTPError`
that converts a login-302 to `GbsAuthExpiredError` and re-raises others, or short-circuit:
if a 3xx has no `Location`, raise `GbsApiError` (or return the stopped response) rather
than attempting a follow that cannot succeed.

### WR-03: Re-issue branch correctness (reading the redirect target body) is untested

**File:** `tests/test_gbs_api.py:528-552`
**Issue:** `test_open_with_cookies_non_login_302_does_not_raise_auth_expired` only
asserts that a non-login 302 does **not** raise `GbsAuthExpiredError`; it explicitly
accepts "any other exception" as a pass (lines 551-552). The `_fake_do_open` returns
the *same* `fake_resp` object on the re-issue call, whose `http.client.HTTPResponse`
socket buffer is already consumed — so the test cannot and does not verify that the
re-issued request returns a *readable* body. The entire happy-redirect path
(non-login 3xx → follow → 200 body readable by the caller) added by this change has
no positive assertion. Combined with WR-01/WR-02, the most behaviorally novel branch
of the rewrite ships unverified.
**Fix:** Add a test where `do_open` returns a 302-to-`/somewhere-else` on the first
call and a fresh readable 200 response on the second, asserting `resp.read()` yields
the expected body. This pins the re-issue contract and would have surfaced WR-01's
double-fetch (the test would observe two `do_open` invocations).

## Info

### IN-01: Docstring references stale line numbers for `_open_with_cookies`

**File:** `musicstreamer/gbs_api.py:368-369`
**Issue:** `fetch_user_tokens`'s docstring says "The `_open_with_cookies` helper at
lines 146-160 already converts 302→login" but the helper now lives at lines 158-228
after the rewrite. Stale line references rot quickly and mislead future readers.
**Fix:** Replace the line-number citation with a symbolic reference
(`see _open_with_cookies`), which does not drift on edits.

### IN-02: `_NoRedirect` is defined identically in two functions (duplication)

**File:** `musicstreamer/gbs_api.py:183-189` and `musicstreamer/gbs_api.py:246-256`
**Issue:** The inner `_NoRedirect(HTTPRedirectHandler)` class with the
`http_error_302 = http_error_301 = http_error_307 = http_error_303` override is now
defined twice (once in `_open_with_cookies`, once in `_open_no_redirect`) with
identical bodies. The two copies can silently diverge (e.g., one gaining a 308
override and the other not), and re-defining the class on every call is needless.
**Fix:** Hoist a single module-level `_NoRedirectHandler` class and instantiate it in
both helpers.
