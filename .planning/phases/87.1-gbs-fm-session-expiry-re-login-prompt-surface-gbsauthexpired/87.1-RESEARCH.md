## Summary

Detection of GBS.FM session expiry is ALREADY complete. `gbs_api.py:86` defines `GbsAuthExpiredError`; `gbs_api.py:171` raises it on a `302 → /accounts/login/` redirect from any authenticated call. The gap is entirely in handling/UX: the `_on_gbs_playlist_error` slot in `now_playing_panel.py` (~line 3088) silently hides the `_gbs_playlist_widget`, stops the poll timer, and flips `_gbs_ajax_disabled = True` — with no user-visible indication.

The fix has three interlocking parts: (1) a new inline expiry prompt widget that occupies the `_gbs_playlist_widget` slot (D-01/D-03), (2) a shared `GbsReloginHandler` QObject that owns the `oauth_helper --mode gbs` QProcess and de-dupes concurrent expiry signals (GBS-AUTH-EXP-02), and (3) a success signal that the playlist poller, marquee worker, and eventually Phase 87b zero-token add all connect to for post-login refresh (D-04, GBS-AUTH-EXP-02).

The implementation is entirely within `now_playing_panel.py` + a new `musicstreamer/ui_qt/gbs_relogin_handler.py`. No changes to `gbs_api.py` or `accounts_dialog.py` public contracts. The QProcess subprocess wiring from `accounts_dialog.py` is the canonical pattern to mirror — not import or reuse directly (to avoid circular deps), but copy the shape exactly: `_gbs_login_proc: QProcess | None`, `_make_oauth_launch_args("gbs")`, finished + errorOccurred wiring, and the `_parse_oauth_stderr` → `_validate_gbs_cookies` success check.

**Primary recommendation:** Implement `GbsReloginHandler` as a `QObject` (not `QThread`) living on the main thread, owned by `NowPlayingPanel` (or `MainWindow`). It owns a `QProcess`, de-dupes via a `_login_in_flight: bool` flag, emits `relogin_succeeded` on success, and `relogin_failed` on cancel/crash. `NowPlayingPanel` connects its inline prompt's "Log in again" button to `handler.request_relogin()`. On `relogin_succeeded`, the panel hides the prompt, resets `_gbs_ajax_disabled`, and calls `_refresh_gbs_visibility()` to resume polling.

## Shared Expiry→Re-login Handler Design

### Recommended Shape: `GbsReloginHandler` QObject (main-thread resident)

**Module:** `musicstreamer/ui_qt/gbs_relogin_handler.py`

This is a `QObject` (NOT `QThread`) that lives on the main thread. It owns a single `QProcess` for the oauth_helper subprocess and provides the single-flight de-dup gate. Ownership: constructed once by `NowPlayingPanel.__init__` (or by `MainWindow`, passed to the panel). Lifetime: same as the panel.

```python
class GbsReloginHandler(QObject):
    """Shared GBS session expiry → re-login handler.

    Lives on the main thread. Provides single-flight de-duplication so
    concurrent expiry signals from multiple workers (playlist poller,
    marquee worker, Phase 87b zero-token add) trigger only one
    oauth_helper subprocess.

    Signals:
        relogin_succeeded: Emitted after the subprocess writes valid
            Netscape cookies and they are persisted to gbs_cookies_path().
        relogin_failed: Emitted if the subprocess exits non-zero, was
            cancelled, or fails to start. Payload is a short human-readable
            reason string.
    """
    relogin_succeeded = Signal()
    relogin_failed = Signal(str)   # reason — for optional surface

    def __init__(self, parent: QObject | None = None) -> None:
        super().__init__(parent)
        self._login_proc: QProcess | None = None  # WR-03 guard

    def notify_expiry_detected(self) -> None:
        """Called by any consumer that detects GbsAuthExpiredError.

        No-op if a login subprocess is already in flight (single-flight).
        Safe to call from the main thread only (no cross-thread calls here).
        """
        if self._login_proc is not None:
            return  # WR-03: already in flight
        self._launch()

    def _launch(self) -> None:
        self._login_proc = QProcess(self)
        self._login_proc.finished.connect(self._on_finished)  # QA-05
        self._login_proc.errorOccurred.connect(self._on_error)  # QA-05
        from musicstreamer.subprocess_utils import _make_oauth_launch_args
        program, args = _make_oauth_launch_args("gbs")
        self._login_proc.start(program, args)

    def _on_finished(self, exit_code: int, exit_status: QProcess.ExitStatus) -> None:
        proc = self._login_proc
        self._login_proc = None
        # Same success check as AccountsDialog._on_gbs_login_finished
        from musicstreamer.gbs_api import _validate_gbs_cookies
        from musicstreamer import paths
        netscape_text = ""
        if proc is not None:
            try:
                netscape_text = proc.readAllStandardOutput().data().decode("utf-8", errors="replace")
            except Exception:
                pass
        last_event = AccountsDialog._parse_oauth_stderr(proc)  # static method, no instance needed
        success_category = last_event is None or last_event.get("category") == "Success"
        if exit_code == 0 and netscape_text and success_category and _validate_gbs_cookies(netscape_text):
            import os
            cookies_path = paths.gbs_cookies_path()
            os.makedirs(os.path.dirname(cookies_path), exist_ok=True)
            with open(cookies_path, "w", encoding="utf-8") as fh:
                fh.write(netscape_text)
            os.chmod(cookies_path, 0o600)
            self.relogin_succeeded.emit()
        else:
            self.relogin_failed.emit("Login was cancelled or failed")

    def _on_error(self, error: QProcess.ProcessError) -> None:
        if error == QProcess.ProcessError.FailedToStart:
            self._login_proc = None
            self.relogin_failed.emit("Could not start login helper")
```

**Note on `_parse_oauth_stderr` reuse:** `AccountsDialog._parse_oauth_stderr` is a `@staticmethod` — it can be called as `AccountsDialog._parse_oauth_stderr(proc)` without an instance. However, to avoid importing `AccountsDialog` into the new module (which could create coupling), duplicate the ~15-line parse logic inline in `GbsReloginHandler._on_finished`. The pattern is: `proc.readAllStandardError()` → decode → splitlines → `json.loads` each line → keep last dict with `"category"` key. [VERIFIED: accounts_dialog.py:457-494]

**Alternatively** (recommended for testability): extract `_parse_oauth_stderr` into `musicstreamer/subprocess_utils.py` alongside `_make_oauth_launch_args`. Both are subprocess-layer utilities and neither has Qt widget dependencies. [ASSUMED — planner decides]

### Single-Flight De-duplication

The `_login_proc is not None` check in `notify_expiry_detected` is sufficient. The playlist poller fires first (it's the fastest cycle), the marquee fires seconds later — both call `notify_expiry_detected()`. The second call sees `_login_proc is not None` and returns immediately. Both consumers will receive `relogin_succeeded` when the ONE subprocess completes.

### Consumer Connections

self._gbs_relogin_handler.relogin_failed.connect(

`GbsMarqueeWorker` runs on a `QThread`. It cannot call `handler.notify_expiry_detected()` directly (main-thread-only method). Two options:

**Option A (recommended):** Add a `auth_expired = Signal()` to `GbsMarqueeWorker`. In `_fetch_marquee()`, replace the silent log with `self.auth_expired.emit()`. In `attach_gbs_marquee_worker()` (or `MainWindow`), connect `worker.auth_expired` → `gbs_relogin_handler.notify_expiry_detected` with `Qt.QueuedConnection`. This crosses the thread boundary correctly.

**Option B:** `GbsMarqueeWorker` continues to catch and log; MainWindow detects re-login and calls `marquee_worker.force_poll()` after `relogin_succeeded`. This is acceptable for the marquee refresh use case (GBS-AUTH-EXP-02 says "reused" — the marquee refreshes after success, which is achievable without the marquee feeding into the handler). ## Login Launch + Success Detection

### What `_make_oauth_launch_args("gbs")` Does

[VERIFIED: subprocess_utils.py:34-72]

Returns `(program, args)` where:
- Frozen bundle: `(str(Path(sys.executable).parent / "oauth_helper" / "oauth_helper.exe"), ["--mode", "gbs"])`
- Source/dev/CI: `(sys.executable, ["-m", "musicstreamer.oauth_helper", "--mode", "gbs"])`

The subprocess writes Netscape cookie text to stdout on success. The caller reads it via `proc.readAllStandardOutput()`. `finished` signal provides `(exit_code, exit_status)`.

### Success Criteria (mirror AccountsDialog exactly)

[VERIFIED: accounts_dialog.py:584-610]

All four conditions must hold:
1. `exit_code == 0`
2. `netscape_text` is non-empty
3. `last_event is None OR last_event.get("category") == "Success"` (stderr parse)
4. `_validate_gbs_cookies(netscape_text)` returns True

Write to `paths.gbs_cookies_path()` with `os.chmod(path, 0o600)`. Then emit `relogin_succeeded`.

### FailedToStart Handling

[VERIFIED: accounts_dialog.py:627-652]

`QProcess.errorOccurred` fires INSTEAD of `finished` on `FailedToStart`. The handler must wire `errorOccurred` as well. On `FailedToStart`, reset `_login_proc = None` and emit `relogin_failed`. Other `ProcessError` values (Crashed) will ALSO emit `finished`, so the existing `_on_finished` failure path handles them — only `FailedToStart` requires the guard.

### Direct-launch vs Open-AccountsDialog

**Recommendation: direct launch** (CONTEXT.md's preferred direction). The QProcess wiring is 15–20 lines of proven code with established error paths. Opening AccountsDialog requires a parent widget, focuses on the wrong surface (all accounts vs GBS specifically), and adds an unnecessary dialog layer for a one-click re-login affordance. [ASSUMED — planner confirms]

The only cost of direct-launch: the handler must duplicate the stderr-parse logic (~15 lines). Worth it for the UX gain.

## Poll-Timer Stop/Resume Mechanics

### Current Timer Lifecycle

[VERIFIED: now_playing_panel.py:698-700, 2914-2957, 3088-3116]

```
Construction:
  _gbs_poll_timer = QTimer(self)
  _gbs_poll_timer.setInterval(15000)
  _gbs_poll_timer.timeout.connect(_on_gbs_poll_tick)

_refresh_gbs_visibility (should_show=True):
  → clears cursor, clears list, kicks _on_gbs_poll_tick(), starts timer

_refresh_gbs_visibility (should_show=False):
  → _gbs_poll_timer.stop()

_on_gbs_playlist_error (auth_expired):
  → _gbs_poll_timer.stop()  ← D-04 already satisfied

bind_station (new station):
  → calls _refresh_gbs_visibility()
```

### Resume Mechanic (D-04: resumes only after successful re-login)

On `relogin_succeeded`, the panel's `_on_gbs_relogin_succeeded` slot:
1. Hides `_gbs_expiry_widget`
2. Resets `_gbs_ajax_disabled = False`
3. Resets `_gbs_label_source = None` (fresh state for the resumed session)
4. Calls `_refresh_gbs_visibility()` — this re-checks `_is_gbs_logged_in()` (cookie file now exists with fresh content) and `is_gbs` (station still bound) → `should_show=True` → kicks the poll and starts timer

This is the cleanest path: reuse the existing visibility-gate logic rather than directly calling `_gbs_poll_timer.start()`. [ASSUMED — planner confirms]

### Cancel Path (D-02/GBS-AUTH-EXP-03)

On `relogin_failed` (user closed the login window, subprocess failed to start):
- The prompt stays visible (D-02: non-dismissive)
- Log the failure
- Optionally: update the prompt sub-line to "Login failed or was cancelled — try again" and re-enable the "Log in again" button (which was disabled while the subprocess was in flight)
- The poll timer stays stopped (D-04: only resumes on success)
- User sees a clear state, not an empty widget — satisfies GBS-AUTH-EXP-03

## Post-Login Refresh/Retry Wiring

### Active-Playlist (primary, auto-refreshes)

`relogin_succeeded` → `_on_gbs_relogin_succeeded` → `_refresh_gbs_visibility()` → timer starts + immediate poll tick. Full auto-refresh.

### Marquee Worker (GBS-AUTH-EXP-02)

Option A (marquee emits `auth_expired` signal): on `relogin_succeeded`, `NowPlayingPanel` calls `self._gbs_marquee_worker.force_poll()` — this is already a safe cross-thread call (it emits `cadence_changed_internal` via `Qt.QueuedConnection`).

Option B: same as above, marquee just gets a force_poll on success regardless.

### Vote, Search, Submit (user-action call sites)

Per CONTEXT.md discretion: no auto-retry. User re-clicks the vote/search/submit button. The vote buttons will be visible again after `_refresh_gbs_visibility` resumes the poll and `_apply_vote_buttons_enabled` fires on the next successful `/ajax` response.

## Common Pitfalls

### Pitfall 1: Multiple QProcess subprocesses from concurrent expiry

**What goes wrong:** Playlist poller and marquee both call `notify_expiry_detected()` in the same 15s window. Without the `_login_proc is not None` guard, two `oauth_helper` processes launch simultaneously, both write to the same cookies file, and both emit `relogin_succeeded` — the panel refreshes twice.

**Prevention:** `GbsReloginHandler._login_proc is not None` guard in `notify_expiry_detected()`. The second caller's signal is silently dropped; both consumers receive the ONE `relogin_succeeded` emission.

**Warning signs:** Two `oauth_helper` windows appear. Test by monkeypatching `QProcess.start` to count calls.

### Pitfall 2: QProcess `errorOccurred` fires INSTEAD of `finished` on FailedToStart

**What goes wrong:** If `oauth_helper.exe` is missing (frozen bundle) or the module path is wrong (dev), `FailedToStart` fires `errorOccurred` but NOT `finished`. Without the `errorOccurred` handler, `_login_proc` stays non-None permanently, blocking all future re-login attempts for the session.

**Prevention:** Wire `errorOccurred` in `_launch()`. On `FailedToStart`, reset `_login_proc = None`, emit `relogin_failed`. [VERIFIED: accounts_dialog.py:627-652 — exact same issue and fix]

**Warning signs:** "Log in again" button appears to do nothing after first failure. No `relogin_failed` emission.

### Pitfall 3: Cookie file race — reading stale cookies after successful write

**What goes wrong:** `_on_gbs_relogin_succeeded` calls `_refresh_gbs_visibility()` → `_on_gbs_poll_tick()` → `gbs_api.load_auth_context()` loads the cookie file. If the file write in `_on_finished` completed on the main thread (it did — QProcess callbacks are on main thread), the read in `_on_gbs_poll_tick` sees the fresh cookies. No race.

**Prevention:** Keep the entire success flow (write cookies → emit relogin_succeeded → refresh visibility → load cookies) on the main thread. QProcess `finished` signal fires on the main thread. No threading issue.

**Warning signs:** First poll after re-login returns `cookies = None` → `_refresh_gbs_visibility()` runs again → hides the widget. Diagnose by logging `gbs_api.load_auth_context()` result in `_on_gbs_poll_tick`.

### Pitfall 4: Expiry prompt shown for non-GBS stations or when logged out initially

**What goes wrong:** `_gbs_expiry_widget.setVisible(True)` is called by `_on_gbs_playlist_error`. If a stale token somehow arrives after station re-bind to a non-GBS station, the prompt shows on the wrong station.

**Prevention:** `_on_gbs_playlist_error` already has `if token != self._gbs_poll_token: return` at ~line 3097. Stale tokens are already discarded. The prompt only shows when the token matches the current GBS poll cycle. Additionally, `_refresh_gbs_visibility(should_show=False)` hides `_gbs_expiry_widget` when leaving GBS context.

**Warning signs:** Expiry prompt appears on a SomaFM station. Test by binding GBS (trigger expiry), then binding a non-GBS station — verify prompt hides.

### Pitfall 5: Re-entrancy on the "Log in again" button while subprocess is in flight

**What goes wrong:** User clicks "Log in again" → subprocess starts → user clicks again → second subprocess would launch.

**Prevention:** `GbsReloginHandler.notify_expiry_detected()` checks `_login_proc is not None` → returns. Additionally, the button can be `setEnabled(False)` while `_login_proc` is in flight, re-enabled on `relogin_succeeded` or `relogin_failed`.

### Pitfall 6: Expiry prompt persists after user manually re-logs-in via Accounts dialog

**What goes wrong:** User opens Accounts dialog, connects GBS manually. The `_gbs_expiry_widget` stays visible and the poll timer stays stopped — the panel doesn't know about the external login.

**Prevention:** `AccountsDialog._on_gbs_login_finished` emits `self._toast_callback("GBS.FM logged in.")`. MainWindow could connect this path to also trigger `gbs_relogin_handler.notify_succeeded()` (a method that emits `relogin_succeeded` directly without a subprocess). OR: the existing `_update_status` call in AccountsDialog could be wired to trigger panel refresh. [ASSUMED — planner should decide: recommended to add a `MainWindow._on_gbs_login_externally_completed` slot that calls `panel._refresh_gbs_visibility()` or `relogin_handler.emit_succeeded()`]

### Pitfall 7: Vote buttons stay disabled after re-login

**What goes wrong:** After `relogin_succeeded`, the poll resumes and shows the playlist. But vote buttons are enabled only after a successful `/ajax` response stamps `_gbs_current_entryid` (the `_apply_vote_buttons_enabled` call in `_on_gbs_playlist_ready`). This is CORRECT — buttons re-enable on the next poll success. No action needed; document this for UAT.

**Warning signs:** Vote buttons never re-enable. Diagnose by checking that `_on_gbs_playlist_ready` is firing and calling `_apply_vote_buttons_enabled(True)`.

### Pitfall 8: QA-05 — no self-capturing lambdas on signal connections

**What goes wrong:** `self._gbs_expiry_btn.clicked.connect(lambda: self._on_gbs_relogin_clicked())` creates a reference cycle.

**Prevention:** Use `self._gbs_expiry_btn.clicked.connect(self._on_gbs_relogin_clicked)` (bound method, QA-05 invariant).

### Pitfall 9: Marquee `auth_expired` signal crosses thread boundary incorrectly

**What goes wrong:** If `GbsMarqueeWorker._fetch_marquee()` called `handler.notify_expiry_detected()` directly, it would call a main-thread method from a worker thread — unsafe.

**Prevention:** `GbsMarqueeWorker` emits a Signal (with `Qt.QueuedConnection` to the handler's slot). The signal enqueues a call on the main event loop. [VERIFIED: marquee's existing `set_cadence` and `force_poll` methods both emit signals for the same reason — gbs_marquee.py:486-505]

## Open Questions

1. **GbsReloginHandler ownership — panel vs MainWindow?**
   - What we know: `GbsMarqueeWorker` is owned by `MainWindow` (constructed there, injected into panel via `attach_gbs_marquee_worker`). `GbsReloginHandler` needs to receive marquee's `auth_expired` signal AND emit `relogin_succeeded` back to the panel.
   - What's unclear: Whether to own the handler in `NowPlayingPanel` (simpler wiring, panel owns all GBS state) or in `MainWindow` (mirrors marquee worker ownership, easier for future Phase 87b wiring).
   - Recommendation: Own in `NowPlayingPanel`. The panel already owns all GBS poll/vote state. MainWindow only needs to forward the marquee `auth_expired` signal (or wire it in `attach_gbs_marquee_worker`).

2. **Should `_parse_oauth_stderr` be extracted to `subprocess_utils.py`?**
   - What we know: The logic is 15 lines, pure stdlib (json + string ops), no Qt dependency. `subprocess_utils.py` already has `_make_oauth_launch_args`.
   - Recommendation: Yes, extract. Clean separation, easy to test, removes the only coupling between `GbsReloginHandler` and `AccountsDialog`. This is a one-line import change in `AccountsDialog` after extraction.

3. **Should the "Log in again" button be disabled while the subprocess is in flight?**
   - Recommendation: Yes. On `_on_gbs_relogin_clicked`, disable the button; on `relogin_succeeded` or `relogin_failed`, re-enable it. Prevents user confusion about whether the click registered.
