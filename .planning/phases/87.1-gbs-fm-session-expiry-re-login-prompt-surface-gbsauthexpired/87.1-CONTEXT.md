# Phase 87.1: GBS.FM Session-Expiry Re-login Prompt - Context

**Gathered:** 2026-06-17
**Status:** Ready for planning

<domain>
## Phase Boundary

When a GBS.FM session cookie has expired, the app surfaces a clear, actionable "GBS session expired — please log in again" prompt that launches the existing in-app GBS login (`oauth_helper --mode gbs`) and refreshes on success — instead of the active playlist silently failing to load.

Detection ALREADY exists (`gbs_api.py` raises `GbsAuthExpiredError` on a `302 → /accounts/login/` redirect). This phase is about HANDLING/UX: catching that exception at the GBS call sites and routing to a re-login affordance, plus a SHARED expiry→re-login handler the Phase 87 `GbsMarqueeWorker` poller and Phase 87b zero-token add can reuse.

**Out of scope:** detecting expiry (done), changing the login subprocess itself, any new GBS capability.
</domain>

<decisions>
## Implementation Decisions

### Prompt surface & form (locked)
- **D-01:** The re-login prompt for the failed active-playlist load is an **inline panel state** — replace the empty/failed active-playlist sub-region of the now-playing panel with a persistent inline message + "Log in again" button. NOT a toast, NOT a dismissible banner. (Replaces the current silent path at `now_playing_panel.py:_on_gbs_playlist_error`, ~line 3088, which hides the widget and flips to ICY with no indication.)
- **D-02:** The prompt is **non-dismissive** — it stays in place until re-login succeeds (or the user navigates away from the GBS station). This satisfies the roadmap's "no silent dead-ends" requirement; there is no × / dismiss that returns to an empty state.
- **D-03:** **Playback is not interrupted.** Audio keeps streaming and live ICY now-playing metadata stays visible; only the GBS active-playlist sub-region is swapped for the inline prompt. Expiry of the GBS overlay must not interrupt the stream the user is hearing.
- **D-04:** While the prompt is shown, the **active-playlist poll timer stops** until re-login succeeds (it currently stops on error — keep that). No background re-polling of an endpoint that will 302 every interval; the user's re-login is the retry trigger. Resume polling only after a successful re-login.
- **D-05:** **Wording:** primary message "GBS session expired" with a sub-line along the lines of "Log in again to keep the playlist updating", and a primary button "Log in again". Plain, non-jargony, consistent with the existing GBS toast voice. Exact micro-copy may be refined in planning but must preserve cause (expired) + action (log in again).

### Claude's Discretion (resolve in research/planning, guided by roadmap + existing patterns)
- **Login launch path** — How the prompt launches login. Recommended direction: a **shared launcher that starts `oauth_helper --mode gbs` directly** (extract/reuse the QProcess wiring currently living only in `AccountsDialog`), matching the roadmap's "launches the existing GBS login flow." Opening `AccountsDialog` focused on GBS is the fallback if direct extraction is too invasive — but the phase intent is a one-click re-login, not "go to Accounts." This is the core of the SHARED handler (GBS-AUTH-EXP-02), so design it once and let the playlist loader, Phase 87 marquee poller, and Phase 87b zero-token add all consume it.
- **Post-login retry/refresh** — What auto-recovers on successful re-login. Recommended: the shared handler emits a "re-login succeeded" signal; the active-playlist loader reloads + resumes its poll timer (D-04); the marquee worker refreshes. Whether other call sites (vote/search/submit) auto-retry the specific failed action vs require a manual re-trigger is left to planning — bias toward refreshing passive/auto surfaces (playlist, marquee) and letting explicit user actions (a vote, a submit) be re-triggered by the user.
- **Scope: unify existing call sites** — Whether to migrate the existing passive "GBS.FM session expired — reconnect via Accounts" toasts (search/vote/submit/import in `gbs_search_dialog.py`, `now_playing_panel.py` vote, `main_window.py` import) onto the new shared actionable handler. Recommended: keep this phase anchored on the **playlist loader + a reusable shared handler** that 87 marquee / 87b zero-token consume (GBS-AUTH-EXP-01/02); opportunistically route the other call sites through the same handler if low-cost, but do not let a broad toast-migration balloon the phase. Marquee currently fails silently (D-18, `gbs_marquee.py:355`) — planning should decide whether the shared handler changes that.
- **Concurrent expiry / single-flight** — Multiple workers (poller, marquee, vote) can hit `GbsAuthExpiredError` at once. The shared handler should de-dupe so only one re-login affordance/launch is active at a time (avoid prompt spam / multiple `oauth_helper` subprocesses). Implementation detail, but call it out in the plan.
</decisions>

<specifics>
## Specific Ideas

- The reported symptom that motivates this phase: the active playlist would not load with **no indication why**; the user had to manually log out and back in to recover. The inline prompt (D-01..D-05) is the direct fix for that "no indication" gap.
- Reuse, don't reinvent: detection (`GbsAuthExpiredError`), the login subprocess (`oauth_helper --mode gbs`), and cookie storage (`paths.gbs_cookies_path`) all already exist.
</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`musicstreamer/gbs_api.py`** — `GbsAuthExpiredError` + all GBS endpoint functions. The exception type to catch at every call site.
- **`musicstreamer/ui_qt/accounts_dialog.py`** — owns the only existing `oauth_helper --mode gbs` launcher (QProcess `_launch_gbs_login_subprocess`, success via `_on_gbs_login_finished` checking `paths.gbs_cookies_path()` existence). The shared launcher should be factored out of / from this wiring.
- **`musicstreamer/ui_qt/now_playing_panel.py`** — `_GbsPollWorker` (active-playlist poller, ~line 107) and `_on_gbs_playlist_error` (~line 3088, the current silent-fail path to replace); `_GbsVoteWorker` / `_on_gbs_vote_error` (~3290) already emit an "auth_expired" toast.
- **`musicstreamer/gbs_marquee.py`** — `GbsMarqueeWorker` (QThread, ~line 437); `_fetch_marquee` catches `GbsAuthExpiredError` and currently logs silently (~line 355, D-18). Must consume the shared handler (GBS-AUTH-EXP-02).
- **`musicstreamer/ui_qt/gbs_search_dialog.py`** — multiple `_Gbs*Worker`s already catch the error and emit an "auth_expired" sentinel → passive toast + `_refresh_login_gate()`.
- **`musicstreamer/ui_qt/toast.py`** / **`announcement_banner.py`** — existing surfaces; explicitly NOT chosen for this prompt (D-01), but inform the inline-state visual treatment.
- **`paths.gbs_cookies_path()`** + `cookie_utils.is_cookie_file_corrupted` — cookie storage / refresh-detection.

### Established Patterns
- GBS workers run on QThread/QObject and surface errors via Qt signals (sentinel "auth_expired") connected in `main_window` / panel slots — the shared handler should follow this signal pattern.
- Login-success detection across the app = "does `paths.gbs_cookies_path()` now load a valid auth context?" (`gbs_api.load_auth_context()`), then re-gate dependent UI.

### Integration Points
- The shared expiry→re-login handler is the new seam: consumed by the active-playlist loader (now_playing_panel), the Phase 87 marquee poller (gbs_marquee), and the Phase 87b zero-token add. Design it as a single reusable unit (signal in: expiry detected; signal out: re-login succeeded) rather than per-call-site logic.
</code_context>
