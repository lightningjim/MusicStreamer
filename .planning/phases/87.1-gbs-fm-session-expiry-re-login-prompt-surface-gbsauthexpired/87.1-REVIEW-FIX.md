## Fixed Issues

### WR-01: Non-login redirect re-issue double-fetches GET-with-side-effects (vote) — violates Pitfall 7 no-retry invariant

**Files modified:** `musicstreamer/gbs_api.py`
**Commit:** 28311df0
**Applied fix:** Removed the blind re-issue branch in `_open_with_cookies`. On a
non-login 3xx, the helper now returns the STOPPED (no-redirect) response unchanged
and logs a warning, rather than building a fresh redirect-following opener and
re-issuing the same GET. This is the review's preferred "treat any non-login 3xx
the same way `_open_no_redirect` does and return the stopped response" option. The
second outbound request that would have double-fired a side-effecting vote/ajax GET
is eliminated; the 200-JSON happy path never enters the 3xx branch and is
unaffected; a genuine non-login redirect now surfaces as a JSON-decode failure in
the caller (fail loud), which is correct for an endpoint that today returns 200 or
a login-302. Verified: `python3 ast.parse` OK; full `tests/test_gbs_api.py` (57
tests) pass with the changed behavior — the existing non-login-302 guard test still
passes because it only fails on `GbsAuthExpiredError`.

### WR-02: Malformed (Location-less) non-login 3xx escapes as raw HTTPError past the try/except

**Files modified:** `musicstreamer/gbs_api.py`
**Commit:** 28311df0 (resolved by the WR-01 fix — same edit)
**Applied fix:** WR-02's defect was that the re-issue `follow_opener.open(...)`
(lines 223-226) sat OUTSIDE any try/except, so a Location-less non-login 3xx would
make the default `HTTPRedirectHandler` raise an uncaught `HTTPError` that propagated
out to `fetch_active_playlist`/`vote_now_playing` (which have no `HTTPError`
handler). The WR-01 fix deletes the entire re-issue branch, so there is no longer
any `.open()` call outside the try/except — the offending code path no longer
exists. A Location-less non-login 3xx now returns the stopped response (empty
Location) and the caller fails loud on the JSON decode. Confirmed via
`grep follow_opener musicstreamer/gbs_api.py` → no matches remain. No separate edit
was introduced, since doing so would have re-added dead code; WR-02 is fully
addressed by the WR-01 commit.

### WR-03: Re-issue branch correctness (reading the redirect target body) is untested

**Files modified:** `tests/test_gbs_api.py`
**Commit:** f8460d3d
**Applied fix:** Added `test_open_with_cookies_non_login_3xx_does_not_re_issue`,
adapted to the post-WR-01 contract. The review originally asked for a test proving
the re-issue returns a readable body AND surfacing the double-fetch; because WR-01
removed re-issuing entirely, the correct contract to pin is now "exactly ONE
do_open call, returning the stopped 302." The test patches
`AbstractHTTPHandler.do_open` with an invocation counter, asserts it is called
exactly once (this would have caught the original double-fetch — it observes two
invocations under the old code), asserts the returned response is the stopped 302
(status 302), and asserts the `Location` header is readable. Verified: new test
passes in isolation and the full `tests/test_gbs_api.py` suite is 58 passed (was 57).
