The bug reproduction requires the systemd-session-inherited PATH (which strips fnm/nvm/volta/asdf shims). The cleanest way to get that:
- Log out of the GNOME session (or reboot).

- DO NOT open a terminal first. Launch MusicStreamer ONLY by clicking the icon in the GNOME dock / Activities / app grid.

(Alternative if logout is inconvenient: launch via `gtk-launch musicstreamer.desktop` from a terminal. This goes through GNOME's launcher and inherits the systemd-session PATH instead of the terminal PATH. Note this in the SUMMARY if used.)

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User → GNOME shell | User clicks the `.desktop` launcher icon. The launch context is the systemd user session (stripped PATH). |
| pipx shim → musicstreamer process | The shim binary inherits the launcher's environment; the child Python process sees the same (stripped) PATH. |
| musicstreamer → yt-dlp library API | `Player._youtube_resolve_worker` constructs `yt_dlp.YoutubeDL(opts)` with `opts["js_runtimes"]["node"]["path"] = NodeRuntime.path`. yt-dlp then `subprocess.run([<path>, "--version"])`. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-79-05-01 | Tampering | UAT execution itself (the user clicks the .desktop launcher) | accept | The UAT is a manual verification step. The user is the legitimate operator; no untrusted input crosses any boundary that wasn't already in play during normal app use. |
| T-79-05-02 | Information Disclosure | journalctl excerpt captured in SUMMARY includes the absolute Node path | accept | The SUMMARY lives in `.planning/phases/79-...` which is committed to the project's git repo (QNAP Gitea + GitHub mirror per user-memory `reference_qnap_github_mirror.md`). The absolute path reveals the user's home-directory layout and the version-manager choice (fnm vs nvm vs volta vs asdf).
