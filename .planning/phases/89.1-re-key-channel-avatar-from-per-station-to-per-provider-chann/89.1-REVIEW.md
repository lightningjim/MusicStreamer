# Phase 89.1: Code Review Report

**Reviewed:** 2026-06-16
**Depth:** deep (cross-file call-chain tracing)
**Files Reviewed:** 6
**Status:** issues-found

## Summary

The phase correctly re-keys channel avatars from per-station to per-provider across all six files. The backfill in `db_init` is crash-safe (shutil.copy2, not os.rename), idempotent (NOT EXISTS guard), skips NULL provider_id stations, and runs the copy loop in full before any per-station file deletion — the ordering is safe. The four row-to-Station mappers all include the `p.avatar_path AS provider_avatar_path` LEFT JOIN alias. `update_provider_avatar_path` is a correctly isolated single-column UPDATE. `_channel_avatar_lookup` is purely synchronous, reads only one field, and never raises. The `bind_station` reset-before-set ordering (L936 before L943) prevents stale-station bleed, and the `os.path.isfile` guard is correct.

One blocker was found: the avatar fetch worker can be launched with `provider_id=None` for stations that lack a provider, producing a junk `None.png` file on disk and a silent DB no-op (the UPDATE matches zero rows). The user sees a false-success status message. No exception is raised, so the failure is invisible.

## Critical Issues

### CR-01: `_AvatarFetchWorker` launched with `provider_id=None` — junk file + silent DB no-op

**File:** `musicstreamer/ui_qt/edit_station_dialog.py:1306-1327` (launch site) and `1476,1485` (result handler)

**Issue:** In `_on_url_timer_timeout`, the avatar fetch worker is launched whenever the URL contains `youtube.com` or `youtu.be`, without checking whether `self._station.provider_id` is non-null. The D-07 reuse-on-open gate (L1309-1313) short-circuits only when `provider_avatar_path` is already set; it does not guard against a null `provider_id`.

When `provider_id` is `None`:

1. `_AvatarFetchWorker.run()` calls `assets.write_provider_avatar(None, data)` (L176).
2. `write_provider_avatar` formats the filename as `f"{None}.png"` → `"None.png"` and writes that file to disk — a junk artifact that is never cleaned up.
3. `_on_avatar_fetched` (L1485) calls `self._repo.update_provider_avatar_path(None, rel_path)`, which executes `UPDATE providers SET avatar_path = ? WHERE id = ?` with `(rel_path, None)`. In SQLite, `WHERE id = NULL` matches zero rows — silent no-op.
4. The in-memory `self._station.provider_avatar_path` is updated (L1476), so the preview label shows the avatar correctly.
5. No exception is raised anywhere in this path, so the `except` block at L1486 never fires. The status text is set to the "Avatar saved — all ... stations updated" success message even though nothing was persisted to the DB.

The blast radius: any YouTube-URL station with `provider_id = None` (a station inserted with an empty `provider_name`) will silently accumulate a `None.png` orphan file on every avatar fetch, and the avatar will never persist across restarts.

**Fix:** Add a `provider_id` null guard at the worker launch site, before the avatar token increment:

```python
# In _on_url_timer_timeout, after the D-07 reuse gate, before token increment:
if self._station.provider_id is None:
    self._avatar_status.setText("Cannot fetch avatar: station has no provider")
    return

self._avatar_fetch_token += 1
...
```

Alternatively, add a defensive guard at the top of `write_provider_avatar`:

```python
def write_provider_avatar(provider_id: int, data: bytes) -> str:
    if provider_id is None:
        raise ValueError("provider_id must not be None")
    ...
```

The launch-site guard is preferred because it gives the user visible feedback and avoids wasting network time fetching an avatar that cannot be stored.
