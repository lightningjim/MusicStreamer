## Summary

Phase 89.1 re-keys the channel-avatar from the `stations` table to the `providers` table so that sibling YouTube streams sharing the same channel reuse one avatar fetch and one cached file. The existing Phase 89 mechanism (per-station `channel_avatar_path`) is fully in place; this phase migrates that data to a new `providers.avatar_path` column, rewires the four consumer sites (now_playing_panel, cover_art._channel_avatar_lookup, EditStationDialog, and the repo persist method), and backfills existing users' avatars at `db_init()` startup.

The migration follows established Phase 89A / 89 patterns exactly: idempotent additive-ALTER, atomic `mkstemp`+`os.replace` file writes, dedicated non-silent-reset persist method, and a `_root_override`-friendly path accessor. The only novel element is the file-copy backfill step inside `db_init()`, which requires careful handling because `db_init` receives a raw `sqlite3.Connection` — no Repo wrapper — but CAN import and use `paths` and `assets` helpers safely.

**Primary recommendation:** Structure the phase as four clean task groups: (1) schema + storage, (2) backfill migration, (3) repo/model re-wiring + new persist method, (4) UI consumer re-wiring + dialog changes. The backfill belongs inside `db_init()` using a nested file-copy helper that imports `paths` and calls `os.rename`/`shutil.copy2` + `os.unlink`. No Repo wrapper is needed for the backfill — raw `con.execute` is appropriate.

## Detailed Findings

### 1. Schema: `providers` table has NO rebuild block

[VERIFIED: grep of repo.py] The `providers` table is defined at `repo.py:91–95` as a plain `CREATE TABLE IF NOT EXISTS` inside the `executescript` block. There is NO `providers_new` rebuild — confirmed by exhaustive search (`grep -n "providers_new\|providers.*rebuild" repo.py` returned nothing). The additive-ALTER Pitfall-2 ordering constraint that applies to `stations` does **not** apply here.

The `providers` table columns are `id INTEGER PRIMARY KEY AUTOINCREMENT, name TEXT NOT NULL UNIQUE`. The new ALTER:

```python
# Phase 89.1 D-11 — provider-level channel avatar path; nullable TEXT no DEFAULT.
# NULL means no provider avatar stored. providers has NO legacy rebuild block
# (unlike stations), so ordering constraint does not apply. Idempotent via the
# same try/except OperationalError idiom as the Phase 73/82/83/89A station blocks.
try:
    con.execute("ALTER TABLE providers ADD COLUMN avatar_path TEXT")
    con.commit()
except sqlite3.OperationalError:
    pass  # column already exists — idempotent
```

This block should be placed AFTER the `stations.channel_avatar_path` ALTER block (end of `db_init` at L324) to maintain chronological ordering with prior phases. No technical dependency on ordering within `db_init` for `providers`, but convention is to append.

### 2. Backfill mechanics: query shape and seam analysis

**Key constraint:** `db_init(con)` receives a raw `sqlite3.Connection`, NOT a `Repo` wrapper. The `Repo` wrapper is constructed one line after `db_init` returns in `__main__.py:287`. However, `db_init` CAN safely import and use `paths` and `assets` helpers — they are pure Python with no Qt dependency. `write_channel_avatar` (assets.py:32) calls `os.makedirs(dst_dir, exist_ok=True)` on L43, so it is self-sufficient even if `ensure_dirs()` has not been called.

[ASSUMED] Option A: Run the backfill entirely inside `db_init()`, using raw `con.execute` for DB operations and importing `paths`/`os`/`shutil` directly for file operations. This is the cleanest seam because `db_init` already does the ALTER, and the backfill is logically part of the same migration pass.

[ASSUMED] Option B: Run the backfill in a new `Repo` method called from `__main__.py` after `db_init` and `Repo(con)` construction. This gives the backfill access to the full Repo API but adds complexity to the startup sequence.

**Recommendation (Claude's discretion — D-02/D-03):** Option A. The backfill is a one-time migration; placing it in `db_init` mirrors the established pattern and keeps it atomic with the schema change. The file operations are safe at that point because `write_channel_avatar` handles its own `makedirs`.

**Exact SQL for the backfill (D-02 most-recently-updated tiebreak):**

`updated_at` is stored as `TEXT DEFAULT (datetime('now'))` — SQLite `datetime('now')` produces ISO-8601 strings (`'2026-06-15 12:34:56'`), which sort correctly as text because the format is lexicographically ordered. `MAX(updated_at)` or `ORDER BY updated_at DESC LIMIT 1` per provider group both work.

```sql
-- Select the single best avatar row per provider (most-recently-updated station wins)
SELECT s.provider_id, s.channel_avatar_path, s.id AS station_id
FROM stations s
WHERE s.channel_avatar_path IS NOT NULL
  AND s.provider_id IS NOT NULL
GROUP BY s.provider_id
HAVING s.updated_at = MAX(s.updated_at)
-- Note: HAVING on aggregated MAX is valid in SQLite, but use a subquery for clarity:
```

The safest / most readable form avoids `GROUP BY + HAVING` on non-aggregated columns (SQLite is permissive but behavior is undefined for non-aggregate columns in GROUP BY without explicit aggregation):

```sql
SELECT s.provider_id, s.channel_avatar_path, s.id AS station_id
FROM stations s
WHERE s.channel_avatar_path IS NOT NULL
  AND s.provider_id IS NOT NULL
  AND s.updated_at = (
      SELECT MAX(s2.updated_at)
      FROM stations s2
      WHERE s2.provider_id = s.provider_id
        AND s2.channel_avatar_path IS NOT NULL
  )
```

If two stations have identical `updated_at` (edge case: both imported in the same second), take the one with the highest `station_id` (the later-inserted sibling). Use `ORDER BY s.updated_at DESC, s.id DESC LIMIT 1` per provider in a subquery.

**Complete backfill idiom for `db_init`:**

```python
# Phase 89.1 D-01/D-02/D-03: one-time idempotent backfill.
# For each provider whose providers.avatar_path is still NULL, find the
# most-recently-updated station with a channel_avatar_path and copy its
# PNG to the provider-keyed location.
import os, shutil
from musicstreamer import paths as _paths

avatar_dir = _paths.channel_avatars_dir()
rows = con.execute(
    """
    SELECT s.provider_id, s.channel_avatar_path, s.id AS station_id
    FROM stations s
    WHERE s.channel_avatar_path IS NOT NULL
      AND s.provider_id IS NOT NULL
      AND NOT EXISTS (
          SELECT 1 FROM providers p
          WHERE p.id = s.provider_id AND p.avatar_path IS NOT NULL
      )
    ORDER BY s.provider_id, s.updated_at DESC, s.id DESC
    """
).fetchall()
seen_providers: set = set()
for row in rows:
    pid = row[0]
    if pid in seen_providers:
        continue  # only process the first (best) row per provider
    seen_providers.add(pid)
    rel_path: str = row[1]
    # rel_path e.g. 'assets/channel-avatars/12.png'
    src = os.path.join(_paths.data_dir(), rel_path)
    if not os.path.isfile(src):
        continue  # avatar on disk missing — skip, don't corrupt DB
    dst = os.path.join(avatar_dir, f"{pid}.png")
    provider_rel = os.path.relpath(dst, _paths.data_dir())
    try:
        os.makedirs(avatar_dir, exist_ok=True)
        shutil.copy2(src, dst)
    except OSError:
        continue  # skip this provider on copy failure — don't abort db_init
    con.execute(
        "UPDATE providers SET avatar_path = ? WHERE id = ?",
        (provider_rel, pid),
    )
    con.commit()

# Clean up per-station files whose provider now has an avatar
cleanup_rows = con.execute(
    """
    SELECT s.channel_avatar_path
    FROM stations s
    JOIN providers p ON p.id = s.provider_id
    WHERE s.channel_avatar_path IS NOT NULL
      AND p.avatar_path IS NOT NULL
    """
).fetchall()
for row in cleanup_rows:
    old_abs = os.path.join(_paths.data_dir(), row[0])
    try:
        os.unlink(old_abs)
    except OSError:
        pass  # already gone — idempotent
```

**Idempotency proof:** The backfill query filters `NOT EXISTS (... p.avatar_path IS NOT NULL)` — once `providers.avatar_path` is written, the row is excluded on the next run. The cleanup pass deletes files that may already be gone (swallows `OSError`). Double-run is a no-op.

**Rollback safety:** If `db_init` crashes mid-backfill (e.g. after copying the file but before the `UPDATE providers` commit), on the next startup the `NOT EXISTS` filter correctly re-runs for that provider (because `providers.avatar_path` is still NULL) and copies the PNG again (harmless overwrite via `shutil.copy2`). No partial state is observable by callers.

### 6. `write_provider_avatar` — new or re-keyed writer in `assets.py` (D-10)

**Recommendation (Claude's discretion — D-10):** Add a sibling `write_provider_avatar(provider_id, data)` rather than re-keying `write_channel_avatar` in place. This avoids breaking the existing `_AvatarFetchWorker.run()` call at `edit_station_dialog.py:175` that still uses the station-keyed writer during the transition.
```python
def write_provider_avatar(provider_id: int, data: bytes) -> str:
    """Write avatar PNG bytes atomically keyed by provider_id (Phase 89.1 D-10).

    Returns path relative to paths.data_dir(), e.g. 'assets/channel-avatars/7.png'.
    Uses tempfile.mkstemp + os.replace for atomicity (same as write_channel_avatar).
    provider_id is an int from SQLite — no traversal risk (T-89.1-01).
    """
    dst_dir = paths.channel_avatars_dir()
    os.makedirs(dst_dir, exist_ok=True)
    dst = os.path.join(dst_dir, f"{provider_id}.png")
    fd, tmp = tempfile.mkstemp(dir=dst_dir, suffix=".png.tmp")
    try:
        os.write(fd, data)
        os.close(fd)
        os.replace(tmp, dst)
    except Exception:
        try:
            os.close(fd)
        except OSError:
            pass
        try:
            os.unlink(tmp)
        except OSError:
            pass
        raise
    return os.path.relpath(dst, paths.data_dir())
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Avatar column write | Broad `UPDATE providers SET ... WHERE id = ?` covering ALL columns | Dedicated `update_provider_avatar_path(provider_id, path)` | Silent-reset hazard (Pitfall 5): a future broad provider update would overwrite `avatar_path` back to NULL if it doesn't know about the column |
| Atomic file write | `open(dst, 'wb').write(data)` | `tempfile.mkstemp(dir=...)` + `os.replace` | Non-atomic write leaves a partial file visible to readers if the process dies mid-write |
| Path resolution | Hardcoded `/home/user/.local/share/musicstreamer/assets/channel-avatars/` | `paths.channel_avatars_dir()` | Tests use `_root_override`; hardcoded paths break all test isolation |
| Migration idempotency | Ad-hoc "already done" flag | `NOT EXISTS (SELECT 1 FROM providers WHERE id=? AND avatar_path IS NOT NULL)` in backfill query | SQL filter is the correct atomic gate; a separate flag would race |

**Key insight:** The "dedicated single-column update" pattern was established precisely because Phase 89 discovered that routing avatar writes through the broad `update_station` silently reset `channel_avatar_path` whenever the user saved other station metadata. The same hazard applies to providers — enforce the same guard.

## Common Pitfalls

### Pitfall 1: Placing the `providers` ALTER before verifying there is no rebuild
**What goes wrong:** If a `providers_new` rebuild existed, placing the ALTER before it would cause the column to be silently dropped (same as the `stations` Pitfall-2 from Phase 89A).
**Why it happens:** Cargo-culting the Pitfall-2 warning without verifying whether providers has an analogous rebuild.
**How to avoid:** Confirmed by direct grep: `grep "providers_new" repo.py` returns nothing. No rebuild block exists. The ALTER can be placed at the end of `db_init` without ordering concerns. [VERIFIED: repo.py grep]

### Pitfall 2: Backfill reading `station.channel_avatar_path` after D-04 cutover
**What goes wrong:** If new code in the same phase also stops writing `stations.channel_avatar_path`, the backfill might find nothing to migrate (the column would be empty for stations imported after the cutover but before a user's next app restart).
**Why it happens:** The cutover (stop writing to `stations.channel_avatar_path`) and the backfill (read from `stations.channel_avatar_path`) must be ordered correctly.
**How to avoid:** The backfill runs at `db_init` startup, BEFORE any UI code can write new avatar paths. The write-path cutover (D-04) removes the `update_channel_avatar_path(station_id, ...)` call from the dialog — which only fires at dialog open after startup. So `db_init` always runs before any dialog can change the state. Safe as designed.

### Pitfall 3: Non-idempotent file-rename on partial backfill
**What goes wrong:** Using `os.rename` to move `{station_id}.png` to `{provider_id}.png` deletes the source atomically on Linux — if the `UPDATE providers` commit fails afterward, the file is at the new location but the DB still says NULL. On re-run, the backfill selects the same station (DB still NULL) but the source file is gone.
**Why it happens:** `os.rename` is destructive and not a two-phase operation.
**How to avoid:** Use `shutil.copy2` (non-destructive copy) for the backfill step. The old file is deleted by the cleanup pass ONLY AFTER `providers.avatar_path` is committed. This way, a crash between copy and commit leaves both files; the next run re-copies (harmless overwrite) and retries the commit. The cleanup pass then removes the old file. [ASSUMED — no official doc, this is the correct concurrency analysis]

### Pitfall 4: `_set_avatar_pixmap_from_path` in `bind_station` without file-existence check
**What goes wrong:** `_set_avatar_pixmap_from_path` loads a `QPixmap` from the path. If the file doesn't exist, `QPixmap.isNull()` is true, and it falls back to `_show_station_logo_in_cover_slot()`. However, `bind_station` has already called `_show_station_logo_in_cover_slot()` at L932 — the fallback is a no-op overwrite of the same content. The real risk is `_last_avatar_path` being set to a non-null bad path, causing `_apply_art_tier` to retry the bad path on resize.
**Why it happens:** `_set_avatar_pixmap_from_path` at L2212–2213 correctly clears `_last_avatar_path = None` on `isNull()` — the current code handles this correctly. BUT: the new resolution reads `provider_avatar_path`, which is a relative path from the DB. If the file was deleted out-of-band, the DB says non-null but the file is gone.
**How to avoid:** Add an explicit `os.path.isfile()` check in `bind_station` before calling `_set_avatar_pixmap_from_path` (as shown in the code example in §Finding 3). This avoids a QPixmap construction attempt for a known-missing file.

### Pitfall 5: Non-silent-reset hazard for providers
**What goes wrong:** If `update_provider_avatar_path` is NOT a dedicated method, and instead the avatar path is cleared whenever any other provider field is updated (e.g. `providers.name` via `ensure_provider`), all avatars get wiped.
**Why it happens:** `ensure_provider` uses `INSERT OR IGNORE` — it never updates an existing row. But a future phase adding a broad `update_provider` method could silently reset `avatar_path` if the writer doesn't include it.
**How to avoid:** Add the `update_provider_avatar_path` as a dedicated method (D-09) with a doc comment citing Pitfall 5. No broad `UPDATE providers SET ... WHERE id = ?` should include `avatar_path` except this dedicated method. [ASSUMED — forward-looking design concern]

### Pitfall 6: Four mappers, one missed
**What goes wrong:** The four row→Station mappers in repo.py (`list_stations` L544, `get_station` L584, `list_recently_played` L693, `list_favorite_stations` L811) must ALL include `p.avatar_path AS provider_avatar_path` in their SELECT and populate `station.provider_avatar_path`. Missing even one causes some consumers to see `None` for a station that has an avatar.
**Why it happens:** Copy-paste oversight. Phase 89 already had to thread `channel_avatar_path` through all four.
**How to avoid:** The planner's task list must explicitly enumerate all four mappers. A drift-guard test (`test_provider_avatar_path_in_all_mappers`) should verify that `list_stations`, `get_station`, `list_recently_played`, and `list_favorite_stations` all return the correct `provider_avatar_path`.

### Pitfall 7: `_channel_avatar_lookup` tests reference the old field name
**What goes wrong:** `test_cover_art_avatar.py` tests `station.channel_avatar_path` directly. After re-pointing `_channel_avatar_lookup` to use `provider_avatar_path`, all those tests pass a mock with `channel_avatar_path` set — but the production code now reads `provider_avatar_path`. Tests pass with wrong attribute, production breaks.
**Why it happens:** Tests use `MagicMock()` which creates attributes on access — `station.channel_avatar_path` and `station.provider_avatar_path` both return a `MagicMock` truthy value.
**How to avoid:** Update all `_channel_avatar_lookup` tests to set `station.provider_avatar_path` (not `channel_avatar_path`) and use `spec=[...]` or `spec=Station` to prevent attribute auto-creation. The planner must update `test_cover_art_avatar.py` as part of the re-wire tasks.

```python
           s.channel_avatar_path,
           s.id AS station_id

               WHERE p.id = s.provider_id AND p.avatar_path IS NOT NULL
           )
    ORDER BY s.provider_id, s.id DESC
    """
).fetchall()
```
