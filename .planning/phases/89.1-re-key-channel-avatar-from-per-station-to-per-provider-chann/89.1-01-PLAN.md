<objective>
Re-key the channel avatar from the `stations` table to the `providers` table at the data layer. Add `providers.avatar_path`, backfill existing per-station PNGs to provider-keyed files at `db_init()` startup, add a provider-keyed atomic writer and a dedicated non-silent-reset persist method, carry `provider_avatar_path` on the `Station` model through all four row→Station mappers, and stop writing the deprecated `stations.channel_avatar_path`.

Purpose: One avatar fetch + one cached file per YouTube channel instead of per station (the Lofi-Girl-siblings dedup goal). This plan is the data foundation; Plan 02 rewires the UI consumers to read it.
Output: `providers.avatar_path` column + idempotent backfill, `assets.write_provider_avatar`, `repo.update_provider_avatar_path`, `Station.provider_avatar_path` populated by all four mappers, and the Wave-0 test suite that locks all of it.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SQLite row → filesystem path | `provider_id` (int from DB) is formatted into `{provider_id}.png`; the avatar bytes originate from a prior network fetch stored on disk |
| Backfill copy/delete during db_init | files are copied and old files deleted on disk during the one-time migration |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-89.1-01 | Tampering | assets.write_provider_avatar / backfill `{provider_id}.png` | mitigate | `provider_id` is an integer from SQLite (PK AUTOINCREMENT), never a user-controlled string — `f"{provider_id}.png"` cannot traverse out of channel_avatars_dir(). Confirm the key is integer-typed before formatting (it comes from providers.id / station.provider_id, both INTEGER). |
| T-89.1-02 | Tampering / Denial | repo.db_init backfill copy + delete | mitigate | Use `shutil.copy2` (NOT os.rename) so a crash between copy and the `UPDATE providers` commit leaves the source intact and the next run re-copies (idempotent). Delete the old per-station file ONLY after providers.avatar_path is committed (cleanup pass keyed on `p.avatar_path IS NOT NULL`). Never deletes a file whose provider lacks a committed avatar — no data loss on partial failure. |
| T-89.1-03 | Tampering / Info Disclosure | atomic PNG write | mitigate | tempfile.mkstemp in the destination dir + os.replace (same filesystem) — no torn/partial PNG visible to readers (Phase 89A D-12 mechanic). |
| T-89.1-04 | Denial of Service | backfill on malformed/missing on-disk avatar | accept | Missing source file is skipped (`if not os.path.isfile(src): continue`); copy OSError is swallowed per-provider and does not abort db_init. Low severity — worst case one provider keeps its NULL avatar and falls through to existing behavior. |

All dispositions are mitigate/accept; no high-severity threat — this surface is an int-keyed local-file migration with documented copy-not-rename and atomic-write mitigations.
</threat_model>
