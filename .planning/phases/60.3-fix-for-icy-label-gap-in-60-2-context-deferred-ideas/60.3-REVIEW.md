# Phase 60.3: Code Review Report

**Reviewed:** 2026-05-07T14:46:38Z
**Depth:** standard
**Files Reviewed:** 2
**Status:** issues_found

## Summary

Phase 60.3 wires three new behaviours onto `NowPlayingPanel`: a tri-state `_gbs_label_source` flag, an "ICY-then-/ajax" upgrade helper (`_apply_gbs_icy_label`), a kick-on-title-change debouncer (`_gbs_poll_in_flight`), a no-downgrade guard, a bridge-window star/cover-art gate, and an `auth_expired` -> `'icy'` fallback flip. The non-GBS code paths are correctly fenced behind the `provider_name == "GBS.FM"` guard, so SomaFM/DI.fm/etc. behaviour is unchanged.

That said, several decisions in the implementation do not actually deliver the contract that the CONTEXT/PLAN documents promise. The most serious are:

- **D-05 over-fires on track change** — the no-downgrade guard returns early on _every_ ICY arrival once `_gbs_label_source == 'ajax'`, including after a server-side track transition. The user keeps seeing the previous track's full Artist - Title for up to 15 s instead of upgrading to even the bare ICY title, AND the kick that D-03 was meant to deliver is suppressed in the same path. The CONTEXT explicitly described D-05 as a _title-suffix match_; the implementation uses a coarse flag that does not distinguish "ICY for the same track" from "ICY for a new track."
- **D-08 is broken when cookies remain on disk** — `_is_gbs_logged_in()` only checks file existence. `auth_expired` flips the flag but does not delete the cookie file, so subsequent `_update_star_enabled` and `in_bridge_window` checks still see `logged_in=True && flag != 'ajax'` and keep the star disabled / cover-art suppressed. The test author noticed this and removed the corresponding assertion (lines 1532-1544 of the test file are an extended apology rather than coverage).
- **Favourites display is not refreshed when /ajax stamps the canonical title** — `_apply_gbs_icy_label` updates `_last_icy_title` to the full Artist - Title but never re-queries `is_favorited` or updates the star icon/checked state/tooltip. The user can see a "non-starred" icon for a track that IS in favourites (or vice versa).
- **MagicMock truthy-by-default contaminates the existing playlist test** — `test_gbs_playlist_populates_from_mock_state` was patched to mock `_fetch_cover_art_async` but `_make_gbs_station()` returns a MagicMock whose auto-attribute `icy_disabled` is truthy. `_apply_gbs_icy_label` therefore early-returns on line 1140, exercising **none** of the new D-01/D-06 stamping behaviour the test was modified to cover. The mock-cover-art guard turns out to be unnecessary for the wrong reason.

The Qt threading reasoning in the implementation comment at lines 530-532 is correct (`title_changed.emit` at `player.py:446` is auto-queued to the main thread), so the direct call to `_on_gbs_poll_tick` is safe from the slot.

## Critical Issues

### CR-01: D-05 no-downgrade guard suppresses kick + label upgrade across track changes

**File:** `musicstreamer/ui_qt/now_playing_panel.py:542-543` (and the kick site at `:582-583`)
**Issue:** The no-downgrade guard reads:

```python
if is_gbs and self._gbs_label_source == 'ajax':
    return
```

This early-returns out of `on_title_changed` whenever the source flag is 'ajax', which means it fires on:

1. The intended case — the same-track ICY-after-/ajax race (CONTEXT line 129).
2. The unintended case — a server-side **track transition** where the new ICY tag arrives before the next /ajax tick. The flag is still 'ajax' from the prior track because nothing flips it back. Result: the label keeps showing the previous track's full Artist - Title, `_last_icy_title` is not updated, AND the D-03 kick at lines 582-583 never fires (it lives inside the same function that just returned). The user must wait for the 15 s `_gbs_poll_timer` tick — exactly the gap D-03 was designed to close.

The CONTEXT doc (line 41) explicitly described D-05 as "skips writing if the bare ICY title matches the title-suffix of the current /ajax-stamped label, ... or, equivalently, if a label-source flag indicates the label is currently /ajax-owned **for the active entryid**." The implementation dropped the entryid coupling without replacing it.

There's also a subtler version: when `_apply_gbs_icy_label` is called by `_on_gbs_playlist_ready` with `state.get("icy_title") or ""` and the server has cleared the title (mid-transition), the helper early-returns at line 1138 (`if not icy_title: return`) **without flipping the flag back to None or 'icy'**. The flag stays 'ajax' across the transition.

**Fix:** Either (a) carry the entryid into the source-flag state — `_gbs_label_source = ('ajax', entryid)` — and treat ICY-for-a-different-entryid as a fresh write; or (b) flip the flag when `_on_gbs_playlist_ready` sees a track change, before delegating to the helper:

```python
# in _on_gbs_playlist_ready, just after the track_changed computation:
if track_changed:
    self._gbs_label_source = None  # let the next ICY (or /ajax) re-stamp
```

If you take option (b), make sure the kick path still fires on track transitions. Simplest: split the no-downgrade early-return out of the kick path — i.e. let the kick at lines 582-583 run unconditionally in GBS context, and only short-circuit the label/cover-art writes:

```python
if not (is_gbs and self._gbs_label_source == 'ajax'):
    self.icy_label.setText(title or "")
    self._last_icy_title = title or ""
    if is_gbs:
        self._gbs_label_source = 'icy'
    self._update_star_enabled()
    # ... favorites + cover-art block ...
# Kick always runs on tag change in GBS context:
if is_gbs and not self._gbs_poll_in_flight():
    self._on_gbs_poll_tick()
```

### CR-02: D-08 auth-expired fallback does not actually relax the bridge gate when cookies file is still present

**File:** `musicstreamer/ui_qt/now_playing_panel.py:922-925, 668-675, 564-568, 1063-1083`
**Issue:** The CONTEXT D-08 promise is "when /ajax is impossible (auth-expired OR logged-out), star and cover-art remain enabled." The implementation only delivers the logged-out half. After `_on_gbs_playlist_error(token, "auth_expired")`:

- `self._gbs_label_source = 'icy'` is set.
- The cookie file is NOT deleted — it remains stale on disk.
- `_is_gbs_logged_in()` (line 922-925) is `os.path.exists(paths.gbs_cookies_path())`, which still returns True.
- On the next ICY tag:
  - `_update_star_enabled` evaluates `is_gbs && _is_gbs_logged_in() && flag != 'ajax'` -> `True && True && True` -> star **disabled**.
  - `in_bridge_window` in `on_title_changed:564-568` is the same predicate -> cover-art **suppressed**.
- Worse, `_on_gbs_poll_tick` still gets kicked from `on_title_changed` because the panel still believes it's in GBS-logged-in context. Each kick will re-raise `auth_expired` from the worker, re-flip the flag (no-op), and the user is stuck in a UX deadlock — a track is playing, the bare ICY shows in the label, but the star is greyed out and no cover art appears.

The test file acknowledges this at lines 1532-1544 in `test_gbs_auth_expired_relaxes_bridge_gate`:
> "the gate may still close. Document this nuance: this test asserts the FLAG flip + the ICY write succeeding; the star-enabled D-08 promise is verified by the next test."

The "next test" is `test_gbs_logged_out_icy_writes_normally`, which tests a totally different scenario (cookies never existed). The auth-expired path is not covered by any test.

**Fix:** Either delete the cookie file on `auth_expired` (matches the "session is gone" semantics that the rest of `gbs_api` raises), OR introduce a separate "ajax-impossible" flag that both the logged-out and auth-expired paths set:

```python
# In _on_gbs_playlist_error, auth_expired branch:
if msg == "auth_expired":
    self._gbs_playlist_widget.setVisible(False)
    self._gbs_poll_timer.stop()
    self._gbs_playlist_widget.clear()
    self._gbs_label_source = 'icy'
    self._gbs_ajax_disabled = True   # NEW
    # Optional: blow away the cookie file so _is_gbs_logged_in() also returns False
    # (matches the "session is dead" semantics of GbsAuthExpiredError).
```

Then change the bridge-gate predicate in `_update_star_enabled` and `on_title_changed`:

```python
# was: is_gbs and self._is_gbs_logged_in() and self._gbs_label_source != 'ajax'
# becomes:
ajax_possible = self._is_gbs_logged_in() and not getattr(self, '_gbs_ajax_disabled', False)
in_bridge_window = is_gbs and ajax_possible and self._gbs_label_source != 'ajax'
```

Reset `_gbs_ajax_disabled` in `_refresh_gbs_visibility` on station change so a fresh re-login isn't blocked.

### CR-03: `_apply_gbs_icy_label` does not refresh favourites display state on the star button

**File:** `musicstreamer/ui_qt/now_playing_panel.py:1118-1152` (helper omits favourites lookup that lives at `:551-560`)
**Issue:** The helper updates `self.icy_label`, `self._last_icy_title`, `self._gbs_label_source`, calls `_update_star_enabled`, and fires the cover-art lookup. It does NOT call `is_favorited(...)` or update `star_btn.setChecked / setIcon / setToolTip`.

Concrete defect path:
1. Bare ICY arrives: `on_title_changed("Track Y")` runs the favourites block at lines 551-560 with the BARE title and sets the star icon/check state based on `is_favorited(station_name, "Track Y")`. During the bridge window the star is disabled, but the **icon and `setChecked` state are still mutated** based on the bare-title query.
2. /ajax arrives: `_apply_gbs_icy_label("Artist X - Track Y")` writes the canonical string to `_last_icy_title` and re-enables the star. But it does NOT re-query `is_favorited(station_name, "Artist X - Track Y")` — so the star icon and `isChecked()` state still reflect the bare-title query result.
3. The icon may show "non-starred" while the canonical track is in favourites (or vice versa). The `_on_star_clicked` handler at lines 683-711 reads `_last_icy_title` (canonical) when the user clicks, so the click-action is correct, but the user's mental model ("the highlighted star means this is a favourite") is broken.

This contradicts D-06 ("all three downstream consumers — display, favorites key, cover-art query — read the same full Artist - Title value").

**Fix:** In `_apply_gbs_icy_label`, after the cover-art block, mirror the favourites block from `on_title_changed`:

```python
def _apply_gbs_icy_label(self, icy_title: str) -> None:
    if not icy_title:
        return
    if self._station is not None and self._station.icy_disabled:
        return
    self.icy_label.setText(icy_title)
    self._last_icy_title = icy_title
    self._gbs_label_source = 'ajax'
    self._update_star_enabled()
    # NEW: refresh favourites display so the star icon matches the canonical key.
    if self._station is not None:
        is_fav = self._repo.is_favorited(self._station.name, icy_title)
        self.star_btn.setChecked(is_fav)
        icon_name = "starred-symbolic" if is_fav else "non-starred-symbolic"
        self.star_btn.setIcon(
            QIcon.fromTheme(icon_name, QIcon(f":/icons/{icon_name}.svg"))
        )
        self.star_btn.setToolTip(
            "Remove track from favorites" if is_fav else "Save track to favorites"
        )
    if (
        not is_junk_title(icy_title)
        and self._station is not None
        and icy_title != self._last_cover_icy
    ):
        self._last_cover_icy = icy_title
        self._fetch_cover_art_async(icy_title)
```

Consider extracting the favourites-display block into a private helper to avoid duplication with `on_title_changed`.

### CR-04: `test_gbs_playlist_populates_from_mock_state` does not actually exercise the new `_apply_gbs_icy_label` path it was modified to defend

**File:** `tests/test_now_playing_panel.py:1005-1058`
**Issue:** The diff added `monkeypatch.setattr(panel, "_fetch_cover_art_async", MagicMock())` at line 1027 to "preserve offline-CI invariant." The intent is that `_on_gbs_playlist_ready` now calls `_apply_gbs_icy_label`, which fires cover-art. But:

- `_make_gbs_station` returns a `MagicMock()` — `s.icy_disabled` is a MagicMock auto-attribute, **which is truthy**.
- `_apply_gbs_icy_label` line 1140: `if self._station is not None and self._station.icy_disabled: return`. This **early-returns** on every MagicMock-backed test that does not explicitly set `icy_disabled = False`.
- Net effect: the new `_apply_gbs_icy_label` invocation at line 1061 of the panel does NOTHING in this test. The cover-art mock is unneeded because cover-art is never reached. The test passes only because its assertions are scoped to `_gbs_playlist_widget.item(...)` — the widget population, not the label stamping.

This is a silent regression hole: a future bug in `_apply_gbs_icy_label`'s widget interaction won't surface here. The test is also misleading — its diff comment claims the helper "now fires cover-art lookup," but with the current setup the helper short-circuits before cover-art.

**Fix:** Add `gbs_station = _make_gbs_station(); gbs_station.icy_disabled = False; panel.bind_station(gbs_station)` (matching the pattern used in the new Phase 60.3 tests at lines 1253, 1301, 1385, 1419, 1453, 1482, 1520, 1559) and add an assertion on `panel.icy_label.text() == "Crippling Alcoholism - Templeton"` and `panel._gbs_label_source == 'ajax'` so the test actually exercises the helper.

Better: fix `_make_gbs_station` once at the helper level so all 22 GBS tests benefit:

```python
def _make_gbs_station(provider_name: str = "GBS.FM", name: str = "GBS.FM"):
    s = MagicMock()
    s.id = 99
    s.name = name
    s.provider_name = provider_name
    s.tags = ""
    s.streams = []
    s.icy_disabled = False  # NEW: defeat MagicMock truthy default for the icy_disabled gate
    return s
```

Then drop the `gbs_station.icy_disabled = False` lines from each new test (and keep the explicit `True` override in `test_gbs_icy_disabled_suppresses_ajax_stamp` at line 1342, which is testing the LOCK).

## Warnings

### WR-01: `_apply_gbs_icy_label` accepts non-GBS stations — silent semantic drift

**File:** `musicstreamer/ui_qt/now_playing_panel.py:1118-1152`
**Issue:** The helper has no `provider_name == "GBS.FM"` guard. If something in the future calls `_apply_gbs_icy_label` from a non-GBS context (e.g., a spike that reuses the helper), it will set `self._gbs_label_source = 'ajax'` on a non-GBS station, which then leaks into the bridge-gate evaluations the next time `on_title_changed` fires (wait — the bridge gate is itself fenced on `provider_name == "GBS.FM"`, so the leak is limited; but the flag-state machine is now describing an impossible state).

Today the only call site is `_on_gbs_playlist_ready` which is only wired in GBS context, so this is latent rather than active. Still, defensive programming for a method named `_apply_gbs_*`:

**Fix:** Add an early-return:

```python
def _apply_gbs_icy_label(self, icy_title: str) -> None:
    if not icy_title:
        return
    if self._station is None or self._station.provider_name != "GBS.FM":
        return  # NEW — defensive: helper is GBS-only by name
    if self._station.icy_disabled:
        return
    ...
```

### WR-02: Source-flag wipe in `_refresh_gbs_visibility` overwrites the 'icy' state set in `_on_gbs_playlist_error`

**File:** `musicstreamer/ui_qt/now_playing_panel.py:927-966`
**Issue:** Sequence:
1. User is bound to GBS, logged in, /ajax has stamped, `_gbs_label_source == 'ajax'`.
2. Mid-session, the next poll returns `auth_expired`. `_on_gbs_playlist_error` flips to `'icy'`, hides widget, stops timer.
3. User triggers any code path that re-runs `_refresh_gbs_visibility` for the SAME GBS station — e.g., `bind_station` is called again (Phase 64 D-04 says it runs once per bind, but note `_on_play_pause_clicked` at line 634 calls `bind_station(self._station)` after a stop/play cycle).
4. `_refresh_gbs_visibility` evaluates `should_show = is_gbs && logged_in`. The cookie file is still on disk (CR-02), so `should_show=True`. Line 944 wipes `self._gbs_label_source = None`, undoing the `'icy'` flip from step 2.
5. The bridge-gate predicate is now evaluated with flag=None, which behaves the same as flag != 'ajax' — star/cover-art still gated. But subsequent ICY arrivals will flip flag to 'icy' again. So the practical effect is: the `'icy'` state set in `_on_gbs_playlist_error` does not survive a re-bind. Couples to CR-02 — if cookies were also deleted there, this becomes a non-issue.

**Fix:** Either delete the cookie file in the auth_expired branch (preferred — see CR-02 fix), or move the source-flag reset out of the `should_show=True` branch and only reset on first-bind-to-this-station. Simplest patch:

```python
if should_show:
    self._gbs_poll_cursor = {}
    # Don't reset _gbs_label_source if it's already 'icy' (auth_expired path
    # set this) — the next ICY or /ajax response is the right point to flip.
    if self._gbs_label_source not in ('icy', 'ajax'):
        self._gbs_label_source = None
    ...
```

### WR-03: Direct `self._station.icy_disabled` access in `on_title_changed` and `_apply_gbs_icy_label` differs from defensive `getattr` access in `bind_station`

**File:** `musicstreamer/ui_qt/now_playing_panel.py:533, 1140` (vs. `:504`)
**Issue:** `bind_station` reads `getattr(station, "icy_disabled", False)` (line 504). `on_title_changed` (line 533) and `_apply_gbs_icy_label` (line 1140) read `self._station.icy_disabled` directly. If a hand-rolled test double or a Station-shaped object lacks the attribute, `bind_station` silently treats it as False but the slot/helper raises `AttributeError`. The slot is connected via `Qt.QueuedConnection` from `player.title_changed`, and `_apply_gbs_icy_label` is reached via `_on_gbs_playlist_ready` (also a Qt slot via queued signal). Slots-never-raise contract is established at lines 781-787 (cover-art) and 859-867 (siblings). An AttributeError here would propagate out of the slot.

In production, all `Station` instances carry `icy_disabled` (it's a model field — see test fixtures at line 125 etc.), so the active risk is low. Test-double risk is real (see `_make_gbs_station` in CR-04). Defensive consistency is cheap.

**Fix:** Use `getattr(self._station, "icy_disabled", False)` at both sites.

### WR-04: `_apply_gbs_icy_label` widens the favourites/cover-art coupling without preserving the bare-ICY favourites display

**File:** `musicstreamer/ui_qt/now_playing_panel.py:1118-1152` and `:551-560`
**Issue:** During the bridge window, `on_title_changed` writes `_last_icy_title = "Track Y"` and queries favourites with the bare title. Then `_apply_gbs_icy_label` overwrites `_last_icy_title = "Artist X - Track Y"`. If the user had favourited the bare title in a non-GBS context (e.g., earlier when the station's `icy_disabled` was True or before the user logged in to GBS), the bare-title favourite is invisible to the post-/ajax UI.

This is more of a data-model concern than a per-commit defect, but the new helper makes it more visible — it "promotes" the favourites key without migrating prior bare-title favourites. Worth a comment in the helper noting that bare-title and full-title favourites are distinct rows.

**Fix:** Either document the invariant in the helper docstring, or add a soft-migration step (when `_apply_gbs_icy_label` runs and the bare title was favourited, also favourite the full title). Soft-migration changes data semantics — probably a future-phase decision. For now, a docstring note suffices.
