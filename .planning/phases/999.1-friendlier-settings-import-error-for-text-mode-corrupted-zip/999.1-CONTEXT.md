# Phase 999.1: Friendlier settings-import error for text-mode-corrupted zips - Context

**Gathered:** 2026-06-19
**Status:** Ready for planning

<domain>
## Phase Boundary

When a settings-backup `.zip` fails to open in `settings_export.preview_import`
(currently `zipfile.BadZipFile` → generic `ValueError("Not a valid ZIP archive")`),
detect the fingerprint of a **text-mode / CRLF-corrupted file transfer** and surface
an actionable recovery message telling the user to re-copy the file in binary mode.

**In scope:** detection of the text-mode-corruption signature + clearer user-facing
guidance on the file-picker settings-import path.

**Out of scope (locked by ROADMAP):**
- **No auto-repair.** Do NOT collapse `0x0D 0x0A` → `0x0A` — that would clobber
  legitimate binary byte pairs and cause silent data loss. Detection + guidance ONLY.
- No changes to the export/zip-build side (the bug is transfer corruption, not a defect).
- The Flatpak import wizard path is explicitly excluded (see D-04).

</domain>

<decisions>
## Implementation Decisions

### Detection (how tightly to fingerprint)
- **D-01:** Use a **strict CRLF fingerprint**, not a loose PK-magic-only check. Only
  classify as text-mode corruption when the file starts with the `PK` zip magic AND
  the raw bytes show evidence of injected `0x0D 0x0A` runs (CR inserted before LF
  bytes, as a text-mode/CRLF transfer would produce). Goal: minimize false positives so
  a genuinely truncated/damaged binary zip is NOT mislabeled as a transfer problem.
  - Byte-scan window size (the "first N bytes") and the exact injection-run test are
    left to research/planning — see RESEARCH NOTE below.

### Presentation (how the message shows)
- **D-02:** Surface the actionable message as a **modal dialog (`QMessageBox`)**, NOT a
  toast. Rationale: the current toast path (`_on_import_preview_error`,
  `main_window.py:1534`) truncates at 80 chars and is transient; the recovery
  instruction (~95 chars) needs to be fully readable and acknowledged. Follow the
  existing actionable-error dialog precedent at `main_window.py:955` (playback error
  `QMessageBox`).

### Fallback (non-text-mode BadZipFile)
- **D-03:** Keep a **two-tier** outcome:
  - text-mode signature detected → the specific "re-copy in binary mode" dialog (D-02).
  - any other `BadZipFile` (genuine truncation/damage, or no `PK` magic) → keep the
    generic message **plus a softer hint**: e.g. `"Not a valid ZIP archive — the file
    may be incomplete or corrupted."` (Generic + hint, not bare generic.)

### Scope of code paths
- **D-04:** Apply detection + dialog to the **file-picker settings-import path only**
  (`main_window.py:118`, `_PreviewWorker` → `_on_import_preview_error`). The Flatpak
  import wizard (`flatpak_import_wizard.py:135`) reads from a sandbox/`build_zip` copy it
  controls, where text-mode transfer corruption cannot realistically occur — leave it
  unchanged.

### Wording (locked)
- **D-05:** Dialog wording, as approved:
  - **Title:** `Settings import failed`
  - **Body:** `This backup looks corrupted by a text-mode file transfer. Re-copy it in
    binary mode and try again.`

### Claude's Discretion
- Exact byte-scan window length and the precise CRLF-injection detection algorithm
  (the mechanism that satisfies D-01's "strict" intent).
- The mechanism by which the worker/error path carries a discriminator so the UI knows
  to render a dialog (D-02) vs the existing toast for other errors — see RESEARCH NOTE.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `QMessageBox` actionable-dialog pattern at `main_window.py:955` (Icon.Warning,
  custom buttons) — template for the D-02 recovery dialog.
- `settings_export.preview_import` already raises typed `ValueError`s with distinct
  strings ("Missing settings.json", "Unsupported version: …", "Unsafe path in
  archive: …") — the new case extends this established error-string contract.

### Established Patterns
- Import preview runs off the UI thread in `_PreviewWorker` (QThread); errors come back
  via the `error: Signal(str)` → `_on_import_preview_error`. Any new discriminator for
  "show dialog vs toast" must survive that thread→signal hop (a plain `str(exc)` today).
- `_on_import_preview_error` currently truncates to 80 chars — the dialog branch must
  bypass that truncation.

### Integration Points
- Detection logic belongs in `settings_export.py` (OS-agnostic stdlib `zipfile`,
  testable without Qt); presentation belongs in `main_window.py` UI layer.

## RESEARCH NOTE (for gsd-phase-researcher / gsd-planner)
The UI shows errors as a toast keyed off `str(exc)`. To render a **modal dialog for
the text-mode case only** (D-02) while keeping the toast for other errors (D-03),
the error path needs a **discriminator** that survives `_PreviewWorker.error.emit(str(exc))`.
Options to evaluate: a dedicated exception subclass + re-classification in
`_on_import_preview_error`, a sentinel-prefixed error string, or a structured error
payload. Pick the lowest-friction option consistent with the existing typed-ValueError
contract. Also determine the byte-scan window length and the concrete 0x0D0A-injection
test that satisfies D-01's "strict, low-false-positive" intent (e.g., detecting CR
bytes that are not part of legitimate zip-structure bytes within the local file header).

</code_context>

<specifics>
## Specific Ideas

- The corruption signature to target: file begins with `PK` (`0x50 0x4B`) but raises
  `BadZipFile`, with injected `0x0D 0x0A` runs characteristic of an ASCII/text-mode
  transfer that rewrote lone `0x0A` bytes as `0x0D 0x0A`.
- Tone: actionable and plain — the approved wording uses "binary mode" deliberately
  (user accepted the term over a jargon-free rephrase).

</specifics>
