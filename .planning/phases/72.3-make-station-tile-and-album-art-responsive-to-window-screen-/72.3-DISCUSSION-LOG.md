## Sizing scheme

| Option | Description | Selected |
|--------|-------------|----------|
| Tiered breakpoints (Recommended) | E.g., right-pane <700px → 140×140; 700–1100px → 180×180 (current); ≥1100px → 240×240. Deterministic, predictable, easy to test. | ✓ |
| Continuous proportional | Size = clamp(pane_width × ratio, min, max). Smooth scaling on every resize. More fluid but harder to test, may flicker, costs pixmap re-renders on every resize tick. | |
| Fill-and-cap | Logo grows to fill available column up to a hard ceiling (e.g., 260px). Center column gets remaining room. | |

**User's choice:** Tiered breakpoints (Recommended).
**Notes:** Testability and predictability picked over fluidity. Three discrete tier states are easier to assert in pytest-qt and easier to reason about.

## Tier values (follow-up to Sizing scheme)

| Option | Description | Selected |
|--------|-------------|----------|
| 3 tiers: 140 / 180 / 240 (Recommended) | Small (pane <700) = 140×140; Medium (700–1100) = current values; Large (≥1100) = 240×240. Modest growth. | ✓ |
| 3 tiers: 128 / 192 / 280 | More aggressive: smaller floor, bigger ceiling. Bigger jumps. | |
| 4 tiers: 120 / 160 / 200 / 260 | Finer granularity at thresholds ~640/850/1100/1400. More states to test. | |

**User's choice:** 3 tiers: 140 / 180 / 240.
**Notes:** Middle tier matches the current logo size (180×180); the cover gets a +20px bump from 160 → 180 at the middle tier as the equal-size invariant kicks in.

## Trigger metric

| Option | Description | Selected |
|--------|-------------|----------|
| Now-playing pane width (Recommended) | Use the panel's own width() in a resizeEvent override. Matches Phase 72.1's pattern exactly. Tier changes when user drags splitter or resizes window. Works for compact and expanded modes. | ✓ |
| Main window width | Use MainWindow.width(). Subtler since left column sometimes hidden (compact mode) — same window width can mean very different right-pane widths. | |
| Screen size at launch (one-time) | Detect QScreen size once at startup and lock a tier. No resize responsiveness. | |

**User's choice:** Now-playing pane width.
**Notes:** Mirrors Phase 72.1's width-driven layout pattern (D-01, D-05 of 72.1-CONTEXT.md). Works correctly in both compact and expanded modes because the panel's own width reflects whatever space the splitter hands it.

## Claude's Discretion

- **Exact threshold widths between tiers** (~700px narrow→medium, ~1100px medium→large suggested; planner finalizes).
- **Hysteresis dead-band** around tier boundaries — recommendation: ship without, observe (same as Phase 72.1).
- **Helper-function shape** — recommended pattern: pure `_current_art_tier_size()` returning the int, plus `_apply_art_tier()` orchestrator.
- **`resizeEvent` integration with Phase 72.1** — share the same override if 72.1 shipped first.
- **Tier change detection** — only re-render pixmaps when the tier value actually changes from the cached previous tier.
- **Initial-tier bootstrap** — call `_apply_art_tier()` at end of `__init__` (or via `QTimer.singleShot(0, ...)`) so the first paint uses the correct tier.

## Deferred Ideas

- Responsive sidebar tile icons (`STATION_ICON_SIZE`) — sidebar stays at 32×32.
- Responsive Recently-Played icons — same constant; out of scope.
- Continuous proportional sizing — rejected this phase; could revisit if tiered feels too steppy.
- Fill-and-cap sizing — rejected this phase.
- Window-width or screen-size trigger — rejected this phase.
- Hysteresis dead-band on tier thresholds — ship without; observe; add only if flicker is real.
- Persisted "user prefers tier X" override — pane width is authoritative.
- Animation / fade on tier change — instant resize, matching 72/72.1 precedent.
- Per-display-geometry tier memory — declined as over-engineered.
- Auto-suggest toast on tier switch — declined by precedent.
- Symmetric scaling that preserves the existing 180/160 asymmetry — rejected; equal-size at every tier instead.
