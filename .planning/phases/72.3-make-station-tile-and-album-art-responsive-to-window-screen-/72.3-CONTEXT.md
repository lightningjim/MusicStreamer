# Phase 72.3: Make station tile and album art responsive to window/screen size - Context

**Gathered:** 2026-05-19
**Status:** Ready for planning

<domain>
## Phase Boundary

The now-playing pane's station logo (`logo_label`, currently `setFixedSize(180, 180)` at `now_playing_panel.py:339`) and album cover slot (`cover_label`, currently `setFixedSize(160, 160)` at `now_playing_panel.py:745`) are hard-coded pixel sizes. On a small-screen high-resolution display (Wayland DPR=1.0 deployment), both elements look disproportionately small relative to the surrounding pane and the rendering of the rest of the UI. Phase 72.3 replaces the fixed sizes with a three-tier responsive scheme driven by the now-playing panel's own width — both labels scale together based on a tiered breakpoint table.

**In scope:**

- Replace the `setFixedSize(180, 180)` and `setFixedSize(160, 160)` calls with a single tier-driven sizing helper that updates both labels in lockstep.
- Three tiers (D-04): **140×140 / 180×180 / 240×240**. The middle tier matches the existing logo dimensions; the cover gets a +20px bump (to 180) at the middle tier as part of the new equal-size invariant (D-05). Logo and cover always render at the same dimensions at every tier.
- Tier selection driven by `NowPlayingPanel.width()` evaluated in a `resizeEvent` override (D-06), mirroring the Phase 72.1 width-driven layout pattern (D-05 of `72.1-CONTEXT.md`). Threshold values between tiers (e.g., ~700px / ~1100px right-pane width) are planner-discretion (see Claude's Discretion).
- On a tier change, both `logo_label` and `cover_label` call `setFixedSize(N, N)` with the new tier value AND the bound pixmap is re-rendered through `_load_scaled_pixmap(path, QSize(N, N))` so the image fills the new box. The two existing call sites that re-render the logo/cover slot (`_show_station_logo`, line 1493; `_show_station_logo_in_cover_slot`, line 1497; the cover-art ready slot rescale at line 1468) are refactored to read the current tier size instead of hard-coded `QSize(180, 180)` / `QSize(160, 160)`.
- The existing aspect-ratio handling stays intact: `_load_scaled_pixmap` uses `Qt.KeepAspectRatio + Qt.SmoothTransformation` (line 213), so YouTube 16:9 logos still letterbox inside the (now variable) square box via Qt's CONTAIN-style fit. No aspect-ratio logic changes.
- Single source of truth: the tier value is computed centrally (e.g., a `_current_art_tier_size()` helper) and read everywhere logos/covers are rendered. Both labels and all rescale call sites read the same source.
- Tests at the pytest-qt level: simulate the panel at three representative widths (narrow → 140, default → 180, wide → 240) and assert `logo_label.size()`, `cover_label.size()`, AND the rendered pixmap dimensions all match the tier. Cover-art-ready replay test (re-emit cover_art_ready after a resize) confirms the rescale picks up the new tier.
- Behavior holds in both expanded mode and compact mode (the trigger is panel width, not mode — same invariant as Phase 72.1 D-01).
- Cooperates with Phase 72.1's stream-picker reflow: both behaviors share the same `resizeEvent` override on `NowPlayingPanel`. Planner decides whether to extend the existing `resizeEvent` (if 72.1 has shipped) or add it new, but only one override exists per class.

**Out of scope:**

- Sidebar station-tile icons (`STATION_ICON_SIZE = 32` in `_theme.py:52`, consumed by `station_list_panel.py:174,325` and `station_star_delegate.py:91`). User explicitly excluded from the "Scope of art" question — sidebar tiles stay at 32×32. (Despite the phase title saying "station tile", the user clarified the term refers to the now-playing **logo**, not the sidebar tree's row icons.)
- "Recently Played" QListView icons (`recent_view.setIconSize`, `station_list_panel.py:174`). Same 32px icon; out of scope.
- Continuous proportional sizing (e.g., `size = clamp(pane_width × 0.3, min, max)`). User picked tiered breakpoints over continuous on the "Sizing scheme" question — testability and predictability over fluidity.
- Fill-and-cap sizing (logo grows to fill its column up to a ceiling). User picked tiered over fill-and-cap on the same question.
- Window-width-driven or screen-size-driven trigger. User picked pane-width on the "Trigger" question; window-width was rejected because compact mode decouples it from the actual right-pane area, and screen-size-at-launch was rejected because it doesn't adapt to resize.
- Preserving the existing 180/160 logo-bigger-than-cover asymmetry. User picked equal-at-every-tier on the "Symmetry" question; the cover gets a +20px bump at the middle tier (160 → 180) as the deliberate change.
- Persisting the current tier across app restarts. No SQLite write; tier is recomputed every resize event. Aligns with Phase 72 D-09 (session-only) and Phase 72.1 (no persistence).
- Animation or transition on tier change. Instant resize — matches Phase 72.1 D-06 ("no animation on reparent") and Phase 72 D-Claude-Discretion ("opens INSTANTLY").
- Hysteresis dead-band on tier thresholds. Wayland coalesces resize events; ship without and observe (same deferral as Phase 72.1 D-Claude-Discretion).
- Per-display-geometry memory ("remember the user's small screen always uses tier 140"). The trigger is dynamic on every resize; no persistence.
- Changes to the `MainWindow` minimum width (`main_window.py:310` = 560px) or to `NowPlayingPanel.setMinimumWidth(560)` (`now_playing_panel.py:326`).
- Restructuring the now-playing outer three-column layout (`outer = QHBoxLayout`, lines 331–342, 742–747). The tier change only swaps `setFixedSize` values; the outer layout pattern (16/24px margins/spacing, logo-left + center-stretch + cover-right) is preserved.
- Changes to `cover_art.fetch_cover_art`, junk detection, or the iTunes/MusicBrainz upstream. The tier resize only affects how fetched pixmaps are scaled into the label, not how they're fetched.

<decisions>
## Implementation Decisions

### Scope of "art" — what scales

- **D-01:** **Only the now-playing logo (`logo_label`) and album cover (`cover_label`) are responsive.** The sidebar tile icons (`STATION_ICON_SIZE = 32`) and the Recently-Played QListView icons stay at their existing fixed 32×32. (User Q on Scope: only the two now-playing elements selected; sidebar and Recently-Played explicitly unchecked.) The phrase "station tile" in the phase title refers to the now-playing logo, not the sidebar tree's row tiles.
- **D-02:** **Single-tier source of truth.** Both `logo_label.setFixedSize(N, N)` and `cover_label.setFixedSize(N, N)` are called with the same `N` value derived from one central tier function. No independent sizing logic per label.
- **D-03:** **Three pixmap re-render call sites must read the central tier.** `_show_station_logo` (line 1493), `_show_station_logo_in_cover_slot` (line 1497), and the cover-art-ready rescale at line 1468 currently hard-code `QSize(180, 180)` and `QSize(160, 160)`. Each is refactored to fetch the current tier dimension and pass it to `_load_scaled_pixmap`. Otherwise the labels would resize but the bound pixmaps would still be at the old dimensions and Qt would either upscale (blurry) or letterbox (wasted space).

### Sizing scheme

- **D-04:** **Tiered breakpoints with three tiers: 140 / 180 / 240.** Small (narrow pane) = 140×140; Medium (default current size) = 180×180; Large (wide pane) = 240×240. (User Q on Sizing scheme: "Tiered breakpoints"; User Q on Tier values: "3 tiers: 140 / 180 / 240".) Deterministic, predictable, and easy to assert in tests; no continuous re-scaling on every resize tick.
- **D-05:** **Logo and cover are equal-sized at every tier.** (User Q on Symmetry: "Equal at every tier".) The cover gets a +20px bump at the medium tier (160 → 180) as the deliberate divergence from the current asymmetric 180/160. Simplifies the central tier function (one `N`, both labels), produces a visually balanced two-square framing, and is easier to test.

### Trigger metric

- **D-06:** **Pane width on `resizeEvent`.** Tier selection reads `NowPlayingPanel.width()` (or `self.width()` from inside the panel) in a `resizeEvent` override. (User Q on Trigger: "Now-playing pane width".) Mirrors Phase 72.1's width-driven layout pattern. Works correctly in both compact and expanded modes because the panel's own width reflects whatever space the splitter has handed it, regardless of whether the left column is hidden. Phase 72.1's stream-picker reflow uses the same hook — planner extends the existing `resizeEvent` override (if 72.1 has shipped before 72.3) or builds a new one shared by both behaviors.
- **D-07:** **Bound-method connect / override, no lambdas (QA-05).** The `resizeEvent` override calls a bound method on the panel for the tier recalculation; no lambda slots.

### Claude's Discretion

- **Exact threshold widths between tiers.** Phase 72.1 D-01 used the panel's own width directly; suggested thresholds for 72.3:
  - Narrow → Medium: ~700px right-pane width (between the 560px floor and the typical default).
  - Medium → Large: ~1100px right-pane width (room for the larger 240×240 squares without crowding the center column).
  Planner finalizes. Compute from widget minimums (preferred) or hand-tune from Phase 72 UAT-04's "560–700 px wide" overlap-range observation.
- **Hysteresis dead-band** around tier boundaries to suppress flicker if the user drags the splitter exactly at a threshold. Recommendation: ship without; observe; only add if flicker is real. Wayland's coalesced resize events tend to suppress oscillation. Same deferral as Phase 72.1 D-Claude-Discretion.
- **Helper-function shape.** Two viable patterns: (a) `_current_art_tier_size() -> int` on `NowPlayingPanel` that reads `self.width()` and returns 140/180/240; the `resizeEvent` override calls a separate `_apply_art_tier()` that fetches the size and updates labels + bound pixmaps. (b) A single `_reflow_art_tier()` that does both. Planner picks; (a) is more testable (the size function is pure) and is the recommendation.
- **Resize event integration with Phase 72.1.** If Phase 72.1 has shipped and added a `resizeEvent` override for the stream-picker reflow, 72.3 extends that same override with one extra call to the tier-reflow helper. If 72.1 has not shipped, planner picks an integration that minimizes future-72.1 churn (e.g., factor the resize-driven work into a small `_on_panel_resized()` dispatcher that both behaviors register with).
- **Tier change detection.** Recompute the tier on every resize event; only call `setFixedSize` + rescale pixmaps when the tier value actually changes from the cached previous tier. Saves pixmap re-renders during continuous drag within a single tier band.
- **Recently-played's relationship to scope.** Out of scope per D-01, but planner may note in PLAN.md that if the user later wants Recently-Played icons to track the same tier, the `STATION_ICON_SIZE` constant in `_theme.py` is the single point of change.
- **Test pattern for resize.** Use the same `qtbot.resize(panel, QSize(w, h))` / direct `widget.resize(w, h); qtbot.wait(0)` idiom as `tests/test_phase72_now_playing_panel.py` and (if shipped) `tests/test_phase72_1_stream_picker_reflow.py`.

</decisions>

<specifics>
## Specific Ideas

- **Symptom** — On a small-screen high-resolution display, the 180×180 logo and 160×160 cover look visually small relative to the rendering of the rest of the now-playing pane. The Wayland DPR=1.0 deployment means there's no automatic DPR upscaling to compensate; the fix is to size based on available pane width instead of hard-coded pixels.
- **Term clarification** — "Station tile" in the phase title refers to the now-playing **logo** (the 180×180 left-column image), NOT the sidebar tree's per-row 32px icons. The sidebar tiles stay at 32×32 (D-01).
- **Phase 72.1 alignment** — 72.3 reuses 72.1's exact pattern (width-driven, `resizeEvent`, no animation, no persistence). The two phases ideally share a single `resizeEvent` override on `NowPlayingPanel` with one dispatcher calling both helpers.
- **Cover gets a +20px bump at the middle tier** — the deliberate consequence of D-05 (equal sizes at every tier). At the medium tier, the cover goes from 160 → 180 to match the logo. This is the most visible behavior change in expanded mode at the default window size.
- **Tier change is rare in practice** — once the user picks a window size, tier rarely flips. The optimization to "only re-render when tier changes" (D-Claude-Discretion) is mostly about preventing thrash during continuous drag, not about runtime cost in steady state.
- **YouTube 16:9 logos still work** — `Qt.KeepAspectRatio` produces a letterboxed pixmap inside whatever square the tier provides. The CONTAIN behavior described in PROJECT.md is preserved exactly; no aspect-handling code changes.

</specifics>
