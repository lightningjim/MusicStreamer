## When the row splits

| Option | Description | Selected |
|--------|-------------|----------|
| Width-responsive (Recommended) | Row wraps to a second line whenever available width is insufficient, regardless of compact mode. Matches the roadmap's "on smaller screens" wording — wide windows stay one-line, narrow windows wrap. No hand-tuned thresholds; layout decides. | ✓ |
| Always when multi-stream visible | Whenever stream_combo is visible (a multi-stream station is selected), force the two-row layout regardless of window width. Simpler/predictable: the picker always gets its own row when present. Trade-off: takes extra vertical space on wide windows too. | |
| Only in compact mode + multi-stream | Two-row layout only when compact mode is ON AND stream_combo is visible. Narrowest fix — targets exactly the bug scenario, leaves all other states untouched. Trade-off: if user resizes window narrow WITHOUT entering compact mode (e.g. peek-overlay disabled), the bug is unfixed. | |

**User's choice:** Width-responsive
**Notes:** Captured as D-01. Trigger is panel width, not compact-mode coupling. Single-stream stations stay one-row at all widths (D-02), since `_populate_stream_picker` already hides the combo when `len(streams) ≤ 1`.

## What migrates to row 2

| Option | Description | Selected |
|--------|-------------|----------|
| Stream picker alone on row 2 (Recommended) | Row 1 stays: play, stop, edit, star, eq, volume, compact-toggle. The stream picker is split out into a dedicated row 2 container, shown only when stream_combo is visible AND there's not enough width. Cleanest visual hierarchy — transport controls always stay together; the picker reads as a station-attribute selector rather than mixed in with transport. | ✓ |
| Picker + volume slider on row 2 | The two widest non-transport widgets move down together. Row 1 = play, stop, edit, star, eq, compact-toggle. Row 2 = stream picker + volume. Frees more width on row 1; visually pairs the "continuous" controls (picker dropdown + slider). | |
| Flat FlowLayout — Qt decides | All widgets sit in a single FlowLayout; Qt wraps whichever don't fit (typically the rightmost ones — compact-toggle, volume, eq). No pre-planning of what goes where. Simplest code; risk is unpredictable visual grouping (compact-toggle could end up on row 2 looking detached). | |

**User's choice:** Stream picker alone on row 2
**Notes:** Captured as D-03 (picker is the only widget that moves) and D-04 (no flat FlowLayout — preserves "transport always together" invariant).

## Layout mechanism

| Option | Description | Selected |
|--------|-------------|----------|
| Reparent picker between row 1 / row 2 on resize (Recommended) | Keep stream_combo in row 1 by default. On QResizeEvent, compute whether row 1 has room for the picker at its 140px min; if not (and multi-stream visible), reparent it to a row 2 QHBoxLayout under controls. Two explicit layouts — row 1 stays clean, row 2 is hidden unless populated. Deterministic test surface: width thresholds become assertable. Trade-off: ~20 lines of resize-handler logic. | ✓ |
| Always-have row 2; pop picker in/out | A QHBoxLayout for row 2 is constructed once and parented under controls. stream_combo lives in row 1 OR row 2 depending on a single boolean state derived from width. Same end result as option 1 but the row 2 container is constructed eagerly (visible only when picker is in it). Slightly more code but more inspectable in the widget tree. | |
| FlowLayout for entire controls row | Replace controls QHBoxLayout with FlowLayout (already exists in-repo). All buttons wrap naturally; we'd order widgets so the picker is rightmost, making it the first to wrap. Less code; depends on FlowLayout's wrap order being stable across Qt versions. Doesn't strictly guarantee "picker alone" — if the row is even narrower, more widgets wrap with it. | |

**User's choice:** Reparent picker between row 1 / row 2 on resize
**Notes:** Captured as D-05. Instant reparent (D-06) — no animation, matches Phase 72's peek-overlay precedent.

## Stream-picker width on its own line

| Option | Description | Selected |
|--------|-------------|----------|
| Stretch to fill row 2 (Recommended) | Picker fills the full width of the now-playing pane minus margins. Easiest to read at a glance — stream labels (e.g. "320 kbps MP3 — EU Mirror") get full breathing room. Equivalent to setting size policy to Expanding when in row 2, then restoring to Preferred when back in row 1. | ✓ |
| Keep min 140, no growth | Picker keeps its 140px minimum, sits left-aligned on row 2 with empty space to its right (or center-aligned via addStretch). Consistent with row-1 sizing; visually shows the picker is the same widget that was in row 1. Trade-off: row 2 looks sparse on wide windows. | |
| Stretch in row 2, fixed in row 1 | Picker uses QSizePolicy.Expanding when reparented to row 2 (fills width), and reverts to Preferred when reparented back to row 1 (sits at min 140). Single widget, two policies depending on parent row. Adds 2-3 lines to the reparent logic but matches both contexts naturally. | |

**User's choice:** Stretch to fill row 2
**Notes:** Captured as D-07 / D-08. Picker stretches when alone on row 2, reverts to its existing 140px-min Preferred sizing when back in row 1.
