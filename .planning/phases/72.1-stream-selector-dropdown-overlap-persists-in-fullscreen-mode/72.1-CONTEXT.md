# Phase 72.1: Stream-selector dropdown overlap persists in fullscreen mode for multi-stream stations - Context

**Gathered:** 2026-05-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 72 hid the left column to free width for the now-playing pane's bottom-bar controls, fixing the overlap at narrow widths (560–700px) for single-stream stations. For **multi-stream stations** the bug remains: `stream_combo` (`QComboBox`, `setMinimumWidth(140)`, visible only when a station has >1 streams) is appended to the same controls row, pushing the total minimum widget width above the right pane's design floor. Phase 72.1 splits the controls row into two lines on smaller screens so the picker can move to its own row when the single-row layout would overflow.

**Measured overflow source** (`now_playing_panel.py:440–545`, post-Phase 72 layout):
Control row order = `play(36) → stop(36) → edit(36) → stream_combo(min 140, multi-stream only) → star(28) → eq(28) → volume(120 fixed) → compact_toggle(28) → addStretch(1)`
Total fixed-widget widths = 508px + 7×8px spacing (56px) = **~564px row minimum**, before any container margins or splitter handle width. At the 560px MainWindow minimum (`main_window.py:310`) the row exceeds the available right-pane width, producing visible overlap. Single-stream stations have stream_combo hidden, removing 140+8 = 148px and the row fits.

**In scope:**
- Width-responsive reflow of the now-playing control row: when the row's required minimum width exceeds the available width AND `stream_combo` is visible, reparent the picker from the existing row 1 (`controls` QHBoxLayout) into a new row 2 QHBoxLayout that sits directly underneath row 1 inside the same `center` QVBoxLayout. When width allows, reparent back to row 1.
- Row 1 always contains the same widget set whether the picker is present or not: play, stop, edit, star, eq, volume, compact_toggle, addStretch. The picker is the ONLY widget that moves between rows.
- When the picker is on row 2 it stretches to fill the full row width (minus margins). When it's back in row 1 it returns to its existing 140px-min Preferred sizing.
- Single-stream stations stay one-row at all widths (no behavior change — `stream_combo.setVisible(False)` already keeps it out of the layout via Qt's `QHBoxLayout` zero-size handling for hidden widgets).
- The reparent decision is driven from the panel's resize event (QResizeEvent on `NowPlayingPanel`, or `eventFilter` on the panel's parent — planner picks the cleanest hook). The decision input is the panel's current `width()` versus a computed row-1-with-picker minimum; planner determines the exact threshold expression.
- Tests at the pytest-qt level: simulate the panel at three widths (wide → 1-row, narrow → 2-row, threshold ± hysteresis), assert `stream_combo.parent()` / layout membership, assert no other widgets moved, assert visibility unchanged when stream_combo itself is hidden (single-stream station).
- Behavior holds in BOTH expanded mode and compact mode. The trigger is width, not mode.

**Out of scope:**
- Changes to `stream_combo` content/population logic (`_populate_stream_picker`, `_sync_stream_picker`, `_on_stream_selected`). Existing dropdown behavior is preserved as-is.
- Restructuring the rest of the control row. Star, eq, volume, compact-toggle keep their current positions in row 1.
- Generalizing to a multi-widget responsive layout. ONLY the stream picker reparents. Future overflow polish belongs in its own phase.
- Animation or transition for the reparent. Reparent is instant (matches Phase 72's chosen "no animation" for the peek overlay).
- New widget-level affordances on the picker (e.g., search-as-you-type, grouped quality presets). Phase 72.1 is layout-only.
- Per-display-geometry memory ("remember that user's small screen always shows two rows"). The trigger is dynamic on every resize; no persistence.
- A hysteresis layer to suppress flicker if the user drags the splitter exactly at the threshold. Planner-discretion (D-06 below) — instrumentation first, hysteresis only if observed flicker is real.
- Changes to the volume slider (still 120px fixed width). Volume is NEVER what wraps; if width is below row 1's minimum even without the picker, that's a separate phase.
- Changes to MainWindow minimum width (`main_window.py:310` = 560px). Phase 72.1 makes the existing floor honored for multi-stream too; it does not lower the floor further.

</domain>

<decisions>
## Implementation Decisions

### Wrap trigger (when row splits)

- **D-01:** **Width-responsive trigger, not mode-coupled.** The reparent decision is driven by the panel's available width versus the computed row-1-with-picker minimum width. Compact mode is NOT an input. Rationale: the roadmap explicitly says "on smaller screens"; coupling to compact mode misses the case where a user resizes the window narrow without entering compact mode. Wide windows stay one-row; narrow windows wrap, regardless of compact-mode state. (User Q on Trigger: "Width-responsive".)
- **D-02:** **Single-stream stations stay one-row at all widths.** When `stream_combo` is hidden (single-stream station, the existing `len(streams) > 1` gate at `now_playing_panel.py:1135`), the row 2 container stays empty/hidden — no reparent logic runs. Single-stream behavior is unchanged from Phase 72.

### What wraps to row 2

- **D-03:** **ONLY the stream picker moves to row 2.** Row 1 always contains play, stop, edit, star, eq, volume, compact-toggle, addStretch — that set is invariant. The picker is the only widget that moves between rows. Rationale: transport controls (play/stop) read as a unit; star/eq/volume/compact form the secondary cluster; the picker is the station-attribute selector and reads naturally as a separate concern. Keeping the picker as the sole wrap target gives a deterministic visual hierarchy that's easy to test. (User Q on Row 2: "Stream picker alone on row 2".)
- **D-04:** **No flat FlowLayout for the entire row.** A FlowLayout would let any rightmost widget wrap when narrowed further, breaking the "transport always together" invariant from D-03. The reparent approach gives an explicit two-state layout that's easier to reason about and test. (User Q on Mechanism: rejected "Flat FlowLayout".)

### Layout mechanism

- **D-05:** **Reparent picker between row 1 / row 2 on QResizeEvent.** Construct both `controls` (row 1 QHBoxLayout) and a new `controls_row2` (QHBoxLayout) eagerly inside `center` (the NowPlayingPanel center QVBoxLayout, parent of the existing controls layout). By default, stream_combo lives in row 1 (preserving Phase 72's layout exactly when single-stream). On a panel resize event, compute `should_wrap = (stream_combo.isVisible() AND panel_width < row1_with_picker_min)`. If state changes, remove the picker from its current layout and add it to the other layout. Row 2 container is hidden when empty; shown when populated. (User Q on Mechanism: "Reparent picker between row 1 / row 2 on resize".)
- **D-06:** **No animation on reparent.** Reparent is instant — matches Phase 72's chosen "no animation" for the peek overlay (Phase 72 D-Claude-Discretion / UAT-02 "peek opens INSTANTLY"). Consistent UX precedent within the now-playing pane.

### Stream-picker sizing on its own row

- **D-07:** **Picker stretches to fill row 2 width.** When reparented into row 2, the picker uses `QSizePolicy.Expanding` horizontally so the dropdown grows to the panel's full width minus margins. Improves readability of stream labels (e.g. "320 kbps MP3 — EU Mirror") when given room. (User Q on PickerWidth: "Stretch to fill row 2".)
- **D-08:** **Picker reverts to its existing Preferred sizing when back in row 1.** The 140px `setMinimumWidth(140)` from `now_playing_panel.py:485` is preserved as the floor; the size policy is restored to Preferred (Qt's default for QComboBox) on reparent back. The picker reads identically to its Phase 72 row-1 appearance when wide. Implementation: planner picks between toggling `setSizePolicy(...)` on reparent OR using a per-parent sizing wrapper; either is acceptable.

### Claude's Discretion

- **Exact width threshold for the reparent decision.** Candidates: (a) `stream_combo.minimumSizeHint().width() + sum_of_other_widget_minimums + spacing + margins` computed once and cached; (b) a hand-tuned constant (e.g., 700px) chosen from the Phase 72 UAT-04 observation that "560–700 px wide" was the overlap range; (c) `controls.sizeHint().width() > panel.width()`. Recommendation: compute from widget minimums (option a) for self-correcting behavior if other widget sizes ever change. Planner finalizes.
- **Hysteresis to avoid flicker at the threshold boundary.** When the user drags the splitter exactly at the threshold width, the picker could oscillate between rows on each resize tick. Recommendation: do not pre-emptively add hysteresis. Ship without; observe in real use (Wayland GNOME Shell resize events tend to be coalesced, so flicker may not manifest). If flicker is observed, add a small dead-band (e.g., +20px on the "shrink → row 2" trigger vs. the "grow → row 1" trigger) in a follow-up polish.
- **Where exactly the new row 2 QHBoxLayout sits inside `center`.** Most-natural insertion is `center.addLayout(controls_row2)` immediately after the existing `center.addLayout(controls)` at `now_playing_panel.py:548`. Planner confirms there's no widget between line 548 and the stats widget (`_stats_widget`, line 553) that should sit above the row 2.
- **Resize-event hook choice.** Options: (a) override `NowPlayingPanel.resizeEvent`, (b) install an `eventFilter` on the panel watching for QEvent.Resize, (c) subscribe to the splitter's `splitterMoved` signal in addition to plain resize. Recommendation: override `resizeEvent` (simplest, panel is the natural owner of its own layout decisions). Splitter signal is unnecessary because resizing the splitter triggers a resize event on the child panel.
- **Row 2 vertical spacing.** Recommend: match the existing `controls.setSpacing(8)` for visual consistency; no additional `setContentsMargins` on row 2 (inherit `center`'s margins). Planner verifies visual result.
- **Test pattern for QResizeEvent.** Use pytest-qt `qtbot.resize(panel, QSize(w, h))` or direct `panel.resize(w, h); qtbot.wait(0)` — pick whichever the existing Phase 72 tests use (see `tests/test_phase72_now_playing_panel.py`).
- **Handling the multi-stream → single-stream transition mid-narrow.** When a station change causes `stream_combo.setVisible(False)` while the picker was on row 2, the picker becomes invisible but its layout membership doesn't auto-clean. Recommendation: in `_populate_stream_picker` (which already handles visibility at line 1135), when setting visible=False, also call the reparent helper to move the picker back to row 1 (its default home) and hide row 2's container. Keeps state consistent.

</decisions>

## Existing Code Insights

### Reusable Assets

- **`controls` QHBoxLayout** (`now_playing_panel.py:440`) — existing row 1 container; Phase 72.1 keeps it intact and adds a sibling `controls_row2` QHBoxLayout immediately below in the same `center` QVBoxLayout.
- **`stream_combo`** (`now_playing_panel.py:484`) — the only widget that reparents. Its signal connection (`currentIndexChanged.connect(self._on_stream_selected)`), its data model (populated by `_populate_stream_picker`), and its sync logic (`_sync_stream_picker`) are unaffected by reparenting in Qt — `QLayout.addWidget()` reparents the widget to the layout's parent without disturbing signal/slot wiring.
- **`_populate_stream_picker` / `_sync_stream_picker`** (`now_playing_panel.py:1114, 1149`) — existing show/hide and selection-sync logic. ### Established Patterns

- **Bound-method slot connects (QA-05)** — apply to any new resize-handler / reparent slot wiring.
- **No `lambda` in connect calls** — same.
- **Single source of truth for state-derived visibility** (Phase 47.1 WR-02, Phase 67 M-02) — `_populate_stream_picker` owns `stream_combo.setVisible(...)`; the reparent helper only owns `stream_combo`'s layout parent. Two distinct concerns, two distinct owners.
- **No SQLite persistence for layout state** (Phase 72 D-09) — row 1/2 state is recomputed on every resize event; no `Repo.set_setting` call.
- **Two-layout swap pattern** — not previously used in this codebase. Phase 72.1 establishes it. Future phases that need conditional layout reshuffling (e.g., a future "compact icy row" or "stacked badge row") can adopt the same pattern.
- **Reparenting widgets across QHBoxLayouts via `removeWidget()` + `addWidget()`** — standard Qt idiom. Widget's signal connections, model data, and state survive the reparent automatically. The only thing that changes is the widget's parent layout (and thus its geometry on the next layout pass).

<specifics>
## Specific Ideas

- **Bug repro recipe** — Open MusicStreamer; select a station with multiple streams (an AudioAddict station or any radio-browser station that has multiple stream URLs in its `streams` table); drag the splitter or resize the window to ~600–700px right-pane width. Observe `stream_combo` overlapping star/eq/volume/compact-toggle on its right.
- **Roadmap wording is the design brief** — "consider splitting into two lines on smaller screens" — width-responsive, two-line layout. Phase 72.1 implements exactly this.
- **Why not FlowLayout for the whole row** — FlowLayout's left-to-right wrap order would let star/eq/volume/compact-toggle wrap independently as width shrinks further, breaking the "transport always together" mental model. The reparent approach gives a deterministic two-state outcome that's easy to test.
- **Why width, not compact-mode** — Compact mode addresses the left-column-claiming-space problem. The multi-stream overflow problem is purely a row-content-width problem. Coupling them would miss the case where a user (a) doesn't enter compact mode but (b) narrows the window — the bug would still reproduce. Width-trigger is the correct invariant.
- **Picker stretches when alone** — A 140px combo on a 500px row 2 looks unintentionally short. Stretching to fill makes the longer stream labels readable and signals "this is the focused control on this row" visually. Cheap and obvious.
- **No animation** — Matches Phase 72's peek overlay choice ("opens INSTANTLY", UAT-02). Instant reparent reads as a layout adjustment, not a transition; transitions invite questions ("what just happened?") that instant doesn't.

</specifics>

<deferred>
## Deferred Ideas

- **Hysteresis dead-band on the wrap threshold** — to suppress potential flicker when the user drags the splitter exactly at the threshold width. Recommendation in this phase: ship without; observe; only add if flicker is real. Could become a 5-line follow-up polish.
- **Generalized responsive layout for the entire control row** — multi-widget overflow handling (e.g., volume slider also wraps if the row is below row-1-minus-picker minimum). Out of scope here; Phase 72.1 is explicitly picker-only. If the user ever drives the window below ~420px (row 1's minimum without the picker), a follow-up phase would handle it.
- **Animation / fade on reparent** — out of scope (D-06). Could become a micro-polish if the instant reparent feels jarring; would mirror the "slide animation for peek overlay" deferral from Phase 72.
- **Picker search-as-you-type / quality-grouped dropdown** — orthogonal to layout; belongs in a "stream picker UX" phase if ever desired.
- **Showing the active stream's quality in the row-1 collapsed state** — e.g., a small "320kbps" badge next to the compact-toggle when the picker is on row 2 (so the user always sees what's playing without scrolling to row 2). Not requested; layout-pure phase. Future polish if the row 2 dropdown feels too far from the user's eye.
- **Mid-resize coalescing** — debouncing the reparent decision across rapid resize events. Wayland tends to coalesce these already; revisit only if observed unnecessary churn.
- **Persisted "user prefers two rows" toggle** — explicit setting to force two-row layout regardless of width. Not requested; width is the authoritative trigger.
- **Auto-suggest toast** when the row first wraps ("Stream picker moved to second row to fit screen"). Declined by precedent — Phase 72 deferred the same noisy-toast pattern for compact-mode discoverability.
