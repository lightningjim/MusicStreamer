# Phase 72.1: Code Review Report

**Reviewed:** 2026-05-16T23:55:51Z
**Depth:** standard
**Files Reviewed:** 2
**Status:** issues_found

## Summary

Phase 72.1 adds a width-responsive reparent of `stream_combo` between row 1 (`self.controls`) and a new row 2 (`self._controls_row2`) inside `NowPlayingPanel`. The mechanism (resizeEvent override + four helpers + lazy threshold cache) is structurally sound: super().resizeEvent() is chained correctly (Pitfall 9), the helpers are idempotent (Pattern 2), QObject parent is preserved during reparent (Pattern 2 — signal/model survival is real), insertWidget with a pinned `_picker_row1_index` preserves row-1 ordering (Pitfall 6), and the documented `isHidden()` shift (Plan 02 Rule 1 deviation) is the correct intent-vs-actual visibility choice.

However, one Critical bug exists in the threshold-cache invariant: the cache can be recomputed while the picker is OUT of `self.controls`, producing a stale-low threshold that defeats the wrap predicate for the duration of the bound station. Several Warning-tier concerns surround a parallel `isVisible()` vs `isHidden()` inconsistency in `_populate_stream_picker`, a pre-existing second writer of `stream_combo.setVisible(False)` in `_on_stop_clicked` that now leaves layout state inconsistent until the next resize, an assertion helper that misses negative-membership checks, and a "compact-mode independence" test that passes by accident at the chosen widths. Two Info-tier code smells (dead local, redundant `outer_widget_count`; gap-tracking comment that does not match the code) round out the list.

The Critical finding is not exercised by any of the 10 new tests — multi-stream→multi-stream rebinds while narrow are missing from the matrix.

## Critical Issues

### CR-01: Threshold cache recomputed while picker is OUT of row 1 produces stale-low value

**File:** `musicstreamer/ui_qt/now_playing_panel.py:1144-1180, 1214-1281`
**Issue:**
`_picker_row_min_width()` walks `self.controls` and sums each contained widget's minimum width — including `stream_combo` (140px floor) and its inter-widget spacing (8px). The cache is invalidated only by `_populate_stream_picker` (line 1170, `self._cached_row1_min = None`).

When the user is in narrow mode with a multi-stream station, the picker has been reparented OUT of `self.controls` and INTO `self._controls_row2`. If `_populate_stream_picker` then fires — which it does on **every** `bind_station` call (line 809), e.g. station-list double-click on another multi-stream station, or a `_sync_now_playing_station` after edit (`main_window.py:1312`) — the cache is invalidated. On the **next** `resizeEvent`, `_picker_row_min_width()` walks `self.controls` while the picker is still in row 2. The sum drops by ~148px (140 picker minimum + 8 spacing). The cached threshold is now too small for the actual row-1-with-picker requirement.

Concrete consequence: at intermediate widths (e.g. ~820–960px depending on real font metrics), the predicate `panel_width < threshold` returns False even though row 1 cannot fit the picker. `_reflow_stream_picker_row` moves the picker back to row 1, where it produces the exact overlap that Phase 72.1 is supposed to fix. Reverse direction: if the user then narrows again, the now-stale-but-correctly-low cache may keep the picker on row 1 instead of moving it to row 2.

The multi→single transition (`test_multi_to_single_mid_narrow_rehomes_picker`) accidentally hides the bug for that path because the `if not isVisible(): _move_stream_picker_to(self.controls)` block at lines 1177–1179 rehomes the picker BEFORE the cache is consulted on the next resize — so by the time recompute happens, the picker is back in row 1. The multi→multi case has no such pre-rehome and is untested.

**Fix:** Recompute the threshold against a layout state that always contains the picker, regardless of current layout membership. Two reasonable options:

```python
# Option A: temporarily re-home before compute (matches the cache's intent
# of "row 1 WITH picker minimum")
def _picker_row_min_width(self) -> int:
    if self._cached_row1_min is not None:
        return self._cached_row1_min
    # ... existing margins/spacing init ...
    layout = self.controls
    # Sum row-1 widgets PLUS picker's contribution unconditionally —
    # the picker is conceptually a row-1 member for sizing purposes.
    picker_in_row1 = layout.indexOf(self.stream_combo) >= 0
    widget_count = 0
    total = margins.left() + margins.right()
    for i in range(layout.count()):
        w = layout.itemAt(i).widget()
        if w is None:
            continue
        total += max(w.minimumWidth(), w.minimumSizeHint().width())
        widget_count += 1
    if not picker_in_row1:
        # Picker currently in row 2 — add its row-1 contribution explicitly.
        total += max(
            self.stream_combo.minimumWidth(),
            self.stream_combo.minimumSizeHint().width(),
        )
        widget_count += 1
    if widget_count > 1:
        total += spacing * (widget_count - 1)
    # ... outer-chrome walk unchanged ...
```

```python
# Option B: invalidate cache inside _move_stream_picker_to AND ensure
# every recompute path is gated on "picker currently in row 1." Simpler
# but requires careful test coverage of every recompute trigger.
```

Add a regression test that bind_station-rebinds a different multi-stream station while in narrow mode and asserts the picker stays on row 2 across a subsequent resize-no-op (or a width that previously triggered wrap continues to trigger it).

## Warnings

### WR-01: `_populate_stream_picker` uses `isVisible()` while reflow uses `isHidden()` — asymmetric semantics

**File:** `musicstreamer/ui_qt/now_playing_panel.py:1177`
**Issue:**
`_reflow_stream_picker_row` deliberately uses `self.stream_combo.isHidden()` (line 1365) with a documented rationale: `isVisible()` returns False when an ancestor has not been shown, which fires false positives in pre-show test contexts. The fix is the Plan 02 Rule 1 deviation — accepted.

But the rehome block at line 1177 in `_populate_stream_picker` reads `if not self.stream_combo.isVisible():` — the very pattern Plan 02 rejected. Behaviour: in any context where `_populate_stream_picker` runs before the panel ancestor is shown (e.g. constructor flow if bind_station were ever called pre-show; or a peek-overlay reparent that briefly leaves the panel detached), `not isVisible()` evaluates True even for a multi-stream station, and the rehome runs unnecessarily. The rehome is idempotent (`_move_stream_picker_to` early-returns when already in row 1), so this is not currently a behavior bug — but it is a maintenance hazard: a future change that makes the rehome non-idempotent (e.g. emitting a signal, logging, or modifying state) would surface the asymmetry immediately. It also contradicts the documented Plan 02 lesson, making the codebase inconsistent.

**Fix:**
```python
if self.stream_combo.isHidden():
    self._move_stream_picker_to(self.controls)
    self._set_picker_size_policy(expanding=False)
```
And update the surrounding comment (lines 1171–1176) to call out the same isHidden() rationale as line 1355.

### WR-02: `_on_stop_clicked` is a second writer of `stream_combo.setVisible(False)` — layout state can drift

**File:** `musicstreamer/ui_qt/now_playing_panel.py:988-1003` (existing) — interaction with Phase 72.1 lines 1330–1377
**Issue:**
CONTEXT.md / Pattern E explicitly identifies `_populate_stream_picker` as the **single writer** of `stream_combo.setVisible(...)`. Phase 72.1's reflow predicate is built on that invariant (the `isHidden()` short-circuit at line 1365 trusts visibility to encode multi-vs-single-stream-ness).

But `_on_stop_clicked` (line 994) directly calls `self.stream_combo.setVisible(False)` — a pre-existing violation that becomes load-bearing under Phase 72.1. Scenario: user has a multi-stream station playing in narrow mode → picker on row 2. User clicks stop → line 994 hides the picker → picker is invisible but **still in row 2 layout slot**. No resize follows, so `_reflow_stream_picker_row` does not run. Until the next resize, layout state (picker in row 2) disagrees with visibility state (hidden). Visually fine on Qt (Qt collapses zero-effective-content rows), but it leaves a hidden child in row 2's QHBoxLayout.

On the next user resize, `isHidden()` is True → picker is rehomed to row 1 → state reconciles. So the user-visible bug is small. But this is the exact "layout state diverges from visibility state" failure mode that the multi→single rehome hook at lines 1177–1179 was designed to prevent — and Phase 72.1 did not extend the same hook into `_on_stop_clicked`. Either fix the second-writer violation or accept it explicitly in CONTEXT.

**Fix:** Add the same rehome call at the end of `_on_stop_clicked`, or (preferable) trigger the reflow:
```python
def _on_stop_clicked(self) -> None:
    # ... existing body ...
    self.stream_combo.setVisible(False)
    # Phase 72.1: keep layout state consistent with the just-flipped
    # visibility state — same rationale as _populate_stream_picker's
    # multi→single rehome hook at lines 1177-1179.
    self._move_stream_picker_to(self.controls)
    self._set_picker_size_policy(expanding=False)
    # ... rest of body ...
```

### WR-03: `_assert_stream_combo_in_layout` has no negative-membership check

**File:** `tests/test_phase72_1_stream_picker_reflow.py:236-252`
**Issue:**
The helper asserts `expected_layout.indexOf(stream_combo) >= 0`. It does not assert that the **other** layout returns `-1`. If a regression caused `_move_stream_picker_to` to add the picker to the target without removing it from the source (e.g., omitted `removeWidget` branch), the picker would be in BOTH layouts and every test would still pass. Qt's behavior for a widget being a member of two QHBoxLayouts simultaneously is undefined and produces the exact "overlap" bug Phase 72.1 fixes — so the test matrix should detect it.

**Fix:**
```python
def _assert_stream_combo_in_layout(panel, expected_layout, msg=""):
    other_layout = (
        panel._controls_row2
        if expected_layout is panel.controls
        else panel.controls
    )
    assert expected_layout.indexOf(panel.stream_combo) >= 0, (
        f"{msg}stream_combo not in expected layout"
    )
    assert other_layout.indexOf(panel.stream_combo) == -1, (
        f"{msg}stream_combo unexpectedly present in {other_layout!r} too "
        f"(double-membership bug)"
    )
    assert panel.stream_combo.parent() is panel
```

### WR-04: `test_compact_mode_and_picker_wrap_independent` passes by accident — the test does not exercise the independence claim

**File:** `tests/test_phase72_1_stream_picker_reflow.py:403-470`
**Issue:**
The test resizes the window to 560×800 and toggles compact ON/OFF, asserting the picker stays in row 2 across all three states. But at 560px window width, even compact-ON (station_panel hidden, splitter gives all space to now_playing) leaves the panel at ~550px — still well below the ~944px threshold this code computes. The test could not produce any other outcome at this width, so it does not actually verify the D-01 "width-only, not mode-coupled" claim.

A genuine independence test needs a window width chosen such that:
- Compact OFF: panel.width() < threshold (picker on row 2)
- Compact ON: panel.width() > threshold (picker would move to row 1 if mode were an input — but should NOT, per D-01)

This requires the window to be just wide enough that compact-mode-induced panel expansion crosses the threshold. With station_panel ≈ 360px and the rough ~944px threshold, the right window width is around 1000–1100px so that:
- compact OFF: panel ≈ 640px (< 944, row 2)
- compact ON: panel ≈ 1050px (> 944, row 1 IF mode were coupled; row 1 is also correct here under width-only, so the assertion needs inversion)

A cleaner version: pick two widths that straddle the threshold and verify compact-toggle never changes the row decision when width is held constant. The current test gives false confidence.

**Fix:** Re-derive the test widths from `panel._picker_row_min_width()` at runtime (computed via a one-shot call after `bind_station`) and pick widths that produce the desired threshold-relative panel widths in both compact states. Or split into two tests: one at 560px (current — verifies "narrow stays narrow"), one at a width chosen so compact-ON would push panel above threshold, verifying the picker still stays on row 1 in both compact states (consistent with width-only trigger when panel is wide). The current test only verifies one quadrant.

## Info

### IN-01: `outer_widget_count` is dead code in `_picker_row_min_width`

**File:** `musicstreamer/ui_qt/now_playing_panel.py:1264-1272`
**Issue:**
```python
outer_widget_count = 0
for i in range(outer.count()):
    item = outer.itemAt(i)
    w = item.widget()
    if w is not None:
        wmin = max(w.minimumWidth(), w.minimumSizeHint().width())
        total += wmin
        outer_widget_count += 1
```
`outer_widget_count` is assigned but never read. Likely a copy-paste from the row-1 loop above (line 1243-1252) where `widget_count` IS used to compute spacing. The subsequent spacing math at line 1276-1277 uses `outer.count() - 1` instead — which is also slightly wrong since it counts gaps between ALL items (widgets + layouts) rather than between widgets only. For the current 3-item outer layout (logo widget, center layout, cover widget), `outer.count() - 1 == 2`, which happens to match the correct widget-gap count of 2 by coincidence; a future change to add another layout sibling would break this silently.

**Fix:**
```python
total += outer.spacing() * max(0, outer.count() - 1)
# Note: gap count = number of items - 1, which includes layouts because
# Qt allocates a gap between every adjacent item regardless of type.
```
Either remove `outer_widget_count` or use it for the spacing math; the dead-variable form is the worst of both choices.

### IN-02: Comment claims to count widget gaps but code counts all-item gaps

**File:** `musicstreamer/ui_qt/now_playing_panel.py:1273-1277`
**Issue:**
The comment at lines 1273-1275 reads "Outer spacing between every item (including center layout) is outer.count() - 1 gaps. center is one of the items so the gap count is correct." This is correct ONLY if Qt inserts a spacing gap between every adjacent item regardless of whether the item is a widget or a layout. That is in fact Qt's behavior (`QBoxLayout` adds spacing between every adjacent item), so the math is right — but the comment buries the load-bearing assumption. Combined with IN-01's dead `outer_widget_count`, this section reads as if the author was uncertain which counting scheme applies. Tighten the comment or assert the assumption with a constant.

**Fix:** Replace the comment with a one-liner: `# Qt's QBoxLayout inserts spacing between every adjacent item (widget or sub-layout), so gap count = outer.count() - 1.`

### IN-03: `test_no_lambda_in_reflow_wiring` does not include all reflow helpers

**File:** `tests/test_phase72_1_stream_picker_reflow.py:570-580`
**Issue:**
The QA-05 source-grep lint scans lines containing `_reflow_stream_picker_row`, `_move_stream_picker_to`, or `resizeEvent`. It does NOT scan `_set_picker_size_policy` or `_picker_row_min_width`. While those two helpers do not currently use any `.connect()` wiring (they are pure-compute / setter helpers), a future regression that adds e.g. `setSizePolicy_changed.connect(lambda: ...)` to `_set_picker_size_policy` would slip past this test.

**Fix:** Add the two missing helper names to the `targets` tuple, or replace with a broader pattern that catches any line containing "stream_combo" AND "connect(" with a lambda. The current scope is narrower than the QA-05 invariant warrants.
