**Findings:**

**Verdict:** **Hypothesis 1 REFUTED.** The server emits the Artists/Albums panels for multi-word queries with matches. The existing `_ArtistAlbumParser.handle_data` discriminator (`lower.startswith("artists")` / `lower.startswith("albums")`) recognizes both leading text markers in this fixture, and the existing `<p class="artists">` gate at `handle_starttag` line 522 matches both blocks (the second block is also `<p class="artists">`, despite containing albums — the discriminator is text-based, not class-based).

**Implication:** No parser change needed for this fixture. `test_search_returns_artist_links_multiword` was GREEN on first run in Plan 01 and remains GREEN in Plan 02. The user's reported Issue A ("multi-word click only surfaces song results") may have been live-production-specific (e.g., a different multi-word query that DOES trigger one of the other hypotheses, or a path-of-execution bug in the dialog tier rather than the parser tier). **Plan 04 UAT must re-confirm Issue A reproduces against live production with multi-word queries; if it does not, the issue is closed by this fixture-confirmed behavior.**

## Decisions Made

- **Hypothesis 1 REFUTED via fixture inspection** — captured `search_multiword_p1.html` contains valid Artists/Albums blocks. No `_ArtistAlbumParser` patch was applied. The Plan 01 RED test that "passed unexpectedly" is the canonical truth for this fixture; Plan 04 UAT re-confirms against live production.
- **Per-row artist semantics differ between artist-page and album-page** — artist page emits the page-title artist ("Testament") for every song row (catalog of one artist's discography); album page emits per-row artist from the `<a href="/artist/Y">` anchor in td 0 (compilations / various-artists albums have different per-row artists). Both flow into the same `{songid, artist, title, duration, add_url}` dict shape so Plan 03's dialog `_render_results()` does not need surface-aware branching.
- **D-02 belt-and-suspenders explicit reject-list over implicit** — `if "artist" in cls or "songs" in cls: return` is explicit and grep-able. Dropping this in favor of an implicit "first matching table wins" gate would silently fail when a future markup change adds another `<table width="620">` inside `<div class="playlist">`. The regression test `test_album_parser_rejects_artist_page_table` is the permanent guard.
- **Position-based duration capture in `_AlbumPageParser`** — `self._capture = "duration"` is set on `<td>` start when `self._tds_in_row == 5`, then cleared on the matching `</td>`. This mirrors `_SongRowParser`'s established pattern and avoids endtag-time complexity.
- **No `<div>` depth tracking in `_AlbumPageParser`** — per RESEARCH §Pitfall 2, the `_table_locked` one-shot covers the case (subsequent `<table>` tags inside `<div class="playlist">` are ignored). Adding depth tracking would only complicate the state machine without changing the outcome.
