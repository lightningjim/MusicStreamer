## Issues Encountered

- **SC1 FAIL — query-specific Issue A reproduces for `bad religion`.** This is the genuine defect that motivated Phase 60.1 in the first place. Plan 02's fixture-first triage of `foo+fighters` showed the parser handles multi-word fine — and indeed, `foo fighters` and `pearl jam` are confirmed working in UAT. But `bad religion` returns no artist/album panels and only song results in the live app. **This is query-specific, not a blanket multi-word issue.** Phase 60.1.1 must capture `tests/fixtures/gbs/search_bad_religion_p1.html` and triage the live HTML to determine whether (a) the server actually omits the panels for this query (server-side oddity — perhaps too many song matches? a special-cased substring like "bad"?), or (b) the existing `_ArtistAlbumParser` has an edge-case bug that the foo+fighters fixture didn't surface.
- **NEW UX DEFECT — artist drill-down flat rendering.** Even when an artist drill succeeds, the rendered table loses the album grouping. `_ArtistPageParser` is the cause: it deliberately skips `<tr class="albumTitle">` separator rows and emits flat song dicts without an `album` field, per Plan 02's design. The dialog renders that flat list with no Album column and no section headers. Tester reports this as "undifferentiated tables of songs". This was NOT in the Phase 60.1 contract — Plan 02 explicitly chose flat output — so it's a separate UX gap, not a Phase 60.1 contract violation. Phase 60.1.1 should add an `album` field per row (carry the most-recent albumTitle text as parser state) and either render an Album column in the artist drill view or insert visual album section dividers.
