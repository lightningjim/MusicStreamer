# Phase 88.3: Bundle QtWebEngine in Frozen Windows Build — Research

**Researched:** 2026-06-10
**Domain:** PyInstaller + PySide6 QtWebEngine — Windows conda-forge build env + frozen bundle
**Confidence:** HIGH

---

## Summary

Phase 88 UAT G6 confirmed a two-layer failure: (1) the `musicstreamer-build` conda env does NOT have
PySide6 QtWebEngineWidgets installed, and (2) even if it were installed, `MusicStreamer.spec` has zero
WebEngine collection directives, so PyInstaller's modulegraph would not have collected the ~130 MB
runtime (QtWebEngineProcess.exe, resources/*.pak, locales, ICU data). The oauth helper subprocess
immediately crashes with exit=2 when the frozen bundle lacks either dependency.

The fix has four concrete parts that must all land together:

1. **Env provisioning**: `pip install PySide6-Addons==<pinned-pyside6-version>` in the
   `musicstreamer-build` env adds QtWebEngineWidgets/Core. This CANNOT be `conda install qt6-webengine`
   because conda-forge's pyside6 package excludes WebEngine (it is not in the conda-forge pyside6
   distribution; only the PyPI PySide6-Addons wheel ships it for pip-based installs). However there is
   a critical ABI conflict risk when mixing pip-installed wheels into a conda-forge Qt env: the ICU
   shared library that conda-forge's GStreamer pulls is different from pip's PySide6 wheel's ICU. The
   existing README explicitly warns against this (Phase 43.1 Pitfall #1). Research identifies a path
   that avoids this conflict (see Architecture Patterns § Env provisioning strategy).

2. **MusicStreamer.spec**: Add explicit WebEngine bundling — `hiddenimports` for
   `PySide6.QtWebEngineWidgets` and `PySide6.QtWebEngineCore`, PLUS explicit collection of the WebEngine
   runtime files that the PyInstaller hook may not auto-collect when the import is wrapped in
   `try/except`. Mirror the existing 88.1 winrt `collect_all` + merge pattern.

3. **Deepen step-4d build guard**: The current `--self-test` short-circuits before oauth_helper imports
   — add a new step-4e `--check-webengine` invocation of the frozen exe that actually imports
   `PySide6.QtWebEngineWidgets` (like the 88.1 `--check-mediakeys` pattern).

4. **Documentation**: Document QtWebEngine as a build-env precondition in `packaging/windows/README.md`
   and add a preflight check in `build.ps1`.

**Primary recommendation:** Follow the fix_shape in 88-UAT.md G6 verbatim, implementing all four
parts in a single wave to avoid partial-fix states that pass the shallow guard but fail at runtime.

### Pattern 1: try/except import + modulegraph blind spot — explicit hiddenimport required

**What:** When an import is wrapped in `try/except ImportError`, PyInstaller's static analysis
(modulegraph) does not reliably follow the import edge. The module exists in the build env but is
never collected because the graph never visits it. This is exactly the oauth_helper.py:62-71 situation.

**When to use:** Add explicit `hiddenimports` + `collect_all` output for every module behind
`try/except ImportError` that must be present in the frozen bundle.

**Example (from existing spec pattern for winrt):**
```python
# Source: packaging/windows/MusicStreamer.spec (88.1 pattern to mirror)
_wec_datas, _wec_binaries, _wec_hiddenimports = collect_all("PySide6.QtWebEngineCore")
# NOTE: collect_all argument is the Python PACKAGE path, not a PyPI dist name.
# For WebEngine the package IS the module (unlike winrt-runtime vs winrt/).
```
Then in Analysis:
```python
binaries=[...] + _wec_binaries,
datas=[...] + _wec_datas,
hiddenimports=[...
    "PySide6.QtWebEngineWidgets",
    "PySide6.QtWebEngineCore",
    "PySide6.QtWebEngineQuick",   # sometimes needed by Core
] + _wec_hiddenimports,
```

**Rationale for collect_all vs hiddenimports-only:**
`collect_all("PySide6.QtWebEngineCore")` captures the PyInstaller hook's binaries output (which
calls `get_qt_webengine_binaries_and_data_files()` — the function that collects QtWebEngineProcess.exe,
pak files, locales directory). [CITED: github.com/pyinstaller/pyinstaller/pull/6412]

**Important caveat:** The argument to `collect_all` here is the Python package name
`"PySide6.QtWebEngineCore"` (dot-separated), not a PyPI distribution name like `"winrt-runtime"`.
The 88.1 winrt pattern used PyPI dist names because winrt is a namespace package; PySide6 modules
use the Python import path directly. [ASSUMED — need to verify collect_all accepts package path vs dist name]

**Alternative — use the PyInstaller hook directly via hiddenimport trigger:**
The simpler approach that is more in line with how the hooks-contrib hook is designed to fire:
```python
hiddenimports=[
    "PySide6.QtWebEngineWidgets",   # triggers hook-PySide6.QtWebEngineWidgets
    "PySide6.QtWebEngineCore",      # triggers hook-PySide6.QtWebEngineCore
],
```
When PyInstaller sees these hiddenimports, it fires the `hook-PySide6.QtWebEngineCore.py` hook from
hooks-contrib, which calls `get_qt_webengine_binaries_and_data_files()` and collects everything
needed. This is the minimal-change approach that mirrors how `PySide6.QtNetwork` and `PySide6.QtSvg`
are already declared in the spec. [ASSUMED — need to verify hook fires from hiddenimports alone]

### Pattern 2: --check-webengine frozen-exe guard (mirror of --check-mediakeys from Phase 88.1)

**What:** A new `--check-webengine` flag in `__main__.py` that imports `QWebEngineView` inside the
frozen runtime, exits 0 on success and exits 13 on ImportError. Build.ps1 step-4e runs it after
PyInstaller completes.

**When to use:** Immediately after step-4d (existing oauth-helper --self-test guard).

**Example:**
```python
# musicstreamer/__main__.py — mirrors _run_check_mediakeys()
def _run_check_webengine() -> int:
    """Phase 88.3: frozen-bundle guard — confirms QWebEngineView importable.

    Exits 0 if PySide6.QtWebEngineWidgets is available in the frozen bundle.
    Exits 13 if not (build.ps1 step-4e will catch this and fail the build).
    Called from build.ps1: MusicStreamer.exe --check-webengine
    """
    try:
        from PySide6.QtWebEngineWidgets import QWebEngineView  # noqa: F401
        print("WEBENGINE_GUARD OK")
        return 0
    except ImportError as exc:
        print(f"WEBENGINE_GUARD FAIL: {exc}", file=sys.stderr)
        return 13
```

```powershell
# build.ps1 step-4e (after step-4d OAUTH HELPER GUARD)
if (-not $SkipWebEngineGuard) {
    Write-Host "=== WEBENGINE GUARD: assert QWebEngineView importable in frozen bundle (Phase 88.3) ==="
    Invoke-Native {
        & "..\..\dist\MusicStreamer\MusicStreamer.exe" --check-webengine 2>&1 | Out-Host
    }
    if ($LASTEXITCODE -ne 0) {
        Write-Host "BUILD_FAIL reason=webengine_not_bundled hint='QWebEngineView not importable in frozen bundle; check PySide6-Addons install and MusicStreamer.spec WebEngine collection'" -ForegroundColor Red
        exit 13
    }
    Write-Host "WEBENGINE GUARD OK"
} else {
    Write-Host "WEBENGINE GUARD skipped (-SkipWebEngineGuard)"
}
```

### Pattern 3: Env provisioning with version pinning (critical — ICU ABI constraint)

**What:** The `musicstreamer-build` conda env has `pyside6` from conda-forge. This uses conda-forge's
ICU library. If you do `pip install PySide6-Addons` WITHOUT a version pin, pip will upgrade PySide6
to the latest (currently 6.11.1) which is built against a different ICU ABI, breaking the Qt6Core.dll
load. The fix is to pin PySide6-Addons to EXACTLY the conda-forge pyside6 version already installed.

**Steps:**
```powershell
# On the Windows build VM, with musicstreamer-build activated:
$pyside_ver = python -c "from PySide6 import __version__; print(__version__)"
pip install "PySide6-Addons==$pyside_ver"
```
This installs only the Addons wheel for the same version — it does NOT upgrade PySide6-Essentials
or the core PySide6 package (which would trigger the ICU conflict).

**Verification:**
```powershell
python -c "from PySide6.QtWebEngineWidgets import QWebEngineView; print('WebEngine OK')"
```

**Risk if ICU ABI mismatch:** Qt6Core.dll crash at import time, same symptoms as Phase 43.1 Pitfall #1.
If the pip Addons version doesn't match exactly, Qt DLL load will fail. [CITED: packaging/windows/README.md Pitfall #1 + pyproject.toml comment]

### Anti-Patterns to Avoid

- **`--self-test` as the only WebEngine guard**: The existing step-4d guard deliberately returns 0
  BEFORE importing oauth_helper, so it cannot detect a missing WebEngine. Never rely on it for
  WebEngine presence verification.

- **`conda install qt6-webengine`**: This installs the Qt C++ WebEngine library but NOT the
  Python bindings (PySide6.QtWebEngineWidgets). Python code will still fail with ImportError.
  [ASSUMED — conda-forge pyside6 appears to exclude WebEngine bindings]

- **`pip install PySide6` (without version pin)**: Upgrades PySide6-Essentials to latest, triggering
  ICU ABI mismatch with conda-forge GStreamer. Documented breaking change from Phase 43.1 Pitfall #1.

- **Using `--single-process` Chromium flag with frozen WebEngine**: Known incompatibility with VM
  / software rendering environments. The existing `oauth_helper.py` sets
  `QTWEBENGINE_CHROMIUM_FLAGS` with `--user-agent` — do NOT add `--single-process`. [CITED: github.com/pyinstaller/pyinstaller/issues/7665]

- **`collect_all("PySide6")` for the whole binding**: Collects the entire ~600 MB PySide6 distribution.
  Targeted collection is sufficient and keeps bundle size manageable. [ASSUMED]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Collecting QtWebEngineProcess.exe + pak files | Manual Tree() blocks for each resource file | `hiddenimport = ["PySide6.QtWebEngineCore"]` (triggers hook-PySide6.QtWebEngineCore which calls `get_qt_webengine_binaries_and_data_files()`) | The hooks-contrib function already handles the QtWebEngineProcess helper, resources/*.pak, locales, and qt.conf generation for the subprocess [CITED: github.com/pyinstaller/pyinstaller/blob/.../PyInstaller/utils/hooks/qt.py] |
| Checking WebEngine availability | Custom PyInstaller analysis script | `--check-webengine` flag pattern (mirror of --check-mediakeys from Phase 88.1) | Frozen-exe self-test is the proven pattern in this codebase |
| ICU library management | conda/pip gymnastics | `pip install "PySide6-Addons==<exact-pinned-version>"` | Version pin ensures the Addons wheel's ICU matches conda-forge's already-installed ICU |

**Key insight:** The hooks-contrib `hook-PySide6.QtWebEngineCore.py` already handles the complex
WebEngine runtime collection (QtWebEngineProcess helper, pak files, locales, qt.conf for subprocess).
The only reason it didn't fire is that the import was in a `try/except` block, hiding it from
PyInstaller's modulegraph. An explicit `hiddenimports` entry is the minimal, correct fix.

## Common Pitfalls

### Pitfall 1: try/except ImportError hides the import from modulegraph

**What goes wrong:** oauth_helper.py:62 wraps `from PySide6.QtWebEngineWidgets import QWebEngineView`
in a `try/except ImportError`. PyInstaller's modulegraph performs static analysis; when the import
is in a `try/except`, the analysis may not follow the import edge or may consider it optional and
skip collection.

**Why it happens:** PyInstaller uses modulegraph to trace import dependencies. Imports inside
`try/except` blocks are the documented primary cause of "hidden imports" that require explicit
`--hidden-import` or `hiddenimports=` entries in the spec file.

**How to avoid:** Add `"PySide6.QtWebEngineWidgets"` and `"PySide6.QtWebEngineCore"` to the
`hiddenimports` list in `Analysis()`. This explicitly tells PyInstaller to collect these modules
regardless of static analysis findings.

**Warning signs:** Exit code 2 from the oauth helper subprocess (ImportError triggers `sys.exit(2)`
in oauth_helper.py:71).

### Pitfall 2: ICU ABI mismatch on PySide6-Addons pip install without version pin

**What goes wrong:** `pip install PySide6-Addons` (without `==<version>`) installs the latest
(6.11.x as of 2026-06-10). The pip PySide6 wheel family uses a different ICU build than the
conda-forge GStreamer package's ICU. Qt6Core.dll fails to load at runtime.

**Why it happens:** conda-forge's GStreamer bumps ICU versions on its own schedule. pip's PySide6
wheels are built by the Qt Company against the ICU bundled in the Qt offline installers. Phase 43.1
Pitfall #1 documented this exact failure mode. [CITED: packaging/windows/README.md, pyproject.toml comment]

**How to avoid:** `pip install "PySide6-Addons==$(python -c "from PySide6 import __version__; print(__version__)")"` — pin to the exact conda-forge pyside6 version already in the env.

**Warning signs:** `DLL load failed while importing QtCore` or `ImportError: DLL load failed` at any
PySide6 import. If PySide6.QtCore imports fine but QtWebEngineCore fails, the issue is different
(runtime collection, not ICU).

### Pitfall 3: conda-forge pyside6 package does NOT include QtWebEngineWidgets

**What goes wrong:** `conda install pyside6` (from conda-forge) installs the PySide6 bindings but
WITHOUT QtWebEngineWidgets. Running `python -c "from PySide6.QtWebEngineWidgets import QWebEngineView"`
raises ModuleNotFoundError in the conda env, which is exactly what the VM diagnostic confirmed.

**Why it happens:** The conda-forge pyside6 package (as of 2026-06-10) does not bundle
QtWebEngineWidgets. This is a known limitation documented in Qt Forum issues. [CITED: forum.qt.io/topic/146325]

**How to avoid:** After `conda install pyside6`, additionally run
`pip install "PySide6-Addons==<pinned-version>"`. This is a pip-into-conda install, which carries
the ICU risk above — hence the exact version pin.

**Warning signs:** `ModuleNotFoundError: No module named 'PySide6.QtWebEngineWidgets'` in the
build env itself (catch this in the build.ps1 preflight, before PyInstaller runs).

### Pitfall 4: step-4d --self-test short-circuit masks missing WebEngine

**What goes wrong:** The existing step-4d guard (`MusicStreamer.exe --oauth-helper --self-test`)
returns exit 0 immediately at `__main__.py:65` BEFORE importing oauth_helper. So even if WebEngine
is completely absent from the bundle, this guard passes.

**Why it happens:** The `--self-test` branch was designed to verify only that the `--oauth-helper`
dispatch path is reachable — not that the actual login window can open. It is documented as a
"headless smoke guard."

**How to avoid:** Add a separate step-4e `--check-webengine` guard that actually imports
`QWebEngineView` in the frozen exe. Keep step-4d (which guards a different failure mode — missing
dispatch path).

**Warning signs:** Build passes all guards; user gets "Login helper crashed unexpectedly" exit=2.

### Pitfall 5: QtWebEngineProcess.exe location in the bundle

**What goes wrong:** PyInstaller's WebEngine hook collects `QtWebEngineProcess.exe` into the
top-level PySide6 package directory (to match the PyPI wheel layout). If the hook doesn't fire
(because the import was hidden from modulegraph), `QtWebEngineProcess.exe` is absent and WebEngine
silently fails to render pages.

**Why it happens:** WebEngine uses a separate subprocess for the Chromium renderer. Qt6 locates
this subprocess via the `browser-subprocess-path` at runtime. In a frozen bundle, the hook generates
a `qt.conf` file so the subprocess finds its resources. Without the hook firing, neither the exe
nor qt.conf is present. [CITED: PyInstaller changelog notes on QtWebEngine/qt.conf generation]

**How to avoid:** After the fix, verify `dist\MusicStreamer\_internal\PySide6\QtWebEngineProcess.exe`
exists in the bundle (add this as a post-bundle assertion in build.ps1 or the test suite).

**Warning signs:** WebEngine window opens but renders a blank page; no Chromium subprocess visible
in Task Manager when the login window is open.

### Pitfall 6: QTWEBENGINE_DISABLE_SANDBOX is not needed on Windows (it IS on Linux Flatpak)

**What goes wrong:** Developers familiar with the Flatpak path (Phase 86, PKG-LIN-FP-05) may add
`QTWEBENGINE_DISABLE_SANDBOX=1` to the Windows env configuration.

**Why it happens:** The Flatpak manifest requires this env var because of sandboxing constraints.
Windows frozen apps do NOT need it (Chromium sandbox works in frozen PyInstaller bundles on Windows).

**How to avoid:** Do not add `QTWEBENGINE_DISABLE_SANDBOX` to `runtime_hook.py` for Windows.

## Open Questions (RESOLVED)

> All three open questions were resolved during planning (Phase 88.3 plans 88.3-01 / 88.3-02). Each carries a functional answer wired into a plan task; the VM-bound assumptions are guarded by explicit Test-Path / import-verify gates so a wrong assumption fails the build, not the user.

1. **collect_all vs hiddenimports-only for WebEngine collection**
   - What we know: The winrt fix used `collect_all("winrt-runtime")` where the argument is the PyPI distribution name. PySide6 modules are part of the `PySide6` PyPI distribution, not separate distributions — so `collect_all("PySide6.QtWebEngineCore")` may fail or behave unexpectedly since that is not a PyPI dist name.
   - What's unclear: Whether `collect_all` on a Python package path (not PyPI dist) works. The PyInstaller docs note `collect_all` is designed for PyPI distribution names.
   - Recommendation: Prefer the simpler approach — add `"PySide6.QtWebEngineWidgets"` and `"PySide6.QtWebEngineCore"` to `hiddenimports=` in the spec. This is the standard way to trigger existing hooks and is exactly how `PySide6.QtNetwork` and `PySide6.QtSvg` are handled in the current spec. Verify on the VM that the hook fires and QtWebEngineProcess.exe appears in the bundle.
   - **RESOLVED:** Locked to hiddenimports-only. Plan 88.3-01 Task 1 adds the two hiddenimports and its acceptance criteria forbid `collect_all` for WebEngine. Plan 88.3-02 Task 1 step 4 asserts `_internal/PySide6/QtWebEngineProcess.exe` exists after the build; if the hook does not fire from hiddenimports alone, that assertion fails and Plan 01's spec is revised to add an explicit `Tree()` collection (documented fallback).

2. **Does PySide6-Addons version 6.10.2 match exactly what conda-forge ships?**
   - What we know: `pip index versions PySide6-Addons` shows `6.10.1, 6.10.2, 6.10.3, 6.11.0, 6.11.1`. The Linux dev env has PySide6 6.10.2 (confirmed: `python3 -c "from PySide6 import __version__; print(__version__)"`). The Windows VM's conda-forge pyside6 version is what matters, and it may differ from the Linux dev env.
   - What's unclear: The exact pyside6 version currently in `musicstreamer-build` on the Windows VM. The diagnostic result from 88-UAT.md did not record the version number.
   - Recommendation: Task 1 should dynamically read the installed version and pin Addons to it.
   - **RESOLVED:** Handled dynamically. Plan 88.3-02 Task 1 step 1 reads the installed pyside6 version on the VM (`python -c "from PySide6 import __version__; print(__version__)"`) and pins `pip install "PySide6-Addons==<that version>"` — no hard-coded version.

3. **ICU conflict risk when installing Addons-only pip wheel**
   - What we know: Phase 43.1 Pitfall #1 documented that pip's PySide6 wheel family has a different ICU ABI than conda-forge's GStreamer. The fix was to use conda-forge pyside6.
   - What's unclear: Whether installing ONLY PySide6-Addons (not upgrading PySide6-Essentials) avoids the ICU conflict. The Addons wheel imports QtWebEngineCore which has its own DLLs that may load a bundled ICU.
   - Recommendation: Prioritize testing `pip install "PySide6-Addons==<same-version>"` on the VM first. If it causes ICU crashes at import time, escalate to a conda-level fix. The strict version pin is the primary risk mitigation.
   - **RESOLVED:** Mitigated by the strict version pin (OQ2) plus an import-verification gate. Plan 88.3-02 Task 1 step 2 imports `PySide6.QtWebEngineWidgets` before building; an ICU/DLL load failure stops the task with an escalation note to a conda-level fix (`conda install qt6-webengine`). The README precondition (Plan 88.3-01 Task 2) records the ICU hazard and the pin requirement.
