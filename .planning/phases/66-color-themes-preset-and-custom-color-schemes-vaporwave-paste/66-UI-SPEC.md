# Phase 66 — UI Design Contract

> Visual and interaction contract for the new ThemePickerDialog (`musicstreamer/ui_qt/theme_picker_dialog.py`) and ThemeEditorDialog (`musicstreamer/ui_qt/theme_editor_dialog.py`). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Stack note:** This is a PySide6 6.11 / Qt 6 desktop app on Linux Wayland (GNOME Shell, DPR=1.0) with Windows as a secondary target. The shadcn / 8-point-grid / 60-30-10 palette framing in the template is adapted to a Qt-native context. Most pixel-level visual properties are determined by the *active theme's* QPalette (the very thing this phase introduces); we explicitly call out OWN vs INHERIT below.
>
> **Phase 59 idiom inheritance:** Both dialogs in this phase follow the modal-QDialog pattern established by `accent_color_dialog.py` (Phase 59). Spacing, typography, snapshot/restore semantics, button-row shape, bound-method connects, modality and Esc/X behavior are inherited verbatim. Where this phase deviates from Phase 59, the deviation is called out explicitly.

## Spacing Scale

Multiples of 4 — applies to wrapper layouts and tile sizing. Both dialogs share the Phase 59 8/8/8/8 contentsMargins + 8px spacing token.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inner padding inside a `_ThemeTile` between the 4-color stripe and the name label; vertical padding inside a `_ColorRow` between the role label and the swatch row |
| sm | 8px | Wrapper `QVBoxLayout` content margins on all four edges (both dialogs); spacing between sibling widgets in wrapper `QVBoxLayout` (matches Phase 59 `accent_color_dialog.py:91-93`); horizontal grid gap between tiles in the picker `QGridLayout` |
| md | 16px | Vertical spacing between the tile grid and the bottom button row in the picker (visual separation of selection area vs commit area); top padding above the "Customize…" button if rendered as a separate left-aligned button row |
| lg | 24px | Editor dialog: vertical spacing between groups of color rows if the planner chooses to visually group "background roles" (Window / Base / AlternateBase) vs "foreground roles" (WindowText / Text / Link) vs "control roles" (Button / ButtonText / HighlightedText). Optional grouping; planner may flatten into a single uniform-spacing column. |

Exceptions: **none**. No icon-only touch targets; no fractional pixels.

**Locked tile dimensions (`_ThemeTile`):**

| Property | Value | Rationale |
|----------|-------|-----------|
| Tile total size (fixed) | **120 × 100 px** | RESEARCH Q8 recommendation. Fits 4-across × 2 rows = 480px grid width + 8px sm gaps + 8px sm margins ⇒ ≥ 504px dialog width — comfortably within Linux Wayland DPR=1.0 deployment target and below typical 1366×768 screens. |
| 4-color stripe (drawn inside tile) | **96 × 16 px** centered horizontally, 8px from top edge | Wide enough to show 4 visually distinct 24×16 swatches at 1x DPR |
| Name label area | top-edge of label is 8px below stripe; label is centered in remaining vertical space (~60px high for the label cell — Qt default label sizing) | Single-line text; no truncation expected for any preset name (longest is "GBS.FM After Dark" — fits at 12pt body) |
| Active checkmark overlay | **16 × 16 px**, anchored top-right corner with 4px (xs) padding from each edge | `QStyle.StandardPixmap.SP_DialogApplyButton` rendered at tile size |
| Active border | **3px** solid using `palette.highlight()` (the *current* theme's Highlight, post-accent-override if any) | Q14 a11y — colorblind-friendly redundant cue alongside checkmark |
| Inactive border | 1px solid using `palette.mid()` | Subtle delineation against dialog background |
| Disabled (empty Custom) border | 1px **dashed** using `palette.mid()` | Non-clickable cue; reinforces the hint label |
| Disabled (empty Custom) opacity | tile rendered at **0.5 effective opacity** via Qt's `setEnabled(False)` (Qt's stock disabled-palette tinting) | Q14 a11y — still legible enough to read the hint |

1. **Reading order matches palette mental model.** Background → foreground pairs (Window/WindowText, Base/Text, Button/ButtonText) read top-to-bottom; a 2-column grid would scatter the pair across columns and break the mental model.
2. **9 rows at ~28px each = ~252px content height + button row + margins ≈ 320px total height.** Fits comfortably; no need to economize via a grid.
3. **Matches Phase 59 dialog visual density** — single vertical column of fields + button row at bottom.

## Color

This phase introduces a *family* of palettes (the 6 presets + Custom slot), not a single palette. The dialogs themselves render in whichever theme is currently active (live preview during Cancel-restorable interaction). The 60/30/10 framing below describes the dialogs' OWN visual hierarchy *as inherited from whatever theme is active at any moment*, not new tokens.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `QPalette.Window` (active theme) | Wrapper `QDialog` background; tile widget background under the stripe; editor background |
| Secondary (30%) | `QPalette.Button` (active theme) | `QDialogButtonBox` button surfaces; editor row swatch button background **fallback** (when swatch's QSS-painted background is not yet set) |
| Accent (10%) | `QPalette.Highlight` (active theme, post-accent-override) | Active-tile 3px border; focus rectangle on focused tile or button; selected text in any QLineEdit-like widgets inside the inner QColorDialog |
| Destructive | `_theme.ERROR_COLOR_HEX = "#c0392b"` | NOT USED in this phase — kept here for the reserved-list audit. Reset is a recoverable, dialog-stay-open action; it is NOT styled with the destructive color. |

**Accent reserved for (this phase only — system-wide consumers documented in Phase 59 spec):**

- Active-tile 3px border in the picker grid (drawn using `palette.highlight().color()` in the tile's paintEvent — re-resolves automatically when palette changes during live preview)
- The default-button focus state on `Apply` (picker) and `Save` (editor) — Qt-default `setDefault(True)` rendering uses the active palette's Highlight for the button border
- The 4th swatch in every preset's tile mini-stripe (Window / Base / Text / **Highlight-baseline-or-ACCENT_COLOR_DEFAULT**) — see Tile Stripe Composition below

**Destructive actions in this phase: NONE.**

- Reset (editor) is a recoverable action: it reverts the editor's 9 rows to the source preset hex values and re-applies them to the running palette; the editor stays open. NOT styled destructive.
- Cancel (picker or editor) is non-destructive: snapshot is restored.
- Saving overwrites the user's previous Custom palette. **No destructive confirmation is shown** — the action is non-catastrophic (user can re-edit any time) and the running app's accent_color setting (which is the only data-loss-bearing setting that the editor *could* have touched) is intentionally NOT in the editor's surface. See "Save+apply+switch flow" below for the concrete UX.

**Color-flash guard (RESEARCH Pitfall 3 — same as Phase 59 D-17):**

- When the editor opens with `source_preset='custom'` AND `theme_custom` JSON is corrupt or empty, the editor MUST fall back to `THEME_PRESETS[<active-theme-or-system>]` for prefill (NOT to `QColor()` defaults which would produce black flashes).
- When `apply_theme_palette` reads `theme_custom`, every hex passes through `_is_valid_hex` before reaching `QColor` (RESEARCH Q12, Pitfall 3).

**Tile Stripe Composition (4 swatches, left → right, each 24×16 px):**

| Slot | Source | Notes |
|------|--------|-------|
| 1 | preset's `Window` | The dominant background of the theme |
| 2 | preset's `Base` | The content/list background |
| 3 | preset's `Text` | Foreground text color (gives a contrast hint vs Base) |
| 4 | preset's `Highlight` baseline if non-empty, else `ACCENT_COLOR_DEFAULT` (`#3584e4`) | For Dark/Light presets and System default tile, this is `ACCENT_COLOR_DEFAULT` (D-07). For System default tile, the stripe shows ACCENT_COLOR_DEFAULT for the Highlight slot AND uses Qt-default Window/Base/Text colors for slots 1-3 (read from a freshly-constructed `QPalette()` at tile-render time). |

The Custom tile, when `theme_custom` is populated, uses the saved JSON's role values for slots 1-3 + `ACCENT_COLOR_DEFAULT` for slot 4 (Highlight is not in the editor surface). When empty, the Custom tile renders **no stripe** — the tile body shows only the italic hint label "Click Customize…".

## Interaction & Feedback Contract

Owned by Phase 66; load-bearing UX shape.

### Modality & lifecycle

- Both ThemePickerDialog and ThemeEditorDialog are **modal**: `setModal(True)` (D-20). User must Apply / Save / Cancel before resuming main window interaction.
- **Window-manager close (X button, Alt+F4, super-q on some Wayland WMs)** routes through `QDialog.reject()` — same code path as Cancel. Snapshot is restored.
- **Esc key** routes through `reject()` (Qt default for dialogs with a RejectRole button) — restores snapshot.
- **Enter / Return** routes through the default button:
  - Picker: Apply (`_apply_btn.setDefault(True)`)
  - Editor: Save (`_save_btn.setDefault(True)`)

### Modal stacking (picker + editor)

The editor opens *on top of* the picker. While the editor is modal:

- The picker is **disabled at the WM level** (Qt parent-modal behavior — clicks on the picker beep on Wayland or are visually disabled; the picker cannot be interacted with directly).
- The picker remains **visible** as the parent surface — the user can see the tile grid behind the editor's modal dialog, which gives spatial context (matches Phase 59-style nested-dialog idiom from `gbs_search_dialog.py` precedent).
- Esc / X on the editor dismisses ONLY the editor and returns the user to the still-open picker (back-button behavior is "cancel inner-most modal first"). This matches Qt's default cascading modality.
- Esc / X on the picker (after the editor was dismissed) dismisses the picker and snapshot-restores. **CANNOT happen while the editor is open** — the modality layer prevents it.

**Important (RESEARCH Pitfall 1 — Stale snapshot when editor saves while picker is still open):** when the editor's Save fires, the picker's snapshot becomes "stale" because the running palette is no longer what was active when the picker opened. **Solution (locked):** the editor's `_on_save` callback BOTH (a) persists `theme = 'custom'` + `theme_custom` JSON to repo, AND (b) accepts the editor (`self.accept()`), AND (c) sets a flag on the parent picker (`self.parent()._save_committed = True`) which the picker's `reject()` method consults — if `_save_committed` is True, the picker skips the snapshot-restore on Cancel. This means: after the user clicks Save in the editor, even if they then click Cancel on the picker, the saved Custom palette stays applied. This is the correct behavior — "save means save". See **State Machine** section below for the full state graph.

### Save+apply+switch perceptual flow (LOCKED)

The user perceives the following sequence when they click Save in the editor:

1. Editor closes (instantaneous — no animation we own)
2. Picker is now visible, with Custom tile shown as the active selection (3px border + checkmark + bold label) — the picker MUST refresh its active-tile rendering when control returns from the editor (planner: the picker's `_open_editor` slot calls `self._refresh_active_tile()` after `editor.exec()` returns regardless of accepted/rejected)
3. Main window (visible behind both dialogs as a third surface) has been showing the new palette live since the user's last color-row edit in the editor; nothing changes in the main window at the moment of Save
4. User can now click Apply on the picker (which is a no-op for `theme = 'custom'` because `theme` was already persisted by the editor — but Apply still commits the picker's own choice of Custom-as-active and accepts the picker, closing it) OR Cancel on the picker (which is also a no-op for the saved Custom palette per the `_save_committed` flag, and Cancel still closes the picker) OR pick a different tile (which discards Custom-as-active for the next picker session — `theme_custom` JSON is preserved).

**The one perceptually subtle thing:** after Save in the editor, the picker is still in "uncommitted preview" mode — the user *must* click Apply or Cancel to dismiss the picker. They can also click another tile to change theme away from Custom. This is consistent with the picker's normal flow.

**Alternative considered (REJECTED):** automatically accepting the picker when the editor saves — i.e., closing both dialogs and returning the user to the main window. Rejected because: (a) it short-circuits the user's mental model of "I'm in a dialog, I commit when I'm done" — they may want to compare the new Custom against another preset before committing, (b) it deviates from the picker's documented interaction (Apply / Cancel close the picker), creating a third "Save-from-editor" path the user has to learn.

### Tile interaction (picker)

- `_ThemeTile` is a `QPushButton` subclass with custom `paintEvent` (RESEARCH Q8 recommendation — Approach A).
- `clicked` signal is connected via **bound method** (`tile.clicked.connect(self._on_tile_clicked)` — QA-05 ban on self-capturing lambdas; the slot reads the tile's stored `_theme_id` attribute via `self.sender()` or via a per-tile bound partial wrapping the theme_id — planner picks).
- Click semantics:
  1. **Live preview**: `apply_theme_palette` for the clicked tile's theme is run against the current `QApplication.instance()`. The accent override is RE-IMPOSED on top via `apply_accent_palette(app, repo.get_setting("accent_color", ""))` IF the accent setting is non-empty AND `_is_valid_hex` passes (RESEARCH Pitfall 2).
  2. **Active-tile UI update**: the picker's `_active_tile_id` member is updated; all tiles repaint (the previously-active tile loses its 3px border + checkmark; the newly-active tile gains them).
  3. **No persistence**: `repo.set_setting("theme", ...)` is NOT called. Persistence happens only on Apply.
- Hover semantics: Qt-default QPushButton hover state (subtle background lift). **No hover preview** — explicit reject per CONTEXT.md "Deferred Ideas" + RESEARCH Pitfall 8. Clicking is the only way to preview a theme.
- Disabled tile (empty Custom) interaction: `setEnabled(False)` causes Qt to ignore clicks at the QPushButton layer; the tile's `paintEvent` renders with the disabled-state visual treatment (0.5 effective opacity, dashed border, italic hint label). Tooltip `Click Customize… to create a Custom theme` is shown on hover.

### Editor row interaction

- Each `_ColorRow` is a `QWidget` containing: `QLabel` (role label) + `QPushButton` (color swatch — clickable; QSS-painted with the current hex as background) + `QLabel` (current hex display, monospace).
- Swatch button click semantics:
  1. Open `QColorDialog.getColor(QColor(self._current_hex), self, f"Choose {role_label} color", QColorDialog.ColorDialogOption.DontUseNativeDialog)` (RESEARCH Q9 — static call, modal, blocking).
  2. On accept (returned QColor is valid): update `self._current_hex = chosen.name()` (lowercase `#rrggbb` per Qt convention), refresh the swatch button background QSS, refresh the hex display label, refresh the swatch button's `accessibleName`, emit `color_changed(role_name, new_hex)` signal to the parent editor dialog.
  3. The parent editor's `_on_role_color_changed` slot updates the running `QApplication` palette for that role + re-imposes the accent override (RESEARCH Pitfall 2 — Q9 wiring).
  4. On reject (returned QColor is invalid): no-op; row state unchanged.
- `DontUseNativeDialog` matches Phase 59 (`accent_color_dialog.py:67`) — required for consistent rendering on Wayland (RESEARCH Q15) and to avoid the xdg-desktop-portal native dialog's slowness on Wayland.

#### Picker tile click → palette mutation

```python
def _on_tile_clicked(self, theme_id: str) -> None:
    self._selected_theme_id = theme_id
    app = QApplication.instance()
    # 1. Build & apply the theme's palette
    if theme_id == "system":
        # System branch: reset to a fresh QPalette (Linux: no setStyle; Windows: setStyle("Fusion") + _apply_windows_palette)
        ...
    elif theme_id == "custom":
        raw = self._repo.get_setting("theme_custom", "")
        role_hex = json.loads(raw) if raw else {}
        if not isinstance(role_hex, dict):
            role_hex = {}
        palette = build_palette_from_dict(role_hex)
        app.setPalette(palette)
    else:
        palette = build_palette_from_dict(THEME_PRESETS[theme_id])
        app.setPalette(palette)
    # 2. Re-impose accent override (RESEARCH Pitfall 2)
    accent = self._repo.get_setting("accent_color", "")
    if accent and _is_valid_hex(accent):
        apply_accent_palette(app, accent)
    # 3. Update active-tile UI
    self._refresh_active_tile()
```

#### Editor row color change → palette mutation

```python
def _on_role_color_changed(self, role_name: str, new_hex: str) -> None:
    app = QApplication.instance()
    palette = app.palette()
    role = getattr(QPalette.ColorRole, role_name, None)
    if role is None or not _is_valid_hex(new_hex):
        return  # defense-in-depth — should never trigger
    palette.setColor(role, QColor(new_hex))
    app.setPalette(palette)
    # Re-impose accent override (RESEARCH Pitfall 2)
    accent = self._repo.get_setting("accent_color", "")
    if accent and _is_valid_hex(accent):
        apply_accent_palette(app, accent)
    # Update internal role-hex dict (for Save persistence)
    self._role_hex_dict[role_name] = new_hex
```

**No throttling on either path.** Tile clicks are discrete events (no drag-through-tiles). Editor color changes are per-QColorDialog-accept (one event per Edit… click). Phase 59's "no QTimer coalesce" rationale (D-11) applies a fortiori here.

### Snapshot & restore (D-12 + D-17 + RESEARCH Q10)

- **Picker snapshot** taken in `__init__` BEFORE any wiring:
  ```python
  app = QApplication.instance()
  self._original_palette = app.palette()    # QPalette is value-copy
  self._original_qss = app.styleSheet()     # QString-equivalent value-copy
  self._save_committed = False              # set True if editor saves during this picker session
  ```
- **Editor snapshot** taken in `__init__` (independent of picker's snapshot — RESEARCH Q10):
  ```python
  self._original_palette_at_editor_open = app.palette()
  self._original_qss_at_editor_open = app.styleSheet()
  ```
  The editor's snapshot captures whatever the picker has live-previewed (e.g., if the user clicked Vaporwave in the picker, then clicked Customize, the editor's snapshot is Vaporwave + accent override). Editor Cancel restores to this snapshot, NOT to the picker's pre-tile-click snapshot.
- **Picker `reject()`**:
  ```python
  def reject(self) -> None:
      if not self._save_committed:
          app = QApplication.instance()
          app.setPalette(self._original_palette)
          app.setStyleSheet(self._original_qss)
      super().reject()
  ```
- **Editor `reject()`**:
  ```python
  def reject(self) -> None:
      app = QApplication.instance()
      app.setPalette(self._original_palette_at_editor_open)
      app.setStyleSheet(self._original_qss_at_editor_open)
      super().reject()
  ```

**Picker states (P prefix):**

|-------|---------|--------------|-------------------------|

| **P-Previewing** | User clicks a tile | `_active_tile_id` updated; all tiles repaint; main window live-updates to the clicked theme's palette + accent override re-imposed | none — `QApplication.palette()` mutated in-process only |
| **P-Editor-open** | User clicks Customize… | Editor opens on top of picker (modal); picker still visible behind editor but disabled | depends on editor's exit state |
| **P-Editor-saved** | Editor saved (returns through editor `accept()`) | Editor closes; `_save_committed = True`; `_active_tile_id = 'custom'`; tile grid repaints (Custom tile now active); main window already shows saved Custom palette via editor's live-preview | `theme = 'custom'` + `theme_custom = <JSON>` already persisted by editor's Save |
| **P-Editor-cancelled** | Editor cancelled (returns through editor `reject()`) | Editor closes; `_save_committed` stays False; `_active_tile_id` reverts to whatever it was BEFORE Customize was clicked (planner: stash `_active_tile_id` before opening editor, restore on editor reject) — actually, **simpler:** re-read repo's `theme` setting? **No:** repo wasn't mutated. Just leave `_active_tile_id` unchanged from its pre-Customize value (which was the live-previewed theme); editor's Cancel restores the palette via its own snapshot, so the visual state matches the `_active_tile_id` automatically | none |

| State | Trigger | Visual signal | Persistence side effect |
|-------|---------|--------------|-------------------------|
| **E-Idle (just opened)** | `__init__` complete; `source_preset` (the parent picker's selected theme) prefilled into 9 rows | All 9 swatches show source preset's hex; hex labels match | none — read-only |

### Pre-population on open

**Picker:**

1. `saved_theme = repo.get_setting("theme", "system")`.
2. `self._selected_theme_id = saved_theme` (default to "system" if unset).
3. `self._active_tile_id = saved_theme` (active tile renders with checkmark + 3px border).
4. `theme_custom_raw = repo.get_setting("theme_custom", "")`. If empty/corrupt JSON: Custom tile is disabled. Otherwise: Custom tile is enabled and renders its 3-color stripe from the saved JSON.

**Editor:**

1. Receive `source_preset: str` constructor argument from the picker (one of `system`, `vaporwave`, `overrun`, `gbs`, `gbs_after_dark`, `dark`, `light`, `custom`).
2. If `source_preset == 'custom'` AND `theme_custom` JSON is valid: prefill 9 rows from the saved JSON. Stash the JSON dict as `self._source_preset_palette` for Reset semantics.
3. Else if `source_preset == 'custom'` AND JSON is invalid/empty: prefill from `THEME_PRESETS['system']` (or active app palette as fallback) and stash same as `_source_preset_palette`. **Should never happen in practice** — the Custom tile is disabled when JSON is empty, so the user can't reach the editor with `source_preset='custom' AND no JSON`. Defense-in-depth fallback.
4. Else: prefill from `THEME_PRESETS[source_preset]` (skipping the `Highlight` key which isn't editable here). Stash same as `_source_preset_palette`.
5. Special case `source_preset == 'system'`: prefill from a freshly-constructed Qt-default `QPalette()` for each role (the "system" theme has no preset hex). Stash same as `_source_preset_palette`. The user effectively edits a copy of the Qt-default palette into a Custom theme — useful for users who want "almost system, but tweak Window".

- Do NOT call `setFocusPolicy(Qt.NoFocus)` on enabled tiles or any button — breaks keyboard navigation.
- Do NOT hide the inner `QColorDialog`'s numeric fields, hex field, or eyedropper to "simplify" — defeats Q14 a11y.

**Wayland DPR=1.0 deployment target — accessibility caveats:**

- Tile rendering is pixel-exact at 120×100 px DPR=1.0 (no fractional scaling concerns per project memory).
- `QColorDialog.ColorDialogOption.DontUseNativeDialog` ensures consistent rendering on Wayland (avoids the xdg-desktop-portal slow-native-dialog path — RESEARCH Q15).
