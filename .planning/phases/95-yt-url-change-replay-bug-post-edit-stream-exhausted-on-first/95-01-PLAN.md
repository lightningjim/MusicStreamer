<objective>
Fix the YouTube URL-change replay bug: after a station's stream URL is edited, the
first play attempt fails with "stream exhausted" and only the second play picks up
the new URL. The DB is correctly updated on save, but the Player's in-memory
`_streams_queue`, `_current_stream`, the loaded playbin3 URI, and any in-flight
YouTube resolution all still carry the OLD URL. This plan invalidates that stale
player state on edit and adds a generation guard so a late YouTube resolution
cannot resurface the old URL.

Purpose: deliver D-01..D-05 — first play after an edit always uses the saved URL,
without disturbing metadata-only edits or non-playing stations.

Output: `Player.invalidate_for_edit(station, is_playing)` + `_youtube_resolve_seq`
guard; MainWindow wiring at the existing edit junction; FakePlayer parity + stub;
unit and integration coverage for V1-V7, V10.

Approach is wiring + one guard + one decision method. Every mechanism already
exists in-repo (per 95-PATTERNS.md / 95-RESEARCH.md). Do NOT hand-roll a partial
player reset — reuse `Player.play(fresh_station)` for the D-01 restart.

Resolved open implementation choices (baked into tasks):
- Q1 → gate the D-01-vs-D-05 decision on the panel's `is_playing`, passed into
  `invalidate_for_edit` (Player has no playback-state field; adding one is out of scope).
- Q2 → deleted-playing-stream while audio is live → re-issue `play(updated_station)`
  on a surviving stream (RESEARCH A4; aligns with D-01 "hear the new stream"); the
  existing `play()` no-streams guard handles the all-deleted case gracefully.
- Q3 → Signal-carry for the YouTube resolve generation guard (widen
  `youtube_resolved` to `Signal(str, bool, int)`), mirroring the `_preroll_seq`
  idiom exactly. This trips the FakePlayer parity drift-guard, so `tests/_fake_player.py`
  is updated in this same wave (the drift-guard makes the parity edit self-enforcing).
</objective>
