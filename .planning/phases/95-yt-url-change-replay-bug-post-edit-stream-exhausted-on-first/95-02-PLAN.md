<objective>
Close the residual Phase 95 UAT gap: after Plan 01's fix, editing a playing
YouTube station's stream URL correctly replays the new URL on first play, but a
spurious "Stream exhausted" toast still flashes briefly before the new URL plays.

Root cause (from .planning/debug/95-spurious-stream-exhausted-toast.md): a QUEUED
error-recovery from the OLD exhausted YouTube stream — posted via QueuedConnection
from the GStreamer bus thread — runs deferred on the main loop AFTER the
synchronous edit→play() restart has already emptied `_streams_queue` for the new
(still-resolving) URL and reset `_recovery_in_flight=False`. With the coalescing
guard cleared and the queue empty, `_handle_gst_error_recovery` calls
`_try_next_stream()` → `failover.emit(None)` → `show_toast("Stream exhausted")`.
The error-recovery path has NO edit/restart generation guard, unlike the
`_youtube_resolve_seq` guard Plan 01 added to the resolve path.

Purpose: add an edit/restart generation guard to the ERROR-RECOVERY path,
mirroring the existing `_youtube_resolve_seq` / `_preroll_seq` idiom, so a stale
pre-restart recovery no-ops — WITHOUT suppressing a genuine current-generation
exhaustion (which must still toast).

Output: a `_recovery_seq` monotonic counter bumped at the top of `play()` and in
`invalidate_for_edit`'s restart path; `_error_recovery_requested` widened to
`Signal(int)` so the stamp is captured at error-POST time (bus thread) and checked
at recovery-RUN time (main thread); a stale early-return in
`_handle_gst_error_recovery`; FakePlayer parity for the widened signal; and unit
coverage V11/V12/V13.

Approach mirrors the Plan 01 `_preroll_seq` / `_youtube_resolve_seq` mechanism
exactly — capture the generation where the cross-thread event is POSTED, reject
deliveries whose stamp != current at RUN time. Do NOT suppress the toast globally;
do NOT touch the legitimate `_recovery_in_flight` cascade-coalescing behavior.

Resolved implementation choices (baked into tasks):
- Q1 (dedicated vs reused counter) → dedicated `_recovery_seq`. The existing
  `_youtube_resolve_seq` is resolve-path-specific and is bumped on metadata-only
  edits too (Plan 01 bumps it unconditionally first in invalidate_for_edit); reusing
  it would over-suppress. A dedicated `_recovery_seq` makes the recovery-path
  semantics independent and auditable, matching the one-counter-per-async-path
  precedent (`_preroll_seq` for preroll, `_youtube_resolve_seq` for resolve).
- Q2 (stamp at post-time vs run-time) → POST time. The stamp MUST be captured where
  `_error_recovery_requested` is emitted (`_on_gst_error`, bus thread, player.py:991),
  carried on the Signal, and compared at run time. Capturing at run time would read
  the post-restart seq and let a pre-restart recovery pass the guard — defeating the
  fix. Post-time capture rejects exactly the recoveries posted before the most recent
  restart and accepts genuine current-generation ones.
- Q3 (where to bump) → bump `_recovery_seq` at the TOP of `play()` (covers every
  restart path, including invalidate_for_edit's `self.play(station)` delegation) AND
  redundantly in invalidate_for_edit before the restart is unnecessary because
  invalidate's restart goes through `play()`; bump it ONCE at play() entry. (The
  D-04/D-05 no-restart branches of invalidate_for_edit do NOT bump — they leave the
  current generation valid so a legitimate same-session recovery still toasts.)
</objective>
