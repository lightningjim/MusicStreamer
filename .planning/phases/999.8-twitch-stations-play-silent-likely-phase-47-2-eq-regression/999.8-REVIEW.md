# Phase 999.8: Code Review Report

**Reviewed:** 2026-04-24
**Depth:** standard
**Files Reviewed:** 1
**Status:** issues_found

## Summary

Reviewed `musicstreamer/player.py` with focus on the Twitch resolve worker
(`_twitch_resolve_worker`) and its sibling `_youtube_resolve_worker`. The
Phase 999.8 one-line fix (`session.set_option("twitch-api-header", ...)`)
is correctly applied and does not introduce new defects.

The phase summary's daemon-thread error-masking concern is **confirmed and
material**. Both resolve workers run in `daemon=True` threads with narrow
exception coverage — unhandled exceptions print to stderr and die silently,
leaving the UI stuck (no `playback_error`, no failover advance, no `offline`
signal). This is the same failure mode that masked the original 999.8
regression: had `set_plugin_option(...)` raised `AttributeError`/`TypeError`
on the new streamlink API instead of becoming a silent no-op, the user-visible
symptom (silent Twitch playback) would have been identical because the
exception would have escaped to the thread's default handler.

The findings below recommend tightening the worker exception envelopes so
unexpected errors surface as `playback_error` and advance the failover queue.
Performance issues are out of v1 scope and not flagged.

## Warnings

### WR-01: `_twitch_resolve_worker` has unguarded code paths that silently kill the daemon thread

**File:** `musicstreamer/player.py:551-588`
**Issue:** Only `streamlink.session.streams(url)` is wrapped in try/except
(lines 569-582). The following calls are entirely unguarded:

- Line 555: `session = Streamlink()` — constructor can raise.
- Lines 565-568: `session.set_option("twitch-api-header", ...)` — this is the
  exact call the 999.8 fix changed. A future API rename or a typo would raise
  silently with no user-visible signal.
- Line 588: `self.twitch_resolved.emit(streams["best"].url)` — `streams["best"]`
  was just confirmed truthy, but property access on the stream object is
  unguarded.

Because the thread is `daemon=True` and has no top-level handler, any of
these raise→die paths produce identical UX to the original 999.8 bug: the
pipeline is in NULL state (set by `_play_twitch` line 545), no failover
timer is armed, and the user sees a stuck UI with no error toast.

**Fix:** Wrap the worker body in a top-level try/except that emits
`playback_error` and advances the queue on any unexpected exception:

```python
def _twitch_resolve_worker(self, url: str) -> None:
    from streamlink.session import Streamlink
    from streamlink.exceptions import NoPluginError, PluginError
    try:
        session = Streamlink()
        token = ""
        try:
            with open(paths.twitch_token_path()) as fh:
                token = fh.read().strip()
        except OSError:
            pass
        if token:
            session.set_option(
                "twitch-api-header",
                [("Authorization", f"OAuth {token}")],
            )
        try:
            streams = session.streams(url)
        except NoPluginError:
            self.playback_error.emit("streamlink: no plugin")
            QTimer.singleShot(0, self._try_next_stream)
            return
        except PluginError as e:
            if "No playable streams" in str(e) or "offline" in str(e).lower():
                channel = url.rstrip("/").split("/")[-1]
                self.offline.emit(channel)
                return
            self.playback_error.emit(f"streamlink: {e}")
            QTimer.singleShot(0, self._try_next_stream)
            return
        if not streams or "best" not in streams:
            channel = url.rstrip("/").split("/")[-1]
            self.offline.emit(channel)
            return
        self.twitch_resolved.emit(streams["best"].url)
    except Exception as e:  # noqa: BLE001 — daemon worker must surface ALL failures
        self.playback_error.emit(f"twitch resolve crashed: {e!r}")
        QTimer.singleShot(0, self._try_next_stream)
```

Note: `QTimer.singleShot` is being called from the worker thread here — that
is the existing pattern in lines 573 and 581, so the fix preserves it. If
that pattern is suspect on its own (no Qt loop on this thread), prefer
emitting a queued signal that runs `_try_next_stream` on the main thread.
See WR-03.

### WR-02: `_youtube_resolve_worker` has unguarded prologue/epilogue code

**File:** `musicstreamer/player.py:467-523`
**Issue:** The `extract_info` call is correctly wrapped (lines 508-513), but
the surrounding code is not:

- Line 477: `import yt_dlp` (ImportError possible if the venv is misconfigured).
- Lines 480-485: `paths.cookies_path()`, `os.path.exists`, `cookie_utils.is_cookie_file_corrupted`,
  `constants.clear_cookies()` — any of these can raise on filesystem permission errors,
  malformed paths, or unexpected cookie file shapes.
- Line 505: `cookie_utils.temp_cookies_copy()` `__enter__` — file copy can raise
  PermissionError/OSError before reaching the inner try.

Same daemon-thread failure mode as WR-01: silent death, no `youtube_resolution_failed`,
no failover advance, UI stuck.

**Fix:** Wrap the entire worker body in a top-level try/except that emits
`youtube_resolution_failed` for unexpected exceptions:

```python
def _youtube_resolve_worker(self, url: str) -> None:
    try:
        import yt_dlp
        canonical = paths.cookies_path()
        if os.path.exists(canonical) and cookie_utils.is_cookie_file_corrupted(canonical):
            constants.clear_cookies()
            self.cookies_cleared.emit(
                "YouTube cookies cleared — re-import via Accounts menu."
            )
        opts = { ... }  # unchanged
        info = None
        with cookie_utils.temp_cookies_copy() as cookiefile:
            if cookiefile is not None:
                opts["cookiefile"] = cookiefile
            try:
                with yt_dlp.YoutubeDL(opts) as ydl:
                    info = ydl.extract_info(url, download=False)
            except Exception as e:
                self.youtube_resolution_failed.emit(str(e))
                return
        # ... existing format-extraction logic ...
        self.youtube_resolved.emit(resolved)
    except Exception as e:  # noqa: BLE001
        self.youtube_resolution_failed.emit(f"youtube resolve crashed: {e!r}")
```

The existing inner `except Exception` for `extract_info` can stay; the outer
catch is a backstop for everything else.

### WR-03: `QTimer.singleShot` invoked from worker thread (lines 573, 581)

**File:** `musicstreamer/player.py:573, 581`
**Issue:** Inside `_twitch_resolve_worker` (a non-Qt daemon thread),
`QTimer.singleShot(0, self._try_next_stream)` is called. The class docstring
and bus-handler comments (lines 76-83, 323-325, 361-363) explicitly state
that singleShot from non-Qt threads "vanishes" because there is no Qt event
loop on the worker thread. The class already introduces queued signals
(`_cancel_timers_requested`, `_error_recovery_requested`) precisely to solve
this on the GLib bus-loop thread.

The Twitch worker is in the same situation. While `playback_error.emit(...)`
on lines 572 and 580 correctly auto-queues to the main thread, the
`QTimer.singleShot` lines that immediately follow may not run at all,
leaving the failover queue un-advanced after a `NoPluginError` or generic
`PluginError`.

**Fix:** Add a queued signal `_try_next_stream_requested` and emit it instead:

```python
# class scope:
_try_next_stream_requested = Signal()

# __init__:
self._try_next_stream_requested.connect(
    self._try_next_stream, Qt.ConnectionType.QueuedConnection
)

# in worker, replace QTimer.singleShot(0, self._try_next_stream) with:
self._try_next_stream_requested.emit()
```

Verify with a quick test: trigger `NoPluginError` (e.g. point at a non-Twitch
URL routed through `_play_twitch`) and confirm the next stream in the queue
is attempted.
