## Pitfall-In-The-Wild Notes

### #1: PySide6 via pip is ABI-incompatible with conda-forge GStreamer env (BLOCKER → FIXED)

**Symptom:** `ImportError: DLL load failed while importing QtCore: The specified procedure could not be found.` on every PySide6 submodule (QtCore, QtGui, QtWidgets). Even `ctypes.WinDLL("Qt6Core.dll")` failed.

**Root cause:** PySide6 wheels on PyPI are built against a different VC runtime / C++ toolchain than the one conda-forge uses for GStreamer and friends. Mixing them in one env causes Qt's low-level DLLs to fail symbol resolution against conda-installed system DLLs on `os.add_dll_directory`-registered paths.

**Diagnosis ruled out:**
- Windows 11 24H2 build 26200 — new enough for Qt 6.11
- VC++ redist already installed system-wide — didn't help
- DLL shadowing — `os.add_dll_directory(PySide6_dir)` before import didn't help
- PySide6 version — both 6.11.0 and 6.10.3 failed identically

**Fix (used this UAT):**
```powershell
pip uninstall -y PySide6 PySide6-Addons PySide6-Essentials shiboken6
conda install -c conda-forge pyside6
```
This installs PySide6 built with the same toolchain as GStreamer. `python -m musicstreamer` launches.

**Action items:**
- **pyproject.toml**: loosen or clarify `PySide6>=6.11` pin — conda-forge may lag pypi. Keep the version floor only if genuinely needed.
- **Phase 44 packaging**: use conda-forge PySide6 in the build env; PyInstaller bundles from there. pip PySide6 must not be present in the build env.
- **Dev runbook**: add a line to the Windows setup instructions: "Install PySide6 via `conda install -c conda-forge pyside6`, NOT pip. pip-installed PySide6 will not load in a conda-forge GStreamer env."
