## Pitfall-In-The-Wild Notes

### #1: PySide6 via pip is ABI-incompatible with conda-forge GStreamer env (BLOCKER → FIXED)

**Symptom:** `ImportError: DLL load failed while importing QtCore: The specified procedure could not be found.` on every PySide6 submodule (QtCore, QtGui, QtWidgets). Even `ctypes.WinDLL("Qt6Core.dll")` failed.

**Root cause:** PySide6 wheels on PyPI are built against a different VC runtime / C++ toolchain than the one conda-forge uses for GStreamer and friends. Mixing them in one env causes Qt's low-level DLLs to fail symbol resolution against conda-installed system DLLs on `os.add_dll_directory`-registered paths.

**Diagnosis ruled out:**
- Windows 11 24H2 build 26200 — new enough for Qt 6.11
- VC++ redist already installed system-wide — didn't help
- DLL shadowing — `os.add_dll_directory(PySide6_dir)` before import didn't help
- PySide6 version — both 6.11.0 and 6.10.3 failed identically

**Fix (used this UAT):**
```powershell
pip uninstall -y PySide6 PySide6-Addons PySide6-Essentials shiboken6
conda install -c conda-forge pyside6
```
This installs PySide6 built with the same toolchain as GStreamer. `python -m musicstreamer` launches.

**Action items:**
- **pyproject.toml**: loosen or clarify `PySide6>=6.11` pin — conda-forge may lag pypi. Keep the version floor only if genuinely needed.
- **Phase 44 packaging**: use conda-forge PySide6 in the build env; PyInstaller bundles from there. pip PySide6 must not be present in the build env.
- **Dev runbook**: add a line to the Windows setup instructions: "Install PySide6 via `conda install -c conda-forge pyside6`, NOT pip. pip-installed PySide6 will not load in a conda-forge GStreamer env."

### #2: QTimer.singleShot from a non-Qt thread silently drops (BLOCKER → FIXED)

**Symptom:** Every Shoutcast/Icecast stream dies at exactly 10 s on both Linux and Windows after the 43.1 bus-signal-watch marshal fix landed. Pause button reverts to play, elapsed counter keeps ticking, GStreamer log shows a clean external `unlock()` → `stop()` → `dispose()` teardown with no error message. Reproduced on SomaFM (HTTPS) and DI.fm (HTTP) — not stream-specific.

**Root cause:** Phase 43.1 bus-watch fix (`5827062`) correctly moved `bus.add_signal_watch()` onto the bridge thread so its GSource attaches to the bridge's private MainContext. Bus handlers now reliably dispatch on the bridge thread (pure GLib `MainLoop`, not a `QThread`). `_on_gst_tag` used `QTimer.singleShot(0, self._cancel_timers)` to cancel the 10 s failover timer when ICY data arrived — but `QTimer.singleShot(0, callable)` posts to the **calling thread's** Qt event loop, and the bridge thread has no Qt loop. The cancel vanished. Timer fired at 10 s → `_on_timeout` → `_try_next_stream` → `pipeline.set_state(NULL)` → "Stream exhausted" toast.

Pre-43.1 on Linux, `add_signal_watch` ran on main, so the GSource attached to whatever iterated the default MainContext (usually main via PyGObject's default-loop integration) → handlers dispatched on main → `singleShot` worked. The bug was latent until the watch was deliberately moved off main.

**Diagnosis ruled out:**
- Error-driven recovery (`_handle_gst_error_recovery`) — GStreamer debug log showed no error message at the drop point
- `souphttpsrc` timeout / TLS / ICY parse — log showed healthy byte reads up to the stop instant
- SMTC feedback loop (OS auto-sending stop events) — Linux reproduced the 10 s death with no SMTC involvement at all

**Fix (commit `f1333ed`):** Replace cross-thread `QTimer.singleShot(0, bare_callable)` with a Qt `Signal` + `QueuedConnection` — the same pattern already used for `title_changed`, `twitch_resolved`, and `youtube_resolved`. Two new internal signals route the bus-handler cancel/recovery work onto the main thread correctly regardless of which thread-default MainContext the bus watch is attached to.

**Forward-looking rule:** any `QTimer` / `QWidget` interaction from a GStreamer bus handler MUST go through a queued Qt Signal. Never pass a bare callable to `QTimer.singleShot` from the bus-loop thread. Documented in the project spike-findings skill.
