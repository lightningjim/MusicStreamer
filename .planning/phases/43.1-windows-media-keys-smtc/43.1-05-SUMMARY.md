# Phase 43.1 Plan 05 Summary — shutdown + factory fallback

## What was built

`WindowsMediaKeysBackend.shutdown()` per CONTEXT D-09, symmetric with `LinuxMprisBackend.shutdown`. Idempotent, partial-failure resilient, mitigates T-43.1-01 (SMTC session leak) and Pitfall #4 (token required for `remove_button_pressed`).

Phase 43.1 code-only scope is now complete: `WindowsMediaKeysBackend` implements all four abstract methods (`publish_metadata`, `set_playback_state`, `_apply_playback_state`, `shutdown`) + `create_windows_backend` factory.

Covers MEDIA-03 (failure path + lifecycle closure) → ROADMAP success criterion #5.

## Key decisions honored

- **D-09 (lifecycle)**: construct in MainWindow.__init__ (unchanged — wired in Phase 41), shutdown in closeEvent. `self._media_keys.shutdown()` is already called by main_window.py:289.
- **Pitfall #4**: `remove_button_pressed(self._bp_token)` uses the exact token stored in `__init__` — removal would be a no-op otherwise.
- **T-43.1-01**: `self._smtc.is_enabled = False` explicitly in shutdown() as belt-and-braces beyond `MediaPlayer.close()` — prevents session lingering if COM GC is slow.
- **Idempotency**: `self._shutdown_complete` sentinel flag — second `shutdown()` returns immediately. Critical for error paths where construction partially fails.
- **Partial-failure resilience**: each of the three shutdown steps is independently wrapped in try/except. If `remove_button_pressed` fails, `is_enabled=False` and `close()` still run. Mirrors `LinuxMprisBackend.shutdown()` exception-swallow pattern.
