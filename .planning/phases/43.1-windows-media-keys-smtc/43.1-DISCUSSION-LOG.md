## winrt API + event bridge

### Q1: Which winrt API pattern for driving SMTC?

| Option | Description | Selected |
|--------|-------------|----------|
| MediaPlayer + SMTC (Recommended) | Construct winrt MediaPlayer, grab .system_media_transport_controls off it, disable command_manager, subscribe to button_pressed. Standard pattern, no HWND interop. | ✓ |
| Direct SystemMediaTransportControls | ISystemMediaTransportControlsInterop with QMainWindow HWND. More boilerplate, COM interop, tied to a window. | |
| winsdk (older projection) | Older Python namespace projection. | |

**User's choice:** MediaPlayer + SMTC (Recommended)

### Q2: How should button_pressed callbacks reach the Qt event loop?

| Option | Description | Selected |
|--------|-------------|----------|
| Qt signal auto-queue (Recommended) | Call self.play_pause_requested.emit() directly from winrt threadpool thread; Qt auto-marshals via queued connection. | ✓ |
| QMetaObject.invokeMethod queued | Explicit QueuedConnection invocation. More verbose. | |
| Dedicated QThread dispatcher | Spawn a QThread for winrt events. Duplicated machinery. | |

**User's choice:** Qt signal auto-queue (Recommended)

## Cover art transport

### Q1: How to transport cover art to SMTC?

| Option | Description | Selected |
|--------|-------------|----------|
| Reuse _art_cache.py PNG on disk (Recommended) | write_cover_png stable per-station path; wrap file URI via RandomAccessStreamReference. | ✓ |
| In-memory RandomAccessStream from QPixmap bytes | Serialize QPixmap → QByteArray → InMemoryRandomAccessStream. No disk I/O but re-encodes every title_changed. | |
| Hybrid: disk cache, stream from bytes on failure | Premature, extra branch. | |

**User's choice:** Reuse _art_cache.py PNG on disk (Recommended)

### Q2: Rename _art_cache cache subdir from mpris-art/ for cross-platform use?

| Option | Description | Selected |
|--------|-------------|----------|
| Rename to media-art/ (Recommended) | Shared cache dir; migrate existing mpris-art/ on first launch. | ✓ |
| Keep mpris-art/ for both platforms | Tech-debt naming. | |
| Separate per-platform dirs | Duplicate helpers. | |

**User's choice:** Rename to media-art/ (Recommended)

## winrt package + install

### Q1: How does winrt-Windows.Media.Playback get installed?

| Option | Description | Selected |
|--------|-------------|----------|
| pyproject optional-deps.windows + pip (Recommended) | Add winrt-* to [project.optional-dependencies].windows; pip install in conda-forge env. | ✓ |
| winsdk (single-package projection) | Older monolithic package. | |
| Request conda-forge package | Unnecessary — pip wheels work inside conda envs. | |

**User's choice:** pyproject optional-deps.windows + pip (Recommended)

### Q2: If winrt wheel isn't installed at runtime on Windows, what happens?

| Option | Description | Selected |
|--------|-------------|----------|
| Catch ImportError → NoOp with warning (Recommended) | Factory catches ImportError, logs install hint, returns NoOp. Matches Phase 41 D-06. | ✓ |
| Hard fail with install instructions | Contradicts D-06 (blocks startup over nice-to-have). | |
| Silent NoOp | Hard to diagnose. | |

**User's choice:** Catch ImportError → NoOp with warning (Recommended)

### Q3: Set SMTC playback type to Music on DisplayUpdater?

| Option | Description | Selected |
|--------|-------------|----------|
| PlaybackType=Music + MusicDisplayProperties (Recommended) | Music-specific Windows overlay UI (title/artist layout + thumbnail). | ✓ |
| PlaybackType=Other + generic display | Less rich overlay, no benefit. | |

**User's choice:** PlaybackType=Music + MusicDisplayProperties (Recommended)

### Q4: When does the WindowsMediaKeysBackend get constructed/destroyed?

| Option | Description | Selected |
|--------|-------------|----------|
| Construct in MainWindow __init__, shutdown() in closeEvent (Recommended) | Symmetric with Phase 41 MPRIS2 lifecycle. | ✓ |
| Lazy construct on first play | Breaks symmetry, complicates factory contract. | |

**User's choice:** Construct in MainWindow __init__, shutdown() in closeEvent (Recommended)
