## Summary

Phase 43.1 lands a clean, well-documented Windows SMTC backend, a mpris-art/ → media-art/ cache rename with one-shot migration, and a `[windows]` optional-deps group. Correctness of the core SMTC logic is solid — pitfall resolutions from UAT are encoded with clear comments, shutdown is idempotent with independent try/except per step, and the Linux-importability contract (D-06) is respected via deferred winrt imports. Test coverage is strong: 100% of public API paths, both win32 factory fallback and Linux NoOp degradation paths are exercised.

Three warnings concern (1) a subtle re-entrancy hazard in `_build_thumbnail_ref` when called while an asyncio loop is already running, (2) pywinrt deps with no lower bounds (locks in a moving target), and (3) SMTC session visibility leaking before the first `publish_metadata` call. Info items are mostly cosmetic — inline imports, a non-thread-safe module sentinel whose only caller is the main thread, and a test that mutates a MagicMock class.

No critical findings. No security issues (T-41-09 path-traversal guard is preserved and tested).

## Warnings

### WR-01: `asyncio.run()` inside `_build_thumbnail_ref` raises if a loop is already running

**File:** `musicstreamer/media_keys/smtc.py:230-231`
**Issue:** `asyncio.run()` refuses to execute when an asyncio event loop is already running in the current thread (`RuntimeError: asyncio.run() cannot be called from a running event loop`). Today Qt's main loop is not an asyncio loop so the call succeeds, but if the app ever integrates a qasync/qtinter bridge (not uncommon for async HTTP in Qt apps) or if `publish_metadata` is ever called from an asyncio context, this path will raise and fall through to the `except Exception` on line 238 — thumbnails silently vanish with only a WARNING log. The existing comment ("asyncio.run never re-enters Qt's loop") is correct for today but becomes false the day an asyncio bridge is added.

**Fix:** Prefer a loop-aware pattern that degrades gracefully:
```python
import asyncio
try:
    loop = asyncio.get_running_loop()
except RuntimeError:
    loop = None

if loop is None:
    asyncio.run(_await_store(writer))
else:
    # Rare path — schedule and wait. For Qt/STA main thread today this branch
    # is never taken; it future-proofs against qasync integration.
    fut = asyncio.run_coroutine_threadsafe(_await_store(writer), loop)
    fut.result(timeout=5.0)
```
Alternatively, document the constraint explicitly in the module docstring so the next engineer sees it before touching async integration.

### WR-02: `[windows]` optional deps pin no minimum version

**File:** `pyproject.toml:24-29`
**Issue:** pywinrt 3.x is a relatively young project with breaking namespace reshuffles between minor versions (e.g. 2.x → 3.x moved `winrt.windows.*` layout). Declaring `winrt-Windows.Media.Playback` with no floor means `pip install -e ".[windows]"` will grab whatever pywinrt is current — a future release that renames an enum or restructures modules will silently break the SMTC backend with a confusing ImportError that looks indistinguishable from "winrt not installed" to the `__init__.py` factory. Phase 43.1 UAT pinned down the 3.2.x namespace layout (smtc.py lines 18-24 document this); codify it.

**Fix:**
```toml
windows = [
    "winrt-Windows.Media.Playback>=3.2,<4",
    "winrt-Windows.Media>=3.2,<4",
    "winrt-Windows.Storage.Streams>=3.2,<4",
    "winrt-Windows.Foundation>=3.2,<4",
]
```
Also consider a runtime version probe in `create_windows_backend` that logs the observed winrt version at INFO on first import so UAT logs surface drift quickly.

### WR-03: SMTC session becomes visible before first metadata publish

**File:** `musicstreamer/media_keys/smtc.py:114`
**Issue:** `__init__` sets `self._smtc.is_enabled = True` at construction time, which is correct per Pitfall #7 so the overlay sees the session — but `publish_metadata` has not yet run, so `display_updater.music_properties.title/artist` are empty strings (winrt defaults). For the window between app launch and the first station selection, the Windows Media overlay will show a blank "MusicStreamer" entry with no title, no artist, no thumbnail. Functionally harmless but looks broken to the user.

**Fix:** Either (a) defer `is_enabled = True` until the first `publish_metadata(station, ...)` call with a non-None station, or (b) seed a neutral placeholder in `__init__` so the overlay always shows something meaningful:
```python
# After configuring buttons, before is_enabled = True:
du = self._smtc.display_updater
du.type = self._type_enum.MUSIC
du.music_properties.title = "MusicStreamer"
du.music_properties.artist = "Idle"
du.update()
self._smtc.is_enabled = True
```
Option (a) is cleaner but requires moving the enable flag into `publish_metadata` with a "first call" sentinel. UAT on the Win11 VM should inform the choice.

## Info

### IN-01: `import asyncio` inside hot path

**File:** `musicstreamer/media_keys/smtc.py:230`
**Issue:** `import asyncio` is inlined inside `_build_thumbnail_ref`. Stdlib imports are cached after first use so there's no runtime cost beyond the first call, but the inline placement obscures the dependency and doesn't match the module's own convention (only *winrt* imports are deferred per D-06; asyncio is unconditional stdlib).
**Fix:** Move `import asyncio` to module scope with other stdlib imports near `import logging`.

### IN-02: Module-level `_migration_attempted` sentinel is not thread-safe

**File:** `musicstreamer/media_keys/_art_cache.py:30`
**Issue:** Two concurrent calls to `cover_path_for_station` from different threads can both pass the `if _migration_attempted: return` check before either sets the flag, leading to a duplicate `os.rename` call. In practice `cover_path_for_station` is only called from the Qt main thread (metadata publish), so this is benign today — but the module docstring doesn't state the single-thread assumption.
**Fix:** Either (a) add a one-line comment on line 30 clarifying the main-thread-only assumption, or (b) guard with a `threading.Lock`. Given the contract is best-effort and wrapped in try/except, (a) is sufficient.

### IN-03: `isinstance(station_id, int)` accepts `bool`

**File:** `musicstreamer/media_keys/_art_cache.py:61`
**Issue:** `bool` is a subclass of `int` in Python, so `cover_path_for_station(True)` returns a path ending in `1.png` without raising. The T-41-09 path-traversal guard is fine (bool can't inject path separators) but the type check is slightly misleading.
**Fix:** Optional — tighten to `if not isinstance(station_id, int) or isinstance(station_id, bool)`. Low priority; station IDs come from SQLite PKs which are never bools.

### IN-04: `_build_thumbnail_ref` passes winrt classes as positional args

**File:** `musicstreamer/media_keys/smtc.py:196-203`
**Issue:** The method takes `InMemoryRandomAccessStream, DataWriter, RandomAccessStreamReference` as positional parameters to keep the winrt imports in the caller (D-06). This works but creates a 5-arg signature that's hard to read. Since `_build_thumbnail_ref` is a private method and already inside a class that has a reliable `__init__` lifecycle, the winrt classes could be stashed on `self` in `__init__` the same way enums are (lines 121-123) — `self._stream_cls`, `self._writer_cls`, `self._ref_cls`.
**Fix:** Cache the three storage.streams classes on `self` in `__init__` (alongside `_button_enum`/`_status_enum`/`_type_enum`) and shrink the helper signature to `_build_thumbnail_ref(self, cover_pixmap, station_id)`. No behavioral change, just cleaner.

### IN-05: Test mutates MagicMock class property (potential cross-test bleed)

**File:** `tests/test_media_keys_smtc.py:306`
**Issue:** `type(args).button = property(lambda self: (_ for _ in ()).throw(RuntimeError("boom")))` installs a property on `MagicMock` itself (since `type(MagicMock())` is `MagicMock`). In practice pytest runs each test in isolation and `args` is a fresh instance, but `type(args)` is the shared `MagicMock` class — the property persists for the duration of the test process. If another test in the same file later accesses `.button` on an unrelated MagicMock instance, it will also raise.
**Fix:** Use `args.button = <descriptor>` via `PropertyMock` on the instance:
```python
from unittest.mock import PropertyMock
args = MagicMock()
type(args).button = PropertyMock(side_effect=RuntimeError("boom"))
```
Then explicitly delete or reset after the test. Alternatively, construct `args` as a non-MagicMock object (e.g., a dataclass with a property). The current tests pass, so this is a latent hazard rather than a confirmed bug.

### IN-06: `uv.lock` not reviewed (generated lockfile)

**File:** `uv.lock`
**Issue:** Lockfile is generated from `pyproject.toml` by `uv lock`; it should not be hand-edited and code review of its diff adds no value.
**Fix:** Exclude `uv.lock` from future review scopes (alongside `package-lock.json`, `poetry.lock`). Noting here so this filter can be added to `.planning/config.json` or the review workflow defaults.
