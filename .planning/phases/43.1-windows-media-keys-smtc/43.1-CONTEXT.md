# Phase 43.1: Windows Media Keys (SMTC) - Context

**Gathered:** 2026-04-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Implement `WindowsMediaKeysBackend` in `musicstreamer/media_keys/smtc.py` so Windows System Media Transport Controls drive the Player (keyboard media keys + on-screen overlay) and display station + ICY title + cover art. Validated live on the user's Windows 11 VM.

The factory, abstract base class, signal surface, cover-art PNG helper, and all `main_window.py` Player integration points were already built in Phase 41. This phase replaces the `NotImplementedError` stub with a real winrt-backed implementation. No new contract, no new wiring ŌĆö just the Windows leaf of the platform split.

**Carried forward from Phase 41 (locked, not re-discussed):**
- No-op Next/Previous (no queue concept) ŌĆö SMTC `IsNextEnabled = IsPreviousEnabled = false`
- Publish metadata on every `Player.title_changed` + `bind_station()` ŌĆö no debounce
- Construction failure ŌåÆ `NoOpMediaKeysBackend` fallback; app startup never blocks (D-06)
- Symmetric lifecycle to MPRIS2: construct in `MainWindow.__init__`, `shutdown()` in `closeEvent`

**Carried forward from Phase 43 spike:**
- Target runtime: Windows 11, conda-forge Python 3.12 env, PyInstaller 6.19+ bundle (Phase 44 will wrap this)
- UAT happens on the user's Win11 VM used during the Phase 43 spike

</domain>

<decisions>
## Implementation Decisions

### winrt API surface
- **D-01:** Use the **MediaPlayer + SMTC** pattern.
  - `winrt.windows.media.playback.MediaPlayer()` ŌåÆ `.system_media_transport_controls`
  - Set `.command_manager.is_enabled = False` so buttons flow through the SMTC we control (not MediaPlayer's auto-command routing)
  - Set `smtc.is_play_enabled = is_pause_enabled = is_stop_enabled = True`; `is_next_enabled = is_previous_enabled = False` (D-03 from Phase 41)
  - Subscribe to `smtc.add_button_pressed(handler)`; the handler inspects `SystemMediaTransportControlsButton` and emits the matching backend signal
  - Rejected: direct `ISystemMediaTransportControlsInterop` with HWND (COM interop boilerplate, ties backend to a window); rejected `winsdk` (older projection, Microsoft's recommended projection is now `winrt-*`)

### Event bridge (winrt ŌåÆ Qt)
- **D-02:** Backend emits Qt signals directly from the winrt `button_pressed` callback.
  - Callback runs on a winrt threadpool thread; `self.play_pause_requested.emit()` is thread-safe and Qt auto-queues to the main thread because the backend QObject was constructed on the main thread
  - No `QMetaObject.invokeMethod`, no QThread dispatcher, no QTimer polling ŌĆö simplest pattern, matches MPRIS2 precedent
  - Unit tests that exercise the callback on non-Qt threads must rely on Qt's queued-connection semantics (or bypass the wire by calling the internal `_on_smtc_button(btn)` helper directly)

### Cover art transport
- **D-03:** Reuse the existing `_art_cache.write_cover_png` pattern (file-on-disk, stable per-station path).
  - PNG is written to `user_cache_dir()/media-art/{station.id}.png` (renamed from `mpris-art/`, see D-04)
  - Publish via `RandomAccessStreamReference.create_from_uri(Uri("file:///..."))` so SMTC reads the same file MPRIS2 already caches
  - Stable path, no tmp-file churn, cross-platform code reuse
  - Rejected: `InMemoryRandomAccessStream` from `QByteArray` bytes (re-encodes every title_changed, more winrt plumbing); rejected hybrid fallback (premature)

### Cache directory rename
- **D-04:** Rename `_art_cache.py` cache subdir `mpris-art/` ŌåÆ `media-art/`.
  - Both Linux MPRIS2 and Windows SMTC write to the same dir ŌĆö `mpris-art/` is Linux-protocol-specific branding
  - Best-effort one-shot migration on first launch: if `{user_cache_dir}/mpris-art/` exists and `{user_cache_dir}/media-art/` does not, `os.rename()` with exception swallow (PNGs are regenerated on next `publish_metadata` anyway if migration fails)
  - Update `cover_path_for_station()` + `write_cover_png()` docstrings + existing Phase 41 tests to assert against `media-art/`
  - Docstring at top of `_art_cache.py` updated from "Phase 41, D-04" to reference both phases

### winrt package + install strategy
- **D-05:** Declare winrt projection packages under `[project.optional-dependencies].windows` in `pyproject.toml`, install via `pip` into the conda-forge env.
  - New dependency group: `windows = ["winrt-Windows.Media.Playback", "winrt-Windows.Media", "winrt-Windows.Storage.Streams", "winrt-Windows.Foundation"]`
  - Install command on the Win11 VM inside the activated conda env: `pip install -e ".[windows]"` (pip wheels coexist with conda-forge GStreamer ŌĆö proven by Phase 43 spike)
  - Linux installs never pull these packages; `pyproject.toml` already has `[project.optional-dependencies]` scaffolding
  - Rejected: `winsdk` monolithic package (older projection); rejected conda-forge recipe request (unnecessary ŌĆö pip wheels work inside conda envs, avoids recipe-turnaround dependency)

### Module-scope import guard
- **D-06:** All winrt imports stay inside function bodies, never at module scope.
  - `smtc.py` must remain importable on Linux for test-discovery and factory-introspection purposes (Phase 41 deliberately kept the stub Linux-importable)
  - The factory function `create_windows_backend(player, repo)` does `from winrt.windows.media.playback import MediaPlayer, ...` inside its body; catches `ImportError` ŌåÆ raises for the outer factory to handle

### Missing dependency handling
- **D-07:** If winrt wheels aren't installed at runtime on Windows, fall back to `NoOpMediaKeysBackend` with a hint-bearing warning.
  - `smtc.create_windows_backend` catches `ImportError`, re-raises (or raises a subclass) so the outer factory in `media_keys/__init__.py` logs `"Windows media keys unavailable ŌĆö install with pip install -e '.[windows]'"` and returns NoOp
  - Matches Phase 41 D-06: app startup never blocks on media keys
  - Distinct from `NotImplementedError` (which the current stub raises) ŌĆö now the package dependency error is the only path

### SMTC DisplayUpdater type
- **D-08:** Set `display_updater.type = MediaPlaybackType.MUSIC`; populate `music_properties.title` (ICY title) and `music_properties.artist` (station name).
  - Enables Windows' music-specific overlay UI (title + artist layout + thumbnail slot)
  - Mirror of MPRIS2 `xesam:title` / `xesam:artist` mapping from Phase 41 mpris2.py
  - Thumbnail via `display_updater.thumbnail = RandomAccessStreamReference.create_from_uri(file_uri)` using the media-art PNG (D-03)
  - Call `display_updater.update()` after every property mutation so SMTC picks up changes

### Backend lifecycle
- **D-09:** Symmetric with Phase 41 MPRIS2 lifecycle ŌĆö no wiring changes to `main_window.py`.
  - Construct in `MainWindow.__init__` via `media_keys.create(player, repo)`
  - `MediaPlayer` instance stored as `self._media_player` on the backend; kept alive for app lifetime
  - `shutdown()` in `closeEvent`: detach button_pressed handler, call `self._media_player.close()` (releases winrt), set `smtc.is_enabled = False`
  - Idempotent shutdown (same contract as MPRIS2)

### Playback state mapping
- **D-10:** Map the base class `Literal["playing", "paused", "stopped"]` to `MediaPlaybackStatus`:
  - `playing` ŌåÆ `PLAYING`
  - `paused` ŌåÆ `PAUSED`
  - `stopped` ŌåÆ `STOPPED`
  - Set via `smtc.playback_status = MediaPlaybackStatus.PLAYING` inside `_apply_playback_state()` (base class has already validated the string)

### Test strategy
- **D-11 (Claude's Discretion ŌĆö unit-test surface):**
  - Linux test runs: the Windows-specific tests skip via `@pytest.mark.skipif(sys.platform != "win32", ...)` OR run with winrt fully mocked via `unittest.mock.patch` on the deferred-import sites.
  - Recommendation: pure-Python unit tests that mock the winrt module at import time (call the factory, verify it catches ImportError ŌåÆ returns NoOp). Exercise `_on_smtc_button` by calling it directly with a mock `args.button` object ŌĆö no winrt required.
  - Full integration validation happens as UAT on the user's Win11 VM (step 3 of success criteria). CI can't run SMTC.

### Claude's Discretion
- Exact winrt import granularity (flat vs submodule imports) ŌĆö implementer picks what reads cleanest
- Logger messages and their severity levels
- Whether button_pressed handler dispatches via `match` statement or dict lookup
- Timing of `display_updater.update()` call (single batched update per `publish_metadata` vs per-field)
- Unit-test mocking strategy (see D-11)

</decisions>

## Existing Code Insights

### Reusable Assets
- `MediaKeysBackend` (base.py): abstract class to subclass; signal surface fixed, `_apply_playback_state` hook already validated
- `NoOpMediaKeysBackend` (base.py): the fallback when winrt is unavailable
- `_art_cache.write_cover_png` / `cover_path_for_station`: PNG-on-disk helper; rename cache subdir to `media-art/` (D-04), otherwise reuse as-is
- `LinuxMprisBackend` (mpris2.py): structural template ŌĆö `_MprisRootAdaptor`/`_MprisPlayerAdaptor` correspond to the SMTC/DisplayUpdater pair; copy the `_build_metadata_dict` shape and `publish_metadata`/`_apply_playback_state` surface
- `MainWindow` wiring (main_window.py:247ŌĆō395): 15 call sites already present; signal connections `play_pause_requested`, `stop_requested`, `next_requested`, `previous_requested` and `publish_metadata`/`set_playback_state` calls all ready for the Windows backend

### Established Patterns
- **Lazy-import in factory**: `media_keys/__init__.py` already does `from musicstreamer.media_keys.smtc import create_windows_backend` inside a try block; `smtc.py` does the winrt import inside the factory body (D-06)
- **NoOp fallback on any failure**: Phase 41 D-06 pattern ŌĆö every construction exception collapses to `NoOpMediaKeysBackend()` with a single `_log.warning` line
- **Stable cache path per station**: `_art_cache` keys PNGs by `station.id`; MPRIS2 publishes `file://` URIs; Windows SMTC will consume the same files via `RandomAccessStreamReference.create_from_uri`
- **Metadata-publish cadence**: every `title_changed` signal and every `bind_station()` rebind; no throttling (D-05 from Phase 41)

### Integration Points

<specifics>
## Specific Ideas

- "Same file the Linux MPRIS2 backend already caches" ŌĆö the Windows backend should not have its own cache logic; reuse `_art_cache.py` after the D-04 rename
- Mirror `mpris2.py` structure: docstring block at top documenting winrt constants (namespaces, button enum), then the backend class, then helpers. Reader opens either file and sees the same shape.
- Match Phase 41's construction-failure contract precisely ŌĆö the factory's `try/except` in `__init__.py` already handles it, so `create_windows_backend` just needs to raise on failure (any exception), not catch and NoOp internally

</specifics>

<deferred>
## Deferred Ideas

- **SMTC `Raise` / window-focus action** ŌĆö SMTC overlays don't have a "focus main window" button that MPRIS2 might eventually wire; not needed for v2.0
- **Volume round-trip via SMTC** ŌĆö Windows SMTC doesn't expose volume the way MPRIS2 does; volume stays Player-internal. Revisit with SEED-001 volume-slider work if/when it lands
- **Seek / position properties** ŌĆö MusicStreamer plays live streams; meaningless on both platforms
- **GlobalSystemMediaTransportControlsSessionManager observation** ŌĆö this would make MusicStreamer observe *other* apps' SMTC sessions; out of scope
- **Multiple concurrent instances** ŌĆö Linux MPRIS2 has the same limitation (single service name); punt until someone actually asks
- **Testing SMTC button_pressed on Linux CI** ŌĆö genuinely cannot be done without Windows; UAT on the Win11 VM is the integration proof

</deferred>
