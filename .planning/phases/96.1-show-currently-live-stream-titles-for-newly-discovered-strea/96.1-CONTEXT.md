# Phase 96.1: Show Live Titles for Newly-Discovered Streams - Context

**Gathered:** 2026-06-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Refine the Phase 96 `LiveRefreshDialog` (`musicstreamer/ui_qt/live_refresh_dialog.py`) so that **newly-discovered currently-live streams** — live streams on the provider's channel that don't correspond to an existing flagged station — are surfaced with a **reliable, identifiable title** and can be **mapped/merged onto an existing station** (not only added as brand-new).

**Confirmed root cause (the actual bug):** the channel scan (`yt_import.scan_playlist`, `extract_flat='in_playlist'` on the `/streams` tab) populates `title` reliably for finished VODs but frequently returns blank / `"Untitled"` for *just-started* live streams. Those streams therefore appear unidentifiable in the dialog, so the user cannot map or merge them. So this phase has both a **data-fetch** dimension (resolve the real title) and a **display/action** dimension (show it, act on it).

**Scope anchor:** YouTube only (inherited from Phase 96). Still manual review-and-confirm; no auto-apply. This is a targeted refinement of the existing dialog — not a new capability.
</domain>

<decisions>
## Implementation Decisions

### Title resolution (the core fix)
- **D-01:** Root cause is confirmed: the flat scan returns blank/`"Untitled"` titles for newly-started live streams. Fix with a **hybrid** approach — keep the fast flat `scan_playlist`, then for **only** the entries whose title is blank/`"Untitled"`, do a targeted **per-video metadata fetch** to resolve the real current title. Do NOT per-video-resolve every entry (slower, more failure surface); resolve just the blanks.
- **D-02:** Per-video title resolution MUST run **off the UI thread** (extend the existing `_LiveRefreshScanWorker` / Phase 96 D-09 threading discipline) and MUST thread the resolved `node_runtime` / `check_node()` through to yt-dlp. This is a hard landmine — YouTube extraction fails under `.desktop` launchers (minimal PATH, no Node) without it. See memory `yt-dlp-callsites-need-resolved-node-runtime` and `yt-streams-tab-flat-scan-live-status-none`.
- **D-03:** **Fallback when a real title still can't be resolved** (private / just-starting / fetch error): show the watch URL / video ID (e.g. `Untitled — watch?v=abc123`) so the stream stays identifiable and mappable. A row must **never** show a bare blank.

### Newly-discovered section & display
- **D-04:** Add a dedicated **"Newly discovered on channel"** section. Each newly-discovered live stream displays its current YouTube title as **read-only text** (rendered `Qt.PlainText` — scan titles are untrusted; carries forward Phase 96 CR-01 / T-39-01) **alongside** the existing editable Name field. The read-only title is "what YouTube calls it live right now"; the editable name is "what the station will be called."
- **D-05:** List **every** currently-live stream in this section — **no dedup hiding** of streams already chosen as a flagged station's REMAP target. The user wants full visibility; incidental double-listing (a stream appearing both in a REMAP dropdown and the new-stream section) is acceptable.

### Actions on a newly-discovered row
- **D-06:** Each newly-discovered row offers two per-row actions: **Add as a new station** (existing behavior) OR **Map/merge onto an existing station** (re-point that station's stale primary stream URL — and refresh its title anchor — to this live URL).
- **D-07:** The map/merge **target** is chosen via a **dropdown of this provider's flagged stations** (the same set the dialog already loads via `repo.list_flagged_stations_for_provider`). Conflicting double-maps (same target station, or same live URL targeted by multiple rows) are caught **fail-closed by the existing WR-03 duplicate-target guard** in `apply_refresh` at Apply time. No proactive in-UI greying this phase.
- **D-08:** Conservative defaults carry forward (Phase 96 D-10): newly-discovered rows are **unchecked by default**; nothing is applied without an explicit tick; the pre-Apply change summary behavior is unchanged.

### Claude's Discretion
- Exact "blank" predicate (empty vs literal `"Untitled"` vs whitespace) and whether to cache resolved titles for the dialog session.
- Whether per-video blank-resolution runs concurrently (bounded pool) or sequentially, and any progress affordance while blanks resolve (e.g. "Resolving titles…").
- Exact placement/ordering of the new section relative to the REMAP rows, and the precise row layout for the read-only-title + name + action controls.
- Where the single-video title-resolution helper lives (a new `yt_import` helper vs. reusing the `player._youtube_resolve_worker` extract_info pattern).
</decisions>

<specifics>
## Specific Ideas

- The lived symptom: after the channel restarts a 24/7 live stream, the new stream shows up "blank/Untitled" in the refresh dialog and can't be told apart or acted on. Making the real title visible is the whole point.
- "Mapped/merged" means: a newly-discovered stream can take over an existing flagged station (replacing its dead stream), not just spawn a new station.
</specifics>

## Existing Code Insights

### Reusable Assets
- `LiveRefreshDialog` + `_RowWidget` + `apply_refresh` — the entire surface being extended; the new section is additive to `_populate_rows`.
- `player._youtube_resolve_worker` — drop-in template for single-video `extract_info` with correct node-runtime threading (per-video title fetch, D-01/D-02).
- `repo.list_flagged_stations_for_provider` — already returns the exact set needed for the map-target dropdown (D-07); no new query needed.
- WR-03 duplicate-target guard in `apply_refresh` (208-222) — already enforces fail-closed conflict handling, so the new stream→station map path needs no separate guard (D-07).

### Established Patterns
- Off-UI-thread scan, main-thread persistence on Apply (Phase 96 D-09) — the blank-title resolution must extend the worker, not block the UI (D-02).
- Untrusted scan titles rendered as `Qt.PlainText` (Phase 96 CR-01 / T-39-01) — applies to the new read-only title labels (D-04).
- Conservative unchecked-by-default rows; empty-apply guard (Phase 96 D-10) — carries forward (D-08).

### Integration Points
- `_LiveRefreshScanWorker.run` (or a follow-on resolution step) — where blank entries get a per-video title fetch.
- `_populate_rows` — where the "Newly discovered on channel" section + per-row add/map controls render.
- `apply_refresh` — already handles `remap`/`add`; the new stream→station "map" reuses the `remap` change shape (station_id + stream_id + scan_result).
</code_context>

<deferred>
## Deferred Ideas

- **Map/merge into ANY station (not just flagged)** — user chose flagged-only this phase. A new stream targeting an unflagged station whose stream died is out of scope here.
- **Proactive in-UI conflict greying** — disabling an already-targeted station in the dropdown before Apply. Deferred; rely on the Apply-time WR-03 guard (D-07).
- **Thumbnail / "started X ago" disambiguators** — user chose URL/ID fallback only (D-03); richer sibling-disambiguation visuals are not in scope.
- **Always-resolve-every-entry titles** — rejected for cost/fragility (D-01); only blanks are resolved.

</deferred>
