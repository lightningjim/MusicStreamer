## Summary

Reviewed the per-video YouTube title resolution backend (`resolve_live_title`) and
the new "Newly discovered on channel" section of the live-refresh dialog. The
phase-specific landmines are all handled correctly:

- **Threading (D-02):** `_LiveRefreshScanWorker.run()` touches no widgets — it only
  calls `scan_playlist` / `resolve_live_title` and emits Signals. It threads
  `self._node_runtime` into both calls. Verified clean.
- **Untrusted titles (D-04 / T-39-01):** every QLabel rendering a scan-derived title,
  station name, or anchor calls `setTextFormat(Qt.PlainText)` (`_DiscoverRowWidget`
  L492, `_RowWidget` L335/L340). The discover-section header label (L785) renders a
  *static* string only, so its `<b>` rich-text is safe. No untrusted title rendered
  as rich text.
- **yt-dlp callsite (D-02):** `resolve_live_title` uses `build_js_runtimes(node_runtime)`
  (never inline), omits `extract_flat`, sets `socket_timeout=10`, and wraps yt-dlp in
  `cookie_utils.temp_cookies_copy()`. Verified.
- **Fallback (D-03):** `resolve_live_title` never returns a bare-blank title — on any
  exception or blank result it returns `_fallback_title(url)`.
- **Map action (D-07):** `_DiscoverRowWidget.build_staged_change` emits the existing
  `remap` shape and relies on the `apply_refresh` WR-03 duplicate-target guard rather
  than re-implementing it. Verified.

Two real defects survive adversarial refutation: a silent last-write-wins collision
when the same station is targeted by both its REMAP row and a discover MAP row (the
WR-03 guard only dedups by URL, not by station_id), and the dead `QFormLayout` /
`QPushButton` imports Pyright flagged. Two minor maintainability items round it out.

## Warnings

### WR-01: Same-station double-targeting silently collapses to last-write-wins

**File:** `musicstreamer/ui_qt/live_refresh_dialog.py:218-234, 236-269`

**Issue:** The WR-03 duplicate-target guard in `apply_refresh` fails closed only on
duplicate *target URLs*. It does NOT detect two staged changes that point at the same
*station*. This is now reachable because the new "Newly discovered" section adds a
second independent path to remap an existing flagged station:

1. Flagged station X gets a `_RowWidget` (action `remap`, **checked by default** per
   `default_check_state("remap") == True`) targeting its anchor-closest live URL_A.
2. The user also takes a `_DiscoverRowWidget`, switches it to "Map onto existing
   station", and selects station X with a different live stream URL_B.

Both rows produce `{"action": "remap", "station_id": X, ...}` with *distinct* URLs, so
`dupes` is empty and the guard passes. `apply_refresh` then runs both remaps against
station X's primary stream sequentially:

```python
repo.update_stream(primary.id, URL_A, ...)   # from _RowWidget
repo.set_live_url_title_anchor(X, title_A)
...
repo.update_stream(primary.id, URL_B, ...)   # from _DiscoverRowWidget — overwrites
repo.set_live_url_title_anchor(X, title_B)
```

The first remap is silently discarded (last-write-wins on `primary.id`), and the
anchor is left pointing at whichever change applied last — an outcome the user never
chose. Because the default-checked REMAP row makes this easy to trigger by accident,
this is a data-integrity defect, not a theoretical one.

**Fix:** Extend the fail-closed guard to also reject duplicate station targets across
remap rows, before any mutation:

```python
remap_station_ids = [
    change.get("station_id")
    for change in staged_changes
    if change.get("action") == "remap" and change.get("station_id") is not None
]
station_dupes = sorted({
    sid for sid in remap_station_ids if remap_station_ids.count(sid) > 1
})
if station_dupes:
    raise ValueError(
        "The same station is mapped to more than one live stream "
        f"(station ids: {station_dupes}). Each station must map to a single stream."
    )
```

### WR-02: Dead imports `QFormLayout` and `QPushButton`

**File:** `musicstreamer/ui_qt/live_refresh_dialog.py:39, 44`

**Issue:** Both `QFormLayout` and `QPushButton` are imported but never referenced
anywhere in the module (confirmed by grep — the only occurrences are the import lines
themselves). Pyright flags these as unused. They are leftover from the replaced ADD
section. The Apply/Cancel buttons are created via `QDialogButtonBox.addButton`, not
`QPushButton`, and no `QFormLayout` is constructed.

**Fix:** Remove both names from the `PySide6.QtWidgets` import block:

```python
from PySide6.QtWidgets import (
    QButtonGroup,
    QCheckBox,
    QComboBox,
    QDialog,
    QDialogButtonBox,
    QHBoxLayout,
    QLabel,
    QLineEdit,
    QProgressBar,
    QRadioButton,
    QScrollArea,
    QSizePolicy,
    QVBoxLayout,
    QWidget,
)
```

## Info

### IN-01: `_on_add_selection_changed` only fills name when field is empty, diverging from `_DiscoverRowWidget`

**File:** `musicstreamer/ui_qt/live_refresh_dialog.py:383-389`

**Issue:** In `_RowWidget`, the ADD-row selection handler updates the name field only
when `self._name_edit.text() == ""`. The new `_DiscoverRowWidget` has no equivalent
"update name on combo change" behavior at all — its name field is set once at
construction from the entry title and never re-synced. This is not a correctness bug
(the discover row's name is one-shot by design and `build_staged_change` falls back to
the entry title), but the two row types now handle ADD-name population with subtly
different rules, which is an inconsistency a future maintainer can trip over. Worth a
one-line comment noting the intentional difference, or aligning the behavior.

**Fix:** Add a comment in `_DiscoverRowWidget.__init__` near L501 documenting that the
name field is intentionally one-shot (not re-synced on combo/radio change), to make
the divergence from `_RowWidget._on_add_selection_changed` explicit.

### IN-02: `replace` action is declared but never produced or applied

**File:** `musicstreamer/ui_qt/live_refresh_dialog.py:322-325, 422-440`

**Issue:** `_RowWidget` renders a `"replace": "REPLACE"` action label and
`build_staged_change` has a `# remap / replace` branch, but no code path ever
constructs a `_RowWidget` with `action="replace"` (rows are created only with
`"remap"` at L771), and `apply_refresh` has no `elif action == "replace"` handler — a
`replace` change would silently fall through all branches and be dropped. This is dead
/ aspirational scaffolding rather than a live bug, but it invites confusion about
whether REPLACE is a supported, tested action (it is not).

**Fix:** Either remove the `"replace"` label entry and the `replace` references from
the `build_staged_change` docstring/branches, or add a tracking comment that REPLACE
is reserved for a future phase and is intentionally unreachable today.
