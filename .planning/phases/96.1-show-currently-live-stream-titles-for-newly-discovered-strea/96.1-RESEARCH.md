## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** Hybrid approach — keep fast flat `scan_playlist`, then do targeted per-video metadata fetch **only** for entries whose title is blank/`"Untitled"`. Do NOT per-video-resolve every entry.
- **D-02:** Per-video title resolution MUST run off the UI thread (extend `_LiveRefreshScanWorker` threading discipline) AND MUST thread the resolved `node_runtime` / `check_node()` through to yt-dlp. Hard landmine — YouTube extraction fails under `.desktop` launchers without it.
- **D-03:** Fallback when title still can't be resolved: show `Untitled — watch?v=<id>` (the watch URL / video ID). A row must never show a bare blank.
- **D-04:** Add a dedicated **"Newly discovered on channel"** section. Each row: read-only current YouTube title (Qt.PlainText — untrusted), editable Name field, add/map controls.
- **D-05:** List **every** currently-live stream — no dedup hiding. Acceptable that a stream appears both in a REMAP dropdown and the new section.
- **D-06:** Each newly-discovered row offers two per-row actions: **Add as a new station** OR **Map/merge onto an existing station** (re-point that station's stale primary stream URL + refresh title anchor).
- **D-07:** Map/merge target chosen via **dropdown of this provider's flagged stations** (same set as `repo.list_flagged_stations_for_provider`). Conflicts caught fail-closed by existing WR-03 duplicate-target guard in `apply_refresh`. No proactive in-UI greying.
- **D-08:** Conservative defaults carry forward — newly-discovered rows **unchecked by default**; no auto-apply.

## Summary

Phase 96.1 is a targeted refinement of the existing `LiveRefreshDialog` introduced in Phase 96. The confirmed root cause is that `yt_import.scan_playlist` (which uses `extract_flat='in_playlist'` on the `/streams` tab) returns `entry.get("title", "Untitled")` — yielding either `""` or `"Untitled"` for just-started live streams because yt-dlp's flat scan does not resolve per-video metadata for live entries.

The fix has two independent halves: (1) a new `yt_import.resolve_live_title(url, node_runtime)` helper that runs a targeted per-video `extract_info` (no `extract_flat`, but with `playlist_items="0"` style bounds and `socket_timeout=10` for safety) using the same cookie/node-runtime/`temp_cookies_copy` pattern established in `fetch_channel_avatar` and `player._youtube_resolve_worker`; (2) extension of the dialog to replace the current "Add currently-live stream" ADD section with a richer "Newly discovered on channel" section where each row shows the resolved title (read-only, Qt.PlainText) alongside an editable Name field and a per-row add-or-map action toggle.

The threading model is straightforward: extend `_LiveRefreshScanWorker.run()` to perform blank resolution inline after the flat scan completes (sequential, capped at a few entries per real-world usage), emitting the fully-resolved results via the existing `finished` Signal. No new QThread class is needed. The WR-03 duplicate-target guard in `apply_refresh` already covers the new map action shape (it's a `remap` change record with `station_id` from the MAP dropdown).

**Primary recommendation:** Add `yt_import.resolve_live_title(url, *, node_runtime=None) -> str` as a new module-level function in `yt_import.py`; extend `_LiveRefreshScanWorker.run()` to call it for blanks after the flat scan; replace the ADD section in `_populate_rows` with a `_DiscoverRowWidget` that supports both add and map modes.

### Pattern 1: Per-Video Title Resolution Helper (new `yt_import.resolve_live_title`)

**What:** A new module-level function in `yt_import.py` that calls `yt_dlp.YoutubeDL.extract_info` on a single video URL with no `extract_flat`, bounded by `socket_timeout=10`, using the standard `temp_cookies_copy` + `js_runtimes` + `remote_components` pattern. Returns the resolved title string, or the fallback string on failure.

**When to use:** Only for entries where `_is_blank_title(entry["title"])` is True after the flat scan.

**Template (modeled after `fetch_channel_avatar` in `yt_import.py`):** [VERIFIED: read from yt_import.py lines 207-276]

```python
# Source: yt_import.py fetch_channel_avatar pattern (lines 207-276)
def resolve_live_title(
    url: str,
    *,
    node_runtime: "NodeRuntime | None" = None,
) -> str:
    """Fetch the current live title for a single YouTube video URL.

    Returns the resolved title, or a fallback string "Untitled — <url>"
    if the title cannot be resolved (D-03).

    MUST be called off the UI thread (D-02).
    node_runtime MUST be threaded through (BUG-11 / D-02 landmine).
    """
    opts = {
        # No extract_flat — we need per-video metadata (title is absent in flat scan)
        "socket_timeout": 10,         # safety: don't hang the worker thread
        "quiet": True,
        "no_warnings": True,
        "skip_download": True,
        "js_runtimes": yt_dlp_opts.build_js_runtimes(node_runtime),  # LANDMINE: must thread
        "remote_components": {"ejs:github"},
    }
    with cookie_utils.temp_cookies_copy() as cookiefile:
        if cookiefile is not None:
            opts["cookiefile"] = cookiefile
        try:
            with yt_dlp.YoutubeDL(opts) as ydl:
                info = ydl.extract_info(url, download=False)
        except Exception:
            return _fallback_title(url)
    title = (info or {}).get("title", "")
    return title if title and title.strip() else _fallback_title(url)


def _fallback_title(url: str) -> str:
    """D-03: derive a stable fallback title from the URL."""
    # Extract video ID for a compact label: "Untitled — watch?v=abc123"
    from urllib.parse import urlparse, parse_qs
    parsed = urlparse(url)
    vid = parse_qs(parsed.query).get("v", [None])[0]
    if vid:
        return f"Untitled — watch?v={vid}"
    return f"Untitled — {url}"
```

### Pattern 2: Blank Title Predicate

**What:** A pure function `_is_blank_title(title: str) -> bool` that returns True when the scan-produced title needs per-video resolution.

**When to use:** Applied to every entry after `scan_playlist` returns, before resolving.

**Rationale for the predicate choices** (Claude's Discretion):

`yt_import.py` line 135 returns `entry.get("title", "Untitled")`. Two cases produce a needs-resolution title:
- yt-dlp returns `None` → `entry.get("title", "Untitled")` yields `"Untitled"`
- yt-dlp returns `""` (empty string) → `entry.get("title", "Untitled")` yields `""` (the default only fires for missing/None, not `""`)
- yt-dlp returns `"  "` (whitespace) → yields whitespace

The predicate should match all three: [ASSUMED — based on yt-dlp behavior observed in CONTEXT.md and memory `yt-streams-tab-flat-scan-live-status-none`]

```python
_BLANK_TITLES = frozenset({"", "untitled"})

def _is_blank_title(title: str) -> bool:
    """Return True when a flat-scan title needs per-video resolution (D-01)."""
    return not title or title.strip().lower() in _BLANK_TITLES
```

### Pattern 3: Extending `_LiveRefreshScanWorker.run()` for Blank Resolution

**What:** After calling `yt_import.scan_playlist`, the worker iterates live entries and resolves any blank titles sequentially (the real-world case is 1-2 blank entries per scan). [VERIFIED: read _LiveRefreshScanWorker lines 86-96]

```python
# Source: live_refresh_dialog.py _LiveRefreshScanWorker.run() (lines 86-96)
def run(self):
    """Blocking scan + blank resolution — MUST NOT touch any widget."""
    try:
        results = yt_import.scan_playlist(
            self._url,
            toast_callback=self._toast,
            node_runtime=self._node_runtime,
        )
        # D-01: resolve blank titles for live entries only
        for entry in results:
            if yt_import._is_blank_title(entry.get("title", "")):
                entry["title"] = yt_import.resolve_live_title(
                    entry["url"],
                    node_runtime=self._node_runtime,  # D-02 LANDMINE: must thread
                )
        self.finished.emit(results)
    except Exception as exc:
        self.error.emit(str(exc))
```

**Sequential vs concurrent:** Sequential is correct here. Real-world usage: 1-5 live streams per channel scan, of which at most 1-2 have blank titles. The sequential overhead is < 5 seconds per blank. A progress affordance ("Resolving titles…") on `_status_label` during the scan (already visible as "Scanning channel for currently-live streams…") is sufficient — no separate progress bar needed.

### Anti-Patterns to Avoid

- **Calling `extract_info` with `extract_flat='in_playlist'` for per-video resolution:** This is the flat scan mode — it skips per-video metadata and will return the same blank title. ALWAYS omit `extract_flat` for per-video fetches.
- **Omitting `socket_timeout`:** Without it, a stalled connection blocks the worker thread indefinitely. `fetch_channel_avatar` uses `socket_timeout=10` for exactly this reason.
- **Omitting `js_runtimes` / `node_runtime` threading:** The landmine. YouTube extraction under `.desktop` launchers fails silently. Every yt-dlp callsite MUST thread the resolved `node_runtime` via `yt_dlp_opts.build_js_runtimes(node_runtime)`.
- **Omitting `remote_components: {"ejs:github"}`:** Required for yt-dlp 2026.03.17+ when YouTube account cookies are present (authenticated code path). Already present in `scan_playlist` and `_youtube_resolve_worker`.
- **Calling yt-dlp OUTSIDE `temp_cookies_copy`:** yt-dlp's `save_cookies()` side effect corrupts the canonical cookies file. Always nest `yt_dlp.YoutubeDL` inside the `temp_cookies_copy()` context manager.
- **Touching widgets from `_LiveRefreshScanWorker.run()`:** The method MUST NOT access any Qt widget. Emit Signals only.
- **Using HTML in the read-only title label:** Must be Qt.PlainText. A scan-provided title like `<img src=http://attacker/>` cannot trigger a remote-resource load.
- **Auto-applying a map based on anchor similarity:** D-05 and Phase 96 D-05 both prohibit auto-apply. The NAME PRE-FILL is fine; silently staging a change is not.

## Common Pitfalls

### Pitfall 1: `extract_flat='in_playlist'` in the resolve helper

**What goes wrong:** The new `resolve_live_title` accidentally includes `"extract_flat": "in_playlist"` in opts. yt-dlp short-circuits per-entry JS solving for flat scans (documented in `yt_import.py` comment at line 100-104), so the result is the same blank title as before.

**Why it happens:** Copy-paste from `scan_playlist` opts without removing `extract_flat`.

**How to avoid:** The per-video resolve opts MUST NOT include `extract_flat`. Use `fetch_channel_avatar` (lines 207+) as the template, not `scan_playlist`.

**Warning signs:** Tests pass on VODs (which have titles even in flat scan) but fail on live entries.

### Pitfall 2: Blank predicate too narrow (`== ""` only)

**What goes wrong:** Only checks `title == ""` but yt-dlp sometimes returns `None` (which becomes `"Untitled"` via `entry.get("title","Untitled")`). Entries with title `"Untitled"` don't get resolved.

**Why it happens:** CONTEXT.md says "blank/`Untitled`" but the predicate only guards one case.

**How to avoid:** `_is_blank_title` must catch `""`, `"Untitled"` (case-insensitive), and whitespace-only strings.

**Warning signs:** Just-started live streams show `"Untitled"` in the dialog even after the resolution step.

### Pitfall 3: `node_runtime` not threaded through `resolve_live_title`

**What goes wrong:** `_LiveRefreshScanWorker.run()` calls `resolve_live_title(url)` (no `node_runtime`) or `yt_import` function uses a hardcoded `node_runtime=None`. Works in terminal but fails silently in `.desktop` launchers.

**Why it happens:** This is the documented BUG-11 landmine (memory `yt-dlp-callsites-need-resolved-node-runtime`). Easy to miss when adding a new callsite.

**How to avoid:** `_LiveRefreshScanWorker.__init__` already stores `self._node_runtime`; pass it to `resolve_live_title(entry["url"], node_runtime=self._node_runtime)`.

**Warning signs:** Title resolution works when launched from terminal but returns fallback string when launched from `.desktop`.

### Pitfall 4: `_DiscoverRowWidget` "map" staged change missing `stream_id`

**What goes wrong:** The map action builds `{"action": "remap", "station_id": ..., "scan_result": ...}` but omits `"stream_id"`. In `apply_refresh`, the `stream_id` key is used to find the correct stream: `(s for s in streams if s.id == stream_id)`. Without it, the `next()` falls through to `streams[0]` (defensive fallback), which may or may not be position==1.

**Why it happens:** The `stream_id` comes from iterating `station.streams`, which requires reading the `Station.streams` list. Easy to skip if copying from a simpler template.

**How to avoid:** In `_DiscoverRowWidget.build_staged_change()` when mode is "map", find `primary_stream_id` by iterating `chosen_station.streams` for `position == 1`, fallback to `streams[0].id` if none found. This is the exact same logic as `_RowWidget.build_staged_change()` lines 416-421. [VERIFIED: live_refresh_dialog.py lines 414-428]

**Warning signs:** Map applies successfully but updates the wrong stream on stations with multiple streams.

### Pitfall 5: Widget-touching inside `_LiveRefreshScanWorker.run()`

**What goes wrong:** Adding a progress affordance ("Resolving titles…") by updating `_status_label` inside `run()`. Qt widget access from non-main thread causes a crash or silent corruption.

**Why it happens:** The existing `run()` method doesn't touch widgets; new resolution code might inadvertently be written to do so.

**How to avoid:** Only emit Signals from `run()`. If a progress affordance is desired during resolution, add a new `Signal(str)` (e.g. `status_update = Signal(str)`) and connect it to `_status_label.setText` with `Qt.QueuedConnection`.

**Warning signs:** Segfault or `QObject::setParent: Cannot set parent, new parent is in a different thread` warnings.

```python
# Source: modeled from yt_import.py fetch_channel_avatar + player._youtube_resolve_worker
# [VERIFIED: yt_import.py lines 207-276, player.py lines 2026-2064]

def _is_blank_title(title: str) -> bool:
    """Return True when a flat-scan title needs per-video resolution (D-01)."""
    return not title or title.strip().lower() in frozenset({"", "untitled"})


def _fallback_title(url: str) -> str:
    """D-03: derive a stable fallback title from a YouTube watch URL."""
    from urllib.parse import urlparse, parse_qs
    parsed = urlparse(url)
    vid = parse_qs(parsed.query).get("v", [None])[0]
    return f"Untitled — watch?v={vid}" if vid else f"Untitled — {url}"


def resolve_live_title(
    url: str,
    *,
    node_runtime: "NodeRuntime | None" = None,
) -> str:
    """Fetch the current live title for a single YouTube video URL.

    Returns the resolved title, or a fallback string on any error (D-03).
    MUST be called off the UI thread (D-02).
    node_runtime MUST be threaded through (BUG-11 / D-02 landmine).
    """
    node_path = node_runtime.path if node_runtime else None
    _log.info("resolve_live_title: url=%s node_path=%s", url, node_path)

    opts = {
        # OMIT extract_flat — it suppresses per-video title resolution (Pitfall 1)
        "socket_timeout": 10,          # prevent hanging the worker thread
        "quiet": True,
        "no_warnings": True,
        "skip_download": True,
        "js_runtimes": yt_dlp_opts.build_js_runtimes(node_runtime),  # D-02 LANDMINE
        "remote_components": {"ejs:github"},   # required for authenticated path
    }

    with cookie_utils.temp_cookies_copy() as cookiefile:
        if cookiefile is not None:
            opts["cookiefile"] = cookiefile
        try:
            with yt_dlp.YoutubeDL(opts) as ydl:
                info = ydl.extract_info(url, download=False)
        except Exception as exc:
            _log.warning("resolve_live_title: extraction failed for %s: %s", url, exc)
            return _fallback_title(url)

    title = (info or {}).get("title", "")
    if title and title.strip():
        return title.strip()
    return _fallback_title(url)
```

```python
        # D-01: resolve blank titles sequentially (real-world: 1-2 blank entries)
        for entry in results:
            if yt_import._is_blank_title(entry.get("title", "")):
                entry["title"] = yt_import.resolve_live_title(
                    entry["url"],
                    node_runtime=self._node_runtime,  # D-02 LANDMINE: must thread
                )
        self.finished.emit(results)
    except Exception as exc:
        self.error.emit(str(exc))
```

### `_DiscoverRowWidget.build_staged_change()` — Map Mode

```python
# Source: modeled from _RowWidget.build_staged_change() lines 382-428 [VERIFIED]
def build_staged_change(self) -> Optional[dict]:
    if not self.is_staged():
        return None

    name = self._name_edit.text().strip()
    entry = self._entry  # the scan result dict

    if self._action_is_add():
        return {
            "action": "add",
            "name": name or entry.get("title", ""),
            "scan_result": entry,
            "provider_name": self._provider_name,
        }

    # Map mode: build a remap change targeting the chosen flagged station
    chosen_station: Optional[Station] = self._map_combo.currentData(Qt.UserRole)
    if chosen_station is None:
        return None

    # Find primary stream_id — same logic as _RowWidget lines 416-421
    primary_stream_id: Optional[int] = None
    if hasattr(chosen_station, "streams") and chosen_station.streams:
        for s in chosen_station.streams:
            if s.position == 1:
                primary_stream_id = s.id
                break
        if primary_stream_id is None:
            primary_stream_id = chosen_station.streams[0].id

    return {
        "action": "remap",
        "station_id": chosen_station.id,
        "stream_id": primary_stream_id,
        "scan_result": entry,
    }
```
