## Summary

Phase 60.4 layers two thin informational surfaces onto Phase 60's existing GBS.FM machinery. The data already flows through the system on both surfaces — `state["queue_summary"]` is populated by `_fold_ajax_events` (`gbs_api.py:279-280`) but its renderer was removed in 60-10 D-10c; the token count is parseable from a single HTML snippet (`<th class="plinfo">`) on the GBS home page (`/`) but is not yet fetched. No new architectural primitives are introduced; every pattern is a clone-and-adapt of an existing Phase 60.x asset.

The work decomposes into three additive code surfaces, each independent enough to ship as its own wave:
1. **`gbs_api.fetch_user_tokens(cookies)`** — new HTTP fetcher + parser, mirrors `fetch_active_playlist`'s shape (`_open_with_cookies` + `GbsAuthExpiredError` propagation). Pure module-level addition; no UI consumer needed for the unit tests.
2. **Search-dialog token label** — new `_GbsTokenWorker` (mirrors `_GbsSubmitWorker` 30 LoC), new `_token_label` QLabel placed after `_breadcrumb_label`, new `WARNING_COLOR_HEX` token in `_theme.py`, two new slots, four call sites (`showEvent`, `_refresh_login_gate`, `_on_submit_finished` success branch, `_on_submit_error` auth-expired branch). Optimistic-decrement state on `self._last_known_tokens: Optional[int]`.
3. **Playlist-summary row** — single guarded `addItem` line in `_on_gbs_playlist_ready` between `_gbs_playlist_widget.clear()` and the `▶ {icy}` row. Reverses 60-10 D-10c: jargon (`dongs`) preserved verbatim. One existing test assertion flips polarity.

**Primary recommendation:** Three plans (Wave A: api, Wave B: search-dialog, Wave C: panel-summary). Wave A is API-only and unit-testable in isolation; Wave B depends on Wave A's `fetch_user_tokens` symbol; Wave C is fully independent of A/B (different file, different data path). Waves A and B can land in the same merge window; Wave C can land before, after, or in parallel.

### Pattern 1: HTML scrape with `_open_with_cookies` + module-level regex

**What:** Mirror `fetch_active_playlist`'s HTTP shape for the home page; mirror `_LINKED_SONGID_RE`'s module-level regex for parsing.
**When to use:** Any new HTML-scrape capability in `gbs_api.py`.
**Example (regex path — recommended):**
```python
# Source: gbs_api.py existing patterns at lines 211, 219-240, 875

# Module-level regex (sibling of _LINKED_SONGID_RE at line 211)
_TOKEN_RE = re.compile(r"You have (\d+) tokens?", re.IGNORECASE)


def fetch_user_tokens(cookies: http.cookiejar.MozillaCookieJar) -> Optional[int]:
    """Phase 60.4: GET / (home page); parse <th class="plinfo"> for integer.

    Returns int on success, None on parse miss / network error / 5xx.
    Raises GbsAuthExpiredError on 302 → /accounts/login/ (mirrors search()).

    Pitfalls (60-RESEARCH):
      6 — HTML scrape fragility: regex anchors on stable prose, not class names.
      7 — GET-with-side-effects: NO retries on transient failure (this is a GET
          but it has no side effect, so technically Pitfall 7 doesn't bite us;
          but the no-retry convention still applies as a project-wide invariant).
    """
    url = f"{GBS_BASE}/"
    try:
        with _open_with_cookies(url, cookies, timeout=_TIMEOUT_READ) as resp:
            html_str = resp.read().decode("utf-8", errors="replace")
    except urllib.error.HTTPError as e:
        if e.code in (301, 302) and "/accounts/login/" in (e.headers.get("Location") or ""):
            raise GbsAuthExpiredError(f"Session expired (home 302→login)") from e
        # Defensive: 5xx / 4xx other than 302→login → return None (D-T2)
        _log.warning("fetch_user_tokens HTTPError %s for %s", e.code, url)
        return None
    except (urllib.error.URLError, OSError) as e:
        # Network failure → None (D-T2 silent log)
        _log.warning("fetch_user_tokens network failure: %s", e)
        return None
    m = _TOKEN_RE.search(html_str)
    if not m:
        # Empty playlist / not-logged-in / template change → None (D-T2)
        _log.warning("fetch_user_tokens parse miss for %s", url)
        return None
    try:
        return int(m.group(1))
    except (TypeError, ValueError):
        _log.warning("fetch_user_tokens malformed integer in match %r", m.group(0))
        return None
```

**Note on `_open_with_cookies` exception path:** `_open_with_cookies` itself catches the 302→login case AND raises `GbsAuthExpiredError` directly (line 158-159). So the `except urllib.error.HTTPError` block above will only catch other HTTP errors (5xx, 4xx). The `except` should NOT re-catch `GbsAuthExpiredError` — let it propagate. **[VERIFIED: `gbs_api.py:146-160`]**

```python
      - typed finished signal carrying the integer count
      - error signal carrying 'auth_expired' sentinel OR raw message
      - QA-05 bound-method connection convention applied at the call site

    Pitfall 5+7 (60-RESEARCH): NO retry on transient failure. Caller's next
    trigger (next submit, next dialog open) is the retry.
    """
    # finished payload: token count (int)
    finished = Signal(int)
    # error: ('auth_expired' sentinel OR raw message)
    error = Signal(str)

    def __init__(self, cookies, request_id: int, parent=None):
        super().__init__(parent)
        self._cookies = cookies
        # request_id mirrors _GbsSearchWorker.search_version (line 132): each
        # kick stamps a monotonic id; slot discards if mismatched against the
        # dialog's current id. Phase 60-RESEARCH §Pitfall 1 token-discard idiom.
        self.request_id = request_id

    def run(self):
        try:
            count = gbs_api.fetch_user_tokens(self._cookies)
            if count is None:
                # D-T2 failure path: emit error so dialog can show '—' placeholder
                self.error.emit("parse_failed")
            else:
                self.finished.emit(int(count))
        except Exception as exc:
            if isinstance(exc, gbs_api.GbsAuthExpiredError):
                self.error.emit("auth_expired")
            else:
                self.error.emit(str(exc))
```

### Pattern 4: Token-discard guard mirroring `_search_version` / `_gbs_poll_token`

**What:** Monotonic int incremented on each kick; slots discard callbacks when their stamped id doesn't match the dialog's current id.
**When to use:** Any worker whose result might land out-of-order with respect to subsequent worker kicks.
**Example:**
```python
# Source: gbs_search_dialog.py:131-132, 891-911 (_GbsSubmitWorker.search_version
#         + _on_submit_finished discard guard);
#         now_playing_panel.py:402-403, 1077-1078 (_gbs_poll_token).

def _kick_token_worker(self, cookies) -> None:
    self._token_request_id += 1
    request_id = self._token_request_id
    self._token_worker = _GbsTokenWorker(cookies, request_id, parent=self)
    self._token_worker.finished.connect(self._on_token_fetched)  # QA-05
    self._token_worker.error.connect(self._on_token_error)        # QA-05
    self._token_worker.start()

def _on_token_fetched(self, count: int) -> None:
    # Pitfall 1: discard if the request id is stale (a newer kick ran while we
    # were in-flight — e.g., user submitted twice rapidly).
    worker = self.sender()
    if worker is not None and getattr(worker, "request_id", -1) != self._token_request_id:
        return  # stale — newer kick is authoritative
    self._last_known_tokens = count
    self._token_label.setText(f"Tokens: {count}")
    self._apply_token_color(count)
    self._token_worker = None
```

### Anti-Patterns to Avoid

- **Lambdas in `.connect()` calls:** Forbidden by QA-05. Use bound methods only. Existing test `test_no_self_capturing_lambdas_in_dialog` (`tests/test_gbs_search_dialog.py:185`) greps the source — any lambda will break it.
- **Re-parsing tokens out of `_token_label.text()`:** Don't. Use `self._last_known_tokens: Optional[int]` instance attribute. Display-string parsing breaks if D-T5 wording ever changes.
- **Toast on token-fetch failure:** Forbidden by D-T2 + Pitfall 3. Silent log + `Tokens: —` placeholder only.
- **Periodic poll of `fetch_user_tokens`:** Out of scope per D-T1. Refresh is on-open + after-each-successful-submit only.
- **Sanitization of "dongs" / gbs.fm jargon:** Forbidden by D-S2. The user explicitly chose to preserve site voice.
- **Carrying `state["queue_summary"]` value through the bridge state machine:** Out of scope. The summary row is a single `addItem` per render tick — no flag, no source-of-truth coupling, no interaction with `_gbs_label_source` / `_gbs_ajax_disabled` / `_last_icy_title`.

## Common Pitfalls

### Pitfall A (60.4-specific): Token-fetch race during rapid submit

**What goes wrong:** User clicks "Add!" twice in rapid succession. First submit success spawns refetch worker A. Before A lands, second submit success spawns refetch worker B. Worker B lands first (network jitter); worker A lands second, OVERWRITING the just-fetched accurate count with a stale one.
**Why it happens:** No token-discard machinery on the new `_GbsTokenWorker` callbacks.
**How to avoid:** Apply the `_search_version` / `_gbs_poll_token` idiom: monotonic int incremented on each kick; slot discards if the worker's stamped id doesn't match the current id. Pattern 4 above shows the shape. **[VERIFIED: existing at `gbs_search_dialog.py:131-132, 891-911` and `now_playing_panel.py:402-403, 1077-1078`]**
**Warning signs:** After two rapid submits, the token count display "bounces" to a stale higher value (e.g., 48 → 47 → 46 → 47 visible), or settles on a count that differs from what a manual page-reload shows.

### Pitfall B (60.4-specific): Optimistic decrement crossing the color-tier boundary

**What goes wrong:** `_last_known_tokens = 4`. User submits successfully. Optimistic decrement: 4 → 3. The color tier should now flip from default (4+) to amber (1-3). If the implementation only calls `setText` and not `_apply_token_color`, the color stays default while the number is amber-zone.
**Why it happens:** Forgetting to invoke the color-tier helper after the optimistic decrement.
**How to avoid:** The optimistic-decrement path MUST call BOTH `_token_label.setText(...)` AND `_apply_token_color(n)`. A small helper `_render_token_count(n: Optional[int])` that does both is recommended.
**Warning signs:** Color displays one tier behind the displayed number; particularly visible on the 4→3 and 1→0 transitions.

### Pitfall C (60.4-specific): Auth-expired during in-flight token fetch

**What goes wrong:** User opens dialog while logged in; token-fetch worker spawned. While in-flight, user logs out via Accounts dialog (clears cookies). Token-fetch worker proceeds with stale cookie jar; gbs.fm 302→login; `GbsAuthExpiredError` raised; worker emits `error.emit('auth_expired')`. Slot fires. But `_refresh_login_gate` was ALREADY called (by AccountsDialog cookie-clear), so the dialog is already in not-logged-in state.
**Why it happens:** The token-fetch worker outlives the cookies file's existence.
**How to avoid:** `_on_token_error('auth_expired')` should set `_token_label` to `Tokens: —` + clear stylesheet, then call `_refresh_login_gate()` (idempotent — no-op if already in not-logged-in state). The login-gate transition will hide `_token_label` via D-T7. **No new lifecycle code needed beyond the existing pattern at `_on_search_error` (`gbs_search_dialog.py:444-453`).**
**Warning signs:** Stale `Tokens: 48` text remains visible briefly after Accounts→Disconnect.

### Pitfall D (60.4-specific): Singular vs plural in regex

**What goes wrong:** When the user has exactly 1 token, gbs.fm renders `"You have 1 token"` (singular, no `s`). A regex `r"You have (\d+) tokens"` would miss this case → parse miss → `Tokens: —`.
**Why it happens:** English plural-by-default in dev fixtures (the captured fixture has 48 tokens; never tested with 1).
**How to avoid:** Use `r"You have (\d+) tokens?"` — the optional `s` quantifier matches both forms. Add a unit test for the singular case using a hand-crafted fragment fixture (or modify a copy of `home_playlist_table.html` to swap `48 tokens` → `1 token`).
**Warning signs:** Token label displays `—` exactly when the count hits 1; user (correctly) wonders why their single remaining token suddenly shows as a fetch failure.

### Pitfall E (60.4-specific): Empty-playlist home page renders different HTML

**What goes wrong:** When the gbs.fm playlist is empty (no songs queued), the `<th class="plinfo">` line may be absent OR replaced with a different message. Parser → no match → `Tokens: —`. But the user may legitimately have 48 tokens — just no songs queued.
**Why it happens:** gbs.fm conditionally renders the plinfo row based on playlist state.
**How to avoid:** This is the documented D-T2 behavior — parse miss → `—` placeholder, silent log. Acceptable degradation for an edge case (empty queue is rare on a public radio station). If user reports this as a bug post-ship, reopen as a follow-on phase that captures an empty-playlist home-page fixture and parses a fallback location.
**Warning signs:** User reports "tokens column shows `—` even though I'm logged in and active." — verify by visiting gbs.fm in a browser.

D-T2 and D-T8 both forbid toasts. The `auth_expired` toast at `_on_submit_error` (line 937) is ALREADY emitted by the existing submit machinery — D-T8 does NOT add a second toast; it only flips the label state BEFORE the existing `_refresh_login_gate()` call.

### Token fetcher with regex (recommended)

```python
# Source: gbs_api.py existing patterns at lines 211, 219-240, 875

# Module-level compiled regex (sibling of _LINKED_SONGID_RE at line 211).
# Singular `1 token` and plural `N tokens` both matched via the optional `s`.
_TOKEN_RE = re.compile(r"You have (\d+) tokens?", re.IGNORECASE)


def fetch_user_tokens(cookies: http.cookiejar.MozillaCookieJar) -> Optional[int]:
    """Phase 60.4 D-T1: GET /; parse <th class="plinfo"> for token count.

    Returns int on success.
    Returns None on parse miss / network error / 5xx / malformed integer (D-T2).
    Raises GbsAuthExpiredError on 302 → /accounts/login/ (mirrors search()).

    Pitfalls (60-RESEARCH §Common Pitfalls):
      6 — HTML scrape fragility: regex anchors on stable English prose,
          not on CSS class names that may evolve.
      7 — GET-with-side-effects: NO retries (stylistic — this fetch is
          idempotent, but the project-wide convention is no-retry).
    """
    url = f"{GBS_BASE}/"
    try:
        with _open_with_cookies(url, cookies, timeout=_TIMEOUT_READ) as resp:
            html_str = resp.read().decode("utf-8", errors="replace")
    except urllib.error.HTTPError as e:
        # _open_with_cookies already raises GbsAuthExpiredError on 302→login;
        # this except catches OTHER HTTPErrors (5xx, 4xx).
        _log.warning("fetch_user_tokens HTTPError %s for %s", e.code, url)
        return None
    except (urllib.error.URLError, OSError) as e:
        _log.warning("fetch_user_tokens network failure: %s", e)
        return None

    m = _TOKEN_RE.search(html_str)
    if not m:
        _log.warning("fetch_user_tokens parse miss for %s", url)
        return None
    try:
        return int(m.group(1))
    except (TypeError, ValueError):
        _log.warning("fetch_user_tokens malformed integer in match %r", m.group(0))
        return None
```

1. **Singular `1 token` rendering verification.**
   - What we know: regex `tokens?` covers both singular and plural English.
   - What's unclear: whether gbs.fm actually renders `1 token` (singular) or always pluralises (`1 tokens`).
   - **Resolution:** ship with `tokens?`; add UAT step to verify at-1-token state on the user's account.

2. **Empty-playlist home page parser miss.**
   - What we know: `<th class="plinfo">` is conditionally rendered; absence → `—` placeholder.
   - What's unclear: whether absence is the intended UX or whether we should fall back to a different parse location.
   - **Resolution:** ship with current behavior (D-T2 `—` placeholder); revisit only if user reports as bug.

3. **Color tier transition during in-flight refetch race.**
   - What we know: optimistic decrement applies tier transition immediately; refetch overwrite applies tier again.
   - What's unclear: whether a fast refetch that lands BEFORE the user sees the optimistic value visibly causes a `Tokens: 48 → Tokens: 47 → Tokens: 47` flicker.
   - **Resolution:** not a concern in practice — the optimistic write is synchronous in the slot; the refetch is on a worker thread. The visible state will be `48 → 47 (optimistic) → 47 (server, no-op)` in 99% of cases. Only a network round-trip-zero (impossible in practice) could cause flicker. Document but don't gate on it.

4. **Whether the optimistic decrement floor should be 0 or no floor.**
   - What we know: D-T3 says "decrement by 1" without specifying floor behavior.
   - What's unclear: if `_last_known_tokens == 0` and user submits anyway (ignoring `Tokens: 0` red warning), should optimistic decrement go to -1?
   - **Resolution:** floor at 0 via `max(0, n - 1)`. The success branch should not be entered if the server rejected the submit for "not enough tokens" (that's the `is_error` branch). So `_last_known_tokens` going to -1 is impossible in healthy gbs.fm behavior — but `max(0, ...)` is defensive against gbs.fm-side state desync. Test by injecting an `_last_known_tokens = 0` initial state, firing success, asserting display reads `Tokens: 0` (not `-1`). The next refetch overrides anyway.
