## Token source & refresh

### Q1 — Refresh trigger

| Option | Description | Selected |
|---|---|---|
| On open + after each successful submit | Refetch on dialog open + on `_on_submit_finished` success branch. ~50ms extra HTTP per submit. No periodic poll. | ✓ |
| On open only (fire-and-forget) | One fetch when dialog opens; never refresh until reopen. Counter goes stale immediately after first submit. | |
| On open + periodic poll (every 30s) | Catches token-grants from elsewhere. Extra HTTP for slow-changing value. | |

**User's choice:** On open + after each successful submit.

### Q2 — Fetch failure UX

| Option | Description | Selected |
|---|---|---|
| Hide the label entirely | Honest-data; silent log; no fake count. | |
| Show 'Tokens: —' (em-dash placeholder) | Layout-stable; explicit "unknown". | ✓ |
| Show toast 'Couldn't fetch token count' | Visible feedback. Trade-off: noisy (Pitfall 3). | |

**User's choice:** Em-dash placeholder. Note: prefers layout stability over the hide-entirely option I recommended.

### Q3 — Post-submit window (refetch in flight)

| Option | Description | Selected |
|---|---|---|
| Optimistic decrement: 48 → 47 immediately, then server-confirm | Counter drops by one the moment submit succeeds; refetch overwrites. | ✓ |
| Show 'Tokens: —' until refetch lands | Same code path as initial-load and failure. Brief flash of unknown. | |
| Keep the old count visible until refetch lands | Stale 48 until server confirms 47. Looks like submit was free. | |

**User's choice:** Optimistic decrement.
