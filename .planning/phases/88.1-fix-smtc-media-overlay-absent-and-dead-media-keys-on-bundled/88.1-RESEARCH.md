## Summary

The Phase 88 UAT gap G2 is a **PyInstaller namespace-package bundling defect**. The pywinrt 3.2.x library distributes Windows Runtime bindings as multiple separate PyPI distributions that all contribute files into the same `winrt/` namespace package. Each distribution ships one or more **compiled `.pyd` extension modules** (e.g. `winrt/_winrt.cp312-win_amd64.pyd`, `winrt/_winrt_windows_media_playback.cp312-win_amd64.pyd`) that PyInstaller's static import analysis cannot discover because they are loaded dynamically via namespace package machinery, not via conventional `from package import name` statements analyzable at build time.

The current spec declares `winrt.windows.media`, `winrt.windows.media.playback`, `winrt.windows.storage.streams`, and `winrt.windows.foundation` as `hiddenimports`. This tells PyInstaller to include those Python subpackage `__init__.py` files, but it does **not** cause collection of the compiled `.pyd` extensions sitting at the top of the `winrt/` namespace root (`winrt/_winrt.cp312-win_amd64.pyd` from `winrt-runtime`, `winrt/_winrt_windows_media_playback.cp312-win_amd64.pyd` from `winrt-Windows.Media.Playback`, etc.). When the frozen exe attempts `from winrt.windows.media.playback import MediaPlayer`, the `__init__.py` succeeds to load (it was collected), but it immediately imports `winrt._winrt_windows_media_playback` — a compiled extension that was never bundled — causing `ImportError`. The factory catches this silently and degrades to NoOp.

The fix is `collect_all("winrt-runtime")` + `collect_all` for each installed pywinrt namespace distribution. The `collect_all` function, unlike `hiddenimports`, returns a `(datas, binaries, hiddenimports)` triplet where `binaries` includes all `.pyd` compiled extensions collected via `collect_dynamic_libs()`. This is the same pattern already used successfully for `charset_normalizer`, `streamlink`, and `yt_dlp` in the existing spec (lines 25-33).

**Primary recommendation:** Apply `collect_all("winrt-runtime")` + `collect_all` for each of the four installed pywinrt namespace distributions in `MusicStreamer.spec`, following the exact same pattern already proven for charset_normalizer/streamlink/yt_dlp. Add backend-selection logging in the factory. Add a Linux static spec test. Add a `build.ps1` SMTC smoke assertion.

### Pattern 1: collect_all for namespace package with compiled extensions

**What:** Collect all datas + binaries + submodule imports from a named PyPI distribution. For pywinrt, this is the ONLY way to pick up compiled `.pyd` modules that live in the namespace root (`winrt/`) and are invisible to modulegraph.

**When to use:** Any time a package ships compiled `.pyd` extensions at a namespace root level that are loaded dynamically (not via static `import pkg.ext` analyzable by modulegraph).

**Example (following the existing spec pattern for charset_normalizer/streamlink/yt_dlp):**
```python
# Source: wheel inspection of pywinrt 3.2.1 + collect_all PyInstaller docs
# [VERIFIED: PyPI registry + PyInstaller docs]

from PyInstaller.utils.hooks import collect_all, copy_metadata

# pywinrt 3.2.x: each distribution ships a compiled .pyd at winrt/_winrt_<namespace>.pyd
# that __init__.py dynamically imports. hiddenimports alone cannot collect .pyd files.
# Must collect each distribution that is installed, since each has its own .pyd.
_wr_datas,  _wr_binaries,  _wr_hiddenimports  = collect_all("winrt-runtime")
_wmp_datas, _wmp_binaries, _wmp_hiddenimports = collect_all("winrt-Windows.Media.Playback")
_wm_datas,  _wm_binaries,  _wm_hiddenimports  = collect_all("winrt-Windows.Media")
_wss_datas, _wss_binaries, _wss_hiddenimports = collect_all("winrt-Windows.Storage.Streams")
_wf_datas,  _wf_binaries,  _wf_hiddenimports  = collect_all("winrt-Windows.Foundation")

a = Analysis(
    ...
    binaries=... + _wr_binaries + _wmp_binaries + _wm_binaries + _wss_binaries + _wf_binaries,
    datas=...   + _wr_datas    + _wmp_datas    + _wm_datas    + _wss_datas    + _wf_datas,
    hiddenimports=[
        ...
        # Remove the old winrt.* hiddenimport lines — collect_all provides them
    ] + _wr_hiddenimports + _wmp_hiddenimports + _wm_hiddenimports
      + _wss_hiddenimports + _wf_hiddenimports,
    ...
)
```

**Important:** The existing `hiddenimports` for `winrt.windows.media`, `winrt.windows.media.playback`, `winrt.windows.storage.streams`, `winrt.windows.foundation` (spec lines 122-125) should be REMOVED when `collect_all` is added, to avoid duplicate entries. The `collect_all` hiddenimports output already covers these.

**Why the distribution names not the module names:** PyInstaller's `collect_all` argument is the **PyPI distribution name** (hyphenated, case-insensitive for the lookup), not the Python import name. `collect_all("winrt-runtime")` and `collect_all("winrt-Windows.Media.Playback")` are the correct invocations because these are the `pip install` names. [VERIFIED: PyInstaller docs + project pyproject.toml]

### Pattern 2: Backend-selection logging (D-03)

**What:** Log which backend was selected (or why it fell back) at WARNING or INFO level in the factory. Preserves the "never raises" contract.

**When to use:** Every time the factory selects a backend — both the success and failure paths.

**Example:**
```python
# media_keys/__init__.py — augmented factory (D-03)
if sys.platform == "win32":
    try:
        from musicstreamer.media_keys.smtc import create_windows_backend
        backend = create_windows_backend(player, repo)
        _log.info("Media keys: WindowsMediaKeysBackend selected (SMTC active)")
        return backend
    except Exception as e:
        _log.warning(
            "Media keys: falling back to NoOp on win32 — %s: %s",
            type(e).__name__, e,
        )
        return NoOpMediaKeysBackend()
```

**Why WARNING not DEBUG for the failure path:** The silent degradation is exactly what let G2 ship undetected. The failure log must be visible at the default `logging.WARNING` level that `__main__.py` sets globally. `_log.warning(...)` is correct. [ASSUMED — the existing spec already uses _log.warning for the failure path; this is a content improvement]

### Pattern 3: build.ps1 smoke assertion for backend identity (D-05)

**What:** Extend the frozen exe's `--smoke` invocation (or add a sibling `--check-mediakeys` flag) to print the backend class name, then assert it in build.ps1 before exit.

**When to use:** After PyInstaller completes, before Inno Setup compiles the installer.

**Recommended approach:** Extend `__main__.py`'s existing `--smoke` path with a `--check-mediakeys` flag that:
1. Runs without a Qt event loop (QCoreApplication is fine, matching the existing `_run_smoke` signature)
2. Calls `media_keys.create(...)` with a mock/None player+repo
3. Prints `MEDIAKEYS_BACKEND=WindowsMediaKeysBackend` or `MEDIAKEYS_BACKEND=NoOpMediaKeysBackend`
4. Exits 0 on success, exits non-zero on NoOp

build.ps1 invocation:
```powershell

Write-Host "=== SMTC SMOKE GUARD: assert WindowsMediaKeysBackend in frozen bundle ==="
Invoke-Native {
    & "..\..\dist\MusicStreamer\MusicStreamer.exe" --check-mediakeys 2>&1 | Out-Host
}
if ($LASTEXITCODE -ne 0) {
    Write-Host "BUILD_FAIL reason=smtc_backend_not_loaded hint='MusicStreamer.exe --check-mediakeys returned non-zero; winrt import failed in frozen bundle; check build.log for ImportError'" -ForegroundColor Red
    exit 11
}
Write-Host "SMTC SMOKE GUARD OK"
```

The CONTEXT.md says to follow exit-code-guard style; 10 is the existing maximum (post-bundle plugin guard), so 11 is the natural next slot. [ASSUMED — could also be 12 if another guard is planned; confirm by reading build.ps1 header comments]

Keeping them separate keeps the smoke check fast and reliable in the build environment. The existing `_run_smoke` harness is kept intact. ### Anti-Patterns to Avoid

- **`hiddenimports`-only for compiled namespace extensions:** Listing `winrt.windows.media.playback` as a hidden import tells PyInstaller to include `winrt/windows/media/playback/__init__.py`. It does NOT collect the `.pyd` at `winrt/_winrt_windows_media_playback.cp312-win_amd64.pyd`. The `.pyd` is a sibling of `__init__.py` at the DISTRIBUTION root, not a subpackage.

- **`collect_all("winrt")`:** The bare `winrt` identifier is NOT a PyPI distribution name — there is no `winrt` package on PyPI (pip index versions winrt returns nothing). The distribution names are `winrt-runtime`, `winrt-Windows.Media`, etc. Using `collect_all("winrt")` would silently fail to find any distribution and collect nothing.

- **UPX compression on `.pyd` files:** The spec already has `upx=False` (correct). UPX-compressing `.pyd` extension modules breaks Windows DLL loading. Do not introduce UPX for pywinrt .pyd files.

- **`collect_all` on the wrong argument type (module name vs distribution name):** `collect_all` accepts a PyPI distribution name (the `pip install` name). For pywinrt, the module import path (`winrt.windows.media.playback`) differs from the distribution name (`winrt-Windows.Media.Playback`). Use the distribution name.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Discovering which .pyd files to bundle | Manual file enumeration or regex scan of site-packages | `collect_all("winrt-runtime")` etc. | collect_all uses PyInstaller's authoritative binary collection logic; manual enumeration misses edge cases (version-suffixed filenames, DLL deps) |

## Common Pitfalls

### Pitfall 1: `collect_all` argument is the distribution name, not the Python module name

**What goes wrong:** Developer writes `collect_all("winrt.windows.media.playback")` (module name) instead of `collect_all("winrt-Windows.Media.Playback")` (PyPI distribution name). PyInstaller silently collects nothing because no distribution with a dotted name matches.

**Why it happens:** The pywinrt distribution names use title-case with dots and dashes (`winrt-Windows.Media.Playback`) while the import paths use lowercase with dots (`winrt.windows.media.playback`). The mapping is not obvious.

**How to avoid:** Always use the PyPI distribution name — the same name used in `pip install` or `pyproject.toml [optional-dependencies].windows`. [VERIFIED: PyPI registry]

**Warning signs:** `collect_all` returns three empty lists `([], [], [])` and PyInstaller emits no error.

### Pitfall 2: `_winrt.pyd` is in the winrt-runtime distribution, not a winrt/* namespace-package distribution

**What goes wrong:** Collector adds `collect_all` for the four WinRT Windows namespace distributions but omits `winrt-runtime`. The `_winrt.cpXX-win_amd64.pyd` is never bundled. All four namespace `__init__.py` files execute `from winrt._winrt import ...` (via `winrt.system`) and crash with ImportError.

**Why it happens:** The pywinrt architecture has a core runtime (`winrt-runtime`) that ships `winrt/_winrt.pyd` (the compiled Python/C++ bridge), and the namespace distributions each ship their own `winrt/_winrt_<ns>.pyd`. The core is a distinct distribution and must be collected explicitly.

**How to avoid:** Always include `collect_all("winrt-runtime")` in addition to the four namespace distributions.

**Warning signs:** Frozen exe crashes with `ImportError: cannot import name '_winrt' from 'winrt' (unknown location)` — this is the exact error reported in pywinrt issue #41. [CITED: github.com/pywinrt/pywinrt/issues/41]

### Pitfall 3: UPX stripping .pyd files (spec already correct — regression guard)

**What goes wrong:** UPX compression is applied to `.pyd` compiled extension modules. Windows DLL loader cannot load compressed `.pyd` files.

**Why it happens:** PyInstaller's `upx=True` applies UPX globally to all binaries.

**How to avoid:** The spec already has `upx=False` on both EXE and COLLECT. The new winrt binaries are automatically covered by this setting. Do not change `upx=False`. [VERIFIED: reading MusicStreamer.spec lines 174, 187]

### Pitfall 4: The winrt namespace package has no `winrt/__init__.py` — it is a PEP 420 namespace package

**What goes wrong:** Developer assumes `collect_all("winrt-runtime")` collects a `winrt/__init__.py` that serves as a registry for all subpackages. No such file exists. Each distribution independently contributes to the `winrt/` namespace. PyInstaller must collect each one separately.

**Why it happens:** Modern Python namespace packages (PEP 420) allow multiple distributions to install into the same package path without a shared `__init__.py`. Each `collect_all` call is required because no single distribution "owns" the namespace root.

**How to avoid:** Call `collect_all` for EACH installed pywinrt distribution (runtime + 4 namespace packages = 5 total for MusicStreamer's use).

**Warning signs:** Only some winrt imports work; others fail with ImportError at import time in the frozen exe.

### Pitfall 5: PyInstaller modulegraph cannot follow dynamic imports from compiled extensions

**What goes wrong:** Developer adds `winrt.windows.media.playback` as a `hiddenimport` and assumes PyInstaller will now analyze `winrt/windows/media/playback/__init__.py` for its imports and transitively bundle the dependencies. It does NOT: modulegraph scans Python source but stops at compiled `.pyd` boundaries.

**Why it happens:** `hiddenimports` tells PyInstaller "include this module", but PyInstaller's analysis does not recursively scan a compiled `.pyd` for its Python-side imports. `winrt/windows/media/playback/__init__.py` imports `from winrt._winrt_windows_media_playback import ...` — that import requires the `.pyd` to be present, which modulegraph never discovers.

**How to avoid:** Use `collect_all` which includes the `binaries` output (physical `.pyd` files) via `collect_dynamic_libs()`. [VERIFIED: PyInstaller docs on `collect_all`]

### Pitfall 6: `build.ps1` SMTC guard exit code conflicts

**What goes wrong:** The new exit code for the SMTC smoke guard conflicts with an existing documented exit code (0=ok, 1=env missing, 2=pyinstaller failed, 3=smoke, 4=pkg03, 5=version parse, 6=iscc fail, 7=spec entry guard, 8=pre-bundle clean fail, 9=post-bundle distinfo, 10=plugin presence guard).

**How to avoid:** Use exit code **11** (next available after 10). Update the exit-code table comment at the top of `build.ps1` (line 4-7 block).

### Pitfall 7: The `--check-mediakeys` smoke check requires `QCoreApplication`, not `QApplication`

**What goes wrong:** Using `QApplication` in the `--check-mediakeys` path on a headless build machine causes a crash or hangs waiting for a display.

**How to avoid:** Use `QCoreApplication` only, matching the existing `_run_smoke` headless pattern. The `media_keys.create()` factory only requires a player and repo (both can be None), not a full Qt GUI. [VERIFIED: reading `_run_smoke` source in `__main__.py`]

### Pitfall 8: D-03 logging must not add `exc_info=True` for the normal import-error path

**What goes wrong:** Adding `exc_info=True` to the fallback warning log prints a full stack trace on every Windows launch that has the `[windows]` extras missing (e.g. development environments on Windows without pywinrt installed). This is noise.

**How to avoid:** Use `_log.warning("Media keys: falling back to NoOp on win32 — %s: %s", type(e).__name__, e)` without `exc_info=True`. The error type and message are sufficient for diagnosis. Reserve `exc_info=True` for unexpected exception types (not ImportError). [ASSUMED — this is a style judgment; planner confirms]

## Open Questions

1. **Does msvcp140.dll from winrt-runtime conflict with any other msvcp140.dll in the bundle?**
   - What we know: `winrt-runtime 3.2.1` ships `winrt/msvcp140.dll` inside the `winrt/` subdirectory, which is an unusual location for a CRT DLL.
   - What's unclear: Whether PyInstaller's dedup logic handles two copies of msvcp140.dll (one from GStreamer/conda-forge, one from winrt-runtime) correctly when they land in different subdirectories.
   - Recommendation: Accept it for now; if the bundle smoke test fails with a DLL conflict, examine both copies' versions and keep the newer one.

2. **Does the conda-forge Windows build env have all five pywinrt distributions installed?**
   - What we know: `pyproject.toml [windows]` declares all four namespace packages; `pip install -e ".[windows]"` is run in step 3 of build.ps1. The `winrt-runtime` package is a transitive dependency of the namespace packages (each depends on it).
   - What's unclear: Whether `pip install` of the namespace packages automatically installs `winrt-runtime` as a declared dependency.
   - Recommendation: `collect_all("winrt-runtime")` is safe to call even if it was auto-installed transitively; PyInstaller handles missing distributions gracefully (returns empty lists). Verify during the VM session.
