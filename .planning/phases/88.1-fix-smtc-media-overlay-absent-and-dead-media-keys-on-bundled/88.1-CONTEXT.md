# Phase 88.1: Fix SMTC media overlay absent and dead media keys on bundled Windows build - Context

**Gathered:** 2026-06-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Make the **bundled** (PyInstaller-frozen) Windows build's SMTC media session actually
register, so the Windows media overlay shows "MusicStreamer" and hardware media keys
control play/pause/stop. This fixes Phase 88 UAT gap **G2** (UAT-3, UAT-7) and unblocks
**WIN-02** / **VER-02-J**.

Scope is the **bundling and registration** of the already-working `smtc.py` backend — NOT
rewriting the SMTC backend logic (that code passed Phase 43.1 UAT in the dev/conda env).
G2 is a "works in dev, dies in the frozen bundle" packaging defect: the win32 factory
catches the winrt `ImportError` and silently degrades to `NoOpMediaKeysBackend`, so no
media session is presented at all.

**Out of scope:** G3 (GBS.FM login — separate phase 88.2), the AUMID shortcut mechanism
(already wired — see Code Context), and any new SMTC features (next/previous, seek).
</domain>

<decisions>
## Implementation Decisions

### Root-cause approach — diagnose-first, but ship a provisional fix
- **D-01:** Treat `collect_all("winrt")` + `winrt-runtime` collection in the PyInstaller
  spec as the **leading hypothesis** for why winrt fails to import in the frozen exe
  (the spec currently only declares winrt namespaces as `hiddenimports`, with no
  `collect_all` for the compiled pywinrt projection or the winrt runtime core).
- **D-02:** Phase 88.1 ships **diagnostics AND a provisional fix together** (chosen over
  diagnostics-only). Land the backend-selection logging and the leading-hypothesis
  bundling fix in the same phase. The diagnostic Windows build (in 88-03) then either
  confirms the fix green via the smoke guard, or the logging pinpoints exactly what else
  must be collected. Diagnosis **validates** the fix rather than gating it.
- **D-03:** Add **backend-selection logging** to `media_keys/__init__.py` (and/or
  `smtc.create_windows_backend`): on win32, log which backend was selected and, on
  fallback to NoOp, log the underlying winrt import error at a visible level — so the
  silent-degradation that hid G2 can never again ship without a diagnosable trace.

### Build-time regression guard — both layers
- **D-04:** Add a **Linux-runnable static spec test** (analog of
  `tests/test_packaging_linux_spec.py`) asserting `packaging/windows/MusicStreamer.spec`
  collects the winrt runtime + projection (e.g. asserts `collect_all("winrt")` / the
  winrt-runtime distribution is wired). Catches spec regressions in normal CI without a
  Windows build.
- **D-05:** Extend the **`build.ps1` / `--smoke` runtime step** to assert the frozen exe
  constructs the **real `WindowsMediaKeysBackend` (not `NoOpMediaKeysBackend`)** on
  win32, failing the Windows build with a **new exit code** if SMTC didn't load. Follows
  the existing exit-code-guard house style (exit 3 = smoke, 8/9/10 = bundle guards).
- **Why both:** belt-and-suspenders — the static test guards the spec in CI; the runtime
  smoke assertion proves the frozen exe actually imports winrt on the build machine.

### Phase completion boundary — Linux-side now, VM proof deferred to 88-03
- **D-06:** Phase 88.1 **closes on the Linux/CI side**: the `.spec` fix + factory logging
  + the Linux static spec test land and pass in CI. The executor runs on Linux and
  **cannot** run the Windows build or the frozen smoke test.
- **D-07:** The Windows build's `build.ps1` smoke guard **and** the interactive UAT-3 /
  UAT-7 re-test both run together in the **consolidated 88-03 Win11 VM session**
  (Option C, 2026-06-09). `must_haves` MUST be split into:
  - **Linux-verifiable now:** spec collects winrt; static spec test passes; factory logs
    backend selection + fallback reason; `build.ps1` contains the new SMTC smoke
    assertion + documented exit code.
  - **VM-deferred (88-03):** Windows build smoke guard passes (SMTC backend constructs in
    the frozen exe); UAT-3 overlay shows "MusicStreamer"; UAT-7 media keys control
    playback.

### Claude's Discretion
- Exact `collect_all` invocation shape and which winrt sub-distributions to enumerate
  (research the pywinrt 3.2.x packaging layout — winrt-runtime `_winrt` core, per-namespace
  compiled `.pyd` modules, projection metadata).
- The new `build.ps1` exit code number and the precise smoke-harness assertion mechanism
  (extend the existing `--smoke` path in `__main__.py` vs. a dedicated `--check-mediakeys`
  flag) — keep it Qt-light, consistent with the current `_run_smoke` harness.
- Log level/wording for backend-selection diagnostics.
- Whether the runtime hook (`runtime_hook.py`) needs any winrt-related env setup.
</decisions>

<specifics>
## Specific Ideas

- The silent-degradation behavior (`media_keys/__init__.py` catching the winrt ImportError
  and returning NoOp with only `_log.warning`) is the reason G2 shipped undetected. The
  fix should make a Windows-build SMTC failure **loud at build time** (D-05) and
  **diagnosable at runtime** (D-03), not just silent.
- pywinrt is pinned `>=3.2,<4` in `pyproject.toml` `[project.optional-dependencies].windows`;
  the spec's winrt namespace layout (`winrt.windows.media`, `.playback`, `.storage.streams`,
  `.foundation`) was validated against the 3.2.x layout in Phase 43.1 (see `smtc.py` lines 18-24).
</specifics>

## Existing Code Insights

### Reusable Assets
- **`--smoke` harness** (`musicstreamer/__main__.py` `_run_smoke`): already runs the
  `media_keys.create()` factory under QCoreApplication. Extend it (or add a sibling check)
  to assert a non-NoOp Windows backend for D-05 — no new harness needed.
- **`collect_all` pattern** (`MusicStreamer.spec` lines 24-33): charset_normalizer,
  streamlink, yt_dlp are already collected this way; winrt fix mirrors this exactly.
- **Exit-code-guard pattern** (`build.ps1`): exit 3/8/9/10 guards with `Invoke-Native` +
  `Write-Host`/`exit N` (NOT `Write-Error`, per Phase 65 WR-01) — D-05's new exit code
  must follow this to avoid PS 5.1 swallowing the code.
- **`test_packaging_linux_spec.py`**: direct structural analog for D-04.

### Established Patterns
- winrt imports are deferred to function bodies (D-06 contract in smtc.py) so the module
  stays Linux-importable — the Linux static spec test (D-04) asserts spec wiring, it does
  NOT import winrt.
- Factory NEVER raises (D-06/T-41-01) — all failure modes return NoOp. D-03 logging must
  preserve this contract (log + degrade, never block startup).

### Integration Points
- The AUMID labeling mechanism is already complete (`.iss` `[Icons]` + `__main__.py`
  ctypes call, UAT-2 passed). Once SMTC sessions register, UAT-3's "MusicStreamer" label
  follows automatically — do NOT re-open AUMID work.
- The diagnostic + smoke-guard results surface during the **consolidated 88-03 Win11 VM
  session**; 88.1's executor work is Linux-only.
