## Summary

This phase adds five coherent changes: (1) `collect_all` for five winrt distributions in the spec, (2) INFO/WARNING logging in the win32 factory branch, (3) `_run_check_mediakeys` / `--check-mediakeys` in `__main__.py`, (4) step 4c SMTC smoke guard in `build.ps1`, and (5) two new test functions in the existing SMTC test file plus a new drift-guard test module.

The core architecture is sound: the "never raises" factory contract is preserved, the `except Exception` was not narrowed, QCoreApplication is used (not QApplication) in `_run_check_mediakeys`, the exit-code contract is correct, Write-Host (not Write-Error) is used in the step 4c failure path, and exit 11 is correctly documented in the header. The winrt distribution names in `collect_all` calls use the correct PyPI distribution name format.

One critical gap was found: `pyproject.toml [project.optional-dependencies].windows` does not include `winrt-runtime`, the core pywinrt bridge. This omission means `pip install -e ".[windows]"` relies on `winrt-runtime` arriving only as a transitive dependency of the namespace packages, which is currently true on PyPI but is not guaranteed. The spec correctly calls `collect_all("winrt-runtime")` but the package is not explicitly declared as a runtime dependency.

Three warnings and one info item are also documented below.

## Critical Issues

### CR-01: `winrt-runtime` missing from `pyproject.toml` `[windows]` optional-deps

**File:** `pyproject.toml` (not in the diff, but directly caused by this phase's spec change)

**Issue:** `MusicStreamer.spec` calls `collect_all("winrt-runtime")` at line 42, correctly identifying it as the core pywinrt bridge whose `.pyd` is required in the frozen bundle. However, `pyproject.toml [project.optional-dependencies].windows` only lists the four namespace packages (`winrt-Windows.Media.Playback`, `winrt-Windows.Media`, `winrt-Windows.Storage.Streams`, `winrt-Windows.Foundation`) — `winrt-runtime` is absent. On PyPI, pywinrt 3.x namespace packages declare `winrt-runtime` as a transitive dependency, so `pip install -e ".[windows]"` happens to pull it in today. If pywinrt ever drops that transitive declaration (as happened in past pywinrt 2.x → 3.x reshuffles, which is why the `>=3.2,<4` pin was added in the first place), a developer install with `.[windows]` would fail to install the core bridge, the `collect_all("winrt-runtime")` call in the spec would fail with a `PackageNotFoundError`, and the bundle build would break with no actionable error pointing at the pyproject gap.

The test `test_pyproject_has_windows_optional_deps` (line 140, `tests/test_media_keys_smtc.py`) explicitly enumerates `expected_pkg_prefixes` but omits `winrt-runtime`, so this gap has no regression coverage.

**Fix:** Add `winrt-runtime>=3.2,<4` to the `[project.optional-dependencies].windows` list in `pyproject.toml` and add `"winrt-runtime"` to `expected_pkg_prefixes` in `test_pyproject_has_windows_optional_deps`:

```toml
# pyproject.toml
windows = [
    "winrt-runtime>=3.2,<4",
    "winrt-Windows.Media.Playback>=3.2,<4",
    "winrt-Windows.Media>=3.2,<4",
    "winrt-Windows.Storage.Streams>=3.2,<4",
    "winrt-Windows.Foundation>=3.2,<4",
]
```

```python
# tests/test_media_keys_smtc.py, line 157
expected_pkg_prefixes = {
    "winrt-runtime",
    "winrt-Windows.Media.Playback",
    "winrt-Windows.Media",
    "winrt-Windows.Storage.Streams",
    "winrt-Windows.Foundation",
}
```

## Warnings

### WR-01: `MEDIAKEYS_BACKEND=` diagnostic print is silenced in frozen windowed bundle

**File:** `musicstreamer/__main__.py:38`

**Issue:** `_run_check_mediakeys` calls `print(f"MEDIAKEYS_BACKEND={class_name}", flush=True)`. The bundle is built with `console=False` (spec line 188), which means PyInstaller sets `sys.stdout` to `None` in the frozen process. In Python 3, `print()` to `None` stdout is a documented no-op (it does not raise), so the exit-code contract (0/1) works correctly and the SMTC guard functions as intended. However, the diagnostic output `MEDIAKEYS_BACKEND=WindowsMediaKeysBackend` is silently dropped when `build.ps1` runs `MusicStreamer.exe --check-mediakeys 2>&1 | Out-Host`. A build engineer investigating a step 4c failure sees only the exit code and the BUILD_FAIL hint message, not the actual backend class name that was returned. When the guard fails due to a partial winrt collection (e.g., only some `.pyd` files land), the class name would confirm whether the factory fell back to `NoOpMediaKeysBackend` or crashed entirely.

**Fix:** Guard the print against `None` stdout, or redirect diagnostic output to stderr (which is captured by `2>&1` in the pipe regardless of windowed mode):

```python
# Option A: write diagnostic to stderr so it appears in build.log
import sys as _sys
print(f"MEDIAKEYS_BACKEND={class_name}", file=_sys.stderr, flush=True)

# Option B: guard both streams
if _sys.stdout is not None:
    print(f"MEDIAKEYS_BACKEND={class_name}", flush=True)
print(f"MEDIAKEYS_BACKEND={class_name}", file=_sys.stderr, flush=True)
```

### WR-02: SMTC smoke guard (step 4c) has no skip switch — cannot be bypassed in offline CI

**File:** `packaging/windows/build.ps1:321-329`

**Issue:** Steps 4a (dist-info assertion) and 4b (plugin guard) are mandatory with no bypass switches, which is appropriate since they inspect the bundle contents directly. Step 4c, by contrast, actually *executes* the frozen `MusicStreamer.exe`. On a Windows build machine that has network/display restrictions (e.g., a headless Windows Server CI runner where `QCoreApplication` fails to initialize due to missing platform plugins), step 4c would fail with a non-zero exit code and the build would die at exit 11 even though the bundle contents are correct. The `$SkipSmoke` switch (line 13) only gates step 5, not step 4c. No `$SkipSmtcGuard` or equivalent switch exists.

The comment "Only runs on Windows" is also misleading: there is no `if ($IsWindows)` guard in the script; the comment refers to the fact that `.ps1` is a Windows-only script in practice, not a runtime gate.

**Fix:** Add a `$SkipSmtcGuard` parameter (defaulting `$false`) parallel to `$SkipSmoke`, and wrap step 4c:

```powershell
param(
    ...
    [switch]$SkipSmtcGuard = $false
)

# step 4c
if (-not $SkipSmtcGuard) {
    Write-Host "=== SMTC SMOKE GUARD ..."
    Invoke-Native {
        & "..\..\dist\MusicStreamer\MusicStreamer.exe" --check-mediakeys 2>&1 | Out-Host
    }
    if ($LASTEXITCODE -ne 0) {
        Write-Host "BUILD_FAIL reason=smtc_backend_not_loaded ..." -ForegroundColor Red
        exit 11
    }
    Write-Host "SMTC SMOKE GUARD OK"
} else {
    Write-Host "SMTC SMOKE GUARD skipped (SkipSmtcGuard=true)"
}
```

### WR-03: No unit test for `_run_check_mediakeys` exit-code contract

**File:** `musicstreamer/__main__.py:20-42` / `tests/test_media_keys_smtc.py`

**Issue:** `_run_check_mediakeys` implements a specific exit-code contract: returns `0` only when `type(backend).__name__ == "WindowsMediaKeysBackend"`, returns `1` for any other class name. This string-equality check is the hinge of the entire SMTC smoke guard — if the class is ever renamed or the check logic drifts, the guard silently inverts. No unit test exercises this function directly. The drift-guard test `test_build_ps1_smtc_smoke_guard_present` (line 162, `test_packaging_winrt_spec.py`) only checks that `--check-mediakeys` appears as a string in `build.ps1`; it does not invoke `_run_check_mediakeys` with a mock factory that returns `NoOpMediaKeysBackend` and verify exit code `1`.

**Fix:** Add tests to `tests/test_media_keys_smtc.py` or a new `tests/test_main_check_mediakeys.py`:

```python
def test_run_check_mediakeys_returns_0_for_windows_backend(mock_winrt_modules, monkeypatch, qtbot):
    """_run_check_mediakeys returns 0 when factory selects WindowsMediaKeysBackend."""
    import sys
    from musicstreamer.__main__ import _run_check_mediakeys
    monkeypatch.setattr(sys, "platform", "win32")
    # clear cached modules so factory re-runs
    for m in list(sys.modules):
        if m.startswith("musicstreamer.media_keys"):
            del sys.modules[m]
    result = _run_check_mediakeys([__file__])
    assert result == 0

def test_run_check_mediakeys_returns_1_for_noop(monkeypatch, qtbot):
    """_run_check_mediakeys returns 1 when factory falls back to NoOpMediaKeysBackend."""
    import sys
    from musicstreamer.__main__ import _run_check_mediakeys
    # Linux: factory always returns NoOp
    result = _run_check_mediakeys([__file__])
    assert result == 1
```

**File:** `tests/test_media_keys_smtc.py:11`

**Issue:** Line 11 imports `unittest.mock` as `mock` with a `# noqa: F401` suppression. The `mock` alias is never referenced in the file; all mock usage goes through the directly imported `AsyncMock` and `MagicMock` names (line 13). The `# noqa` suppression hides what would otherwise be a lint signal to remove the dead import. This is a pre-existing line that predates Phase 88.1, but it appears in the reviewed file.

**Fix:** Remove the unused import alias:
