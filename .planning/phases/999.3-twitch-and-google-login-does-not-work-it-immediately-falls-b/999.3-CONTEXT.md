<decisions>
## Implementation Decisions

### Twitch OAuth Redesign (REVISED post-research — 2026-04-24)

**Context for revision:** Research (RESEARCH.md) discovered two blocking facts after discussion:
1. Twitch does NOT support PKCE for Authorization Code flow (confirmed via Twitch dev forums Dec 2024). Original D-04 unexecutable.
2. Tokens from a newly-registered dev.twitch.tv app are Helix Bearer tokens and are NOT accepted by streamlink's private GQL endpoint (streamlink upstream maintainer statement). So registering our own Twitch app (original D-01) would yield a token useless for MusicStreamer's Twitch playback path. The piggyback web client_id (kimne78...) is the ONLY token source compatible with streamlink.

**User decision after research:** Path X — keep piggyback + rebuild capture via local HTML bounce page. D-01 through D-06 are superseded.

- **D-01 (REVISED):** Keep Twitch's web client_id `kimne78kx3ncx6brgo4mv6wki5h1ko` hardcoded in `oauth_helper.py`. Do NOT register a new MusicStreamer Twitch app. Rationale: it's the only client_id whose tokens streamlink accepts on the private GQL API.
- **D-02 (REVISED):** Stay on **implicit flow** (`response_type=token`). The token arrives in the URL fragment after successful login (`#access_token=...`). Scope remains empty (`scope=`) — matches current working pattern.
- **D-03 (NEW):** Redirect URI is `http://localhost:PORT` (fixed port — planner picks e.g. 17823 or 49152+, verify not IANA-registered). A real HTTP listener runs inside the `oauth_helper.py` subprocess on 127.0.0.1 only (RFC 8252 §8.3). Serves a single local HTML page in response to the redirect.
- **D-04 (NEW — replaces failed urlChanged pattern):** The loopback listener serves a local HTML page whose JS reads `window.location.hash`, extracts `access_token`, and POSTs it to the listener via `fetch('/capture', { method: 'POST', body: token })`. This works because the fragment IS available to JS running on the redirect page (fragments are never sent to servers in the HTTP request line, but JS on the loaded page can read them). Listener receives the token via POST body and signals the Qt main thread.
- **D-05:** Cross-thread signaling: HTTP server runs in a background `threading.Thread`; receipt of the token triggers a `pyqtSignal.emit` (or a `threading.Event` + `QTimer` poll) on the main Qt thread, which closes the QWebEngineView and prints the token to stdout. Parent (`AccountsDialog`) receives it via existing `QProcess` stdout capture.
- **D-06:** State parameter (`state=<secrets.token_urlsafe(16)>`) added for CSRF protection. Validated on redirect before accepting the token.
- **D-07:** `oauth_helper.py` `--mode twitch` is rewritten. `--mode google` remains UNTOUCHED (working).

### Failure UX

- **D-08:** Failure dialog shows **category + detail** (not just "Try again"). Categories: `LoginTimeout`, `WindowClosedBeforeLogin`, `TwitchRejectedRequest` (error query param from redirect), `InvalidTokenResponse` (state mismatch, empty token), `SubprocessCrash`. Detail line carries the underlying error text (stderr excerpt, exception message).
- **D-09:** Failure dialog has **Retry** + **Close** buttons. Retry relaunches the OAuth subprocess inline without closing `AccountsDialog` — user does not need to re-click "Connect Twitch" from the parent dialog.
- **D-10:** Persistent log file at `~/.local/share/musicstreamer/oauth.log` (path determined by `paths.py` — planner adds an `oauth_log_path()` helper). Size-rotated via `logging.handlers.RotatingFileHandler`. Permissions `0o600`.
- **D-11:** Log entries include **failure events** (timestamp, category, detail, provider=twitch) **and success events** (timestamp, provider=twitch, "connected"). Redirect URLs and any token material are NOT logged — scrubbed at log-write time.
- **D-12:** Subprocess (`oauth_helper.py`) emits structured diagnostics on stderr (JSON lines) so the parent can parse category + detail for the dialog and forward to the log file. Parent captures `readAllStandardError()` on `QProcess.finished`.

### Claude's Discretion

- Exact fixed port number for the loopback listener (planner picks an uncommon port in 49152+ dynamic range; verify not IANA-registered).
- Log format details (JSON lines preferred).
- HTTP listener implementation (`http.server.BaseHTTPRequestHandler` stdlib — no new deps).
- Exact HTML bounce page content/styling ("Logged in — you can close this window" + auto-close JS).
- Whether the subprocess exits after first successful `/capture` POST or serves until timeout.
- Timeout duration for login (suggest 120s).
- Cross-thread signal mechanism (pyqtSignal vs threading.Event + QTimer poll).

<specifics>
## Specific Ideas

- **Scope narrowing mid-discussion:** User confirmed YouTube/Google login is now working — remove from phase scope. Twitch-only fix.
- **Root failure mode (user's words):** "it still can't complete login capture" — browser login succeeds, but the subprocess never captures the token. Consistent with `urlChanged` not firing for `http://localhost` redirects when no listener is running (connection-refused URLs never complete navigation, so the signal may never carry the final URL).
- **Piggyback client_id reversal:** User initially rejected keeping `kimne78...`, but research proved it's the only token source streamlink accepts. User re-opted for Path X (keep piggyback, fix capture mechanism). See D-01 (REVISED) for rationale.
- **No new Twitch app registration needed** — no manual step from Kyle; planner can execute end-to-end.
- **Why JS bounce page works where urlChanged doesn't:** URL fragments (`#access_token=...`) are never transmitted to servers in the HTTP request line, but ARE accessible to JavaScript running on the loaded page via `window.location.hash`. Serving a real HTML page from the loopback listener guarantees the page loads (no connection-refused), and its JS can read the fragment and POST it back.

</specifics>
