<decisions>
## Implementation Decisions

### Twitch OAuth Redesign

- **D-01:** Register a proper MusicStreamer Twitch app on dev.twitch.tv. **Kyle** registers the app manually (requires Twitch account login — Claude cannot do this step). Kyle provides the client_id + configured redirect_uri back to the planner/executor.
- **D-02:** The new client_id is embedded in code (public OAuth client pattern — Twitch permits this for native/desktop apps). Stored as a constant in `musicstreamer/constants.py` or `oauth_helper.py` alongside the existing Twitch URL constants.
- **D-03:** Redirect URI is `http://localhost:PORT` (fixed port TBD by planner — e.g., 17823 or similar unlikely-to-collide value). A real HTTP listener runs inside the `oauth_helper.py` subprocess to receive the redirect reliably. This removes the fragile dependency on `urlChanged` catching an unloadable URL.
- **D-04:** OAuth flow is **Authorization Code + PKCE** (not implicit token, not plain auth code). Subprocess generates PKCE code_verifier + code_challenge, opens Twitch authorize URL with `response_type=code&code_challenge=...&code_challenge_method=S256`, listens on the loopback port, receives `?code=...` from redirect, exchanges code for access token via POST to `https://id.twitch.tv/oauth2/token` with `grant_type=authorization_code` + verifier (no client_secret). Writes token to `paths.twitch_token_path()` with `0o600` as before.
- **D-05:** `oauth_helper.py` `--mode twitch` is rewritten for the new flow. `--mode google` is untouched (working).
- **D-06:** Planner must verify Twitch currently supports PKCE for native clients. If PKCE is not supported (Twitch historically required client_secret for code flow), fall back to either:
  - (a) ship client_secret embedded (acceptable for desktop apps per OAuth 2.1 discussion — public clients), or
  - (b) return to implicit flow but switch from `urlChanged`-fragment-capture to loading a local HTML page that parses the fragment via JS and POSTs it to the loopback listener.
  Decision deferred to research phase.

### Failure UX

- **D-07:** Failure dialog shows **category + detail** (not just "Try again"). Categories: `LoginTimeout`, `WindowClosedBeforeLogin`, `TwitchRejectedRequest` (4xx from token endpoint), `InvalidTokenResponse`, `SubprocessCrash`. Detail line carries the underlying error text (stderr excerpt, HTTP status + body, or exception message).
- **D-08:** Failure dialog has **Retry** + **Close** buttons. Retry relaunches the OAuth subprocess inline without closing `AccountsDialog` — user does not need to re-click "Connect Twitch" from the parent dialog.
- **D-09:** Persistent log file at `~/.local/share/musicstreamer/oauth.log` (path determined by `paths.py` — planner adds a helper). Size-rotated (keep last N KB or last N entries — planner picks). Permissions `0o600`.
- **D-10:** Log entries include **failure events** (timestamp, category, detail, provider=twitch) **and success events** (timestamp, provider=twitch, "connected"). Redirect URLs and any token material are NOT logged — scrubbed at log-write time.
- **D-11:** Subprocess (`oauth_helper.py`) emits structured diagnostics on stderr (JSON lines or key=value) so the parent can parse category + detail for the dialog and forward to the log file. Parent captures `readAllStandardError()` on `QProcess.finished`.

### Claude's Discretion

- Exact fixed port number for the loopback listener (planner picks an uncommon port; verify not in IANA registered range).
- Log rotation strategy (size-based, count-based, or single-file-with-truncation).
- Log format (JSON lines vs plain text). Structured preferred.
- HTTP listener implementation (`http.server.BaseHTTPRequestHandler` stdlib vs `socketserver`). Stdlib only — no new deps.
- HTML response served after successful redirect ("You can close this window").
- Whether the subprocess exits after first successful redirect or serves until timeout.
- Timeout duration for login (suggest 120s).

</decisions>

## Existing Code Insights

### Reusable Assets
- **`paths.twitch_token_path()`** — token storage path, reuse unchanged.
- **`paths.cookies_path()`, `paths.accent_css_path()`** — establishes the pattern for adding `oauth_log_path()`.

- **`constants.clear_twitch_token()`** — existing disconnect helper, unchanged.
- **`PySide6.QtWebEngineWidgets`** — verified installed on dev machine (not the cause of current failure).

### Established Patterns
- Subprocess-isolated QWebEngineView (Phase 40 D-02) — keep. Don't pull 130MB QtWebEngine into main process.
- Bound-method Qt signal slots (QA-05 convention).
- `sys.executable` + `-m musicstreamer.oauth_helper` for subprocess launch (T-40-05 — no PATH injection).
- `0o600` permissions on credential files immediately after write (T-40-03).
- `Qt.TextFormat.PlainText` on labels showing user-visible error text (T-40-04 — no rich-text injection).

### Integration Points
- `AccountsDialog._on_oauth_finished` — replace "failed, try again" QMessageBox with the new category-detail dialog with Retry.
- `oauth_helper.py` module-level `TWITCH_AUTH_URL` constant — becomes function that builds URL with PKCE challenge + new client_id + new redirect_uri.
- New class `_TwitchLoopbackServer` (or similar) inside `oauth_helper.py` for the HTTP listener. Stdlib `http.server` — no new deps.

### Security Threat Surface
- New: loopback HTTP server in subprocess. Must bind to `127.0.0.1` only (never `0.0.0.0`). Accept single request then shut down. Validate `state` parameter in redirect to prevent CSRF.
- New: PKCE code_verifier lives in subprocess memory only — never persisted, never logged.
- Unchanged: token file permissions, subprocess isolation, `sys.executable` launch.

</code_context>
